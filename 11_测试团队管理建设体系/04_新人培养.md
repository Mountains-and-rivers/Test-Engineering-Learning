## 一、测试新人90天培养计划

### 1.1 90天培养全景图

```
第1周          第2-4周          第2个月          第3个月          第90天
入门阶段        基础阶段          成长阶段          独立阶段          转正评估
  │              │                │                │                │
  ▼              ▼                ▼                ▼                ▼
环境与工具    测试理论与      独立负责小      主导一个模块      转正答辩
文化融入      流程培训        型需求测试      完整测试交付      + 考核评语
  │              │                │                │                │
  ▼              ▼                ▼                ▼                ▼
目标：        目标：           目标：           目标：           目标：
认识团队      掌握基础技能     能在指导下       能独立完成       证明胜任力
了解业务      了解业务流程     执行测试        需求测试         成为正式员工
```

### 1.2 第一阶段：入门期（第1周）

**目标：** 融入团队、熟悉环境、了解业务

| 天数 | 学习/工作内容 | 产出物 | 指导人 |
|-----|------------|-------|-------|
| Day 1 | 入职办理、团队介绍、分配工位和电脑 | - | HR+导师 |
| Day 2 | 账号权限开通、开发环境搭建、阅读团队Wiki | 环境搭建记录 | 导师 |
| Day 3 | 业务培训：公司产品全景介绍、核心业务流程 | 业务笔记 | 导师+产品 |
| Day 4 | 技术培训：技术架构概览、测试环境介绍、工具链入门 | 技术笔记 | 导师+开发 |
| Day 5 | 跑通一个完整的测试流程Demo（从需求到Bug提交） | Demo执行记录 | 导师 |

**第一周每日检查清单：**

```markdown
# 新人第一周每日Check-in（导师填写）

## Day 1
- [ ] 入职手续完成
- [ ] 团队介绍完成（认识每位团队成员）
- [ ] 电脑/显示器等设备就绪
- [ ] 新人是否有不清楚的问题需要解答？

## Day 2
- [ ] 账号开通完毕（邮箱/IM/Jira/Wiki/Git/数据库等）
- [ ] 开发环境搭建完成
- [ ] 至少看完了团队Wiki首页

## Day 3
- [ ] 完成业务培训，能画出核心业务流程图
- [ ] 了解产品的主要用户群体和核心场景

## Day 4
- [ ] 了解系统技术架构（前端/后端/数据库/中间件）
- [ ] 能独立登录测试环境并访问功能

## Day 5
- [ ] 完成测试流程Demo（需求→用例→执行→Bug提报）
- [ ] 提交至少1个模拟Bug（练习Bug提交流程）
```

### 1.3 第二阶段：基础期（第2-4周）

**目标：** 掌握基础技能，能执行简单测试任务

| 周次 | 学习内容 | 实践任务 | 产出物 |
|-----|---------|---------|-------|
| 第2周 | 测试方法培训、用例设计练习 | 在导师指导下设计简单模块用例 | 用例设计练习 |
| 第3周 | 接口测试入门（Postman）、SQL查询练习 | 执行已有用例，参与实际项目测试 | 测试执行记录 |
| 第4周 | Bug定位技巧、抓包工具（Charles/Fiddler） | 独立执行一个简单需求的测试 | 测试总结 |

**第二-四周技能达标要求：**

| 技能项 | 达标标准 |
|-------|---------|
| 用例执行 | 能准确执行预设用例，正确判断Pass/Fail |
| Bug提交 | 能规范提交Bug，包含完整复现步骤和附件 |
| 接口测试 | 能使用Postman发送请求并验证响应 |
| SQL查询 | 能写SELECT + WHERE + JOIN查询验证数据 |
| 抓包工具 | 能使用Charles/Fiddler查看请求和响应 |

### 1.4 第三阶段：成长期（第5-8周）

**目标：** 能够在指导下负责小型需求测试

| 周次 | 核心任务 | 能力提升重点 |
|-----|---------|------------|
| 第5周 | 在导师指导下，独立负责一个小型需求（2-3人天）的测试 | 需求分析、用例设计 |
| 第6周 | 参与接口自动化编写（如团队有自动化框架） | 编程能力、框架使用 |
| 第7周 | 独立负责一个中等需求（3-5人天）的测试 | 独立测试全流程 |
| 第8周 | 参与需求评审、参与用例评审（作为旁听+建议者） | 评审技巧、问题发现 |

### 1.5 第四阶段：独立期（第9-12周）

**目标：** 独立完成完整需求的测试交付

| 周次 | 核心任务 | 能力提升重点 |
|-----|---------|------------|
| 第9-10周 | 独立负责一个完整需求（5-8人天）的测试全流程 | 全流程管控能力 |
| 第11周 | 总结90天工作，整理岗位学习笔记 | 知识沉淀 |
| 第12周 | 准备转正答辩材料 + 参加转正答辩 | 综合展现 |

## 二、导师制实施方案

### 2.1 导师选拔标准

| 选拔维度 | 标准 |
|---------|-----|
| 职级要求 | P1（中级）及以上 |
| 入职年限 | 入职1年以上 |
| 绩效要求 | 最近两次绩效评估≥B+ |
| 软技能 | 沟通能力强，有耐心，乐于分享 |
| 意愿度 | 自愿报名，非强制指派 |

### 2.2 导师职责

```markdown
# 导师职责清单

## 入职前准备
- [ ] 确认新人电脑、账号、工位已就绪
- [ ] 制定90天培养计划（针对新人背景定制）
- [ ] 准备第1周学习资料（文档清单、视频链接）

## 第1-4周（密集辅导期）
- [ ] 每日至少与新人沟通1次（面对面/IM）
- [ ] 每日检查新人学习进度和学习笔记
- [ ] 回答新人遇到的所有问题（或引导找到答案）
- [ ] 每周输出1份新人成长周报给Leader
- [ ] 带新人一起吃饭，帮助融入团队

## 第5-8周（指导期）
- [ ] 每日可通过IM随时沟通
- [ ] 每周至少1次面对面1on1（30min）
- [ ] 评审新人设计的用例，给出修改建议
- [ ] 逐步放手让新人独立完成任务
- [ ] 每两周输出新人成长报告

## 第9-12周（辅助期）
- [ ] 每周至少1次1on1
- [ ] 帮助新人准备转正答辩
- [ ] 输出新人最终评价报告

## 转正后（跟踪期）
- [ ] 转正后持续跟踪3个月
- [ ] 帮助新人制定第一个半年的成长计划
```

### 2.3 导师激励措施

| 激励方式 | 说明 |
|---------|-----|
| 导师津贴 | 每月X元导师补贴（辅导期间） |
| 绩效加分 | 导师工作纳入绩效"团队贡献"维度 |
| Mentor of the Quarter | 每季度评选最佳导师，给予荣誉+奖金 |
| 晋升加分 | 担任导师是晋升P2及以上职级的参考条件之一 |

### 2.4 导师工作质量评估

**评估方式（每季度）：**

| 评估维度 | 评估方式 | 权重 |
|---------|---------|-----|
| 新人转正通过率 | 辅导新人转正通过情况 | 30% |
| 新人满意度 | 新人匿名评价（5分制） | 30% |
| 新人成长速度 | 90天成长目标达成度 | 25% |
| Leader评估 | 团队Leader对导师工作的评价 | 15% |

### 3.4 培训效果评估

**四级评估模型（Kirkpatrick Model）：**

| 级别 | 评估内容 | 评估方式 | 时间 |
|-----|---------|---------|-----|
| Level 1：反应 | 学员对培训的满意度 | 培训后问卷调查 | 培训当天 |
| Level 2：学习 | 学到多少知识/技能 | 培训后测验/实操考核 | 培训后1周内 |
| Level 3：行为 | 是否将所学应用到工作 | 导师观察/工作产出检查 | 培训后1-3个月 |
| Level 4：结果 | 对团队/业务的影响 | 绩效数据变化 | 培训后3-6个月 |

## 四、实践项目安排

### 4.1 实践项目设计原则

- **渐进式难度：** 从简单到复杂，逐步提升
- **真实场景：** 基于真实业务需求，不是虚拟练习
- **可交付性：** 每个项目都有明确的交付物
- **有反馈：** 每个项目完成后导师给出详细反馈

### 4.2 实践项目路线图

```
项目1：Bug探索（第2周）
  ├── 难度：★☆☆☆☆
  ├── 内容：在测试环境中自由探索，提交10个Bug
  └── 目标：熟悉产品、练习Bug提交流程

项目2：用例设计（第3-4周）
  ├── 难度：★★☆☆☆
  ├── 内容：针对一个已上线的简单功能，设计20条测试用例
  └── 目标：掌握用例设计方法，理解功能测试流程

项目3：简单需求测试（第5-6周）
  ├── 难度：★★★☆☆
  ├── 内容：独立负责一个小型需求（2-3人天）的完整测试
  └── 目标：实践测试全流程（需求分析→用例设计→执行→报告）

项目4：接口测试实战（第7-8周）
  ├── 难度：★★★☆☆
  ├── 内容：使用Postman编写30条接口测试用例并执行
  └── 目标：掌握接口测试方法，能独立验证接口逻辑

项目5：中等需求独立交付（第9-10周）
  ├── 难度：★★★★☆
  ├── 内容：独立负责一个中等需求（5-8人天）的测试全流程
  └── 目标：证明独立工作的能力，为转正做准备

项目6（可选）：自动化脚本编写（第11-12周）
  ├── 难度：★★★★☆
  ├── 内容：使用公司自动化框架编写5-10条自动化用例
  └── 目标：建立自动化测试基础能力
```

### 4.3 实践项目评价标准

| 评价维度 | 权重 | 优秀标准 |
|---------|-----|---------|
| 任务完成度 | 30% | 100%完成计划任务 |
| 产出质量 | 30% | 用例设计合理，Bug描述清晰 |
| 主动性 | 15% | 主动发现需求中的问题/主动承担额外任务 |
| 学习总结 | 15% | 每个项目输出学习笔记/总结 |
| 时间管理 | 10% | 按时完成，对工作量有合理预估 |

## 五、新人考核与转正标准

### 5.1 试用期考核节点

```
入职     第30天      第60天      第90天      转正
  │        │          │          │          │
  ▼        ▼          ▼          ▼          ▼
入职    月度Review  月度Review  转正答辩   转正
启动      (摸底)      (追踪)     (决定)   (发正式offer)
```

### 5.2 各阶段考核重点

**第30天 Review：**

| 考核项 | 通过标准 | 评价 |
|-------|---------|-----|
| 工具环境 | 能独立使用Jira/Postman/Charles等基础工具 | Pass/Fail |
| Bug提交 | 提交的Bug格式规范，被退回率<30% | Pass/Fail |
| 用例执行 | 能准确执行已有用例，效率和准确度合格 | Pass/Fail |
| 学习能力 | 完成了第1个月所有学习任务 | Pass/Fail |
| 团队融入 | 与团队成员建立了基本的工作关系 | Pass/Fail |

**第60天 Review：**

| 考核项 | 通过标准 |
|-------|---------|
| 需求理解 | 能理解需求文档并独立设计简单模块的测试用例 |
| 测试执行 | 能独立负责小型需求的测试执行 |
| 沟通能力 | 能就Bug问题与开发进行有效沟通 |
| 进度管理 | 能在计划时间内完成测试任务 |

**第90天转正答辩：**

```
转正答辩材料清单：
1. 个人述职PPT（15-20页）
   ├── 90天工作总结（做了什么，产出什么）
   ├── 重点项目介绍（选1-2个项目详细讲）
   ├── 学习与成长（学到了什么，哪些需要继续提升）
   ├── 自我评价（优点和不足）
   └── 未来规划（下一个半年的成长目标）

2. 工作产出汇总
   ├── 提交的Bug列表（数量+质量分析）
   ├── 编写的测试用例集
   ├── 测试报告
   └── 学习笔记/技术文章

3. 导师评价报告
```

### 5.3 转正评估标准

| 评估维度 | 权重 | 优秀（90-100分） | 良好（75-89分） | 及格（60-74分） | 不及格（<60分） |
|---------|-----|----------------|----------------|----------------|----------------|
| 工作能力 | 30% | 能独立完成中等需求测试，质量高 | 能独立完成简单需求测试 | 需较多指导 | 无法独立完成工作 |
| 学习能力 | 25% | 主动学习，快速掌握新技能 | 按要求学习，能掌握 | 学习速度偏慢 | 学习效果差，技能未达标 |
| 沟通协作 | 20% | 主动沟通，跨角色协作顺畅 | 能有效沟通 | 基本沟通OK | 沟通障碍 |
| 工作态度 | 15% | 积极主动，责任心强 | 认真负责 | 基本尽责 | 态度敷衍 |
| 文化契合 | 10% | 完全融入团队价值观 | 基本融入 | 无明显不适 | 价值观冲突 |

## 六、5个新人培养实战案例

### 案例1：校招新人培养——从零到独立

**背景：** 计算机专业应届毕业生，无测试经验，有Python基础

**培养方案：**
```markdown
第1-2周：密集培训期
- 安排TEST-101～108基础课程（前2周学完）
- 导师每天陪跑（每日1on1 15min）
- 重点：从"学生思维"转变为"工程师思维"

第3-4周：手把手指导期
- 导师与新人结对测试（Pair Testing）
- 导师先演示，新人模仿；新人操作，导师实时指导
- 重点：规范的Bug提交、准确的用例执行

第5-8周：逐步放手期
- 先从简单模块开始独立（看看日志、检查数据）
- 逐步过渡到中等模块
- 导师从"手把手"变为"结果检查"

第9-12周：独立交付期
- 独立负责一个完整需求
- 导师只需Review最终产出

关键成功因素：
- 校招新人学习意愿强，培训效果明显
- 导师耐心细致，建立了良好的信任关系
- 逐步放手的节奏控制得当，新人始终在"学习区"
```

### 案例2：社招中级跳槽——快速融入

**背景：** 3年测试经验（外包），跳槽到互联网大厂，需要快速适应

**培养方案：**
```markdown
差异化策略（与校招不同）：
- 不安排基础测试理论培训（已有经验）
- 重点放在：公司工具链、业务知识、技术规范
- 期望第2周就能开始参与实际项目

第1周：
- 业务全景培训（公司产品线、技术架构）
- 工具链培训（Jira/TestRail/GitLab CI）
- 技术规范学习（代码规范、用例规范、Bug规范）

第2-3周：
- 跟随导师参与一个正在进行的项目
- 先执行导师设计的用例，快速上手
- 输出一份"与原公司流程对比"的观察报告（帮助团队发现改进点）

第4周起：
- 独立负责需求测试
- 每月与导师一次1on1回顾

关键挑战与应对：
- 挑战：有经验的习惯难以改变（如Bug提交不规范）
- 应对：第1个月严格检查规范性，不符合的退回重写
- 挑战：外包思维→主人翁思维的转变
- 应对：让TA参与需求评审，鼓励提出项目级改进建议
```

### 案例3：跨领域转岗——开发转测试

**背景：** Java开发2年，因个人兴趣转岗测试开发

**培养方案：**
```markdown
优势利用：
- 代码能力强，自动化上手快
- 理解开发思维，与开发沟通顺畅
- 能看懂代码逻辑，Bug定位快

待补短板：
- 测试思维欠缺（习惯于"让它能用"而非"让它不能出错"）
- 测试方法论缺失（不知道边界值、等价类等方法）
- 质量意识不够（对Bug敏感度不如测试老手）

培养路径：
Month 1：建立测试思维
- 学习测试基础理论（2周）
- 跟随高级测试工程师学习用例设计（1周）
- 用开发技能写测试工具（1周）

Month 2：发挥代码优势
- 主导接口自动化编写
- 维护测试框架
- 开发小工具辅助测试团队

Month 3：独立交付
- 独立负责一个后端模块的测试
- 同时维护自动化框架

转正结果：代码能力和测试思维融合，成为团队的"技术担当"
```

### 案例4：远程新人培养

**背景：** 测试团队有远程办公成员，新人入职后远程培养

**培养方案：**
```markdown
远程培养的额外挑战：
- 缺乏面对面沟通，沟通效率降低
- 团队归属感难以建立
- "有问题不好意思问"更严重

针对策略：
1. 沟通机制加强：
   - 每日固定30min视频1on1（前2周）
   - 重要事项视频沟通，日常问题IM随时沟通
   - 建立"新人问题日志"共享文档，问题+答案持续记录

2. 工具支撑：
   - 远程桌面共享做实操演示
   - 录制操作视频供反复观看
   - 使用在线白板工具做用例设计协作

3. 文化融入：
   - 线上团队活动邀请参加（周会/分享会/团建游戏）
   - 给其他团队成员布置"带新人"任务（轮流线上交流）
   - 定期安排出差到办公室（如每月1周）

4. 进度管理：
   - 日清日结：每天下班前在IM汇报当日进展
   - 任务拆解更细：每个任务不超过4小时
   - 产出物要求更明确：减少歧义

关键经验：远程培养需要更细致的流程和更频繁的沟通
```

### 案例5：多新人同时入职——批量培养

**背景：** 业务扩张，同一季度入职5名新人（2校招+3社招）

**培养方案：**
```markdown
挑战：
- 导师资源不足（已有导师只有3人）
- 新人间容易相互干扰或产生比较心理
- 批量培训质量难以保证

解决方案：

1. 建立"新人班"机制：
   - 5名新人组成一个学习小组（同批培训，降低导师压力）
   - 集中授课：通用课程统一面授，不用导师各自讲
   - 建立"新人学习进度看板"：公开透明各自进度

2. 导师分配策略：
   - 每位导师最多带2人
   - 按能力匹配：高级测试带校招生，中级测试带社招生
   - 导师之间横向交流（每周导师碰头会15min）

3. 竞争+合作：
   - 个人任务：各自独立完成分配的需求测试
   - 小组任务：共同完成一个功能模块的探索性测试
   - Peer Review：新人间互相评审用例和Bug

4. 里程碑同步：
   - 统一30天/60天/90天考核日期
   - 转正答辩集中安排（节省评委时间）

结果：5名新人全部通过转正，批量培养节省导师总投入约30%
```
