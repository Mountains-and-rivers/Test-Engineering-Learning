## 二、需求评审流程规范

### 2.1 需求评审目的

- 确保需求的可测试性（可验证、可量化、无歧义）
- 提前识别技术风险和业务风险
- 对齐各角色对需求的理解
- 避免需求缺陷流入开发测试阶段（缺陷修复成本：需求阶段1x → 开发阶段10x → 测试阶段100x）

### 2.2 需求评审准入标准

在进入需求评审前，产品经理必须满足以下条件：

| 准入项 | 标准 | 不满足时处理 |
|-------|-----|------------|
| PRD文档完整 | 包含背景、目标、功能描述、交互原型、异常处理 | 打回补充 |
| 交互稿/设计稿 | 交互原型完成 | 打回补充 |
| 数据埋点需求 | 如涉及数据分析，需明确埋点需求 | 打回补充 |
| 前置依赖确认 | 上下游依赖已确认 | 记录风险 |
| 技术可行性 | 开发初步评估可行 | 开发参加预评审 |

### 2.3 需求评审流程

```
Step 1：产品发起评审
├── 产品提前2天发送PRD文档给参与人员
├── 参与人员：产品、开发、测试、UI/UX、运营（可选）
└── 测试人员提前阅读并准备问题清单

Step 2：需求评审会议
├── 产品讲解需求背景、目标、功能细节（15-30min）
├── 开发评估技术方案（10-15min）
├── 测试提出问题与建议（10-15min）
├── 讨论与决议（15-20min）
└── 会议纪要输出（记录决策结果和待办事项）

Step 3：评审结果
├── 通过：进入开发阶段
├── 有条件通过：记录待修改项，产品修改后可进入开发
└── 不通过：产品修改后需重新评审

Step 4：跟踪确认
├── 产品修改PRD后，测试确认修改是否覆盖评审问题
└── 需求变更时，走变更评审流程
```

### 2.4 测试人员需求评审检查清单

| 检查维度 | 检查项 | 状态 |
|---------|-------|-----|
| 需求明确性 | 功能描述是否有歧义 | [ ] |
| 需求完整性 | 是否覆盖正常流程 + 异常流程 + 边界场景 | [ ] |
| 需求可测性 | 验收标准是否可量化 | [ ] |
| 界面交互 | 交互状态是否完整（加载、空数据、错误） | [ ] |
| 数据逻辑 | 数据流转、存储、展示逻辑是否清晰 | [ ] |
| 权限控制 | 不同角色的权限区分是否明确 | [ ] |
| 兼容性 | 是否考虑浏览器/系统版本兼容 | [ ] |
| 性能要求 | 是否有性能指标要求 | [ ] |
| 安全性 | 是否涉及敏感数据/操作 | [ ] |
| 兼容历史 | 是否影响已有功能 | [ ] |
| 数据埋点 | 埋点需求是否明确 | [ ] |
| 异常处理 | 异常情况的交互和提示是否明确 | [ ] |

### 2.5 需求评审常见问题与应对

| 常见问题 | 应对策略 |
|---------|---------|
| PRD写得过于简单 | 制定PRD模板标准，不达标不打回不评审 |
| 评审会变成需求讨论会 | 要求产品会前充分准备，会中只做决策不做发散讨论 |
| 开发主导讨论，测试没有发言机会 | 设定明确的发言顺序和时长分配 |
| 评审时间长，效率低 | 设定会议时限（≤60min），超时则分多次评审 |
| 评审后需求仍然频繁变更 | 建立需求变更流程，变更需走评审，统计变更率作为产品考核 |

## 三、用例评审流程规范

### 3.1 用例评审目的

- 检查用例设计的完整性（是否覆盖所有需求点）
- 检查用例的正确性（测试方法、预期结果是否正确）
- 发现需求理解偏差（测试人员的理解是否与产品一致）
- 知识共享（团队成员互相学习设计思路）

### 3.2 用例评审准入标准

| 准入项 | 标准 |
|-------|-----|
| 需求已评审通过 | 对应需求的PRD已评审通过 |
| 用例设计完成 | 按需求功能点100%设计用例，自检通过 |
| 用例格式规范 | 遵循团队用例模板，包含必要字段 |
| 前置条件准备 | 测试分析文档（脑图/测试要点）已完成 |

### 3.3 三级评审机制

| 级别 | 适用场景 | 参与人 | 评审重点 |
|-----|---------|-------|---------|
| L1：自评审 | 所有用例 | 设计者自己 | 完整性、格式、逻辑自洽 |
| L2：组内评审 | 中等复杂度需求 | 测试团队成员 | 用例覆盖度、测试方法 |
| L3：跨角色评审 | 核心功能、高风险需求 | 测试 + 产品 + 开发 | 业务逻辑正确性、技术可行性 |

### 3.4 用例评审流程

```
Step 1：发起评审
├── 提前1天发送用例（Excel/Xmind）给评审参与人
└── 标注重点评审区域

Step 2：评审会议（≤60min）
├── 用例设计者讲解设计思路和覆盖策略（5-10min）
├── 逐模块走查用例（重点关注：正常流程 / 异常流程 / 边界值 / 状态流转）
├── 评审人提出补充/修改建议
└── 记录评审意见

Step 3：评审结果
├── 通过：用例可直接执行
├── 有条件通过：修改后邮件确认即可
└── 不通过：大幅修改后需重新评审

Step 4：跟踪修改
├── 用例设计者修改用例（评审后2个工作日内）
└── 评审人确认修改内容
```

### 3.5 用例评审检查清单

**功能覆盖维度：**

| 检查项 | 说明 | 状态 |
|-------|-----|-----|
| 正向流程 | 核心业务流程是否完整覆盖 | [ ] |
| 逆向流程 | 异常操作/中断操作是否覆盖 | [ ] |
| 边界值 | 输入边界（最大/最小/特殊值）是否覆盖 | [ ] |
| 状态流转 | 所有状态及其转换是否覆盖 | [ ] |
| 权限控制 | 不同角色权限是否分别设计用例 | [ ] |
| 数据验证 | 数据库/缓存/消息队列数据是否验证 | [ ] |
| 接口测试 | 核心接口是否有对应的接口用例 | [ ] |
| 兼容性 | 浏览器/系统版本是否考虑 | [ ] |
| 性能场景 | 大数据量/高并发场景是否覆盖 | [ ] |

**用例质量维度：**

| 检查项 | 说明 | 状态 |
|-------|-----|-----|
| 前置条件 | 是否明确测试执行的前提条件 | [ ] |
| 测试步骤 | 步骤是否清晰可执行 | [ ] |
| 测试数据 | 是否指定了具体的测试数据 | [ ] |
| 预期结果 | 是否明确且可验证 | [ ] |
| 优先级 | P0/P1/P2优先级是否合理 | [ ] |

## 四、提测规范（开发提测标准）

### 4.1 提测准入标准

**开发在提测前必须完成以下检查（自测Checklist）：**

```markdown
# 开发提测自检清单

## 代码质量
- [ ] 代码已通过Code Review
- [ ] 代码已合并到提测分支
- [ ] 静态代码扫描通过（无严重/阻塞告警）
- [ ] 单元测试通过率100%，核心模块覆盖率≥60%

## 功能自测
- [ ] 主流程功能自测通过
- [ ] 所有接口（含异常返回）自测通过（Postman/Swagger）
- [ ] 页面适配主流浏览器/分辨率

## 文档与信息
- [ ] 提测邮件已发送，包含：
  - 提测版本号/Git Commit ID
  - 需求列表（含变更说明）
  - 部署说明（配置变更、SQL变更、依赖服务）
  - 自测结果截图
  - 已知风险与限制
- [ ] 数据库变更脚本已提供（DBA评审通过）
- [ ] 配置文件变更已同步

## 环境准备
- [ ] 提测环境已部署并可正常访问
- [ ] 测试所需Mock服务/测试账号已就绪
```

### 4.2 提测退回标准

**以下情况测试有权直接退回提测（打回）：**

| 打回条件 | 处理方式 |
|---------|---------|
| 主流程无法走通（P0级别阻塞） | 开发修复后重新提测 |
| 提测邮件信息缺失严重影响测试 | 补齐信息后重新提测 |
| 环境不可用（服务起不来、数据库版本错误） | 环境就绪后重新提测 |
| 版本错误（提测版本与预期不符） | 确认后重新提测 |
| 提测后1小时内发现3个以上P0 Bug | 开发加强自测后重新提测 |

### 4.3 提测流程

```
Step 1：开发完成自测
    ↓
Step 2：开发发送提测邮件 + 更新Jira状态
    ↓
Step 3：测试进行冒烟测试（提测验收）
    ↓
Step 4：冒烟通过 ──→ 进入正式测试
    冒烟不通过 ──→ 打回开发（打回次数纳入考核统计）
    ↓
Step 5：测试过程中发现 → Bug提交 → 开发修复 → 验证
    ↓
Step 6：测试完成 → 输出测试报告 → 进入发布评审
```

### 4.4 提测质量度量

| 度量指标 | 计算方式 | 目标值 |
|---------|---------|-------|
| 提测一次通过率 | 冒烟通过次数 / 总提测次数 | ≥80% |
| 提测打回率 | 打回次数 / 总提测次数 | ≤20% |
| 提测后首日Bug数 | 提测后第1天发现的Bug数 | ≤5个 |
| 严重Bug密度 | P0+P1 Bug数 / 功能点数 | 持续下降趋势 |

## 五、发布上线流程规范

### 5.1 发布审核准入标准

```markdown
# 发布前Checklist

## 测试完成度
- [ ] 所有计划内测试用例已执行完毕
- [ ] P0、P1级Bug已全部关闭（修复+验证通过）
- [ ] P2级Bug关闭率 ≥ 95%
- [ ] 回归测试已执行通过
- [ ] 性能测试通过（如涉及）
- [ ] 安全测试通过（如涉及）

## 文档与审批
- [ ] 测试报告已输出（含质量评估 + 上线建议）
- [ ] 发布上线审批单已完成签批
- [ ] 产品验收通过确认

## 发布准备
- [ ] 发布版本确认（Git Tag / Release版本号）
- [ ] 数据库变更脚本已准备
- [ ] 配置文件已准备（生产环境）
- [ ] 回滚方案已准备
- [ ] 上线负责人已指定
- [ ] 监控告警已配置
```

### 5.2 发布流程

```
阶段一：发布评审（T-2天）
├── 参加人：开发、测试、运维、产品
├── 评审内容：发布内容、影响范围、风险评估、回滚方案
└── 输出：发布评审纪要 + 审批通过

阶段二：灰度发布（T-1天 ~ T日）
├── 先发布到1-2台服务器，观察15-30分钟
├── 灰度验证：核心流程冒烟测试
└── 灰度期间有异常立即回滚

阶段三：全量发布（T日）
├── 分批发布，每批间隔5-10分钟
├── 监控核心指标无异常后继续下一批
└── 全部节点发布完成，观察30分钟

阶段四：上线验证（T日）
├── 线上冒烟测试：核心业务流程验证
├── 线上监控：错误日志、接口响应时间、业务指标
└── 验证通过后，通知相关方

阶段五：线上观察（T日 ~ T+3日）
├── 重点监控时间窗口：发布后72小时
├── 每日线上巡检
└── 异常即时响应处理
```

### 5.3 紧急发布流程（Hotfix）

```
触发条件：线上出现P0级事故（影响核心业务/资金安全/大量用户）

Step 1：立即响应（事故发生后15min内）
├── 拉紧急群 + 同步初步影响
├── 技术负责人判定是否需要Hotfix
└── 如需Hotfix，启动紧急流程

Step 2：修复与测试（2-4小时内）
├── 开发修复 → Code Review（可简化但不可跳过）
├── 测试只验证修复内容 + 核心回归冒烟（精简版）
├── 紧急审批（可电话/IM审批，事后补签）

Step 3：发布（修复完成后）
├── 发布验证（核心流程冒烟）
├── 持续监控至少2小时
└── 事故报告输出（48小时内）
```

### 5.4 发布后质量度量

| 指标 | 计算方式 | 目标值 |
|-----|---------|-------|
| 发布成功率 | 成功发布次数 / 总发布次数 | ≥99% |
| 发布回滚率 | 回滚次数 / 总发布次数 | ≤2% |
| 发布后72h事故数 | 发布后72h内发生的P0/P1事故 | 0 |
| Hotfix频率 | Hotfix次数 / 月 | ≤1次 |

## 六、缺陷管理流程规范

### 6.1 缺陷生命周期

```
                    ┌──────────────────────────┐
                    │         已关闭            │
                    └──────────▲───────────────┘
                               │（验证通过）
                    ┌──────────┴───────────────┐
                    │      待验证（已修复）      │
                    └──────────▲───────────────┘
                               │（开发修复完成）
创建 → 新建 → 开发确认 → 处理中 → 待验证 → 已关闭
                    │
                    ▼
              拒绝/重复/暂不处理
```

### 6.2 缺陷优先级定义

| 优先级 | 定义 | 示例 | 处理时效 |
|-------|-----|-----|---------|
| P0-紧急 | 阻塞核心流程，影响线上用户 | 支付失败、登录崩溃、数据丢失 | 4小时内修复 |
| P1-高 | 主要功能不可用，但有替代方案 | 某个支付方式不可用、某个页面加载异常 | 24小时内修复 |
| P2-中 | 非核心功能异常，不影响主流程 | 界面样式问题、非核心功能报错 | 本迭代修复 |
| P3-低 | 轻微问题、优化建议 | 文案错误、交互体验优化 | 排期修复 |

### 6.3 缺陷严重程度定义

| 严重程度 | 定义 | 示例 |
|---------|-----|-----|
| Blocker | 系统崩溃、死锁、数据损坏 | 服务OOM、数据库死锁 |
| Critical | 主要功能失效、数据计算错误 | 订单金额计算错误、权限失效 |
| Major | 功能部分失效、有临时绕过方案 | 某筛选条件失效 |
| Minor | 界面/交互/提示问题 | 错别字、图标不清晰 |
| Trivial | 优化建议 | 性能优化、体验优化 |

### 6.4 Bug提交规范

**Bug标题规范：**
```
[模块名][功能点] 简短描述问题现象

示例：
[订单][退款] 退款成功后订单状态未更新为"已退款"
[登录][微信授权] iOS端微信授权登录后App闪退
```

**Bug描述模板：**


### 6.5 Bug处理流程规则

| 规则 | 说明 |
|-----|-----|
| **Bug确认时限** | 开发需在新Bug提交后24小时内确认（标记为"处理中"或"拒绝"） |
| **Bug修复时限** | 按优先级：P0-4h / P1-24h / P2-本迭代 / P3-排期 |
| **拒绝Bug必须附理由** | 开发拒绝Bug时必须写明技术原因或PRD依据 |
| **争议处理** | 开发与测试意见不一致时，升级至技术负责人/产品经理仲裁 |
| **Bug重开** | 验证不通过时重新打开，标记为"reopened"，原修复人继续处理 |
| **Bug转交** | 修复人变更时需在系统中转交并知会测试 |

### 6.6 缺陷度量关键指标

| 指标 | 计算方式 | 用途 |
|-----|---------|-----|
| Bug发现率 | 测试发现Bug数 / 总Bug数 | 评估测试有效性 |
| Bug泄漏率 | 线上发现Bug数 / 总Bug数 | 评估测试质量 |
| Bug修复率 | 已修复Bug数 / 总有效Bug数 | 评估开发修复效率 |
| Bug重开率 | 重开Bug数 / 已修复Bug数 | 评估开发修复质量 |
| Bug平均修复时长 | Bug从"处理中"到"待验证"的平均时间 | 评估团队响应速度 |
| Bug分布 | 按模块/严重程度/原因分类的Bug数量 | 识别质量薄弱环节 |

## 七、流程落地与持续改进

### 7.1 流程落地五步法

```
Step 1：共识达成
├── 与管理层、团队骨干充分沟通流程价值
├── 收集各方反馈，调整流程细节
└── 正式发布，全员宣贯

Step 2：工具承载
├── 将流程固化到工具中（Jira/TAPD/飞书审批）
├── 自动化流转与提醒
└── 确保每个流程节点都有工具支撑

Step 3：试点运行
├── 选择1-2个项目试点运行新流程
├── 收集问题与反馈
└── 快速迭代优化

Step 4：全面推广
├── 试点成功后在全团队推广
├── 安排流程培训（操作指南 + 常见问题）
└── 设定流程QA负责人

Step 5：持续监控
├── 定义流程执行监控指标
├── 定期检查流程执行情况
└── 每季度回顾优化一次
```

### 7.2 流程执行度量指标

| 指标 | 计算方式 | 目标 |
|-----|---------|-----|
| 提测规范执行率 | 有提测邮件的提测次数 / 总提测次数 | 100% |
| 用例评审覆盖率 | 经过评审的用例需求数 / 总需求数 | ≥90% |
| 流程违规次数 | 月度违规事件数 | ≤3次 |
| 流程满意度 | 团队匿名调研评分（1-5分） | ≥4.0 |

### 7.3 持续改进机制

**定期回顾会议：**
- 月度流程回顾（30min）：检视流程执行情况
- 季度流程优化会（60min）：讨论流程改进提案
- 年度流程大回顾（半天）：全面审视流程体系

**改进来源：**
- 线上事故复盘（每次事故必须反思流程是否有改进空间）
- 团队匿名调研（每季度收集改进建议）
- 行业最佳实践对标（每年进行一次基准对比）

### 7.4 流程裁剪指南

不同项目应灵活裁剪流程，避免"一刀切"：

| 项目类型 | 需求评审 | 用例评审 | 提测规范 | 发布审批 |
|---------|---------|---------|---------|---------|
| 核心业务（P0） | 必须 | 必须（L3） | 严格 | 多层审批 |
| 一般业务（P1） | 必须 | 必须（L2） | 标准 | 标准审批 |
| 小型优化（P2） | 简化 | L1自审 | 简化 | 简化审批 |
| Hotfix | 跳过 | 跳过 | 简化 | 紧急通道 |

## 八、5个流程规范建设案例

### 案例1：从零建设——创业公司测试流程

**背景：** 20人研发团队，测试2人，之前没有任何正式流程

**建设策略：**
```markdown
第一阶段（第1个月）：建立最小可行流程
- 需求评审：不强制形式，但测试必须参与需求讨论
- Bug管理：统一使用TAPD，制定Bug提交模板
- 提测：开发口头/IM提测 → 统一为提测邮件

第二阶段（第2-3个月）：流程规范化
- 制定正式的需求评审规范
- 建立用例评审机制（小型需求自审，大型需求组内评审）
- 制定提测准入标准

第三阶段（第4-6个月）：流程完善
- 建立发布上线审批流程
- 建立线上事故复盘机制
- 引入流程度量指标

结果：6个月后，提测打回率从60%降至20%，线上Bug减少50%
```

### 案例2：流程优化——互联网大厂敏捷团队

**背景：** 双周迭代，测试团队10人+，流程过重导致敏捷效率下降

**优化策略：**
```markdown
问题：流程太多太细，测试人员60%时间花在流程上

优化措施：
1. 需求评审瘦身：
   - 不再要求所有需求都开正式评审会
   - 小需求（<3人天）改为异步评审（在线文档评论+IM确认）
   - 评审会时长限制在30min内

2. 用例评审分级：
   - L1自审+AI辅助评审：小需求用AI进行用例覆盖度检查
   - L2组内评审：中等需求保留
   - L3跨角色评审：仅核心功能需要

3. 提测流程精简：
   - 提测邮件改为禅道自动通知（关联Git提交自动触发）
   - 冒烟测试自动化：核心用例集自动执行+报告

结果：测试人员流程负担降低40%，迭代交付速度提升30%
```

### 案例3：金融企业——强合规测试流程

**背景：** 银行科技子公司，受银保监会监管，系统变更需审计追溯

**流程特点：**
```markdown
1. 全流程留痕：
   - 所有评审会议必须有正式纪要+签字
   - 测试用例与需求一一追溯（需求追溯矩阵RTM）
   - 测试执行记录必须可审计（谁、何时、执行了什么、结果如何）

2. 多级审批：
   - 测试计划：测试经理 → 项目经理 → 质量总监
   - 测试报告：测试经理 → 项目群经理 → 技术总监
   - 发布审批：技术负责人 → 运维负责人 → 业务负责人

3. 审计要求：
   - 所有测试文档保留至少5年
   - 定期接受内部审计+外部监管审计
   - 合规检查项必须100%执行

关键经验：流程严格但高效的秘诀是"自动化留痕"
- 使用Jira+Xray实现需求→用例→执行→缺陷全链路追溯
- 审批流程线上化，移动端可审批
- 文档自动归档，减少手工操作
```

### 案例4：多产品线测试流程统一

**背景：** 公司3条产品线，各自有独立的测试流程，导致资源调配困难

**统一策略：**
```markdown
Step 1：现状调研
- 调研各产品线现有流程
- 识别共通点（需求评审、Bug管理流程基本一致）
- 识别差异点（用例评审方式、提测标准差异大）

Step 2：制定统一框架
- 制定"测试流程总纲"（强制要求，各产品线必须遵守）
- 允许各产品线在总纲下制定适配细节（自由裁量）

Step 3：统一工具平台
- 统一使用Jira管理需求和Bug
- 统一使用TestLink管理测试用例
- 统一CI/CD平台（GitLab CI）

Step 4：建立流程委员会
- 跨产品线的流程负责人每月沟通一次
- 流程变更需要流程委员会审批
- 共享优秀实践

结果：测试人员跨产品线调配效率提升50%，新人上手时间从2周缩短至3天
```

### 案例5：测试左移——流程前置实践

**背景：** 传统流程中测试在开发之后才介入，Bug修复成本高

**左移策略：**
```markdown
1. 需求阶段左移：
   - 测试参与需求讨论（不是评审时才介入，而是在需求构思阶段就参与）
   - 测试提供可测性设计建议（如日志埋点、异常处理规范）
   - 测试输出验收标准（AC）作为需求的一部分

2. 设计阶段左移：
   - 测试参与技术方案评审（关注接口设计、数据一致性、异常处理）
   - 测试输出测试要点供开发自测参考
   - 测试提前准备测试数据和测试环境

3. 开发阶段左移：
   - 开发自测用例由测试提供（测试驱动自测）
   - 接口Mock由测试提前准备
   - 测试在开发阶段就开始编写自动化用例

4. 集成交付左移：
   - 代码提交自动触发静态扫描+单元测试
   - 每日构建自动触发接口自动化测试
   - 冒烟测试自动化，提测即自动执行

效果数据：
- 提测质量：P0/P1 Bug密度下降60%
- 测试周期：缩短30%
- 线上故障率：下降45%
```
