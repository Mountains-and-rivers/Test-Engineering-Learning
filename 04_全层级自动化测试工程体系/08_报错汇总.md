## 1. 框架搭建常见错误（15+）

### 1.1 配置文件加载顺序混乱

**问题描述：** 配置了多个配置文件（application.yaml, application-test.yaml, 环境变量），但最终生效的值不是预期的。

**报错信息：** 无明显报错，但数据库连接到错误的地址，或API请求发到了错误的环境。

**根本原因：** 多个配置源的加载顺序未明确定义，低优先级的配置意外覆盖了高优先级的配置。Spring的PropertySource加载顺序与开发者的直觉不一致。

**解决方案：**
```java
// 1. 明确配置加载优先级
@PropertySource(value = {
    "classpath:application.yaml",
    "classpath:application-${env}.yaml"
})
// 2. 启动时打印所有配置（验证期望值）
@PostConstruct
public void printActiveConfig() {
    log.info("Active Profile: {}", environment.getActiveProfiles());
    log.info("DB Host: {}", environment.getProperty("datasource.host"));
}
// 3. 使用配置断言：不符合预期则阻止启动
```

### 1.2 依赖版本冲突（NoClassDefFoundError）

**问题描述：** Maven/Gradle项目中引入了多个版本的同一个依赖，运行时出现类找不到的方法。

**报错信息：**
```
java.lang.NoClassDefFoundError: com/google/common/collect/ImmutableList
java.lang.NoSuchMethodError: org.apache.commons.io.FileUtils.readFileToString
```

**根本原因：** 多个第三方库各自依赖了commons-io的不同版本，Maven依赖仲裁选择了旧版本。

**解决方案：**
```xml
<!-- pom.xml: 使用dependencyManagement统一管理版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.3-jre</version>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 查看依赖树排查冲突 -->
<!-- mvn dependency:tree -Dincludes=com.google.guava -->
```

### 1.3 TestNG/JUnit混用导致注解失效

**问题描述：** 项目中同时引入了TestNG和JUnit，`@Test`注解来源于错误的包。

**报错信息：**
```
org.testng.TestNGException: Cannot find class in classpath
@Test annotation not recognized / No tests found
```

**根本原因：** 类路径上同时存在TestNG和JUnit，IDE自动导入时选择了错误的`@Test`注解（`org.junit.Test` vs `org.testng.annotations.Test`）。

**解决方案：**
```xml
<!-- 在pom.xml中明确排除不需要的测试框架 -->
<dependency>
    <groupId>org.testng</groupId>
    <artifactId>testng</artifactId>
    <version>7.8.0</version>
    <scope>test</scope>
    <exclusions>
        <exclusion>
            <groupId>junit</groupId>
            <artifactId>junit</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

### 1.4 WebDriver与浏览器版本不匹配

**问题描述：** Selenium WebDriver版本与本地Chrome版本不匹配，导致无法启动浏览器。

**报错信息：**
```
org.openqa.selenium.SessionNotCreatedException: 
  Could not start a new session. 
  This version of ChromeDriver only supports Chrome version 119
  Current browser version is 125.0.6422.141
```

**根本原因：** Chrome浏览器自动更新到新版本，但ChromeDriver还是旧版本。

**解决方案：**
```java
// 使用WebDriverManager自动管理驱动版本（推荐）
import io.github.bonigarcia.wdm.WebDriverManager;

@BeforeMethod
public void setupDriver() {
    WebDriverManager.chromedriver().setup();  // 自动匹配版本
    driver = new ChromeDriver();
}

// Maven依赖
// <dependency>
//   <groupId>io.github.bonigarcia</groupId>
//   <artifactId>webdrivermanager</artifactId>
//   <version>5.6.2</version>
// </dependency>
```

### 1.5 连接池耗尽导致测试hang住

**问题描述：** 高并发执行数据库相关测试时，测试突然卡死不动。

**报错信息：**
```
java.sql.SQLTransientConnectionException: 
  HikariPool-1 - Connection is not available, 
  request timed out after 30000ms
```

**根本原因：** 连接池配置过小，并发测试把连接池榨干了，新的请求等待连接释放但永远等不到。未正确关闭数据库连接导致连接泄漏。

**解决方案：**
```yaml
# 合理配置连接池
datasource:
  hikari:
    maximum-pool-size: 20       # 匹配并发线程数
    minimum-idle: 5
    connection-timeout: 30000
    idle-timeout: 600000
    max-lifetime: 1800000
    leak-detection-threshold: 10000  # 检测连接泄漏
```

### 1.6 日志文件爆炸撑满磁盘

**问题描述：** 运行几天后服务器磁盘被日志文件撑满，导致所有服务异常。

**报错信息：**
```
java.io.IOException: No space left on device
```

**根本原因：** 日志配置没有滚动策略，或DEBUG级别产生大量日志。无限追加写入同一个日志文件。

**解决方案：**
```xml
<!-- logback.xml: 添加滚动和清理策略 -->
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/test.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
        <fileNamePattern>logs/test.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
        <maxFileSize>100MB</maxFileSize>          <!-- 单文件上限 -->
        <maxHistory>30</maxHistory>               <!-- 保留30天 -->
        <totalSizeCap>3GB</totalSizeCap>          <!-- 总大小限制 -->
    </rollingPolicy>
</appender>
```

### 1.7 时间格式/时区问题

**问题描述：** 测试用例在不同时区的服务器上执行，时间比较断言失败。

**报错信息：**
```
Expected: "2025-01-15 10:00:00" but was: "2025-01-15 02:00:00"
java.lang.AssertionError: 期望时间不匹配
```

**根本原因：** 测试环境/CI服务器的时区是UTC，而开发环境是UTC+8。日期比较没有指定时区。

**解决方案：**
```java
// 永远使用带时区的日期时间
LocalDateTime now = LocalDateTime.now(ZoneOffset.UTC);
ZonedDateTime zdt = ZonedDateTime.now(ZoneId.of("Asia/Shanghai"));

// CI中强制设置时区
// -Duser.timezone=Asia/Shanghai
// docker run -e TZ=Asia/Shanghai
```

### 1.8 内存溢出（OOM）

**问题描述：** 大量用例执行后JVM崩溃，或测试越来越慢。

**报错信息：**
```
java.lang.OutOfMemoryError: Java heap space
java.lang.OutOfMemoryError: GC overhead limit exceeded
```

**根本原因：** 
- 测试结果/日志对象未释放，持续堆积在内存中
- ThreadLocal未清理导致内存泄漏
- WebDriver实例创建后未关闭

**解决方案：**
```java
// 1. 及时清理ThreadLocal
@AfterMethod
public void cleanup() {
    TestContextHolder.clear();  // ThreadLocal.remove()
}

// 2. 确保WebDriver被关闭
@AfterMethod
public void tearDown() {
    if (driver != null) {
        try { driver.quit(); } 
        catch (Exception e) { log.warn("关闭driver失败", e); }
    }
}

// 3. 调整JVM参数
// -Xmx2048m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./heapdumps/
```

### 1.9 硬编码路径导致跨平台问题

**问题描述：** 代码中使用Windows风格路径（反斜杠），在Linux/Mac上执行失败。

**报错信息：**
```
java.io.FileNotFoundException: 
  D:\testdata\input.xlsx (No such file or directory)
```

**根本原因：** 路径分隔符硬编码为`\`（Windows），Linux/macOS使用`/`。

**解决方案：**
```java
// 使用File.separator或Path API
Path path = Paths.get("testdata", "input.xlsx");  // 跨平台
// 或使用正斜杠（Java在所有平台都能处理）
String path = "testdata/input.xlsx";

// CI中统一使用Linux风格
String reportPath = ConfigManager.get("report.path", "./reports")
    .replace("\\", "/");
```

### 1.10 HTTP连接未关闭导致端口耗尽

**问题描述：** 大量API测试后，新请求开始报连接错误。

**报错信息：**
```
java.net.BindException: Address already in use
java.net.SocketException: Too many open files
```

**根本原因：** 每次请求创建新的HttpClient但没有关闭，连接处于TIME_WAIT状态。

**解决方案：**
```java
// 1. 复用HttpClient实例（单例模式）
private static final OkHttpClient CLIENT = new OkHttpClient.Builder()
    .connectionPool(new ConnectionPool(20, 5, TimeUnit.MINUTES))
    .build();

// 2. 注意关闭Response Body
try (Response response = CLIENT.newCall(request).execute()) {
    // 使用response
}  // try-with-resources自动关闭

// 3. 设置操作系统级别限制
// ulimit -n 65535  (Linux)
```

### 1.11 静态变量污染跨用例数据

**问题描述：** 测试用例A修改了某个静态变量，测试用例B因此失败。但单独跑B却是通过的。

**报错信息：** 无明显报错，但数据不符合预期。

**根本原因：** 测试代码中使用了static变量存储测试状态，不同用例之间共享了状态。

**解决方案：**
```java
// 不要用static存储测试状态！
// 错误示例
private static String currentToken;  // 会被并发用例覆盖

// 正确做法：使用ThreadLocal或测试框架的上下文
private static final ThreadLocal<String> currentToken = new ThreadLocal<>();

// 或在 @BeforeMethod 中重新初始化
@BeforeMethod
public void init() {
    currentToken.set(null);
}
```

### 1.12 隐式等待与显式等待混用

**问题描述：** Selenium中同时设置了隐式等待和显式等待，导致等待时间不可预测。

**报错信息：** 无明显报错，但有时等待时间异常长（两个等待叠加）。

**根本原因：** Selenium官方明确不建议混用隐式等待和显式等待，可能导致不可预测的等待时长。

**解决方案：**
```java
// 推荐：只用显式等待（WebDriverWait）
driver.manage().timeouts().implicitlyWait(0, TimeUnit.SECONDS); // 禁用隐式等待

WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement element = wait.until(
    ExpectedConditions.elementToBeClickable(By.id("submit"))
);
```

### 1.13 测试数据硬编码导致大规模修改

**问题描述：** 测试数据直接写在测试代码中，接口参数变更时需要在几十个文件里逐个修改。

**报错信息：** 无直接报错，大量编译错误或断言失败。

**根本原因：** 测试数据没有与测试代码分离。

**解决方案：**
```java
// 错误做法
@Test
public void testLogin() {
    httpClient.post("/api/login", 
        """{"username":"admin","password":"Admin@123"}""");  // 硬编码
}

// 正确做法：使用外部数据文件 + DataFactory
@Test(dataProvider = "loginData")
public void testLogin(LoginData data) {
    httpClient.post("/api/login", JsonUtil.toJson(data));
}
```

### 1.15 @Transactional测试导致数据不可见

**问题描述：** @Transactional + @Test导致在测试方法内插入的数据，被测应用查询不到。

**报错信息：** 无直接报错，断言失败：期望查到数据但查询结果为空。

**根本原因：** 测试方法的事务未提交，被测应用（独立进程/线程）查询不到未提交的数据。

**解决方案：**
```java
// 不要依赖@Transactional回滚（API测试不是单元测试）
// 测试API时，事务在服务端，测试端的事务回滚不作用于服务端

// 正确做法：使用数据清理
@AfterMethod
public void cleanup() {
    dataFactory.cleanupCreatedData();
}

// 或者：使用独立的数据库连接（不用Spring事务管理）
@BeforeMethod
public void setup() {
    jdbcTemplate.execute("INSERT INTO ...");
    jdbcTemplate.execute("COMMIT");  // 显式提交
}
```

### 1.16 使用assert而不是Assert.assertEquals

**问题描述：** 在TestNG测试中使用了Java的`assert`关键字，但生产环境运行时JVM默认禁用了assert。

**报错信息：** 无报错，断言被静默跳过，明明应该失败的测试却通过了。

**根本原因：** Java的`assert`关键字默认被禁用，需要`-ea`虚拟机参数才能启用。

**解决方案：**
```java
// 永远使用测试框架的断言
import org.testng.Assert;

@Test
public void testExample() {
    // 错误
    // assert result != null;  
    
    // 正确
    Assert.assertNotNull(result, "结果不应为空");
}
```

## 2. CI/CD集成常见坑（10+）

### 2.1 CI环境缺少必要的系统依赖

**问题描述：** 本地能跑的测试到CI上就报错，提示找不到某个库或工具。

**报错信息：**
```
error while loading shared libraries: libgtk-3.so.0
java.awt.AWTError: Can't connect to X11 window server
```

**根本原因：** CI服务器（通常是Linux）没有安装图形界面库（Chrome/Firefox依赖）。

**解决方案：**
```dockerfile
# Dockerfile中安装必要的系统依赖
RUN apt-get update && apt-get install -y \
    libgtk-3-0 libgbm1 libasound2 \
    libnspr4 libnss3 libxss1 \
    fonts-liberation libappindicator3-1 \
    xdg-utils

# Chrome headless模式
ChromeOptions options = new ChromeOptions();
options.addArguments("--headless=new");
options.addArguments("--no-sandbox");
options.addArguments("--disable-dev-shm-usage");
```

### 2.2 CI超时设置不合理

**问题描述：** CI构建总是在固定的时间点超时失败，但实际上测试还在正常运行。

**报错信息：**
```
The job exceeded the maximum log length, 
and has been terminated.
Build timed out (after 60 minutes)
```

**根本原因：** CI平台默认超时时间（GitHub Actions: 360分钟, Jenkins默认无限制）与测试实际所需时间不匹配。或者日志输出太多导致构建被终止。

**解决方案：**
```yaml
# GitHub Actions
jobs:
  test:
    timeout-minutes: 120  # 设置合理的超时
    steps:
      - run: mvn test -q  # -q减少日志输出

# Jenkins
options {
    timeout(time: 90, unit: 'MINUTES')
}
```

### 2.3 环境变量在CI中未正确传递

**问题描述：** CI中设置的环境变量，在测试代码中读取为null或默认值。

**报错信息：**
```
ConfigManager.get("api.base-url") returns null
```

**根本原因：** 
- Jenkins中环境变量需要export或使用withEnv

- Docker中env需要用-e传递

```groovy
// Jenkins: 在pipeline中正确设置
environment {
    API_BASE_URL = 'https://test-api.example.com'
}
// 或
withEnv(["API_BASE_URL=https://test-api.example.com"]) {
    sh 'mvn test'
}
```

### 2.4 Artifact路径错误导致报告丢失

**问题描述：** CI构建成功但找不到测试报告文件。

**报错信息：**
```
Warning: No artifacts were uploaded. 
Pattern 'target/allure-results/' matched no files.
```

**根本原因：** Artifact路径相对于工作目录不正确，或测试失败后没有生成报告文件。

**解决方案：**
```groovy
// 1. 始终使用绝对路径（相对于workspace）
archiveArtifacts artifacts: '**/allure-results/**', fingerprint: true

// 2. 确保无论测试成功失败都上传报告
post {
    always {
        archiveArtifacts artifacts: 'target/allure-results/**'
    }
}

// 3. 先验证文件是否存在
sh 'ls -la target/allure-results/'
```

### 2.5 Jenkins Slave节点环境不一致

**问题描述：** 测试在某些节点通过，在另一些节点失败。

**报错信息：** 因节点而异，常见：浏览器版本不匹配、Java版本不对、依赖缺失。

**根本原因：** Jenkins Slave节点没有统一的环境配置，手动安装的工具版本不一致。

**解决方案：**
```groovy
// 1. 使用Docker agent确保环境一致
agent {
    docker {
        image 'test-automation:2.0'
        alwaysPull true  // 始终拉取最新镜像
    }
}

// 2. 使用label精确指定节点
agent { label 'ubuntu20-java17-chrome125' }

// 3. 在pipeline开始时验证环境
stage('Verify Environment') {
    steps {
        sh 'java -version'
        sh 'mvn --version'
        sh 'google-chrome --version'
    }
}
```

### 2.6 并发构建导致资源冲突

**问题描述：** 多个分支同时触发CI构建，同时操作同一个数据库/测试环境，互相干扰。

**报错信息：**
```
SQL Error: 1062, Duplicate entry 'test_user' for key 'username'
Cannot acquire lock on test resource
```

**根本原因：** 多个构建共享同一测试环境，未做资源隔离。

**解决方案：**
```groovy
// 1. 使用资源锁
lock(resource: 'test-environment') {
    sh 'mvn test'
}

// 2. 每个构建使用不同的数据前缀
env.TEST_DATA_PREFIX = "build_${BUILD_NUMBER}_"
sh "mvn test -Dtest.prefix=${TEST_DATA_PREFIX}"

// 3. 使用Docker Compose为每个构建创建独立环境
sh """
    docker compose -p build-${BUILD_NUMBER} up -d
    mvn test
    docker compose -p build-${BUILD_NUMBER} down -v
"""
```

### 2.8 CI构建排队时间过长

**问题描述：** CI触发后等待很长时间才开始执行。

**报错信息：** 无直接报错，但在Jenkins队列中等待数小时。

**根本原因：** Agent资源不足、并发构建数达到上限。

**解决方案：**
```groovy
// 1. 动态扩展Agent（K8s）
agent {
    kubernetes {
        yaml '''
        apiVersion: v1
        kind: Pod
        spec:
          containers:
          - name: maven
```

### 3.10 数据库备份/恢复失败

**问题描述：** 回归测试依赖数据库快照恢复，但有时恢复失败。

**报错信息：**
```
mysqldump: Got error: 1045: Access denied
ERROR 2002: Can't connect to MySQL server
```

**根本原因：** 数据库凭据过期、磁盘空间不足、MySQL服务异常。

**解决方案：**
```bash
#!/bin/bash
# 备份脚本中加入检查

# 检查磁盘空间
AVAILABLE=$(df /backup | awk 'NR==2 {print $4}')
if [ "$AVAILABLE" -lt 5242880 ]; then  # < 5GB
    echo "磁盘空间不足，清理旧备份"
    find /backup -name "*.sql.gz" -mtime +7 -delete
fi

# 检查数据库连接
if ! mysqladmin ping -h"$DB_HOST" -u"$DB_USER" -p"$DB_PASS" --silent; then
    echo "数据库不可达"
    exit 1
fi

# 执行备份
mysqldump -h"$DB_HOST" -u"$DB_USER" -p"$DB_PASS" \
    --single-transaction --quick --routines \
    "$DB_NAME" | gzip > "/backup/snapshot_$(date +%Y%m%d).sql.gz"
```

```java
                "\"" + key + "\"\\s*:\\s*\"[^\"]*\"",
                "\"" + key + "\": \"******\"");
        }
        return json;
    }
}

    SensitiveDataMasker.mask(requestBody), ".txt");
```

## 5. 用例维护常见痛点（10+）

### 5.1 用例腐化（Bit Rot）

**问题描述：** 功能没变，但自动化用例逐渐失败，因为没有人维护用例代码。

**报错信息：** 各种零散的失败，如元素定位失败、API字段名变更、断言过时。

**根本原因：** 只增加新用例不维护旧用例，框架变更后旧用例未同步更新。

**解决方案：**
```java
// 1. 用例健康检查
@Scheduled(cron = "0 0 4 * * 1")  // 每周一凌晨4点
public void testHealthCheck() {
    List<TestResult> results = runAllTests();
    long consecutiveFailures = results.stream()
        .filter(r -> r.getConsecutiveFailures() >= 7)
        .count();
    
    if (consecutiveFailures > 0) {
        notificationService.sendAlert(
            String.format("发现 %d 个用例连续失败超过7天，可能已经腐化", 
                consecutiveFailures));
    }
}

// 2. 自动禁用长期失败的用例（避免干扰）
@Test(enabled = false)  // 手动或自动设置
public void deprecatedTestCase() { ... }
```

### 5.3 用例间隐式依赖

**问题描述：** 用例B的单跑失败，但在完整套件中能通过（因为用例A给B准备了数据）。

**报错信息：** 单独运行时失败，但套件中通过。

**根本原因：** 用例之间通过共享的数据库状态传递数据，没有显式声明依赖。

**解决方案：**
```java
// 1. 使用显式依赖注解
@Test
public void testCreateUser() {
    String userId = createUser();
    TestContext.set("createdUserId", userId);
}

@Test(dependsOnMethods = "testCreateUser")  // 显式声明依赖
public void testGetUser() {
    String userId = TestContext.get("createdUserId");
    getUser(userId);
}

// 2. 或使每个用例完全独立（推荐）
@Test
public void testGetUser() {
    String userId = dataFactory.createUser().getUserId();  // 自己准备数据
    getUser(userId);
}
```

### 5.4 环境依赖导致Flaky用例

**问题描述：** 同一个用例今天通过明天失败，没有任何代码变更。

**报错信息：** 各类异常随机出现：TimeoutException、AssertionError、ConnectException。

**根本原因：** 
- 网络延迟抖动
- 数据库查询时间不固定
- 异步任务完成时间随机
- 第三方服务不稳定

**解决方案：**
```java
// 1. 使用显式等待替代固定sleep
// 错误: Thread.sleep(3000);
// 正确:
await().atMost(10, TimeUnit.SECONDS).until(() -> {
    return dbHelper.queryOne("SELECT status FROM orders WHERE id = ?", orderId)
        .get("status").equals("PAID");
});

// 2. 增加重试机制 + Flaky标签
@Test(retryAnalyzer = RetryAnalyzer.class)
public void testAsyncPaymentCallback() { ... }

// 3. Mock不稳定的外部依赖
@BeforeMethod
public void mockThirdParty() {
    WireMock.stubFor(post("/mock/payment/callback")
        .willReturn(ok("""{"code":0}""")));
}
```

### 5.5 测试数据竞争

**问题描述：** 并发执行的用例操作同一条测试数据，导致数据状态不确定。

**报错信息：**
```
SQL Error: 1213, Deadlock found when trying to get lock
OptimisticLockException
Unexpected row count: expected 1, got 0 (数据已被其他用例修改)
```

**根本原因：** 并发用例共享了测试数据，没有做数据隔离。

**解决方案：**
```java
// 1. 每个并发用例使用唯一的数据
@BeforeMethod
public void createUniqueData() {
    String uniqueId = UUID.randomUUID().toString().substring(0, 8);
    user = dataFactory.createUser("testuser_" + uniqueId);
    product = dataFactory.createProduct("product_" + uniqueId);
}

// 2. 使用数据库行级锁
SELECT * FROM test_resource WHERE resource_type = 'coupon' 
FOR UPDATE SKIP LOCKED LIMIT 1;

// 3. 或不同用例使用不同的数据分区
String partition = "shard_" + (Thread.currentThread().getId() % 10);
```

### 5.6 断言不精确

**问题描述：** 用例通过了但实际功能有问题，因为断言太宽松。

**报错信息：** 无报错（假通过——最危险的失败）。

**根本原因：** 
- 断言只检查HTTP 200，不检查返回内容
- 使用了模糊匹配（contains而非equals）
- 完全没有断言

**解决方案：**
```java
// 错误：无意义断言
response.assertStatus(200);
Assert.assertTrue(true);

// 正确：精确断言
HttpResponse response = httpClient.post("/api/orders", orderJson);
response.assertStatus(201);
JsonNode order = response.toJson();
Assert.assertNotNull(order.get("orderId"), "订单ID不应为空");
Assert.assertEquals(order.get("status").asText(), "PENDING", "订单状态应为待支付");
Assert.assertEquals(order.get("totalAmount").asDouble(), 198.0, 0.01);
Assert.assertTrue(order.get("orderId").asText().startsWith("ORD"), "订单号格式验证");

// 数据库落库验证
Map<String, Object> dbOrder = dbHelper.queryOne("default",
    "SELECT * FROM orders WHERE order_id = ?", order.get("orderId").asText());
Assert.assertNotNull(dbOrder, "订单应已落库");
Assert.assertEquals(dbOrder.get("status"), "PENDING");
```

### 5.7 测试代码重复严重

**问题描述：** 10个测试类中有8个都包含相同的登录初始化代码。

**报错信息：** 无直接报错，但修改登录逻辑时需要改8个文件。

**根本原因：** 缺少公共的测试基类和辅助方法。

**解决方案：**
```java
// 1. 提取公共操作到基类
public abstract class BaseTest {
    protected User loginAsDefaultUser() {
        return loginAs("default_user", "Test@123");
    }
    
    protected User loginAs(String username, String password) {
        HttpResponse response = httpClient.post("/api/login",
            String.format("""{"username":"%s","password":"%s"}""", 
                username, password));
        response.assertStatus(200);
        String token = response.toJson().get("token").asText();
        httpClient.setAuthToken(token);
        return new User(username, token);
    }
    
    protected Product createTestProduct(double price) {
        return dataFactory.createProduct(
            ProductTemplate.builder().price(new BigDecimal(price)).build());
    }
}

// 2. 使用Fixture/DataFactory减少重复数据准备代码
// 3. 提取公共断言到AssertHelper
```

### 5.9 测试环境数据污染

**问题描述：** 测试环境中有大量"脏数据"——不完整的订单、半截的用户、逻辑上不可能的数据组合。

**报错信息：**
```
测试数据异常: 订单状态为"PAID"但没有支付记录
```

**根本原因：** 测试用例创建了数据但没有完整清理，或中间出错中断导致数据不完整。

**解决方案：**
```java
// 1. try-finally确保清理
@Test
public void testCreateOrder() {
    User user = null;
    Order order = null;
    try {
        user = dataFactory.createUser();
        order = dataFactory.createOrder(user);
        // 测试逻辑
    } finally {
        if (order != null) dataFactory.cleanupOrder(order);
        if (user != null) dataFactory.cleanupUser(user);
    }
}

// 2. 使用回调/钩子注册清理操作
TestCleanup.register(() -> dataFactory.cleanupAll());

// 3. 定期执行数据完整性检查
SELECT * FROM orders WHERE user_id NOT IN (SELECT user_id FROM users);
```

### 5.11 测试执行时间不受控

**问题描述：** 回归测试时间从1小时增长到3小时，没有人注意到这个趋势。

**报错信息：** 无直接报错，但回归时间越来越长导致反馈变慢。

**根本原因：** 不断新增用例但从不优化或删除旧用例，执行时间线性增长。

**解决方案：**
```java
// 1. 监控执行时间趋势并告警
@Scheduled(cron = "0 0 8 * * 1")
public void checkExecutionTimeTrend() {
    long currentAvg = getAverageExecutionTime(7);  // 最近7天
    long baselineAvg = getAverageExecutionTime(30, 60); // 30-60天前
    
    if (currentAvg > baselineAvg * 1.3) {
        alertService.sendAlert(String.format(
            "回归执行时间增长超过30%%: 当前%dm vs 基线%dm",
            currentAvg / 60000, baselineAvg / 60000));
    }
}

// 2. 定期Review慢速用例
// 3. 删除重复或低价值用例
// 4. 使用并行执行提高效率
```

### 5.13 Python多版本兼容

**问题描述：** Python测试代码在某台机器（Python 3.10）上能跑，在另一台（Python 3.12）上报错。

**报错信息：**
```
AttributeError: module 'asyncio' has no attribute 'coroutine'
DeprecationWarning: distutils is deprecated
```

**根本原因：** Python版本升级后的breaking changes影响了测试依赖。

**解决方案：**
```bash
# 1. 使用虚拟环境锁定Python版本
# .python-version
3.10.13

# 2. 锁定依赖版本
# requirements.txt
pytest==7.4.3
requests==2.31.0

# 3. CI中使用固定版Python
# GitHub Actions:
- uses: actions/setup-python@v4
  with:
    python-version: '3.10'

# 4. 定期测试多Python版本（矩阵测试）
strategy:
  matrix:
    python-version: ['3.9', '3.10', '3.11', '3.12']
```

## 总结

```
         ┌──────────────────────────────────┐
         │        避坑核心原则               │
         ├──────────────────────────────────┤
         │ 1. 环境一致性 > 方便性            │
         │    Docker化一切，消除"我机器上好的"│
         │                                  │
         │ 2. 显式 > 隐式                    │
         │    依赖显式声明，数据显式准备      │
         │                                  │
         │ 3. 隔离 > 共享                    │
         │    数据隔离，环境隔离，并发隔离    │
         │                                  │
         │ 4. 可观测 > 静默                  │
         │    日志充分，报告详尽，告警及时    │
         │                                  │
         │ 5. 自动化 > 人工                  │
         │    自动清理、自动重试、自动告警    │
         │                                  │
         │ 6. 持续改善 > 一次性搭建          │
         │    定期Review，持续优化，永不腐化  │
         └──────────────────────────────────┘
```

> 每一个报错都是一次学习机会。掌握了这些常见问题和解决方案，你就能在自动化测试工程的道路上走得更稳、更远。
