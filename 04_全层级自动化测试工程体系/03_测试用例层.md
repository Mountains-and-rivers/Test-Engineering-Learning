# 测试用例层

> **测试用例层**（Test Case Layer）：自动化测试框架的执行核心层，负责测试用例的组织、管理、调度和执行。这一层决定了测试用例如何被组织（按模块、按流程）、如何被参数化（数据驱动）、如何被调度（并发、依赖、重试）以及如何被标记（标签、分组、优先级）。

## 1. 测试用例组织结构

### 通俗释义

测试用例组织就像图书馆的图书分类——如果没有好的分类体系，几千个用例会像乱堆的书一样找不到。好的组织方式让你能快速定位某个功能的所有用例、快速筛选出一个冒烟测试套件、快速找出P0级的核心用例。

### 详细解释

#### 三种主流组织方式

| 组织方式 | 说明 | 适用场景 | 示例 |
|----------|------|----------|------|
| **按模块组织** | 按系统功能模块划分 | 功能清晰、模块独立 | 用户模块/订单模块/支付模块 |
| **按流程组织** | 按端到端业务流程划分 | 业务耦合度高、流程驱动 | 下单流程/退货流程/注册流程 |
| **按优先级组织** | 按用例重要性划分 | 需要区分核心与边缘 | P0核心/P1重要/P2一般 |

#### 按模块组织（推荐主结构）

#### 基类设计（BaseTest）

```java
// BaseTest.java - 测试用例基类
@Listeners({TestListener.class, RetryListener.class, ReportListener.class})
public abstract class BaseTest {
    
    protected static final Logger log = LoggerFactory.getLogger(BaseTest.class);
    
    // 全局测试数据上下文
    protected static final ThreadLocal<Map<String, Object>> testContext = 
        ThreadLocal.withInitial(HashMap::new);
    
    // 注入全局工具
    @Autowired protected HttpClientManager httpClient;
    @Autowired protected DatabaseHelper dbHelper;
    @Autowired protected CacheHelper cacheHelper;
    @Autowired protected ConfigManager config;
    
    @BeforeSuite
    public void globalSetup() {
        log.info("========== 测试套件初始化 ==========");
        ConfigManager.init();
        log.info("环境: {}", config.getActiveEnv());
        log.info("API地址: {}", config.get("api.base-url"));
    }
    
    @BeforeMethod
    public void testSetup(Method method) {
        log.info("========== {} 开始 ==========", method.getName());
        testContext.get().clear();
        testContext.get().put("testName", method.getName());
        testContext.get().put("startTime", System.currentTimeMillis());
    }
    
    @AfterMethod
    public void testTeardown(Method method) {
        long elapsed = System.currentTimeMillis() - 
            (long) testContext.get().get("startTime");
        log.info("========== {} 结束 ({}ms) ==========", method.getName(), elapsed);
        testContext.remove(); // 防止内存泄漏
    }
    
    @AfterSuite
    public void globalTeardown() {
        log.info("========== 测试套件结束 ==========");
    }
    
    // 辅助方法
    protected void setTestData(String key, Object value) {
        testContext.get().put(key, value);
    }
    
    @SuppressWarnings("unchecked")
    protected <T> T getTestData(String key) {
        return (T) testContext.get().get(key);
    }
}
```

## 2. 用例依赖管理

### 通俗释义

用例依赖管理解决"A用例的输出是B用例的输入"的问题——比如"登录"用例生产的token，需要传递给"下单"用例使用。好的依赖管理让用例之间既能协作，又不会因为一个用例失败导致整个套件崩溃。

### 详细解释

#### 依赖管理策略

```
依赖处理的三种策略：

策略1：显式依赖（TestNG dependsOnMethods/dependsOnGroups）
    ├── 优点：精确控制执行顺序
    ├── 缺点：依赖链断裂时后续用例全部跳过
    └── 场景：有明确前置条件的用例（如：登录 -> 下单）

策略2：隐式依赖（共享Fixture/测试数据上下文）
    ├── 优点：解耦，前置失败不影响后续
    ├── 缺点：前置数据需要预先准备
    └── 场景：大多数业务场景（用数据工厂准备前置数据）

策略3：自包含（每个用例独立准备自己需要的数据）
    ├── 优点：完全独立，互不影响
    ├── 缺点：执行慢（每个用例都要准备数据）
    └── 场景：单元测试、并发执行的用例
```

#### TestNG依赖管理实现

```java
public class OrderTests extends BaseTest {
    
    // ===== 方式1：方法级依赖 =====
    @Test
    public void testUserLogin() {
        // 登录获取token
        HttpResponse response = httpClient.post("/api/login", 
            """{"username":"testuser","password":"Test123"}""");
        
        String token = response.toJson().get("token").asText();
        setTestData("authToken", token); // 存入测试上下文
        response.assertStatus(200);
    }
    
    @Test(dependsOnMethods = "testUserLogin")
    public void testCreateOrder() {
        // 依赖登录生成的token
        String token = getTestData("authToken");
        httpClient.addDefaultHeader("Authorization", "Bearer " + token);
        
        HttpResponse response = httpClient.post("/api/orders", 
            """{"productId":"P001","quantity":2}""");
        
        String orderId = response.toJson().get("orderId").asText();
        setTestData("orderId", orderId);
        response.assertStatus(201);
    }
    
    @Test(dependsOnMethods = "testCreateOrder")
    public void testPayOrder() {
        String token = getTestData("authToken");
        String orderId = getTestData("orderId");
        httpClient.addDefaultHeader("Authorization", "Bearer " + token);
        
        HttpResponse response = httpClient.post("/api/payments",
            String.format("""{"orderId":"%s","method":"ALIPAY"}""", orderId));
        
        response.assertStatus(200);
    }
    
    // ===== 方式2：分组级依赖 =====
    @Test(groups = "preparation")
    public void prepareTestData() {
        // 准备测试数据
        dbHelper.update("default", 
            "INSERT INTO products VALUES (?,?,?)", "P999", "测试商品", 99.00);
    }
    
    @Test(groups = "order", dependsOnGroups = "preparation")
    public void testOrderWithPreparedData() {
        // 使用准备好的数据
        boolean exists = dbHelper.recordExists("default", 
            "products", "product_id = ?", "P999");
        Assert.assertTrue(exists, "测试商品应存在");
    }
    
    // ===== 方式3：软依赖（alwaysRun = true） =====
    @Test(dependsOnMethods = "testUserLogin", alwaysRun = true)
    public void testGetUserInfo() {
        // 即使登录失败也尝试执行（使用缓存的token或默认token）
        String token = getTestData("authToken");
        if (token == null) {
            token = ConfigManager.get("test.fallback.token", "");
        }
        httpClient.addDefaultHeader("Authorization", "Bearer " + token);
        // ... 测试逻辑
    }
}
```

```python
import pytest

class TestOrderFlow:
    """订单流程（有依赖关系）"""
    
    @pytest.fixture(scope="class")
    def auth_token(self, http_client):
        """前置fixture：获取认证token"""
        resp = http_client.post("/api/login",

        assert resp.status_code == 200
        token = resp.json()["token"]
        return token
    
    @pytest.fixture(scope="class")
    def created_order(self, http_client, auth_token):
        """前置fixture：创建订单"""
        http_client.set_auth_token(auth_token)
        resp = http_client.post("/api/orders", 
            json={"productId": "P001", "quantity": 2})
        assert resp.status_code == 201
        order_id = resp.json()["orderId"]
        return order_id
    
    def test_create_order_success(self, created_order):
        """验证订单创建成功"""
        assert created_order is not None
        assert created_order.startswith("ORD")
    
    def test_pay_order(self, http_client, auth_token, created_order):
        """支付订单"""
        http_client.set_auth_token(auth_token)
        resp = http_client.post("/api/payments",
            json={"orderId": created_order, "method": "ALIPAY"})
```

## 3. 用例参数化（数据驱动）

### 通俗释义

用例参数化让"一个测试方法"能跑"多组测试数据"——比如登录测试，你用同一个测试方法，传入10组不同的用户名密码组合（正确/错误/空/超长/特殊字符），框架自动跑10次并生成10条测试结果。

### 详细解释

#### TestNG DataProvider

```java
public class LoginParameterizedTest extends BaseTest {
    
    // ===== 方式1：方法级DataProvider =====
    @DataProvider(name = "loginData")
    public Object[][] loginDataProvider() {
        return new Object[][]{
            // {username, password, expectedStatus, description}
            {"admin",      "Admin@123",  200, "正确的用户名密码"},
            {"admin",      "wrongpass",  401, "错误的密码"},
            {"notexist",   "Admin@123",  404, "不存在的用户"},
            {"",           "Admin@123",  400, "空用户名"},
            {"admin",      "",           400, "空密码"},
            {"admin",      "Admin@123",  200, "正确密码-重复验证"},
            {"ADMIN",      "Admin@123",  200, "用户名大小写不敏感"},
            {" admin ",    "Admin@123",  200, "用户名含空格（应去空格）"},
            {"admin",      "12345",      400, "密码过短（5位）"},
            {"admin",      "a".repeat(256), 400, "密码超长（256位）"},
        };
    }
    
    @Test(dataProvider = "loginData")
    public void testLogin(String username, String password, 
                         int expectedStatus, String description) {
        log.info("测试场景: {}", description);
        
        HttpResponse response = httpClient.post("/api/login",
            String.format("""{"username":"%s","password":"%s"}""", username, password));
        
        response.assertStatus(expectedStatus);
        if (expectedStatus == 200) {
            response.assertBodyContains("token");
        }
    }
    
    // ===== 方式2：从Excel文件读取DataProvider =====
    @DataProvider(name = "excelLoginData")
    public Object[][] excelLoginDataProvider() {
        List<Map<String, String>> data = ExcelHelper.readAsMap(
            "testdata/login_test_data.xlsx", "Sheet1");
        
        Object[][] result = new Object[data.size()][4];
        for (int i = 0; i < data.size(); i++) {
            Map<String, String> row = data.get(i);
            result[i] = new Object[]{
                row.get("username"),
                row.get("password"),
                Integer.parseInt(row.get("expectedStatus")),
                row.get("description")
            };
        }
        return result;
    }
    
    // ===== 方式3：从YAML文件读取DataProvider =====
    @DataProvider(name = "yamlLoginData")
    public Object[][] yamlLoginDataProvider() {
        Map<String, Object> config = YamlConfigLoader.load("testdata/login_data.yaml");
        List<Map<String, Object>> cases = (List<Map<String, Object>>) config.get("testCases");
        
        Object[][] result = new Object[cases.size()][4];
        for (int i = 0; i < cases.size(); i++) {
            Map<String, Object> tc = cases.get(i);
            result[i] = new Object[]{
                tc.get("username"), tc.get("password"),
                tc.get("expectedStatus"), tc.get("description")
            };
        }
        return result;
    }
}
```

## 4. 用例标签与分组

### 通俗释义

用例标签和分组就像给测试用例贴"标签贴纸"——冒烟测试、回归测试、P0优先级、慢速用例、需要数据库、依赖第三方...通过这些标签，你可以快速筛选出不同场景下应该跑哪些用例。

### 详细解释

#### 标签体系设计

| 标签类别 | 标签名称 | 用途 | 适用筛选场景 |
|----------|----------|------|-------------|
| **执行阶段** | smoke | 冒烟测试 | 代码提交后立即执行（5-10分钟） |
| | regression | 回归测试 | 每日/每周全量回归 |
| | e2e | 端到端测试 | 发版前全流程验证 |
| **优先级** | p0 | 核心功能 | 阻塞发版，必须100%通过 |
| | p1 | 重要功能 | 失败需评估是否修复 |
| | p2 | 一般功能 | 可接受失败 |
| | p3 | 边缘场景 | 可选执行 |
| **执行特性** | slow | 执行慢（>1分钟） | 并行调度时避开高峰期 |
| | flaky | 不稳定 | 失败时自动重试，不计入通过率 |
| | manual | 需人工介入 | CI中跳过 |
| **技术依赖** | db-required | 需要数据库 | 数据库不可用时不执行 |
| | third-party | 依赖第三方服务 | 第三方不可用时不执行 |
| | mock-only | 仅使用Mock | 离线环境可执行 |

#### TestNG分组实现


## 5. 用例失败重试机制

### 通俗释义

用例失败重试机制就像是"摔倒后爬起来再试一次"——自动化测试中经常会遇到网络抖动、环境不稳定等瞬时问题导致的假失败（Flaky Test）。合理的重试机制能有效过滤掉这些假失败，让测试结果更可信。

### 详细解释

#### 重试策略设计

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **即时重试** | 失败后立即重试，无等待 | 网络瞬时抖动 |
| **延迟重试** | 失败后等待固定时间再重试 | 数据库写入延迟 |
| **退避重试** | 等待时间指数增长 | 第三方服务恢复 |
| **条件重试** | 仅特定异常才重试 | 区分真BUG和假失败 |

#### TestNG重试实现

```java
// RetryAnalyzer.java - 重试分析器
public class RetryAnalyzer implements IRetryAnalyzer {
    
    private int retryCount = 0;
    private static final int MAX_RETRY = 2;
    
    // 需要重试的异常类型（白名单）
    private static final Set<Class<? extends Throwable>> RETRYABLE_EXCEPTIONS = Set.of(
        AssertionError.class,           // 断言失败（可能是环境问题）
        TimeoutException.class,         // 超时
        WebDriverException.class,       // WebDriver异常
        ConnectException.class,         // 网络连接异常
        SocketTimeoutException.class    // Socket超时
    );
    
    // 不需要重试的异常（黑名单）
    private static final Set<String> NO_RETRY_MESSAGES = Set.of(
        "404", "401", "403",          // HTTP明确错误
        "NullPointerException",       // NPE通常是代码Bug
        "DataNotFoundException"       // 数据不存在
    );
    
    @Override
    public boolean retry(ITestResult result) {
        if (retryCount < MAX_RETRY) {
            Throwable throwable = result.getThrowable();
            
            // 检查是否在白名单中
            boolean shouldRetry = RETRYABLE_EXCEPTIONS.stream()
                .anyMatch(ex -> ex.isInstance(throwable));
            
            // 检查是否在黑名单中
            if (throwable != null) {
                String message = throwable.getMessage();
                if (message != null) {
                    for (String noRetry : NO_RETRY_MESSAGES) {
                        if (message.contains(noRetry)) {
                            shouldRetry = false;
                            break;
                        }
                    }
                }
            }
            
            if (shouldRetry) {
                retryCount++;
                LoggerFactory.getLogger(RetryAnalyzer.class)
                    .warn("重试用例: {} (第{}次重试)", 
                        result.getName(), retryCount);
                return true;
            }
        }
        return false;
    }
}

// 使用方式1：注解
@Test(retryAnalyzer = RetryAnalyzer.class)
public void testFlakyApi() {
    // 不稳定用例
}

// 使用方式2：基类统一配置
public abstract class BaseTest {
    @BeforeMethod
    public void setupRetry(Method method) {
        // 通过配置文件控制是否启用重试
        if (!ConfigManager.getBoolean("execution.retry.enabled", true)) {
            RetryAnalyzer.disableForCurrentThread();
        }
    }
}

// RetryListener.java - 全局重试监听器
public class RetryListener implements IInvokedMethodListener {
    
    private static final Logger log = LoggerFactory.getLogger(RetryListener.class);
    
    @Override
    public void afterInvocation(IInvokedMethod method, ITestResult result) {
        if (result.getStatus() == ITestResult.FAILURE) {
            // 记录重试信息到报告中
            if (result.getMethod().getRetryAnalyzer(result) != null) {
                int retryCount = result.getMethod().getCurrentInvocationCount();
                log.warn("用例 {} 失败，已重试{}次", 
                    result.getName(), retryCount);
                
                // 超过重试次数后发送告警
                if (retryCount >= 2) {
                    NotificationService.sendAlert(
                        "用例持续失败: " + result.getName(),
                        result.getThrowable().getMessage()
                    );
                }
            }
        }
    }
}
```

## 6. 用例并发执行策略

### 通俗释义

并发执行让几百个测试用例"分头行动"同时跑——8个线程一起跑，比单线程快8倍。但并发也带来新问题：线程安全、数据隔离、资源竞争。好的并发策略需要在速度和稳定性之间找到平衡。

### 详细解释

#### TestNG并发策略

```java
// 资源池管理（数据库连接、浏览器实例等）
public class ResourcePool<T> {
    
    private final BlockingQueue<T> available;
    private final Set<T> inUse = ConcurrentHashMap.newKeySet();
    private final int poolSize;
    
    public ResourcePool(int size, Supplier<T> factory) {
        this.poolSize = size;
        this.available = new LinkedBlockingQueue<>();
        for (int i = 0; i < size; i++) {
            available.add(factory.get());
        }
    }
    
    public T borrow(long timeoutMs) {
        try {
            T resource = available.poll(timeoutMs, TimeUnit.MILLISECONDS);
            if (resource == null) {
                throw new RuntimeException("资源池耗尽，等待超时");
            }
            inUse.add(resource);
            return resource;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("获取资源被中断", e);
        }
    }
    
    public void release(T resource) {
        inUse.remove(resource);
        available.add(resource);
    }
    
    public int availableCount() { return available.size(); }
    public int inUseCount() { return inUse.size(); }
}
```

## 7. 用例超时控制

### 通俗释义

超时控制是测试用例的"安全绳"——如果一个用例跑了10分钟还没结束，很可能陷入了死循环或等待永远不会出现的元素。超时机制强制中断"卡住"的用例，避免整个测试套件被一个用例拖垮。

### 详细解释

#### 超时分层控制

```
超时控制的三个层级：

Level 1: 框架全局超时
    └── 所有用例的默认超时上限（如300秒）

Level 2: 套件超时
    └── 整个测试套件的总超时（如30分钟）

Level 3: 用例级超时
    └── 单个用例的超时覆盖（如特定业务场景需要更长时间）
```

#### TestNG超时实现

### 案例2：用例依赖自动排序

**背景：** 当用例之间有复杂依赖时，自动计算执行拓扑顺序。

```python
# dependency_sorter.py
from collections import defaultdict, deque

class TestCaseScheduler:
    """基于拓扑排序的用例执行顺序计算"""
    
    def __init__(self):
        self.dependencies = defaultdict(set)  # A -> {B, C} 表示A依赖B和C
        self.reverse_deps = defaultdict(set)  # 反向依赖（用于通知）
    
    def add_dependency(self, test: str, depends_on: list):
        """添加用例依赖"""
        for dep in depends_on:
            self.dependencies[test].add(dep)
            self.reverse_deps[dep].add(test)
    
    def topological_sort(self) -> list:
        """拓扑排序（Kahn算法）"""
        # 计算入度
        in_degree = defaultdict(int)
        all_tests = set()
        
        for test, deps in self.dependencies.items():
            all_tests.add(test)
            in_degree[test] = len(deps)
            for dep in deps:
                all_tests.add(dep)
        
        # 入度为0的优先执行
        queue = deque([t for t in all_tests if in_degree[t] == 0])
        result = []
        
        while queue:
            test = queue.popleft()
            result.append(test)
            
            # 通知依赖此用例的后续用例
            for dependent in self.reverse_deps.get(test, set()):
                in_degree[dependent] -= 1
                if in_degree[dependent] == 0:
                    queue.append(dependent)
        
        if len(result) != len(all_tests):
            remaining = all_tests - set(result)
            raise ValueError(f"存在循环依赖: {remaining}")
        
        return result
    
    def get_execution_batches(self) -> list:
        """获取并发执行批次（每批内的用例可并发执行）"""
        sorted_tests = self.topological_sort()
        batches = []
        current_batch = []
        current_batch_deps = set()
        
        for test in sorted_tests:
            # 检查是否与当前批次有依赖
            if current_batch_deps & self.dependencies.get(test, set()):
                # 有依赖，开始新批次
                batches.append(current_batch)
                current_batch = [test]
                current_batch_deps = set(self.dependencies.get(test, set()))
            else:
                current_batch.append(test)
                current_batch_deps.update(self.dependencies.get(test, set()))
        
        if current_batch:
            batches.append(current_batch)
        
        return batches


# 使用示例
scheduler = TestCaseScheduler()
scheduler.add_dependency("test_pay", ["test_create_order", "test_login"])
scheduler.add_dependency("test_create_order", ["test_login"])
scheduler.add_dependency("test_refund", ["test_pay"])
scheduler.add_dependency("test_cancel", ["test_create_order"])

# 输出: ['test_login', 'test_create_order', 'test_pay', 'test_cancel', 'test_refund']
```

### 案例3：动态并发度调整

**背景：** 根据系统资源使用情况动态调整并发线程数。

```java
@Component
public class AdaptiveConcurrencyController {
    
    private final AtomicInteger currentThreads = new AtomicInteger(8);
    private static final int MIN_THREADS = 1;
    private static final int MAX_THREADS = 32;
    
    @Scheduled(fixedDelay = 10000) // 每10秒检查一次
    public void adjustConcurrency() {
        // 1. 检查系统CPU使用率
        double cpuUsage = getSystemCpuUsage();
        // 2. 检查被测试系统响应时间
        double avgResponseTime = getAverageResponseTime();
        // 3. 检查失败率
        double failureRate = getRecentFailureRate();
        
        int adjustment = 0;
        
        // CPU过载 -> 减少线程
        if (cpuUsage > 0.8) adjustment -= 2;
        
        // 响应变慢 -> 减少线程
        if (avgResponseTime > 5000) adjustment -= 1;
        
        // 失败率低且资源充足 -> 增加线程
        if (failureRate < 0.02 && cpuUsage < 0.5 && avgResponseTime < 1000) {
            adjustment += 1;
        }
        
        // 应用调整
        if (adjustment != 0) {
            int newThreads = Math.max(MIN_THREADS, 
                Math.min(MAX_THREADS, currentThreads.get() + adjustment));
            currentThreads.set(newThreads);
            log.info("动态调整并发数: {} -> {} (CPU:{:.0f}% RT:{:.0f}ms Fail:{:.1f}%)",
                currentThreads.get() - adjustment, newThreads,
                cpuUsage * 100, avgResponseTime, failureRate * 100);
        }
    }
    
    public int getCurrentConcurrency() {
        return currentThreads.get();
    }
}
```

### 案例4：用例执行时间预测与排序

**背景：** 根据历史执行时间智能排序用例执行顺序，先跑快的用例，让慢的用例在后台跑。

```python
# test_scheduler.py
import json
from pathlib import Path
from collections import defaultdict

class TestDurationPredictor:
    """基于历史数据的用例执行时间预测"""
    
    def __init__(self, history_file: str = "test_history.json"):
        self.history_file = Path(history_file)
        self.history = self._load_history()
    
    def _load_history(self) -> dict:
        if self.history_file.exists():
            return json.loads(self.history_file.read_text())
        return defaultdict(lambda: {"times": [], "avg": 0})
    
    def record(self, test_name: str, duration_ms: float):
        """记录执行时间"""
        record = self.history[test_name]
        record["times"].append(duration_ms)
        # 只保留最近20次
        if len(record["times"]) > 20:
            record["times"] = record["times"][-20:]
        # 加权平均（最近权重高）
        weights = list(range(1, len(record["times"]) + 1))
        record["avg"] = sum(w * t for w, t in zip(weights, record["times"])) / sum(weights)
    
    def predict(self, test_name: str) -> float:
        """预测执行时间（毫秒）"""
        return self.history.get(test_name, {}).get("avg", 30000)
    
    def sort_by_duration(self, test_list: list, fast_first: bool = True) -> list:
        """按预测时间排序"""
        return sorted(test_list, 
            key=lambda t: self.predict(t), 
            reverse=not fast_first)
    
    def estimate_total_time(self, test_list: list, parallel: int = 1) -> float:
        """估算总执行时间（考虑并行）"""
        if parallel == 1:
            return sum(self.predict(t) for t in test_list)
        
        # 贪心分配：将用例分配到最空闲的线程
        sorted_tests = self.sort_by_duration(test_list, fast_first=False)
        threads = [0] * parallel
        
        for test in sorted_tests:
            # 找最空闲的线程
            min_idx = min(range(parallel), key=lambda i: threads[i])
            threads[min_idx] += self.predict(test)
        
        return max(threads)
    
    def save(self):
        """保存历史数据"""
        self.history_file.parent.mkdir(parents=True, exist_ok=True)
        self.history_file.write_text(json.dumps(self.history, indent=2))


# 使用示例
predictor = TestDurationPredictor()
predictor.record("test_login", 2500)
predictor.record("test_search", 4500)
predictor.record("test_batch_import", 30000)

tests = ["test_batch_import", "test_login", "test_search"]
sorted_tests = predictor.sort_by_duration(tests)
print(f"推荐执行顺序: {sorted_tests}")
# 输出: ['test_login', 'test_search', 'test_batch_import']

estimated = predictor.estimate_total_time(tests, parallel=4)
print(f"4线程预估总时间: {estimated/1000:.1f}秒")
```

### 案例5：用例自动分配执行环境

**背景：** 不同用例需要不同的执行环境（有的需要Chrome浏览器，有的只需要API，有的需要移动端），自动将用例分配到合适的环境。

```python
# env_assigner.py
from dataclasses import dataclass, field
from typing import List, Set
import pytest

@dataclass
class ExecutionEnvironment:
    """执行环境"""
    name: str
    capabilities: Set[str]  # 具备的能力
    max_slots: int = 1
    current_slots: int = 0
    
    def can_execute(self, required_caps: Set[str]) -> bool:
        return required_caps.issubset(self.capabilities)
    
    def has_capacity(self) -> bool:
        return self.current_slots < self.max_slots

class EnvironmentAwareScheduler:
    """环境感知的用例调度器"""
    
    def __init__(self):
        self.environments = []
    
    def register_environment(self, env: ExecutionEnvironment):
        self.environments.append(env)
    
    def assign(self, test_cases: List[dict]) -> dict:
        """为用例分配执行环境"""
        assignments = {}
        pending = []
        
        for tc in test_cases:
            required = set(tc.get("requirements", ["api"]))
            candidates = []
            
            for env in self.environments:
                if env.can_execute(required) and env.has_capacity():
                    candidates.append(env)
            
            if candidates:
                # 选择负载最轻的环境
                best = min(candidates, key=lambda e: e.current_slots)
                best.current_slots += 1
                assignments[tc["name"]] = best.name
            else:
                pending.append(tc["name"])
        
        if pending:
            print(f"WARNING: {len(pending)} 个用例无法分配: {pending}")
        
        return assignments


# 使用示例
scheduler = EnvironmentAwareScheduler()
scheduler.register_environment(ExecutionEnvironment(
    "api-node-1", {"api"}, max_slots=10))
scheduler.register_environment(ExecutionEnvironment(
    "chrome-node-1", {"api", "browser", "chrome"}, max_slots=4))
scheduler.register_environment(ExecutionEnvironment(
    "mobile-node-1", {"api", "mobile", "android"}, max_slots=2))

test_cases = [
    {"name": "test_login", "requirements": ["api"]},
    {"name": "test_ui_order", "requirements": ["api", "browser"]},
    {"name": "test_mobile_login", "requirements": ["api", "mobile"]},
]

assignments = scheduler.assign(test_cases)
print(assignments)
# {'test_login': 'api-node-1', 'test_ui_order': 'chrome-node-1', 'test_mobile_login': 'mobile-node-1'}
```

## 本章小结

| 管理维度 | 核心要点 | 关键技术/工具 |
|----------|----------|-------------|
| 用例组织 | 按模块为主，按流程为辅，多维度索引 | TestNG XML / pytest conftest |
| 依赖管理 | 优先共享Fixture，显式依赖备选 | dependsOnMethods / pytest fixture |
| 参数化 | 数据与代码分离，支持外部文件 | DataProvider / @pytest.mark.parametrize |
| 标签分组 | 多维度标签体系，灵活筛选 | groups / pytest.mark |
| 失败重试 | 条件重试，区分真失败和假失败 | IRetryAnalyzer / pytest-rerunfailures |
| 并发执行 | 粒度可控，线程安全，资源池化 | parallel / thread-count / pytest-xdist |
| 超时控制 | 全局默认+用例覆盖+套件上限 | timeOut / pytest-timeout |

> 测试用例层是自动化测试的"指挥官"——它决定了谁（哪些用例）、什么时候（执行顺序/定时）、怎么跑（并发/重试/超时）、用什么数据（参数化）、在什么环境（标签筛选）执行。一个好的用例管理层让测试执行井然有序，而非混乱不堪。
