## 1. 性能测试类型

### 1.1 六大性能测试类型对比

| 测试类型 | 目的 | 负载特点 | 持续时间 | 关键指标 | 典型场景 |
|----------|------|----------|----------|----------|----------|
| **基准测试** | 建立性能基线 | 低负载 | 短（5-15分钟） | 单用户响应时间、资源基准 | 新系统上线前、每次大版本发布前 |
| **负载测试** | 验证预期负载下的性能 | 逐步增加至预期值 | 中（30分钟-2小时） | 响应时间、TPS、资源利用率 | 日常性能回归验证 |
| **压力测试** | 找到系统性能拐点和极限 | 持续增加直至系统崩溃 | 短-中（直到崩溃） | 最大TPS、崩溃点、恢复能力 | 大促前容量评估 |
| **稳定性测试** | 验证长期运行的稳定性 | 预期负载的70-80% | 长（8-72小时） | 内存泄漏、GC情况、响应时间波动 | 长期运行系统的可靠性验证 |
| **并发测试** | 验证并发处理能力 | 瞬时高并发 | 短（5-15分钟） | 并发用户数、响应时间、错误率 | 秒杀、抢购场景 |
| **容量测试** | 确定系统的最大容量 | 逐步增加 | 中 | 最大并发用户数、最大TPS | 容量规划和扩容决策 |

### 1.2 负载模型设计

负载模型决定了性能测试的逼真度。常见的负载模型包括：

| 模型类型 | 描述 | 适用场景 | 示例 |
|----------|------|----------|------|
| **线性递增** | 负载随时间线性增长 | 找到拐点 | 每5分钟增加50并发用户 |
| **阶梯递增** | 分阶段增加负载 | 观察每阶段的稳定表现 | 100→200→300→400 VU，每阶段10分钟 |
| **脉冲/尖峰** | 瞬间大流量涌入 | 秒杀/抢购 | 在5秒内从0到10000并发 |
| **随机波动** | 模拟真实用户的随机行为 | 在线商城/社交应用 | 负载在500-800VU间随机波动 |
| **基于场景** | 根据用户行为模型分配比例 | 混合业务场景 | 70%浏览+20%搜索+8%下单+2%支付 |

### 1.3 测试环境与生产环境的差异处理

| 方面 | 测试环境 | 生产环境 | 换算策略 |
|------|----------|----------|----------|
| **硬件配置** | 通常低于生产 | 高配/集群 | 等比例缩容或按CPU核心数换算 |
| **数据量** | 百万级 | 亿级 | 按数据量增长曲线外推 |
| **网络** | 局域网 | 公网/多地域 | 考虑网络延迟差异 |
| **第三方依赖** | Mock | 真实 | 排除第三方干扰或约定SLA |

**容量换算公式**（近似）：
```
生产预估TPS = 测试环境TPS × (生产CPU核心数 / 测试CPU核心数) × 架构因子
架构因子：单体0.7-0.8，微服务集群0.5-0.6（考虑网络开销）
```

## 2. 性能指标体系

### 2.1 核心指标详解

#### 2.1.1 吞吐量指标

| 指标 | 全称 | 计算方式 | 含义 | 目标值参考 |
|------|------|----------|------|-----------|
| **TPS** | Transactions Per Second | 总事务数 / 总时间(秒) | 每秒处理的事务数 | Web应用通常100-10000+ |
| **QPS** | Queries Per Second | 总查询数 / 总时间(秒) | 每秒处理的查询请求数 | 与TPS类似，一个事务可能包含多个查询 |
| **HPS** | Hits Per Second | 总请求数 / 总时间(秒) | 每秒HTTP请求数 | 通常HPS > TPS |
| **吞吐量** | Throughput | 数据量 / 时间 | 单位时间内传输的数据量 | MB/s或GB/s |

**TPS与QPS的区别**：
- TPS关注业务层面的事务（如一个下单操作 = 登录+浏览+加购+下单+支付）
- QPS关注技术层面的单个请求
- 一个事务通常包含多个查询，所以 QPS >= TPS

#### 2.1.2 响应时间指标

| 指标 | 含义 | 计算方式 | 重要性 |
|------|------|----------|--------|
| **Avg（平均值）** | 所有请求响应时间的算术平均 | Sum(rt) / N | 参考价值有限，容易被长尾拖高 |
| **P50（中位数）** | 50%的请求响应时间低于此值 | 排序后第50分位 | 反映"典型用户"的体验 |
| **P90** | 90%的请求响应时间低于此值 | 排序后第90分位 | 反映大多数用户的体验上限 |
| **P95** | 95%的请求响应时间低于此值 | 排序后第95分位 | 核心SLA指标之一 |
| **P99** | 99%的请求响应时间低于此值 | 排序后第99分位 | 反映极端情况，长尾优化的关键 |
| **P999** | 99.9%的请求响应时间低于此值 | 排序后第99.9分位 | 高可靠性系统的核心指标 |
| **Min/Max** | 最小和最大响应时间 | — | 发现极值问题 |

**示例分析**：
```
假设100次请求的响应时间（ms）：
正常：90, 95, 98, 100, 102, 105, 108, 110, 115, 120... (共95个，在90-200之间)
异常：500, 1000, 2000, 5000, 30000 (共5个)

Avg = 约600ms  ← 被长尾严重拉高
P50 = 110ms    ← 大部分用户体验良好
P95 = 200ms    ← 95%的用户在200ms内得到响应
P99 = 5000ms   ← 存在严重长尾问题！
```

**长尾问题分析**：如果P50和P99差距过大（如P50=50ms, P99=3000ms），说明存在间歇性的性能问题，可能原因：
- GC停顿（Full GC导致STW）
- 锁竞争
- 网络抖动
- 数据库慢查询
- 缓存失效/击穿

#### 2.1.3 并发指标

| 指标 | 含义 | 计算公式 |
|------|------|----------|
| **并发用户数** | 同一时刻与系统交互的用户数 | 在线用户数 × 活跃比例 |
| **在线用户数** | 已登录系统的用户总数 | 从连接数/会话数统计 |
| **思考时间** | 用户两次操作之间的间隔 | 模拟真实用户的行为节奏 |

**Little定律**：
```
并发用户数(VU) = TPS × 平均响应时间(RT)

例如：TPS = 500, RT = 0.5秒
VU = 500 × 0.5 = 250个并发用户

反过来：如果VU = 1000, RT = 0.5秒
需要的TPS = 1000 / 0.5 = 2000
```

### 2.2 SLA指标体系参考

| 业务场景 | P95响应时间 | P99响应时间 | 错误率 | 可用性 |
|----------|-------------|-------------|--------|--------|
| **搜索/查询** | < 200ms | < 500ms | < 0.1% | 99.9% |
| **下单/支付** | < 500ms | < 1s | < 0.01% | 99.99% |
| **消息推送** | < 1s | < 3s | < 1% | 99.9% |
| **报表导出** | < 10s | < 30s | < 0.5% | 99.5% |
| **视频播放** | 首帧<1s | 首帧<2s | < 0.5% | 99.9% |
| **实时交易** | < 10ms | < 50ms | < 0.001% | 99.999% |

### 3.2 JMeter核心使用

#### 3.2.1 基本组件

| 组件 | 用途 | 关键配置 |
|------|------|----------|
| **Thread Group** | 模拟用户线程组 | 线程数、Ramp-Up时间、循环次数 |
| **HTTP Request** | 发送HTTP请求 | 协议、域名、端口、方法、路径、参数、Body |
| **CSV Data Set Config** | 参数化数据源 | 文件路径、变量名、分隔符、循环模式 |
| **Response Assertion** | 响应断言 | 响应码、响应文本、JSON Path |
| **Constant/Uniform Random Timer** | 模拟用户思考时间 | 固定/随机延迟 |
| **View Results Tree** | 查看请求详情 | 调试用，压测时禁用 |
| **Summary Report** | 汇总报告 | 聚合数据 |
| **Aggregate Report** | 聚合报告 | Avg/Min/Max/Pct响应时间、吞吐量、错误率 |

### 3.3 Locust使用示例

```python
from locust import HttpUser, task, between, TaskSet
import random
import json

class BrowseBehavior(TaskSet):
    """浏览行为 - 占比70%"""
    
    @task(5)  # 权重5
    def view_homepage(self):
        with self.client.get("/api/v1/home", catch_response=True) as resp:
            if resp.status_code != 200:
                resp.failure(f"首页加载失败: {resp.status_code}")

    @task(3)  # 权重3
    def search_product(self):
        keywords = ["手机", "电脑", "耳机", "键盘", "鼠标"]
        kw = random.choice(keywords)
        with self.client.get(f"/api/v1/search?keyword={kw}&page=1", 
                            name="/api/v1/search",
                            catch_response=True) as resp:
            if resp.status_code == 200:
                data = resp.json()
                if data.get("code") != 200:
                    resp.failure(f"搜索接口业务错误: {data.get('message')}")
            else:
                resp.failure(f"搜索接口HTTP错误: {resp.status_code}")

    @task(2)  # 权重2
    def view_product_detail(self):
        product_id = random.randint(1, 10000)
        self.client.get(f"/api/v1/products/{product_id}", name="/api/v1/products/{id}")

class OrderBehavior(TaskSet):
    """下单行为 - 占比20%"""
    
    @task(1)
    def create_order(self):
        # 加入购物车
        cart_resp = self.client.post("/api/v1/cart/add",
            json={"product_id": random.randint(1, 5000), "quantity": 1},
            name="/api/v1/cart/add")
        
        if cart_resp.status_code != 200:
            return
        
        # 创建订单
        order_resp = self.client.post("/api/v1/orders",
            json={
                "address_id": 1,
                "payment_method": random.choice(["alipay", "wechat", "card"]),
                "remark": "性能测试订单"
            },
            name="/api/v1/orders")
        
        if order_resp.status_code != 201:
            return
        
        try:
            order_id = order_resp.json()["data"]["id"]
            # 支付
            self.client.post(f"/api/v1/orders/{order_id}/pay",
                json={"payment_method": "alipay", "password": "123456"},
                name="/api/v1/orders/{id}/pay")
        except (KeyError, json.JSONDecodeError):
            pass

class AdminBehavior(TaskSet):
    """管理行为 - 占比10%"""
    
    def on_start(self):
        """管理员登录"""
        resp = self.client.post("/api/v1/auth/login",
            json={"username": "admin", "password": "Admin@123"})
        if resp.status_code == 200:
            self.token = resp.json()["data"]["token"]
        else:
            self.token = None
    
    @task(1)
    def view_dashboard(self):
        if not self.token:
            return
        self.client.get("/api/v1/admin/dashboard",
            headers={"Authorization": f"Bearer {self.token}"})

class EcommerceUser(HttpUser):
    wait_time = between(1, 5)  # 用户思考时间1-5秒
    tasks = {BrowseBehavior: 70, OrderBehavior: 20, AdminBehavior: 10}
```

### 3.4 k6使用示例

```javascript
import http from 'k6/http';
import { check, sleep, group } from 'k6';
import { Rate, Trend, Counter } from 'k6/metrics';

// 自定义指标
const errorRate = new Rate('errors');
const orderTrend = new Trend('order_create_time');
const paymentCounter = new Counter('payment_success');

export const options = {
    stages: [
        { duration: '2m', target: 100 },   // 热身：2分钟升至100VU
        { duration: '5m', target: 100 },   // 保持100VU 5分钟
        { duration: '3m', target: 500 },   // 加压：3分钟升至500VU
        { duration: '10m', target: 500 },  // 保持500VU 10分钟
        { duration: '3m', target: 1000 },  // 加压：3分钟升至1000VU
        { duration: '10m', target: 1000 }, // 保持1000VU 10分钟
        { duration: '2m', target: 0 },     // 冷却
    ],
    thresholds: {
        'http_req_duration': ['p(95)<500', 'p(99)<1000'],  // P95<500ms, P99<1s
        'errors': ['rate<0.01'],                            // 错误率<1%
        'http_req_failed': ['rate<0.01'],
    },
};

export default function () {
    const BASE_URL = 'https://api-staging.example.com/v1';
    const params = {
        headers: {
            'Content-Type': 'application/json',
            'Authorization': 'Bearer test-token-xxx',
        },
    };

    // 场景1：浏览首页
    group('浏览首页', () => {
        const res = http.get(`${BASE_URL}/home`, params);
        check(res, {
            '首页200': (r) => r.status === 200,
            '首页响应时间<200ms': (r) => r.timings.duration < 200,
        }) || errorRate.add(1);
    });

    sleep(Math.random() * 3 + 1);  // 1-4秒思考时间

    // 场景2：搜索
    group('搜索商品', () => {
        const keyword = ['手机', '电脑', '耳机'][Math.floor(Math.random() * 3)];
        const res = http.get(`${BASE_URL}/search?keyword=${keyword}`, params);
        check(res, {
            '搜索200': (r) => r.status === 200,
            '搜索返回数据': (r) => r.json().data != null,
        }) || errorRate.add(1);
    });

    sleep(Math.random() * 2 + 1);

    // 场景3：下单（概率20%）
    if (Math.random() < 0.2) {
        group('创建订单', () => {
            const payload = JSON.stringify({
                product_id: Math.floor(Math.random() * 5000) + 1,
                quantity: Math.floor(Math.random() * 3) + 1,
                address_id: 1,
            });
            const orderStart = Date.now();
            const res = http.post(`${BASE_URL}/orders`, payload, params);
            orderTrend.add(Date.now() - orderStart);
            
            const success = check(res, {
                '下单成功': (r) => r.status === 201,
            });
            
            if (success && res.json().data && res.json().data.id) {
                paymentCounter.add(1);
            } else {
                errorRate.add(1);
            }
        });
    }

    sleep(Math.random() * 2);
}

// 测试结束时输出自定义摘要
export function handleSummary(data) {
    return {
        'summary.json': JSON.stringify(data),
        stdout: `
=============================================
  性能测试完成
=============================================
总请求数: ${data.metrics.http_reqs.values.count}
平均响应时间: ${data.metrics.http_req_duration.values.avg.toFixed(2)}ms
P95响应时间: ${data.metrics.http_req_duration.values['p(95)'].toFixed(2)}ms
P99响应时间: ${data.metrics.http_req_duration.values['p(99)'].toFixed(2)}ms
错误率: ${(data.metrics.errors.values.rate * 100).toFixed(2)}%
订单创建平均时间: ${data.metrics.order_create_time?.values.avg?.toFixed(2) || 'N/A'}ms
支付成功数: ${data.metrics.payment_success?.values.count || 0}
=============================================
`,
    };
}
```

## 4. 性能测试流程

### 4.1 标准八步流程

```
需求分析 → 场景设计 → 脚本开发 → 环境准备 → 测试执行 → 监控分析 → 性能调优 → 回归验证
```

### 4.2 各阶段详解

#### 阶段1：需求分析

| 分析维度 | 具体内容 | 产出物 |
|----------|----------|--------|
| **业务目标** | 支持多少用户？日均PV/UV？峰值QPS？ | 性能需求文档 |
| **技术架构** | 服务拓扑、中间件、数据库、缓存、CDN | 架构图 + 调用链 |
| **历史数据** | 线上监控数据、历史峰值、增长趋势 | 数据报告 |
| **SLA要求** | 响应时间、可用性、错误率要求 | SLA指标表 |
| **风险识别** | 可能的瓶颈点（新功能、复杂查询、大促场景） | 风险清单 |

**性能需求推导示例**：
```
业务需求："双11支持100万用户同时在线，50万人同时抢购"

推导过程：
1. 在线用户100万，假设活跃比例10% = 10万活跃用户
2. 抢购高峰持续时间约10分钟，50万人参与 = 5000单/秒
3. 一个下单操作包含约5个后端请求
4. 后端需要支撑：5000 × 5 = 25000 TPS（下单链路）
5. 加上浏览、搜索等请求，预估总QPS约10万
6. 考虑系统的当前能力，决定扩容倍数
```

#### 阶段2：场景设计

| 场景元素 | 设计要点 |
|----------|----------|
| **用户模型** | 用户角色分布、行为路径、操作频率 |
| **数据模型** | 数据量级、数据分布（冷热数据比例） |
| **负载模型** | 并发数增长曲线、各场景的配比 |
| **监控模型** | 需要监控的指标和组件列表 |

#### 阶段3：脚本开发

| 开发要点 | 说明 |
|----------|------|
| **参数化** | 用户名、商品ID等使用CSV数据源，避免缓存命中 |
| **关联处理** | 提取响应中的Token、SessionID、订单ID用于后续请求 |
| **断言校验** | 不只检查HTTP状态码，更要检查业务逻辑正确性 |
| **思考时间** | 模拟真实用户行为的间隔，不要太快或太慢 |
| **数据准备** | 准备足够的测试数据（商品、用户、库存等） |

#### 阶段4：环境准备

| 检查项 | 要点 |
|--------|------|
| **配置一致性** | 中间件连接池、JVM参数、超时配置与生产一致 |
| **数据预热** | 提前执行常见查询让缓存/缓冲池预热 |
| **网络隔离** | 确保测试流量不与真实流量混合 |
| **监控就绪** | APM、服务器监控、日志采集确认可用 |

#### 阶段5：测试执行

执行顺序：**基准测试 → 负载测试 → 压力测试 → 稳定性测试**

```
执行时间线：
09:00 - 09:15  环境检查 & 数据预热
09:15 - 09:30  基准测试（单用户、10并发）
09:30 - 11:00  负载测试（逐步升到预期负载）
11:00 - 12:00  压力测试（找到性能拐点和极限）
12:00 - 13:00  午休/环境恢复
13:00 - 21:00  稳定性测试（8小时持续运行）
21:00 - 22:00  报告整理
```

#### 阶段6：监控分析

关键分析思路：**从外到内，逐层排查**

```
终端/客户端
  ↓ 网络延迟
CDN / 负载均衡器
  ↓ 请求分发
API网关
  ↓ 路由/限流
应用服务（CPU、内存、线程、GC、连接池）
  ↓ 数据访问
缓存层（Redis/Memcached）
  ↓ miss
数据库（MySQL/PostgreSQL/MongoDB）
  ↓ 磁盘IO
存储
```

#### 阶段7：性能调优

| 层级 | 常见优化手段 | 预期收益 |
|------|-------------|----------|
| **应用层** | 缓存策略、异步处理、批量操作、连接池调优 | 2-10倍 |
| **数据库层** | 索引优化、SQL改写、读写分离、分库分表 | 2-100倍 |
| **中间件层** | 消息队列削峰、缓存预热、限流降级 | 1.5-5倍 |
| **系统层** | JVM GC调优、内核参数优化、连接数调整 | 1.2-3倍 |
| **架构层** | 水平扩展、CDN加速、多级缓存 | 线性可扩展 |

#### 阶段8：回归验证

- 调优后重新执行相同场景
- 对比调优前后的关键指标变化
- 确认没有引入新的问题
- 更新性能基线

## 5. 服务器监控

### 5.1 关键监控指标

#### 5.1.1 CPU监控

| 指标 | 含义 | 正常范围 | 告警阈值 |
|------|------|----------|----------|
| **CPU使用率** | 整体CPU利用率 | 30-70% | >85%（持续5分钟） |
| **CPU User** | 用户态CPU占用 | — | 过高说明应用计算密集 |
| **CPU System** | 内核态CPU占用 | <30% | 过高说明频繁系统调用 |
| **CPU IOWait** | 等待IO的CPU时间 | <10% | >20%说明IO瓶颈 |
| **CPU Steal** | 虚拟化环境被宿主机偷走的CPU | <1% | >5%说明宿主机超卖 |
| **Load Average** | 1min/5min/15min平均负载 | <CPU核心数 | >CPU核心数的2倍 |

**Load Average解读**：
```
单核CPU: load average > 1 说明有进程在等待CPU
4核CPU:  load average < 4 正常; > 8 严重过载

关键在于趋势：load持续上升 vs 短暂尖峰
```

#### 5.1.2 内存监控

| 指标 | 含义 | 告警阈值 |
|------|------|----------|
| **内存使用率** | 已用物理内存占比 | >85% |
| **Swap使用** | 是否发生内存交换 | 任何Swap使用都值得关注 |
| **OOM Kill** | 是否触发Out of Memory | 发生即为严重问题 |
| **Cache/Buffer** | 文件系统缓存和缓冲区 | 可回收，不算真正的使用 |

**JVM内存（Java应用专用）**：

| 指标 | 说明 | 告警 |
|------|------|------|
| **Heap Used/Max** | 堆内存使用率 | >85% |
| **GC Frequency** | GC频率（次/分钟） | 突然增加 |
| **GC Pause Time** | GC停顿时间 | Full GC > 1s |
| **Metaspace** | 元空间使用 | >90% |
| **Thread Count** | 线程数 | 持续增长（泄漏） |

#### 5.1.3 磁盘IO监控

| 指标 | 含义 | 告警阈值 |
|------|------|----------|
| **ioutil** | 磁盘IO使用率 | >80% |
| **iowait** | CPU等待IO的时间 | >20% |
| **r_s / w_s** | 每秒读写次数 | 对比基线 |
| **rkB/s / wkB/s** | 每秒读写数据量 | 对比基线 |
| **await** | IO请求的平均等待时间 | >50ms（SSD）/ >200ms（HDD） |
| **avgqu-sz** | 平均IO队列长度 | >1说明有积压 |

#### 5.1.4 网络监控

| 指标 | 含义 | 告警阈值 |
|------|------|----------|
| **网卡流量** | 入/出流量 Mbps | >80%带宽 |
| **TCP连接数** | ESTABLISHED/TIME_WAIT/CLOSE_WAIT | TIME_WAIT过多→端口耗尽 |
| **丢包率** | 网络丢包比例 | >0.1% |
| **Socket等待队列** | Recv-Q/Send-Q | >0说明有积压 |

## 6. 常见性能瓶颈与调优

### 6.1 瓶颈分类与解决方案

| 瓶颈类型 | 典型表现 | 诊断方法 | 解决方案 |
|----------|----------|----------|----------|
| **CPU瓶颈** | CPU使用率持续>85%，响应时间增加 | top查看CPU分布，perf分析热点函数 | 代码优化、缓存、异步、水平扩展 |
| **内存泄漏** | 内存使用持续上升，Full GC频繁 | heap dump分析、GC日志 | 修复泄漏代码、调整GC策略 |
| **数据库慢查询** | 数据库CPU高，慢查询日志增长 | 慢查询日志、EXPLAIN分析 | 加索引、SQL改写、读写分离 |
| **连接池耗尽** | 获取连接超时，请求排队 | 查看连接池监控 | 增加池大小、检查连接泄漏、缩短等待超时 |
| **网络瓶颈** | 带宽打满、丢包 | iftop/nethogs/iperf | 升级带宽、CDN、数据压缩 |
| **锁竞争** | 线程阻塞，响应时间毛刺 | 线程dump、死锁检测 | 减小锁粒度、乐观锁、无锁数据结构 |
| **缓存问题** | 缓存命中率低、缓存穿透/击穿 | Redis监控 | 调整过期策略、布隆过滤器、缓存预热 |
| **线程池耗尽** | 请求被拒绝或排队，RejectedExecutionException | 线程dump | 调整线程池参数、异步处理、降级 |

### 6.2 数据库常见性能优化

```sql
-- 常见优化SQL示例

-- 问题1：全表扫描
-- 优化前：SELECT * FROM orders WHERE user_id = 12345;
-- 优化方案：添加索引
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 问题2：索引失效（隐式类型转换）
-- 优化前：SELECT * FROM orders WHERE phone = 13800138000;  -- phone是varchar类型
-- 优化方案：使用正确的类型
SELECT * FROM orders WHERE phone = '13800138000';

-- 问题3：回表查询过多
-- 优化前：SELECT * FROM orders WHERE user_id = 123 AND status = 'paid';
-- 优化方案：使用覆盖索引
CREATE INDEX idx_orders_user_status ON orders(user_id, status, created_at, amount);
SELECT user_id, status, created_at, amount FROM orders WHERE user_id = 123 AND status = 'paid';

-- 问题4：大偏移量分页
-- 优化前：SELECT * FROM orders ORDER BY id LIMIT 100000, 20;
-- 优化方案：基于ID的游标分页
SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 20;

-- 问题5：N+1查询
-- 优化方案：使用JOIN或批量查询替代循环查询
SELECT o.*, u.username 
FROM orders o 
LEFT JOIN users u ON o.user_id = u.id 
WHERE o.created_at > '2025-01-01';
```

### 6.3 缓存策略优化

| 缓存策略 | 适用场景 | 优缺点 |
|----------|----------|--------|
| **Cache Aside** | 读多写少 | 逻辑简单，可能存在短暂不一致 |
| **Read/Write Through** | 缓存层完全代理 | 一致性更好，但缓存故障影响大 |
| **Write Behind** | 写操作频繁 | 写性能好，但有数据丢失风险 |
| **Refresh Ahead** | 热点数据 | 减少缓存失效时的延迟 |

**缓存常见问题及应对**：

| 问题 | 描述 | 解决方案 |
|------|------|----------|
| **缓存穿透** | 查询不存在的key，每次都穿透到DB | 布隆过滤器、空值短期缓存 |
| **缓存击穿** | 热点key过期，瞬时大量请求到DB | 互斥锁（Mutex Key）加载、逻辑过期 |
| **缓存雪崩** | 大量key同时过期，DB压力骤增 | 过期时间加随机值、多级缓存、限流降级 |
| **热点key** | 单个key请求量极大，单节点瓶颈 | 本地缓存、读写分离、key拆分 |

## 7. 实战案例

### 案例1：电商大促性能测试

**背景**：某电商平台备战双11，需要验证系统能否支撑预估的峰值流量：50万QPS。

**测试策略**：
1. **全链路压测**：从CDN→网关→应用→缓存→数据库完整压测
2. **容量预估**：基于历史数据和增长趋势，预估需要扩容的规模
3. **限流降级验证**：验证超出容量时，限流和降级策略是否正确生效

**关键发现**：
- Redis集群热点key问题：首页商品缓存的单个key QPS达到5万，导致单节点过载
  - 解决：本地缓存(Guava Cache) + Redis Cluster + key分片
- 数据库连接池不足：高并发下连接池耗尽导致大量超时
  - 解决：调整连接池大小(20→100)，优化慢SQL减少连接占用时间
- 下单接口P99达到5秒：分布式事务回滚逻辑过重
  - 解决：采用TCC模式 + 异步通知替代同步回滚

**压测数据**：
| 并发数 | TPS | P95(ms) | P99(ms) | 错误率 | CPU | 内存 |
|--------|-----|---------|---------|--------|-----|------|
| 1000 | 5000 | 120 | 250 | 0% | 40% | 55% |
| 3000 | 12000 | 180 | 400 | 0.01% | 60% | 65% |
| 5000 | 18000 | 350 | 800 | 0.05% | 78% | 70% |
| 8000 | 22000 | 800 | 3000 | 1.2% | 92% | 85% |
| 10000 | 21000 | 2000 | 8000 | 5% | 98% | 92% |

系统拐点在8000并发左右。

### 案例2：金融交易系统性能测试

**背景**：某证券交易所低延迟交易系统，对响应时间要求极高。

**性能要求**：
- P99延迟 < 10ms
- 最大延迟 < 50ms
- TPS > 100,000

**测试方法**：
1. 使用定制化压测工具直接发送二进制协议消息
2. 高精度时间测量（纳秒级）
3. GC策略调优验证（ZGC/Shenandoah）
4. CPU绑核（CPU Affinity）避免上下文切换

**调优结果**：
| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| P50延迟 | 3.2ms | 1.1ms |
| P99延迟 | 28ms | 5ms |
| Max延迟 | 150ms | 18ms |
| TPS | 45000 | 120000 |

### 案例3：SaaS平台多租户性能隔离测试

**背景**：某SaaS平台，多个企业租户共享同一套集群，需要验证租户间的性能隔离。

**测试策略**：
1. **单租户基准**：单个租户在独占资源下的性能
2. **多租户并行**：10/50/100个租户同时运行各自业务
3. **"吵闹邻居"测试**：一个大租户运行大量消耗资源的操作，观察其他租户是否受影响
4. **数据隔离验证**：租户A的数据查询不应该消耗租户B的数据库资源

**关键发现**：
- 数据库层面缺乏有效隔离：一个租户的大批量导出操作消耗了30%的数据库CPU，影响其他租户
  - 解决：实现数据库连接级别的资源隔离 + 慢SQL自动限流

### 案例4：视频直播系统弹幕性能测试

**背景**：某直播平台需要验证弹幕系统在千万级同时在线的表现。

**性能要求**：
- 弹幕分发延迟 < 200ms
- 弹幕吞吐量 > 100万条/秒
- 支持单个直播间100万用户同时接收弹幕

**测试方案**：
1. 使用WebSocket协议进行弹幕长连接测试
2. 自定义压测工具模拟大量WebSocket连接
3. 测试消息广播的扩散效率（1条弹幕→N个客户端的时间差）
4. 测试不同人气值直播间的弹幕频率差异

**架构优化**：
- 长连接管理：分层架构（接入层→逻辑层→分发层）
- 消息分发：按直播间ID一致性Hash进行分片
- 热门直播间：独立集群承载，自动隔离

### 案例5：API网关限流性能测试

**背景**：某开放平台API网关需要对不同API Key实现精确的限流控制。

**限流策略验证**：
| 策略 | 验证方法 | 工具 |
|------|----------|------|
| 固定窗口 | 在1秒窗口内发送N+1个请求，验证第N+1个被限流 | Locust |
| 滑动窗口 | 跨窗口边界发送请求，验证精确度 | k6 |
| 令牌桶 | 突发流量测试：前0.5秒全速→后半秒限速 | 自研 |
| 漏桶 | 恒定速率输入→验证输出是否被平滑 | JMeter |
| 分布式限流 | 多台压力机同时发送，验证全局配额准确性 | 多agent JMeter |

**发现的问题**：
- Redis实现的分布式限流在高并发下存在计数竞态条件
  - 解决：使用Redis Lua脚本保证原子性

## 8. 本章小结

1. **性能测试是系统工程**：不是简单的"跑一下JMeter"，而是从需求分析到调优回归的完整闭环
2. **指标体系是核心**：关注P95/P99而非平均值，关注业务TPS而非仅HTTP QPS
3. **工具只是手段**：JMeter适合传统企业，k6适合云原生，Locust适合Python团队——根据技术栈选择
4. **监控先行**：没有完善的监控体系，性能测试就是"盲测"
5. **调优需要分层**：从应用层→数据库层→中间件层→系统层→架构层逐层分析优化
6. **性能基线化**：每次重大变更前建立性能基线，防止性能退化
7. **左移**：性能测试不再是大促前的"一次性项目"，应融入日常CI/CD流程
8. **真实为本**：负载模型、测试数据、环境配置越接近生产，测试结果越有价值
