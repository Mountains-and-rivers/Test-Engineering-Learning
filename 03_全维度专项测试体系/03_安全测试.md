## 1. OWASP Top 10详解

OWASP（Open Web Application Security Project）每3-4年发布一次Web应用十大安全风险。以下内容基于最新版本，结合测试视角深入解析。

### 1.1 OWASP Top 10 速览表

| 排名 | 风险名称（中/英） | 风险简述 | 风险等级 |
|------|-------------------|----------|----------|
| **A01** | 访问控制失效 / Broken Access Control | 未正确实施权限控制，用户可以访问或操作非授权资源 | 高 |
| **A02** | 加密失效 / Cryptographic Failures | 敏感数据未加密、弱加密算法、密钥管理不当 | 高 |
| **A03** | 注入 / Injection | SQL注入、NoSQL注入、OS命令注入、LDAP注入等 | 严重 |
| **A04** | 不安全设计 / Insecure Design | 设计阶段缺少安全考量，缺乏威胁建模 | 中-高 |
| **A05** | 安全配置错误 / Security Misconfiguration | 默认配置未修改、错误信息泄露、未修补漏洞 | 高 |
| **A06** | 使用含已知漏洞组件 / Vulnerable Components | 使用了已知漏洞的第三方库、框架、中间件 | 高 |
| **A07** | 认证与鉴权失效 / Identification and Authentication Failures | 弱密码策略、暴力破解防护不足、会话管理缺陷 | 严重 |
| **A08** | 软件与数据完整性失效 / Software and Data Integrity Failures | 不安全的CI/CD、未验证的更新、不受信的反序列化 | 高 |
| **A09** | 安全日志与监控失效 / Security Logging and Monitoring Failures | 日志不足、监控缺失、告警响应不及时 | 中 |
| **A10** | 服务端请求伪造(SSRF) / Server-Side Request Forgery | 服务端未校验用户提供的URL即发起请求 | 高 |

### 1.2 A01 - 访问控制失效

**典型漏洞场景**：
1. 水平越权（IDOR - Insecure Direct Object Reference）：用户A通过修改URL参数访问用户B的数据
2. 垂直越权：普通用户执行管理员操作
3. CORS配置不当：允许任意源跨域访问敏感接口
4. 目录遍历：访问不应被访问的路径

**测试方法**：
```
测试案例1 - IDOR测试：
用户A登录 → 获取自己的订单 /api/orders/1001 → 成功
用户A登录 → 修改URL访问 /api/orders/1002 → 是否返回其他用户数据？
预期：返回404或403

测试案例2 - 垂直越权：
普通用户登录 → 访问 /api/admin/users → 应返回403
普通用户登录 → 访问 /api/admin/settings → 应返回403

测试案例3 - 未授权访问：
不登录 → 访问 /api/sensitive/data → 应返回401

测试案例4 - CORS测试：
从恶意域名发送跨域请求 → 检查Access-Control-Allow-Origin
是否包含恶意域名？通配符*是否被滥用？
```

**防护措施**：
- 默认拒绝：所有资源默认不可访问，显式授权
- 服务端统一鉴权：每个API请求都在服务端校验权限
- CORS白名单：仅允许受信域名
- 速率限制：防止暴力枚举ID

### 1.3 A02 - 加密失效

**常见问题**：
| 问题 | 说明 | 示例 |
|------|------|------|
| 明文传输 | 敏感数据通过HTTP明文传输 | 登录接口使用HTTP而非HTTPS |
| 弱哈希算法 | 使用MD5/SHA1存储密码（注：MD5/SHA1是哈希函数，而非加密算法） | password = MD5(password) |
| 硬编码密钥 | 代码中包含密钥/证书 | API_SECRET = "sk-12345abc" 写在代码中 |
| 密钥管理不当 | 密钥未轮换、权限过大 | 所有服务共享同一密钥 |
| 不安全的随机数 | 使用非密码学安全的随机数 | Math.random()生成Token |

**测试方法**：
```python
# 测试密码存储强度
import hashlib

def test_password_hashing():
    # 检测：如果数据库中两个相同密码的用户hash不同（salt不同），则是好的
    # 检测：是否使用了bcrypt/scrypt/argon2，而非MD5/SHA1/SHA256
    
    # 弱实现
    weak_hash = hashlib.md5("password123".encode()).hexdigest()
    # "482c811da5d5b4bc6d497ffa98491e38" — 可被彩虹表秒破
    
    # 强实现（bcrypt示例）
    import bcrypt
    strong_hash = bcrypt.hashpw(b"password123", bcrypt.gensalt(rounds=12))
    # $2b$12$... — 每次不同，计算消耗大

# 测试HTTPS强制
def test_https_enforcement():
    import requests
    # HTTP请求应该被301重定向到HTTPS
    resp = requests.get("http://example.com/api/login", allow_redirects=False)
    assert resp.status_code in [301, 302, 403]
    assert resp.headers.get("Location", "").startswith("https://")
```

### 1.4 A03 - 注入

**注入类型全景**：

| 注入类型 | 攻击面 | 测试Payload示例 | 影响 |
|----------|--------|----------------|------|
| **SQL注入** | 数据库查询参数 | `' OR '1'='1' --` | 数据泄露、数据篡改 |
| **NoSQL注入** | MongoDB查询 | `{"$ne": null}` | 绕过认证、数据泄露 |
| **OS命令注入** | 系统命令调用 | `; cat /etc/passwd` | 服务器被控制 |
| **LDAP注入** | LDAP查询 | `*)(uid=*))(|(uid=*` | 认证绕过 |
| **XPath注入** | XML查询 | `' or '1'='1` | XML数据泄露 |
| **模板注入(SSTI)** | 模板引擎 | `{{7*7}}` | 远程代码执行 |
| **表达式注入** | SpEL/OGNL | `${7*7}` | 远程代码执行 |

**SQL注入详细测试**：
```sql
-- 测试1：基础注入 - 绕过登录
-- 输入用户名：admin' --
-- 输入密码：（任意）
-- 生成SQL：SELECT * FROM users WHERE username='admin' --' AND password='...'
-- 如果单引号未转义且存在admin用户，则绕过密码验证

-- 测试2：UNION注入 - 窃取数据
' UNION SELECT username, password FROM users --

-- 测试3：布尔盲注 - 逐字符猜解
' AND SUBSTRING((SELECT password FROM users WHERE username='admin'), 1, 1) = 'a' --

-- 测试4：时间盲注 - 无回显场景 (MySQL特有: IF()和SLEEP(); PostgreSQL等价: CASE WHEN ... THEN pg_sleep(5) ELSE 0 END)
' AND IF(SUBSTRING((SELECT password FROM users WHERE username='admin'), 1, 1) = 'a', SLEEP(5), 0) --

-- 测试5：堆叠查询
'; DROP TABLE users; --
```

**自动化注入测试（sqlmap）**：
```bash
# 基本扫描
sqlmap -u "https://target.com/api/users?id=1" --batch

# 指定参数扫描POST请求
sqlmap -u "https://target.com/api/search" \
  --data="keyword=test&category=1" \
  --method=POST

# 使用Cookie认证
sqlmap -u "https://target.com/api/user/profile?id=1" \
  --cookie="session=abc123" \
  --level=3 --risk=2

# 获取数据库表列表
sqlmap -u "https://target.com/api/users?id=1" --tables

# 导出指定表数据
sqlmap -u "https://target.com/api/users?id=1" \
  -D target_db -T users --dump
```

### 1.5 A04 - 不安全设计

**测试要点**：
| 设计缺陷 | 测试方法 | 风险 |
|----------|----------|------|
| 缺少速率限制 | 短时间内大量发送请求（如登录、注册、短信） | 暴力破解、资源耗尽 |
| 批量分配漏洞 | 在请求中添加不应被修改的字段（如role=admin） | 权限提升 |
| 缺少业务逻辑校验 | 负数量下单、零金额支付、优惠券无限叠加 | 业务损失 |
| 工作流绕过 | 跳过必需步骤直接执行后续操作 | 业务流程被破坏 |

### 1.6 A05 - 安全配置错误

**配置检查清单**：
```
[ ] 是否仍在使用默认账号密码（admin/admin, root/root）
[ ] 错误页面是否暴露了技术栈信息（异常堆栈、框架版本）
[ ] 不必要的HTTP方法是否启用（PUT/DELETE/TRACE）
[ ] 是否暴露了敏感端点（/actuator, /swagger-ui.html, /phpinfo.php）
[ ] 目录列表功能是否关闭（Directory Browsing）
[ ] 安全相关的HTTP头是否配置：
    - Strict-Transport-Security (HSTS)
    - X-Content-Type-Options: nosniff
    - X-Frame-Options: DENY
    - Content-Security-Policy
    - X-XSS-Protection（注：此头部已在现代浏览器中弃用，建议使用Content-Security-Policy替代）
[ ] CORS配置是否过于宽松
[ ] TLS是否使用了安全版本（TLS 1.2+，禁用SSLv3/TLS 1.0/1.1）
[ ] Cookie是否设置了Secure/HttpOnly/SameSite
[ ] 服务器是否隐藏了版本信息（Server头、X-Powered-By）
```

### 2.2 Burp Suite核心功能

**五大核心模块**：

| 模块 | 功能 | 典型操作 |
|------|------|----------|
| **Proxy** | HTTP/HTTPS拦截代理 | 拦截请求→修改参数→放行→观察响应 |
| **Repeater** | 手动修改和重发请求 | 修改参数逐个测试，观察响应变化 |
| **Intruder** | 自动化参数爆破 | 对参数位置进行字典/Payload遍历 |
| **Scanner** | 自动漏洞扫描（Pro版） | 自动检测SQL注入、XSS等漏洞 |
| **Decoder** | 编码/解码工具 | Base64/URL/HTML编解码 |

**Repeater实战示例**：
```
1. Proxy拦截一个请求后，右键Send to Repeater
2. 在Repeater中修改参数：
   原始：username=user1&role=viewer
   修改：username=user1&role=admin
3. Send，观察响应
4. 如果返回了admin权限的数据，则存在批量分配漏洞
```

**Intruder攻击类型**：
| 模式 | 说明 | 示例 |
|------|------|------|
| Sniper | 一个Payload Set，逐个替换每个位置 | 依次测试每个参数 |
| Battering Ram | 一个Payload Set，同时替换所有位置 | 所有位置使用相同的Payload |
| Pitchfork | 多个Payload Set，一一对应替换 | username字典 + password字典对应 |
| Cluster Bomb | 多个Payload Set，笛卡尔积组合 | 遍历所有username × password组合 |

#### 3.1.3 JWT安全测试

```python

import base64
import json

    payload = {"user_id": 1, "role": "admin"}

        base64.urlsafe_b64encode(json.dumps(header).encode()).rstrip(b'=') +
        b'.' +
        base64.urlsafe_b64encode(json.dumps(payload).encode()).rstrip(b'=') +
        b'.'
    ).decode()

    # 测试3：弱HMAC密钥
    weak_keys = ['secret', 'jwt_secret', 'your-256-bit-secret', 'changeme']
    for key in weak_keys:
```

## 5. 会话管理测试

### 5.1 Session/Cookie安全测试

| 测试项 | 检测方法 | 风险 |
|--------|----------|------|
| Cookie Secure | 检查Set-Cookie是否包含Secure属性 | Cookie通过HTTP传输时被窃取 |
| Cookie HttpOnly | 检查是否包含HttpOnly | JavaScript可读取Cookie，XSS风险 |
| Cookie SameSite | 检查SameSite=Lax/Strict | CSRF攻击风险 |
| Session固定 | 登录前后Session ID是否变化 | 攻击者预置Session ID |
| Session超时 | 长时间不活动后Session是否过期 | 会话被恶意利用 |
| 登出失效 | 登出后Session是否服务端失效 | Session未真正销毁 |

```python
def test_session_management():
    """会话管理完整测试"""
    
    # 测试1：Session固定攻击
    # 步骤：获取一个未登录的Session ID → 让受害者用这个ID登录 → 检查ID是否改变

    pre_login_cookie = session.cookies.get("JSESSIONID")

        json={"username": "test", "password": "Test@123"})

        "登出后Session仍有效！"
    
    # 测试3：并发会话限制
    # 同一账号在不同设备/浏览器登录
    token1 = login("test@example.com", "Test@123")
    token2 = login("test@example.com", "Test@123")
    
    # 根据业务规则：可能是允许（多设备）或踢下线（单设备）
    # 需验证实现的策略是否正确
```

## 6. 实战案例

### 案例1：电商平台安全渗透测试

**背景**：某电商平台上线前进行安全渗透测试。

**测试过程和发现**：

| 阶段 | 发现 | 严重程度 | 修复方案 |
|------|------|----------|----------|
| 信息收集 | 网站暴露了/actuator端点，泄露了配置信息和内部API地址 | 中 | 关闭生产环境actuator暴露 |
| 认证测试 | 找回密码接口存在用户枚举：已注册和未注册邮箱返回不同的信息 | 中 | 统一返回"已发送重置邮件" |
| 授权测试 | 普通用户可通过修改ID访问其他用户的订单详情（水平越权） | 高 | 增加数据所有权校验 |
| 注入测试 | 商品搜索接口存在SQL注入（like模糊查询未参数化） | 严重 | 使用参数化查询 |
| 业务逻辑 | 优惠券可重复使用（并发请求绕过幂等性校验） | 高 | 数据库唯一约束+分布式锁 |
| 配置检查 | S3存储桶设置为公开可读，部分用户上传的身份证照片可被任何人访问 | 严重 | 设置私有Bucket+预签名URL |
| 加密测试 | 数据库中用户密码使用MD5存储，可被彩虹表破解 | 高 | 迁移至bcrypt |

**收获**：此次测试共发现32个安全漏洞，其中2个严重、8个高、15个中、7个低。关键教训是数据权限校验必须在服务端每层都执行。

### 案例2：金融App客户端安全测试

**背景**：某银行App客户端安全评估。

**测试重点**：
1. **本地数据存储**：SharedPreferences/UserDefaults是否存储了敏感信息
2. **网络通信**：是否实现SSL Pinning防止中间人攻击
3. **代码混淆**：反编译后是否清楚暴露了业务逻辑
4. **越狱/Root检测**：检测App在越狱设备上的行为
5. **内存数据**：内存中是否有明文密码、密钥

**关键发现**：
- App的SharedPreferences中明文存储了登录Token
- SSL Pinning未正确实现，使用Charles可抓包解密HTTPS流量
- 反编译后发现了硬编码的AES密钥
- 手势密码本地仅使用简单的异或加密

## 7. 本章小结

1. **安全测试是持续的**：不是上线前的一次性评估，应融入DevSecOps流程
2. **OWASP Top 10是基础**：至少覆盖Top 10的检测，但不局限于此
3. **工具是辅助**：自动化扫描工具能发现约30-50%的漏洞，手动测试不可替代
4. **纵深防御**：不要依赖单一安全控制，网络层→应用层→数据层都需要防护
5. **业务逻辑漏洞**：这是自动化工具最难发现的，需要深入理解业务
6. **安全左移**：从需求阶段就考虑安全，威胁建模应成为标准动作
7. **关注数据**：数据是安全的核心，知道敏感数据在哪里、如何保护
8. **持续学习**：安全领域发展极快，保持对新威胁和防御技术的关注
