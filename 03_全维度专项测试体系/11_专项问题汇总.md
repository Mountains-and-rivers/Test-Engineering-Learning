## 1. 接口测试常见问题

### 问题1：只校验HTTP状态码，不校验业务响应体

**问题描述**：许多测试仅检查`response.status_code == 200`，但不验证返回数据的正确性。当后端代码有Bug但返回了200时（如返回了空列表、错误数据），测试仍然通过。

**典型案例**：
```python
# 不好的做法 - 只检查状态码
def test_get_user():
    resp = requests.get("/api/users/123")
    assert resp.status_code == 200  # 即使返回了错误数据，这个测试也会通过

# 好的做法 - 全面校验
def test_get_user():
    resp = requests.get("/api/users/123")
    assert resp.status_code == 200
    data = resp.json()
    assert data["code"] == 200
    assert data["data"]["id"] == 123
    assert data["data"]["username"] is not None
    assert "@" in data["data"]["email"]
    assert data["data"]["status"] in ["active", "inactive"]
```

### 问题2：请求参数未覆盖NULL和空字符串的区别

**问题描述**：测试用例仅覆盖"有值"和"不传该参数"的情况，但NULL（JSON的null）、空字符串("")、缺少字段在接口层面行为可能完全不同。

**测试矩阵**：
| 传参方式 | 示例 | 可能的服务端行为 |
|----------|------|----------------|
| 正常值 | `"name": "test"` | 正常处理 |
| 空字符串 | `"name": ""` | 可能被拒绝、可能被接受、可能存为空字符串 |
| null | `"name": null` | 可能报错、可能被忽略、可能覆盖原值 |
| 字段不传 | `{}` (没有name字段) | 可能使用默认值、可能报错 |
| 空格 | `"name": "   "` | 是否trim？是否判空？ |

### 问题3：Token/Session过期场景遗漏

**问题描述**：绝大多数接口测试使用"刚登录的Token"，没有覆盖Token即将过期、已经过期、刷新Token等场景。

**应覆盖的场景**：
- Token过期后接口返回401（非200或500）
- 刷新Token接口在Token过期前/后分别调用
- 使用过期Token + 刷新后的Token不会串号
- 并发刷新Token不会产生多个有效Token

### 问题4：并发请求的幂等性问题

**问题描述**：接口文档中声明了幂等性，但并发场景下可能出现：重复创建、库存重复扣减、优惠券多次核销。

**测试方法**：同时发送N个相同的请求（同一幂等性Key），验证最终只有1个生效。

### 问题5：文件上传接口的MIME类型校验不足

**问题描述**：仅检查文件后缀名而非真实MIME类型，导致.jpg后缀的.exe文件上传成功，可能引发安全问题。

**应校验**：文件真实MIME类型（magic bytes检查）、文件大小限制、文件名长度和特殊字符、文件内容扫描（防病毒）。

### 问题6：分页参数的极端值

**问题描述**：未覆盖`page=0`, `page=-1`, `page_size=0`, `page_size=10000`, `page=999999`等极端值。

**应验证的行为**：
- 极端值不导致服务崩溃（500）
- 返回合理的错误提示（400/422）
- `page_size`有合理上限（如100或200），超限自动限制

### 问题7：JSON响应中的数字精度丢失

**问题描述**：JavaScript中Number类型精度有限（53位），当后端返回超过`2^53`的整数时，JSON.parse会导致精度丢失。

**示例**：订单ID `9007199254740993` 在JavaScript中会被解析为 `9007199254740992`。

**验证方法**：检查后端是否将大整数以字符串形式返回（`"id": "9007199254740993"`）。

### 问题8：接口超时配置不合理

**问题描述**：测试环境和生产环境的超时配置不一致，测试环境10秒超时不暴露问题，生产环境3秒超时导致大量失败。

### 问题9：依赖接口不Mock导致测试不稳定

**问题描述**：接口测试直接依赖真实的下游服务（如支付、短信），下游服务故障时测试大量失败。Mock层应在环境异常时保证测试可执行。

### 问题10：多语言/国际化接口测试遗漏

**问题描述**：接口返回的文案没有覆盖多语言场景，只在中文环境测试。不同语言下错误信息的格式和内容可能不同。

### 问题11：(Bonus) WebSocket/SSE长连接测试遗漏

大多数接口测试只覆盖了短连接HTTP，忽略了WebSocket和SSE(Server-Sent Events)的长连接场景：断线重连、心跳保活、消息顺序性。

## 2. 性能测试常见坑

### 坑1：用平均响应时间代替分位数

**问题描述**："平均响应时间200ms，性能OK"——掩盖了5%的请求超过5秒的问题。平均值对长尾不敏感，应使用P95/P99。

### 坑2：压测环境与生产环境差异巨大

**问题描述**：在4核8G测试环境压测得到"TPS=5000"，推断64核128G生产环境"TPS=80000"。实际由于数据库锁竞争、网络开销等非线性因素，生产TPS可能只有20000。

**正确做法**：在生产环境（或等比缩容环境）进行压测，或采用生产环境流量复制。

### 坑3：测试数据量不足，缓存全命中

**问题描述**：压测环境中只有几千条数据，全部被缓存命中，TPS极高。生产环境亿级数据，缓存命中率低，TPS大幅下降。

**正确做法**：压测数据量应与生产接近（至少模拟1-6个月的数据积累），冷热数据比例也应模拟真实情况。

### 坑4：忽略GC停顿的影响

**问题描述**：Java应用在压测前几分钟TPS正常，之后突然下降——Full GC触发。压测时间太短错过了GC周期。

**正确做法**：性能测试至少持续30分钟以上，覆盖多次GC周期。

### 坑5：模拟的并发用户行为不真实

**问题描述**：压测脚本中所有虚拟用户执行完全相同的操作序列（如都搜索"手机"），完全命中了缓存。真实用户搜索的关键词高度分散。

**正确做法**：使用真实用户行为日志回放，或至少使用大量不同的参数值。

### 坑6：连接池/线程池配置被忽略

**问题描述**：压测时发现TPS上不去，CPU和内存使用率都不高——数据库连接池默认20个连接成为瓶颈。

**排查清单**：
- 数据库连接池大小（HikariCP/Druid）
- HTTP客户端连接池（RestTemplate/HttpClient）
- Redis连接池（Lettuce/Jedis）
- 服务器线程池（Tomcat/Netty）
- 消息队列消费者线程数

### 坑7：仅压测单个接口而非混合场景

**问题描述**：单独压测每个接口TPS都达标，但所有接口混合压测时，共享资源（数据库、Redis）成为瓶颈。

**正确做法**：按真实流量比例混合压测（如70%浏览+20%搜索+8%下单+2%支付）。

### 坑8：未设置合理的断言导致"假TPS"

**问题描述**：压测脚本没有设置响应断言（如状态码=200、业务码=200），服务器返回了大量500错误但压测工具认为都是"成功"的，TPS虚高。

### 坑9：忽略了预热的重要性

**问题描述**：冷启动时JIT未编译、缓存为空、连接池未建立——前几分钟的性能数据不具备参考价值。

### 坑10：仅关注服务端，忽略端到端延迟

**问题描述**：服务端P99=50ms，"性能很好"——但客户端实际体验P99=2秒，瓶颈在CDN、DNS、SSL握手、网络延迟等。

### 坑11：(Bonus)性能测试工具自身的瓶颈

JMeter单机只能产生约5000 TPS的HTTP请求。需要更高TPS时，必须使用分布式压测或换用Gatling/wrk/k6等更高性能的工具。

## 3. 安全测试常见误区

### 误区1：只做自动化扫描，不做手动测试

**问题描述**：完全依赖OWASP ZAP/Burp Scanner的自动扫描结果。自动化扫描器能发现约30-50%的漏洞，业务逻辑漏洞（越权、优惠券叠加等）必须手动测试。

### 误区2：只测Web应用，不测API

**问题描述**：安全测试聚焦在Web页面的XSS、CSRF上，但移动App/SPA对应的API接口可能完全没有认证（以为只有App才能调用）。

**真实案例**：App的API接口没有认证，通过抓包获取接口地址后，可直接批量爬取所有用户数据。

### 误区3：仅在测试环境做安全测试

**问题描述**：测试环境有WAF、有限流、有监控，生产环境可能因为配置差异缺少这些防护。安全测试应在预发布/生产环境验证。

### 误区4：只关注外部攻击者，忽略内部威胁

**问题描述**：所有安全测试从"未登录的外部攻击者"角度出发，忽略了"已登录的恶意用户"（水平越权）和"恶意员工"（内部数据泄露）。

### 误区5：认证测试只测"无Token"，不测"伪造Token"

**问题描述**：测试了不带Token返回401，但未测试伪造Token（修改payload但保持签名不变）、过期Token、跨用户Token等场景。

### 误区6：加密测试=检查是否用了HTTPS

**问题描述**：确认了HTTPS就认为"传输加密OK"，忽略了TLS版本（是否禁用1.0/1.1）、加密套件强度、证书有效性（是否自签名、是否过期）、HSTS配置。

### 误区7：只做黑盒渗透，不做白盒代码审查

**问题描述**：纯黑盒测试无法发现硬编码密钥、不安全的随机数生成、后门代码等问题。白盒SAST（静态分析）和人工代码审查是必要补充。

### 误区8：安全测试是一轮式而非持续式

**问题描述**：只在"上线前做一次安全测试"。新功能、新第三方依赖、新配置变更都可能引入新的安全漏洞。

### 误区9：误以为WAF/防火墙能解决所有问题

**问题描述**：WAF可以防御常见的SQL注入和XSS，但无法防御业务逻辑漏洞（如IDOR越权）、零日漏洞、应用层配置错误。

### 误区10：安全测试报告=漏洞列表

**问题描述**：输出一个漏洞列表就结束了，没有对漏洞进行风险评估（CVSS评分）、没有提供修复建议、没有验证修复效果（复测）。

### 误区11：(Bonus) 忽视安全响应计划测试

不仅要测试系统是否安全，还要测试安全事件响应流程是否有效：发现漏洞→报告→评估→修复→验证的闭环是否畅通。

## 4. 兼容测试常见遗漏

### 遗漏1：只测最新版浏览器，不测旧版

**问题描述**：只测最新Chrome和Safari，忽略了仍在使用旧版浏览器的用户（如企业环境中Safari 14、Chrome 90之前的版本）。

**建议**：基于用户数据分析确定需要支持的浏览器最低版本。

### 遗漏2：只测主流设备，忽略中低端设备

**问题描述**：使用iPhone 15 Pro/Samsung S24测试，没考虑中低端设备（2-4GB内存、慢速CPU）上的性能表现。

### 遗漏3：弱网环境测试缺失

**问题描述**：在WiFi/5G环境下测试一切正常，但2G/3G/弱4G下的体验可能极差。应使用Chrome DevTools的网络限速或Charles进行弱网模拟。

### 遗漏4：离线/断网场景遗漏

**问题描述**：App在网络切换（WiFi→4G→无网络→恢复网络）过程中的行为未测试。可能导致数据丢失、状态错乱。

### 遗漏5：横竖屏切换测试遗漏

**问题描述**：只在竖屏测试，横屏下UI布局可能崩溃或关键按钮被遮挡。尤其在视频播放、游戏中横竖屏切换频繁。

### 遗漏6：系统字体大小修改后的适配

**问题描述**：Android/iOS系统设置中将字体调大，App中文字溢出、截断、重叠。老年用户群体尤其受影响。

### 遗漏7：输入法兼容性遗漏

**问题描述**：在中文输入法（搜狗/百度/讯飞）、日文假名输入、韩文输入等不同输入法下，输入框行为可能异常（如输入法候选区遮挡输入框）。

### 遗漏8：辅助功能模式下的表现遗漏

**问题描述**：开启TalkBack/VoiceOver、放大手势、高对比度模式后，App/网站可能完全无法使用。WCAG合规不仅是法律要求，也影响真实用户。

### 遗漏9：打印/PDF导出样式遗漏

**问题描述**：屏幕显示正常的页面，打印或导出PDF时布局混乱、关键内容缺失。`@media print`样式往往被忽略。

### 遗漏10：通知权限和系统弹窗

**问题描述**：App首次启动时的权限请求弹窗（通知、定位、相机、相册）在不同系统版本和不同厂商ROM上的表现差异巨大。拒绝权限后App的行为需要逐一验证。

### 遗漏11：(Bonus) 国际化：只测翻译，不测文本膨胀和RTL

翻译不仅是文本替换，德语等语言文本长度膨胀30-50%，阿拉伯语等RTL语言需要镜像布局。这些往往被忽略。

## 5. 微服务测试常见挑战

### 挑战1：测试环境搭建的"依赖地狱"

**问题描述**：一个服务依赖10+个其他服务，搭建完整环境极其复杂。本地开发只能启动自己的服务，其他全部Mock——但Mock行为可能与真实服务不一致。

**缓解方案**：TestContainers + Docker Compose + 契约测试 + 服务虚拟化工具（如WireMock/Mountebank/Sandbox）。

### 挑战2：分布式调试困难

**问题描述**：一个请求跨越5个服务，其中某个服务返回了错误。在日志中追踪这个请求的完整链路非常困难。

**缓解方案**：强制全链路Trace ID传播、日志聚合(ELK/Loki)、分布式追踪(Jaeger/Zipkin/SkyWalking)。

### 挑战3：异步通信的测试复杂性

**问题描述**：同步调用的接口测试只需"发请求→等响应→验证"。异步消息则需要"发消息→等消费→等处理→验证副作用"，时序控制困难。

### 挑战4：多版本共存时的兼容性

**问题描述**：微服务独立部署，可能出现订单服务v2调用用户服务v1的情况。契约测试和灰度路由策略需要覆盖多版本共存。

### 挑战5：数据一致性的最终验证

**问题描述**：最终一致性意味着"立即验证"可能得到不一致的结果。测试需要在合理的时间窗口内轮询等待，但窗口大小难以确定。

### 挑战6：服务网格（Service Mesh）的测试

**问题描述**：Istio/Linkerd引入的sidecar代理增加了网络跳数和延迟。测试需要覆盖：mTLS是否生效、流量拆分是否准确、故障注入策略是否正确。

### 挑战7：跨服务的限流和熔断

**问题描述**：服务A的限流可能因为服务B的慢响应而触发。测试熔断时需要精确控制下游响应时间和错误率。

### 挑战8：容器化环境中的资源限制

**问题描述**：K8s Pod的资源limits/requests设置不当，导致服务在压测时被OOMKilled。测试需要覆盖资源限制下的行为。

### 挑战9：跨团队协调成本

**问题描述**：测试服务A的新功能需要服务B团队配合修改Mock、服务C团队提供测试数据。跨团队协调成本可能占测试时间的30-50%。

### 挑战10：端到端测试的不可靠性

**问题描述**：微服务的E2E测试失败时，需要逐个排查是哪个服务出了问题。5个服务每个99%可用→E2E测试最多95%可用率。

### 挑战11：(Bonus) 事件版本演进

事件驱动架构中，事件Schema的演进需要向前兼容。旧版消费者遇到新字段、新版消费者收不到旧字段——这些场景很难在测试中覆盖。

## 6. 高可用测试常见盲区

### 盲区1：只测单点故障，不测级联故障

**问题描述**：测试了"数据库主库挂掉"→"自动切换从库"，但没有测试"数据库主库挂掉→应用重试风暴→应用线程池耗尽→上游服务超时→雪崩"。

### 盲区2：恢复后的数据一致性验证不足

**问题描述**：故障恢复后只验证"服务可以访问了"，没有验证"数据完全正确"。特别是故障期间正在处理的事务可能丢失或重复。

### 盲区3：监控告警本身的有效性

**问题描述**：故障注入后系统正确恢复了，但监控系统没有发出告警（告警规则配置错误、告警通道失效）。没有告警的高可用是假的高可用。

### 盲区4：人工干预的依赖

**问题描述**：演练时"自动故障切换"实际上有运维人员在后台手动执行了部分步骤。真正的自动切换能力被高估。

### 盲区5：只演练"干净"的故障

**问题描述**：演练数据库故障时，先优雅地停止了数据库。真实故障是突然崩溃，没有机会刷写缓冲区——数据丢失情况完全不同。

### 盲区6：备份恢复只验证"能恢复"而不验证"恢复后能用"

**问题描述**：从备份恢复了数据库→恢复了所有表→恢复了所有行——但应用程序连接恢复后的数据库可能因为权限、序列号、存储过程版本等问题无法正常工作。

### 盲区7：DNS/负载均衡器自身的故障

**问题描述**：高可用演练聚焦在应用和数据库层面，忽略了DNS故障（域名无法解析）、负载均衡器故障（单点）、API网关故障等基础设施。

### 盲区8：配置中心/注册中心的故障

**问题描述**：Nacos/Consul/Eureka等配置/注册中心挂掉后，服务能否使用本地缓存继续运行？新服务能否注册？

### 盲区9：人为错误的预防和恢复

**问题描述**：高可用设计能应对机器故障，但无法应对运维人员的`rm -rf /`或`DROP TABLE`。需要测试操作审计、危险命令二次确认、变更回滚等机制。

### 盲区10：容量规划中的"协调开销"

**问题描述**：从1个实例扩展到10个实例，理论TPS应增加10倍。但分布式协调（选主、锁、一致性协议）的开销随节点数增加可能呈O(n^2)增长。

### 盲区11：(Bonus) 没有定义"可用"的标准

"系统可用"的定义不清晰：是API返回200就算可用？还是核心业务功能正常才算可用？不同定义导致高可用目标完全不同。

## 7. 自动化测试常见陷阱

### 陷阱1：追求100%自动化覆盖率

**问题描述**："所有用例都要自动化"——这是不切实际且ROI极低的。部分测试（探索性测试、可用性测试、一次性验证）天生不适合自动化。

### 陷阱2：自动化用例越多越好

**问题描述**：盲目追求自动化用例数量而非质量。大量重复覆盖、僵尸用例（测试已删除功能）、Flaky Tests会拖垮CI/CD流水线。

### 陷阱3：UI自动化优先于接口自动化

**问题描述**：因为是"看得见摸得着的"，先从UI开始做自动化。UI自动化维护成本最高、执行最慢、最不稳定——应该从接口层开始建设。

### 陷阱4：测试数据硬编码在脚本中

**问题描述**：用户名、密码、商品ID等硬编码在脚本中。换一个测试环境、数据过期后全部脚本失效。应使用数据工厂或测试数据管理平台。

### 陷阱5：忽略测试环境的稳定性

**问题描述**：大量自动化用例失败是因为"环境问题"——数据库连接超时、第三方服务不可用、磁盘空间满。环境稳定性是自动化稳定性的前提。

### 陷阱6：Flaky Tests不处理，放任自流

**问题描述**：偶尔失败偶尔通过的Flaky Tests不分析不修复。团队的自动化测试信任度逐渐降低，最终"测试失败了先重跑几次再说"。

**处理策略**：
1. 标记Flaky Tests
2. 分类分析根本原因（时序问题？数据冲突？环境问题？）
3. Sprint中分配专项时间修复
4. 无法短期内修复的暂时隔离（不参与CI门禁）
5. 建立Flaky率指标（>2%需要关注）

### 陷阱7：测试代码质量低，无Code Review

**问题描述**：测试代码没有纳入Code Review流程，重复代码泛滥、没有使用设计模式、一次改动需要修改几十个文件。

### 陷阱8：自动化测试没有纳入CI/CD门禁

**问题描述**：自动化测试存在于"某个人的电脑上"，定期手动运行。没有集成到CI/CD中作为质量门禁，自动化测试的价值大打折扣。

### 陷阱9：测试报告只有"通过/失败"，没有可操作信息

**问题描述**：测试失败时输出`AssertionError: Expected 200, got 500`，没有任何上下文：请求参数是什么？响应体是什么？哪个环境？哪个版本？

### 陷阱10：测试数据依赖执行顺序

**问题描述**：测试用例间通过共享数据相互依赖（如Test1创建用户→Test2使用该用户→Test3删除用户）。单独运行Test2必定失败。

### 陷阱11：(Bonus) 没有定期清理和优化测试套件

每季度应审查自动化测试套件：删除无效用例、合并重复覆盖、重构低质量代码、更新过时的定位策略。

---

## 总结

各专项测试中的常见问题和陷阱往往不是技术问题，而是**意识问题**和**流程问题**：

1. **接口测试**：别只满足于状态码校验，深入业务逻辑和边界场景
2. **性能测试**：别用平均值忽悠自己，关注分位数和真实负载模型
3. **安全测试**：自动化工具是辅助，业务逻辑漏洞必须人脑发现
4. **兼容测试**：你的用户在用的设备/浏览器/网络环境远比测试覆盖的复杂
5. **微服务测试**：分布式系统的复杂度是指数级的，不要用单体思维做测试
6. **高可用测试**：演练要"真"，故障要"突然"，恢复后要"验证"
7. **自动化测试**：少即是多，质量优于数量，稳定性是生命线
