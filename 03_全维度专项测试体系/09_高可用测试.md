# 高可用测试 (High Availability Testing)

> **高可用测试**：通过模拟各种故障场景，验证系统在组件失效、网络中断、资源耗尽等异常情况下的容错能力、恢复能力和持续服务能力。高可用测试不是测试"系统会不会出问题"（一定会出问题），而是测试"系统出问题后能否继续服务"。

## 1. 故障注入测试

### 1.1 故障注入方法论

故障注入（Fault Injection）是有意将故障引入系统，以验证系统的容错能力和弹性。

| 故障类型 | 注入方式 | 影响范围 | 工具 |
|----------|----------|----------|------|
| **网络故障** | 延迟、丢包、断网、DNS故障 | 服务间通信 | Chaos Mesh, Toxiproxy, tc (traffic control) |
| **资源故障** | CPU满载、内存耗尽、磁盘满 | 单节点/单Pod | Chaos Mesh(StressChaos), stress-ng |
| **进程故障** | Kill进程、Pod删除、JVM OOM | 单实例 | Chaos Mesh(PodChaos), systemctl stop |
| **时钟故障** | 时间跳跃、时钟偏差 | 节点级 | Chaos Mesh(TimeChaos), libfaketime |
| **IO故障** | 磁盘IO延迟、文件只读 | 存储层 | Chaos Mesh(IOChaos), dm-delay |
| **依赖故障** | 下游服务返回错误/超时 | 服务依赖链 | Toxiproxy, WireMock(延迟/错误响应) |

### 1.2 Chaos Mesh故障注入

Chaos Mesh是CNCF项目，专为Kubernetes环境设计的混沌工程平台。

### 1.3 故障注入测试场景

```python
import requests
import time
import subprocess

class TestFaultInjection:
    """故障注入场景测试"""
    
    BASE_URL = "http://api-gateway"
    
    def test_order_creation_during_payment_service_failure(self):
        """
        故障注入：支付服务不可用
        预期：订单仍可创建（状态为待支付），不应崩溃
        """
        # 1. 注入网络故障（模拟支付服务不可用）
        subprocess.run([
            "kubectl", "apply", "-f", "payment-network-down.yaml"
        ])
        time.sleep(10)  # 等待故障生效
        
        try:
            # 2. 尝试创建订单
            resp = requests.post(
                f"{self.BASE_URL}/api/v1/orders",
                json={
                    "user_id": 123,
                    "items": [{"product_id": 1, "quantity": 1}],
                    "total_amount": 99.00,
                },
                headers={"Authorization": "Bearer test_token"},
                timeout=10,
            )
            
            # 3. 验证订单创建成功（虽然支付不可用）
            assert resp.status_code == 201, \
                f"支付服务不可用时订单应仍可创建，状态码：{resp.status_code}"
            assert resp.json()["data"]["status"] == "pending_payment", \
                "订单状态应为待支付"
            
        finally:
            # 4. 恢复故障
            subprocess.run(["kubectl", "delete", "-f", "payment-network-down.yaml"])
            time.sleep(10)
    
    def test_user_service_pod_kill_resilience(self):
        """
        故障注入：随机杀死50%用户服务Pod
        预期：服务无错误，或错误率<1%
        """
        # 注入PodChaos
        subprocess.run(["kubectl", "apply", "-f", "user-pod-kill.yaml"])
        
        errors = 0
        total = 0
        start = time.time()
        
        # 在故障期间持续发送请求
        while time.time() - start < 60:  # 持续1分钟
            try:
                resp = requests.get(
                    f"{self.BASE_URL}/api/v1/users/100",
                    timeout=5,
                )
                total += 1
                if resp.status_code != 200:
                    errors += 1
            except Exception:
                errors += 1
                total += 1
            time.sleep(0.5)
        
        error_rate = errors / total if total > 0 else 1
        assert error_rate < 0.01, \
            f"Pod被Kill期间错误率应<1%，实际：{error_rate:.2%} (错误{errors}/{total})"
        
        # 清理
        subprocess.run(["kubectl", "delete", "-f", "user-pod-kill.yaml"])
    
    def test_cpu_stress_degradation(self):
        """
        故障注入：订单服务CPU高负载
        预期：服务响应变慢但不应完全不可用，触发告警
        """
        # 注入CPU压力
        subprocess.run(["kubectl", "apply", "-f", "cpu-stress.yaml"])
        time.sleep(30)  # 等待CPU压力生效
        
        latencies = []
        errors = 0
        for _ in range(100):
            start = time.time()
            try:
                resp = requests.get(
                    f"{self.BASE_URL}/api/v1/orders?page=1&size=10",
                    timeout=10,
                )
                latencies.append(time.time() - start)
                if resp.status_code != 200:
                    errors += 1
            except requests.Timeout:
                errors += 1
                latencies.append(10)
            time.sleep(0.2)
        
        p95 = sorted(latencies)[int(len(latencies) * 0.95)]
        error_rate = errors / 100
        
        print(f"CPU压力下P95延迟: {p95:.2f}s")
        print(f"错误率: {error_rate:.1%}")
        
        # CPU压力下P95延迟应该明显高于基线，但不应该完全不可用
        assert error_rate < 0.3, f"CPU压力下错误率过高：{error_rate:.1%}"
        
        subprocess.run(["kubectl", "delete", "-f", "cpu-stress.yaml"])
```

### 1.4 混沌工程原则

1. **从假设开始**：先定义"稳态"是什么（如错误率<0.1%，P95<200ms）
2. **最小爆炸半径**：从最小的故障范围开始（1个Pod、1%流量）
3. **多样化真实事件**：不只是Kill Pod，还要注入延迟、错误、资源耗尽
4. **在生产环境运行**：在非生产环境做的混沌实验价值有限（但这需要强大的信心基础）
5. **自动化持续运行**：将混沌实验集成到CI/CD中，持续验证系统弹性

## 2. 灾难恢复测试

### 2.1 灾难恢复(DR)指标

| 指标 | 全称 | 定义 | 测试目标 |
|------|------|------|----------|
| **RTO** | Recovery Time Objective | 从故障发生到恢复服务的时间 | 越短越好（如<15分钟） |
| **RPO** | Recovery Point Objective | 可接受的数据丢失时间窗口 | 越短越好（如<1分钟、0丢失） |
| **WRT** | Work Recovery Time | 业务数据恢复到可操作状态的时间 | 在RTO内完成 |

### 2.2 DR测试场景矩阵

| 灾难类型 | 恢复策略 | RTO目标 | RPO目标 | 测试频率 |
|----------|----------|---------|---------|----------|
| **单机房断电** | 切换到备机房 | <5分钟 | 0（同步复制） | 每月 |
| **数据库主库损坏** | 从库提升为主库 | <3分钟 | <1分钟 | 每月 |
| **云服务区域故障** | 切换到另一区域 | <30分钟 | <5分钟 | 每季度 |
| **人为误删数据** | 从备份恢复 | <2小时 | 恢复到误删前一刻 | 每季度 |
| **勒索软件攻击** | 从隔离备份恢复 | <4小时 | <1小时 | 每半年 |
| **完全站点瘫痪** | 切换到DR站点 | <1小时 | <15分钟 | 每年 |

### 2.3 DR演练测试流程

```
阶段1：准备（Pre-DR）
  ├── 数据备份状态确认
  ├── DR环境健康检查
  ├── 通知相关团队（避免误告警）
  └── 准备回切方案

阶段2：故障模拟
  ├── 宣布"灾难"发生（开始计时）
  ├── 模拟主站点不可用
  └── 记录故障时间点

阶段3：恢复执行
  ├── 执行切换流程（DNS修改、数据库主从切换）
  ├── 验证DR环境可用
  ├── 恢复数据（从最新备份/快照）
  └── 验证数据完整性

阶段4：业务验证
  ├── 关键业务路径验证
  ├── 数据一致性校验
  ├── 性能验证（DR站点可能配置较低）
  └── 记录RTO和RPO实际值

阶段5：回切与总结
  ├── 故障恢复后切回主站点
  ├── 数据同步回主站点
  ├── 记录改进项
  └── 更新DR Runbook
```

## 3. 主从切换测试

### 3.1 切换类型

| 切换类型 | 触发方式 | 数据影响 | 切换时间 | 适用场景 |
|----------|----------|----------|----------|----------|
| **计划内切换(Switchover)** | 人工触发 | 无数据丢失 | 秒-分钟级 | 维护、升级 |
| **故障切换(Failover)** | 自动检测+触发 | 可能有数据丢失 | 秒-分钟级 | 主库故障 |
| **强制切换** | 人工强制触发 | 有数据丢失风险 | 分钟级 | 主库无法恢复 |

        self._stop_master_db()

        assert failover_detected, \
            f"自动故障切换未在{max_wait}秒内完成"
        
        print(f"自动故障切换时间：{failover_time:.1f}秒")
        assert failover_time < 30, "自动切换时间应<30秒"

## 4. 限流降级测试

### 4.1 限流策略测试

| 限流算法 | 测试方法 | 预期行为 |
|----------|----------|----------|
| **令牌桶** | 以恒定速率发送N+1个请求（N为桶容量） | 第N+1个被限流 |
| **滑动窗口** | 跨窗口边界发送请求 | 精确限流，无窗口边界漏洞 |
| **漏桶** | 突发流量 | 输出速率平滑，超出排队或拒绝 |
| **分布式限流** | 多节点同时发送 | 全局限流准确（误差<5%） |

```python
def test_rate_limiting():
    """限流测试"""
    
    # 测试1：令牌桶 - 正常流量不触发限流
    responses = []
    for _ in range(50):  # QPS限制为100
        resp = requests.get(f"{BASE_URL}/api/v1/data", 
                          headers={"X-API-Key": "test_key"})
        responses.append(resp.status_code)
        time.sleep(0.02)  # 约50 QPS，低于100的限流
    
    assert all(s == 200 for s in responses), "正常流量不应被限流"
    
    # 测试2：超限触发
    responses = []
    for _ in range(150):
        resp = requests.get(f"{BASE_URL}/api/v1/data",
                          headers={"X-API-Key": "test_key"})
        responses.append(resp.status_code)
        # 不等待，迅速发送（>100 QPS）
    
    limited_count = sum(1 for s in responses if s == 429)
    assert limited_count > 0, "超限应触发429"
    print(f"限流响应数：{limited_count}/{len(responses)}")
    
    # 测试3：限流恢复
    time.sleep(1.1)  # 等待1秒窗口过期
    resp = requests.get(f"{BASE_URL}/api/v1/data",
                       headers={"X-API-Key": "test_key"})
    assert resp.status_code == 200, "窗口过期后应恢复正常"
    
    # 测试4：检查Retry-After头
    resp = requests.get(f"{BASE_URL}/api/v1/data",
                       headers={"X-API-Key": "test_key"})
    if resp.status_code == 429:
        assert 'Retry-After' in resp.headers or 'X-RateLimit-Reset' in resp.headers, \
            "限流响应应包含重试时间信息"
```

```python
def test_backup_and_restore():
    """备份恢复验证"""
    
    # 1. 创建测试数据（含时间标记）
    test_records = []
    for i in range(100):
        record = {"table": "test_backup", "data": f"record_{i}_{int(time.time())}"}
        self._insert_record(record)
        test_records.append(record)
        time.sleep(0.1)
    
    # 2. 在备份过程中持续写入（验证备份一致性）
    backup_start = time.time()
    self._start_backup()
    
    # 模拟备份期间有新写入
    records_during_backup = []
    for i in range(20):
        record = {"table": "test_backup", "data": f"during_backup_{i}"}
        self._insert_record(record)
        records_during_backup.append(record)
        time.sleep(0.5)
    
    self._wait_for_backup_completion()
    backup_duration = time.time() - backup_start
    print(f"备份耗时：{backup_duration:.0f}秒")
    
    # 3. 记录备份后持续写入的数据（用于验证RPO）
    records_after_backup = []
    for i in range(50):
        record = {"table": "test_backup", "data": f"after_backup_{i}"}
        self._insert_record(record)
        records_after_backup.append(record)
        time.sleep(0.1)
    
    # 4. 执行恢复
    restore_start = time.time()
    self._restore_from_backup()
    restore_duration = time.time() - restore_start
    print(f"恢复耗时：{restore_duration:.0f}秒")
    
    # 5. 验证恢复后的数据一致性
    # 备份前的数据应该完整恢复
    for record in test_records:
        assert self._record_exists(record), \
            f"备份前写入的记录缺失：{record}"
    
    # 备份期间的数据可能部分恢复（取决于备份方式）
    recovered_during = sum(1 for r in records_during_backup 
                          if self._record_exists(r))
    print(f"备份期间数据恢复率：{recovered_during}/{len(records_during_backup)}")
    
    # 备份后的数据应该丢失（验证RPO）
    recovered_after = sum(1 for r in records_after_backup 
                         if self._record_exists(r))
    assert recovered_after == 0, \
        f"备份后写入的数据应全部丢失，但恢复了{recovered_after}条"
    
    # 6. 验证恢复后的系统功能正常
    assert self._verify_database_functional(), "恢复后数据库功能异常"
```

def test_horizontal_scalability():
    """水平扩展性测试"""
    
    test_configs = [
        {"replicas": 1, "expected_max_tps": None},
        {"replicas": 2, "expected_max_tps": None},
        {"replicas": 4, "expected_max_tps": None},
        {"replicas": 8, "expected_max_tps": None},
    ]
    
    baseline_tps = None
    
    for config in test_configs:
        replicas = config["replicas"]
        
        # 1. 扩展到指定副本数
        self._scale_deployment("order-service", replicas)
        self._wait_for_ready("order-service")
        
        # 2. 执行压测找到该副本数下的最大TPS
        max_tps = self._find_max_tps_with_error_rate("order-service", 
                                                      max_error_rate=0.01)
        
        print(f"副本数={replicas}, 最大TPS={max_tps}")
        
        if replicas == 1:
            baseline_tps = max_tps
        
        # 3. 验证扩展效率
        if baseline_tps and replicas > 1:
            expected_tps = baseline_tps * replicas
            efficiency = max_tps / expected_tps
            
            print(f"  预期TPS={expected_tps}, 扩展效率={efficiency:.1%}")
            
            # 扩展效率应>70%（考虑协调开销）
            assert efficiency > 0.7, \
                f"扩展效率过低：{efficiency:.1%} (副本数={replicas})"

## 8. 本章小结

1. **故障是必然的**：不测试故障，就是等着故障来测试你
2. **Chaos Engineering是验证高可用的最佳实践**：从Chaos Mesh/Toxiproxy开始
3. **DR测试不只是纸面演练**：必须定期在真实环境中执行，RTO/RPO要量化
4. **主从切换要自动化**：手动切换的RTO远大于自动切换，训练团队也要训练系统
5. **限流降级是一套体系**：不是单独的功能，需要配合熔断、重试、降级一起验证
6. **备份不等于能恢复**：定期验证恢复流程，确保备份文件可用
7. **扩展性不是线性的**：从1→2→4→8副本，TPS不会是完美的2x增长
8. **在生产环境做混沌工程是终局**：在此之前，先让预发布环境接近生产配置
