## MySQL Workbench/DBeaver/Navicat 数据库管理

### 2.1 MySQL Workbench

MySQL Workbench 是MySQL官方出品的可视化数据库设计和管理工具。

**主要功能：**
- SQL开发（代码补全、语法高亮、执行历史）
- 数据建模（EER图、正向/逆向工程）
- 服务器管理（用户权限、状态监控、日志查看）
- 数据迁移（支持多种数据库源）
- 性能仪表板

**常用操作：**

```sql
-- 1. 导出数据库结构
-- Server → Data Export → 选择Schema → Export to Self-Contained File

-- 2. 查看表结构
DESCRIBE users;
SHOW CREATE TABLE users;

-- 3. 查看索引
SHOW INDEX FROM users;

-- 4. 查看表大小
SELECT 
    table_name AS '表名',
    ROUND(((data_length + index_length) / 1024 / 1024), 2) AS '大小(MB)'
FROM information_schema.TABLES
WHERE table_schema = 'your_database'
ORDER BY (data_length + index_length) DESC;

-- 5. 查看当前连接和状态
SHOW PROCESSLIST;
SHOW STATUS LIKE '%Connections%';
SHOW STATUS LIKE 'Threads_%';
```

**性能监控面板解读：**
| 指标 | 正常值 | 需要关注 |
|------|--------|---------|
| InnoDB Buffer Pool Hit Rate | >99% | <95% |
| Connections | 稳定 | 持续增长 |
| Slow Queries | 0 | >0持续 |
| Lock Waits | 0 | >0 |
| Table Locks Waited | <10% | >10% |

### 2.2 DBeaver

DBeaver 是开源的多数据库管理工具，支持几乎所有主流数据库。

**安装：**
```bash
# Windows: https://dbeaver.io/download/
# macOS
brew install --cask dbeaver-community

# Linux (Ubuntu/Debian)
sudo add-apt-repository ppa:serge-rider/dbeaver-ce
sudo apt update
sudo apt install dbeaver-ce

# 或通过Flatpak
flatpak install flathub io.dbeaver.DBeaverCommunity
```

**支持的数据库类型：**
```
关系型:
  MySQL, PostgreSQL, Oracle, SQL Server, SQLite, MariaDB,
  DB2, H2, Derby, Firebird, ClickHouse, TiDB, OceanBase

NoSQL:
  MongoDB, Redis, Cassandra, Couchbase, InfluxDB

云数据库:
  AWS RDS/Aurora/Redshift, Google Cloud SQL/BigQuery,
  Azure SQL, Snowflake

其他:
  Elasticsearch, Hive, Presto/Trino, Phoenix
```

**实用功能：**

```
1. ER图 (Entity Relationship Diagram)
   - 右键数据库 → View Diagram
   - 显示表之间的关系

2. 数据导出
   - 右键表/查询结果 → Export Data
   - 支持格式: CSV, JSON, SQL INSERT, HTML, XML, Excel, Markdown

3. 数据对比
   - 右键表 → Compare/Migrate → Compare Data
   - 不同数据库之间的数据对比

4. SQL模板
   - 支持预定义的SQL模板
   - Window → Preferences → Editors → SQL Editor → Templates

5. 代码生成
   - 右键表 → Generate SQL
   - 自动生成 SELECT/INSERT/UPDATE/DELETE语句

6. Mock数据生成
   - 右键表 → Generate Mock Data
   - 按列配置生成规则
```

### 2.3 Navicat

Navicat 是商业数据库管理工具，以界面精美和功能全面著称。

**核心特点：**
- 数据同步（结构同步 + 数据同步）
- 数据传输（跨数据库迁移）
- 数据建模（可视化设计表结构）
- 自动备份（定时备份支持）
- 报表生成（自定义数据报表）
- Navicat Cloud（跨设备同步连接设置）

**Navicat 常用场景 - 测试数据生成：**
```
步骤：
1. 右键目标表 → 数据生成
2. 配置每个字段的生成规则
   - 姓名字段: 从姓名列表随机选择
   - 邮箱: 随机生成 <name>@<domain>.com
   - 手机号: 1[3-9]xxxxxxxxx 正则
   - 时间: random_date(2023-01-01, 2024-12-31)
3. 设置生成数量（如10000条）
4. 预览后点击生成
```

```
[ ] 持久化策略（RDB/AOF配置验证）
[ ] 主从同步延迟（INFO replication检查）
[ ] 集群分片均衡性
```

### 4.3 MongoDB常用查询 (测试用)

```javascript
// 连接
mongo "mongodb://user:pass@host:27017/database"

// 基础查询
db.users.find({ status: "active" })
db.users.findOne({ _id: ObjectId("...") })
db.users.find({ age: { $gt: 18 } }).limit(10)
db.users.countDocuments({ status: "active" })

// 复杂查询
db.orders.aggregate([
    { $match: { status: "completed" } },
    { $group: {
        _id: "$userId",
        totalAmount: { $sum: "$amount" },
        orderCount: { $sum: 1 }

    { $sort: { totalAmount: -1 } },
    { $limit: 10 }

// 索引
db.users.createIndex({ email: 1 }, { unique: true })
db.users.createIndex({ name: "text", description: "text" })
db.users.getIndexes()
db.users.dropIndex("email_1")

// 查询分析
db.users.find({ email: "test@example.com" }).explain("executionStats")
// 关注: executionTimeMillis, totalDocsExamined, totalKeysExamined

// 性能监控
db.serverStatus()
db.stats()
db.currentOp()
db.setProfilingLevel(2)  // 开启慢查询日志，记录所有操作

// 备份恢复 (mongodump/mongorestore)
// 导出
mongodump --uri="mongodb://localhost:27017" --db=testdb --out=/backup/
// 导入
mongorestore --uri="mongodb://localhost:27017" --db=testdb /backup/testdb/
```

## Flyway/Liquibase 数据库版本管理

### 5.1 数据库版本管理的重要性

```
数据库Schema纳入版本控制的好处：
1. 可追溯 - 每次变更都有记录
2. 可复现 - 任意环境可重建相同的Schema
3. 可回滚 - 问题变更可以回退
4. 团队协作 - 多人同时开发不会冲突
5. 自动化 - CI/CD流程中自动迁移
```

### 5.2 Flyway

Flyway 是最流行的数据库迁移工具之一。

**Maven集成：**
```xml
<plugin>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-maven-plugin</artifactId>
    <version>10.4.1</version>
    <configuration>
        <url>jdbc:mysql://localhost:3306/myapp</url>
        <user>root</user>
        <password>password</password>
    </configuration>
</plugin>
```

**迁移脚本命名规范：**
```
src/main/resources/db/migration/
├── V1__create_users_table.sql
├── V2__add_email_column.sql
├── V3__create_orders_table.sql
├── V4__add_foreign_key.sql
├── V5__add_indexes.sql
├── R__repeatable_view.sql        # 可重复执行
├── U4__rollback_foreign_key.sql  # 回滚脚本(需付费版或自定义)
```

**迁移脚本示例：**
```sql
-- V1__create_users_table.sql
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- V2__add_email_column.sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20) AFTER email;
ALTER TABLE users ADD UNIQUE INDEX idx_phone (phone);

-- V3__create_orders_table.sql
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_number VARCHAR(32) NOT NULL UNIQUE,
    total_amount DECIMAL(10,2) NOT NULL,
    status ENUM('PENDING', 'PAID', 'SHIPPED', 'COMPLETED', 'CANCELLED') DEFAULT 'PENDING',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**常用命令：**
```bash
# Maven命令
mvn flyway:migrate         # 执行迁移
mvn flyway:info            # 查看迁移状态
mvn flyway:validate        # 验证迁移脚本一致性
mvn flyway:repair          # 修复checksum
mvn flyway:clean           # 清空数据库(谨慎！)
mvn flyway:undo            # 回滚最近一次迁移(需付费版)

# CI中集成
mvn flyway:migrate -Dflyway.url=jdbc:mysql://test-db:3306/myapp \
    -Dflyway.user=ci_user -Dflyway.password=${DB_PASSWORD}
```

### 5.3 Liquibase

Liquibase 支持SQL/XML/YAML/JSON多种格式定义迁移。

**Maven集成：**
```xml
<plugin>
    <groupId>org.liquibase</groupId>
    <artifactId>liquibase-maven-plugin</artifactId>
    <version>4.25.1</version>
    <configuration>
        <changeLogFile>src/main/resources/db/changelog/db.changelog-master.xml</changeLogFile>
        <url>jdbc:postgresql://localhost:5432/myapp</url>
        <username>postgres</username>
        <password>password</password>
    </configuration>
</plugin>
```

**Changelog示例 (XML格式)：**
```xml
<!-- db.changelog-master.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<databaseChangeLog
    xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
        http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.25.xsd">

    <changeSet id="1" author="zhangsan">
        <createTable tableName="users">
            <column name="id" type="BIGINT" autoIncrement="true">
                <constraints primaryKey="true"/>
            </column>
            <column name="username" type="VARCHAR(50)">
                <constraints unique="true" nullable="false"/>
            </column>
            <column name="email" type="VARCHAR(100)"/>
            <column name="created_at" type="TIMESTAMP" defaultValueComputed="CURRENT_TIMESTAMP"/>
        </createTable>
    </changeSet>

    <changeSet id="2" author="lisi">
        <addColumn tableName="users">
            <column name="phone" type="VARCHAR(20)"/>
        </addColumn>
    </changeSet>

    <changeSet id="3" author="wangwu">
        <createIndex tableName="users" indexName="idx_users_email">
            <column name="email"/>
        </createIndex>
    </changeSet>
</databaseChangeLog>
```

**Liquibase SQL格式：**
```sql
--liquibase formatted sql

--changeset zhangsan:1
CREATE TABLE products (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock INT DEFAULT 0
);

--rollback DROP TABLE products;

--changeset zhangsan:2
ALTER TABLE products ADD COLUMN description TEXT;

--rollback ALTER TABLE products DROP COLUMN description;
```

### 5.4 Flyway vs Liquibase

| 维度 | Flyway | Liquibase |
|------|--------|-----------|
| **脚本格式** | SQL（命名约定） | SQL/XML/YAML/JSON |
| **回滚支持** | 付费版 | 内置(需手动写rollback) |
| **学习曲线** | 低 | 中 |
| **灵活性** | 低 | 高 |
| **Spring Boot集成** | 原生支持 | 需引入依赖 |
| **校验** | Checksum校验 | MD5校验 |
| **适合场景** | SQL为主的项目 | 需要多格式/自动化回滚 |

## SQL性能分析工具

### 6.1 EXPLAIN 执行计划分析

```sql
-- MySQL EXPLAIN
EXPLAIN SELECT u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.status = 'active'
GROUP BY u.id;

-- EXPLAIN关键字段解读：
-- id: SELECT的序号
-- select_type: 查询类型 (SIMPLE/PRIMARY/SUBQUERY/DERIVED)
-- type: 访问类型 (从上到下性能递减)
--   system > const > eq_ref > ref > range > index > ALL
--   ALL表示全表扫描，需要优化！
-- possible_keys: 可能使用的索引
-- key: 实际使用的索引
-- key_len: 使用的索引长度
-- ref: 索引比较的列
-- rows: MySQL估算需要扫描的行数
-- Extra: 额外信息
--   Using index: 覆盖索引（好）
--   Using where: 使用了WHERE过滤
--   Using temporary: 使用了临时表（需要优化）
--   Using filesort: 使用了文件排序（需要优化）

-- 优化后的查询分析
EXPLAIN FORMAT=JSON SELECT ...;
EXPLAIN ANALYZE SELECT ...;  -- MySQL 8.0.18+ 实际执行分析
```

### 6.3 数据库性能监控SQL

```sql
-- MySQL性能监控查询集

-- 1. 查看当前正在执行的所有查询
SELECT 
    id, user, host, db, command, time, state,
    LEFT(info, 200) as query_preview
FROM information_schema.PROCESSLIST
WHERE command != 'Sleep'
ORDER BY time DESC;

-- 2. 找出未使用索引的查询
SELECT 
    DIGEST_TEXT,
    COUNT_STAR as exec_count,
    AVG_TIMER_WAIT / 1000000000 as avg_time_ms,
    SUM_ROWS_EXAMINED as total_rows_examined
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_NO_INDEX_USED > 0 
   OR SUM_NO_GOOD_INDEX_USED > 0
ORDER BY AVG_TIMER_WAIT DESC
LIMIT 20;

-- 3. 找出最耗时的查询(Top 10)
SELECT 
    DIGEST_TEXT,
    COUNT_STAR as exec_count,
    SUM_TIMER_WAIT / 1000000000000 as total_time_sec,
    AVG_TIMER_WAIT / 1000000000 as avg_time_ms,
    MIN_TIMER_WAIT / 1000000000 as min_time_ms,
    MAX_TIMER_WAIT / 1000000000 as max_time_ms
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;

-- 4. 查看锁等待
SELECT 
    r.trx_id waiting_trx_id,
    r.trx_mysql_thread_id waiting_thread,
    r.trx_query waiting_query,
    b.trx_id blocking_trx_id,
    b.trx_mysql_thread_id blocking_thread,
    b.trx_query blocking_query
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b ON b.trx_id = w.blocking_trx_id
JOIN information_schema.innodb_trx r ON r.trx_id = w.requesting_trx_id;

-- 5. 查看表锁情况
SHOW OPEN TABLES WHERE In_use > 0;

-- 6. 查看连接数使用情况
SELECT 
    VARIABLE_VALUE as max_connections
FROM performance_schema.global_status
WHERE VARIABLE_NAME = 'Max_used_connections';
```

```python
import pandas as pd
from sqlalchemy import create_engine

def verify_migration(source_conn, target_conn):
    """比较两个数据库的数据一致性"""

    # 获取所有表
    tables_query = "SELECT table_name FROM information_schema.tables WHERE table_schema = %s"

    source_tables = pd.read_sql(tables_query, source_conn, params=('source_db',))
    target_tables = pd.read_sql(tables_query, target_conn, params=('target_db',))

    # 检查表数量
    assert len(source_tables) == len(target_tables), "表数量不一致"

    for table in source_tables['table_name']:
        # 比较行数
        source_count = pd.read_sql(f"SELECT COUNT(*) as cnt FROM {table}", source_conn)
        target_count = pd.read_sql(f"SELECT COUNT(*) as cnt FROM {table}", target_conn)

        if source_count['cnt'][0] != target_count['cnt'][0]:
            print(f"表 {table} 行数不一致: 源={source_count['cnt'][0]}, 目标={target_count['cnt'][0]}")

        # 对于关键表，比较checksum
        if table in CRITICAL_TABLES:
            source_checksum = pd.read_sql(f"SELECT CHECKSUM_AGG(BINARY_CHECKSUM(*)) as chk FROM {table}", source_conn)
            target_checksum = pd.read_sql(f"SELECT CHECKSUM_AGG(BINARY_CHECKSUM(*)) as chk FROM {table}", target_conn)
            assert source_checksum['chk'][0] == target_checksum['chk'][0], f"表 {table} 数据不一致"
```

## 总结

数据库测试工具选择建议：

| 场景 | 推荐工具 |
|------|---------|
| 日常查询/管理 | DBeaver (免费多库) |
| MySQL建模样| MySQL Workbench (官方免费) |
| 专业数据库管理 | Navicat (商业) |
| Redis管理 | RedisInsight (官方免费) |
| MongoDB管理 | MongoDB Compass (官方免费) |
| Schema版本管理 | Flyway (简单) / Liquibase (灵活) |
| SQL性能分析 | EXPLAIN + pt-query-digest |
| 压测 | sysbench (MySQL) / pgbench (PostgreSQL) |

**核心原则：**
1. Schema变更必须版本化（Flyway或Liquibase）
2. 每个SQL变更上线前必须分析EXPLAIN计划
3. 生产数据库定期执行慢查询分析（至少每周）
4. 测试数据与生产数据严格隔离
5. 数据库备份恢复必须定期演练
