## 性能测试概述

### 性能测试类型

| 类型 | 目的 | 典型场景 |
|------|------|---------|
| **负载测试** | 验证系统在预期负载下的表现 | 日常峰值流量 |
| **压力测试** | 找到系统瓶颈和最大容量 | 大促前摸底 |
| **稳定性测试** | 验证长时间运行的稳定性 | 7x24小时运行 |
| **峰值测试** | 验证系统应对突发流量的能力 | 秒杀/抢购 |
| **容量测试** | 规划系统扩容方案 | 增长预测 |

### 核心指标

- **TPS:** 每秒事务数 (Transactions Per Second) —— 衡量业务操作吞吐量，一个事务可包含多个请求（如"登录"事务 = 请求页面 + 提交表单 + 验证跳转）
- **QPS:** 每秒查询数 (Queries Per Second) —— 衡量单一请求吞吐量，无业务上下文（如DNS查询、数据库查询、HTTP请求）
- **注意: TPS与QPS是两个不同概念。** 在简单场景下（一个事务仅包含一次请求时），TPS ≈ QPS；但在复杂业务中（如一个下单事务包含认证、库存、支付多次请求），TPS < QPS。不可混用。
- **响应时间:** 平均响应时间、P50/P90/P95/P99
- **并发用户数:** 同时在线或活跃用户数
- **吞吐量:** 网络吞吐量(Byte/s)
- **错误率:** 失败请求占比
- **资源使用率:** CPU、内存、磁盘I/O、网络带宽

## JMeter 详细使用教程

### 2.1 安装与启动

```bash
# 下载解压
wget https://dlcdn.apache.org/jmeter/binaries/apache-jmeter-5.6.3.tgz
tar -xzf apache-jmeter-5.6.3.tgz

# 启动GUI模式（Windows: jmeter.bat, Linux: jmeter.sh）
cd apache-jmeter-5.6.3/bin
./jmeter.sh

# 启动非GUI模式（压测推荐）
./jmeter.sh -n -t test_plan.jmx -l results.jtl -e -o report/

# 命令行参数说明
# -n: 非GUI模式
# -t: 测试计划文件
# -l: 结果文件输出
# -e: 生成HTML报告
# -o: 报告输出目录
# -J: 设置JMeter属性 -Jthreads=100
# -G: 设置全局属性（分布式）
```


**test_users.csv:**
```csv
username,password,expected_role
```

#### 关联(Correlation) - 提取器和变量传递

```
场景：登录 → 提取Token → 后续请求使用Token

1. 登录请求 → 添加后置处理器 → JSON Extractor
   Names of created variables: auth_token
   JSON Path expressions: $.data.token
   Default Values: NOT_FOUND

2. 后续请求 → HTTP Header Manager → 添加
   Name: Authorization
   Value: Bearer ${auth_token}
```

**常用 JSON Path 表达式：**
```
$.data.token                    # 获取token
$.data.userId                   # 获取用户ID
$.data.items[0].id              # 获取第一个元素的id
$.data.items[*].name            # 获取所有name
$.data.items[?(@.price > 100)]  # 过滤价格大于100的
```

### 2.4 JMeter 分布式测试

当单机无法产生足够负载时，需要分布式压测。

**架构：**
```
┌─────────────────┐
│  Master (控制机)  │  ← 运行jmeter，不产生负载
│  发送指令、收集结果 │
└────────┬────────┘
         │ (RMI)
    ┌────┴────┬─────────┐
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│ Slave1 │ │ Slave2 │ │ Slave3│  ← 执行压测、产生负载
│ 100并发 │ │ 100并发 │ │ 100并发 │
└───────┘ └───────┘ └───────┘
```

**配置步骤：**

1. Master配置 (`jmeter.properties`)：
```properties
remote_hosts=192.168.1.101:1099,192.168.1.102:1099
server.rmi.ssl.disable=true
mode=Standard
```

2. Slave启动：
```bash
# 在每个Slave机器上启动
./jmeter-server.sh
# 或指定网卡
./jmeter-server.sh -Djava.rmi.server.hostname=192.168.1.101
```

3. 执行分布式压测：
```bash
# 从Master发起
./jmeter.sh -n -t test_plan.jmx -r -l results.jtl -e -o report/

# -r: 启动所有远程Slave
# -R host1,host2: 指定特定Slave
```


## Gatling Scala DSL 性能测试

### 3.1 概述

Gatling 是基于 Scala/Akka 的高性能压测工具，使用 DSL 编写测试脚本。

**优势：**
- 异步非阻塞IO（极其高效的资源利用）
- 代码即配置（DSL可读性好）
- 内置HTML报告
- Maven/Gradle集成

**Maven依赖：**
```xml
<dependency>
    <groupId>io.gatling.highcharts</groupId>
    <artifactId>gatling-charts-highcharts</artifactId>
    <version>3.11.1</version>
    <scope>test</scope>
</dependency>
```

### 3.2 测试脚本示例

```scala
// src/test/scala/LoginSimulation.scala
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class LoginSimulation extends Simulation {

  // HTTP Protocol 配置
  val httpProtocol = http
    .baseUrl("https://api.example.com")
    .acceptHeader("application/json")
    .contentTypeHeader("application/json")
    .userAgentHeader("Gatling/3.0")

  // 数据喂入
  val feeder = csv("users.csv").random

  // 场景定义
  val scn = scenario("用户登录场景")
    .feed(feeder)
    .exec(
      http("登录请求")
        .post("/api/v1/login")
        .body(StringBody("""{"username":"${username}","password":"${password}"}"""))
        .check(status.is(200))
        .check(jsonPath("$.data.token").saveAs("authToken"))
    )
    .pause(2, 5)  // 随机等待2-5秒
    .exec(
      http("获取用户信息")
        .get("/api/v1/user/profile")
        .header("Authorization", "Bearer ${authToken}")
        .check(status.is(200))
        .check(jsonPath("$.data.name").exists)
    )

  // 注入策略
  setUp(
    scn.inject(
      rampUsers(100).during(10.seconds),     // 10秒内递增到100用户
      constantUsersPerSec(20).during(60.seconds), // 稳定20用户/秒，持续60秒
      stressPeakUsers(500).during(20.seconds) // 20秒内达到峰值500
    )
  ).protocols(httpProtocol)
    .assertions(
      global.responseTime.mean.lt(2000),     // 平均响应时间<2s
      global.responseTime.percentile(95).lt(3000), // P95<3s
      global.successfulRequests.percent.gt(95),    // 成功率>95%
      global.requestsPerSec.gt(100)          // RPS>100
    )
}
```

### 3.3 运行与报告

```bash

mvn gatling:test -Dgatling.simulationClass=LoginSimulation

# Gradle 运行

# 报告位置: target/gatling/loginsimulation-xxxxx/index.html
```

## Locust Python 性能测试框架

### 4.1 概述

Locust 是用 Python 编写的性能测试框架，通过编写 Python 代码来定义用户行为。

**安装：**
```bash
pip install locust
```

### 4.2 基础测试脚本

```python
# locustfile.py
from locust import HttpUser, task, between, events
import random

class WebsiteUser(HttpUser):
    wait_time = between(1, 3)  # 请求间隔1~3秒
    host = "https://api.example.com"

    def on_start(self):
        """每个虚拟用户启动时执行一次（登录）"""
        response = self.client.post("/api/v1/login", json={
            "username": f"testuser_{random.randint(1, 1000)}",
            "password": "Test@123"
        })
        if response.status_code == 200:
            self.token = response.json()["data"]["token"]
        else:
            self.token = None

    @task(3)  # 权重3（执行频率高）
    def browse_products(self):
        """浏览商品列表"""
        page = random.randint(1, 10)
        self.client.get(
            f"/api/v1/products?page={page}&size=20",
            headers={"Authorization": f"Bearer {self.token}"},
            name="/api/v1/products"
        )

    @task(2)  # 权重2
    def view_product_detail(self):
        """查看商品详情"""
        product_id = random.randint(1, 500)
        self.client.get(
            f"/api/v1/products/{product_id}",
            headers={"Authorization": f"Bearer {self.token}"},
            name="/api/v1/products/{id}"
        )

    @task(1)  # 权重1（执行频率低）
    def place_order(self):
        """下单"""
        self.client.post(
            "/api/v1/orders",
            json={
                "product_id": random.randint(1, 500),
                "quantity": 1,
                "address_id": 1001
            },
            headers={"Authorization": f"Bearer {self.token}"},
            name="/api/v1/orders"
        )

# 自定义事件监听
@events.test_start.add_listener
def on_test_start(environment, **kwargs):
    print("=== 性能测试开始 ===")

@events.request.add_listener
def on_request(request_type, name, response_time, response_length,
               exception, context, **kwargs):
    if exception:
        print(f"请求失败: {name}, 异常: {exception}")
```

### 4.4 高级特性

```python
# 自定义负载形状
from locust import LoadTestShape

class StagesShape(LoadTestShape):
    """
    自定义压测阶段：
    - 阶段1: 5分钟逐步增加到500用户
    - 阶段2: 10分钟保持500用户
    - 阶段3: 2分钟减少到0
    """
    stages = [
        {"duration": 300, "users": 500, "spawn_rate": 10},
        {"duration": 600, "users": 500, "spawn_rate": 10},
        {"duration": 120, "users": 0, "spawn_rate": 10},
    ]

    def tick(self):
        run_time = self.get_run_time()
        cumulative_time = 0
        for stage in self.stages:
            cumulative_time += stage["duration"]
            if run_time < cumulative_time:
                return (stage["users"], stage["spawn_rate"])
        return None
```

### 5.2 测试脚本

```javascript
// test_script.js
import http from 'k6/http';
import { sleep, check, group } from 'k6';
import { Rate, Trend, Counter } from 'k6/metrics';

// 自定义指标
const errorRate = new Rate('errors');
const checkoutTime = new Trend('checkout_time');
const orderCount = new Counter('orders_placed');

// 测试配置
export const options = {
    stages: [
        { duration: '2m', target: 100 },   // 2分钟爬升到100用户
        { duration: '5m', target: 100 },   // 保持100用户5分钟
        { duration: '2m', target: 200 },   // 2分钟爬升到200用户
        { duration: '5m', target: 200 },   // 保持200用户5分钟
        { duration: '1m', target: 0 },     // 1分钟降至0
    ],
    thresholds: {
        http_req_duration: ['p(95)<2000', 'p(99)<5000'],  // P95 < 2s, P99 < 5s
        errors: ['rate<0.05'],                // 错误率 < 5%
        http_req_failed: ['rate<0.01'],       // HTTP错误率 < 1%
    },
};

// 初始化（每个VU执行一次）
export function setup() {
    const loginRes = http.post('https://api.example.com/api/v1/login', JSON.stringify({
        username: 'test_admin',
        password: 'Test@123456',
    }), { headers: { 'Content-Type': 'application/json' } });

    const token = loginRes.json('data.token');
    return { token };
}

// 主测试函数
export default function (data) {
    const params = {
        headers: {
            'Authorization': `Bearer ${data.token}`,
            'Content-Type': 'application/json',
        },
    };

    group('浏览商品', () => {
        const res = http.get(
            `https://api.example.com/api/v1/products?page=${Math.floor(Math.random() * 10)}`,
            params
        );
        check(res, {
            '状态码200': (r) => r.status === 200,
            '响应时间<500ms': (r) => r.timings.duration < 500,
        });
    });

    sleep(Math.random() * 3 + 1);  // 1-4秒随机等待

    group('下单', () => {
        const orderRes = http.post(
            'https://api.example.com/api/v1/orders',
            JSON.stringify({
                product_id: Math.floor(Math.random() * 500) + 1,
                quantity: 1,
            }),
            params
        );

        const success = orderRes.status === 200 || orderRes.status === 201;
        errorRate.add(!success);

        if (success) {
            orderCount.add(1);
        }

        checkoutTime.add(orderRes.timings.duration);
    });

    sleep(2);
}
```

## 5个性能测试最佳实践

### 实践1: 逐步加压策略

```
正确做法：阶梯式加压
阶段1: 0 - 100用户 (5分钟)
阶段2: 100用户稳态 (10分钟)
阶段3: 100 - 200用户 (5分钟)
阶段4: 200用户稳态 (10分钟)
阶段5: 200 - 400用户 (5分钟)
阶段6: 400用户稳态 (10分钟) → 找到拐点
阶段7: 0用户 (1分钟) → 观测恢复

错误做法：瞬间1000并发 → 系统直接崩溃，无法分析瓶颈
```

### 实践2: 监控指标关联分析

在压测过程中，必须同时监控应用服务器：

```
关键监控层面：
├── 应用层: TPS, 响应时间, 错误率, 线程池
├── JVM层: GC频率, 堆内存, Young/Old区
├── 数据库: 连接数, 慢查询, 死锁, QPS
├── 操作系统: CPU, 内存, 磁盘IO, 网络带宽
└── 中间件: Redis命中率, MQ积压, Nginx连接数
```

**常见瓶颈模式：**
| 现象 | 可能原因 |
|------|---------|
| TPS不再增长，CPU接近100% | 代码计算密集或死循环 |
| TPS不再增长，CPU低，内存高 | 内存泄漏或GC频繁 |
| TPS不再增长，CPU和内存正常 | 数据库/外部服务瓶颈或线程池满 |
| 响应时间逐步增加 | 连接池泄漏、缓存失效 |
| 间歇性错误 | 连接超时、线程竞争 |

### 实践5: 结果分析与报告

**JMeter HTML报告关键图表解读：**

```
1. APDEX (应用性能指数)
   - APDEX = (满意数 + 容忍数/2) / 总数
   - > 0.9: 优秀; 0.7-0.9: 可接受; < 0.7: 需改进

2. Response Time Percentiles
   - P50 < P90 < P95 < P99
   - 如果 P99 >> P95，说明有"长尾"问题
   - P99 过大通常是GC停顿或外部服务超时

3. Transactions Per Second
   - 观察TPS曲线是否平滑
   - 是否有周期性波动（定时任务干扰）
   - 拐点位置（系统最大吞吐量）

4. Response Codes Per Second
   - 关注5xx错误的出现时机
   - 错误率是否随压力增大而增加
```

**报告模板关键信息：**
```
【测试结论】
- 最大TPS: 1250/s (400并发时)
- 平均响应时间: 345ms (P95: 890ms, P99: 2100ms)
- 系统瓶颈: 数据库连接池（当前100连接，满载）
- 建议: 增加连接池至200，优化慢查询(user_search接口)

【风险项】
- P99响应时间2.1s，超过SLA要求的1.5s
- 500并发时错误率3.2%，超过可接受的1%
```

## 总结

性能测试工具的选择核心原则：

1. **按团队技能选** - Python团队用Locust，JS开发者用k6，传统企业用JMeter
2. **按并发规模选** - 单机万级用k6/Gatling，分布式用JMeter/Locust
3. **按协议需求选** - HTTP-only可用任一款，多协议必选JMeter
4. **按集成需求选** - CI/CD集成优先k6，监控平台集成优先JMeter(插件丰富)

性能测试最重要的不是工具本身，而是**合理的测试方案、充分的监控覆盖、以及科学的分析能力**。工具只是手段，解决问题才是目的。
