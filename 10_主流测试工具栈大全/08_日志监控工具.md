## 可观测性概述

### 三大支柱

```
可观测性 (Observability)

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   Logging    │  │   Metrics    │  │   Tracing    │
│   (日志)     │  │   (指标)     │  │   (追踪)     │
├──────────────┤  ├──────────────┤  ├──────────────┤
│ ELK Stack    │  │ Prometheus   │  │ SkyWalking   │
│ Graylog      │  │ Grafana      │  │ Jaeger       │
│ Loki         │  │ InfluxDB     │  │ Zipkin       │
│ Splunk       │  │ DataDog      │  │ Pinpoint     │
└──────────────┘  └──────────────┘  └──────────────┘
```

**三者关系：**
- **Logging:** 记录离散的事件（谁，做了什么，什么时间）
- **Metrics:** 聚合的数值数据（多少，多快，多频繁）
- **Tracing:** 请求在分布式系统中的完整路径（经过了哪些服务，每个环节耗时多少）

> **注意：** Splunk 和 DataDog 并非仅属于 Logging 或 Metrics 单一类别——它们是**全栈可观测性平台**，同时覆盖日志（Logging）、指标（Metrics）和链路追踪（Tracing）三大支柱。在上表中按其最初的核心优势归类，实际上它们已发展为完整可观测性解决方案。

### 2.3 Logstash 管道配置

```
# logstash/pipeline/logstash.conf
input {
  beats {
    port => 5044
  }

  # 也可以直接从文件读取
  file {
    path => "/var/log/app/*.log"
    start_position => "beginning"
    sincedb_path => "/dev/null"
  }
}

filter {
  # 1. 解析JSON格式日志
  if [log][file][path] =~ "application.json" {
    json {
      source => "message"
    }
  }

  # 2. 自定义Grok模式解析Java日志
  if "JAVA" in [tags] {
    grok {
      match => {
        "message" => "%{TIMESTAMP_ISO8601:timestamp}\s+%{LOGLEVEL:level}\s+\[%{DATA:thread}\]\s+%{DATA:logger}\s+-\s+%{GREEDYDATA:msg}"
      }
    }

    # 解析异常堆栈
    if [msg] =~ "Exception" {
      multiline {
        pattern => "^\s+at\s"
        what => "previous"
      }
    }
  }

  # 3. 提取Nginx访问日志
  if "NGINX" in [tags] {
    grok {
      match => {
        "message" => '%{IPORHOST:client_ip} - %{DATA:remote_user} \[%{HTTPDATE:timestamp}\] "%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}" %{NUMBER:status} %{NUMBER:body_bytes_sent} "%{DATA:referrer}" "%{DATA:user_agent}" %{NUMBER:request_time}'
      }
    }
  }

  # 4. 添加地理位置信息
  geoip {
    source => "client_ip"
    target => "geoip"
  }

  # 5. 添加时间戳处理
  date {
    match => ["timestamp", "ISO8601"]
    target => "@timestamp"
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
    # 按天创建索引
  }

  # 可选：同时输出到控制台（调试用）
  stdout {
    codec => rubydebug
  }
}
```

### 2.4 Filebeat 配置

```yaml
# filebeat/filebeat.yml
filebeat.inputs:
  # 收集应用日志
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    fields:
      app: my-application
      environment: production
    fields_under_root: true
    # 多行日志合并 (Java异常堆栈)
    multiline.pattern: '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
    multiline.negate: true
    multiline.match: after

  # 收集Nginx日志
  - type: log
    enabled: true
    paths:
      - /var/log/nginx/access.log
    fields:
      app: nginx
    tags: ["NGINX"]

  # 通过Docker socket收集容器日志
  - type: container
    enabled: true
    paths:
      - /var/lib/docker/containers/*/*.log

# 输出到Logstash
output.logstash:
  hosts: ["logstash:5044"]
```

```
Prometheus 数据模型:
<metric_name>{<label_name>="<label_value>", ...} <value> [<timestamp>]

http_requests_total{method="GET", endpoint="/api/users", status="200"} 12345
```

### 3.4 告警规则

```yaml
# prometheus/alerts/application.yml
groups:
  - name: application_alerts
    rules:
      # 1. 应用宕机
      - alert: ApplicationDown
        expr: up{job="spring-boot-app"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "应用 {{ $labels.instance }} 已宕机"
          description: "应用已停止响应超过1分钟"

      # 2. 高错误率
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) > 0.1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "高错误率: {{ $labels.instance }}"
          description: "5xx错误率超过10%，当前值: {{ $value }}"

      # 3. 高响应时间
      - alert: HighResponseTime
        expr: histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m])) > 2
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "P95响应时间超过2秒"
          description: "P95响应时间: {{ $value }}s"

      # 4. 高CPU使用率
      - alert: HighCPUUsage
        expr: rate(process_cpu_usage[5m]) * 100 > 80
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "CPU使用率超过80%"
          description: "当前CPU使用率: {{ $value }}%"

      # 5. 高内存使用率
      - alert: HighMemoryUsage
        expr: (sum(jvm_memory_used_bytes) / sum(jvm_memory_max_bytes)) * 100 > 90
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "JVM内存使用率超过90%"

      # 6. 线程池队列积压
      - alert: ThreadPoolBacklog
        expr: tomcat_threads_busy_threads / tomcat_threads_config_max_threads > 0.9
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Tomcat线程池使用率超过90%"

      # 7. 磁盘空间不足
      - alert: DiskSpaceLow
        expr: node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} < 0.1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "磁盘空间不足10%"

      # 8. 数据库连接池耗尽
      - alert: DatabaseConnectionPoolExhausted
        expr: hikaricp_connections_active / hikaricp_connections_max > 0.9
        for: 3m
        labels:
          severity: critical
        annotations:
          summary: "数据库连接池使用率超过90%"
```

### 4.3 链路追踪核心价值

```
性能问题排查示例：
一个用户查询订单详情的慢请求 → 链路追踪展示：

[用户] → [API Gateway 50ms] → [订单服务 2000ms]
                                      ├── [查询订单表 50ms]
                                      ├── [查询用户服务 100ms]
                                      ├── [查询商品服务 1500ms] ← 瓶颈！
                                      │   └── [Redis缓存未命中]
                                      │   └── [DB查询 特慢SQL]
                                      └── [查询物流服务 350ms]

快速定位：商品服务的Redis缓存失效，导致DB慢查询
修复措施：预热Redis缓存，优化SQL索引
```

## Sentry 错误监控

### 5.1 概述

Sentry 是应用错误监控平台，实时捕获和追踪生产环境的异常。

**部署：**
```yaml
# docker-compose-sentry.yml (基于官方onpremise仓库)
# git clone https://github.com/getsentry/self-hosted
# cd self-hosted
# ./install.sh
```

### 5.2 集成方式

**Java (Spring Boot)：**
```xml
<dependency>
    <groupId>io.sentry</groupId>
    <artifactId>sentry-spring-boot-starter</artifactId>
    <version>7.3.0</version>
</dependency>
```

```yaml
# application.yml
sentry:
  dsn: https://xxx@sentry.example.com/project-id
  environment: production
  release: ${app.version}
  traces-sample-rate: 0.1  # 10%的请求采集性能数据
  enable-tracing: true
  logging:
    minimum-event-level: WARN
    minimum-breadcrumb-level: INFO
```

**Python：**
```python
import sentry_sdk
from sentry_sdk.integrations.flask import FlaskIntegration

sentry_sdk.init(
    dsn="https://xxx@sentry.example.com/project-id",
    environment="production",
    traces_sample_rate=0.1,
    integrations=[FlaskIntegration()],
)

# 手动捕获异常
try:
    risky_operation()
except Exception as e:
    sentry_sdk.capture_exception(e)

# 手动添加标签
with sentry_sdk.configure_scope() as scope:
    scope.set_tag("user_id", "12345")
    scope.set_extra("order_id", "ORD-2024-001")
```

**JavaScript (前端)：**
```javascript
import * as Sentry from "@sentry/browser";

Sentry.init({
    dsn: "https://xxx@sentry.example.com/project-id",
    environment: "production",
    tracesSampleRate: 0.05,
    integrations: [
        new Sentry.BrowserTracing(),
    ],
    beforeSend(event) {
        // 过滤敏感信息
        if (event.request) {
            delete event.request.cookies;
            delete event.request.headers;
        }
        return event;
    },
});
```

### 5.3 Sentry 核心功能

```
1. Issue Grouping (问题分组)
   - 相同堆栈的异常自动聚合
   - 按环境/版本/用户维度过滤

2. Breadcrumbs (面包屑)
   - 错误发生前的用户操作序列
```

## 总结

监控工具选择推荐：

| 场景 | 推荐方案 |
|------|---------|
| 日志分析 | ELK Stack (功能全面) / Loki (轻量低成本) |
| 指标监控 | Prometheus + Grafana (事实标准) |
| 链路追踪 | SkyWalking (Java生态) / Jaeger (多语言) |
| 错误监控 | Sentry (应用错误) |
| 系统监控 | Zabbix (传统IT) / Prometheus + Node Exporter (云原生) |
| 告警通知 | AlertManager + 钉钉/飞书/企业微信Webhook |
| 容器监控 | Prometheus + cAdvisor |
| 完整方案 | ELK + Prometheus + Grafana + SkyWalking + Sentry |

**实施建议：**
1. 从最重要的指标开始，逐步完善
2. 告警要有明确的优先级和处理指南(runbook)
3. 定期review告警规则，消除噪音告警
4. 将监控数据集成到测试报告中(压测时必配)
5. 做好权限控制和审计日志(特别是ELK中的敏感数据脱敏)
