## 概述与分类

接口测试是软件测试的核心领域之一。随着微服务架构和前后端分离的普及，接口测试的重要性与日俱增。主流的接口测试工具可以分为以下几类：

| 分类 | 代表工具 | 适用场景 |
|------|---------|---------|
| GUI桌面工具 | Postman, Insomnia, SoapUI | 手动调试、接口探索、快速验证 |
| 命令行工具 | curl, httpie | 脚本自动化、CI/CD集成、轻量调用 |
| API文档+测试 | Swagger/OpenAPI, Apifox | 文档驱动的开发测试一体化 |
| 国产集成平台 | Apifox, Apizza, Eolink | 国内团队协作、项目管理集成 |
| 企业级测试 | SoapUI Pro, ReadyAPI, Postman Enterprise | 复杂场景、合规要求、团队协作 |

## Postman 详细使用教程

### 2.1 安装与环境配置

Postman 是当前最流行的 API 开发与测试工具，支持 Windows、Mac、Linux 三大平台。

**下载地址：** https://www.postman.com/downloads/

**Linux 安装示例 (Ubuntu)：**
```bash
# 通过 Snap 安装
sudo snap install postman

# 或通过手动下载 tar.gz
wget https://dl.pstmn.io/download/latest/linux64
tar -xzf linux64 -C /opt/
sudo ln -s /opt/Postman/Postman /usr/bin/postman
```

**Windows 安装：**
- 下载 `.exe` 安装包，双击安装即可
- 推荐使用 winget: `winget install Postman.Postman`

### 2.2 核心概念

#### Workspace (工作区)
工作区是 Postman 中的顶级组织单元，用于隔离不同项目的 API 请求、集合和环境变量。

- **Personal Workspace:** 个人可见
- **Team Workspace:** 团队成员共享
- **Public Workspace:** 对外公开

#### Collection (集合)
Collection 是一组 API 请求的逻辑分组，支持：
- 文件夹结构组织请求
- Collection 级别的 Pre-request Scripts 和 Tests
- Collection Variables（变量）
- Collection Runner（批量执行）

#### Environment (环境变量)
环境变量允许在不同环境（开发、测试、生产）之间切换，无需修改请求本身。

```json
// 开发环境变量示例 (dev environment)
{
  "name": "Development",
  "values": [
    {
      "key": "base_url",
      "value": "http://localhost:8080",
      "enabled": true
    },
    {
      "key": "api_key",
      "value": "dev-api-key-xxxxx",
      "enabled": true
    },
    {
      "key": "db_host",
      "value": "localhost",
      "enabled": true
    }
  ]
}
```

在请求中使用变量：`{{base_url}}/api/v1/users`

### 2.3 Pre-request Script (前置脚本)

Pre-request Script 在请求发送前执行，用于设置动态参数：

```javascript
// 1. 生成时间戳
pm.environment.set("timestamp", Date.now());

// 2. 计算签名（以HMAC-SHA256为例）
const crypto = require('crypto-js');
const secret = pm.environment.get("api_secret");
const body = pm.request.body ? pm.request.body.raw : '';
const sign = crypto.HmacSHA256(body, secret).toString();
pm.environment.set("signature", sign);

// 3. 从响应中提取token并设置到环境变量
const moment = require('moment');
pm.environment.set("request_time", moment().format("YYYY-MM-DD HH:mm:ss"));

// 4. 生成随机数
pm.variables.set("random_email", `test_${Date.now()}@example.com`);
```

### 2.4 Tests / Post-response Script (断言测试)

Tests 脚本在请求完成后执行，用于验证响应：

```javascript
// 1. 基础状态码检查
pm.test("状态码为200", function () {
    pm.response.to.have.status(200);
});

// 2. 响应时间检查
pm.test("响应时间小于500ms", function () {
    pm.expect(pm.response.responseTime).to.be.below(500);
});

// 3. JSON Schema 验证
const schema = {
    "type": "object",
    "properties": {
        "code": { "type": "number" },
        "data": {
            "type": "object",
            "properties": {
                "id": { "type": "number" },
                "name": { "type": "string" }
            },
            "required": ["id", "name"]
        }
    },
    "required": ["code", "data"]
};
pm.test("响应体结构符合预期Schema", function () {
    pm.response.to.have.jsonSchema(schema);
});

// 4. 链式请求 - 提取token用于后续请求
const jsonData = pm.response.json();
if (jsonData.token) {
    pm.environment.set("auth_token", jsonData.token);
}

// 5. 数组长度检查
pm.test("返回数据不少于1条", function () {
    const jsonData = pm.response.json();
    pm.expect(jsonData.data.length).to.be.at.least(1);
});

// 6. 复杂业务校验
pm.test("用户余额校验通过", function () {
    const jsonData = pm.response.json();
    pm.expect(jsonData.balance).to.be.an('number').that.is.greaterThan(0);
});
```

### 2.5 Collection Runner (批量执行)

Collection Runner 用于批量执行整个 Collection 的请求：

**运行方式：**
1. 点击 Collection 右侧的 "Run" 按钮
2. 选择环境、设置迭代次数(iteration)、延迟(delay)
3. 选择数据文件(JSON/CSV)进行数据驱动测试

**数据驱动测试示例 (data.csv):**
```csv
username,password,expected_status
admin,admin123,200
testuser,test123,200
invalid,wrong,401
```

在请求中使用：`{{username}}`, `{{password}}`

### 2.6 Newman 命令行自动化

Newman 是 Postman 的命令行运行器，用于 CI/CD 集成。

**安装：**
```bash
npm install -g newman
npm install -g newman-reporter-htmlextra  # HTML报告插件
```

**基本使用：**
```bash
# 运行单个 Collection
newman run MyCollection.json -e dev_environment.json

# 使用数据文件，5次迭代
newman run MyCollection.json \
  -e dev_environment.json \
  -d testdata.csv \
  -n 5 \
  --delay-request 100 \
  --reporters cli,htmlextra \
  --reporter-htmlextra-export ./reports/report.html

# 设置全局变量
newman run MyCollection.json \
  --env-var "base_url=https://api.example.com" \
  --env-var "api_key=xxxxx"

# 指定文件夹运行
newman run MyCollection.json --folder "用户模块"

# 失败时重试
newman run MyCollection.json --bail --timeout-request 10000
```

**Jenkins 集成 Pipeline 示例：**
```groovy
pipeline {
    agent any
    stages {
        stage('API Test') {
            steps {
                sh '''
                    newman run collections/API_Tests.json \
                      -e environments/qa.json \
                      -d test_data/users.csv \
                      --reporters cli,junit,htmlextra \
                      --reporter-junit-export results/junit-report.xml \
                      --reporter-htmlextra-export results/report.html
                '''
            }
            post {
                always {
                    junit 'results/junit-report.xml'
                    publishHTML(target: [
                        reportName: 'API Test Report',
                        reportDir: 'results',
                        reportFiles: 'report.html'
                    ])
                }
            }
        }
    }
}
```

### 2.7 Postman Mock Server

创建 Mock Server 用于前端开发阶段模拟后端接口：

```
1. 选择 Collection → "..." → "Mock Collection"
2. 配置 Mock Server 名称和响应规则
3. Postman 会生成一个 Mock URL
4. 前端可以直接调用 Mock URL 进行开发
```

### 2.8 版本管理与协作

- **Postman Cloud:** 自动同步到云端，支持版本历史
- **导出为文件:** 可将 Collection 导出为 JSON 文件纳入 Git 管理
- **Postman API:** 可通过 API 编程化管理 Collection

```bash
# 通过 Postman API 获取 Collection
curl -X GET https://api.getpostman.com/collections/{{collection_uid}} \
  -H "X-API-Key: {{postman_api_key}}"
```

## curl/httpie 命令行接口测试

### 3.1 curl 基础与进阶

curl 是 Linux/Unix 系统中极其强大的命令行 HTTP 客户端。

**基础请求：**
```bash
# GET 请求
curl https://api.example.com/users

# 带查询参数
curl "https://api.example.com/users?page=1&limit=10"

# 显示响应头
curl -i https://api.example.com/users

# 仅显示响应头
curl -I https://api.example.com/users

# 跟随重定向
curl -L https://short.link/abc

# 输出到文件
curl -o response.json https://api.example.com/data

# 静默模式（不显示进度条）
curl -s https://api.example.com/health
```

**POST/PUT/DELETE 请求：**
```bash
# POST JSON 数据
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "张三", "email": "zhangsan@example.com"}'

# POST 表单数据
curl -X POST https://api.example.com/login \
  -d "username=admin" \
  -d "password=secret"

# PUT 更新
curl -X PUT https://api.example.com/users/1 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer token123" \
  -d @update_data.json    # 从文件读取请求体

# DELETE
curl -X DELETE https://api.example.com/users/1 \
  -H "Authorization: Bearer token123"
```

**文件上传：**
```bash
curl -X POST https://api.example.com/upload \
  -F "file=@/path/to/file.pdf" \
  -F "description=测试文件"
```

**认证与Cookie：**
```bash
# Basic Auth
curl -u username:password https://api.example.com/secure

# Bearer Token
curl -H "Authorization: Bearer eyJhbGciOi..." https://api.example.com/api

# 保存 Cookie
curl -c cookies.txt https://api.example.com/login

# 使用保存的 Cookie
curl -b cookies.txt https://api.example.com/profile
```

**性能测试相关：**
```bash
# 显示详细时间信息
curl -w "\nDNS: %{time_namelookup}s\nTCP: %{time_connect}s\nTLS: %{time_appconnect}s\nTTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \
  -o /dev/null -s https://api.example.com/endpoint

# 创建 curl-format.txt 模板文件
cat > curl-format.txt << 'EOF'
    time_namelookup:  %{time_namelookup}\n
    time_connect:     %{time_connect}\n
    time_appconnect:  %{time_appconnect}\n
    time_pretransfer: %{time_pretransfer}\n
    time_redirect:    %{time_redirect}\n
    time_starttransfer: %{time_starttransfer}\n
    ----------\n
    time_total:       %{time_total}\n
EOF

curl -w "@curl-format.txt" -o /dev/null -s https://api.example.com
```

### 3.2 HTTPie (httpie)

HTTPie 提供更友好的命令行体验，输出带语法高亮。

**安装：**
```bash
# Ubuntu/Debian
sudo apt install httpie

# macOS
brew install httpie

# 通过 pip
pip install httpie

# Windows
choco install httpie
```

**基础使用：**
```bash
# GET 请求（直接使用 URL）
http https://api.example.com/users

# 带参数
http https://api.example.com/users page==1 limit==10

# 自定义请求头
http https://api.example.com/users Authorization:"Bearer token123"

# POST JSON
http POST https://api.example.com/users name="张三" email="zhangsan@example.com"

# POST 文件
http POST https://api.example.com/upload file@/path/to/file.pdf

# 查看完整请求/响应
http -v GET https://api.example.com/users

# 仅显示响应头
http -h GET https://api.example.com/users

# 仅显示响应体
http -b GET https://api.example.com/users

# 下载文件
http -d https://api.example.com/file.zip

# Session 持久化
http --session=mysession POST https://api.example.com/login username=admin password=secret
http --session=mysession GET https://api.example.com/profile
```

**curl vs HTTPie 对比：**

| 特性 | curl | HTTPie |
|------|------|--------|
| 默认安装 | 几乎全部Linux发行版 | 需要额外安装 |
| 输出格式 | 纯文本 | JSON语法高亮 |
| 表单编码 | 自动检测 | 显式指定 |
| JSON请求 | 手动设置Header和body | 自动处理 |
| 脚本化 | 极其出色 | 一般 |
| CI/CD集成 | 首选 | 可选 |

## Swagger/OpenAPI 接口文档与测试

### 4.1 OpenAPI 规范

OpenAPI Specification (OAS) 是 REST API 的标准描述格式。

**OpenAPI 3.0 文档示例片段：**
```yaml
openapi: "3.0.0"
info:
  title: 用户管理系统 API
  version: "1.0.0"
  description: 提供用户注册、登录、信息管理等接口
servers:
  - url: https://api.example.com/v1
    description: 生产环境
  - url: https://api-test.example.com/v1
    description: 测试环境

paths:
  /users:
    post:
      summary: 创建用户
      operationId: createUser
      tags:
        - 用户管理
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateUserRequest'
      responses:
        '201':
          description: 创建成功
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/UserResponse'
        '400':
          description: 请求参数错误

components:
  schemas:
    CreateUserRequest:
      type: object
      required:
        - username
        - email
      properties:
        username:
          type: string
          minLength: 3
          maxLength: 20
        email:
          type: string
          format: email
```

### 4.2 Swagger UI

将 OpenAPI 文档可视化为交互式API文档：

**通过 Docker 快速部署：**
```bash
docker run -p 8080:8080 \
  -e SWAGGER_JSON=/api/swagger.yaml \
  -v $(pwd)/swagger.yaml:/api/swagger.yaml \
  swaggerapi/swagger-ui
```

**集成到 Spring Boot (Java)：**
```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.1.0</version>
</dependency>
```

访问 `http://localhost:8080/swagger-ui.html` 即可查看交互式文档。

### 4.3 Swagger Codegen

从 OpenAPI 规范自动生成代码：

```bash
# 安装 Swagger Codegen CLI
wget https://repo1.maven.org/maven2/io/swagger/codegen/v3/swagger-codegen-cli/3.0.41/swagger-codegen-cli-3.0.41.jar -O swagger-codegen-cli.jar

# 生成 Python 客户端
java -jar swagger-codegen-cli.jar generate \
  -i api-spec.yaml \
  -l python \
  -o ./python-client

# 生成测试代码
java -jar swagger-codegen-cli.jar generate \
  -i api-spec.yaml \
  -l python \
  -o ./python-tests \
  -DtestPackage=api.tests
```

## SoapUI SOAP/REST 测试

### 5.1 概述

SoapUI 是专为 SOAP 和 REST Web 服务设计的测试工具，分为开源版和商业版(ReadyAPI)。

### 5.2 SOAP 测试示例

```
项目结构:
├── Project
│   ├── WSDL Definition
│   │   ├── Binding
│   │   │   ├── Operation: GetUserInfo
│   │   │   │   ├── Request 1
│   │   │   │   ├── Request 2 (参数化)
│   │   │   ├── Operation: CreateOrder
│   │   ├── TestSuite: 功能测试
│   │   │   ├── TestCase: 正常流程
│   │   │   │   ├── TestStep: GetAuthToken
│   │   │   │   ├── TestStep: CreateOrder
│   │   │   │   ├── TestStep: VerifyOrder
│   │   │   │   ├── Assertions:
│   │   │   │   │   ├── Valid HTTP Status 200
│   │   │   │   │   ├── XPath Match
│   │   │   │   │   ├── Not SOAP Fault
```

**Groovy 脚本示例 (SoapUI 内置脚本)：**
```groovy
// 获取请求/响应
def request = context.expand('${CreateOrder#Request}')
def response = context.expand('${CreateOrder#Response}')

// 解析 XML 响应
def groovyUtils = new com.eviware.soapui.support.GroovyUtils(context)
def holder = groovyUtils.getXmlHolder("CreateOrder#Response")
def orderId = holder.getNodeValue("//orderId")

// 设置项目属性供后续步骤使用
testRunner.testCase.setPropertyValue("orderId", orderId)

// 写入日志
log.info("订单创建成功，订单号：${orderId}")

// 断言
assert orderId != null && !orderId.isEmpty()
```

- 支持 JSON/XML 请求和响应
- 支持 REST Mock Service

## Apifox/Apizza 国产工具

### 6.1 Apifox

Apifox 是集 API 文档、API 调试、API Mock、API 自动化测试于一体的国产工具。

**核心功能：**
1. **API 文档管理：** 可视化编辑，支持导入 Swagger/Postman
2. **API 调试：** 类似 Postman 的调试体验
3. **API Mock：** 根据文档自动生成 Mock 数据
4. **自动化测试：** 可视化编排测试场景
5. **团队协作：** 多人在线协作，权限管理

**Apifox 断言示例：**
```javascript
// 前置脚本
const timestamp = Date.now();
pm.environment.set("timestamp", timestamp);

// 后置断言
pm.test("业务码为0", () => {
    pm.expect(pm.response.json().code).to.equal(0);
});

pm.test("响应中包含token", () => {
    const data = pm.response.json().data;
    pm.expect(data).to.have.property("token");
});
```

### 6.2 Apizza

Apizza 专注于团队协作的 API 管理平台，已更名为 Eolink。

**特点：**
- 自动生成 API 文档
- 支持多协议 (HTTP, WebSocket, gRPC)
- 支持自动化测试编排
- 支持 Mock 服务
- 支持版本管理和变更通知

## 5个使用技巧和最佳实践

### 技巧1: Postman 变量优先级

Postman 变量的优先级从低到高：
```
Global < Collection < Environment < Data < Local
```

**实际应用：** 将不变量设为 Collection 级别，环境相关变量设为 Environment 级别，临时数据用 Local 变量。

### 技巧2: 构建请求链 (Chaining Requests)

利用环境变量在不同请求间传递数据：

```javascript
// 第一步：登录请求的Tests脚本
pm.test("提取token", function() {
    const json = pm.response.json();
    pm.environment.set("token", json.data.token);
    pm.environment.set("user_id", json.data.userId);
});

// 第二步：后续请求自动使用 {{token}}
// Header: Authorization: Bearer {{token}}
```

### 技巧3: curl + jq 组合

```bash
# 发送请求并用 jq 格式化/过滤响应
curl -s https://api.example.com/users | jq '.'

# 提取特定字段
curl -s https://api.example.com/users | jq '.data[].name'

# 过滤数据
curl -s https://api.example.com/users | jq '.data[] | select(.age > 18)'

# 链式请求：先获取token，再发起请求
curl -s https://api.example.com/auth \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"123"}' | \
  jq -r '.token' | \
  xargs -I{} curl -H "Authorization: Bearer {}" \
  https://api.example.com/protected-resource
```

### 技巧5: 测试数据管理策略

```

│  ┌─────────────┐ ┌────────────┐  │

│  ┌─────────────┐ ┌────────────┐  │

└──────────────────────────────────┘
```

**数据隔离原则：**
- 每个测试用例使用独立的测试数据
- 测试结束清理测试数据（Pre-request中生成，Tests中清理）
- 使用时间戳或UUID确保数据唯一性

```javascript
// Pre-request Script: 生成唯一测试数据
const uuid = require('uuid');
pm.variables.set("test_email", `test_${uuid.v4()}@example.com`);
pm.variables.set("test_phone", `138${Math.floor(Math.random() * 100000000).toString().padStart(8, '0')}`);

// Tests: 清理测试数据
pm.test("清理测试用户", function() {
    const userId = pm.environment.get("created_user_id");
    pm.sendRequest({
        url: `${pm.environment.get("base_url")}/users/${userId}`,
        method: 'DELETE',
        header: {
            'Authorization': `Bearer ${pm.environment.get("token")}`
        }
    });
});
```

## 总结

接口测试工具的选择应当基于团队规模、技术栈和具体需求：

1. **入门首选 Postman** - 生态系统最完善
2. **国内团队用 Apifox** - 文档/调试/Mock/自动化一体化
3. **CI/CD 用 curl + Newman** - 脚本化程度最高
4. **SOAP/WebService 用 SoapUI** - 专为此设计
5. **命令行诊断用 curl + jq** - 最灵活的组合

掌握这些工具的核心用法，能够在实际工作中根据场景灵活选择，大幅提升接口测试的效率和覆盖度。
