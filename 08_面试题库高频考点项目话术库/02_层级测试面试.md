### Q1：什么是单元测试？它由谁来做？为什么重要？

**题目类别**：基础概念
**难度**：初级
**出现频率**：高

**考点分析**
单元测试是测试金字塔的底座，理解其价值对于推动开发自测和质量左移至关重要。

**参考答案**

单元测试是对软件中最小的可测试单元（通常是函数、方法或类）进行的测试，目的是验证该单元的行为是否符合设计预期。

**执行者**：主要由开发工程师编写和执行。测试工程师可以提供测试思路、用例评审、以及推动单元测试的覆盖率达标。

**为什么重要（测试金字塔的底座）**：

1. **缺陷发现最早**：在编码阶段就发现问题，修复成本极低
2. **反馈最快**：秒级完成，开发提交前就能得到反馈
3. **重构安全网**：有了充分的单元测试，开发重构代码时才有信心
4. **活文档**：单元测试本身就是代码行为的最佳文档
5. **设计质量指示器**：难以写单元测试的代码往往设计不佳（耦合度高）

**测试金字塔**：
```
         /\
        /E2E\        少量端到端测试（慢、贵、真实环境）
       /------\
      /集成测试 \     适量集成测试
     /----------\
    /  单元测试   \   大量单元测试（快、便宜、隔离）
   /--------------\
```

**加分项 / 深度展开**

- 提到单元测试与TDD的区别：TDD是一种开发模式（先写测试再写代码），单元测试是TDD的产物
- 提到测试工程师在单元测试中的角色：不是在IDE里写单元测试，而是帮助开发设计测试用例、评审测试覆盖、建立覆盖率门禁
- 举例：在我的项目中，推动开发将增量代码的单元测试覆盖率从30%提升到80%后，提测质量明显提升，集成测试阶段的Bug密度下降了40%

### Q2：什么是TDD测试驱动开发？TDD的三个步骤是什么？

**题目类别**：开发模式
**难度**：中级
**出现频率**：中高

**考点分析**
TDD是现代软件开发的重要实践，测试工程师需要理解其理念和方法。

**参考答案**

TDD(Test-Driven Development)测试驱动开发是一种软件开发方法，强调在编写功能代码之前先编写测试代码，然后编写刚好能通过测试的代码，最后重构优化。

**TDD三步骤（Red-Green-Refactor循环）**：

```
1. RED（写一个失败的测试）
   先写一个测试用例，描述期望的行为
   运行测试 -> 失败（红色）
   验证测试用例本身是正确的

2. GREEN（写最少的代码使测试通过）
   编写刚好足够让测试通过的代码
   不写多余的代码、不过度设计
   运行测试 -> 通过（绿色）

3. REFACTOR（重构）
   在保持测试绿色的前提下，优化代码结构
   消除重复、改善可读性
   运行测试 -> 仍然绿色
   
   然后回到第1步继续下一个测试...
```

**TDD的核心原则**：
- 没有测试就不写生产代码
- 只写刚好能让测试失败的测试代码
- 只写刚好能让测试通过的生产代码

**TDD的收益**：
- 从使用者的角度设计接口（可测试性倒逼好设计）
- 每个功能都有测试保护
- 短反馈循环，快速发现问题
- 代码覆盖率自然很高

**加分项 / 深度展开**

- 区分TDD、ATDD和BDD：
  - TDD：开发视角，测试代码单元的正确性
  - ATDD(Acceptance TDD)：业务视角，测试验收条件的满足
  - BDD(Behavior Driven Development)：行为驱动开发，用自然语言描述行为（Given-When-Then）
- 提到TDD的挑战：初期效率下降、对开发技能要求高、遗留代码难以应用
- 举例：在一个接口开发任务中，我看到开发先写了测试用例（描述各种输入和预期输出），然后再写实现代码，最终接口的异常处理覆盖面远超没有TDD的模块

### Q3：JUnit中有哪些常用注解？它们的作用是什么？

**题目类别**：工具
**难度**：初级
**出现频率**：高

**考点分析**
JUnit是Java生态最基础的单元测试框架，也是面试中常考的细节。

**参考答案**

**JUnit 5核心注解**：

| 注解 | 作用 | 说明 |
|------|------|------|
| `@Test` | 标记测试方法 | 被标记的方法作为一个测试用例执行 |
| `@BeforeEach` | 每个测试前执行 | 每个@Test方法执行前运行（JUnit 4中为@Before） |
| `@AfterEach` | 每个测试后执行 | 每个@Test方法执行后运行（JUnit 4中为@After） |
| `@BeforeAll` | 所有测试前执行一次 | 必须为static方法（JUnit 4中为@BeforeClass） |
| `@AfterAll` | 所有测试后执行一次 | 必须为static方法（JUnit 4中为@AfterClass） |
| `@DisplayName` | 测试显示名称 | 可读性好的测试名称，支持中文和Emoji |
| `@Disabled` | 禁用测试 | 暂时跳过该测试（JUnit 4中为@Ignore） |
| `@ParameterizedTest` | 参数化测试 | 使用不同参数多次运行同一测试方法 |
| `@ValueSource` | 提供测试参数值 | 与@ParameterizedTest配合，提供单个参数值 |
| `@CsvSource` | 提供CSV格式参数 | 与@ParameterizedTest配合，提供多参数值 |
| `@MethodSource` | 从方法提供参数 | 与@ParameterizedTest配合 |
| `@RepeatedTest` | 重复测试 | 重复执行指定次数 |
| `@Timeout` | 超时控制 | 测试超过指定时间则失败 |
| `@Nested` | 内嵌测试类 | 组织层级化的测试结构 |

**代码示例**：

```java
import org.junit.jupiter.api.*;

class CalculatorTest {
    private Calculator calculator;

    @BeforeEach
    void setUp() {
        calculator = new Calculator(); // 每个测试方法执行前初始化
    }

    @Test
    @DisplayName("测试两个正整数相加")
    void testAddPositiveNumbers() {
        assertEquals(5, calculator.add(2, 3));
    }

    @Test
    @DisplayName("测试除数为零时抛出异常")
    void testDivideByZero() {
        assertThrows(ArithmeticException.class, () -> {
            calculator.divide(10, 0);
        });
    }

    @ParameterizedTest
    @CsvSource({"1,2,3", "0,0,0", "-1,1,0", "100,200,300"})
    @DisplayName("参数化测试加法")
    void testAdd(int a, int b, int expected) {
        assertEquals(expected, calculator.add(a, b));
    }
}
```

**加分项 / 深度展开**

- 提到JUnit 4 vs JUnit 5的区别（包名、扩展机制等）
- 提到断言库：JUnit自带的Assertions、Hamcrest（更具描述性）、AssertJ（流式断言）
- 提到Mock框架配合：Mockito + JUnit 是目前最主流的单元测试组合

### Q4：什么是Mock？什么时候需要使用Mock？

**题目类别**：测试技术
**难度**：中级
**出现频率**：高

**考点分析**
Mock是单元测试的核心技术，考察候选人是否理解隔离测试的理念。

**参考答案**

Mock（模拟）是在测试中创建一个对象的模拟实现，用于替代真实依赖，使被测单元能够被独立测试。

**需要使用Mock的场景**：

1. **外部依赖**
   - 数据库、消息队列、Redis等
   - Mock后测试不依赖真实基础设施

2. **第三方服务**
   - 支付接口、短信网关、推送服务
   - Mock后测试不依赖外部服务可用性

3. **时间/日期相关**
   - `new Date()`、`System.currentTimeMillis()`
   - Mock后可以控制"现在是什么时间"

4. **文件/网络I/O**
   - 文件系统操作、HTTP请求
   - Mock后测试更快更稳定

5. **未实现的功能**
   - 依赖的其他模块尚未完成
   - Mock后可并行开发

6. **难以构造的状态**
   - 特定异常、特定返回值
   - Mock后可以轻松模拟各种边界情况

**Mock的原则**：
- 只Mock自己的依赖（不Mock被测对象本身）
- 不要Mock值对象（简单数据结构）
- 不要过度Mock（Mock太多说明设计可能有问题）

**代码示例（Mockito）**：

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private PaymentGateway paymentGateway; // Mock外部支付网关

    @Mock
    private UserRepository userRepository; // Mock数据库操作

    @InjectMocks
    private OrderService orderService; // 将被测对象的Mock依赖注入

    @Test
    @DisplayName("支付成功时应返回成功状态")
    void shouldReturnSuccessWhenPaymentSucceeds() {
        // Given
        when(paymentGateway.charge(any(), any(BigDecimal.class)))
            .thenReturn(new PaymentResult("SUCCESS"));
        when(userRepository.findById(1L))
            .thenReturn(Optional.of(new User(1L, "testUser")));

        // When
        OrderResult result = orderService.createOrder(1L, new BigDecimal("100"));

        // Then
        assertEquals("SUCCESS", result.getStatus());
        verify(paymentGateway, times(1)).charge(any(), any()); // 验证支付被调用了1次
    }
}
```

**加分项 / 深度展开**

- 区分Stub、Mock、Fake、Spy、Dummy的概念：Stub返回预设值，Mock验证交互行为，Fake是轻量级替代实现，Spy记录调用信息
- 提到Mockito的核心方法：`when().thenReturn()`（Stub）、`verify()`（Mock验证）、`any()`（参数匹配器）、`doThrow()`（模拟异常）
- 提到过度Mock的坏味道：如果测试中Mock了一大堆对象，说明被测类做了太多事（单一职责问题）

### Q5：单元测试和集成测试的本质区别是什么？

**题目类别**：概念辨析
**难度**：中级
**出现频率**：中高

- 单元测试：测试单个单元（方法/类），依赖全部Mock，快，发现的是单元内部逻辑错误
- 集成测试：测试多个单元之间的协作，使用真实的依赖或Fake，慢一些，发现的是接口不对、数据传递错误等协作问题
- 经典引用："单元测试确保每个零件是对的，集成测试确保零件能拼在一起。"

**参考答案**

| 维度 | 单元测试(Unit Test) | 集成测试(Integration Test) |
|------|-------------------|--------------------------|
| **测试对象** | 单个函数/方法/类 | 多个模块/组件的协作 |
| **依赖处理** | 全部Mock | 使用真实依赖或轻量级替代 |
| **执行速度** | 毫秒到秒级 | 秒到分钟级 |
| **失败根因** | 容易定位（具体的逻辑错误） | 需要排查（集成点出问题） |
| **测试环境** | 无需真实环境 | 可能需要数据库/消息队列等 |
| **编写者** | 开发工程师 | 开发/测试工程师 |
| **反馈周期** | 即时（本地IDE） | 稍长（可能需要部署） |
| **稳定性** | 高（无外部依赖） | 中（受环境影响） |

**通俗理解**：
- 单元测试：检验零件是否是合格的
- 集成测试：检验零件组装后能否协同工作

**经典的"两个模块都通过单元测试但集成失败"案例**：
- 模块A调用模块B时，A传的是（姓名, 年龄），B期望的参数顺序是（年龄, 姓名）。各自的单元测试Mock了对方，都通过了，但集成时就出错了。

**加分项 / 深度展开**

- 提到测试金字塔中的分配比例：单元测试应该占70%，集成测试20%，端到端测试10%
- 提到"集成测试的范围"：可以是很窄的集成（两个类的集成），也可以是很宽的集成（多个服务的集成）。范围越窄越快越稳定，范围越宽越接近真实
- 举例：我们项目中，单元测试(900+个)3分钟内跑完，集成测试(150+个，含真实数据库和消息队列)需要25分钟，但发现了多个线程安全和数据一致性问题

### Q6：如何保证单元测试的质量？有哪些评审维度？

**题目类别**：测试质量
**难度**：中高
**出现频率**：中

**考点分析**
考察候选人是否能评判单元测试的质量。低质量的单元测试等于没写。

**参考答案**

**高质量单元测试的标准**：

1. **F.I.R.S.T原则**
   - Fast（快速）：毫秒级别
   - Independent（独立）：每个测试不依赖其他测试
   - Repeatable（可重复）：任何环境任何次数结果一致
   - Self-Validating（自验证）：不依赖人工判断
   - Timely（及时）：在写生产代码之前或同时写

2. **测试覆盖率不等于测试质量**
   - 100%覆盖率但全都是无断言的测试 = 质量为零
   - 覆盖率是必要条件不是充分条件

3. **测试命名和可读性**
   - 测试名应描述被测场景和预期行为
   - 使用Given-When-Then结构组织测试代码

4. **断言的有效性**
   - 必须有有意义的断言
   - 断言要精确（不要用`assertNotNull`就算了）
   - 正向和负向都要覆盖

5. **测试独立性**
   - 测试之间不共享可变状态
   - 执行顺序不影响结果

**单元测试评审清单**：

| 评审维度 | 检查点 |
|---------|--------|
| **正确性** | 测试是否验证了正确的行为？断言是否合理？ |
| **完整性** | 是否覆盖了Happy Path、边界值、异常路径？ |
| **可读性** | 测试命名是否自解释？结构是否清晰(Given-When-Then)？ |
| **独立性** | 测试是否依赖其他测试的执行结果？ |
| **稳定性** | 测试是否有时过有时不过(Flaky Test)？ |
| **有效性** | 测试是否真的在验证行为，而非实现细节？ |
| **速度** | 单个测试和整个测试套件是否足够快？ |

**加分项 / 深度展开**

- 提到"测试代码也是代码"的理念：测试代码也需要重构，也有可维护性问题
- 提到常见的"坏单元测试"特征：只测getter/setter、Mock一切、测试实现细节而非行为、测试之间暗含依赖
- 举例：评审开发提交的单元测试时，发现一个测试"mock了数据库连接，验证了调用了save方法"，但实际上这个测试只是验证了Mockito的行为——没有任何业务逻辑被测试

### Q7：JUnit中assertEquals和assertSame的区别是什么？

**题目类别**：工具细节
**难度**：初级
**出现频率**：中

**考点分析**
基础工具API的理解，反映Java基本功。

**参考答案**

| | assertEquals | assertSame |
|---|-------------|------------|
| **比较方式** | 使用`equals()`方法比较值 | 使用`==`比较引用 |
| **适用类型** | 比较两个对象的值是否相等 | 比较两个引用是否指向同一对象 |
| **典型场景** | 比较String、数字、POJO的内容 | 验证单例模式、验证同一对象返回 |

**代码示例**：

```java
@Test
void testAssertEqualsVsAssertSame() {
    String str1 = new String("hello");
    String str2 = new String("hello");
    String str3 = str1;

    // assertEquals: 比较值，内容相同则通过
    assertEquals(str1, str2); // 通过！两个String内容都是"hello"

    // assertSame: 比较引用，必须是同一个对象才通过
    assertSame(str1, str3);   // 通过！str3和str1指向同一个对象
    assertSame(str1, str2);   // 失败！str1和str2是不同的对象
}
```

**加分项 / 深度展开**

- 补充其他常用断言方法：`assertTrue/assertFalse`、`assertNull/assertNotNull`、`assertThrows`(验证异常)、`assertAll`(批量断言)、`assertArrayEquals`、`assertIterableEquals`
- 提到AssertJ的流式断言风格更加推荐：`assertThat(result).isNotNull().hasSize(3).contains("expected")`

### Q8：什么是代码覆盖率？JaCoCo的覆盖率类型有哪些？

**题目类别**：工具/度量
**难度**：中级
**出现频率**：中

**考点分析**
代码覆盖率是衡量单元测试充分性的重要指标，考察工具和度量理解。

**参考答案**

JaCoCo(Java Code Coverage)是Java生态主流的代码覆盖率工具，提供了多种覆盖率粒度：

| 覆盖率类型 | 英文名 | 说明 |
|-----------|--------|------|
| 指令覆盖 | Instructions Coverage | 最细粒度，Java字节码指令级别，衡量字节码指令是否被执行过 |
| 分支覆盖 | Branches Coverage | if/else、switch、三元运算符等分支是否都被覆盖 |
| 圈复杂度覆盖 | Cyclomatic Complexity | 方法的圈复杂度信息 |
| 行覆盖 | Lines Coverage | 源代码行是否被覆盖（最直观） |
| 方法覆盖 | Methods Coverage | 方法是否被调用过（粗粒度） |
| 类覆盖 | Classes Coverage | 类是否被测试涉及（最粗粒度） |

**JaCoCo使用方式**：

1. **Maven集成**
```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.10</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal> <!-- 启动时挂载agent -->
                <goal>report</goal>         <!-- 生成报告 -->
            </goals>
        </execution>
    </executions>
</plugin>
```

2. **覆盖率报告解读**
   - 绿色：完全覆盖
   - 黄色：部分覆盖（分支未全覆盖）
   - 红色：未覆盖

**覆盖率门禁**：
```xml
<rule>
    <element>BUNDLE</element>
    <limits>
        <limit>
            <counter>BRANCH</counter>
            <value>COVEREDRATIO</value>
            <minimum>0.80</minimum> <!-- 分支覆盖率不低于80% -->
        </limit>
    </limits>
</rule>
```

**加分项 / 深度展开**

- 提到覆盖率的使用原则：关注增量超出存量、关注未覆盖的代码是核心逻辑还是辅助代码、不要为了数字而写低质量测试
- 具体设置建议：增量代码分支覆盖率>80%、行覆盖率>90%
- 提到Mutation Testing（变异测试）：真正的测试质量度量，比覆盖率更可靠

### Q9：如何处理Flaky Tests（不稳定测试）？

**题目类别**：测试工程
**难度**：中高
**出现频率**：中

**考点分析**
不稳定测试是自动化测试中的慢性病，考察工程化思维。

**参考答案**

Flaky Test（不稳定测试/脆性测试）是指在没有代码变更的情况下，同一个测试有时通过有时失败的现象。

**常见原因和解决方案**：

1. **时间依赖**
   - 原因：测试依赖当前时间、超时设置太紧
   - 方案：Mock时间源；使用`@Timeout`设置合理超时

2. **数据依赖**
   - 原因：测试间的数据污染（共享数据库/共享状态）
   - 方案：每个测试前清理/重建数据；使用事务回滚

3. **顺序依赖**
   - 原因：测试用例之间存在隐式依赖（如TestA创建了数据，TestB依赖该数据）
   - 方案：打破测试间依赖；使用随机执行顺序插件

4. **并发问题**
   - 原因：测试涉及多线程，竞态条件导致结果不稳定
   - 方案：使用CountDownLatch/Awaitility等同步工具

5. **外部依赖不稳定**
   - 原因：依赖的外部服务偶尔不可用
   - 方案：Mock外部依赖；添加Retry机制

6. **环境差异**
   - 原因：本地通过CI失败（或反过来）
   - 方案：统一环境（Docker）；检查Locale、编码等配置

**Flaky Test的处理策略**：

```
发现Flaky Test
    |
    v
记录失败信息（截图、日志、时间）
    |
    v
尝试复现（多次执行）
    |
    v
分析根因 -> 修复 -> 验证稳定
    |
    v
如果短期无法修复：
    - 暂时@Disabled + 创建修复任务
    - 严重时从CI流水线移除
```

**加分项 / 深度展开**

- 提到量化指标：Flaky Test Ratio = 不稳定测试数量 / 总测试数量，理想目标是<1%
- 提到CI中Flaky Test的策略：自动Retry 1-2次（但要记录原因），如果连续失败则判定为真实失败
- 举例：项目中有一个测试在"每个月的最后一天"会失败——原因是日期处理的边界值没考虑跨月。最后通过Mock时间源并补充边界用例解决

### Q10：TDD在实际落地中的挑战有哪些？如何推动团队实践TDD？

**题目类别**：工程实践
**难度**：中高
**出现频率**：中低

**考点分析**
考察候选人是否有推动工程实践落地的经验。

**参考答案**

**TDD落地的主要挑战**：

1. **开发习惯阻力**
   - 传统"先写代码再补测试"的习惯根深蒂固
   - 应对：渐进式推广，先从新模块/核心模块开始

2. **初期效率下降**
   - 学习曲线导致短期产出下降（1-2个迭代）
   - 应对：管理层需要接受短期阵痛，关注中长期质量收益

3. **遗留代码困境**
   - 老代码架构差、耦合高，难以添加测试
   - 应对：遗留代码先做"特征测试"(Characterization Test)，修改时再加单元测试

4. **技能不足**
   - 团队缺乏单元测试/Mock/重构技能
   - 应对：组织培训、Pair Programming、Code Review中指导

5. **缺乏质量文化**
   - 组织只关注交付速度，不关注测试
   - 应对：用数据说话（Bug率、修复成本、线上故障）来证明TDD的价值

**推动策略**：

1. 从小范围试点开始，做出标杆
2. 建立覆盖率门禁（CI流水线检查）
3. 将单元测试质量纳入Code Review标准
4. 定期分享TDD实践经验和收益数据

**加分项 / 深度展开**

- 提到"测试后行(Test-After)"与TDD的比较：测试后行如果能保证质量也是可以的，重要的是"有测试"而非"怎么写测试"
- 举例：我推动TDD从"一上来就让全团队做TDD"失败了，后来改为"新功能必须写测试"（不强制TDD的顺序）成功了。半年后开发主动开始TDD，因为他们自己体验到了先写测试带来的设计改善

### Q11：单元测试中如何处理private方法？应该直接测试private方法吗？

**题目类别**：测试设计
**难度**：中高
**出现频率**：中

**考点分析**
涉及测试设计哲学问题：测试行为还是测试实现细节。

**参考答案**

**核心观点：不应该直接测试private方法。**

**原因**：
1. Private方法是实现细节，不是公开行为
2. Private方法通过public方法的调用间接被测试
3. 直接测试private方法会导致测试与实现细节强耦合（重构private方法时测试会无关失败）
4. 如果private方法复杂到需要独立测试，那它应该被提取为独立的类

**处理方式**：

```java
// 不好的做法：用反射测试private方法
// 使用反射是一种测试坏味道

// 好的做法：通过测试public方法间接覆盖private方法
// 设计能覆盖private方法各种路径的public方法测试用例

// 更好的做法：重构！如果private方法真的需要独立测试
// 说明它应该是一个独立的职责，提取为新的类
class PriceCalculator {
    public BigDecimal calculatePrice(Order order) {
        BigDecimal basePrice = calculateBasePrice(order); // 原来是private
        BigDecimal discount = calculateDiscount(order);   // 原来是private
        return basePrice.subtract(discount);
    }
    // 提取为DefaultPricingStrategy后，public方法可以独立测试
}
```

**例外情况**：
- 极其复杂的算法逻辑private方法，重构成本过高，可以临时使用反射测试（但标记TODO重构）

**加分项 / 深度展开**

- 提到"测试驱动设计"的理念：如果你觉得测试private方法很困难，那可能是设计需要改进的信号
- 提到包级私有(package-private)作为折中方案：将方法从private改为package-private（不加修饰符），使测试类（同包）可以访问，同时仍对外部包隐藏

### Q12：单元测试和代码重构的关系是什么？

**题目类别**：工程实践
**难度**：中级
**出现频率**：中

**参考答案**

单元测试和代码重构是相辅相成的关系。

**单元测试是重构的安全网**：
- 有了充分的单元测试，开发才有信心重构代码
- 重构后全量跑测试：如果绿色 -> 重构没破坏功能；如果红色 -> 及时发现引入的问题

**重构是改善可测试性的手段**：
- 难以测试的代码通常需要重构
- 重构后提升了可测试性 -> 更容易添加测试 -> 更安全的重构 -> 正向循环

**重构安全三步走**：
```
Cover -> Change -> Verify
(先有测试覆盖 -> 再做变更 -> 验证测试通过)
```

**加分项 / 深度展开**

- 提到"没有测试的重构就是重写"——风险巨大
- 提到重构的两个层次：微重构（几分钟完成，如重命名、提取方法）和宏重构（架构层面的变化），单元测试对二者都至关重要

### Q13：什么是代码的可测试性？如何提高？

**题目类别**：代码质量
**难度**：中高
**出现频率**：中

**参考答案**

代码的可测试性(Testability)是指代码在多大程度上能够被有效测试。

**可测试的代码特征**：
- 低耦合、高内聚
- 依赖注入（而非内部new）
- 单一职责
- 明确的输入和输出
- 无隐藏状态

**提升可测试性的方法**：
1. 依赖注入替代直接new
2. 避免静态方法调用（难以Mock）
3. 接口隔离
4. 减少副作用（把计算和IO分离）

**加分项 / 深度展开**

- 提到"可测试性是可维护性的副产品"——好设计的代码自然好测试
- 提到"测试痛苦就是设计信号"

### Q14：BDD（行为驱动开发）与TDD有什么区别？Cucumber是什么？

**题目类别**：开发方法
**难度**：中级
**出现频率**：中

**参考答案**

BDD(Behavior Driven Development)是TDD的演进，强调用自然语言描述软件行为，促进非技术人员参与。

**BDD的核心格式(Gherkin语法)**：
```
Feature: 用户登录
  Scenario: 使用正确的用户名密码登录成功
    Given 用户在登录页面
    When 输入正确的用户名和密码
    And 点击登录按钮
    Then 跳转到首页
    And 显示欢迎消息
```

**TDD vs BDD**：

| 维度 | TDD | BDD |
|------|-----|-----|
| 关注点 | 代码的正确性 | 系统的行为 |
| 语言 | 技术语言（Java/Python等） | 自然语言（Gherkin语法） |
| 参与者 | 开发者 | 开发者+测试+产品 |
| 粒度 | 类/方法级别 | 功能/场景级别 |
| 工具 | JUnit/NUnit/pytest | Cucumber/SpecFlow/Behave |

**Cucumber**是最主流的BDD工具：
- 用.feature文件定义场景
- Step Definitions连接自然语言和代码
- 测试结果以自然语言报告展示

**加分项 / 深度展开**

- 提到BDD的挑战：维护.feature文件和Step Definition的成本
- 举例：在有产品参与的项目中，BDD效果很好（产品能看懂测试场景）；在纯技术项目（如中间件），BDD增加了一层不必要的抽象

### Q15：单元测试的Test Fixture是什么？如何设计好的Test Fixture？

**题目类别**：测试设计
**难度**：初级
**出现频率**：中

**参考答案**

Test Fixture（测试夹具/测试装置）是测试执行前准备的固定环境和数据状态。

**好的Test Fixture设计原则**：
1. 每个测试方法独立运行，不共享可变状态
2. 使用工厂方法/Builder模式创建测试数据
3. 测试数据语义化（容易理解这个数据是干什么的）

**代码示例**：

```java
@BeforeEach
void setUp() {
    // Test Fixture：为每个测试准备干净的初始状态
    user = User.builder()
            .id(1L)
            .name("测试用户")
            .balance(new BigDecimal("1000.00"))
            .build();
    orderRepository = new InMemoryOrderRepository(); // 轻量级内存实现
    orderService = new OrderService(orderRepository, paymentGateway);
}
```

**加分项 / 深度展开**

- 提到Test Fixture过多时的管理策略：提取为共享的ObjectMother模式或Test Data Builder
- 提到"Fixture Setup是测试复杂度的晴雨表"——如果Fixture很复杂，说明被测对象依赖太多

## 二、集成测试

### Q16：什么是集成测试？集成测试的策略有哪些？

**题目类别**：集成测试
**难度**：中级
**出现频率**：高

**考点分析**
集成测试连接单元测试和系统测试，考察对其策略的系统理解。

**参考答案**

集成测试是验证多个软件模块/组件之间交互和协作是否正确的测试。

**集成测试策略**：

1. **大爆炸集成(Big-Bang Integration)**
   - 方式：将所有模块一次性组装在一起测试
   - 优点：简单直接
   - 缺点：失败后定位根因困难，问题相互掩盖
   - 适用：小型项目、模块间耦合小

2. **自顶向下集成(Top-Down Integration)**
   - 方式：从顶层模块开始，逐步向下集成。下层未完成时用Stub（桩）替代
   - 优点：早期能验证主流程和架构
   - 缺点：Stub开发成本高，底层问题发现晚
   - 适用：架构驱动型项目

3. **自底向上集成(Bottom-Up Integration)**
   - 方式：从底层模块开始，逐步向上集成。上层未完成时用Driver（驱动）替代
   - 优点：底层模块稳定，问题容易定位
   - 缺点：顶层功能验证较晚
   - 适用：底层复用性高的项目

4. **三明治集成(Sandwich Integration)**
   - 方式：结合自顶向下和自底向上，中间层最后集成
   - 优点：兼具两者优点，并行度高
   - 缺点：中间层集成可能成为瓶颈
   - 适用：大型分层系统

5. **持续集成(Continuous Integration)**
   - 方式：每次代码提交都触发自动化的集成测试
   - 优点：极早发现问题，持续反馈
   - 缺点：需要CI基础设施和自动化测试
   - 适用：敏捷/DevOps团队

**加分项 / 深度展开**

- 提到微服务架构中的集成测试：服务间契约测试(Contract Testing)和Pact框架
- 举例：在我做的电商项目中，订单服务和库存服务采用自底向上集成——先确保库存服务独立稳定，再集成到订单流程中。使用Pact进行服务间契约验证，提前发现了3次接口不一致的问题

---

### Q17：什么是Stub和Driver？

**题目类别**：集成测试
**难度**：中级
**出现频率**：中高

**参考答案**

在集成测试中，Stub和Driver都是替代未完工组件的测试代码：

| | Stub（桩） | Driver（驱动） |
|---|-----------|-------------|
| **替代对象** | 被调用的下层模块（被依赖方） | 调用上层模块（依赖方） |
| **作用** | 模拟下层返回预设数据 | 模拟上层发起调用 |
| **使用场景** | 自顶向下集成中替代未完成的下层 | 自底向上集成中替代未完成的上层 |
| **复杂度** | 通常较简单（返回固定数据） | 通常较复杂（需要组织调用序列） |

**举例**：
```
被测模块A -> 模块B(未完成) => 用Stub替代B，预设返回值
被测模块C <- 模块D(未完成) => 用Driver替代D，主动调用C
```

**加分项 / 深度展开**

- 提到现代开发中Stub/Driver的角色变化：微服务和Mock框架的普及，使Stub/Driver的使用场景减少（更多用Mock框架）
- 提到WireMock等工具可用于创建HTTP服务的Stub

---

### Q18：数据库集成测试有哪些常见方法？

**题目类别**：集成测试
**难度**：中高
**出现频率**：中

**参考答案**

数据库集成测试的几种主流方案：

1. **内存数据库（H2/HSQLDB）**
   - 优点：速度极快，无需外部依赖
   - 缺点：语法差异可能导致生产环境Bug漏测

2. **Testcontainers（Docker化真实数据库）**
   - 优点：使用与生产一致的数据库
   - 缺点：启动较慢

3. **事务回滚**
   - 每个测试在事务内运行，结束后回滚
   - 不支持测试事务行为本身

4. **每次重建**
   - 每个测试前重建Schema和Data

**推荐方案**：Testcontainers + 事务回滚的组合

**加分项 / 深度展开**

- 提到Testcontainers方案在现代项目中的主流地位
- 提到数据库迁移工具(Flyway/Liquibase)集成

系统测试是在完整的、集成后的系统上进行的测试，目的是验证系统是否满足指定的需求。

9. 配置测试
10. 本地化/国际化测试

- 提到系统测试与非功能测试的关系：系统测试是范围概念，非功能测试是类型概念

### Q20：功能测试和系统测试的关系是什么？

**题目类别**：概念辨析
**难度**：初级
**出现频率**：中高

**参考答案**

功能测试是系统测试的一个子集。

| | 功能测试 | 系统测试 |
|---|---------|---------|
| 范围 | 仅验证功能需求 | 功能+非功能（性能/安全/兼容/易用...） |
| 粒度 | 可针对单个功能 | 必须是完整系统 |
| 关系 | 系统测试的组成部分 | 包含功能测试 |

**加分项 / 深度展开**

- 提到"有些功能测试在集成测试阶段就可以做，不需要等到系统测试"
- 举例：订单提交功能可以在集成测试阶段测基本逻辑，在系统测试阶段测端到端流程和并发性能

## 四、验收测试

---

### Q21：什么是验收测试？验收测试的类型有哪些？

**题目类别**：验收测试
**难度**：中级
**出现频率**：高

**参考答案**

验收测试是确保系统满足业务需求、用户可接受的测试，是软件交付前的最后一道测试关口。

**验收测试类型**：

1. **UAT(User Acceptance Testing，用户验收测试)**
   - 由最终用户/业务方在真实或仿真环境中执行
   - 验证系统是否满足业务需求和工作流程

2. **Alpha测试**
   - 在开发组织内部进行，由内部用户/测试人员执行
   - 环境可控，问题可快速反馈

3. **Beta测试**
   - 在真实用户环境中进行，由外部用户参与
   - 收集真实场景反馈，环境不可控

4. **合同验收测试**
   - 按合同约定的验收标准进行
   - 外包项目常见

5. **法规验收测试**
   - 验证系统符合法规要求

**UAT的主要活动**：
1. 测试用例：基于业务场景的真实操作流程
2. 数据：使用真实或接近真实的业务数据
3. 参与者：业务方/最终用户
4. 关注点：不是技术Bug，而是业务规则、工作流程、可用性

**加分项 / 深度展开**

- 提到UAT常见的失败原因：业务方准备不足（没时间参与）、验收用例不充分、环境数据不真实
- 举例：一个金融系统UAT中，业务方用真实月末结账数据跑了一遍，发现了测试团队未发现的"四舍五入导致的1分钱差异"——这个差异在技术上正确（符合四舍五入规则），但在财务上不可接受（报表不平）

### Q22：Alpha测试和Beta测试有什么区别？

**题目类别**：验收测试
**难度**：初级
**出现频率**：中高

**参考答案**

| 维度 | Alpha测试 | Beta测试 |
|------|----------|---------|
| 地点 | 开发组织内部 | 用户实际环境 |
| 参与者 | 内部员工/测试团队 | 真实外部用户 |
| 环境 | 可控 | 不可控 |
| 目的 | 发现明显的Bug | 收集真实使用反馈 |
| 时间 | Beta之前 | Alpha之后，正式发布前 |
| 数据 | 测试数据 | 用户真实数据 |

**加分项 / 深度展开**

- 提到现代的"内测"和"公测"概念：内测=Alpha，公测=Beta
- 举例：微信大的版本发布前都会经过内部Alpha（几千名内部员工使用2-4周），然后是外部Beta（邀请部分用户灰度使用），最后是全量发布

### Q23：测试人员在UAT阶段的角色和职责是什么？

**题目类别**：验收测试
**难度**：中级
**出现频率**：中

### Q24：如何制定UAT的验收标准？

**题目类别**：验收测试
**难度**：中高
**出现频率**：中

**参考答案**

UAT验收标准的制定方法：

1. **基于需求**：核心业务需求100%通过 -> 验收通过
2. **基于风险**：关键业务流程无阻塞性问题 -> 验收通过
3. **基于缺陷统计**：无P0/P1缺陷，P2缺陷不超过X个
4. **基于用户反馈**：核心用户的正向反馈率 >= X%

**验收标准检查清单**：
- 所有核心业务场景可走通
- 无数据错误/丢失问题
- 关键接口响应时间在可接受范围
- 所有P0/P1缺陷已修复并验证

**加分项 / 深度展开**

- 提到提前与业务方达成验收标准的共识，避免验收阶段的争议
- 举例：项目中双方共同约定的验收标准是："至少3名业务人员完成10个核心业务场景的测试，无P0/P1未解决缺陷，P2缺陷<=5个"

### Q25：回归测试和验收测试的区别是什么？

**题目类别**：概念辨析
**难度**：初级

**参考答案**

| 维度 | 回归测试 | 验收测试 |
|------|---------|---------|
| 目的 | 验证修改未影响已有功能 | 验证系统满足用户需求 |
| 时机 | 每次代码变更后 | 版本交付前 |
| 执行者 | 测试团队 | 用户/业务方 |
| 依据 | 已有测试用例 | 业务场景和需求 |
| 侧重点 | 技术质量 | 业务适用性 |

**加分项 / 深度展开**

- 提到回归测试和验收测试的衔接：功能开发和Bug修复阶段 -> 测试团队做全量回归测试 -> 通过后 -> 业务方做UAT验收
