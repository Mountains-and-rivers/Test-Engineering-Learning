## 案例1：电商平台RESTful API测试项目（全流程）

### 1.1 项目背景

**项目名称**：某B2C电商平台V4.0版本接口测试项目

**业务规模**：
- 日活用户：500万+
- 日均订单量：30万+
- API接口总数：320+个（含商品、订单、用户、支付、物流、营销6大模块）
- 日均API调用量：2.8亿次
- 微服务数量：45个

**项目痛点**：
1. 版本迭代频繁（每2周一个迭代），接口回归测试周期长（5天）
2. 上下游服务依赖复杂，环境不稳定导致测试阻塞
3. 接口文档与实现不一致，联调阶段暴露大量问题
4. 缺乏统一的接口测试规范，各团队测试标准不统一

### 1.2 测试目标

| 目标维度 | 具体指标 | 衡量标准 |
|----------|----------|----------|
| **功能正确性** | 接口功能100%通过P0/P1用例 | 零P0缺陷遗留 |
| **接口覆盖率** | 接口级覆盖率100%，参数级覆盖率>=90% | 覆盖率报告 |
| **自动化效率** | 回归测试时间从5天缩短至4小时 | CI流水线执行时间 |
| **缺陷发现率** | 接口测试阶段发现缺陷占比>=60% | 缺陷统计报表 |
| **文档一致性** | 接口文档与实际实现一致率>=98% | Swagger校验结果 |

### 1.5 用例设计

#### 1.5.1 单接口用例设计方法（以订单创建接口为例）

**接口信息**：
```
POST /api/v2/orders
Content-Type: application/json
Authorization: Bearer {token}
```

**参数覆盖矩阵**（等价类 + 边界值 + 异常场景）：

| 参数名 | 类型 | 必填 | 等价类/边界值设计 |
|--------|------|------|-------------------|
| `sku_id` | Long | 是 | 有效ID / 不存在的ID(999999999) / 负数(-1) / 0 / 超长(9999999999999) / 空/null / 非数字类型("abc") |
| `quantity` | Integer | 是 | 正常范围(1-999) / 边界0 / 边界1000 / 超过库存 / 负数(-5) / 空/null / 字符串("five") |
| `address_id` | Long | 是 | 有效地址ID / 其他用户地址 / 已删除地址 / 0 / 空/null |
| `coupon_code` | String | 否 | 有效优惠券 / 已过期 / 已使用 / 不存在的编码 / 超长字符串(5000字符) / SQL注入("' OR '1'='1") |
| `remark` | String | 否 | 正常备注 / 超长(1001字符) / XSS脚本 / Emoji表情 / 特殊Unicode字符 |
| `pay_type` | String | 是 | 有效值(WECHAT/ALIPAY/BANK_CARD) / 无效值("BITCOIN") / 空/null / 大小写("wechat") |

**用例结构示例**：

```
用例ID: TC_ORDER_CREATE_001
用例名称: 验证使用有效参数创建订单-微信支付
前置条件: 
  1. 获取有效的用户Token（用户已登录且有收货地址）
  2. 准备有效的SKU_ID且库存充足(>5)
  3. 数据库初始订单数已记录
测试步骤:
  1. 构造请求体，包含所有必填有效参数
  2. POST /api/v2/orders 发送请求
  3. 断言HTTP状态码 = 201
  4. 断言响应体code = 200, message = "success"
  5. 断言响应体data.order_id 不为空且为数值
  6. 断言响应体data.status = "PENDING_PAYMENT"
  7. 断言响应体data.total_amount = quantity * sku单价 - 优惠券面额
  8. 数据库断言: orders表新增1条记录，status='PENDING_PAYMENT'
  9. 数据库断言: inventory表对应SKU库存减少quantity
后置清理:
  1. 取消创建的订单，恢复库存
```

#### 1.7.2 典型问题记录

**问题1：库存并发超卖**

```
问题描述: 并发下单时，库存扣减出现负数（超卖）
根因分析: 
  1. 订单创建接口读取库存和写入库存之间存在时间窗口
  2. 库存扣减操作未在数据库层面加行级锁
  3. 应用层未使用Redis分布式锁

定位过程:
  1. 使用JMeter并发20线程提交同一SKU订单（库存=10）
  2. 发现15个请求返回成功，数据库库存=-5
  3. 通过MySQL慢查询日志 + 应用日志定位到非原子操作

解决方案:
  1. 短期: 数据库UPDATE语句增加乐观锁 (WHERE stock >= quantity)
  2. 长期: 引入Redis预扣库存机制 + Lua脚本保证原子性
```

**问题2：金额精度丢失**

```
问题描述: 订单金额计算出现0.01元级误差
根因分析: 接口中使用Double类型存储和计算金额
定位过程: 
  测试数据: 商品单价 19.99元 x 数量3 + 运费8.50元 应=68.47元
  实际返回: 68.46999999999999元
  
  追踪代码发现:
  - 商品单价在Java中用BigDecimal，但优惠计算时转为double
  - 前端展示时toFixed(2)四舍五入掩盖了问题
  
解决方案: 全链路统一使用BigDecimal，金额计算统一在后端完成
```

## 案例2：金融系统接口自动化测试项目（含签名验签）

### 2.1 项目背景

**项目名称**：某互联网金融平台核心交易系统接口测试

**系统特点**：

- 所有API请求必须签名验签
- 监管合规要求（银保监会、央行支付清算要求）
- 系统间调用链长（用户端 → 交易系统 → 风控 → 银行网关 → 清算系统）
- 接口协议多样：REST + SOAP + 银行专有协议

**业务规模**：
- 日均交易笔数：50万+
- 日均交易金额：8亿+
- 对接银行渠道：15家
- 核心接口数：180+

### 2.2 测试目标

| 目标维度 | 具体指标 | 备注 |
|----------|----------|------|
| 功能正确性 | 100%覆盖P0用例，零资金类缺陷 | 最高优先级 |
| 安全合规 | 签名验签、加密传输、日志脱敏全部通过 | 监管要求 |
| 资金对账 | 日终对账差异率<0.001% | 业务红线 |
| 自动化覆盖 | P0/P1自动化率>=95% | 快速回归 |
| 异常容错 | 超时重试、熔断降级、幂等性验证 | 高可用 |

### 2.3 签名验签机制

#### 2.3.1 签名算法流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     API请求签名验签流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  客户端                                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 1. 准备请求参数（排除sign字段）                           │    │
│  │ 2. 对所有参数按Key字母序排序                              │    │
│  │ 3. 拼接为 key1=value1&key2=value2 格式                   │    │
│  │ 4. 拼接AppSecret: string + "&secret=" + AppSecret        │    │
│  │ 5. MD5/SHA256哈希 → sign                                 │    │
│  │ 6. 将sign放入请求体/请求头 → 发送请求                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                           │                                      │
│                           ▼                                      │
│  服务端                                                          │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 1. 接收请求，提取sign字段                                 │    │
│  │ 2. 使用相同算法重新计算sign'                              │    │
│  │ 3. 比较 sign == sign'  ? 通过 : 拒绝                      │    │
│  │ 4. 验证时间戳(防重放攻击): |server_time - req_time| < 5min │    │
│  │ 5. 验证Nonce(防重复请求): Nonce未被使用过                 │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.3.2 签名验证测试用例设计

```python
# 签名验证测试框架核心代码结构
class TestSignatureValidation:
    """签名验签测试套件"""
    
    def test_sign_with_valid_params(self):
        """TC_SIGN_001: 正确参数+正确签名 → 200"""
        
    def test_sign_with_wrong_secret(self):
        """TC_SIGN_002: 正确参数+错误Secret → 401 签名无效"""
        
    def test_sign_without_sign_field(self):
        """TC_SIGN_003: 缺少sign字段 → 400 参数缺失"""
        
    def test_sign_with_empty_sign(self):
        """TC_SIGN_004: sign值为空字符串 → 400"""
        
    def test_sign_with_expired_timestamp(self):
        """TC_SIGN_005: 时间戳过期(>5分钟) → 401 请求已过期"""
        
    def test_sign_with_future_timestamp(self):
        """TC_SIGN_006: 时间戳来自未来(>5分钟) → 401"""
        
    def test_sign_with_replayed_request(self):
        """TC_SIGN_007: 重放攻击(相同Nonce) → 401 重复请求"""
        
    def test_sign_with_parameter_tampered(self):
        """TC_SIGN_008: 参数被篡改(中间人攻击) → 401"""
        
    def test_sign_with_sorted_params(self):
        """TC_SIGN_009: 参数顺序不同但内容相同 → 200 (排序一致性)"""
```

#### 2.3.3 签名自动化核心代码

```python
import hashlib
import time
import uuid
from urllib.parse import urlencode

class SignHelper:
    """API签名工具类"""
    
    @staticmethod
    def generate_sign(params: dict, app_secret: str, sign_type: str = "MD5") -> str:
        """
        生成API请求签名
        
        算法步骤:
        1. 移除sign字段(如果存在)
        2. 按参数名ASCII码升序排序
        3. 拼接为 key1=value1&key2=value2 格式
        4. 追加 &secret=app_secret
        5. 计算摘要
        """
        # 移除sign字段
        sign_params = {k: v for k, v in params.items() if k != "sign"}
        
        # 按Key字典序排序并拼接
        sorted_keys = sorted(sign_params.keys())
        param_str = "&".join([f"{k}={sign_params[k]}" for k in sorted_keys])
        
        # 追加Secret
        raw_str = f"{param_str}&secret={app_secret}"
        
        # 计算签名
        if sign_type.upper() == "MD5":
            return hashlib.md5(raw_str.encode("utf-8")).hexdigest().upper()
        elif sign_type.upper() == "SHA256":
            return hashlib.sha256(raw_str.encode("utf-8")).hexdigest().upper()
        else:
            raise ValueError(f"Unsupported sign_type: {sign_type}")
    
    @staticmethod
    def build_signed_request(base_params: dict, app_id: str, app_secret: str) -> dict:
        """构建带签名的完整请求参数"""
        params = base_params.copy()
        params["app_id"] = app_id
        params["timestamp"] = str(int(time.time()))
        params["nonce"] = str(uuid.uuid4()).replace("-", "")[:16]
        params["sign_type"] = "MD5"
        params["sign"] = SignHelper.generate_sign(params, app_secret)
        return params
```

### 5.3 限流策略测试

#### 5.3.1 多维度限流配​置

```yaml
# 开放平台网关限流配置
rate_limits:
  # 维度1: 按AppKey限流 (每个ISV独立配额)
  - dimension: app_key
    rules:
      - app_key_level: "FREE"        # 免费ISV
        qps: 10
        daily_quota: 10000
      - app_key_level: "BASIC"       # 基础版ISV
        qps: 100
        daily_quota: 100000
      - app_key_level: "ENTERPRISE"  # 企业版ISV
        qps: 1000
        daily_quota: 1000000
      - app_key_level: "VIP"         # VIP ISV
        qps: 5000
        daily_quota: 10000000
        
  # 维度2: 按API限流 (保护后端服务)
  - dimension: api
    rules:
      - api_path: "/api/v3/products/search"
        max_qps: 5000           # 搜索接口全局QPS上限
      - api_path: "/api/v3/orders/create"
        max_qps: 2000           # 下单接口全局QPS上限
        
  # 维度3: 按用户限流 (防止单个用户刷接口)
  - dimension: user
    rules:
      max_qps_per_user: 50
      
  # 限流响应
  throttled_response:
    http_status: 429
    body:
      code: "RATE_LIMIT_EXCEEDED"
      message: "请求过于频繁，请稍后再试"
      retry_after: 60  # 秒
```
