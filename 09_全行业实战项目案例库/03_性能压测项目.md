## 案例1：电商大促全链路压测（双11/618）

### 1.1 项目背景

**项目名称**：某头部电商平台双11大促全链路压测

**业务规模**：
- 双11峰值QPS：85万/秒（零点峰值）
- 日均订单量（日常）：200万+
- 双11目标订单量：5000万+
- 核心链路：首页 → 商品详情 → 购物车 → 下单 → 支付 → 库存扣减
- 微服务数量：200+个
- 数据库实例：MySQL 50+实例，Redis 80+集群
- 接入层：CDN(200+节点) → SLB → Nginx(50台) → API网关(30台)

**压测目标**：

| 目标维度 | 具体指标 | 衡量标准 |
|----------|----------|----------|
| **容量验证** | 支撑85万QPS无降级 | 核心链路错误率<0.1% |
| **响应时间** | 核心接口P99<500ms | 下单/P99<1s，支付/P99<2s |
| **稳定性** | 持续30分钟峰值压力无抖动 | TPS波动<5% |
| **弹性伸缩** | 自动扩容响应<3分钟 | 新节点5分钟内加入服务 |
| **降级方案** | 预案有效性验证 | 限流/熔断/降级全部触发并恢复 |

#### 1.3.2 核心压测脚本 (JMeter JSR223)

```groovy
// JSR223 PreProcessor - 动态构造压测参数
import org.apache.commons.codec.digest.DigestUtils
import java.text.SimpleDateFormat

// 模拟用户行为分布
def userBehaviors = [
    "BROWSE_PRODUCTS",    // 浏览商品 35%
    "VIEW_DETAIL",        // 查看详情 20%
    "CART_ACTION",        // 购物车操作 15%
    "CHECK_COUPON",       // 查询优惠券 10%
    "CREATE_ORDER",       // 创建订单 8%
    "PAY_ORDER",          // 支付 7%
    "QUERY_LOGISTICS"     // 查询物流 5%
]

def random = new Random()
def rand = random.nextInt(100)

def behavior
if (rand < 35) behavior = "BROWSE_PRODUCTS"
else if (rand < 55) behavior = "VIEW_DETAIL"
else if (rand < 70) behavior = "CART_ACTION"
else if (rand < 80) behavior = "CHECK_COUPON"
else if (rand < 88) behavior = "CREATE_ORDER"
else if (rand < 95) behavior = "PAY_ORDER"
else behavior = "QUERY_LOGISTICS"

vars.put("behavior", behavior)

// 模拟用户ID池 (1000万虚拟用户)
def userId = random.nextInt(10000000) + 1
vars.put("userId", String.valueOf(userId))

// 模拟商品ID池 (100万热销商品)
def skuId = random.nextInt(1000000) + 100000
vars.put("skuId", String.valueOf(skuId))

// 生成幂等Key (防重复)
def idempotentKey = "${userId}_${System.currentTimeMillis()}_${random.nextInt(9999)}"
vars.put("idempotentKey", idempotentKey)

// 时间戳
def timestamp = String.valueOf(System.currentTimeMillis() / 1000 as long)
vars.put("timestamp", timestamp)

// 签名生成
def params = "userId=${userId}&skuId=${skuId}&timestamp=${timestamp}&secret=PRESSURE_TEST_KEY_2025"
def sign = DigestUtils.md5Hex(params).toUpperCase()
vars.put("sign", sign)
```

#### 1.5.2 典型问题与调优

**问题1：下单接口P99响应时间从200ms飙升至5s**

```
问题现象: 压测QPS达到5万时，下单接口RT急剧增长
定位过程:
  1. APM定位: 下单服务调用库存服务的Span耗时占比90%
  2. 库存服务: 发现大量"获取库存锁"等待
  3. 根因: 库存扣减使用数据库行锁，热点商品(前100个SKU)锁竞争严重
  
调优方案:
  方案1 (短期): 库存热点SKU拆分 - 将单条库存记录拆分为10个子库存
    UPDATE inventory SET stock = stock - ? 
    WHERE sku_id = ? AND sub_id = ? AND stock >= ?
    应用层Round-Robin选择子库存，分散锁竞争
  
  方案2 (中期): Redis预扣库存
    - 提前将库存加载到Redis (Lua脚本原子扣减)
    - 异步同步到MySQL
    - 失败回滚机制
  
  方案3 (长期): 库存服务独立部署 + 本地缓存
    - 库存热点数据本地缓存(Guava Cache)
    - 定时异步同步
  
调优效果:
  调优前: P99 = 5000ms @ 5万QPS
  调优后: P99 = 180ms @ 5万QPS (方案1+2)
         P99 = 85ms @ 5万QPS (方案1+2+3)
```

**问题2：支付接口偶发超时，错误率0.5%**

```
问题现象: 压测高峰期，支付接口偶发3秒超时
定位过程:
  1. 错误日志: SocketTimeoutException: Read timed out
  2. 支付服务调用银行网关耗时分布: P50=200ms, P99=2800ms, P99.9=3500ms
  3. 银行网关限流: 合作银行侧QPS限制为2000/s
  4. 我方发送QPS峰值达到2500/s，超限部分排队等待+超时

调优方案:
  方案1: 支付请求本地队列削峰
    - 本地内存队列缓冲(容量5000)
    - 控制实际发送速率 (令牌桶2000/s)
    - 超限请求排队等待(最多1s)
  
  方案2: 多银行渠道负载均衡
    - 支持按比例分发到不同银行渠道
    - 单渠道超限自动切换备用渠道
  
  方案3: 支付异步化
    - 用户发起支付 → 受理成功(同步返回) → 后台异步确认
    - 前端轮询获取最终支付结果

调优效果:
  支付超时率: 0.5% → 0.02%
  P99响应时间: 2.8s → 1.2s (同步) / 200ms (异步受理)
```

### 2.1 项目背景

**系统特点**：
- 核心交易系统：账户查询、转账、贷款审批、理财购买

## 案例4：API网关性能瓶颈定位与调优

### 4.1 项目背景

**项目名称**：某企业API网关(Kong)性能瓶颈分析与调优

**系统规模**：
- API网关节点：10台(8C16G)
- 后端路由数：800+个API
- 日均请求量：50亿+
- 峰值QPS：12万/秒
- 插件(TLS终止、限流、认证、日志)：6个全局插件

**问题现象**：
- 业务高峰期网关响应延迟增大(P99从30ms升至500ms)
- 部分请求返回502 Bad Gateway
- 网关CPU使用率飙升至95%

### 4.2 瓶颈定位过程

```
步骤1: 资源监控分析
┌─────────────────────────────────────────────────────────┐
│ 指标            │ 正常值    │ 高峰期值  │ 评估           │
├─────────────────────────────────────────────────────────┤
│ CPU使用率       │ 30%      │ 95%      │ ⚠️ 瓶颈        │
│ 内存使用率      │ 45%      │ 52%      │ ✓ 正常         │
│ 网络带宽        │ 200Mbps  │ 1.2Gbps  │ ⚠️ 接近上限    │
│ 磁盘IO          │ 低       │ 极高     │ ⚠️ 异常        │
└─────────────────────────────────────────────────────────┘

步骤2: 火焰图分析 (perf + FlameGraph)
发现热点函数:
- ssl_encrypt(): 35% CPU  ← TLS加解密消耗
- log_serialize(): 28% CPU  ← 日志序列化消耗
- rate_limit_lookup(): 15% CPU

步骤3: 插件性能逐项排查
关掉日志插件 → CPU降28%
关掉TLS → CPU降35%(不可行,安全要求)

步骤4: 根因确认
1. 日志插件同步写磁盘 → 磁盘IO瓶颈
2. TLS握手未启用会话复用 → 每次请求完整握手
3. 限流插件使用Redis单节点 → Redis连接瓶颈
```

### 4.3 调优方案与效果

| 优化项 | 调优前 | 调优方案 | 调优后 | 改善 |
|--------|--------|----------|--------|------|
| **日志写入** | 同步写磁盘 | 异步批量写 + 环形缓冲区 | CPU降28% | ⬇ 28% |
| **TLS会话复用** | 不复用 | session_tickets + session_cache | CPU降20% | ⬇ 20% |
| **限流Redis** | 单节点 | 集群模式 + 本地二级缓存 | RT降50ms | ⬇ 50ms |
| **连接池** | 默认128 | 调整至512，预创建连接 | 502减少 | ⬇ 95% |
| **Worker数** | 4 worker | 调整为8（匹配CPU核心数） | 吞吐提升 | ⬆ 40% |
| **综合效果** | P99=500ms | - | P99=45ms | ⬇ 91% |
