## 1. 需求评审常见问题

### 问题清单

| 编号 | 问题 | 问题描述 | 根因 | 改进建议 |
|------|------|----------|------|----------|
| **RQ-01** | 异常流程缺失 | PRD只写了正常流程，没有写"如果用户在这一步失败怎么办"。例如：只写了"用户点击支付→支付成功→跳转订单详情"，但用户余额不足、网络超时、支付取消等场景全丢给开发和测试自己脑补 | 产品只关注Happy Path，默认"异常情况开发自己会处理" | 需求评审时测试人员必须逐条追问"如果...怎么办"；制定"PRD异常场景描述模板"要求产品在每条主流程下写出至少3个异常分支 |
| **RQ-02** | 边界条件定义模糊 | "用户每天最多提现3次"——没说"每天"是按自然日0点算还是按24小时滚动窗口算；"最多上传9张图片"——没说是否包含GIF/动图 | 需求描述只写了数字限制，没写起算方式和范围 | 凡是涉及数字限制的需求，必须明确：起始点和结束点（按什么时间窗口？）、是否包含边界值、是否区分文件类型/格式 |
| **RQ-03** | 多个需求之间的冲突未被发现 | PRD-01说"新用户注册送100元优惠券"，PRD-02说"优惠券领取必须先实名认证"。但新用户注册流程中没有实名认证步骤，导致注册后无法领取优惠券 | 需求文档按功能模块分开写，缺少全局一致性检查 | 需求评审时应把所有相关需求放在一起对比审查；建立"需求冲突检查矩阵" |
| **RQ-04** | 可测试性差 | "系统应具备良好的性能"——什么样的性能算良好？怎么验证？谁来验证？"用户体验要好"——如何度量？ | 使用模糊的形容词而非可度量的指标 | 推动SMART需求标准：所有非功能需求必须有数字指标（如"99%的请求在200ms内响应"）；遇到不可测试的需求描述当场要求量化 |
| **RQ-05** | 缺失数据一致性要求 | PRD只说了"展示订单列表"，但没说订单状态变更后，列表中的状态是否要实时刷新？刷新是自动的还是手动下拉？延迟最大是多少？ | 产品不关注数据的实时性和一致性细节 | 需求评审时追问：数据变化后各个展示端多久更新？不一致时的兜底策略？ |
| **RQ-06** | 缺少权限和数据隔离说明 | "用户可以在列表页看到数据"——不同角色的用户看到的是同样的数据吗？部门经理能看到下属的数据还是全部数据？ | 权限设计被作为"技术细节"而在需求阶段被忽略 | 凡是涉及数据查看的需求，必须标注：什么角色看到什么范围的数据？哪些字段对哪些角色不可见？ |
| **RQ-07** | 逆向/撤销流程缺失 | PRD有创建的完整流程，但没有删除/撤销/退出的流程。如："创建一个审批规则"——没说能不能删除、删除后已审批的单据怎么处理 | 产品习惯只设计"正向"操作 | 测试人员应建立"正向/逆向/回滚"的思维，需求评审时主动追问每个正向操作对应的逆向操作 |
| **RQ-08** | 第三方交互细节缺失 | "调用第三方物流接口获取物流信息"——没说第三方接口超时怎么办？返回数据格式异常怎么办？调用频率有限制吗？ | 产品只关注业务逻辑，不关注技术集成细节 | 建立"第三方集成需求模板"，要求产品明确：接口SLA、超时策略、异常返回处理、频率限制 |
| **RQ-09** | 历史数据兼容性被忽略 | "V3.0版本的会员等级从三级改为五级"——V2.0的二级会员数据迁移到V3.0后对应哪个等级？历史订单上的会员等级显示什么？ | 新功能设计时只考虑增量，不考虑存量 | 凡是涉及数据结构变更的需求，必须附加"数据迁移方案"章节 |
| **RQ-10** | 需求粒度不一致 | 一个PRD中，有的功能写到"按钮颜色为蓝色"的粒度，有的功能只写"支持搜索"四个字。粒度差异导致开发和测试对需求理解严重不一致 | 多人协作写PRD，缺乏统一标准 | 建立PRD编写规范和粒度标准；关键功能必须包含：前置条件→操作步骤→业务规则→预期结果→异常处理 |
| **RQ-11** | 缺失国际化/多语言需求 | PRD在设计阶段只做了中文版，临近上线才被告知需要支持英文和日文。英文和日文的UI文案布局完全不同，导致大量返工 | 产品规划时未将国际化纳入范围 | 项目启动时确认是否需要多语言；即使暂时不需要，UI布局也应预留30%的文本扩展空间 |
| **RQ-12** | 缺少数据埋点和运营需求 | 产品上线后运营需要分析"优惠券使用率"，但领券按钮没有埋点，券的使用路径无法追踪 | 产品需求文档不包含运营和数据分析需求 | 需求评审时邀请运营/数据分析参与；将埋点需求作为需求的必填项 |
| **RQ-13** | 缺少验收标准 | PRD描述了一个完整功能，但验收标准只有一句话"功能正常可用"。什么算"正常可用"？哪些场景必须通过？ | 产品没有定义验收标准，或验收标准过于笼统 | 每个需求必须有可测试的验收标准；验收标准应由产品和测试共同制定 |
| **RQ-14** | 设计稿/交互稿与PRD不一致 | PRD写道"点击按钮弹出确认框"，但设计师给的交互稿是"点击按钮直接执行"。到底以谁为准？ | PRD和设计稿由不同人维护，缺乏同步机制 | 需求评审时必须同时核对PRD和设计稿；建立PRD-设计稿-验收标准三位一体的评审机制 |
| **RQ-15** | 需求变更未同步所有相关方 | 产品在开发中期口头跟开发说"把取消订单的时限从30分钟改成15分钟"。开发改了代码，但测试不知道、运营不知道、客服不知道 | 缺少需求变更管理流程 | 建立正式的需求变更流程：变更申请→影响评估→相关方确认→变更记录→测试用例同步更新 |

### 需求评审典型问题场景还原

**场景：一个"商品收藏"功能的PRD评审**

产品经理讲解完PRD后，测试人员逐一发问：

```
测试: "用户最多能收藏多少个商品？"
产品: "嗯...这个没想过，你觉得多少合适？"  ← RQ-02 边界模糊

测试: "收藏的商品如果下架了，收藏列表里怎么展示？"
产品: "这个...应该展示'已下架'吧，我再确认一下"  ← RQ-01 异常流程

测试: "商品加入收藏后，如果商品降价了，要不要通知用户？"
产品: "这个功能不在本迭代" 
测试: "那需要预留接口吗？"
产品: "..."  ← RQ-05 缺失扩展性考虑

测试: "用户把商品从收藏列表删除后，有没有撤销操作？"
产品: "没有，删除就是删了"
测试: "如果用户误删了呢？弹个确认框也好"
产品: "好，加个确认框"  ← RQ-07 逆向流程缺失
```

## 2. 设计评审常见问题

| 编号 | 问题 | 问题描述 | 根因 | 改进建议 |
|------|------|----------|------|----------|
| **DS-01** | 接口文档滞后或不完整 | 开发先写代码后补文档，测试看到的接口文档缺失了错误码定义、字段约束、异常返回格式。测试时只能"盲测"，遇到问题再去问开发 | 开发认为写文档"浪费时间"，或文档工具链不完善 | 推行"文档先行"或"契约先行"模式（如Swagger/OpenAPI先定义再编码）；接口文档纳入提测准入标准——无文档不提测 |
| **DS-02** | 数据库字段长度与业务需求不匹配 | PRD要求"收货地址最多200字符"，但数据库字段设了VARCHAR(50)，因为开发凭经验设的默认值。到测试时才发现用户输入长地址时报错 | 数据库设计和需求文档脱节 | 数据库设计评审时对照PRD逐字段核对长度和类型；建立"字段设计检查表" |
| **DS-03** | 缺少幂等性设计 | 支付接口没有做幂等处理，同一笔订单如果用户连续点击两次"支付"按钮（或客户端超时重试），可能被扣款两次 | 开发在正常流程下忽略了并发和重复场景 | 设计评审时测试人员重点审查：涉及资金/库存/计数的接口是否有幂等设计；接口是否支持幂等Key传入 |
| **DS-04** | 缺少熔断/降级/限流设计 | 下游服务不可用时，调用方没有熔断机制，导致线程池耗尽，整个系统雪崩 | 开发在"一切正常"的假设下设计，缺乏防御性设计思维 | 设计评审时要求给出：依赖服务不可用时的降级方案、高负载时的限流策略、级联故障的熔断机制 |
| **DS-05** | 缓存策略未考虑一致性 | 用户修改了个人信息，但因为Redis缓存了用户信息，页面展示的还是旧数据。缓存过期时间设为1小时，用户困惑"为什么改了没生效" | 开发只考虑了缓存的性能收益，忽略了数据一致性 | 设计评审时明确缓存的更新/失效策略（Cache-Aside / Write-Through / Write-Behind）；重点关注"数据变更时缓存如何同步" |
| **DS-06** | 定时任务设计未考虑幂等和重试 | 每天晚上2:00执行的对账任务，如果执行到一半数据库连接断了，任务重跑时会不会重复计算？会不会漏算？ | 只写了正常执行逻辑，没考虑中断恢复 | 设计评审时追问：任务中断后从哪继续？重复执行是否幂等？是否有防重跑机制（如任务锁）？ |
| **DS-07** | 分布式环境下的时钟不一致 | 订单服务记录的下单时间是服务器A的时间，支付服务记录的支付时间是服务器B的时间。如果两台服务器时间差5秒，可能出现"支付时间早于下单时间"的诡异数据 | 没有统一的时间同步机制 | 关键业务时间戳统一使用数据库时间（NOW()）或使用NTP时钟同步；设计评审时确认各服务的时间源 |
| **DS-08** | 接口版本兼容性未考虑 | App新版本接口增加了必填字段，但旧版本App不传该字段，导致旧版本App报错无法使用 | 接口升级时只考虑了向前兼容，没做向下兼容设计 | 接口设计原则：新增字段必须是选填且有默认值；不删除/不改名已有字段；通过版本号或Header做接口版本路由 |
| **DS-09** | 消息队列的消息丢失/重复 | 订单状态变更通过MQ通知下游，但设计中没有考虑MQ消息丢失（生产者发送失败）和消息重复（消费者重复消费）的处理 | MQ被当作"可靠传输"来用，忽略其"至少一次/至多一次"的语义 | 设计评审时明确MQ的可靠性等级：是否需要手动ACK？是否需要消费端幂等？死信队列如何处理？ |
| **DS-10** | 分页设计在大数据量下性能问题 | 列表查询使用OFFSET+LIMIT分页，数据量小的时候很快，但数据量到100万条时，OFFSET 900000 LIMIT 20需要扫描90万条无用数据 | 开发没有预估数据量增长，用了最简单的分页方案 | 大数据量场景使用游标分页（Cursor-based）或ES等搜索引擎；设计评审时要求给出不同数据量下的性能预估 |
| **DS-11** | 日志设计不当导致排查困难 | 所有异常都只记录了"系统错误"四个字，没有任何上下文信息。线上出问题时，运维看着日志无从下手 | 开发不重视日志的质量和内容 | 建立日志规范：请求必须有TraceID、异常必须记录堆栈+关键参数、业务关键节点必须记录INFO日志 |
| **DS-12** | 配置文件硬编码 | 数据库连接串、第三方API密钥直接写在代码里，每次环境变更需要改代码、重新编译、重新部署 | 缺少配置管理和环境隔离机制 | 使用配置中心（Nacos/Apollo）或环境变量注入；代码评审中硬编码配置视为阻塞项 |
| **DS-13** | 缺少灰度/功能开关设计 | 一个大功能上线后发现问题，因为没有功能开关，只能全量回滚（影响所有用户），而不能只关闭问题功能 | 功能设计时假设"上线了就全量可用"，未考虑灰度发布 | 核心功能和大功能设计时必须包含功能开关（Feature Flag）；开关的状态变更不需要重新部署 |
| **DS-14** | 自增ID作为对外暴露标识 | 订单ID使用数据库自增ID（1, 2, 3...），用户可以通过遍历ID看到其他人的订单（虽然做了权限校验，但暴露了订单总量和增长速度） | 开发图方便使用自增ID，未考虑信息泄露风险 | 对外暴露的标识使用UUID/Snowflake算法/自定义编码；数据库主键仅内部使用 |
| **DS-15** | 缺少数据归档和清理策略 | 日志表数据无限增长，运行半年后表有10亿条数据，导致查询越来越慢，磁盘空间告警 | 设计时没考虑数据的生命周期管理 | 数据设计时必须考虑：保留周期（如日志保留90天）、归档策略（冷数据迁到归档表/对象存储）、清理机制（定时任务自动清理） |

## 3. 用例评审常见问题

| 编号 | 问题 | 问题描述 | 根因 | 改进建议 |
|------|------|----------|------|----------|
| **TC-01** | 覆盖遗漏：重正常路径，轻异常路径 | 用例90%是正向Happy Path，异常场景只有寥寥几条。例如：订单模块30条用例，只有2条是"余额不足"和"网络超时" | 测试人员习惯按"功能怎么用"而非"功能怎么坏"来设计用例 | 建立"每个功能至少2条异常用例"的规则；用例评审时检查每个需求的异常覆盖比例 |
| **TC-02** | 预期结果描述模糊 | 预期结果写"系统正常处理"、"返回正确结果"——什么叫正常？什么叫正确？换了人执行用例就不知道怎么判断 | 测试人员自己知道预期结果，但没写清楚 | 预期结果必须包含：前端展示+接口返回码+数据库落库数据+（如有）通知/消息/日志 |
| **TC-03** | 测试数据无意义 | 测试数据全是"test"、"123"、"aaa"这种无实际意义的字符串，用这些数据找不到真实业务场景的问题 | 图省事随便填数据 | 测试数据应尽量模拟真实业务场景：使用真实的手机号格式、真实的金额、真实的中文姓名/地址 |
| **TC-04** | 用例间存在隐性依赖 | 用例A"创建订单"后，用例B"取消订单"依赖用例A创建的订单ID。当用例A执行失败，用例B也执行不了 | 用例设计时没有考虑独立性 | 每个用例应包含完整的前置数据准备步骤；或使用"数据工厂"统一管理测试数据，用例从数据池中获取而非依赖上一条用例 |
| **TC-05** | 缺少并发场景用例 | 用户模块20条用例，没有1条是关于"同一手机号并发注册"或"同一账号多端同时操作"的 | 并发测试被认为是专项测试而非功能测试的一部分 | 凡是涉及有限资源（库存/名额/优惠券）的功能，必须至少设计1条并发用例 |
| **TC-06** | 用例粒度差异大 | 有的用例一步式"测试整个注册流程"（太粗），有的用例原子级"输入手机号第1位"（太细）。同一份用例集里粒度不统一 | 没有统一的用例粒度标准 | 用例粒度标准：一个用例验证一个独立的业务场景；正常流程可以是"注册全流程"，边界值可以是"密码恰好6位" |
| **TC-07** | 缺少"空值/零值/null"场景 | 列表搜索功能的10条用例，没有1条是"搜索无结果时的空状态"；表单的用例也没有"不填任何内容直接提交" | 测试人员下意识认为"用户总会填点东西" | 建立"空状态必测"原则：每个功能至少包含1条空数据/无结果/无输入的用例 |
| **TC-08** | 边界值测试不完整 | 测试字段长度为6-20位，只测了6位和20位，没有测5位和21位。或者只测了一个边界值，以为就够了 | 边界值分析学得不扎实 | 边界值标准7点：min-1/min/min+1/normal/max-1/max/max+1；用例评审时检查边界值是否完整 |
| **TC-09** | 缺少数据持久化验证 | 用例只验证了"系统提示操作成功"，但没有去数据库里查数据是否真的落库了、字段值是否正确 | 测试人员只关注了"前端表现" | 凡是涉及数据增删改的操作，用例预期结果中必须包含数据库验证步骤 |
| **TC-10** | 用例评审沦为"一人讲众人听" | 评审会上，用例设计人一条一条念用例，其他人玩手机。1小时过去了，没提出几个有效问题 | 评审形式低效，参与者没有参与感 | 改"讲述式评审"为"挑战式评审"：评审前参与者提前阅读用例，会上每人必须提出至少3个问题或建议 |
| **TC-11** | 缺少跨系统/跨模块的端到端场景 | 订单模块的用例到"订单创建成功"就结束了，但订单数据还要流转到仓储系统、物流系统、财务系统。没有验证这些跨系统的数据流转 | 按模块分工后，每个人只关注自己负责的模块 | 分配专人负责端到端场景用例的设计；用例评审时检查关键业务流程是否全程贯通 |
| **TC-12** | 用例与需求版本不对应 | 需求V1.2已经改了"取消订单时限从30分钟改为15分钟"，但用例还是按V1.1的30分钟来设计的 | 需求变更后测试用例没同步更新 | 建立需求-用例双向追溯；需求变更时触发用例更新；用例评审时核对需求版本号 |
| **TC-13** | 缺少"回归影响"分析 | 修改了"用户登录"模块后，只评审了登录模块的新增用例，没有评审修改对"购物车"、"订单"、"支付"等依赖登录态的模块的影响 | 只关注变化的部分，忽略了变化的涟漪效应 | 用例评审时增加"变更影响分析"环节：改了什么→影响了哪些模块→这些模块的回归用例是否需要更新 |
| **TC-14** | 过度依赖自动化生成的用例 | 用工具自动生成的用例（如通过Swagger生成的接口用例），每条都只是"返回200"，没有验证返回数据的业务正确性 | 盲目信任工具，忘记了测试的本质是验证业务正确性 | 自动化生成的用例仅作为基础覆盖；必须手工补充业务验证用例 |
| **TC-15** | 用例维护性差 | 用例写成了"输入手机号13800138001"这种与具体数据强绑定。手机号被占用后，大量用例需要批量修改数据 | 用例编写时没考虑数据的可变性 | 使用变量/占位符管理测试数据；用例中使用逻辑描述（如"输入未注册的有效手机号"）而非具体数值 |

## 4. 上线评审常见遗漏

| 编号 | 问题 | 问题描述 | 根因 | 改进建议 |
|------|------|----------|------|----------|
| **RL-01** | 未确认生产配置与测试环境的一致性 | 测试环境Redis连接数是100，生产环境是50。上线后高并发下Redis连接池耗尽，系统响应超时 | 上线评审时只关注了代码，忽略了配置 | 上线评审必须包含"配置对比"环节：逐项对比测试/预发布/生产环境的配置差异，解释每个差异的原因 |
| **RL-02** | 未通知第三方/合作方 | 上线后，因为API接口的字段名修改了，调用该接口的合作方系统报错，造成业务中断 | 内部系统上线时忽略了对外部依赖方的影响 | 建立"外部依赖方清单"；上线评审时确认所有依赖方是否获知变更并做好准备 |
| **RL-03** | 未准备回滚方案或回滚方案不可行 | 上线前说"有问题就回滚"，但真出问题后：数据库迁移脚本没有回滚脚本、前端CDN缓存无法及时刷新、回滚需要停服2小时用户无法接受 | 回滚方案只是口头说说，没有实际验证 | 上线评审时必须展示回滚方案的验证结果：回滚步骤、负责人、预估时间、是否演练过 |
| **RL-04** | 新建数据库索引未评估对生产的影响 | 新功能需要新建数据库索引，测试环境建索引用了2分钟（数据量小），但生产有10亿条数据，建索引需要2小时，期间表被锁，业务中断 | 测试环境数据量与生产差距太大，无法真实反映操作影响 | 涉及数据库结构变更的上线，必须在评审时给出生产数据量下的耗时估算和锁表风险评估 |
| **RL-05** | 定时任务开关状态未确认 | 新功能上线后需要关闭旧的定时任务、开启新的定时任务。但上线时忘了切换，导致新旧两个定时任务同时运行，数据被重复处理 | 定时任务的启停开关不在代码中而依赖人工操作，容易遗漏 | 定时任务纳入配置管理或代码管理（跟随版本自动上线）；上线操作清单中必须包含定时任务启停确认项 |
| **RL-06** | 缓存未预热/清理 | 新版本上线后，Redis缓存中还是旧版本格式的数据，导致新版代码读取缓存时反序列化失败报错 | 上线操作中遗漏了缓存处理步骤 | 上线方案必须包含缓存策略：哪些缓存需要清理？哪些需要预热？预热需要多长时间？ |
| **RL-07** | 监控告警规则未更新 | 新功能上线后，新增的接口没有配置监控，新增的错误码没有配置告警。测试人员在灰度期间只能手动刷页面看是否正常 | 监控配置工作被视为"运维的事"而延迟到上线后 | 监控告警纳入测试交付物清单；上线评审时检查：新增接口是否配置了监控大盘？新增错误码是否配置了告警？ |
| **RL-08** | 灰度策略不清晰 | 上线时说"先灰度10%用户"，但没定义"什么是灰度成功"——灰度10分钟没报错就算成功？还是10小时？还是需要对比灰度组和非灰度组的业务指标？ | 灰度发布变成了形式化流程 | 上线评审时明确灰度成功标准：观察多久？关键指标是什么？什么信号触发全量/回滚？ |
| **RL-09** | 客服/运营未获得新功能培训 | 新功能上线后，用户打客服电话咨询"这个新功能怎么用"，客服表示"我也不知道有这个功能" | 上线通知范围局限于技术团队 | 建立"上线通知Checklist"：客服培训完成、FAQ就绪、运营话术下发、公告模板确认 |
| **RL-10** | 文件/MQ/定时任务的上线顺序错误 | 上线涉及的发布顺序应该是：数据库→后端→前端→定时任务→配置变更，但实际顺序搞反了，先上线前端再上线后端，导致用户看到新界面但接口报404 | 没有出详细的上线操作顺序清单 | 上线方案必须明确发布顺序和依赖关系；评审时逐步推演，检查是否有循环依赖 |
| **RL-11** | 上线后未立刻执行生产冒烟 | 代码上线完成，开发说"搞定了，应该没问题"，所有人就散了。1小时后用户反馈核心功能异常，才发现问题 | 没有建立"上线后必须立即验证"的纪律 | 上线评审时确认"生产冒烟用例清单"和负责人；上线完成后30分钟内必须完成生产冒烟并汇报结果 |
| **RL-12** | 未备份关键数据 | 上线涉及历史数据清洗（如删除无效订单），但清洗前忘了备份。发现误删了部分有效订单，因无备份无法恢复 | 对数据操作的风险意识不足 | 涉及数据修改/删除的上线操作，操作前必须备份，且备份恢复需验证可用 |
| **RL-13** | 未考虑多机房/多集群的上线节奏 | 系统部署在北京、上海两个机房。北京机房上线后，两个机房的代码版本不一致。用户在北京机房下单，在上海机房查询订单状态（因负载均衡），状态不一致 | 双机房部署时未考虑新旧版本并行窗口 | 两机房上线间隔时间应尽量短（最好同时）；如无法同时，需验证新旧版本的前后兼容性 |
| **RL-14** | 上线时间窗口选择不当 | 周五下午5点上线，上线后发现小问题，但考虑到下班时间，团队决定"周一再说"。结果整个周末用户都在用有问题的版本 | 没有遵循"不在关键时刻上线"的原则 | 上线时间选择原则：避免周五/节假日前一天上线；留足至少1个工作日的观察窗口 |
| **RL-15** | 未做A/B测试和业务指标对比 | 新功能上线后产品问"效果怎么样"，没有数据可以回答。因为没有埋点，没有对照组，无法评估新功能对业务指标的影响 | 数据分析和效果评估不在上线评审范围内 | 涉及体验优化/转化率提升的功能，上线前需确认：埋点是否就绪？对照组是否设置？评估指标是否定义？ |

## 5. 评审会议效率提升建议

### 5.1 评审会议的常见低效问题

| 问题 | 典型表现 | 根本原因 |
|------|----------|----------|
| **会议超长** | 一个需求评审开了3小时，大家精疲力竭 | 被评审的材料太多或太复杂，没做会前阅读 |
| **讨论发散** | 本来在评审登录功能，突然开始讨论"要不要做微信登录"，又拐到了"微信登录的资质审核需要多久" | 缺少议程控制，主持人没有及时制止跑题 |
| **缺乏准备** | 会上大家才第一次看需求文档，一边看一边提问，效率极低 | 评审材料没有提前分发，参会者没有预读 |
| **参与者层次不对** | 该来决策的人没来，来了的人做不了决定 | 未明确评审会需要谁参加、什么角色 |
| **问题无跟踪** | 会上提了很多问题，会后没人记录，没人跟进。下次评审时同样的问题再次出现 | 没有会议纪要，没有问题追踪机制 |

### 5.2 高效评审会议的10条黄金法则

**法则1：提前分发材料（至少提前24小时）**
- 评审组织者必须将被评审的材料（PRD/设计文档/用例）提前发送给所有参会者
- 要求参会者在会前阅读并标注意见和建议
- 会前未阅读者，评审会有权拒绝其参会

**法则2：明确会议目标和议程**
```
【评审会议议程模板】
1. 会议目标: (1分钟) — 本次评审要达成什么结论
2. 范围说明: (2分钟) — 本次评审覆盖哪些内容，不覆盖哪些内容
3. 上下文介绍: (5分钟) — 项目背景、设计思路概述
4. 逐项评审: (核心时间) — 按检查清单逐项过
5. 问题汇总: (5分钟) — 汇总所有提出的问题和建议
6. 结论与行动项: (5分钟) — 评审结论(通过/有条件通过/不通过) + 待办分配
```

**法则3：设定时间盒（Timebox）**
- 需求评审: 单次不超过90分钟（超时分段）
- 用例评审: 单次不超过60分钟（每30条用例约30分钟）
- 上线评审: 单次不超过45分钟
- 超时的部分另约时间，不拖堂

**法则4：区分"评审"和"讨论"**
- 评审是"找问题"，讨论是"解决问题"
- 评审会上发现问题 → 如果30秒内不能给出结论 → 记录为"待讨论项" → 会后再专题讨论
- 禁止评审会上花30分钟讨论一个细节

**法则5：主持人控制节奏**
- 主持人必须敢于打断跑题的讨论："这个问题我们记下来，会后再深入讨论"
- 主持人控制每位参会者的发言时间，避免一两个人占用所有时间
- 主持人确保每个章节都在预定时间内完成

**法则6：使用检查清单而非即兴发挥**
- 评审按预定义的检查清单逐条检查，不以个人经验随机发挥
- 检查清单覆盖了所有该检查的点，避免遗漏
- 检查清单提高了评审的一致性和可复现性

**法则7：记录所有问题并跟踪闭环**
```
【评审问题跟踪表】

┌────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ ID │ 问题描述  │ 提出人    │ 严重程度 │ 责任人    │ 状态     │
├────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ 1  │ 缺少退款  │ 张三     │ 阻塞     │ 产品:李四 │ ✅已解决  │
│    │ 流程设计  │          │          │          │          │
│ 2  │ 并发场景  │ 王五     │ 严重     │ 开发:赵六 │ ⏳处理中  │
│    │ 未覆盖    │          │          │          │          │
│ 3  │ 字段长度  │ 张三     │ 一般     │ 开发:赵六 │ ⬜待处理  │
│    │ 未定义    │          │          │          │          │
└────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

**法则8：确保对的人在对的时间参加**
- **必须全程参加**：决策者（能做决定的人）和核心执行者
- **只需参加相关章节**：领域专家（如安全评审时安全工程师参与）
- **不需要参加**：只需知情但无需决策的人（看会议纪要即可）
- 如果关键决策人无法参加 → 改期，不要在没有决策者的情况下开评审会

**法则9：评审结论必须明确**
- 评审结论只有三种：✅通过、⚠️有条件通过（列明条件）、❌不通过（列明原因）
- 不允许"差不多可以了"、"应该没问题"这种模糊结论
- 有条件通过时，必须明确谁在什么时间点之前完成什么条件

**法则10：定期复盘评审效果**
- 每季度回顾：评审发现了多少问题？这些问题的严重程度？如果没有评审会漏掉多少？
- 根据数据调整评审的投入：如果某个评审长期发现不了有价值的问题，可能是该评审环节可以精简或优化
- 收集参会者反馈：评审会议是否高效？材料是否有用？形式是否合理？

### 5.3 评审文化建议

**建立"对事不对人"的评审文化：**
- 评审的目的是找问题，不是找责任人
- "这个需求的异常流程没写清楚" 而非 "你怎么又忘了写异常流程"
- 设计者/作者应真诚感谢每一个提出问题的人："谢谢你发现了这个，避免了后续返工"

**评审不是"找茬"而是"协作"：**
- 问题不是某个人的问题，而是团队的问题
- 评审会上的每一个问题，都是帮团队节省成本（避免返工 + 避免线上事故）

**心理安全高于流程：**
- 即使是最严格的检查清单，也比不上一个"敢提问题"的团队氛围
- 鼓励初级人员大胆提问——他们看到的可能是"内行"已经麻木的问题

### 5.4 不同项目阶段的评审投入建议

| 项目阶段 | 评审重点 | 建议投入 | 参与人 |
|----------|----------|----------|--------|
| **高不确定性/创新项目** | 需求评审+设计评审 各投入充分时间 | 评审时间占比20%-25% | 全体+领域专家 |
| **成熟产品/稳定迭代** | 侧重上线评审+用例评审 | 评审时间占比10%-15% | 核心成员 |
| **外包项目** | 需求评审+验收评审 重点投入 | 评审时间占比15%-20% | 甲方+乙方负责人 |
| **紧急修复/Hotfix** | 上线评审必做（简化版），其他评审可跳过 | 快速评审30分钟 | 开发+测试+运维 |

## 6. 各角色评审参与指南

### 6.1 测试人员在各类评审中的角色定位

| 评审类型 | 测试人员角色 | 核心任务 | 典型发言 |
|----------|-------------|----------|----------|
| **需求评审** | 挑战者+补齐者 | 挖掘异常场景边界、追问不可测试的描述、发现需求矛盾 | "如果用户在这一步断网了怎么办？" |
| **设计评审** | 防御者+验证者 | 从"哪里可能出Bug"角度审查设计、验证可测试性 | "这个并发场景下会不会出现数据不一致？" |
| **代码评审** | 补充者 | 从常见Bug模式审查代码、发现遗漏的边界处理 | "这个catch块是空的，异常被吞掉了" |
| **用例评审** | 被评审者(提交者)或审查者 | 作为提交者：开放接受反馈；作为审查者：检查覆盖率和正确性 | "这个预期结果没有数据库验证，加上去会更完整" |
| **上线评审** | 守门员 | 逐项确认上线条件、坚持质量标准、敢说"不" | "P1缺陷还没修复，我不同意上线" |

### 6.2 开发人员在评审中的常见误区

| 误区 | 表现 | 正确做法 |
|------|------|----------|
| "需求评审是产品的事，我不用管" | 需求评审不参加或不发言，到开发时发现问题再返工 | 需求评审是开发理解需求的最佳时机，有疑问当场提出比事后返工高效得多 |
| "设计文档写了就够，不需要评审" | 自己写完设计文档直接开始编码 | 再厉害的架构师也会有盲区，设计评审是降低技术风险的最便宜方式 |
| "测试用例评审是测试的事" | 不参加用例评审，等测试提了Bug才发现理解有偏差 | 用例评审可以帮助开发理解测试的覆盖范围和深度，提前发现理解偏差 |
| "代码评审就是检查代码规范" | 代码评审只关注缩进、命名等表面问题 | 代码评审的核心是逻辑正确性、边界处理、异常处理和安全问题 |

### 6.3 产品经理在评审中的常见误区

| 误区 | 表现 | 正确做法 |
|------|------|----------|
| "PRD写了就行，评审就是走个过场" | 评审会上被动回答，不主动挖掘需求盲区 | 需求评审是提升需求质量的最后机会，主动引导技术团队挖掘隐含需求 |
| "技术细节我不懂，你们决定" | 遇到技术实现方案的问题就回避 | 产品应该参与技术方案讨论——技术方案的选择会影响用户体验和功能扩展性 |
| "改了需求口头通知就行" | 需求变更不走正式流程 | 哪怕是一个字段的修改，也要走变更流程：记录→评估影响→通知相关方→更新文档 |

## 7. 评审度量指标与效果评估

### 7.1 评审投入产出度量

| 度量指标 | 计算公式 | 参考目标值 | 说明 |
|----------|----------|-----------|------|
| **评审缺陷发现率** | 评审发现的缺陷数 / (评审发现+测试发现+线上发现) | ≥50% | 越高说明评审越有效 |
| **评审效率** | 评审发现的缺陷数 / 评审投入的人时 | ≥2个/人时 | 衡量评审的ROI |
| **评审覆盖率** | 实际评审的文档/代码行数 / 应评审的总数 | ≥80% | 避免评审点到为止 |
| **评审遗漏率** | 评审后下游发现的缺陷数 / 总缺陷数 | ≤30% | 越低越好，反映评审的全面性 |
| **缺陷严重度分布** | 评审发现的严重缺陷占比 | ≥30% | 如果评审发现的都是小问题，说明评审深度不够 |

### 7.2 评审质量健康度红绿灯

```
【评审质量仪表盘】

指标                  绿灯(优)      黄灯(预警)    红灯(危险)
─────────────────────────────────────────────────────────
评审缺陷发现率       >60%         40%-60%       <40%
评审效率(个/人时)    >3            1.5-3         <1.5
评审遗漏率           <15%         15%-30%       >30%
评审参与率           >90%         70%-90%       <70%
问题闭环率           >95%         80%-95%       <80%

行动建议:
- 绿灯: 维持当前评审机制
- 黄灯: 分析短板项，针对性优化
- 红灯: 评审机制需要全面review，可能有流程性问题
```

### 7.3 如何判断评审是否"走过场"

| 走过场信号 | 具体表现 | 改进方法 |
|------------|----------|----------|
| 评审总是"一致通过" | 连续5次评审都没有提出任何需要修改的问题 | 可能是评审深度不够或评审文化有问题；引入外部交叉评审 |
| 评审后Bug率没有下降 | 评审投入了大量时间，但测试阶段发现的Bug密度没有降低趋势 | 审视评审方法是否对路——可能审了不该审的，没审该审的重点 |
| 同样的Bug反复出现 | 某类缺陷在评审中被指出，但后续版本中仍然出现 | 问题跟踪机制失效：评审发现的问题没有沉淀为检查清单或编码规范 |
| 评审会的参与度下降 | 参会者越来越少，或者越来越多人不发言 | 会议效率有问题——可能是会议太长、内容太泛、或参与者感受不到价值 |

## 8. 评审工具与方法推荐

### 8.1 不同评审类型的工具推荐

| 评审类型 | 推荐工具 | 用途 |
|----------|----------|------|
| **需求评审** | Confluence/Wiki + 内联批注 | 异步评审：评审者在文档上直接添加批注，作者统一回复 |
| **设计评审** | Draw.io/PlantUML + 评审会议 | 架构图、流程图可视化评审 |
| **接口评审** | Swagger Editor/YApi/Apifox | 接口文档在线评审，可直接调试验证 |
| **用例评审** | XMind脑图 + Excel/TestLink + 腾讯文档 | 脑图做快速评审，Excel/ALM做详细追溯 |
| **代码评审** | GitLab MR/PR Review / Gerrit / Crucible | 在线代码diff审阅，行级批注 |
| **上线评审** | 在线Checklist工具 + 会议记录 | 逐项核对，自动记录确认结果 |

### 8.2 异步评审 vs 同步评审的选择

| 维度 | 异步评审（离线） | 同步评审（会议） |
|------|-----------------|-----------------|
| **适用场景** | 文档已经比较成熟，参与者可以自行阅读理解 | 文档有较多不确定点，需要实时讨论 |
| **时间效率** | 高（各自利用碎片时间） | 低（需要整块时间所有人在一起） |
| **讨论深度** | 受限（异步沟通容易产生误解） | 深入（可即时追问和澄清） |
| **参与度** | 取决于个人自律 | 强制参与 |
| **推荐组合** | 异步预审（会前阅读+标注问题）→同步会议（只讨论标注的问题，不再从头读文档） |

### 8.3 评审清单的持续优化方法

```
1. 定期收集评审数据
   - 每次评审后记录：发现了什么类型的问题？用了什么检查项发现的？
   
2. 识别高价值检查项
   - 统计每个检查项发现的缺陷数量和严重程度
   - 高缺陷发现率的检查项 → 保留并强化
   - 连续多次零发现的检查项 → 评估是否可精简

3. 补充新检查项
   - 将测试阶段和线上阶段发现的、评审时遗漏的问题补充为新检查项
   - 每个线上Bug都要做回溯：这个Bug在哪个评审环节应该被发现？为什么漏了？
   
4. 定期清理
   - 每季度review一次检查清单
   - 删除过时的、不适用于当前技术栈的检查项
   - 保持检查清单精简高效（不宜超过50条）
```

## 9. 评审问题深度分析：高频根因与系统性改进

### 9.1 评审问题背后的四大系统性根因

| 根因 | 说明 | 导致的典型问题 | 系统性改进方向 |
|------|------|---------------|---------------|
| **时间压力** | 项目排期紧张，评审被压缩或跳过 | 需求评审不充分、用例评审走过场 | 将评审时间明确纳入项目计划；评审不计入"可压缩"范围 |
| **角色鸿沟** | 产品、开发、测试各自视角孤立，互相不理解对方的关注点 | 需求不完整（产品不懂技术约束）、设计不可测（开发不懂测试需求） | 推行"三驾马车"协作模式；轮岗体验（测试参与需求讨论，开发参与测试设计） |
| **知识断层** | 新人对业务/技术不熟，资深人员的隐性知识没有沉淀 | 用例覆盖遗漏、评审依赖"老人"、人员流动后质量滑坡 | 建立知识库和检查清单；新人导师制；评审发现的问题写入检查清单 |
| **文化问题** | 评审被认为是"找茬"或"形式主义"，缺乏心理安全 | 评审时不敢提尖锐问题、被提问题的人防卫心态 | 领导示范"欢迎找问题"；评审结论归团队不归个人；物质激励"发现最多问题的人" |

### 9.2 建立"评审-测试-线上"三层问题闭环

```
线上发现的问题
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 回溯分析：这个Bug在哪个环节应该被发现？            │
│                                              │
│ 应该被需求评审发现？ → 补充需求评审检查项           │
│ 应该被设计评审发现？ → 补充设计评审检查项           │
│ 应该被用例评审发现？ → 补充用例设计规范             │
│ 应该被代码评审发现？ → 补充代码评审检查项           │
│ 应该被测试发现？     → 补充测试策略/用例            │
│ 谁都发现不了？       → 补充到经验库，新型风险      │
└─────────────────────────────────────────────────┘
    │
    ▼
将改进项加入对应的检查清单 → 下次评审生效
```

---

> **总结**：评审是软件质量保障体系中"投入产出比最高"的活动之一——花1小时在评审上发现的缺陷，可能节省测试阶段10小时、上线后100小时的修复成本。但是，评审的效果不是看"开了多少会"，而是看"发现了多少有价值的问题"和"问题是否被真正解决"。

> **三条核心原则：**
> 1. **评审越早，成本越低**——需求阶段的1个问题，如果漏到上线后，修复成本可能翻100倍
> 2. **检查清单是经验的凝结**——不要靠记忆，要靠系统；每个线上Bug都应该"回馈"到检查清单
> 3. **评审文化决定评审效果**——最完美的检查清单也比不上一个"敢提问题、不怕冲突"的团队氛围

> **评审的质量，决定了产品的质量下限。**
