# 评审检查清单详全指南

> **评审**（Review）：通过结构化的检查活动，在软件开发的各阶段尽早发现问题和缺陷。评审的核心价值在于"缺陷发现得越早，修复成本越低"。据IBM研究数据：需求阶段发现的缺陷修复成本为1，编码阶段为6.5，测试阶段为15，上线后为60-100。

## 1. 需求评审检查清单

### 通俗释义

需求评审的目标是在需求定稿前找出模糊、矛盾、遗漏和不合理的地方。如果需求有歧义，后面所有环节都会跑偏。

### 详细解释

**需求评审的六大维度：**

| 维度 | 检查重点 | 典型问题 |
|------|----------|----------|
| **完整性** | 需求是否覆盖了所有场景 | 只写了正常流程，没说异常流程怎么处理 |
| **明确性** | 需求描述是否清晰无歧义 | "系统应快速响应"——快到什么程度？ |
| **可测试性** | 需求是否可验证、可度量 | "用户体验良好"——如何判断是否达标？ |
| **一致性** | 需求之间是否矛盾冲突 | PRD说"最多上传9张图"，交互稿却显示"支持批量上传" |
| **必要性** | 需求是否能解决真实问题 | 这个功能用户真的需要吗？ROI如何？ |
| **可行性** | 技术、资源、时间上是否可实现 | 技术方案可行吗？时间窗口够吗？ |

### 需求评审检查清单（完整版）

```
【需求评审检查清单】

一、功能需求检查（Functional Requirements）

□ 1. 正常业务流程
   □ 1.1 所有正常业务流程是否完整描述？
   □ 1.2 每个步骤的输入和输出是否明确？
   □ 1.3 业务规则（如折扣规则、优惠叠加规则）是否清晰定义？

□ 2. 异常场景处理
   □ 2.1 输入非法数据时系统的行为是否定义？
   □ 2.2 系统内部异常（如数据库失败、网络超时）的处理方式？
   □ 2.3 "为空"状态（无数据、无搜索结果）的界面表现是否定义？
   □ 2.4 操作失败后的用户引导（重试/联系客服/返回）是否明确？
   □ 2.5 并发操作（如两人同时编辑同一数据）的冲突处理策略？

□ 3. 边界条件
   □ 3.1 所有具有范围限制的字段，边界值是否明确定义？
   □ 3.2 列表/分页的最大数量限制？
   □ 3.3 文件上传的大小、数量、格式限制？
   □ 3.4 时间相关的边界（活动开始/结束、优惠券有效期）是否明确？

□ 4. 状态与生命周期
   □ 4.1 业务对象（订单/工单/申请单）的所有状态是否完整列出？
   □ 4.2 每个状态的进入条件和退出条件是否明确？
   □ 4.3 自动状态转换（如超时自动取消）的条件和时间节点？
   □ 4.4 状态的回退/撤销机制？

□ 5. 权限与角色
   □ 5.1 涉及哪些用户角色？
   □ 5.2 每个角色对功能的访问权限是什么（查看/编辑/删除/审批）？
   □ 5.3 不同角色的数据可见范围（数据权限）？

二、非功能需求检查（Non-Functional Requirements）

□ 6. 性能需求
   □ 6.1 页面/接口的响应时间要求（P95/P99）？
   □ 6.2 系统需要支撑的并发用户数？
   □ 6.3 数据量的预期（日增/总量）？
   □ 6.4 是否有性能基线数据？

□ 7. 安全需求
   □ 7.1 哪些数据需要加密传输和存储？
   □ 7.2 认证和授权机制的明确？
   □ 7.3 敏感操作（支付/权限变更/数据删除）的审计日志要求？
   □ 7.4 是否涉及个人信息保护/数据合规要求？

□ 8. 兼容性需求
   □ 8.1 需要支持的操作系统/浏览器/设备列表？
   □ 8.2 最低支持版本是什么？不支持时的降级策略？

□ 9. 易用性需求
   □ 9.1 错误提示文案是否明确且对用户友好？
   □ 9.2 关键操作的引导和帮助文档需求？
   □ 9.3 无障碍访问需求（无障碍标准WAI）？

□ 10. 可维护性与监控
   □ 10.1 关键指标（如注册量、支付成功率）的监控告警需求？
   □ 10.2 日志记录的范围和格式要求？
   □ 10.3 功能开关/灰度发布的需求？

三、测试视角专项检查

□ 11. 可测试性
   □ 11.1 每个需求是否有明确的可验证标准？
   □ 11.2 需求中是否包含测试所需的信息（输入、输出、规则）？
   □ 11.3 是否有不可测试的描述（如"稳定可靠"、"性能优良"）需要量化？

□ 12. 依赖与接口
   □ 12.1 需求涉及哪些上下游系统/接口？
   □ 12.2 第三方接口的文档/联调环境是否就绪？
   □ 12.3 数据格式和协议是否明确定义？

□ 13. 需求变更痕迹
   □ 13.1 需求变更是否有版本标记和变更说明？
   □ 13.2 变更影响的功能范围是否评估？
```

### 需求评审中测试人员常见提问清单

| 序号 | 典型提问 | 目的 |
|------|----------|------|
| 1 | "如果用户在这一步断网了，重新进入后应该看到什么状态？" | 挖掘异常场景 |
| 2 | "这个列表为空的时候，界面应该怎么展示？" | 明确空状态设计 |
| 3 | "这个字段最多能填多少字符？超了怎么提示？" | 明确输入边界 |
| 4 | "这个优惠券能和会员折扣叠加吗？如果可以，计算顺序是什么？" | 澄清业务规则 |
| 5 | "删除这个数据后，关联的其他数据怎么处理？" | 明确数据一致性规则 |
| 6 | "第一次操作和第二十次操作有没有区别？" | 挖掘隐藏的频次限制 |
| 7 | "不同角色看到的内容有什么不同？" | 明确权限和数据隔离 |
| 8 | "超时后系统自动操作还是等待用户手动操作？" | 明确超时处理策略 |
| 9 | "老用户从旧版本升级到新版本，数据迁移规则是什么？" | 关注兼容性和数据迁移 |
| 10 | "这个需求的验收标准是什么？上线后怎么判断是否成功？" | 明确可度量标准 |

## 2. 设计评审检查清单

### 通俗释义

设计评审的目标是在代码写之前找出架构、接口、数据模型等方面的问题。测试人员参与设计评审，重点是从"可能在哪里出Bug"的角度审查设计。

### 详细解释

**设计评审的三个层面：**

| 层面 | 关注点 | 审查对象 |
|------|--------|----------|
| **架构设计** | 系统分层、模块划分、技术选型 | 架构设计文档、技术方案 |
| **详细设计** | 接口定义、数据模型、类设计 | 接口文档、数据库设计文档 |
| **交互设计** | 页面布局、操作流程、交互逻辑 | 原型图、交互稿 |

### 设计评审检查清单（测试视角）

```
【设计评审检查清单（测试视角）】

一、架构设计检查

□ 1. 系统分层
   □ 1.1 分层是否清晰？职责边界是否明确？
   □ 1.2 各层之间的依赖关系是否合理？
   □ 1.3 是否便于分层测试（可单独测试某一层）？

□ 2. 模块划分
   □ 2.1 模块间的耦合度是否可控？
   □ 2.2 一个模块的修改是否会影响其他模块？
   □ 2.3 核心业务模块是否独立，是否便于隔离测试？

□ 3. 技术选型
   □ 3.1 新技术/组件是否有充分的验证和测试？
   □ 3.2 技术选型是否引入了已知的稳定性或安全风险？
   □ 3.3 是否有对应的测试工具支持？

二、接口设计检查

□ 4. 接口定义
   □ 4.1 所有接口的请求/响应格式是否完整定义？
   □ 4.2 字段类型、长度、是否必填、默认值是否明确？
   □ 4.3 错误码和错误信息格式是否统一？
   □ 4.4 接口版本管理策略是否清晰？

□ 5. 接口异常处理
   □ 5.1 接口超时设置和超时后的处理逻辑？
   □ 5.2 接口调用失败的重试策略（次数、间隔、退避算法）？
   □ 5.3 下游接口不可用时的降级/熔断方案？
   □ 5.4 幂等性设计（防止重复提交/重复扣款）？

□ 6. 接口安全
   □ 6.1 接口是否有认证和授权机制？
   □ 6.2 敏感数据的传输是否加密？
   □ 6.3 是否有防SQL注入/XSS/CSRF的设计？
   □ 6.4 是否有接口限流/防刷机制？

三、数据模型设计检查

□ 7. 数据库设计
   □ 7.1 表结构是否合理？是否有冗余或缺失字段？
   □ 7.2 字段长度是否与实际业务数据匹配？
   □ 7.3 索引设计是否合理（既满足查询需求又不冗余）？
   □ 7.4 主键/外键约束是否正确定义？

□ 8. 数据一致性与完整性
   □ 8.1 跨表/跨库的数据一致性如何保证？
   □ 8.2 金额、库存等关键数据的并发更新策略？
   □ 8.3 数据的软删除与物理删除策略？
   □ 8.4 历史数据的归档和清理策略？

□ 9. 数据迁移
   □ 9.1 是否有数据迁移脚本（表结构变更+数据清洗）？
   □ 9.2 迁移过程中的业务可用性（是否需停服）？
   □ 9.3 迁移失败的回滚方案？

四、交互设计检查（UI/UX）

□ 10. 页面交互
   □ 10.1 所有可点击元素的状态（正常/悬停/点击/禁用）是否完整？
   □ 10.2 异步操作（如文件上传）的加载状态是否有提示？
   □ 10.3 表单校验的触发时机（失焦/提交/实时）？
   □ 10.4 二次确认（如删除、支付）是否有拦截？

□ 11. 多端一致性
   □ 11.1 Web/iOS/Android的交互是否一致？
   □ 11.2 不同分辨率/屏幕尺寸下的布局是否适配？
   □ 11.3 横屏/竖屏切换的适配？

五、非功能设计检查

□ 12. 性能设计
   □ 12.1 是否有缓存策略？缓存失效/更新的机制？
   □ 12.2 是否有数据分页/分片/懒加载设计？
   □ 12.3 是否有异步处理方案（消息队列/定时任务）？

□ 13. 日志与监控
   □ 13.1 关键业务操作是否有日志记录？
   □ 13.2 日志格式是否统一、是否便于检索？
   □ 13.3 是否有异常监控和告警设计？

□ 14. 可测试性
   □ 14.1 设计是否便于自动化测试（接口标准化、UI元素可定位）？
   □ 14.2 是否有测试桩/Mock的预留接口？
   □ 14.3 是否有功能开关/配置开关便于灰度测试？
```

### 设计评审中测试人员的经典提问

| 序号 | 提问 | 发现的潜藏风险 |
|------|------|---------------|
| 1 | "订单状态从'待发货'变为'已发货'时，库存是否已经冻结和扣减？这两步在同一个事务里吗？" | 数据一致性、事务边界 |
| 2 | "定时任务执行到一半如果服务重启了，下次启动是从断点继续还是从头开始？" | 幂等性、容错性 |
| 3 | "这个接口的QPS上限是多少？超过上限后会直接拒绝还是排队？" | 性能上限、限流策略 |
| 4 | "用户A和用户B同时修改同一条数据，后提交的是覆盖还是冲突提示？" | 并发冲突处理 |
| 5 | "这个字段存VARCHAR(100)，但是如果用户填了100个中文和100个英文，存储差异很大，是否考虑了字符编码的影响？" | 边界值、字符编码 |

## 3. 测试用例评审检查清单

### 通俗释义

用例评审是最重要的同行互查活动——一个人设计的用例容易有盲区，多个人一起审查能发现覆盖遗漏、逻辑跳跃、数据不充分等问题。

### 详细解释

**用例评审的五个维度：**

| 维度 | 检查内容 | 占比权重 |
|------|----------|----------|
| **覆盖率** | 需求覆盖是否完整，有无遗漏场景 | 35% |
| **正确性** | 预期结果是否正确，测试逻辑是否合理 | 25% |
| **可执行性** | 前置条件、步骤、数据是否明确可操作 | 20% |
| **高效性** | 用例是否精简，有无冗余或过度设计 | 10% |
| **规范性** | 格式、命名、优先级标注是否符合团队标准 | 10% |

### 测试用例评审检查清单

```
【测试用例评审检查清单】

一、覆盖完整性检查（最重要）

□ 1. 需求覆盖
   □ 1.1 每条需求是否都有对应的用例？（对照RTM追溯矩阵）
   □ 1.2 需求中的所有业务规则是否都有验证用例？
   □ 1.3 PRD中的"备注/例外说明"是否容易被遗漏？

□ 2. 功能覆盖
   □ 2.1 正常流程（Happy Path）：所有成功路径是否覆盖？
   □ 2.2 异常流程：各类异常场景是否覆盖（数据异常、系统异常、网络异常）？
   □ 2.3 边界值：所有输入/输出的边界是否验证？
   □ 2.4 状态转换：所有合法状态转换是否覆盖？非法跳转是否拦截？

□ 3. 场景覆盖
   □ 3.1 端到端业务场景是否覆盖（不仅是孤立的单点功能）？
   □ 3.2 多角色协作场景（如员工提交→领导审批→HR复核）是否覆盖？
   □ 3.3 跨模块联动场景（如购物车→下单→支付→订单查询）是否覆盖？

□ 4. 非功能覆盖
   □ 4.1 性能场景是否包含？（如大量数据时列表加载、高并发下单）
   □ 4.2 安全场景是否包含？（如SQL注入、越权访问、敏感数据泄露）
   □ 4.3 兼容性场景是否包含？（不同浏览器/设备/分辨率）

□ 5. 隐性需求覆盖
   □ 5.1 是否考虑了用户可能的"误操作"路径？
   □ 5.2 是否考虑了并发场景（同时操作时的互相影响）？
   □ 5.3 是否考虑了数据量变化（空→有→多→海量）的梯度？
   □ 5.4 空状态、首次使用、老用户等不同用户旅程是否覆盖？

二、用例正确性检查

□ 6. 测试逻辑
   □ 6.1 测试步骤是否与需求规定的业务流程一致？
   □ 6.2 预期结果是否与需求描述完全匹配？
   □ 6.3 业务规则的计算逻辑是否正确（如折扣、税费计算）？

□ 7. 测试数据
   □ 7.1 测试数据是否真实可信（不使用123、test、aaa等无意义数据）？
   □ 7.2 临界值数据是否精确（如0.01、100.00而非"小额"、"大额"）？
   □ 7.3 测试数据是否覆盖了不同等价类？

□ 8. 预期结果
   □ 8.1 预期结果是否明确、可判定（不是"系统正常"这种模糊表述）？
   □ 8.2 是否包含了前端展示+后端接口+数据库落库的三层验证？
   □ 8.3 异常场景的预期结果（错误码、提示语、数据回滚）是否明确？

三、可执行性检查

□ 9. 前置条件
   □ 9.1 前置条件是否明确可复现？（含具体的账号、状态、数据）
   □ 9.2 前置条件是否依赖其他用例的执行结果？（应尽量避免）

□ 10. 操作步骤
   □ 10.1 步骤是否粒度合适（不太粗"测试注册"也不太细"输入1→输入3→输入8"）？
   □ 10.2 步骤是否顺序正确且无跳跃？
   □ 10.3 步骤中的定位描述是否清晰（具体的按钮文字、页面名称、输入框标签）？

四、高效性检查

□ 11. 冗余检查
   □ 11.1 是否存在重复/高度相似的用例？（可合并）
   □ 11.2 是否可以用正交表减少组合用例？
   □ 11.3 边界值是否和等价类划分合理配合，避免过度设计？

□ 12. 优先级
   □ 12.1 每个用例的优先级(P0/P1/P2/P3)标注是否合理？
   □ 12.2 P0用例是否覆盖了核心流程和阻塞性缺陷场景？
   □ 12.3 是否可以仅通过P0+P1用例就能判断版本是否"可测"？
```

## 4. 测试计划评审检查清单

### 通俗释义

测试计划评审确保测试活动"想清楚了再干"——范围对不对、排期能不能完成、风险有没有应对、资源够不够。

### 测试计划评审检查清单

```
【测试计划评审检查清单】

一、范围与目标

□ 1. 测试范围是否与项目范围一致？
```

## 5. 代码评审检查清单（测试视角）

### 通俗释义

测试人员参与代码评审不是为了替代开发审查代码逻辑，而是从"测试可能会在哪里找到Bug"的角度进行补充审查。

### 代码评审检查清单（测试视角）

```
【代码评审检查清单（测试视角）】

一、输入校验

□ 1. 接口参数校验
   □ 1.1 Controller/Handler层是否有对请求参数的校验？
   □ 1.2 必填参数是否有@NotNull/@NotEmpty等注解验证？
   □ 1.3 参数长度、范围是否有验证（不是只靠前端校验）？
   □ 1.4 枚举类型参数是否只接受合法值？

□ 2. 边界处理
   □ 2.1 循环/数组操作是否有越界检查？
   □ 2.2 数值运算是否有溢出检查？
   □ 2.3 分页参数（pageNum/pageSize）是否有合理范围限制？

二、异常处理

□ 3. 异常捕获
   □ 3.1 是否有空的catch块（吞掉异常不处理也不记录）？
   □ 3.2 异常信息是否清晰（包含上下文，不只是"系统错误"）？
   □ 3.3 是否区分了可恢复异常和不可恢复异常？

□ 4. 资源管理
   □ 4.1 数据库连接/文件流/网络连接是否在finally中关闭？
   □ 4.2 线程池/连接池是否合理配置？

三、业务逻辑

□ 5. 条件判断
   □ 5.1 if/else分支是否完整覆盖？是否遗漏else默认处理？
   □ 5.2 switch语句是否有default分支？
   □ 5.3 ">"和">="、"<"和"<="是否正确？(最常见的bug来源)

□ 6. 状态管理
   □ 6.1 状态转换是否符合业务规则（有没有非法的状态跳转路径）？
   □ 6.2 状态变更时关联数据是否同步更新？

□ 7. 金额/精度
   □ 7.1 金额计算是否使用BigDecimal而非float/double？
   □ 7.2 除法运算是否指定了舍入模式（RoundingMode）？
   □ 7.3 是否考虑了负金额/零金额的场景？

四、安全相关

□ 8. 注入防护
   □ 8.1 SQL查询是否使用参数化查询（PreparedStatement）？
   □ 8.2 用户输入的文本在返回前端时是否做了HTML转义（防XSS）？
   □ 8.3 文件上传是否校验了文件类型（不仅是后缀名，还要检查文件头Magic Number）？

□ 9. 权限与认证
   □ 9.1 每个接口是否有权限验证（不是只靠前端隐藏按钮）？
   □ 9.2 数据查询是否过滤了数据权限（A用户不能看到B用户的数据）？
   □ 9.3 敏感操作（删除/支付/权限变更）是否有二次确认？

五、可测试性

□ 10. 测试友好性
   □ 10.1 核心业务逻辑是否可以从框架层独立出来单独测试？
   □ 10.2 依赖的外部服务是否可以注入Mock？
   □ 10.3 关键方法的可见性是否过于私有导致无法测试？

□ 11. 日志与调试
   □ 11.1 关键业务流程是否有日志记录（含请求ID便于追踪）？
   □ 11.2 日志级别使用是否合理（INFO记录业务，DEBUG记录细节）？
   □ 11.3 异常日志是否包含堆栈信息便于定位？

六、代码变更特殊检查

□ 12. 变更影响
   □ 12.1 公共方法的修改是否影响所有调用方？
   □ 12.2 SQL/数据库变更是否影响历史数据？
   □ 12.3 接口字段变更是否兼容旧版本客户端？
```

## 7. 评审实战案例

### 案例1：需求评审 — 外卖配送超时赔付需求

**需求摘要**：订单超时10分钟以上，自动赔付用户5元优惠券；超时30分钟以上赔付10元优惠券。

**评审中发现的问题清单：**

| 序号 | 问题描述 | 问题类型 | 评审结论 |
|------|----------|----------|----------|
| 1 | "订单超时"的起算时间未明确：按下单时间算还是商家接单时间算？还是骑手取餐时间算？ | 明确性 | 明确为"预计送达时间"为基准 |
| 2 | 用户取消订单后重新下单，超时基准是否重新计算？ | 完整性 | 重新计算，以最新订单为准 |
| 3 | 恶劣天气、交通管制导致的超时是否赔付？ | 边界条件 | 纳入免责条款，不可抗力不赔付 |
| 4 | 系统时间以服务器时间为准还是用户手机时间为准？ | 技术细节 | 以服务器时间为准 |
| 5 | 赔付的优惠券有效期和使用门槛未说明 | 完整性 | 补全：优惠券有效期7天、无使用门槛 |
| 6 | 同一订单是否可能重复赔付？（定时任务重复执行） | 异常场景 | 增加幂等校验，同一订单最多赔付1次 |

**评审效果**：多发现了6个问题，其中问题5和问题6如果遗漏，上线后会导致用户投诉和资金风险。

---

### 案例2：设计评审 — 秒杀系统设计方案

**方案摘要**：采用Redis库存预扣 + 消息队列异步下单的架构。

**测试人员在设计评审中的提问与发现：**

| 序号 | 测试提问 | 发现的设计漏洞 | 改进措施 |
|------|----------|---------------|----------|
| 1 | "Redis库存扣减成功后，如果消息队列发送失败，订单没创建但库存已经扣了，这个存货就'消失'了？" | 缺少消息发送失败的回滚机制 | 增加Redis扣减记录的状态标记：预扣→已确认/已回滚，定时扫描超时未确认的预扣记录自动回滚 |
| 2 | "秒杀开始时Redis的库存是1000件，实际数据库库存也是1000件，如果因为运营配置错误导致Redis和数据库初始值不一致怎么办？" | 缺少初始库存的校验机制 | 增加库存初始化校验：Redis初始库存和数据库库存必须一致，不一致时告警并阻断秒杀启动 |
| 3 | "用户秒杀成功后，如果支付超时30分钟，库存要释放，这期间的库存状态怎么管理？" | 库存生命周期不完整，可能"丢失" | 补充库存状态机：可用→预扣→已售/已释放，每个状态转换都有日志记录 |

**评审效果**：3个问题都是设计层面的严重漏洞，如果在代码层面才发现，修复成本会高很多。

### 案例3：用例评审 — 用户优惠券模块

**评审背景**：测试工程师张三设计了一份包含45条用例的优惠券测试用例集，提交给测试负责人李四评审。

**评审发现的问题分类统计：**

| 问题类别 | 数量 | 具体表现 |
|----------|------|----------|
| **覆盖遗漏** | 5个 | 1.缺少"优惠券过期后自动释放"的场景 2.缺少"同一商品同时满足多个优惠券条件时选最优"的场景 3.缺少"退货时优惠券是否退回"的场景 4.缺少"跨境/虚拟商品不能用优惠券"的校验 5.缺少"并发领取最后一张优惠券"的并发场景 |
| **预期结果不准确** | 3个 | 1.优惠券叠加后的金额计算错误（先折扣券再满减券，顺序写反了） 2.优惠券使用后积分计算，漏了"优惠后金额"作为积分计算基数 3.包邮券+满减券叠加时，邮费计算逻辑不准确 |
| **用例冗余** | 2个 | 1.多个"无效优惠券码"的用例可以合并为1条等价类用例 2.领券入口(商品详情/领券中心/活动页)的测试用例高度重复 |
| **数据不合理** | 1个 | 测试数据使用"满10000减5000"这种不符合实际业务的夸张数据 |

**评审后改进**：用例从45条优化到52条（增加7条覆盖遗漏、合并2条冗余、修正3条错误）。

### 案例4：代码评审 — 订单退款接口

**评审背景**：开发提交了订单退款接口的代码，测试人员参与CR。

**测试视角发现的6个问题：**

```java
// 问题1: 没有校验订单状态——已退款的订单还能再退款！
public void refund(Long orderId, BigDecimal amount) {
    Order order = orderDao.findById(orderId);
    // 缺少: if (order.getStatus() != OrderStatus.PAID) throw Exception
    
// 问题2: 退款金额用的是BigDecimal但构造方式有精度风险
    BigDecimal refundAmount = new BigDecimal(amount);  // 不应该用double构造
    
// 问题3: 退款和恢复库存不在同一事务中
    paymentService.refund(orderId, amount);  // 退款成功
    inventoryService.restore(orderId);       // 如果这步失败，钱退了库存没恢复！
    
// 问题4: 日志没有包含退款金额，出问题难追溯
    log.info("退款成功: orderId={}", orderId);
    // 应该是: log.info("退款成功: orderId={}, amount={}", orderId, amount);
    
// 问题5: 没有做幂等处理，同样的退款请求可能发起多次
    // 缺少基于orderId的幂等校验
    
// 问题6: 退款失败的异常被吞掉了
    try {
        paymentService.refund(orderId, amount);
    } catch (Exception e) {
        // 空catch块，异常被吞
    }
}
```

> **总结**：评审不是"找茬"，而是"协作降低风险"。好的评审文化是：**提出问题的人被感谢，被提出问题的人不防卫**。评审检查清单只是工具，真正起作用的是团队的评审态度和专业素养。
