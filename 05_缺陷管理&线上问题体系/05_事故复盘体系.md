# 事故复盘体系 (Incident Review & Postmortem System)

> **事故复盘的唯一目标是学习与改进，而非追责。** 每一次事故都是系统送给你的一面镜子，让你看到之前不知道的盲区。建立规范的事故复盘体系，可以让团队从"恐惧事故"转变为"拥抱学习"。

## 1. 事故等级定义

### 1.1 P0/P1/P2/P3四级标准

| 等级 | 名称 | 定义 | 响应时间 | 恢复时间 | 典型场景 |
|------|------|------|----------|----------|----------|
| **P0** | 特大事故 | 核心业务完全不可用，造成重大经济损失或品牌声誉严重受损 | 5分钟内 | 30分钟内 | 全站宕机、支付系统全线瘫痪、重大数据泄露 |
| **P1** | 重大事故 | 核心业务部分不可用或严重降级，影响大量用户 | 15分钟内 | 2小时内 | 某个核心功能不可用、支付成功率下降50%、大范围数据不一致 |
| **P2** | 一般事故 | 非核心功能不可用，或核心功能轻微降级影响少量用户 | 1小时内 | 8小时内 | 非核心模块不可用、小范围性能下降、管理后台不可用 |
| **P3** | 轻微事故 | 非核心功能轻微异常，用户感知弱，影响极小 | 4小时内 | 24小时内 | 个别用户偶发报错、非关键页面异常、次要功能间歇性异常 |

### 1.2 P0事故详细判定标准

**满足以下任一条件即为P0：**

1. **全站/核心链路完全不可用**
   - 网站/App首页无法访问超过5分钟
   - 登录/注册/支付/下单任一核心链路成功率<10%超过5分钟
   - CDN/负载均衡全局故障

2. **重大经济损失**
   - 预估直接经济损失 > X万元（根据公司规模设定）
   - 持续1小时以上影响核心交易

3. **重大数据安全事件**
   - 用户核心隐私数据（密码/身份证/银行卡号）批量泄露（>1000条）
   - 数据库被删除/篡改且无有效备份可恢复
   - 核心数据大面积丢失

4. **重大合规/法律风险**
   - 违反法律法规（如数据跨境传输禁令）
   - 触发监管处罚风险的事件

5. **重大舆情事件**
   - 社会媒体大规模负面传播（微博热搜/媒体报道）
   - 引起行业广泛关注的安全漏洞公开

### 1.3 P1事故详细判定标准

**满足以下任一条件即为P1：**

1. **核心功能严重降级**
   - 单一核心功能（如搜索/商品详情/购物车）成功率<50%
   - 支付成功率下降>30%
   - 核心接口P99延迟>正常值5倍

2. **大范围用户影响**
   - 影响用户比例 > 20%（或绝对数 > X万）
   - 某个地域/运营商/设备类型的用户全部受影响

3. **数据一致性问题**
   - 缓存与数据库大范围不一致
   - 订单状态/库存数量出现批量偏差
   - 用户账户余额异常

4. **持续时间长**
   - P2级事故持续超过8小时自动升级为P1

### 1.4 P2事故详细判定标准

**满足以下任一条件即为P2：**

1. **非核心功能不可用**
   - 管理后台/运营工具不可用
   - 非核心页面报错
   - 辅助功能（如收藏/评论/分享）不可用

2. **少量用户影响**
   - 影响用户比例 < 20%
   - 限特定版本/特定设备/特定网络环境

3. **性能轻微降级**
   - 非核心接口P99延迟上升但未超SLA
   - 页面加载时间增加<3秒

### 1.5 P3事故详细判定标准

**满足以下任一条件即为P3：**

1. **偶发轻微异常**
   - 个别用户偶发报错（影响<1%用户）
   - 非关键数据展示异常
   - 自动恢复的短暂波动

2. **预发布/灰度阶段发现**
   - 在灰度发布阶段发现且已止损的异常
   - 在预发布环境发现的生产潜在风险

### 1.6 事故等级快速判定矩阵

```
               影响范围
           小(<5%)  中(5-20%)  大(>20%)
功能影响    │        │         │
───────────┼────────┼─────────┼─────────
核心不可用 │  P1    │   P1    │   P0
核心降级   │  P2    │   P1    │   P1
非核心不可用│  P3    │   P2    │   P2
非核心降级 │  P3    │   P3    │   P2
轻微异常   │  P3    │   P3    │   P3

附加升级规则:
  + 造成经济损失>阈值 → 自动升级一级
  + 涉及数据安全 → 最小P1
  + 媒体/舆情发酵 → 自动升级为P0
  + 持续时间 > SLA → 自动升级一级
```

## 2. 事故处理流程

### 2.1 事故全生命周期流程

```
═══════════════════════════════════════════════════════════
                  事故处理全流程
═══════════════════════════════════════════════════════════

Phase 1: 发现 (Detection) ──── 时间目标: <5分钟
  │
  ├── 来源: 监控告警 / 用户反馈 / 内部发现 / 外部通知
  ├── 动作: 确认事故真实性 → 初步判断等级 → 通知应急团队
  └── 产出: 事故初始通告 (what/where/when/impact)

Phase 2: 响应 (Response) ──── 时间目标: <15分钟 (P0)
  │
  ├── 动作: 应急团队到位 → 建立沟通渠道(应急群/War Room) → 明确指挥链
  ├── 动作: 开始记录时间线(Timeline)
  └── 产出: 应急团队就位，信息通道建立

Phase 3: 止损 (Mitigation) ──── 时间目标: <30分钟 (P0)
  │
  ├── 原则: 先止损后排查！
  ├── 动作: 回滚/降级/限流/切换/下线/扩容
  ├── 动作: 通知相关方(客服/运营/管理层/用户)
  └── 产出: 影响范围被控制住，不再扩大

Phase 4: 根因分析 (Root Cause Analysis) ──── 时间目标: <24小时
  │
  ├── 动作: 收集日志/监控/配置变更记录
  ├── 动作: 5-Why分析 / 鱼骨图 / FTA
  ├── 动作: 确认直接原因 + 根本原因
  └── 产出: 根因分析结果

Phase 5: 修复 (Fix) ──── 根据等级SLA
  │
  ├── 动作: 制定修复方案 → Code Review → 测试验证 → 灰度发布
  ├── 动作: 确认修复不会引入新问题
  └── 产出: 修复代码上线，事故恢复

Phase 6: 复盘 (Postmortem) ──── 时间目标: <3个工作日
  │
  ├── 动作: 召开复盘会议 → 输出复盘报告 → 制定改进措施
  ├── 动作: 改进措施录入跟踪系统
  └── 产出: 事故复盘报告 + Action Items

Phase 7: 改进 (Improvement) ──── 持续跟踪
  │
  ├── 动作: 执行改进措施 → 验证效果 → 关闭Action Item
  ├── 动作: 经验分享(团队/公司级别)
  └── 产出: 改进措施完成，知识沉淀
```

### 2.2 事故处理角色与职责

| 角色 | 职责 | 谁担任 |
|------|------|--------|
| **事故指挥官 (Incident Commander, IC)** | 统筹协调，做出决策，对外沟通 | 当天值班的技术负责人 |
| **处理负责人 (Resolver)** | 负责技术排查和修复 | 相关模块的开发/运维 |
| **沟通负责人 (Communicator)** | 对内对外信息同步，维护状态页面 | 项目经理/技术PM |
| **记录员 (Scribe)** | 记录时间线和关键决策 | 轮值或指定的测试/开发 |

**角色原则：**
- 事故指挥官(IC)不亲自操作排查——保持全局视野
- IC可根据需要交接（如事故跨系统，需要换领域专家）
- 记录员实时记录，不参与排查

### 2.3 止损策略工具箱

| 策略 | 说明 | 适用场景 | 恢复速度 |
|------|------|----------|----------|
| **回滚(Rollback)** | 回退到上一个稳定版本 | 新版本引入Bug | 快（分钟级） |
| **降级(Degrade)** | 关闭非核心功能，保障核心链路 | 部分服务压力大 | 快（开关控制） |
| **限流(Rate Limiting)** | 限制请求速率 | 流量突增/攻击 | 快（配置生效） |
| **切换(Failover)** | 切换到备用集群/机房 | 单个机房/集群故障 | 中（DNS/负载均衡切换） |
| **扩容(Scale Up/Out)** | 增加资源 | 容量不足 | 中（分钟-小时） |
| **下线(Take Offline)** | 下线故障功能 | 某个功能无法快速修复 | 快（开关控制） |
| **数据修复(Data Fix)** | 执行数据修复脚本 | 数据错误 | 慢（需验证） |
| **手动补偿(Manual Compensation)** | 人工处理积压/异常 | 自动化流程中断 | 中 |

### 2.4 事故沟通规范

**沟通时间线：**

```
时间点              沟通内容                      接收方
─────────────────────────────────────────────────────
发现后5分钟内        初始通告(事故概要+影响+预计处理)  技术群/管理层
发现后15分钟内       进展更新(排查进展+确定等级)      同上 + 受影响团队
止损后               止损通告(影响已控制+当前状态)    同上 + 客服/运营
每30分钟             持续更新(如未恢复)              所有相关方
恢复后               恢复通告(事故结束+影响总结)      全公司(视等级)
复盘后               复盘报告                       技术团队+管理层
```

**事故通告模板：**

```
【事故通告】{等级} - {简要描述}

发生时间: YYYY-MM-DD HH:MM
事故等级: P0/P1/P2/P3
影响范围: {受影响的功能/用户/地域}
当前状态: {排查中/止损中/已恢复}
处理团队: {负责人+团队}
预计恢复: {预计时间}
下次更新: {预计下次通报时间}

详细描述:
{简要描述事故现象和已知信息}

用户影响:
{用户能看到什么异常}

当前行动:
{正在做什么}

---
事故ID: INC-YYYY-NNNN
Command Center: {会议链接/群聊链接}
```

## 4. 事故改进措施跟踪

### 4.1 改进措施分类体系

```
改进措施分类

├── 预防类 (Preventive)
│   ├── Code Review检查清单更新

## 5. 无指责文化 (Blameless Postmortem)

### 5.1 为什么需要无指责文化

**指责文化的后果：**
```
发现问题 → 寻找责任人 → 追责/处罚
    │
    ▼
团队恐惧 → 隐藏事故/拖延上报 → 小事故变成大事故
    │
    ▼
不报/晚报/瞒报 → 错失止损窗口 → 更大损失
```

**无指责文化的效果：**
```
发现问题 → 快速上报 → 立即止损 → 深入复盘 → 系统改进
    │
    ▼
团队信任 → 主动暴露问题 → 及早发现 → 持续改善
```

### 5.2 无指责文化的核心原则

**原则1：系统而非个人**

99%的事故根因是系统/流程缺陷，而非个人的恶意或无能。如果一个人"犯了错误"，应该问的是"为什么系统允许这个错误发生"。

```
错误示范: "张三忘记设置超时，导致服务雪崩"
正确示范: "系统缺少统一的超时配置规范，本次事故中依赖人工设置导致了遗漏"
```

**原则2：接受"人都会犯错"**

人不是机器，会疲劳、会分心、会有知识盲区。优秀的事故复盘假定操作者当时已经做到了他们的最好，然后问"我们如何让下一次的'最好'更容易达成"。

**原则3：聚焦学习而非追责**

复盘会议不是"审判庭"，是"学习会"。目标是"我们学到了什么"，而非"谁该背锅"。

**原则4：领导层以身作则**

- 当事故涉及管理层决策时，管理层主动承担责任
- 领导者分享自己的错误经历和学到的教训
- 对主动上报问题的人表示赞赏

**原则5：信息透明**

- 事故报告对全公司公开（涉密信息除外）
- 鼓励跨团队学习和讨论
- 公开"差一点发生的事故(Near Miss)"同样重要

### 5.3 无指责语言指南

| 避免使用的语言 | 推荐使用的语言 |
|---------------|---------------|
| "XX做错了..." | "这个操作导致了..." |
| "XX忘记..." | "XX步骤没有发生..." |
| "人为失误" | "流程设计使得这个操作容易出错" |
| "XX没有考虑到..." | "预案中没有覆盖这个场景" |
| "XX应该..." (指责语气) | "将来我们可以..." |
| "为什么XX没有..." | "什么条件导致了..." |

### 5.4 心理安全建设清单

```
□ 事故复盘报告底部标注"本报告不用于绩效考核"

□ 复盘会议开始时声明: 
  "本次会议的焦点是系统如何改进，而非评价个人表现"

□ 对主动上报事故/风险的员工给予公开表扬

□ 事故复盘不邀请HR（除非涉及重大违规）

□ 管理层的错误决策同样纳入复盘范围

□ 设立"优秀事故报告"奖励——写得最好的复盘报告值得表彰

□ 定期回顾团队是否因为之前的事故而进步——这是最好的正向激励
```

## 6. 事故复盘实战案例（5个）

### 案例1：全站504超时 —— 第三方CDN故障 P0

**事故概要：**
某天14:30起，网站和App陆续出现大面积504 Gateway Timeout。持续2小时15分钟，影响全部用户。

**时间线：**

| 时间 | 事件 |
|------|------|
| 14:30 | 监控告警：网站可用性下降至65% |
| 14:32 | 值班人员确认事故，拉应急群 |
| 14:35 | 初步排查：Nginx日志大量 upstream timed out |
| 14:40 | 发现所有经过CDN的请求超时，直接回源正常 |
| 14:45 | 联系CDN厂商，确认CDN厂商骨干网故障 |
| 14:50 | 执行止损：DNS切换，绕过CDN直连源站 |
| 15:20 | DNS全球生效，服务恢复正常 |
| 16:45 | CDN厂商确认其上游运营商光缆被施工挖断 |

**5-Why分析：**

```
Why 1: 为什么网站504？
  → CDN节点回源超时

Why 2: 为什么CDN回源超时？
  → CDN厂商骨干网光纤中断

Why 3: CDC骨干网中断是第三方不可抗力，但为什么切换花了20分钟？
  → 切换CDN的DNS操作需要人工执行，且团队不熟悉操作流程

Why 4: 为什么没有自动切换方案？
  → 多CDN架构此前因成本考虑被搁置

Why 5 (根因): 为什么没考虑到CDN全局故障的场景？
  → 架构评审时对CDN故障的假设过于乐观，未做充分容灾设计
```

**改进措施：**

| 措施 | 类型 | 责任人 | 期限 |
|------|------|--------|------|
| 配置多CDN互为备份，自动故障切换 | 架构 | 基础架构团队 | 2个月 |
| 编写CDN故障应急操作手册(含DNS切换步骤) | 流程 | 运维团队 | 1周 |
| 每季度CDN故障切换演练 | 流程 | 运维团队 | 持续 |
| CDN故障检测从被动告警改为主动探测+自动切换 | 工具 | 基础架构团队 | 3个月 |
| 关键页面增加CDN可用性主动探测监控 | 监控 | 监控团队 | 1周 |

**核心教训：** 永远不要假设第三方服务100%可靠。所有外部依赖都需要容灾方案。

### 案例2：数据库误删 —— 运维操作失误 P0

**事故概要：**
运维同事在清理测试环境时，误将生产环境的某个非核心数据库删除了。虽然数据可通过备份恢复，但恢复过程中相关功能不可用约4小时。

**时间线：**

| 时间 | 事件 |
|------|------|
| 09:30 | 运维同事登录堡垒机，准备清理测试环境旧数据 |
| 09:32 | 在错误的终端窗口执行了 `DROP DATABASE xxx` |
| 09:33 | 运维同事意识到错误，立即报告 |
| 09:35 | 应急群建立，DBA开始评估影响和恢复方案 |
| 09:50 | 确认最新备份时间为09:00，数据损失<30分钟 |
| 10:00 | 开始从备份恢复数据库 |
| 13:15 | 数据恢复完成，依赖该数据库的服务逐个恢复 |
| 13:30 | 所有服务恢复正常 |

**5-Why分析：**

```
Why 1: 为什么数据库被误删？
  → 运维同事在错误的终端窗口（生产环境连接）执行了DROP命令

Why 2: 为什么能在生产环境执行危险命令？
  → 生产环境数据库账号有DROP权限

Why 3: 为什么生产环境账号有DROP权限？
  → 日常运维脚本需要DDL权限，未做权限分级

Why 4: 为什么没有二次确认？
  → 数据库客户端未开启危险命令确认；堡垒机未设危险命令拦截

Why 5 (根因): 为什么这些安全措施缺失？
  → 缺少数据库操作安全规范，运维操作风险未被系统化评估
```

**改进措施：**

| 措施 | 类型 | 责任人 |
|------|------|--------|
| 生产数据库账号权限分级：日常只读，DDL走审批后临时授权 | 流程 | DBA团队 |
| 堡垒机配置危险命令拦截：DROP DATABASE/TABLE需二次审批 | 工具 | 安全团队 |
| 所有数据库客户端终端提示符显示当前连接的【环境+库名】 | 工具 | 运维团队 |
| 测试环境和生产环境使用不同配色/背景的终端（视觉区分） | 工具 | 运维团队 |
| 生产数据库备份频率评估：关键库备份间隔<30分钟 | 流程 | DBA团队 |

**核心教训：** 不能依赖人的"小心"来防止误操作。系统应该设计为"即使想误操作也很难做到"。

### 案例3：缓存雪崩 —— 大促零点准时崩塌 P1

**事故概要：**
大促活动0点开始，大量用户涌入，首页缓存集体过期，请求全部穿透到数据库，数据库CPU飙升至100%，首页加载超时。

**时间线：**

| 时间 | 事件 |
|------|------|
| 00:00 | 大促活动正式开始 |
| 00:01 | 监控告警：首页API P99延迟 > 5秒 |
| 00:03 | 数据库CPU 100%，首页P99 > 30秒 |
| 00:05 | 执行止损：紧急扩容数据库（升级实例规格）+ 应用限流 |
| 00:08 | 执行止损2：首页缓存预热脚本启动 |
| 00:15 | 缓存预热完成，数据库CPU降到60% |
| 00:20 | 服务恢复正常 |

**根因分析：**
- 直接原因：活动开始时刻大量缓存同时过期（相同的TTL），请求直接打到数据库
- 根本原因：缓存预热方案未考虑"缓存雪崩"场景，所有缓存使用固定TTL（1小时），在0点活动开始时集中过期

**改进措施：**

| 措施 |
|------|
| 缓存TTL增加随机偏移量（如基础TTL 60分钟 + 随机0-10分钟） |
| 大促活动的缓存提前30分钟预热，而非等到0点 |
| 引入本地缓存(如Caffeine) + Redis 两级缓存 |
| 数据库连接池增加限流，防止瞬时流量打垮数据库 |
| 大促前必须通过"缓存击穿/雪崩/穿透"三项测试 |

### 案例4：发布引入Bug —— 黄金周前夜的"惊喜" P1

**事故概要：**
五一黄金周前一天晚上20:00发布，21:00开始陆续有用户反馈无法正常下单。回滚到上一版本后恢复。影响约30%用户，持续40分钟。

**根因：**
- 新版本修改了订单金额计算逻辑，使用了新的促销引擎
- 测试环境验证通过，但测试环境中促销数据很少，覆盖不够
- 生产环境中一个历史遗留的促销规则（三年前创建但从未下线）触发了NPE

**5-Why：**
```
Why 1: 生产NPE → 新促销引擎遇到特殊促销规则
Why 2: 测试环境未发现 → 测试环境促销数据与生产不一致
Why 3: 数据不一致 → 测试数据未定期从生产脱敏同步
Why 4: 没有同步机制 → 数据同步之前做过但太慢，被废弃了
Why 5 (根因): → 测试环境管理缺少规范，环境一致性问题被长期忽视
```

**改进措施：**

| 措施 |
|------|
| 建立生产→测试环境数据脱敏同步机制（每周一次） |
| 生产环境促销规则全面清理（过期规则下线） |
| 发布前灰度验证：按用户比例逐步放量（5%→20%→50%→100%） |
| 黄金周/大促等特殊时期前7天冻结发布（仅修复P0/P1 Bug可发布） |
| 发布回滚SLA：决定回滚后10分钟内完成 |

### 案例5：消息队列消费阻塞 —— 订单通知延迟 P2

**事故概要：**
用户下单后，订单确认推送（App推送+短信）延迟达40分钟。排查发现消息队列消费异常。

**时间线：**

| 时间 | 事件 |
|------|------|
| 17:00 | 运营反馈：客服收到用户投诉"下单后没收到确认" |
| 17:10 | 排查发现RabbitMQ消息积压30万条 |
| 17:15 | 查看消费者：2个消费者中1个CPU 100%卡死 |
| 17:20 | 重启卡死的消费者，积压开始消减 |
| 18:00 | 30万积压消息消化完毕，服务正常 |

**根因：**
- 某个订单包含异常数据（收货地址超过数据库定义长度），导致消息处理死循环（处理失败→重试→再失败→重试...）
- 异常消息未被移入死信队列，一直重试占用了消费者

**改进措施：**

| 措施 |
|------|
| 消息消费者增加处理超时（单条消息处理>30秒则中断） |
| 配置死信队列：重试3次仍失败的消息移入死信队列 |
| 消费者增加并发处理能力（从1个消费者扩展到3个） |
| 死信队列告警：死信队列有消息堆积时即时通知 |
| 消费者增加健康检查：CPU>90%持续1分钟自动重启 |

  □ 进行了5-Why根因分析
  □ 讨论了"为什么更早阶段没发现"

  □ 每项措施明确了责任人和截止日期
  □ 记录了做得好的地方

  □ 24小时内发出复盘报告

```

### C. 事故趋势度量指标

```
1. MTBF (Mean Time Between Failures) - 平均故障间隔
   目标: 持续增长
   计算: 正常运行时间 / 事故次数

2. MTTR (Mean Time To Recover) - 平均恢复时间
   目标: 持续下降
   计算: 事故总恢复时间 / 事故次数

3. MTTD (Mean Time To Detect) - 平均发现时间
   目标: <5分钟 (自动化监控)
   计算: 事故总发现时间 / 事故次数

4. 改进措施完成率
   目标: >90%
   计算: 已完成Action数 / 总Action数

5. 事故复发率
   目标: 0
   计算: 同类根因的事故次数
```

---

*事故复盘不是在寻找"谁的错"，而是在寻找"系统哪里可以更健壮"。每一次事故都是免费的可靠性工程课程。*
