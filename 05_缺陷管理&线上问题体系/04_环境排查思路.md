# 环境排查思路 (Environment Troubleshooting)

> **环境问题**是测试工作中最常见、最耗时的问题之一。"在我的环境上是好的"是开发与测试之间永恒的经典对话。掌握系统化的环境排查方法论，能从"这环境有问题"的模糊抱怨，进化到"端口8080被PID 12345占用，杀掉进程后恢复"的精准诊断。

## 1. 环境排查方法论

### 1.1 从现象到根因的分层排查模型

环境排查采用**OSI分层排查思想**，从底层到上层（或从上层到底层）逐层排除：

```
┌─────────────────────────────────────────────┐
│  第7层 - 应用层                               │
│  排查: 应用日志、错误码、业务逻辑              │
│  工具: 日志平台/Kibana/grep/应用健康检查        │
├─────────────────────────────────────────────┤
│  第6层 - 服务层                               │
│  排查: 进程状态、端口、配置、依赖服务          │
│  工具: ps/netstat/ss/systemctl/docker         │
├─────────────────────────────────────────────┤
│  第5层 - 中间件层                             │
│  排查: Redis/MySQL/MQ/Nginx/ES 状态          │
│  工具: redis-cli/mysql-client/curl/rabbitmqctl │
├─────────────────────────────────────────────┤
│  第4层 - 网络层                               │
│  排查: 连通性、DNS、防火墙、负载均衡           │
│  工具: ping/telnet/nc/traceroute/nslookup     │
├─────────────────────────────────────────────┤
│  第3层 - 系统资源层                           │
│  排查: CPU/内存/磁盘/IO/文件描述符             │
│  工具: top/free/df/iostat/lsof                │
├─────────────────────────────────────────────┤
│  第2层 - 容器/虚拟化层（如适用）               │
│  排查: Docker/K8s Pod状态、资源限制            │
│  工具: docker ps/kubectl get pods             │
├─────────────────────────────────────────────┤
│  第1层 - 硬件/云资源层                         │
│  排查: 云实例状态、磁盘故障、网络故障          │
│  工具: 云控制台/硬件监控                       │
└─────────────────────────────────────────────┘
```

### 1.2 排查黄金法则

**法则1：先近后远**
```
先检查本地环境 → 再检查网络连通性 → 最后检查远程服务
不要一上来就怀疑是第三方服务的问题
```

**法则2：先简后繁**
```
先检查基础连通性（ping/端口） → 再检查配置 → 最后深入日志
避免一开始就看几十MB的日志文件
```

**法则3：一次只改一个变量**
```
当排查需要尝试修改时，每次只改一个配置/参数
改完测试效果 → 记录结果 → 还原 → 再试下一个
同时改多个变量会让你不知道哪个才是根本原因
```

**法则4：保留现场**
```
不急于重启/重装/清日志
先收集信息 → 再尝试修复
在不明原因的情况下重启，可能错失了找到根因的机会
```

**法则5：对比法**
```
有问题的环境 vs 正常的环境（对比差异）
现在的状态 vs 之前正常的状态（对比变化）
A机器 vs B机器（对比配置）
```

### 1.3 环境排查通用流程

```
【Step 1】确认现象
  □ 准确描述异常现象（报错信息/状态码/现象）
  □ 确认异常是必现还是偶发
  □ 确认异常从什么时候开始
  □ 确认最近是否有变更（发布/配置修改/网络变更）

【Step 2】初步定位层级
  □ 检查应用是否能响应（区分是网络不通还是服务挂了）
  □ 检查服务进程是否存在（ps aux | grep xxx）
  □ 检查端口是否在监听（netstat -tlnp | grep PORT）
  □ 快速判断：是网络层问题？服务层问题？资源层问题？

【Step 3】逐层深入排查
  □ 根据Step 2的判断，进入对应层进行深入排查
  □ 收集该层的日志、状态信息
  □ 使用该层的专用排查工具

【Step 4】对比确认
  □ 与正常环境/正常时段对比
  □ 确认差异点即为问题根因

【Step 5】修复验证
  □ 修复问题
  □ 验证修复效果
  □ 记录排查过程和根因（方便日后参考）
```

## 2. 网络问题排查

### 2.1 网络排查分层策略

```
应用层    → curl/wget 测试HTTP(S)可达性
传输层    → telnet/nc 测试TCP端口连通性
网络层    → ping/traceroute 测试IP可达性和路由
链路层    → arp/ethtool 测试MAC和物理链路
DNS层     → nslookup/dig 测试域名解析
```

### 2.2 基础连通性排查

#### ping - 测试IP层连通性

```bash
# 基础ping测试
ping -c 4 192.168.1.100

# 指定发包大小（测试MTU问题）
ping -c 4 -s 1472 192.168.1.100   # 标准以太网MTU 1500 - 28(ICMP头) = 1472

# 指定网卡
ping -c 4 -I eth0 192.168.1.100

# 结果解读:
# 0% packet loss → IP层通
# 100% packet loss → IP层不通，检查网线/网卡/防火墙/路由
# 部分丢包 → 网络不稳定，检查带宽/交换机/网卡
```

#### telnet - 测试TCP端口连通性

```bash
# 测试特定端口是否可达
telnet 192.168.1.100 3306
# 返回: Connected → 端口通
# 返回: Connection refused → 端口未监听或被拒绝
# 返回: Connection timed out → 网络不通或防火墙拦截

# 超时时间设置
telnet 192.168.1.100 3306  # 默认超时较长
# 或使用 nc (netcat) 设置超时:
nc -zv -w 3 192.168.1.100 3306  # 3秒超时
```

#### traceroute - 追踪路由路径

```bash
# Linux/Mac
traceroute 192.168.1.100       # Linux
traceroute -n 192.168.1.100    # 不解析主机名（更快）

# Windows
tracert 192.168.1.100

# 结果解读:
# 每一跳显示三个延迟时间（3次探测）
# * * * → 该节点不响应但数据包可能仍在转发
# 某一跳之后全是 * * * → 该节点或其后路由有问题
# 延迟在某跳突然增大 → 该跳是瓶颈
```

### 2.3 DNS排查

```bash
# DNS解析测试
nslookup api.example.com
dig api.example.com

# 指定DNS服务器测试
nslookup api.example.com 8.8.8.8
dig @8.8.8.8 api.example.com

# 查看DNS配置
cat /etc/resolv.conf
# nameserver 114.114.114.114

# 清除DNS缓存
# Linux (systemd-resolved):
sudo systemd-resolve --flush-caches
# macOS:
sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
# Windows:
ipconfig /flushdns

# 查看DNS缓存
# macOS:
sudo dscacheutil -cachedump
# systemd:
sudo systemd-resolve --statistics
```

### 2.4 网络抓包（tcpdump/Wireshark）

#### tcpdump 常用命令

```bash
# 抓取特定主机+端口的包
sudo tcpdump -i eth0 host 192.168.1.100 and port 8080 -w capture.pcap

# 抓取HTTP请求（端口80或8080）
sudo tcpdump -i eth0 -A 'tcp port 80 or tcp port 8080' | grep -i 'GET\|POST\|Host:'

# 抓取特定时间段的包
sudo tcpdump -i eth0 -w capture_$(date +%Y%m%d_%H%M%S).pcap -G 60 -W 5
# -G 60: 每60秒轮转 -W 5: 保留5个文件

# 读取pcap文件
tcpdump -r capture.pcap -A | less

# 限制抓包数量
sudo tcpdump -i eth0 -c 1000 -w capture.pcap

# 过滤条件组合
sudo tcpdump -i any \
  'host 10.0.1.5 and (port 443 or port 80)' \
  -w capture.pcap
```

#### 抓包分析要点

```
TCP三次握手问题诊断:
  - SYN发出无SYN-ACK → 对端未监听或防火墙拦截
  - SYN发出收到RST → 端口被拒绝（防火墙或服务拒绝）
  - 三次握手后立即RST → 应用层拒绝连接（配置/协议不匹配）
  
HTTP问题诊断:
  - 注意请求头和响应头中的异常
  - 关注状态码: 502/503/504 通常指向上游服务问题
  - Host头是否正确（Nginx虚拟主机匹配）
  - Content-Type是否匹配
  
HTTPS问题诊断:
  - Client Hello后无Server Hello → TLS版本不匹配
  - Certificate错误 → 证书过期/域名不匹配/CA不信任
```

### 2.5 网络排查检查清单

```
□ 本机IP是否正确: ip addr / ifconfig
□ 网关是否可达: ping 网关IP
□ 外网是否可达: ping 8.8.8.8
□ DNS是否正常: nslookup baidu.com
□ 目标IP是否可达: ping 目标IP
□ 目标端口是否可达: telnet/nc 目标IP 目标端口
□ 防火墙规则: iptables -L -n / firewalld
□ 安全组/ACL: 云控制台检查
□ 代理设置: env | grep -i proxy
□ hosts文件: cat /etc/hosts (是否被劫持)
□ 路由表: ip route / route -n (默认路由是否正确)
□ 网络接口状态: ip link show (是否UP)
□ MTU配置: ip link show | grep mtu
```

## 3. 服务问题排查

### 3.1 进程排查

```bash
# 查看进程是否存在
ps aux | grep java     # Java应用
ps aux | grep nginx
ps aux | grep node

# 查看进程详细信息
ps -ef | grep 进程名
ps -p PID -o pid,ppid,user,pcpu,pmem,etime,args

# 查看进程树（父子关系）
pstree -p PID

# 查看进程打开的文件描述符
lsof -p PID
lsof -p PID | wc -l   # 文件描述符数量（检查泄漏）

# 查看进程启动时间
ps -p PID -o lstart

# 实时监控进程
top -p PID
htop -p PID            # 更友好的交互界面

# 查看进程环境变量
cat /proc/PID/environ | tr '\0' '\n'

# 查看进程工作目录
ls -la /proc/PID/cwd
```

### 3.2 端口排查

```bash
# 查看所有监听端口
netstat -tlnp
ss -tlnp                # 更快的替代命令

# 查看特定端口
netstat -tlnp | grep :8080
ss -tlnp | grep :8080

# 查看端口被哪个进程占用
lsof -i :8080
fuser 8080/tcp

# 查看所有TCP连接（含状态）
netstat -tnp
ss -tnp state established

# 查看连接状态统计
netstat -tn | awk '{print $6}' | sort | uniq -c | sort -rn
# 关注 TIME_WAIT / CLOSE_WAIT 过多的情况

# 查看UDP端口
netstat -ulnp
```

**端口状态说明：**

| 状态 | 含义 | 关注点 |
|------|------|--------|
| LISTEN | 正在监听 | 正常 |
| ESTABLISHED | 连接已建立 | 数量过多检查连接池 |
| TIME_WAIT | 主动关闭后等待 | 数量>1000检查短连接和tcp_tw_reuse |
| CLOSE_WAIT | 等待应用层关闭 | 数量>100说明应用未正确关闭连接（Bug） |
| SYN_RECV | 收到SYN未完成握手 | 数量多可能是SYN Flood攻击 |

### 3.3 日志排查

```bash
# 实时查看日志
tail -f /var/log/app/app.log

# 查看最后N行
tail -100 /var/log/app/app.log

# 搜索关键词
grep "ERROR" /var/log/app/app.log
grep -i "exception\|error\|fail" /var/log/app/app.log

# 带上下文
grep -C 5 "ERROR" /var/log/app/app.log    # 前后各5行
grep -B 10 "ERROR" /var/log/app/app.log   # 前10行
grep -A 10 "ERROR" /var/log/app/app.log   # 后10行

# 按时间段查找（假设日志格式包含时间戳）
grep "2024-01-15 14:3" /var/log/app/app.log  # 14:30-14:39
sed -n '/2024-01-15 14:30/,/2024-01-15 14:35/p' /var/log/app/app.log

# 搜索多个文件
grep -r "ERROR" /var/log/app/

# 统计ERROR出现次数
grep -c "ERROR" /var/log/app/app.log
grep "ERROR" /var/log/app/access.log | awk '{print $9}' | sort | uniq -c | sort -rn

# 按TraceId追踪完整请求链（微服务场景）
grep "traceId=abc123" /var/log/app/*.log
```

**日志排查优先级：**

```
第一优先级 → ERROR/CRITICAL/FATAL 日志
第二优先级 → WARN 日志
第三优先级 → 发生时间点前后5分钟的所有日志
第四优先级 → 特定TraceId/RequestId的完整链路日志
```

## 4. 数据库问题排查

### 4.1 数据库连接排查

```bash
# MySQL连接测试
mysql -h 192.168.1.100 -P 3306 -u root -p -e "SELECT 1"
# 返回 1 → 连接正常

# 查看当前连接数
mysql -h ... -e "SHOW PROCESSLIST"
mysql -h ... -e "SHOW STATUS LIKE 'Threads_connected'"
mysql -h ... -e "SHOW VARIABLES LIKE 'max_connections'"

# 计算连接使用率
# 连接数 / max_connections > 80% → 需要关注
mysql -h ... -e "
  SELECT 
    (SELECT COUNT(*) FROM INFORMATION_SCHEMA.PROCESSLIST) AS current_connections,
    (SELECT @@max_connections) AS max_connections,
    ROUND(
      (SELECT COUNT(*) FROM INFORMATION_SCHEMA.PROCESSLIST) 
      / (SELECT @@max_connections) * 100, 2
    ) AS usage_percentage
"

# 杀掉长时间运行的查询
mysql -h ... -e "
  SELECT id, user, host, db, command, time, state, info
  FROM INFORMATION_SCHEMA.PROCESSLIST
  WHERE time > 60 AND command != 'Sleep'
  ORDER BY time DESC
"
# KILL QUERY {id};  -- 杀掉查询但不杀连接
# KILL {id};        -- 杀掉连接
```

### 4.2 慢查询排查

```bash
# 查看慢查询配置
mysql -h ... -e "
  SHOW VARIABLES LIKE 'slow_query%';
  SHOW VARIABLES LIKE 'long_query_time';
"

# 开启慢查询日志（如未开启）
# SET GLOBAL slow_query_log = 'ON';
# SET GLOBAL long_query_time = 1;  -- 1秒以上记录

# 查看慢查询日志
tail -100 /var/log/mysql/slow-query.log

# 分析慢查询日志（mysqldumpslow）
mysqldumpslow -s t -t 10 /var/log/mysql/slow-query.log
# -s t: 按时间排序
# -s c: 按次数排序
# -t 10: 显示前10条

# 直接查看当前正在执行的查询
mysql -h ... -e "SHOW FULL PROCESSLIST" | grep -v "Sleep"

# 查看未使用索引的查询（需开启）
# SET GLOBAL log_queries_not_using_indexes = 'ON';
```

**慢查询常见原因及解决方案：**

| 原因 | 现象 | 解决方案 |
|------|------|----------|
| 缺少索引 | 全表扫描 | EXPLAIN分析，添加合适索引 |
| 索引失效 | 对索引列使用函数/计算/类型转换 | 改写SQL避免索引列运算 |
| 数据量过大 | 单表>千万行 | 分库分表/数据归档 |
| 锁等待 | 大量Waiting for table metadata lock | 检查是否有长事务未提交 |
| SQL写法问题 | 子查询/OR条件 | 改写为JOIN/UNION |
| 返回字段过多 | SELECT * 包含TEXT/BLOB | 只查询需要的字段 |

### 4.3 锁与死锁排查

```bash
# MySQL 查看当前锁等待
mysql -h ... -e "
  SELECT 
    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx_id,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query,
    TIMESTAMPDIFF(SECOND, r.trx_wait_started, NOW()) AS wait_seconds
  FROM information_schema.innodb_lock_waits w
  JOIN information_schema.innodb_trx r 
    ON w.requesting_trx_id = r.trx_id
  JOIN information_schema.innodb_trx b 
    ON w.blocking_trx_id = b.trx_id
"

# 查看最近一次死锁
mysql -h ... -e "SHOW ENGINE INNODB STATUS\G" | grep -A 30 "LATEST DETECTED DEADLOCK"

# 查看长时间未提交的事务
mysql -h ... -e "
  SELECT 
    trx_id, trx_state, trx_started,
    TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS seconds_running,
    trx_mysql_thread_id, trx_query
  FROM information_schema.innodb_trx
  WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 30
  ORDER BY seconds_running DESC
"

# PostgreSQL 查看锁等待
psql -h ... -c "
  SELECT 
    blocked_locks.pid AS blocked_pid,
    blocking_locks.pid AS blocking_pid,
    blocked_activity.query AS blocked_query,
    blocking_activity.query AS blocking_query,
    blocked_activity.query_start AS blocked_since
  FROM pg_catalog.pg_locks blocked_locks
  JOIN pg_catalog.pg_locks blocking_locks 
    ON blocked_locks.locktype = blocking_locks.locktype
    AND blocked_locks.DATABASE IS NOT DISTINCT FROM blocking_locks.DATABASE
    AND blocked_locks.relation IS NOT DISTINCT FROM blocking_locks.relation
    AND blocked_locks.page IS NOT DISTINCT FROM blocking_locks.page
    AND blocked_locks.tuple IS NOT DISTINCT FROM blocking_locks.tuple
    AND blocked_locks.virtualxid IS NOT DISTINCT FROM blocking_locks.virtualxid
    AND blocked_locks.transactionid IS NOT DISTINCT FROM blocking_locks.transactionid
    AND blocked_locks.pid != blocking_locks.pid
  JOIN pg_catalog.pg_stat_activity blocked_activity 
    ON blocked_locks.pid = blocked_activity.pid
  JOIN pg_catalog.pg_stat_activity blocking_activity 
    ON blocking_locks.pid = blocking_activity.pid
  WHERE NOT blocked_locks.granted
"
```

```bash

# us(用户态)高 → 应用代码消耗

# wa(iowait)高 → IO等待, CPU其实不忙
```

## 7. 环境排查实战案例

### 案例1：新部署的服务无法访问

**现象：**
开发刚在测试服务器（192.168.1.50）上部署了订单服务，访问 `http://192.168.1.50:8080/api/health` 无响应。

**排查过程：**

```
Step 1: 本地检查
  本地ping 192.168.1.50 → 通 ✓
  本地telnet 192.168.1.50 8080 → Connection refused ✗
  → 端口不可达，在服务器上继续排查

Step 2: 服务器进程检查
  ssh到192.168.1.50
  ps aux | grep java → 有java进程 PID=12345 ✓
  → 进程存在，但端口不通，检查端口

Step 3: 端口监听检查
  netstat -tlnp | grep 8080 → 无输出 ✗
  netstat -tlnp | grep java → 127.0.0.1:8080 ← 问题找到了!
  → 应用绑定在127.0.0.1而非0.0.0.0，外部无法访问

Step 4: 确认配置
  grep "server.address\|server.port" /app/application.yml
  输出: server.address: 127.0.0.1  ← 确认根因

Step 5: 修复
  修改配置: server.address: 0.0.0.0
  重启服务
  telnet 192.168.1.50 8080 → Connected ✓
  curl http://192.168.1.50:8080/api/health → 200 ✓
```

**教训：** 默认配置检查清单中增加"绑定地址"检查项；部署后自动验证端口可外部访问。

### 案例2：数据库连接池耗尽

**现象：**
应用日志大量 `Cannot get JDBC Connection; nested exception is java.sql.SQLTransientConnectionException: HikariPool-1 - Connection is not available, request timed out after 30000ms`

**排查过程：**

```
Step 1: 确认数据库可连接
  mysql -h 192.168.1.60 -u app -p -e "SELECT 1" → 正常 ✓
  → 数据库本身可用，连接池问题

Step 2: 检查连接数
  mysql -e "SHOW PROCESSLIST" | wc -l → 152
  mysql -e "SHOW VARIABLES LIKE 'max_connections'" → 151
  → 连接数已达上限！

Step 3: 检查哪些连接在占用
  mysql -e "SELECT user, host, db, command, time, state 
             FROM INFORMATION_SCHEMA.PROCESSLIST 
             WHERE command != 'Sleep' 
             ORDER BY time DESC"
  → 发现50+个查询处于"Sending data"状态，已经运行超过120秒

Step 4: 找到慢查询SQL
  mysql -e "SELECT info FROM INFORMATION_SCHEMA.PROCESSLIST 
             WHERE time > 60 AND info IS NOT NULL"
  → 发现是一条未加索引的报表查询

Step 5: 定位发起方
  根据host字段定位到是运营后台发起的查询
  → 运营后台有导出功能，导出近1年数据 → 全表扫描2000万行

Step 6: 临时止血
  KILL 掉长时间运行的查询
  → 连接池恢复

Step 7: 永久修复
  1. 为查询加索引
  2. 运营后台限制导出时间范围（单次最多3个月）
  3. 大量数据导出改为异步模式
  4. 应用配置连接池最大等待时间（避免无限等待）
```

### 案例3：Redis内存满导致服务异常

**现象：**
应用报错 `OOM command not allowed when used memory > 'maxmemory'`，同时用户会话丢失、缓存不可用。

**排查过程：**

```
Step 1: 检查Redis内存
  redis-cli -h 192.168.1.70 INFO memory
  used_memory_human: 4.00G
  maxmemory_human: 4.00G
  maxmemory_policy: noeviction
  → 内存满，且策略是noeviction(不淘汰，写入直接拒绝)

Step 2: 检查是什么占用了内存
  redis-cli --bigkeys
  → 发现一个List Key: "task_queue:export" 有200万条数据，占用3.2GB

Step 3: 确认原因
  → 导出任务队列消费者挂掉了，任务持续积压但无人消费

Step 4: 检查消费者
  ps aux | grep export_consumer → 进程不存在！
  systemctl status export_consumer → inactive (dead)
  tail /var/log/export_consumer/error.log
  → 3天前消费者因OOM被系统kill

Step 5: 临时止血
  redis-cli DEL task_queue:export  # 清空积压队列(确认可删除后)
  redis-cli CONFIG SET maxmemory-policy allkeys-lru  # 临时改为LRU淘汰
  → 内存降到1.2GB，恢复正常

Step 6: 永久修复
  1. 消费者增加进程守护(supervisor/systemd Restart=always)
  2. List类型队列设置最大长度上限(MAXLEN ~ 10000)
  3. Redis maxmemory-policy评估是否应设为allkeys-lru或volatile-lru
  4. 增加Redis内存>80%告警
  5. 队列消费者增加内存限制和健康检查
```

### 案例4：Nginx 502 Bad Gateway

**现象：**
访问网站返回502 Bad Gateway，影响所有用户。

**排查过程：**

```
Step 1: 检查Nginx错误日志
  tail -50 /var/log/nginx/error.log
  → connect() failed (111: Connection refused) while connecting to upstream
  
Step 2: 检查上游服务
  ps aux | grep app-server → 进程不存在！
  → 应用服务挂了

Step 3: 检查应用日志（为什么挂了）
  tail -100 /var/log/app/error.log
  → java.lang.OutOfMemoryError: Java heap space
  → 应用因OOM崩溃

Step 4: 检查为什么OOM
  grep "OutOfMemoryError" /var/log/app/error.log*
  → 最近1小时出现多次Full GC日志
  → 怀疑内存泄漏或大对象

Step 5: 检查JVM参数
  ps aux | grep java (如进程已挂，查看启动脚本)
  → -Xmx512m  (最大堆512MB)

Step 6: 检查请求量
  tail -10000 /var/log/nginx/access.log | wc -l
  → 最近5分钟有8000+请求，QPS约为27
  → 结合JVM堆512M，高并发下OOM合理

Step 7: 修复
  短期:
    - 重启应用服务，临时调大JVM: -Xmx1024m
    - Nginx增加限流: limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
  长期:
    - 分析heap dump，找到内存泄漏点
    - 应用增加水平扩展（多实例+负载均衡）
    - 增加JVM监控告警
```

### 案例5：HTTPS证书过期导致无法访问

**现象：**
用户浏览器提示"您的连接不是私密连接"，证书详情显示昨天到期。

**排查过程：**

```
Step 1: 确认证书到期时间
  echo | openssl s_client -servername www.example.com \
    -connect www.example.com:443 2>/dev/null | \
    openssl x509 -noout -dates
  输出:
    notBefore=Jan 15 00:00:00 2023 GMT
    notAfter=Jan 14 23:59:59 2024 GMT
  → 证书已于昨天过期

Step 2: 检查证书管理流程
  询问运维: 证书到期通知发到哪里了？
  → 发送到了前运维同事的邮箱（已离职）

Step 3: 检查是否有自动续期
  检查是否使用了Let's Encrypt / certbot
  → 购买的是付费证书，一年期，无自动续期

Step 4: 紧急处理
  临时方案: 
    - 如使用云CDN(阿里云/腾讯云CDN)，快速申请免费证书替换
    - 或使用Let's Encrypt临时签发90天证书
  永久方案:
    - 证书到期告警配置到团队邮件组+企业微信/钉钉群
    - 证书信息纳入CMDB
    - 使用ACME自动化续期(如适用)

Step 5: 证书替换后验证
  openssl s_client -connect www.example.com:443 -servername www.example.com \
    2>/dev/null | openssl x509 -noout -dates
  → 确认新证书有效期
  curl -I https://www.example.com → 200 OK
```

**教训：**
- 所有基础设施配置变更通知要发到团队，不能只到个人
- 证书/域名/第三方服务到期检查应自动化
- 建议配置日历提醒：到期前90天/60天/30天/7天/1天

## 本章小结

1. **分层排查法是环境诊断的核心方法论**：从OSI七层模型出发，自底向上（网络→服务→应用→业务）或自顶向下逐层排除，避免盲目重启和"试错式排查"。每层有明确的排查工具和检查清单。
2. **网络问题是环境故障的最高发区之一**：端口占用、防火墙拦截、DNS解析失败、负载均衡配置错误——掌握ping/telnet/curl/netstat/ss/tcpdump等工具是高效排查的前提。
3. **服务和数据库问题是根因的集中地**：服务启动失败、配置错误、日志爆满、数据库连接池耗尽、慢SQL拖垮系统——应从日志和监控入手，建立"现象→工具→诊断→修复"的标准路径。
4. **中间件问题（Redis/MQ/ES/Nginx）需要专项排查能力**：每种中间件有其独特故障模式（Redis OOM、MQ积压、ES集群分片异常、Nginx反向代理超时），需要建立专项排查手册。
5. **系统资源排查是最后兜底**：CPU/内存/磁盘/网络IO异常应通过标准化脚本快速诊断，将排查经验沉淀为自动化脚本而非每次都从头开始。

---

*本文档面向测试人员和初级运维人员，旨在培养"精准诊断"而非"重启试试"的环境排查习惯。*
