# 缺陷提交规范 (Defect Submission Standards)

> **高质量的缺陷报告是高效Bug修复的前提。** 一份好的Bug报告应该让开发人员能够在5分钟内理解问题、10分钟内复现问题、不需要额外沟通就能开始修复。

## 1. 缺陷报告必要元素

### 1.1 缺陷报告标准模板

```
┌─────────────────────────────────────────────────┐
│              缺 陷 报 告 单                      │
├─────────────────────────────────────────────────┤
│ 缺陷ID：        [自动生成]                       │
│ 标题：          [一句话描述问题本质]              │
│ 严重程度：       [S1/S2/S3/S4/S5]                │
│ 优先级：         [P0/P1/P2/P3]                    │
│ 所属模块：       [功能模块名]                      │
│ 发现版本：       [版本号+Build号]                  │
│ 发现环境：       [测试环境/预发布/生产环境]         │
│ 报告人：         [姓名]                           │
│ 报告日期：       [YYYY-MM-DD HH:MM]               │
│ 指派给：         [开发负责人]                      │
│ 缺陷状态：       [新建/确认中/已分配/修复中/        │
│                   已修复/待验证/已验证/关闭/重开]   │
├─────────────────────────────────────────────────┤
│ 复现步骤：                                        │
│   前置条件：                                      │
│   步骤1：                                         │
│   步骤2：                                         │
│   步骤3：                                         │
│                                                  │
│ 预期结果：                                        │
│   1.                                              │
│   2.                                              │
│                                                  │
│ 实际结果：                                        │
│   1.                                              │
│   2.                                              │
│                                                  │
│ 复现概率：    [必现 / 偶发(约X%) / 仅一次]         │
├─────────────────────────────────────────────────┤
│ 测试环境信息：                                    │
│   操作系统：      [Windows 11 / macOS 14 / ...]    │
│   浏览器/版本：    [Chrome 120.0 / Safari 17 / ...]│
│   设备型号：      [iPhone 15 Pro / Pixel 8 / ...]  │
│   网络环境：      [Wi-Fi / 4G / 有线 / VPN]        │
│   账号/数据：     [测试账号/数据ID]                 │
│   服务端版本：     [后端版本/接口版本]               │
│   数据库：        [若相关]                         │
├─────────────────────────────────────────────────┤
│ 附件：                                            │
│   □ 截图/录屏                                    │
│   □ 接口请求/响应 (curl/HAR)                     │
│   □ 服务端日志                                    │
│   □ 客户端日志                                    │
│   □ 测试数据文件                                  │

│   影响范围评估：   [影响哪些功能/用户]              │
└─────────────────────────────────────────────────┘
```

### 1.2 各元素填写要点

| 元素 | 是否必填 | 填写要点 |
|------|----------|----------|
| **标题** | 必填 | 一句话说清"什么模块→什么操作→什么异常"，30字内 |
| **严重程度** | 必填 | 参考《缺陷等级规范》，客观评价技术影响 |
| **优先级** | 必填 | 参考《缺陷等级规范》，或由PM/负责人决定 |
| **模块** | 必填 | 精确到具体功能模块，便于分配和统计 |
| **版本号** | 必填 | 带Build号或Commit Hash，精确溯源 |
| **环境** | 必填 | 多环境项目必须标注，生产Bug需特别标注 |
| **复现步骤** | 必填 | 按Given-When-Then格式编写 |
| **预期结果** | 必填 | 来自需求文档或常识推断 |
| **实际结果** | 必填 | 描述实际观察到的异常表现 |
| **复现概率** | 必填 | 帮助评估排查难度 |
| **测试环境信息** | 必填 | 越详细越好 |
| **附件** | 强烈建议 | 截图+日志+录屏能极大提升沟通效率 |
| **可能原因** | 可选 | 测试人员的初步判断，仅作参考 |
| **关联需求** | 可选 | 关联后可追溯需求变更历史 |

## 2. 优秀Bug标题写法

### 2.1 标题写作公式

```
优秀标题 = [模块/页面] + [操作/场景] + [异常表现]

     ┌──────────┐    ┌──────────────┐    ┌──────────┐
     │  在哪里   │ → │  做了什么     │ → │  发生了什么│
     └──────────┘    └──────────────┘    └──────────┘

好的例子：
  [订单确认页] 使用微信支付时，点击"立即支付"按钮后页面白屏，无法跳转支付

不好的例子：
  支付不了          ← 缺少模块和场景信息
  Bug              ← 完全没有信息
  有个问题         ← 描述过于模糊
```

### 2.2 关键词前置原则

标题开头应包含最核心的信息关键词，便于搜索和归类。

| 关键词类别 | 示例关键词 | 标题示例 |
|------------|-----------|----------|
| 异常类型 | 崩溃/闪退/白屏/报错/卡死/数据错误 | `[登录页] 输入特殊字符后App闪退` |
| 功能模块 | 支付/订单/登录/搜索/购物车 | `[搜索] 输入emoji表情后搜索结果为空` |
| 平台设备 | iOS/Android/Chrome/Safari | `[iOS] 首页轮播图在iPhone 15 Pro上不显示` |
| 用户角色 | 管理员/普通用户/未登录 | `[管理员] 批量导入用户时提示权限不足` |
| 数据量级 | 大数据量/并发/边界值 | `[订单列表] 加载超过500条订单时页面卡死` |

### 2.3 标题好坏对比

| 场景 | 坏标题 | 好标题 |
|------|--------|--------|
| 登录 | 登录报错 | `[登录] 使用已绑定微信的账号在登录页选择"微信登录"，提示"授权失败(code:30001)"` |
| 支付 | 付不了钱 | `[收银台] 选择支付宝支付后金额显示为0.00，确认支付后实际扣款正确` |
| 注册 | 注册有问题 | `[注册] 手机号输入框粘贴带空格的号码（如138 0013 8000）后点击获取验证码无响应` |
| 搜索 | 搜不出来 | `[商品搜索] 搜索"iPhone"时，价格筛选条件"5000-8000"设置后结果数不变，筛选未生效` |
| 订单 | 订单不对 | `[订单详情] 已取消订单的状态标签错误显示为"已完成"，且取消时间字段为空` |
| UI | 页面乱了 | `[个人中心] iOS 17.2 Safari下，用户头像与昵称重叠显示，间距丢失` |
| 接口 | 接口超时 | `[商品列表接口/api/products] 请求参数pageSize=100时响应时间>15秒，预期<2秒` |
| 兼容性 | 华为手机不行 | `[首页] 华为Mate 60 (HarmonyOS 4.0)启动时白屏5秒后才显示内容` |

### 2.4 标题避免的词语（黑名单）

不要使用以下模糊词语作为标题主体：

```
❌ 不可用 / 有问题 / 报错了 / 不行 / 不对 / 好像
❌ 有时候 / 偶尔 / 不知道为什么
❌ Bug / 问题 / 异常（这三个词是冗余的——Bug本身就是问题）

✅ 用具体的异常表现替代：
  - "不可用" → "点击后无响应" / "返回404" / "页面白屏"
  - "有问题" → "金额计算偏差0.02元" / "筛选条件未生效"
  - "不行" → "提交后提示'系统繁忙'" / "文件上传进度卡在99%"
```

## 3. 复现步骤编写规范

### 3.1 Given-When-Then格式

采用BDD风格的Given-When-Then格式编写复现步骤：

```
Given  [前置条件 - 在什么状态下]
When   [操作步骤 - 做了什么操作]
Then   [预期结果 - 应该看到什么]
But    [实际结果 - 实际看到了什么]  ← Bug的点
```

**完整示例：**

```
Given:
  - 用户已登录，账号：test_user_001
  - 购物车中有2件商品（商品A: 50元 × 2 = 100元, 商品B: 30元 × 1 = 30元）
  - 账户中有可用优惠券"满99减10"一张

When:
  1. 进入购物车页面
  2. 勾选商品A和商品B
  3. 点击"去结算"按钮
  4. 在订单确认页，选择优惠券"满99减10"
  5. 观察应付金额

Then（预期结果）:
  - 商品总价：130元 + 运费（如有）
  - 优惠金额：-10元（满99减10）
  - 应付金额：120元（+运费如有）

But（实际结果）:
  - 商品总价显示130元
  - 优惠金额显示-10元
  - 应付金额显示130元（优惠券未实际扣减）
  - 点击应付金额行可看到优惠已选但金额未变
```

### 3.2 复现步骤编写检查清单

在提交前，逐条检查：

```
□ 前置条件是否完整？
  - 是否需要特定账号/权限？
  - 是否需要特定数据状态？
  - 是否需要特定环境配置？
  
□ 步骤是否从零开始？
  - 读者能否从初始状态按步骤走到Bug触发点？
  - 没有跳步或隐含操作（如"先随便操作一下"）？

□ 步骤是否原子化？
  - 每步只做一个操作（点击/输入/选择）
  - "输入XX并点击提交"应拆为两步

□ 数据/参数是否具体？
  - 输入值写明具体内容，而非"输入一些文字"
  - 选择项写明具体选项名，而非"随便选一个"

□ 是否标明了关键检查点？
  - 哪一步之后应该观察什么
  - 异常在哪个环节出现

□ 是否有可替代的简化路径？
  - 如果是复杂操作链，是否提供了最短复现路径
```

### 3.3 偶发Bug的复现步骤

偶发Bug的复现步骤需要额外增加**频率信息**和**模式推测**：

```
【复现频率统计】
  尝试次数：20次
  复现次数：8次
  复现概率：约40%
  分布特征：连续操作第3-5次时最容易复现

【可能的触发条件推测】（测试人员分析）
  - 怀疑与网络延迟有关（WiFi环境复现率高于有线）
  - 怀疑与缓存状态有关（清除缓存后首次操作复现率更高）
  - 怀疑与并发有关（快速连续点击时复现率更高）

【复现操作建议】
  推荐操作节奏：快速连续操作，每秒点击2-3次
  推荐环境：WiFi网络，设备内存占用>80%时
```

### 3.4 复杂操作链的简化技巧

当完整复现路径很长时，同时给出**快速路径**：

```
【完整复现路径】（21步）
  Step 1-5: 用户注册 → 实名认证 → 绑定银行卡 → 设置支付密码 → 首次充值
  Step 6-10: 浏览商品 → 加入购物车 → 进入结算 → 选择地址 → 确认订单
  Step 11-15: 选择支付方式 → 输入支付密码 → 等待回调 → 查看订单 → ...
  ...

【快速复现路径】（通过数据构造跳过前10步）
  □ 使用测试账号: quick_test_001（已实名+绑卡+有余额）
  □ 使用预设购物车数据: cart_fixture_01（已有商品+地址）
  Step 1: 登录 quick_test_001
  Step 2: 进入购物车（已预装测试数据）
  Step 3: 点击"去结算" ← 从此处开始与完整路径一致
  ...
```

## 4. 截图/录屏/日志附件规范

### 4.1 截图规范

**截图必须包含：**

| 要求 | 说明 |
|------|------|
| **完整窗口** | 截图应包含浏览器地址栏/App状态栏，证明环境信息 |
| **标注重点** | 用红框/箭头标注异常点，不要只丢一张原图 |
| **时间戳** | 如果涉及时序问题，截图需包含系统时间 |
| **对比图** | 复杂Bug提供"预期 vs 实际"对比图 |
| **命名规范** | `模块_场景_序号.png`（如 `order_pay_error_01.png`） |

**截图标注示例（文字描述）：**

```
┌─────────────────────────────────────┐
│  [地址栏] https://xxx.com/order/123 │ ← 证明URL
├─────────────────────────────────────┤
│                                     │
│   订单金额：130.00 元               │
│   优惠金额：-10.00 元               │
│   ┌─────────────────────┐          │
│   │ 应付金额：130.00 元  │ ← 红框标注异常
│   │ 预期应为：120.00 元  │ ← 标注预期值
│   └─────────────────────┘          │
│                                     │
│   [时间] 2024-01-15 14:32:05       │ ← 包含时间
└─────────────────────────────────────┘
```

### 4.2 录屏规范

| 规范项 | 要求 |
|--------|------|
| **时长** | 控制在30秒-2分钟，不要超过3分钟 |
| **起点** | 从干净状态开始（清除缓存/重新打开页面） |
| **操作速度** | 慢于正常操作速度，确保每步可追踪 |
| **鼠标/触摸指示** | 开启显示点击位置（Android: 开发者选项→显示触摸操作） |
| **关键帧** | 在关键操作点停留1-2秒再继续 |
| **兼容性** | 横屏Bug录横屏，不要旋转 |
| **格式** | MP4/H.264，控制在10MB以内 |
| **附语音说明** | 可选，简单口头表述操作意图 |

**录屏工具推荐：**

| 平台 | 推荐工具 |
|------|----------|
| macOS | QuickTime Player (自带) / Kap / LICEcap (GIF) |
| Windows | Xbox Game Bar (Win+G) / ScreenToGif / OBS |
| iOS | 控制中心自带录屏 |
| Android | adb screenrecord / AZ Screen Recorder |
| Web | Chrome DevTools Recorder / Loom 浏览器插件 |

### 4.3 日志附件规范

**日志收集原则：**

```
1. 时间定位原则
   - 仅在Bug发生时点前后5分钟的日志即可（不要丢全天日志）
   - 在复现前清空日志或记录开始时间戳

2. 脱敏处理原则
   - 确认日志不包含密码、Token、身份证号等敏感信息
   - 使用 138****8000 格式对手机号等脱敏

3. 分级收集原则
   - 客户端日志：复现前打开Debug级别
   - 服务端日志：提供TraceId/RequestId
   - 网络日志：HAR文件/Charles抓包
```

**日志文件命名规范：**

```
{模块}_{环境}_{日期}_{序号}.{扩展名}

示例：
  order_test_20240115_01.log        ← 服务端日志
  app_android_crash_20240115.logcat ← Android logcat
  payment_pre_20240115_01.har       ← 网络抓包HAR
```

### 4.4 接口/网络信息附件

对于接口相关Bug，提供以下任一：

**方式一：Chrome DevTools 导出 HAR 文件**

```
Chrome DevTools → Network Tab → 复现Bug → 
右键任一请求 → Save all as HAR with content
```

**方式二：curl命令**

```bash
# 包含完整请求头和请求体
curl -X POST 'https://api.example.com/order/create' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer xxx' \
  -H 'X-Request-Id: test-trace-001' \
  -d '{
    "productId": "P001",
    "quantity": 2,
    "couponId": "COUPON_99_10"
  }' \
  -v \
  -o response.json \
  --write-out '\nHTTP_CODE:%{http_code}\nTIME:%{time_total}s\n'
```

**方式三：精简请求摘要**

```
Request:
  POST /api/order/create HTTP/1.1
  Content-Type: application/json
  X-Trace-Id: test-trace-001
  Body: {"productId":"P001","quantity":2,"couponId":"COUPON_99_10"}

Response:
  HTTP/1.1 200 OK
  Body: {"code":0,"data":{"totalAmount":13000,"discountAmount":1000,"payAmount":13000}}
                                 问题在这里 ↑ payAmount应为12000而非13000
```

## 5. 缺陷单生命周期管理

### 5.1 标准生命周期流程图

```
                          ┌─────────────┐
                          │   新建(New)  │  ← 测试人员提交
                          └──────┬──────┘
                                 │
                    ┌────────────┴────────────┐
                    ▼                         ▼
            ┌──────────────┐          ┌──────────────┐
            │ 确认中        │          │ 拒绝(Rejected)│ ← 不是Bug/重复/设计如此
            │ (Acknowledged)│          └──────────────┘
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ 已分配(Assigned)│  ← 指派给开发人员
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ 修复中(In Progress)│
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ 已修复(Fixed) │  ← 开发提交修复代码
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ 待验证        │  ← 等待部署到测试环境
            │ (Ready for QA)│
            └──────┬───────┘
                   │
        ┌──────────┴──────────┐
        ▼                     ▼
  ┌──────────┐        ┌──────────────┐
  │ 已验证通过 │        │ 重开(Reopened)│ ← 修复不完整/引入新问题
  │ (Verified)│        └──────┬───────┘
  └────┬─────┘               │
       │                      │ 重新分配给原开发
       ▼                      └────────→ 回到"已分配"
  ┌──────────┐
  │ 关闭(Closed)│
  └──────────┘
       │
       ▼ (如后续发现回归问题)
  ┌──────────┐
  │ 重开(Reopened)│
  └──────────┘
```

### 5.2 各状态流转条件与职责

| 状态转换 | 触发条件 | 操作人 | 必填信息 |
|----------|----------|--------|----------|
| 新建 → 确认中 | 测试提交Bug | 测试 | 完整Bug报告 |
| 确认中 → 已分配 | 确认是Bug | 测试负责人/PM | 指派给谁 |
| 确认中 → 拒绝 | 不是Bug/重复/设计如此 | 开发/PM | 拒绝原因 |
| 已分配 → 修复中 | 开发开始处理 | 开发 | - |
| 修复中 → 已修复 | 提交修复代码 | 开发 | 修复说明/Commit链接/影响范围 |
| 已修复 → 待验证 | 部署到测试环境 | 开发/运维 | 部署版本号 |
| 待验证 → 已验证 | 测试验证通过 | 测试 | 验证环境/验证结果 |

| 关闭 → 重开 | 线上再次出现/回归发现 | 测试 | 详细重开说明 |

### 5.3 拒绝(Rejected)的处理规范

**合理的拒绝原因：**

| 拒绝原因 | 说明 | 处理方式 |
|----------|------|----------|
| **设计如此** | 当前行为符合PRD设计 | 确认PRD，如PRD有问题则提需求变更 |
| **重复Bug** | 与已有Bug重复 | 关联到原Bug，关闭重复单 |
| **无法复现** | 开发多次尝试无法复现 | 测试提供更多信息/环境/数据，必要时现场复现 |
| **环境问题** | 由测试环境配置引起 | 修复环境后重试，确认是否为代码问题 |
| **第三方问题** | 由第三方SDK/服务引起 | 记录第三方Ticket ID，跟进第三方修复 |

**拒绝时必须填写的信息：**
```
□ 拒绝原因分类
□ 详细说明（为什么拒绝，至少20字）
□ 如为"设计如此"，附需求文档截图或链接
□ 如为"重复Bug"，附原Bug ID
□ 如为"无法复现"，附开发尝试的复现步骤和环境
□ 建议后续动作（转需求/关闭/挂起等）
```

## 本章小结

1. **高质量的Bug报告是高效修复的前提**：一份好的Bug报告应让开发在5分钟内理解问题、10分钟内复现。14元素报告模板确保信息完整，Given-When-Then格式的复现步骤提供清晰的操作路径。
2. **Bug标题是第一印象**：好标题遵循"模块+条件+现象"公式，让开发不看正文就能大致定位问题。避免"XX功能有问题"这类无信息量的标题。
3. **附件是Bug报告的关键证据**：截图标注关键信息、录屏展示操作过程、日志提供技术细节。附件不是"有就行"，而是"能加速理解才行"。
4. **缺陷生命周期管理保持团队共识**：从新建→确认→修复→验证→关闭的标准流程，以及各状态的责任人和SLA，确保缺陷不丢失、不拖延。
5. **开发与测试的Bug协作公约**：等级争议由对应角色裁定、不人身攻击、好的Bug报告值得表扬——这些软性规范比硬性流程更能提升协作效率。

*本文档适用于所有测试人员日常Bug提交参考，配合《01_缺陷等级规范》使用效果更佳。*
