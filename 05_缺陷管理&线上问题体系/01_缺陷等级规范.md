# 缺陷等级规范 (Defect Severity & Priority Standards)

> **缺陷等级规范**是测试团队与项目团队对Bug严重程度和修复优先级达成共识的基石文档。统一的标准可以避免开发与测试之间的"这不是Bug"或"这个不急"的争论，确保高风险的缺陷得到及时处理。

## 1. 缺陷严重程度定义 (Severity)

**严重程度 (Severity)** 描述的是缺陷对系统功能、数据、用户体验造成的**技术影响程度**，由测试人员根据客观标准判定，不随时间推移而改变。

### 1.1 五级严重度标准

| 等级 | 名称 | 英文 | 定义 | 典型表现 |
|------|------|------|------|----------|
| **S1** | 致命 | Blocker | 系统崩溃、数据丢失、核心业务流程完全不可用，导致测试/业务无法继续 | 服务器宕机、数据库被清空、支付流程彻底断裂、无法登录 |

> **IEEE 1044-2009 分类说明**：Blocker（阻断）和 Critical（严重）在IEEE 1044-2009标准中是不同的缺陷严重等级。Blocker指完全阻断测试或业务继续进行的缺陷（如无法登录、系统崩溃），Critical指导致关键功能严重偏离规格但系统仍可部分运行的缺陷（如核心功能计算结果错误、数据不一致）。本文档S1级取Blocker含义，S2级对应Critical含义。部分组织的S1标注为"Critical/Blocker"是对两者的混用，建议区分使用。
| **S2** | 严重 | Critical / Major | 核心功能异常或错误，无规避措施，严重影响用户使用（IEEE 1044定义的Critical等级：关键功能严重偏离规格） | 核心功能计算结果错误、数据不一致、关键接口500错误 |
| **S3** | 一般 | Normal / Medium | 功能存在缺陷但有规避措施，或非核心功能异常 | 非关键页面报错、次要功能不可用但有替代方案、UI显示异常但不影响操作 |
| **S4** | 轻微 | Minor / Low | 小问题，不影响功能使用，用户体验有轻微影响 | 文案错别字、UI样式偏差、提示信息不友好、非关键性能轻微下降 |
| **S5** | 建议 | Trivial / Enhancement | 改进建议，不属于缺陷 | 交互优化建议、功能体验改进、代码规范建议、性能优化建议 |

### 1.2 S1 - 致命 (Blocker) 详细标准

**判定条件（满足任意一条即为S1）：**

1. **系统崩溃/不可用**
   - 服务器进程异常退出或频繁重启
   - 应用无响应（ANR）或白屏超过30秒
   - 核心服务503/502错误率超过50%

2. **核心业务流程完全阻断**
   - 用户无法完成登录/注册
   - 支付流程在任一环节卡死无法继续
   - 下单流程完全不可用
   - 核心API返回100%错误

3. **数据安全/完整性致命问题**
   - 数据库被意外清空或回滚
   - 数据永久丢失且无法恢复
   - 事务提交后数据未持久化
   - 跨用户数据泄露（A用户看到B用户的敏感数据）

4. **安全漏洞**
   - SQL注入、XSS、CSRF等可直接利用的高危漏洞
   - 权限绕过导致未授权访问
   - 敏感信息明文传输或存储

5. **资金损失风险**
   - 支付金额计算错误（多扣/少扣）
   - 重复扣款
   - 账户余额异常

**处理要求：**
- 响应时间：15分钟内
- 修复时间：4小时内
- 如无法快速修复：必须回滚版本或关闭功能

### 1.3 S2 - 严重 (Critical) 详细标准

**判定条件（满足任意一条即为S2）：**

1. **核心功能错误**
   - 金额计算精度偏差（如分转元计算错误）
   - 订单状态流转异常
   - 搜索/筛选结果不准确
   - 核心数据统计报表数据错误

2. **无规避措施的功能异常**
   - 重要功能完全失效且无替代方案
   - 核心页面在特定条件下报500错误
   - 必填项校验失效

3. **数据一致性异常**
   - 主从数据库数据不一致
   - 缓存与数据库数据不一致
   - 前后端状态不一致

4. **兼容性严重问题**
   - 主流浏览器（Chrome/Safari/Edge）下核心功能不兼容
   - iOS/Android主流机型核心功能异常

5. **性能严重不达标**
   - 核心接口响应时间超过规格的3倍
   - 并发量达规格的50%即出现大量超时

**处理要求：**
- 响应时间：1小时内
- 修复时间：24小时内（或下一个发布窗口前）

### 1.4 S3 - 一般 (Normal / Medium) 详细标准

**判定条件：**

1. **有规避措施的功能缺陷**
   - 某操作路径报错但存在替代操作方式
   - 某按钮点击无响应但可通过快捷键完成
   - 某个入口不可用但可通过其他页面进入

2. **非核心功能异常**
   - 帮助文档无法访问
   - 用户头像上传后显示异常
   - 非核心辅助功能报错
   - 特定低频场景下的异常

3. **UI/交互问题影响使用**
   - 页面布局错乱导致信息重叠
   - 按钮位置偏移导致点击困难
   - 列表分页组件失效但数据量小

4. **数据校验非关键问题**
   - 前端校验缺失但后端有兜底
   - 错误提示信息不准确

**处理要求：**
- 响应时间：24小时内
- 修复时间：下一个迭代/Sprint内

### 1.5 S4 - 轻微 (Minor / Low) 详细标准

**判定条件：**

1. **文案/界面问题**
   - 错别字、标点错误
   - 提示文案不通顺
   - 文字溢出/截断
   - 国际化翻译缺失或不准确

2. **UI样式轻微偏差**
   - 颜色值与设计稿有偏差但不影响识别
   - 间距/字体大小与设计稿有少许差异
   - 图标模糊但不影响功能识别
   - 动画效果缺失或不流畅

3. **非关键路径小问题**
   - 操作成功后的提示弹窗样式异常
   - 空数据页面友好提示缺失
   - 非关键数据排序规则异常

4. **轻微性能波动**
   - 非核心接口偶尔超时（<5%）
   - 页面加载时长偶尔超过规格（<10%）

**处理要求：**
- 修复时间：可随版本排期修复，不阻塞发布

### 1.6 S5 - 建议 (Trivial / Enhancement) 详细标准

**判定条件：**

1. **用户体验优化建议**
   - "建议这个按钮放在右边更好操作"
   - "建议添加批量操作功能"
   - "建议增加键盘快捷键支持"

2. **代码质量改进**
   - 变量命名不规范（仅影响可读性）
   - 存在无用代码或注释的代码
   - 日志级别设置不够合理

3. **性能优化建议**
   - "此处可以加缓存优化"
   - "SQL可以优化索引"
   - "可以减少一次API调用"

**处理要求：**
- 进入产品Backlog评审，不保证修复时间

## 2. 缺陷优先级定义 (Priority)

**优先级 (Priority)** 描述的是缺陷应该被修复的**紧急程度和先后顺序**，由项目经理/产品经理/测试负责人综合考虑业务影响、严重程度、修复成本后决定，可能随时间推移而变化。

### 2.1 四级优先级标准

| 等级 | 名称 | 英文 | 定义 | 处理时间 | 决策依据 |
|------|------|------|------|----------|----------|
| **P0** | 立即修复 | Immediate | 必须立即停止当前所有工作，立刻修复 | 4小时内 | 线上服务中断、核心业务流程阻断、影响所有用户 |
| **P1** | 高优先级 | High | 在版本发布前必须修复 | 本版本/24小时内 | 核心功能不可用、影响大部分用户、无规避措施 |
| **P2** | 中优先级 | Medium | 在计划时间内修复，不阻塞当前版本 | 下个版本/本周内 | 有规避措施、影响小部分用户、次要功能 |
| **P3** | 低优先级 | Low | 可在资源允许时修复 | 适时排期 | 轻微问题、优化建议、影响极小 |

### 2.2 各优先级处理策略

#### P0 - 立即修复 (Immediate)

**特征：**
- 线上系统宕机或核心功能完全不可用
- 阻塞测试团队继续执行测试
- 造成直接经济损失
- 影响范围：>80%用户或全量

**处理流程：**
```
发现P0 → 立即在即时通讯群通知 → 召集团队紧急会议 → 
确认影响范围 → 制定修复方案 → 紧急修复/回滚 → 
验证修复 → 通知相关方 → 24小时内输出事故报告
```

**SLA要求：**
- 响应确认：15分钟内
- 修复上线：4小时内
- 24小时内复盘

#### P1 - 高优先级 (High)

**特征：**
- 核心功能存在严重缺陷但系统未完全瘫痪
- 影响范围：20%-80%用户
- 无有效规避措施
- 必须在发布版本前修复

**处理策略：**
- 排到当前迭代最高优先级
- 可能延期发布以确保修复质量
- 每日站会跟踪修复进度

#### P2 - 中优先级 (Medium)

**特征：**
- 存在规避措施
- 影响范围：<20%用户
- 修复成本可控
- 非核心路径问题

**处理策略：**
- 排入下一个版本或迭代
- 累积一定数量后集中修复
- 版本发布前评估是否可延期

#### P3 - 低优先级 (Low)

**特征：**
- 不影响功能使用
- UI/文案问题
- 优化建议类
- 修复成本高于缺陷影响

**处理策略：**
- 进入产品Backlog
- 大版本时统一排期
- 严重程度S5的建议类可直接转产品需求

### 2.3 优先级动态调整机制

优先级不是一成不变的，以下情况需要重新评估优先级：

| 触发条件 | 调整方向 | 举例 |
|----------|----------|------|
| 用户投诉增多 | 升级 | 某S4的UI问题导致大量客服投诉，升级为P1 |
| 影响范围扩大 | 升级 | 某功能最初只有少量用户使用，后续全面推广时升级 |
| 出现有效规避方案 | 降级 | 某P1问题有了临时规避方案，可降为P2 |
| 修复成本超出预期 | 评估 | 某P1问题修复需要重构核心模块，需重新评估 |
| 临近大促/版本 | 升级 | 双11前两周，所有影响用户体验的缺陷至少P1 |

## 3. 严重程度 vs 优先级的关系与差异

### 3.1 核心区别

| 维度 | 严重程度 (Severity) | 优先级 (Priority) |
|------|---------------------|-------------------|
| **定义角度** | 技术影响程度（客观） | 修复紧急程度（主观+客观） |
| **决定者** | 测试人员 | 项目经理/产品/测试负责人 |
| **变化性** | 固定不变 | 可随时间调整 |
| **关注点** | 缺陷本身有多严重 | 缺陷应该什么时候修 |
| **衡量标准** | 对系统功能的破坏程度 | 对业务目标的威胁程度 |
| **类比** | 病情的严重程度 | 挂号的优先级 |

### 3.2 典型关系矩阵

|  | P0-立即 | P1-高 | P2-中 | P3-低 |
|------|---------|-------|-------|------|
| **S1-致命** | 绝大情况 | 罕见（仅影响测试但不影响线上） | 不应出现 | 不应出现 |
| **S2-严重** | 高频（线上核心功能） | 常规匹配 | 极罕见 | 不应出现 |
| **S3-一般** | 罕见（大规模用户投诉） | 常见 | 常规匹配 | 罕见 |
| **S4-轻微** | 不应出现 | 极少（品牌影响大） | 常见 | 常规匹配 |
| **S5-建议** | 不应出现 | 不应出现 | 偶尔 | 常规匹配 |

### 3.3 高严重程度低优先级场景（S2 + P2/P3）

```
场景1：S2严重 + P3低
  - 缺陷：核心报表导出功能在IE11以下浏览器中报错
  - 原因：公司用户中IE11以下占比<0.1%，且提供Chrome推荐引导
  - 结论：严重程度保持S2，但优先级定为P3

场景2：S2严重 + P2中
  - 缺陷：特定复杂查询条件下，搜索结果缺失部分数据
  - 原因：该查询条件为高级功能，使用频率极低，且可通过分步查询规避
  - 结论：严重程度保持S2，优先级定为P2
```

### 3.4 低严重程度高优先级场景（S4 + P1）

```
场景1：S4轻微 + P1高
  - 缺陷：首页Logo图片变形
  - 原因：大促活动期间首页流量暴增，品牌形象至关重要
  - 结论：严重程度S4，但业务优先级定为P1

场景2：S4轻微 + P1高
  - 缺陷：App启动页文案有错别字（"优惠卷"应为"优惠券"）
  - 原因：重大版本发布，媒体关注度高，文案错误影响品牌信任
  - 结论：严重程度S4，但优先级定为P1
```

## 4. 不同行业的缺陷等级标准

### 4.1 金融行业（银行/证券/保险）

金融行业在标准五级之上增加了**合规**和**资金**两个加重因素：

| 加重因素 | 说明 | 调整规则 |
|----------|------|----------|
| 合规风险 | 违反监管规定（央行/银保监/证监会） | 自动升级一档 |
| 资金风险 | 涉及金额计算/转账/清算 | 自动升级一档 |
| 审计追溯 | 影响审计日志完整性 | 自动升级一档 |

**金融行业特化标准：**

| 场景 | 标准分类 | 金融行业分级 | 说明 |
|------|----------|-------------|------|
| 利息计算偏差0.01元 | S2/S3 | **S1** | 涉及资金，零容忍 |
| 客户KYC信息字段缺失 | S3 | **S1** | 合规风险 |
| 交易日志未记录 | S3 | **S1** | 审计追溯 |
| 非核心页面报错 | S3 | S3 | 不变 |
| 管理后台报表延迟 | S4 | S3 | 运营影响 |
| UI文案问题 | S4 | S4 | 不变 |

### 4.2 游戏行业

游戏行业关注**付费**、**体验**、**公平性**三个特殊维度：

| 维度 | S1触发条件 |
|------|-----------|
| 付费 | 充值不到账、重复扣费、道具未发放、货币凭空消失 |
| 核心玩法 | 无法进入游戏、战斗系统崩溃、卡死/闪退 |
| 经济平衡 | 道具/货币可被刷取、交易系统漏洞、拍卖行异常 |
| 公平性 | 外挂漏洞、PvP作弊、利用Bug获取不正当收益 |

**游戏行业特化分级：**

| 问题 | 传统分级 | 游戏行业分级 | 原因 |
|------|----------|-------------|------|
| UI重叠遮挡按钮 | S4 | **S2** | 影响操作体验和留存 |
| 特定技能释放卡顿 | S3 | **S2** | 影响战斗平衡/公平性 |
| 皮肤/时装显示异常 | S4 | **S2** | 付费体验问题 |
| 排行榜更新延迟 | S3 | **S2** | 社交竞争核心体验 |
| 新手引导错别字 | S4 | S3 | 影响新手留存 |

### 4.3 电商行业

电商行业在**大促期间**和**平时**采用双标准：

| 场景 | 平时等级 | 大促期间等级 | 调整原因 |
|------|----------|-------------|----------|
| 商品详情页加载慢3秒 | S4 | **S2** | 转化率影响巨大 |
| 优惠券领取失败 | S3 | **S1** | 直接影响GMV |
| 购物车偶发清空 | S2 | **S1** | 直接影响下单 |
| 推荐算法异常 | S3 | S3 | 非核心路径 |
| 搜索排序偏差 | S3 | **S2** | 影响搜索转化 |
| 评价图片加载失败 | S4 | S3 | 影响决策但不阻断 |

**电商核心链路缺陷从重原则：**
```
浏览 → 搜索 → 商品详情 → 加购 → 下单 → 支付 → 履约

此链路上任何环节的缺陷自动 +1 严重程度
```

### 4.4 医疗行业

医疗行业实行**三档分级**（高于通用标准）：

| 等级 | 定义 | 包含通用标准的 |
|------|------|---------------|
| **A级-致命** | 危及患者安全/生命、违反医疗法规 | S1全部 + 涉及诊疗决策的S2 |
| **B级-严重** | 影响诊疗流程、数据完整性 | S2全部 + 涉及医嘱/处方的S3 |
| **C级-一般** | 不影响诊疗的其他问题 | S3+S4+S5 |

**医疗行业零容忍清单：**
1. 患者信息显示错误（张冠李戴）
2. 药品剂量显示异常
3. 检验结果与患者不匹配
4. 医嘱状态流转错误
5. 过敏信息未正确展示
6. 手术/检查预约时间错误

以上任何一项均自动定为A级致命，P0立即修复。

## 6. 缺陷等级判定实战案例

### 案例1：支付回调丢失导致订单状态不更新

**背景：**
某电商App，用户支付成功（支付宝已扣款），但App内订单一直显示"待付款"，30分钟后仍未更新。经排查，支付宝回调接口偶发500错误。

**分析维度：**
| 维度 | 评估 | 结论 |
|------|------|------|
| 系统崩溃？ | 否 | - |
| 数据丢失？ | 否（支付已成功，仅是状态未更新） | - |
| 核心业务阻断？ | 是（用户无法查看已支付订单、无法退款） | → |
| 有规避措施？ | 是（客服可手动查询支付宝记录确认） | - |
| 影响范围？ | 约5%的支付订单 | 部分用户 |
| 资金风险？ | 是（用户担心资金损失，引发大量投诉） | 加重 |

**最终判定：**
- 严重程度：**S2-严重**（核心业务异常，但有支付宝端可查的规避措施）
- 优先级：**P0-立即修复**（涉及支付和资金安全感知，客服投诉暴增）
- 处理结果：紧急修复回调接口+增加回调重试机制+补偿脚本补推积压订单

---

### 案例2：商品搜索结果显示已下架商品

**背景：**
用户搜索"手机"关键词，搜索结果中包含了大量已下架的商品（商品状态为"已删除"），用户点击后提示"商品不存在"。

**分析维度：**
| 维度 | 评估 |
|------|------|
| 核心功能？ | 是（搜索是电商核心功能） |
| 完全阻断？ | 否（搜索功能可用，只是结果不准确） |
| 有规避措施？ | 部分（用户可浏览分类或直接访问有效商品） |
| 用户体验影响？ | 严重（搜索结果可靠性降低，转化率下降） |
| 影响范围？ | 约30%的搜索结果 |

**最终判定：**
- 严重程度：**S2-严重**（搜索结果数据质量异常，核心功能受损）
- 优先级：**P1-高**（影响用户体验和转化率，26%搜索结果受影响）
- 根因：搜索引擎索引未及时同步商品状态变更
- 修复方案：增加商品状态变更实时同步机制+定时全量重建索引

---

### 案例3：个人中心页面在iOS 17.2上白屏

**背景：**
iPhone 14 Pro升级iOS 17.2后，App个人中心页面白屏，其他页面正常。开发排查发现是iOS 17.2 Safari Webkit对CSS Grid的兼容性变更导致。

**分析维度：**
| 维度 | 评估 |
|------|------|
| 核心功能？ | 否（个人中心是辅助页面，主流程不受影响） |
| 完全阻断？ | 否（查看订单可通过首页入口） |
| 影响范围？ | 约3%用户（iOS 17.2+用户） |
| 有规避措施？ | 是（可通过其他入口访问订单、设置等功能） |
| 兼容性影响？ | 可能随iOS 17.2普及率提升而扩大 |

**最终判定：**
- 严重程度：**S3-一般**（非核心功能，有规避措施）
- 优先级：**P1-高**（虽然影响范围小但涉及兼容性，且iOS新版本用户活跃度高）
- 修复方案：CSS Grid降级方案+全面检查iOS 17.2兼容性

---

### 案例4：运营后台报表导出偶发超时（30秒超时阈值）

**背景：**
运营人员使用后台导出近一年订单报表时，50%概率触发30秒超时报错。重试1-2次后通常可以成功。数据量越大成功率越低。

**分析维度：**
| 维度 | 评估 |
|------|------|
| 核心功能？ | 否（内部运营工具，非用户侧） |
| 完全阻断？ | 否（重试可成功） |
| 影响范围？ | 约5名运营人员 |
| 数据准确性？ | 成功导出的数据准确 |
| 有规避措施？ | 是（分月导出、重试） |

**最终判定：**
- 严重程度：**S3-一般**（非核心功能，有有效规避措施）
- 优先级：**P2-中**（影响小且可规避，下个迭代优化）
- 优化方案：改为异步导出+导出进度通知+SQL优化加索引

---

### 案例5：注册页面提示"用户名已存在"但未说明具体原因

**背景：**
用户注册时使用"张三"作为用户名，提示"注册失败"，未告知具体原因。用户尝试多次更换用户名均失败后放弃注册。实际原因是用户名仅支持英文和数字。

**分析维度：**
| 维度 | 评估 |
|------|------|
| 核心功能？ | 是（注册是用户转化第一步） |
| 完全阻断？ | 间接（错误提示不清晰导致用户放弃） |
| 影响范围？ | 所有使用中文用户名的潜在用户 |
| 有规避措施？ | 无（用户不知道要输入英文数字） |
| 用户体验？ | 极差（可能导致用户流失） |

**最终判定：**
- 严重程度：**S3-一般**（功能可用，但错误提示不足）→ 考虑用户转化影响后判定为 **S2-严重**
- 优先级：**P1-高**（直接影响新用户注册转化率，属于增长关键环节）
- 修复方案：
  1. 前端实时校验+即时提示"用户名仅支持英文和数字"
  2. 后端错误码细化+前端友好展示
  3. 增加用户名输入框Placeholder示例

**关键教训：** 错误提示不足看似是S4文案问题，但放在转化漏斗关键环节，其影响可达到S2级别。判定等级时需要考虑**用户路径位置**。

## 本章小结

1. **严重程度与优先级是两个独立维度**：严重程度（S1-S5）由测试人员根据技术影响客观判定、固定不变；优先级（P0-P3）由PM/测试负责人综合业务影响和修复成本决定、可动态调整。理解两者的关系矩阵是避免开发与测试之间等级争议的关键。
2. **五级严重度标准是缺陷管理的基石**：S1致命（系统崩溃/数据丢失/核心业务阻断）、S2严重（核心功能错误无规避）、S3一般（有规避措施）、S4轻微（UI/文案）、S5建议（改进建议）。每个等级都有明确的判定条件和处理SLA。
3. **不同行业需要特化标准**：金融行业增加合规和资金加重因素；游戏行业关注付费、体验和公平性；电商在大促期间采用升级标准；医疗行业实行三档分级和零容忍清单。
4. **等级判定需要结构化的决策流程**：从"系统崩溃吗？"到"是建议吗？"的决策树帮助团队快速一致地判定，争议时由测试负责人（严重程度）或产品经理（优先级）最终裁定。
