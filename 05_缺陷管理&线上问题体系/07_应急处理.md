# 应急处理 (Emergency Response)

> **线上事故应急处理**是技术团队最重要的实战能力之一。应急处理的质量直接决定了事故的影响范围和持续时间。本章提供从理论到实践的完整应急处理体系。

## 1. 线上事故应急响应流程

### 1.1 应急响应全流程

```
═══════════════════════════════════════════════════════════
                应急响应全流程 (ICS框架)
═══════════════════════════════════════════════════════════

【5秒-发现阶段】
  ┌── 监控告警触发
  ├── 用户/客服反馈
  ├── 内部人员发现
  └── 外部通知(安全机构/监管机构)

         │
         ▼

【5分钟-响应阶段】
  ┌── 值班人员确认事故真实性（排除误报）
  ├── 初步判断严重等级 (P0/P1/P2/P3)
  ├── 建立应急沟通渠道（应急群/War Room/电话会议）
  ├── 明确角色: 事故指挥官(IC) / 处理负责人 / 沟通负责人
  └── 发出初始通告

         │
         ▼

【15分钟-止损阶段】 ← 最关键的阶段
  ┌── 先止损！不要急于找根因！
  ├── 评估止损策略:
  │   □ 回滚? (最近有发布吗?)
  │   □ 降级? (能否关闭问题功能?)
  │   □ 限流? (是不是流量突增?)
  │   □ 切换? (切换到备用集群?)
  │   □ 扩容? (是不是容量不够?)
  │   □ 下线? (能否先下线问题功能?)
  ├── 执行止损操作
  ├── 验证止损效果（核心指标是否恢复？）
  └── 发出止损完成通告

         │
         ▼

【30分钟-排查阶段】 ← 止损后再找根因
  ┌── 收集信息: 日志 / 监控截图 / 配置变更记录 / 发布记录
  ├── 保留事故现场（在日志被轮转/监控过期前保存）
  ├── 技术排查:
  │   □ 最近有无发布/配置变更?
  │   □ 查看关联服务的监控面板
  │   □ 查看错误日志（按时间点）
  │   □ 查看慢查询/异常连接
  │   □ 必要时抓包分析
  ├── 定位根因
  └── 持续更新状态

         │
         ▼

【修复阶段】
  ┌── 制定修复方案
  ├── 评估修复风险（会不会引入新问题?）
  ├── 修复方案评审(Crash Review)
  ├── 执行修复（灰度/分批）
  ├── 验证修复效果（监控对比修复前后）
  └── 发出恢复通告

         │
         ▼

【善后阶段】
  ┌── 数据修复/补偿（如有需要）
  ├── 客服话术更新
  ├── 用户公告(如需要)
  ├── 事故复盘会议安排
  ├── 改进措施录入跟踪系统
  └── 事故报告输出
```

### 1.2 应急响应时间SLA

| 阶段 | P0 | P1 | P2 | P3 |
|------|----|----|----|-----|
| 发现→确认 | 5分钟 | 15分钟 | 30分钟 | 1小时 |
| 确认→止损 | 15分钟 | 30分钟 | 1小时 | 4小时 |
| 止损→恢复 | 30分钟 | 2小时 | 8小时 | 24小时 |
| **总计(TTR)** | **<1小时** | **<3小时** | **<10小时** | **<30小时** |
| 复盘报告 | 24小时 | 3个工作日 | 5个工作日 | 1周 |

### 1.3 事故指挥官(IC)决策框架

```
事故指挥官(IC)的决策清单:

1. 我刚知道这个事故，我需要:
   □ 确认这是一起真事故(不是误报)
   □ 评估影响范围和严重等级
   □ 拉人（开发/运维/测试/产品）
   □ 任命角色（处理人/沟通人/记录员）

2. 止损阶段，我需要决策:
   □ 回滚还是修复？（最近有发布则优先考虑回滚）
   □ 降级哪些功能？（非核心功能果断降级）
   □ 需要通知谁？（客服/运营/管理层/用户）

3. 排查阶段，我需要确保:
   □ 不要一个人闷头排查——保持沟通
   □ 记录时间线（每步操作的时间、执行人、效果）
   □ 每15-30分钟发出状态更新

4. 恢复阶段，我需要确认:
   □ 修复方案经过至少1人Review
   □ 修复后核心指标恢复到正常水平
   □ 不是"看起来好了"而是"数据证明好了"

5. 善后阶段，我需要:
   □ 安排复盘会议时间(24小时内)
   □ 确认是否需要用户补偿/公告
   □ 确认改进措施的负责人和期限
```

## 2. 应急处理原则

### 2.1 核心原则

#### 原则1：先止损后排查 (Stop the Bleeding First)

```
错误做法:
  事故发生了 → 开始看日志分析原因 → 花了30分钟找到根因 → 再修复
  → 事故持续了1小时，影响扩大了

正确做法:
  事故发生了 → 立即判断: 最近有发布吗？ → 有 → 立即回滚
  → 5分钟止损 → 然后再慢慢查根因
  → 事故只持续了10分钟

切记:
  你不一定要知道"为什么坏了"才能修好它。
  回滚到上一个已知稳定的版本永远是最快的止损方式之一。
```

#### 原则2：保留现场 (Preserve the Crime Scene)

```
在应急处理时，保留现场证据：

必须保留的现场信息:
  □ 应用日志 (发生时刻前后30分钟的日志)
  □ 数据库慢查询日志
  □ 监控面板截图 (各项指标的时间序列)
  □ Nginx访问日志/错误日志
  □ 配置变更记录 (配置中心变更历史)
  □ 发布记录 (发布时间、变更内容)
  □ 应急处理操作记录 (谁在什么时候做了什么)

保存方式:
  - 打包日志到独立目录 (不要等日志轮转)
  - 监控截图保存为图片
  - 聊天记录导出
  - 在事故文档中实时记录
```

#### 原则3：变更回滚优先 (Rollback First)

```
事故发生时，检查最近是否有变更:
  □ 代码发布? → 优先回滚
  □ 配置变更? → 优先还原配置
  □ 数据库变更? → 评估是否可回滚DDL/DML
  □ 基础设施变更? → 优先还原
  □ 第三方依赖升级? → 评估降级

"最近的变更是最可能的根因" —— 这是事故排查的铁律。
80%的生产事故与近期变更有关。
```

#### 原则4：单一指挥链 (Single Chain of Command)

```
应急处理中最忌讳的情况:
  - 多个人同时在下指令
  - 决策反复变更
  - 没人知道现在该听谁的

正确的指挥链:
  事故指挥官(IC) → 处理负责人 → 执行人员
  
  - IC做决策(回滚/降级/通知)
  - 处理负责人协调执行
  - 执行人员专注操作

IC的决策是最终的:
  - 可以讨论，但IC一旦做出决策，团队立即执行
  - 如果IC判断失误，事后复盘学习，不是现场争论的时候
```

#### 原则5：充分沟通 (Over-Communication)

```
沟通频率:
  P0: 每15分钟更新一次状态
  P1: 每30分钟更新一次状态
  P2: 每1小时更新一次状态

"没有消息"在事故中不是"好消息"，是"不知道发生了什么"。
即使没有进展，也要更新:"排查仍在进行中，当前已排除XX和YY可能"

状态更新模板:
  [时间] - [状态]
  当前状态: {排查中/止损中/修复中/已恢复}
  影响范围: {不变的/变化的}
  当前行动: {正在做什么}
  预计恢复: {不变/推迟/提前}
  下次更新: {预计时间}
```

### 2.2 常见错误与反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|----------|
| "等我找到根因再修" | 事故时间成倍增长 | 先止损(回滚/降级/限流)，再排查 |
| "这个问题以前也出现过" | 忽视可能的根因变化 | 每次都从零排查，但可以参考历史方案 |
| "让我试试这个" (无计划的试错) | 引入新问题，扩大影响 | 每次只做一件事，记录操作和结果 |
| "不要告诉用户" | 丧失信任，客服电话被打爆 | 诚实沟通，管理预期 |
| "大家都进来" (过多人参与) | 混乱，难以决策 | 核心5-8人，其他人等通知 |
| "我先重启试试" | 丢失现场证据 | 先收集日志和状态，再重启 |
| "修好了但我不知道为什么" | 问题会再次发生 | 必须找到根因，哪怕事后复盘 |

## 3. 应急预案制定

### 3.1 应急预案覆盖范围

```
应急预案应覆盖:

1. 基础设施故障
   ├── 数据库故障（主库宕机/从库延迟/数据损坏）
   ├── 缓存故障（Redis宕机/内存满/集群分裂）
   ├── 消息队列故障（MQ宕机/消息积压）
   ├── 网络故障（断网/DNS故障/负载均衡故障）
   ├── 存储故障（磁盘满/NAS故障/对象存储不可用）
   └── 容器平台故障（K8s节点故障/Pod驱逐）

2. 应用服务故障
   ├── 服务宕机（OOM/死锁/线程池满）
   ├── 性能恶化（慢查询/资源泄漏/雪崩）
   ├── 功能异常（逻辑Bug/数据错误/配置错误）
   └── 第三方依赖故障（支付/短信/地图/推送）

3. 安全事件
   ├── DDoS攻击
   ├── 数据泄露
   ├── 入侵事件（挖矿/勒索/后门）
   └── 证书/密钥泄露

4. 人为操作失误
   ├── 误删除（数据库/文件/配置）
   ├── 错误发布（错误版本/错误环境）
   └── 误操作（错误执行脚本/错误修改数据）
```

### 3.3 各场景速查预案

#### 预案1：数据库主库宕机

```
触发条件: 数据库主库不可达，应用大量数据库连接错误
止损方案A(推荐): 主从切换
  1. 确认主库状态: mysql -h master -e "SELECT 1" 失败
  2. 确认从库数据同步状态: SHOW SLAVE STATUS\G
     Seconds_Behind_Master 应尽量小
  3. 停止应用写入（避免主从切换期间数据不一致）
  4. 提升从库为新主库: STOP SLAVE; RESET SLAVE ALL;
  5. 更新应用数据库连接配置 → 重启或reload应用
  6. 恢复写入
  预计耗时: 5-10分钟

止损方案B(备选): 如果从库数据延迟>1分钟
  1. 优先尝试恢复主库（重启、检查磁盘/网络）
  2. 如主库短期无法恢复，评估数据丢失量后执行方案A
```

#### 预案2：Redis内存满

```
触发条件: Redis报错 OOM command not allowed when used memory > maxmemory
止损方案A(推荐): 分析+清理
  1. redis-cli --bigkeys 找出大Key
  2. 确认大Key是否可以删除 → 可删除 → DEL key
  3. 临时调大 maxmemory（如有内存余量）
  4. 如maxmemory-policy为noeviction → 临时改为allkeys-lru

止损方案B(备选): 重启Redis
  风险: 缓存全清，数据库压力暴增
  仅在前方案无法执行且无持久化(RDB/AOF)时考虑
  重启前: 先扩容应用实例或开启限流保护数据库
```

#### 预案3：DDoS攻击

```
触发条件: 流量突然暴增(>正常10倍)，大量异常请求
止损方案A: 高防IP/清洗
  1. 启用云服务商的高防IP或DDoS清洗服务
  2. 配置流量清洗策略（如已有）

止损方案B: 限流+封IP
  1. Nginx限流: limit_req_zone + limit_req
  2. 防火墙封禁高频IP: iptables 或云安全组
  3. CDN开启WAF(Web应用防火墙)的CC攻击防护

止损方案C: 静态化+降级
  1. 将动态页面切换为静态页面(CDN缓存)
  2. 关闭登录/注册/支付等需要后端的操作
  3. 通知用户"系统维护中"
```

#### 预案4：证书过期

```
触发条件: 浏览器/客户端提示证书错误
止损方案:
  1. 如支持Let's Encrypt: certbot renew (2分钟)
  2. 如使用CDN: 使用云服务商免费证书(5分钟生效)
  3. 如使用负载均衡: 替换证书文件 + reload Nginx (2分钟)
  4. 临时方案: 如为API域名(非用户浏览器访问)，临时切换为HTTP
     注意: 浏览器访问的域名绝不能降级为HTTP
```

#### 预案5：大量慢查询导致服务降级

```
触发条件: 应用响应慢，数据库CPU>80%，大量慢查询
止损方案:
  1. 立即限流: Nginx/QoS限制请求速率，或启用应用限流
  2. 杀掉慢查询: KILL QUERY {id} (先SHOW PROCESSLIST)
  3. 如果慢查询确定来自特定功能(如报表导出):
     临时下线该功能
  4. 扩容数据库连接池或增加只读实例
  5. 如果是索引缺失导致:
     紧急添加索引(评估影响，避免锁表)
```

## 4. 应急演练

### 4.1 演练类型

| 演练类型 | 说明 | 频率 | 参与人 |
|----------|------|------|--------|
| **桌面推演** | 围绕虚拟场景讨论响应流程，不实际操作 | 每月 | 全部On-Call人员 |
| **功能演练** | 针对特定场景实际操作（如主从切换） | 每季度 | 运维+相关开发 |
| **红蓝对抗** | 红队制造故障，蓝队发现并修复 | 每季度 | 蓝队(全员) + 红队(指定) |
| **全链路演练** | 模拟完整的P0事故全流程 | 每半年 | 全技术团队 |
| **混沌工程** | 在生产/预发布环境注入故障 | 持续/定期 | 架构+运维 |

### 4.2 红蓝对抗方案

**角色说明：**

```
红队 (Red Team - 攻击者/故障制造者):
  - 由架构师/资深运维组成（2-3人）
  - 负责设计故障场景
  - 在指定时间窗口注入故障
  - 评估蓝队响应质量
  - 职责：让系统"合理地"出问题

蓝队 (Blue Team - 防守者/应急响应):
  - 由On-Call团队组成
  - 不知道故障的具体内容
  - 通过监控告警发现故障
  - 按应急流程响应
  - 职责：快速发现、快速止损、快速恢复

白队 (White Team - 裁判/观察者):
  - 由技术负责人组成
  - 制定演练规则
  - 确保演练不失控
  - 评估蓝队表现
  - 职责：保证安全，公正评判
```

**红蓝对抗场景示例：**

```
场景1: "电商大促数据库故障"
  红队操作: 
    - 模拟数据库主库CPU 100%（执行高负载SQL）
    - 10分钟后触发监控告警
  观察点:
    - 蓝队多久发现？
    - 蓝队是否执行主从切换？
    - 切换过程是否影响用户？
    - 沟通是否及时？

场景2: "神秘新版本引发的服务雪崩"
  红队操作:
    - 修改某个服务的超时配置为30秒（原3秒）
    - 在上游服务注入慢请求
  观察点:
    - 蓝队能否识别雪崩模式？
    - 蓝队是否使用熔断/降级？
    - 蓝队是否先回滚配置？

场景3: "第三方支付通道中断"
  红队操作:
    - 在防火墙上临时屏蔽支付网关IP
  观察点:
    - 蓝队多久发现支付成功率下降？
    - 蓝队是否有切换支付通道方案？
    - 客服是否被告知？
```

### 4.3 全链路演练计划

**演练准备（演练前1周）：**

```
□ 确定演练日期和时间窗口（建议: 业务低谷期，如周六凌晨）
□ 发布演练通知（告知用户可能短暂影响）
□ 确认参演人员（IC/处理/沟通/记录角色分配）
□ 设计演练场景（包含监控触发→告警→响应→止损→恢复全流程）
□ 准备监控面板和日志收集
□ 设定"停止演习"的安全词（一旦演习影响超出预期立即终止）
□ 准备回滚方案（确保可快速恢复）
```

**演练执行流程：**

```
T-10分钟: 所有人员就位，确认沟通渠道
T-5分钟: 白队重申规则和安全词
T-0: 红队注入故障
T+N分钟: 监控告警触发
T+N+1: 值班人员确认并启动应急
T+N+5: IC就位，应急群建立
T+N+X: 止损完成
T+N+X+Y: 恢复完成
T+演习结束: 白队宣布演习结束
T+结束+30分钟: 即时复盘(Hot Wash)
```

**演习评估维度：**

|----------|-----------|------|

| 止损速度(MTTM) | | 确认到完成止损的时间 |
| 沟通质量 | | 状态更新是否及时、准确、清晰 |
| 决策质量 | | IC的决策是否正确、果断 |
| 协作配合 | | 团队分工是否清晰，配合是否顺畅 |
| 工具使用 | | 是否高效使用监控/排查/止损工具 |
| 规范性 | | 是否按照应急预案和流程执行 |
| 恢复完整性 | | 恢复后是否验证了所有关键指标 |

### 5.4 应急处理工具箱脚本

```bash
#!/bin/bash
# 一键诊断脚本 - 用于应急场景快速收集信息
# usage: bash quick_diag.sh

DIAG_DIR="/tmp/emergency_diag_$(date +%Y%m%d_%H%M%S)"
mkdir -p $DIAG_DIR

echo "===== 快速诊断开始 =====" | tee $DIAG_DIR/summary.txt
date | tee -a $DIAG_DIR/summary.txt
echo "" | tee -a $DIAG_DIR/summary.txt

# 1. 系统资源
echo "--- 系统负载 ---" | tee -a $DIAG_DIR/summary.txt
uptime | tee -a $DIAG_DIR/summary.txt

echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- CPU TOP5 ---" | tee -a $DIAG_DIR/summary.txt
ps aux --sort=-%cpu | head -6 | tee -a $DIAG_DIR/summary.txt

echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 内存 TOP5 ---" | tee -a $DIAG_DIR/summary.txt
ps aux --sort=-%mem | head -6 | tee -a $DIAG_DIR/summary.txt

echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 磁盘使用 ---" | tee -a $DIAG_DIR/summary.txt
df -h | grep -v "^tmpfs\|^devtmpfs" | tee -a $DIAG_DIR/summary.txt

echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 磁盘IO ---" | tee -a $DIAG_DIR/summary.txt
iostat -x 1 2 | tail -n +3 | tee $DIAG_DIR/iostat.txt
cat $DIAG_DIR/iostat.txt | tee -a $DIAG_DIR/summary.txt

# 2. 网络连接
echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- TCP连接状态统计 ---" | tee -a $DIAG_DIR/summary.txt
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn | tee -a $DIAG_DIR/summary.txt

echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 监听端口 ---" | tee -a $DIAG_DIR/summary.txt
ss -tlnp | grep -E "LISTEN" | tee -a $DIAG_DIR/summary.txt

# 3. 应用进程
echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 关键进程状态 ---" | tee -a $DIAG_DIR/summary.txt
ps aux | grep -E "java|nginx|node|python|gunicorn" | grep -v grep | tee -a $DIAG_DIR/summary.txt

# 4. 日志错误（最近30分钟）
echo "" | tee -a $DIAG_DIR/summary.txt
echo "--- 最近错误日志(30分钟) ---" | tee -a $DIAG_DIR/summary.txt
find /var/log -name "*.log" -type f -mmin -30 \
  -exec grep -l -E "ERROR|FATAL|Exception" {} \; 2>/dev/null | tee -a $DIAG_DIR/error_log_files.txt

# 5. JVM信息（如果应用是Java）
JAVA_PID=$(pgrep -f "spring|tomcat|jetty" | head -1)
if [ -n "$JAVA_PID" ]; then
    echo "" | tee -a $DIAG_DIR/summary.txt
    echo "--- JVM 堆内存 ---" | tee -a $DIAG_DIR/summary.txt
    jstat -gc $JAVA_PID | tail -1 | tee -a $DIAG_DIR/summary.txt
    
    echo "" | tee -a $DIAG_DIR/summary.txt
    echo "--- JVM 线程数 ---" | tee -a $DIAG_DIR/summary.txt
    jstack $JAVA_PID | grep "java.lang.Thread.State" | sort | uniq -c | sort -rn | tee -a $DIAG_DIR/summary.txt
fi

echo "" | tee -a $DIAG_DIR/summary.txt
echo "===== 诊断信息收集完成 =====" | tee -a $DIAG_DIR/summary.txt
echo "结果保存在: $DIAG_DIR" | tee -a $DIAG_DIR/summary.txt
```

### 案例1：大促0点全站崩溃 —— 回滚实战

**场景：**
大促0点开始，运营活动页面正式生效。2分钟后，网站和App同时出现大面积超时，核心API错误率达78%。

**应急过程：**

```
00:02 - 监控告警: 网站可用性降至22%
00:03 - 值班人员确认事故，判定P0，拉应急群
00:05 - IC确认: 最近变更 = 0点发布的活动版本 V4.5.0
00:06 - IC决策: "立即回滚到V4.4.0！"
00:07 - 运维执行回滚脚本
00:10 - 回滚完成，服务开始恢复
00:15 - 核心指标: 网站可用性→98%，API错误率→0.2%
00:20 - 全部指标恢复至正常水平

事后根因: 活动版本中新增的限量抢购功能存在死循环Bug，
         特定条件下耗尽CPU资源。

损失因回滚及时: 仅影响约15分钟，错失大促首分钟交易量。
如未及时回滚: 排查Bug+修复+重新发布可能耗时1-2小时。
```

**关键经验：** "先回滚后排查"原则在本次事故中发挥了决定性作用。IC在看到"2分钟前有发布"后，果断决策回滚而不是排查。

### 案例2：Redis大Key导致的间歇性故障 —— 降级+清理

**场景：**
用户反馈App首页有时正常、有时加载很慢（>10秒）。监控显示Redis的GET操作延迟间歇性飙升至5-8秒。

**应急过程：**

```
14:20 - 监控告警: Redis P99延迟 > 3秒
14:22 - 值班人员确认问题，判定P2（部分用户受影响）
14:25 - 排查Redis:
  redis-cli --bigkeys → 发现一个Hash Key "user:behavior:cache" 
  有500万field，占用内存2.8GB
14:28 - 排查该Key的使用场景:
  → 用户行为缓存，每次加载首页需要HGETALL整个Hash
  → 500万field的HGETALL操作导致Redis单线程阻塞
14:30 - IC决策: "先降级——首页暂时不加载行为推荐模块"
14:32 - 执行降级开关，首页使用默认推荐
14:35 - Redis延迟恢复正常 P99 < 5ms
14:40-15:30 - 开发修复:
  1. 将大Hash拆分为按用户粒度的小Hash
  2. 增加过期时间(7天)避免无限增长
16:00 - 修复上线，恢复行为推荐模块
```

**关键经验：** 无法立即修复的功能果断降级，优先保证核心体验（首页能加载），次要功能（个性化推荐）可以事后修复。

### 案例3：机房网络故障 —— 多机房切换

**场景：**
某天下午，整个A机房的网络出现间歇性丢包（30%），导致A机房部署的服务响应超时。但B机房正常运行。

**应急过程：**

```
15:10 - 监控告警: A机房所有服务错误率上升
15:12 - 值班人员发现A机房网络异常（ping B机房正常，ping A机房丢包30%）
15:14 - 联系运维/网络团队确认A机房核心交换机故障

15:25 - DNS生效(部分用户DNS缓存较长)

18:00 - 流量切回A/B双机房负载
```

**关键经验：** 多机房/多活架构在此次故障中展现了价值。单一机房故障时，能在分钟级完成流量切换。

### 案例4：错误的数据修复脚本 —— 止损+数据恢复

**场景：**
运营同事执行了一个数据修复脚本，本应修改100条测试订单，结果由于SQL WHERE条件写错，修改了10万条生产订单的状态（"已完成"被改为"已取消"）。

**应急过程：**

```
11:30 - 客服突然收到大量用户投诉:"我的订单怎么被取消了？"
11:35 - 值班人员排查发现order表大量数据被修改
11:37 - 查询数据库binlog，确认修改范围和修改时间
11:40 - IC决策: "立即暂停该数据库账号的写权限！"
          "从最近备份恢复被修改的数据！"
11:42 - DBA开始从备份恢复:
  1. 确认最近备份: 11:00的自动备份
  2. 将11:00-11:35之间新增的订单导出到临时表
  3. 从备份中恢复被修改的表
  4. 将临时表中的新订单导回
11:50 - 备份恢复完成
12:00 - 数据验证: 订单状态恢复至11:30之前的状态
12:10 - 客服通告: "订单状态已恢复，如仍有异常请联系客服"
12:30 - 全部恢复正常

事后复盘:
  - 生产数据库写权限未做限制（运营同事有直接写权限）
  - 数据修复脚本未经DBA Review
  - 数据修改操作无"仿真执行"（dry-run）环节
```

**关键经验：** 
1. 发现数据被错误修改后，第一反应是"停止继续写入"而非"立即修复"——防止二次破坏
2. 数据恢复优先使用备份+binlog而非手工逐条修复

### 案例5：第三方短信通道故障 —— 切换到备用通道

**场景：**
验证码短信发送成功率突然降至30%（正常>98%），大量用户无法完成注册和登录。

**应急过程：**

```

10:02 - 值班人员确认: 主短信通道（梦网）返回大量超时和拒绝
10:03 - 联系短信供应商确认: 其平台正在遭受DDoS攻击
10:05 - IC决策: "立即切换到备用通道（阿里云短信）"
10:06 - 运维修改短信服务配置:
  # 配置中心修改
  sms.default.channel: aliyun  # 原值: mengwang
10:07 - 推送配置到短信服务
10:08 - 短信服务热加载配置，开始使用备用通道
10:10 - 短信发送成功率恢复至 96%
10:15 - 通知客服和运营，更新用户公告（如需要）
11:30 - 主通道恢复，继续保持备用通道
次日 - 评估是否永久使用双通道方案
```

**关键经验：**
1. 第三方服务必须有多通道冗余（短信、支付、推送等都应如此）
2. 通道切换必须是热切换，不能重启服务（否则增加恢复时间）
3. 切换验证：切换后立即发送测试短信确认可用

### B. On-Call值班手册

```
值班人员职责:
  1. 保持告警接收设备可用（声音+震动，不能静音）
  2. 接到P0/P1告警5分钟内确认
  3. 按照应急流程响应
  4. 如无法处理，立即升级

值班交接:
  交班人:
    □ 交代当前进行中的事故/异常
    □ 交代今天发生的变更(发布/配置修改)
    □ 交代需要关注的监控趋势
  
  接班人:
    □ 确认告警接收设备正常
    □ 确认权限可用(服务器/数据库/配置中心)
    □ 确认应急联系人列表最新

值班不做什么:
  ✗ 不要在值班期间发布（除非是P0修复）
  ✗ 不要关闭告警不处理
  ✗ 不要一个人闷头排查超过15分钟不汇报
```

### C. 应急响应成熟度评估

```
Level 1 - 反应式
  - 用户发现故障，用户通知团队
  - 无正式应急流程
  - 无On-Call机制

Level 2 - 基础
  - 有基本监控告警
  - 有On-Call但响应时间不稳定
  - 应急处理依赖个人能力

Level 3 - 规范
  - 有完整的应急流程和SLA
  - 有应急预案覆盖主要场景
  - 有定期的故障演练
  - 有事故复盘和改进跟踪

Level 4 - 主动
  - 监控覆盖>95%
  - 应急预案覆盖率>90%
  - 季度红蓝对抗
  - 应急平台一键止损
  - MTTR持续下降

Level 5 - 自愈
  - 常见故障自动检测+自动止损
  - 混沌工程常态化
  - 系统韧性达到"故障是常态"水平
  - MTTR < 5分钟
```
