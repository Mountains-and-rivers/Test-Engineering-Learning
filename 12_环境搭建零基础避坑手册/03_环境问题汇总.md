## 1. Java环境常见问题

### 问题1：JAVA_HOME配置了但不生效

**现象**：命令行输入 `java -version` 正确，但 `echo %JAVA_HOME%` 显示的不是刚配置的路径，或Maven报找不到JAVA_HOME。

**根本原因：**
- 环境变量配置后没有重启终端（Windows终端不会自动刷新）
- 系统变量和用户变量存在同名覆盖
- Path中有其他路径包含java.exe（如 `C:\Program Files (x86)\Common Files\Oracle\Java\javapath`），优先级高于 %JAVA_HOME%\bin
- 环境变量中有拼写错误（如 `JAVA_HMOE` → `JAVA_HOME`）

**诊断步骤：**
```bash
# 1. 检查JAVA_HOME是否正确设置
echo %JAVA_HOME%           # Windows
echo $JAVA_HOME            # Mac/Linux

# 2. 检查实际使用的java路径
where java                 # Windows
which java                 # Mac/Linux

# 3. 检查java -version输出的JDK版本和路径
java -verbose -version     # 会输出加载的rt.jar路径

# 4. 检查Path中java相关的所有条目
echo %PATH% | findstr /i java    # Windows
echo $PATH | tr ':' '\n' | grep -i java  # Mac/Linux
```

**解决方案：**
1. **重启终端** — 90%的Windows环境变量问题都是因为没有重启终端
2. 将 `%JAVA_HOME%\bin` 放在Path的**最前面**
3. 删除 `C:\Program Files (x86)\Common Files\Oracle\Java\javapath` 目录中的java.exe
4. 或直接将该目录中的java.exe替换为正确版本的软链接

### 问题2：端口被占用（Address already in use）

**现象**：启动Tomcat/Jetty/Spring Boot应用时报错：
```
java.net.BindException: Address already in use: bind
或
Port 8080 is already in use
```

**根本原因：**
- 上次启动的Java进程未正常退出，端口仍被占用
- 其他程序（Nginx/其他Java应用）占用了相同端口
- IDE中多次运行，旧进程没有被IDE自动杀死

**诊断与解决：**
```bash
# Windows - 查找占用端口的进程
netstat -ano | findstr :8080
# 输出：TCP  0.0.0.0:8080  0.0.0.0:0  LISTENING  12345
# 12345 就是占用端口的PID
taskkill /F /PID 12345

# Mac/Linux - 查找占用端口的进程
lsof -i :8080
# 或
ss -tlnp | grep 8080
kill -9 <PID>

# 一键查找并杀死的脚本（Linux/Mac）
kill -9 $(lsof -t -i:8080) 2>/dev/null
```

### 问题3：OutOfMemoryError: Java heap space

**现象**：运行大型测试套件或性能测试时，Maven/TestNG/JUnit报错：
```
java.lang.OutOfMemoryError: Java heap space
```

**根本原因：**
- Maven Surefire插件默认JVM堆内存太小（通常256M-512M）
- 测试中加载大量测试数据或创建大量对象
- Selenium WebDriver实例未正确关闭导致内存泄漏

**解决方案：**
```xml
<!-- pom.xml中配置surefire插件的JVM参数 -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.2.5</version>
    <configuration>
        <argLine>-Xmx2048m -XX:MaxMetaspaceSize=512m -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=${project.basedir}/target/heapdump</argLine>
    </configuration>
</plugin>
```

```bash
# 或通过 MAVEN_OPTS 环境变量
export MAVEN_OPTS="-Xmx2048m -XX:MaxMetaspaceSize=512m"

# Gradle
# gradle.properties:
org.gradle.jvmargs=-Xmx2048m -XX:MaxMetaspaceSize=512m
```

### 问题4：UnsupportedClassVersionError

**现象**：
```
java.lang.UnsupportedClassVersionError: com/example/TestClass
has been compiled by a more recent version of the Java Runtime
(class file version 61.0), this version of the Java Runtime
only recognizes class file versions up to 55.0
```

**根本原因：** 编译时使用的JDK版本高于运行环境的JDK版本。

**Class版本对照表：**

| Class版本号 | JDK版本 | 说明 |
|-------------|---------|------|
| 52.0 | JDK 8 | |
| 55.0 | JDK 11 | |
| 59.0 | JDK 15 | |
| 61.0 | JDK 17 | 当前主流LTS |
| 65.0 | JDK 21 | 最新LTS |

**解决方案：**
```xml
<!-- pom.xml中指定编译版本 -->
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
</properties>

<!-- 或使用较新的maven-compiler-plugin -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.12.1</version>
    <configuration>
        <release>17</release>  <!-- JDK 9+ -->
    </configuration>
</plugin>
```

### 问题5：IDEA中Maven依赖无法下载（一直Resolving）

**现象**：IDEA打开Maven项目后依赖一直处于Resolving状态，右侧Maven面板显示红色波浪线。

**根本原因：**
- Maven仓库镜像源连接超时（特别是使用国外默认仓库时）
- 公司网络需要代理但Maven未配置
- 本地仓库缓存损坏

**解决方案：**
```xml
<!-- 1. 配置国内镜像（settings.xml） -->
<mirrors>
    <mirror>
        <id>aliyunmaven</id>
        <mirrorOf>central</mirrorOf>
        <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
</mirrors>

<!-- 2. 配置代理（如果有） -->
<proxies>
    <proxy>
        <id>company-proxy</id>
        <active>true</active>
        <protocol>http</protocol>
        <host>proxy.company.com</host>
        <port>8080</port>
    </proxy>
</proxies>
```

```bash
# 3. 清理损坏的本地缓存
# Windows:
rmdir /s /q %USERPROFILE%\.m2\repository\com\specific\package
# Mac/Linux:
rm -rf ~/.m2/repository/com/specific/package

# 4. IDEA中刷新
# Maven面板 -> 点击刷新按钮（Reload All Maven Projects）
```

### 问题6：Could not find or load main class

**现象**：
```
Error: Could not find or load main class com.example.TestRunner
Caused by: java.lang.ClassNotFoundException: com.example.TestRunner
```

**根本原因：**
- 测试类没有被编译（target/classes中无对应的.class文件）
- Maven/Gradle的测试输出路径配置不正确
- 包名与实际路径不一致
- Surefire插件配置的 `<includes>` 模式不包含该类

**解决方案：**
```bash
# 1. 先clean再compile
mvn clean compile

# 2. 检查target/classes是否有对应class
ls target/classes/com/example/TestRunner.class

# 3. 检查surefire的include配置
```
```xml
<plugin>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <includes>
            <include>**/*Test.java</include>
            <include>**/*Tests.java</include>
            <include>**/Test*.java</include>
        </includes>
    </configuration>
</plugin>
```

### 问题7：IDEA中Lombok不生效

**现象**：代码中使用了@Getter/@Setter等Lombok注解，但IDEA报红，编译报找不到getter/setter方法。

**解决方案：**
```
1. 确认pom.xml中已添加lombok依赖
   <dependency>
       <groupId>org.projectlombok</groupId>
       <artifactId>lombok</artifactId>
       <scope>provided</scope>
   </dependency>

2. IDEA安装Lombok插件：
   Settings -> Plugins -> 搜索"Lombok" -> Install -> Restart IDEA

3. 启用注解处理：
   Settings -> Build, Execution, Deployment -> Compiler -> Annotation Processors
   -> 勾选"Enable annotation processing"

4. 如果还不生效：File -> Invalidate Caches and Restart
```

### 问题8：日志框架冲突（SLF4J多重绑定）

**现象**：启动时控制台输出：
```
SLF4J: Class path contains multiple SLF4J bindings.
SLF4J: Found binding in [jar:file:/...logback-classic-1.4.14.jar!/...]
SLF4J: Found binding in [jar:file:/...log4j-slf4j-impl-2.23.0.jar!/...]
```

**根本原因：** Classpath中同时存在Logback和Log4j的SLF4J绑定，SLF4J无法确定使用哪个。

**解决方案：**
```bash
# 1. 使用 mvn dependency:tree 查看依赖来源
mvn dependency:tree | grep -E "logback|log4j|slf4j"

# 2. 排除多余的日志框架
```
```xml
<dependency>
    <groupId>some.library</groupId>
    <artifactId>some-artifact</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.apache.logging.log4j</groupId>
            <artifactId>log4j-slf4j-impl</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

### 问题9：IDEA缓存文件损坏导致各种奇怪报错

**现象**：IDEA突然出现各种之前没有的错误——找不到类、红线满屏、代码补全失效。

**解决方案**：
```
File -> Invalidate Caches...
-> 勾选 "Clear file system cache and Local History"
-> 勾选 "Clear VCS Log caches and indexes"
-> Invalidate and Restart

注意：此操作会清除本地历史记录，但不会影响Git记录
此操作能解决IDEA中80%的"莫名其妙"问题
```

### 问题10：Gradle Daemon内存不足或卡死

**现象**：Gradle构建长时间卡住，或报错 `Gradle build daemon disappeared unexpectedly`。

**解决方案：**
```bash
# 停止所有Gradle Daemon
./gradlew --stop

# 配置daemon内存
# gradle.properties:
org.gradle.daemon=true
org.gradle.jvmargs=-Xmx2048m -XX:MaxMetaspaceSize=512m

# 如果Daemon反复崩溃，可以禁用daemon进行排查
./gradlew build --no-daemon
```

## 2. Python环境常见问题

### 问题1：pip install 报SSL证书错误

**现象**：
```
SSLError: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed
```

**根本原因：**
- 公司网络使用了自签名证书或中间人代理
- 系统CA证书过期
- Python未找到系统的证书存储位置

**解决方案：**
```bash
# 临时绕过（仅限开发环境，不推荐）
pip install --trusted-host pypi.org --trusted-host files.pythonhosted.org <package>

# 永久配置（pip.ini / pip.conf）
[global]
trusted-host = pypi.org
               pypi.python.org
               files.pythonhosted.org

# 更新证书（Mac）
# 在Python安装目录执行：
/Applications/Python 3.12/Install Certificates.command
```

### 问题2：Python多版本共存混乱

**现象**：系统中安装了多个Python版本，`python`、`python3`、`python3.11` 指向各不相同，虚拟环境混乱。

**诊断：**
```bash
# 查看所有Python的位置
where python          # Windows
which -a python python3    # Mac/Linux

# 查看实际版本
python --version
python3 --version
/usr/bin/python3 --version
```

**解决方案：使用pyenv统一管理**
```bash
# 安装pyenv
# Mac:
brew install pyenv
# Linux:
curl https://pyenv.run | bash

# 查看可安装的Python版本
pyenv install --list | grep "^\s*3\.1"

# 安装所需版本
pyenv install 3.10.14
pyenv install 3.11.8
pyenv install 3.12.2

# 设置全局和项目级Python版本
pyenv global 3.11.8          # 全局默认
pyenv local 3.12.2           # 当前目录使用3.12.2（优先级高于global）

# 确保pyenv的shim在PATH最前面
# ~/.bashrc 或 ~/.zshrc:
eval "$(pyenv init -)"
```

### 问题6：import报ModuleNotFoundError但pip list中有包

**现象**：`pip list` 显示已安装，但 `import xxx` 报 ModuleNotFoundError。

**根本原因：**
- 安装到了不同的Python环境（全局Python vs 虚拟环境）
- IDE使用了不同的Python解释器
- 系统PATH中多个Python版本优先级混乱

**诊断与解决：**
```bash
# 1. 确认当前使用的Python解释器
which python
python --version

# 2. 确认包安装到了哪里
pip show <package>
# 查看 Location 字段：确认路径与当前Python匹配

# 3. 确认当前Python的sys.path
python -c "import sys; print('\n'.join(sys.path))"

# 4. 如果包装到错误的Python版本
# 使用正确的pip安装：
python -m pip install <package>     # 始终使用python -m pip
# 而非直接 pip install
```

### 问题7：Pytest找不到测试用例

**现象**：运行 `pytest` 显示 `collected 0 items`，但目录下确实有测试文件。

**诊断与解决：**
```bash
# 1. 检查文件名是否符合pytest发现规则
# 默认规则：test_*.py 或 *_test.py

# 2. 检查类和函数名
# 函数：test_ 开头
# 类：Test 开头（不含__init__方法）

# 3. 使用--collect-only查看哪些用例被收集
pytest --collect-only

# 4. 检查conftest.py中的配置
# conftest.py中是否配置了python_files等

# 5. 检查pytest.ini/pyproject.toml
# [tool.pytest.ini_options]
# testpaths = ["tests"]
# python_files = ["test_*.py"]

# 6. 手动指定测试路径
pytest tests/
```

### 问题8：Python版本导致的语法错误

**现象**：在Python 3.12环境下运行别人的代码报语法错误。

**常见不兼容语法变更：**

| 语法变更 | 影响版本 | 说明 |
|----------|----------|------|
| `async`/`await` 成为关键字 | 3.7+ | 老代码中变量名用了async/await |
| `from __future__ import annotations` | 3.11+ | 一些类型注解行为变化 |
| f-string中反斜杠 | 3.12+ | 3.12之前在f-string中不能直接使用`\` |
| `distutils` 被移除 | 3.12+ | 老package可能依赖distutils |
| `imp` 模块被移除 | 3.12+ | 用importlib替代 |

### 问题9：conda环境与pip混用导致的问题

**现象**：在conda环境中使用pip安装包后，出现依赖冲突或包不可用。

**最佳实践：**
```bash
# conda环境中的包管理优先级：
# 1. 优先用conda install
# 2. conda找不到的包才用pip install

# 不要再次 conda install numpy（会覆盖pip安装的依赖树）

# 导出环境（包含pip包）
conda env export > environment.yml      # 导出conda+所有pip包
```

### 问题10：pip安装报"EnvironmentError: [Errno 13] Permission denied"

**现象**：在Linux/Mac上运行 `pip install` 报权限错误。

**解决方案：**
```bash
# 永远不要用 sudo pip install（风险极高）

# 正确方法1：使用用户安装

# 正确方法2：使用虚拟环境（推荐）

# 如果已经用sudo破坏了系统pip：
# 修复：
python -m pip install --user --upgrade pip
```

## 3. Docker环境常见问题

**现象**：`docker pull` 速度只有几十KB/s，或直接超时报错。

**解决方案：配置镜像加速**
```json
// Docker Desktop -> Settings -> Docker Engine
{
  "registry-mirrors": [

    "https://dockerhub.timeweb.cloud",
    "https://docker.1ms.run",
    "https://docker.xuanyuan.me"

}

// 或配置多个
{
  "registry-mirrors": [
    "https://mirror.ccs.tencentyun.com",
    "https://docker.m.daocloud.io"
  ]
}
```

**当前可用镜像加速器（2024-2025）：**

| 地址 | 提供方 |
|------|--------|

### 问题2：Docker Desktop启动失败（Windows）

**现象**：Docker Desktop 一直停留在"Starting..."，或报 WSL 相关错误。

**解决方案：**
```
1. 确认WSL2已正确安装
   wsl --status
   wsl --update

2. 确认虚拟化已启用（BIOS中）
   systeminfo | findstr "Hyper-V"
   # 如果未启用，在BIOS中开启 Intel VT-x / AMD-V

3. 重置Docker到出厂设置
   Docker Desktop -> 故障排查(Troubleshoot) -> Reset to factory defaults

4. 删除并重建WSL分发版
   wsl --unregister docker-desktop
   wsl --unregister docker-desktop-data
   # 重启Docker Desktop后会自动重建

5. 如果仍然失败，切换回Hyper-V后端
   Settings -> General -> 取消 Use WSL 2 based engine
```

### 问题3：容器之间无法通信

**现象**：docker-compose中定义的多个服务之间无法通过服务名互相访问。

**根本原因：**
- 容器不在同一个Docker网络中
- DNS解析配置错误
- 防火墙规则阻止

**诊断与解决：**

## 4. 数据库连接常见问题

### 问题1：Can't connect to MySQL server on 'localhost' (10061)

**现象**：客户端连接MySQL报 `Can't connect to MySQL server on '127.0.0.1' (10061)`。

**诊断步骤：**
```bash
# 1. 确认MySQL服务是否启动
# Windows:
services.msc -> 查找MySQL -> 状态应为"正在运行"
net start MySQL80      # 手动启动

# Mac:
brew services list | grep mysql
brew services start mysql

# Linux:
systemctl status mysql
sudo systemctl start mysql

# 2. 确认端口是否在监听
netstat -ano | findstr 3306          # Windows
lsof -i :3306                         # Mac
ss -tlnp | grep 3306                  # Linux

# 3. 确认是否绑定了正确的地址
# my.cnf中:
bind-address = 0.0.0.0   # 允许远程连接
# 而非 bind-address = 127.0.0.1
```

### 问题2：Access denied for user

**现象**：
```
Access denied for user 'root'@'localhost' (using password: YES)
```
或
```
Access denied for user 'tester'@'192.168.1.100'
```

**常见原因与解决：**
```sql
-- 1. 密码错误
-- 尝试重置密码（跳过认证模式启动MySQL后执行）
ALTER USER 'root'@'localhost' IDENTIFIED BY 'NewPassword123!';

-- 2. 用户没有从该主机连接的权限
-- 查看用户的host限制
SELECT user, host FROM mysql.user;
-- 创建允许从任意IP连接的用户
CREATE USER 'tester'@'%' IDENTIFIED BY 'password';
GRANT ALL PRIVILEGES ON testdb.* TO 'tester'@'%';
FLUSH PRIVILEGES;

-- 3. MySQL 8.0认证插件问题
-- 老客户端不支持caching_sha2_password，改用mysql_native_password
ALTER USER 'tester'@'%' IDENTIFIED WITH mysql_native_password BY 'password';
```

## 6. 跨平台兼容性问题

### 问题1：Windows和Unix路径分隔符不兼容

**现象**：Windows上编写的测试脚本在Linux上运行路径解析失败。

**解决方案：**
```java
// Java: 使用 File.separator 或 Paths
String path = "data" + File.separator + "testdata" + File.separator + "input.csv";
// 或使用 Paths（自动处理分隔符）
Path path = Paths.get("data", "testdata", "input.csv");

// Python: 使用 pathlib
from pathlib import Path
data_file = Path("data") / "testdata" / "input.csv"

// JavaScript: 使用 path 模块
const path = require('path');
const filePath = path.join('data', 'testdata', 'input.csv');
```

### 问题2：换行符不一致导致测试断言失败

**现象**：测试断言比较两个字符串时失败，对比发现一个有`\r\n`一个只有`\n`。

**解决方案：**
```java
// Java: 统一换行符
String normalized = input.replace("\r\n", "\n").replace("\r", "\n");

// Python: 使用 splitlines()
lines = content.splitlines()    # 自动处理所有换行符类型
```

### 问题3：文件编码问题（GBK vs UTF-8）

**现象**：在Windows上用记事本编辑的CSV文件在Linux上用UTF-8解析出现乱码。

**解决方案：**
```python
# Python: 读取时指定编码
with open('data.csv', 'r', encoding='gbk') as f:
    content = f.read()

# 自动检测编码
import chardet
with open('data.csv', 'rb') as f:
    result = chardet.detect(f.read())
    print(result['encoding'])    # 输出: GB2312 或 UTF-8
```

### 问题4：Windows文件名大小写问题

**现象**：Windows文件系统不区分大小写，但Linux区分。代码中 `import TestData` 在Windows上正常，Linux上报找不到模块。

**解决方案：**
- 文件名和import始终使用统一的大小写规则（推荐全小写+下划线）
- 在CI/CD中加入大小写检查步骤

### 问题5：Apple Silicon上的原生库兼容问题

**现象**：在M1/M2 Mac上运行某些工具崩溃，特别是依赖原生库的工具。

**解决方案：**
```bash

# 2. 对于Python，确认whl包有arm64版本

# 如果没有arm64版本，以x86_64模式运行

# 选择 aarch64 版本的JDK（如Azul Zulu for ARM64）
```

## 通用排查口诀

> **看日志、查端口、核版本、比环境、重启试**

- **看日志**：完整的错误堆栈比直觉更可靠
- **查端口**：`netstat/lsof/ss` 确认服务是否在监听
- **核版本**：Java/Python/Node/数据库版本是否匹配
- **比环境**：出问题的环境和正常环境有什么区别
- **重启试**：重启终端、重启服务、重启电脑——虽然简单但有效
