# 黑盒测试设计方法分类详全汇总

> **黑盒测试**（Black-Box Testing）：不关注程序内部结构和代码逻辑，将程序视为一个"黑盒子"，仅通过输入和输出来验证功能是否符合需求规格的测试方法。

---

## 分类体系说明

本文档将黑盒测试设计方法按照分析维度和组合复杂度划分为三大类，共计25种方法：

- **一、单参数基础分析类（5种）**：聚焦于单个输入参数的取值分析和验证，是黑盒测试的基础方法论。
- **二、多条件参数组合类（14种）**：聚焦于多个参数之间的取值组合与交互关系，核心挑战是参数组合爆炸下的高效用例选取。其中 AC/BC/EC/PC/OC 构成核心组合覆盖缩写体系。
- **三、流程/状态迁移组合类（6种）**：聚焦于业务流程步骤和对象状态变化的复合验证，关注多步骤多状态之间的流转路径覆盖。

三类方法并非互斥而是互补——实际项目中通常以基于需求测试法为框架，根据被测功能特点灵活选用各类方法组合使用。

## 1.1 等价类划分法

### 通俗释义

把数据分成好几组，同一组内数据程序处理逻辑一样，每组随便选 1 个数据做用例；分合法数据、非法数据，同时核对入参、页面返回、数据库存储三处数据。

### 详细解释

等价类划分法（Equivalence Partitioning）是一种系统化的测试用例设计技术，其核心思想是：将程序的输入域（也包括输出域和存储域）划分为若干个子集（等价类），每个子集中的任意数据对程序的处理逻辑是等效的。测试时，从每个等价类中选取一个代表性数据进行测试，就能代表该等价类中所有数据的测试效果。

**等价类划分有两个正交的维度，需要先理解二者的关系：**

**第一维：按"数据合法性"划分（有效 vs 无效）**

| 分类 | 定义 | 关键判断标准 |
|------|------|-------------|
| **有效等价类** | 符合程序/需求规格要求的、有意义的合法数据集合 | 数据应被系统正常接受并处理 |
| **无效等价类** | 不符合程序/需求规格要求的、无意义的非法数据集合 | 数据应被系统拒绝并给出明确错误提示 |

**第二维：按"等价域"划分（输入域 vs 输出域 vs 存储域）**

| 分类 | 关注对象 | 典型验证目标 |
|------|----------|-------------|
| **输入域等价** | 用户/外部系统传入的参数 | 入参校验、格式验证、类型检查 |
| **输出域等价** | 系统处理后返回/展示的结果 | 不同输入产生不同输出分类，输出映射正确性 |
| **存储域等价** | 落库后数据库字段的实际值 | 数据库字段类型、长度、精度、约束校验 |

**两个维度的正交关系（2 × 3 = 6 种组合）—— 等价类划分矩阵：**

|  | 输入域等价 | 输出域等价 | 存储域等价 |
|---|---|---|---|
| **有效等价类** | 合法手机号被接受<br>正常金额通过校验 | 选"博士" → 返回码=5<br>选"本科" → 返回码=3 | 'ACTIVE' 正确存入<br>99.99 精度正确存储 |
| **无效等价类** | 非法字符 → 被拒绝<br>空值 → 必填提示 | 非法输入 → 返回错误码<br>非法输入却返回了 200 | 超长字符串 → 被截断<br>NULL → NOT NULL 拒绝 |

> **如何理解这张矩阵**：每一个等价类同时落在"合法性维度"和"域维度"上。例如——
> - 手机号 "13800138000" 是 **「有效的」+「输入域」** 等价类
> - 超长字符串存入 VARCHAR(20) 字段是 **「无效的」+「存储域」** 等价类
> - 传入 "博士后" 却返回了 200 是 **「无效的」+「输出域」** 等价类（漏测场景）
>
> 划分时：**先定域（在哪个环节验证），再分有效/无效**。

**设计步骤：**
1. 分析需求规格，确定验证环节：输入环节（用户/接口传参）、输出环节（返回值/页面展示）、存储环节（数据库落库）
2. 在每个验证环节中，分别划分有效等价类和无效等价类（每个域至少1个有效、2个以上无效）
3. 为每个等价类编号（建议格式：`域缩写-合法性-序号`，如 `IN-V-01`=输入域-有效-01，`ST-INV-02`=存储域-无效-02）
4. 设计测试用例：同一域的有效等价类可合并到1个用例（减少用例数）；每个无效等价类单独设计1个用例（避免无效数据之间互相屏蔽）
5. 执行验证：入参校验 → 接口/页面返回校验 → 数据库落库校验，三层一致性核对

### 实际使用场景（5个）

**场景1：[输入域] 用户注册 - 手机号输入框**

- **等价域类型**：输入域等价 — 针对用户输入的手机号参数进行等价类划分
- **需求**：手机号为11位中国大陆手机号，以1开头，第二位为3/4/5/6/7/8/9
- **有效等价类（输入域）**：合法的11位手机号（如13800138000）→ 仅1个等价类
- **无效等价类（输入域）**：
  - 少于11位数字（如1380013800，10位）
  - 多于11位数字（如138001380001，12位）
  - 非1开头（如23800138000）
  - 第二位非3-9（如12000138000）
  - 包含非数字字符（如13800a38000）
  - 包含特殊符号（如138-0013-8000）
  - 为空（不输入任何内容）
  - 全角数字（如１２３００１３８０００，全角Unicode字符）
- **用例设计**：1个有效等价类 → 1个用例；8个无效等价类 → 各1个用例，共9个用例
- **验证点**：入参校验提示（输入域）→ 页面错误提示（输出域）→ 数据库 `phone` 字段正确存储（存储域）

**场景2：[输入域] 商品价格输入框**

- **等价域类型**：输入域等价 — 针对用户输入的价格参数进行等价类划分
- **需求**：商品价格范围为 0.01 - 999999.99 元，保留两位小数
- **有效等价类（输入域）**：
  - 最小金额 0.01（边界有效等价类）
  - 中间金额（如 99.99、1234.56，常规有效等价类）
  - 最大金额 999999.99（边界有效等价类）
- **无效等价类（输入域）**：
  - 0（免费商品需特殊标记，语义无效）
  - 负数（如 -1.00，范围无效）
  - 超过最大值（如 1000000.00，范围无效）
  - 超出小数位限制（如 99.999，三位小数，精度无效）
  - 非数字（字母、特殊符号、中文数字，类型无效）
  - 空值（必填校验无效）
  - 科学计数法（如 1e5，格式无效）
- **用例设计**：3个有效等价类 → 可合并为1个用例，或各1个用例；7个无效等价类 → 各1个用例
- **验证点**：前端输入校验（输入域）→ 接口返回提示（输出域）→ 数据库 `price DECIMAL(10,2)` 精度存储（存储域）

**场景3：[输出域] 学历下拉框选择**

- **等价域类型**：输出域等价 — 针对选择不同选项后，系统返回的存储结果和展示结果进行等价类划分
- **需求**：学历可选择 博士/硕士/本科/大专/高中及以下，后端映射为枚举值存储
- **有效等价类（输出域）**：不同合法选择应产生正确的输出映射（存储值 + 展示值）
  - 选择"博士" → 存储值=5，返回值显示"博士"
  - 选择"硕士" → 存储值=4，返回值显示"硕士"
  - 选择"本科" → 存储值=3，返回值显示"本科"
  - 选择"大专" → 存储值=2，返回值显示"大专"
  - 选择"高中及以下" → 存储值=1，返回值显示"高中及以下"
- **无效等价类（输出域）**：非法输入应触发系统产生正确的错误输出，而非静默处理
  - 不选择（空值/null）→ 应输出必填提示（如返回码 400 + "学历为必填项"）
  - 传入非法枚举值（如"博士后"、"其他"）→ 应输出参数非法错误码（如返回码 400 + "无效的学历选项"）
  - 传入数字型值（如 1、2、3）→ 若接口期望字符串型枚举值，应输出类型错误；若接口支持枚举序号映射，则应按序正确映射（需查阅接口规范确定预期行为）
- **用例设计**：5个有效等价类 → 各1个用例验证输出映射正确性；3个无效等价类 → 各1个用例验证错误输出行为
- **验证点**：入参校验（输入域）→ 接口返回 `education` 字段值和描述（输出域）→ 数据库 `education_code` 存储值与返回值一致（存储域）

**场景4：[存储域] 数据库字段约束 - 订单金额字段**

- **等价域类型**：存储域等价 — 针对数据库字段 `total_amount DECIMAL(10,2) NOT NULL DEFAULT 0.00` 的存储约束进行等价类划分
- **需求**：订单金额字段定义 `total_amount DECIMAL(10,2) NOT NULL`，校验写入数据库时各类值的行为
- **有效等价类（存储域）**：
  - 边界最小值 0.01（存储值应为 0.01）— 精度边界有效
  - 中间值 12345.67（存储值应为 12345.67）— 常规业务值有效
  - 边界最大值 99999999.99（存储值精确，8位整数+2位小数=10位总长度）— 位数边界有效
  - 默认值场景（不传值时存储 DEFAULT 0.00）— 默认行为有效
- **无效等价类（存储域）**：
  - 超出总位数（如 100000000.00，9位整数，DECIMAL(10,2)仅支持最多8位整数，应拒绝或截断报错）— 范围无效
  - 超出小数位精度（如 99.999，应四舍五入到 100.00 或报错）— 精度无效
  - NULL值（字段 NOT NULL，应拒绝或报错）— 非空约束无效
  - 负数（如 -50.00，应拒绝或报错）— 业务约束无效
  - 非数值类型（如字符串 "abc"、布尔值 true，应拒绝）— 类型无效
- **用例设计**：4个有效等价类 → 各1个用例；5个无效等价类 → 各1个用例，共9个用例
- **验证点**：直接通过 SQL INSERT/UPDATE 验证数据库约束行为（存储域）→ `SELECT total_amount` 确认最终存储值 → 应用层读取展示一致性（输出域）

**场景5：[存储域] 数据库字段 - 用户状态枚举字段**

- **等价域类型**：存储域等价 — 针对数据库字段 `status VARCHAR(20) NOT NULL` 配合 CHECK 约束的存储验证进行等价类划分
- **需求**：用户状态字段 `status VARCHAR(20) NOT NULL CHECK (status IN ('ACTIVE', 'INACTIVE', 'SUSPENDED', 'DELETED'))`，校验存储有效性
- **有效等价类（存储域）**：
  - 'ACTIVE'（正常存储）— 枚举有效
  - 'INACTIVE'（正常存储）— 枚举有效
  - 'SUSPENDED'（正常存储）— 枚举有效
  - 'DELETED'（正常存储）— 枚举有效
- **无效等价类（存储域）**：
  - 非法枚举值（如 'LOCKED'、'UNKNOWN'，不在CHECK约束列表中）— 枚举约束无效
  - 超过字段长度（如21个字符的字符串）— 长度约束无效
  - 恰好20个字符但不属于CHECK枚举的字符串（如 'PENDING_VERIFICATION'）— 枚举+长度边界组合，应被CHECK拒绝
  - NULL值（NOT NULL约束）— 非空约束无效
  - 空字符串 ''（可能绕过CHECK但不符合业务语义）— 语义无效
  - 大小写变体（如 'active'、'Active'）— 需确认区分行为，隐含BUG风险
  - 含前后空格的合法值（如 ' ACTIVE'、'ACTIVE '）— 格式有效性待确认
- **用例设计**：4个有效等价类 → 可合并同类为1-2个用例；7个无效等价类 → 各1个用例
- **验证点**：SQL INSERT 验证 CHECK 约束行为（存储域）→ `SELECT status` 确认最终存储值 → 应用层读取展示一致性（输出域）

## 1.2 边界值分析法

### 通俗释义

BUG 大多出在数值临界点，除正常临界值外，还要测远超范围的极端值，同时检查返回结果临界、前后端校验规则不一样带来的隐藏边界。

### 详细解释

边界值分析法（Boundary Value Analysis）是对等价类划分法的强力补充。实践证明，大多数软件缺陷都集中在输入/输出域的边界值附近。边界值分析的核心是"边界值、边界值±1"——程序员在写判断条件时最容易在 `<`、`<=`、`>`、`>=`、`==` 这些比较符上出错。

**边界值的四种类型：**

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **常规7点边界** | min-1, min, min+1, normal(中间值), max-1, max, max+1 | 数值范围校验 |
| **健壮边界** | 远超范围的极端值：极大值、极负值、0、空值、特殊字符 | 系统抗极端输入能力 |
| **输出边界** | 系统输出/返回结果的临界值 | 分页、报表、列表数量临界 |
| **隐式前后端边界** | 前端校验和后端校验规则不一致导致的隐藏边界 | js校验通过但后端校验不通过、数据库字段长度>接口限制等 |

**设计步骤：**
1. 确定输入/输出的边界值范围
2. 对每个边界取：边界值、边界值-1、边界值+1
3. 额外补充：空值、极端值（如 Integer.MAX_VALUE、Long.MIN_VALUE）、特殊值（如 0、-1、null）
4. 输出边界同样方法取值
5. 对比前后端校验规则，找出不一致的隐藏边界

### 实际使用场景（5个）

**场景1：分页列表 - 每页显示条数边界**

- **需求**：列表每页默认显示20条，用户可选择10/20/50/100条，数据总量不固定
- **边界测试点**：
  - 数据总量为0条时，列表展示空状态（输出边界：0条）
  - 数据总量为1条时，不应出现分页（输出边界：1条）
  - 数据总量恰好为20条时，应只有1页（输出边界：=pageSize）
  - 数据总量为21条时，应出现第2页（输出边界：pageSize+1）
  - 每页选择10条，数据总量为100条，验证第10页为最后页
  - 每页选择50条，数据总量为100条，验证共2页
  - 每页选择100条，大数据量场景（如10000条），验证分页性能
  - 手动输入每页0条、-1条、非数字、超大值（前端校验但后端默认值）

**场景2：优惠券满减规则边界**

- **需求**：全场满200减30，满500减100，满1000减250（每档优惠不叠加，自动取最高优惠）
- **边界测试点**：
  - 订单金额 199元 → 不享受优惠（min-1）
  - 订单金额 200元 → 满200减30，实付170（=min）
  - 订单金额 200.01元 → 满200减30，实付170.01（min+0.01）
  - 订单金额 499元 → 满200减30，应只减30而非减100（跨档边界验证 min-1）
  - 订单金额 500元 → 满500减100，实付400（=min）
  - 订单金额 500.01元 → 满500减100（min+0.01 档位切换）
  - 订单金额 999元 → 满500减100，验证是否误用1000档（跨档）
  - 订单金额 1000元 → 满1000减250，实付750
  - 订单金额 0元 → 异常场景（健壮边界）
  - 订单金额为负数 → 系统应拦截（健壮边界）

**场景3：文件上传大小限制**

- **需求**：支持上传图片，单文件 ≤ 5MB（注：需确认系统中MB的定义是 5×1024×1024 字节还是 5×1000×1000 字节，两者边界值不同），一次最多上传9张
- **边界测试点**：
  - 文件大小恰好 4.99MB → 上传成功（max-0.01）
  - 文件大小恰好 5.00MB → 上传成功（=max）
  - 文件大小恰好 5.01MB → 上传失败，提示超出限制（max+0.01）
  - 文件大小恰好 0KB → 空文件，能否上传？（min边界）
  - 文件大小 1字节 → 极小文件，能否上传？（min+1）
  - 文件大小 100MB → 远超限制（健壮边界，后端拦截）
  - 文件大小 1GB → 极端大文件（健壮边界，可能引发OOM）
  - 上传数量：9张 → 正好满（=max）
  - 上传数量：10张 → 超出限制（max+1）
  - **隐式前后端边界**：前端限制5MB，但后端Nginx限制1MB → 前端可选中5MB文件但上传到1MB时报413错误

**场景4：密码长度校验**

- **需求**：用户密码长度 6-20 位，必须包含字母和数字
- **边界测试点**：
  - 5位合法密码（如 a1b2c）→ 提示太短（min-1）
  - 6位合法密码（如 a1b2c3）→ 通过（=min）
  - 7位合法密码（如 a1b2c3d）→ 通过（min+1）
  - 19位合法密码 → 通过（max-1）
  - 20位合法密码 → 通过（=max）
  - 21位合法密码 → 提示太长（max+1）
  - 空密码（0位）→ 提示不能为空（健壮边界）
  - 1位密码 → 提示太短（健壮边界）
  - 100位密码 → 远超限制，后端可能截断导致绕过（健壮边界 + 隐式边界）
  - **隐式边界**：数据库字段设为 VARCHAR(20)，如果后端漏校验，写入时报错或静默截断

**场景5：活动时间范围**

- **需求**：限时抢购活动，2025-06-01 10:00:00 至 2025-06-03 23:59:59
- **边界测试点**：
  - 2025-06-01 09:59:59 → 活动未开始，商品显示原价（min-1秒）
  - 2025-06-01 10:00:00 → 活动开始，显示活动价（=min）
  - 2025-06-01 10:00:01 → 活动进行中（min+1秒）
  - 2025-06-03 23:59:58 → 活动进行中（max-1秒）
  - 2025-06-03 23:59:59 → 活动最后一秒，活动价（=max）
  - 2025-06-04 00:00:00 → 活动已结束，恢复原价（max+1秒）
  - **跨时区边界**：用户在不同时区（如UTC+0），系统时间边界是否正确
  - **隐式边界**：前端倒计时和后端时间可能不一致（服务器时间 vs 用户本地时间），导致前端显示活动进行中但后端拒绝下单

## 1.3 错误推测法

### 通俗释义

依靠过往项目踩坑经验、同类产品常见 bug，预判容易出错位置，额外补充测试用例，用来填补别的方法漏测的点。

### 详细解释

错误推测法（Error Guessing）是一种基于经验和直觉的测试技术。它不依赖系统化的规则或模型，而是依赖测试人员对系统的深入理解、对过往项目中积累的缺陷模式的认知，以及对行业中常见错误类型的了解。错误推测法的效果高度依赖于测试人员的经验水平。

> **ISTQB分类说明**：在ISTQB标准测试分类中，错误推测法（Error Guessing）被归类为**基于经验的测试技术（Experience-Based Testing）**，而非**基于规格的测试技术（Specification-Based Testing / 黑盒测试）**。基于经验的测试技术还包括探索性测试（Exploratory Testing）等。本文档将其纳入黑盒测试方法汇总，是因为国内测试实践中习惯将其作为黑盒测试的补充方法使用，但读者应了解在标准分类体系中的正确定位。

**错误推测的知识来源：**

| 来源 | 说明 |
|------|------|
| **历史缺陷数据库** | 本团队过往项目中记录的高频缺陷类型和位置 |
| **同类产品已知问题** | 竞品或同类型系统的已知漏洞和缺陷 |
| **技术栈常见陷阱** | 特定语言、框架、中间件的已知坑点 |
| **通用错误模式** | 如空指针、数组越界、并发问题、精度丢失等 |
| **边界条件遗漏** | 等价类和边界值分析可能覆盖不全的特殊情况 |
| **业务规则冲突** | 看似不相关但实际联动时产生冲突的规则 |

**常见的高风险测试靶点：**

- 输入框：SQL注入、XSS脚本注入、超长字符串、特殊Unicode字符、emoji
- 时间处理：闰年/非闰年、跨时区、夏令时、2038年问题、2月29日
- 金额计算：浮点数精度、负数金额、科学计数法、大额溢出
- 并发操作：重复提交、并发修改同一数据、库存超卖
- 网络异常：超时、断网、弱网、重连后的状态恢复
- 编码问题：不同字符编码乱码、特殊字符处理

**设计步骤：**
1. 回顾历史项目缺陷库和同类产品问题清单
2. 分析被测系统的技术栈和架构，列出常见陷阱
3. 识别业务规则中容易产生冲突的地方
4. 推测每个高风险位置可能的错误类型
5. 设计针对性测试用例，明确输入、操作路径、预期行为
6. 将错误推测用例作为其他系统化方法的补充，而非替代

### 实际使用场景（5个）

**场景1：搜索功能 - 注入与特殊字符攻击**

- **推测错误点**：搜索框是最常见的攻击入口，程序员可能未做参数化查询，或未对特殊字符做转义
- **测试用例**：
  1. SQL注入：输入 `' OR '1'='1` 或 `'; DROP TABLE products; --`，验证是否被防注入过滤
  2. XSS注入：输入 `<script>alert(1)</script>` 或 `<img src=x onerror=alert(1)>`，验证搜索结果页是否执行了脚本（应做HTML转义）
  3. 正则注入（ReDoS）：输入 `((((a)*)*)*)*b` 超长嵌套正则，验证是否导致服务CPU跑满
  4. 空字符注入：输入 `%00`（NULL字符），验证后端对不同语言（C/C++/Java/Python）的空字符处理
  5. LDAP注入：输入 `*)(uid=*))(|(uid=*`，如果搜索底层用到LDAP需验证
  6. 超长输入：输入10000个字符的搜索词，验证是否有长度限制和性能问题
  7. Unicode混淆：输入看起来像正常字符的Unicode特殊字符（如全角字母、零宽空格）

**场景2：支付系统 - 金额精度与并发**

- **推测错误点**：金额计算使用浮点数导致精度丢失，或并发处理不当导致资金问题
- **测试用例**：
  1. 浮点数精度：支付金额 0.1+0.2，验证是否等于0.3而非0.30000000000000004（应用BigDecimal/整数分）
  2. 并发重复支付：模拟用户在网络延迟下连点两次"支付"按钮，验证是否只扣款一次（幂等性验证）
  3. 支付回调与查询并发：模拟第三方支付回调到达的同时用户查询订单，验证订单状态最终一致性
  4. 负金额支付：传入金额=-100，验证系统是否拦截
  5. 科学计数法金额：传入金额=1e10，验证系统是否正确解析或拒绝
  6. 超时后成功：模拟支付请求在服务端已成功但客户端超时，用户再次支付，验证不会重复扣款
  7. 退款金额大于支付金额：验证退款校验逻辑
  8. 零金额支付：金额=0，验证是否允许（如0元购漏洞）

**场景3：登录认证 - 安全绕过**

- **推测错误点**：认证逻辑是安全的核心，程序员可能在Session管理、Token校验、密码策略上出现疏忽
- **测试用例**：
  1. 暴力破解：连续输入错误密码100次，验证是否有锁定机制和验证码
  2. Session未清除：登录后退出，使用之前的Session ID访问接口，验证是否被拒绝
  3. Token篡改：修改JWT Token中的用户ID字段，验证后端是否校验签名而非仅解析Payload
  4. 验证码绕过：抓包后删除验证码参数或不传验证码字段，验证后端是否严格校验
  5. 密码明文传输：抓包检查登录请求中密码是否明文传输（应使用加密或HTTPS）
  6. 记住我功能：勾选"记住我"登录后，删除持久化Cookie，验证是否仍能自动登录
  7. 多端登录：同一账号在手机和电脑同时登录，互踢逻辑是否正确
  8. 弱密码策略：使用123456、password等弱密码注册，验证密码强度校验

**场景4：数据导入导出 - 编码与格式**

- **推测错误点**：导入导出涉及文件格式、编码、大小等多种边界，容易在特殊数据上出问题
- **测试用例**：
  1. 编码冲突：导入UTF-8编码的文件但含GBK特有字符（如中文繁体字），验证是否乱码
  2. BOM头处理：导入带BOM头的UTF-8文件，验证首行是否出现乱码字符
  3. 空文件导入：导入0KB的空文件，验证提示信息是否友好
  4. 超大数据量导入：导入100万行Excel，验证是否超时、OOM、或部分成功部分失败
  5. 循环引用：导入含公式的Excel，单元格A1引用B1，B1引用A1，验证处理逻辑
  6. 特殊字符：单元格中包含换行符、逗号、引号（CSV格式），验证导入后字段是否错位
  7. 合并单元格：导入含合并单元格的Excel，验证数据映射是否正确
  8. 超大字段：某单元格内容超过数据库字段长度（如VARCHAR(255)但导入了500字符），验证是截断还是报错

**场景5：时间日期处理**

- **推测错误点**：日期时间处理BUG无处不在——时区、闰年、跨月、跨年、夏令时等
- **测试用例**：
  1. 闰年2月29日：选择2024-02-29（闰年），验证日期合法性；选择2025-02-29（非闰年），验证是否拒绝
  2. 跨时区：用户在UTC+8创建的数据，在UTC+0时区查看，创建时间显示是否正确（相差8小时）
  3. 夏令时切换：在美国时区选择夏令时切换日（2025-03-09 02:30:00——这个时刻不存在！），验证系统行为
  4. 2038年问题：在32位系统中设置日期为2038-01-20（超过int32范围2038-01-19），验证时间戳溢出
  5. 12月31日→1月1日跨年：在12月31日23:59:59秒下单，验证年份切换后数据统计、订单号生成是否正确
  6. 月末特殊日期：设置日期为每月31日，但在2月/4月/6月/9月/11月，验证自动调整为当月最后一天还是报错
  7. 服务器时间和数据库时间不一致：系统时间不同步导致的时间处理异常

## 1.4 基于需求测试法

### 通俗释义

逐条对照需求文档提炼测试点，是所有黑盒方法的基础，一般搭配其余方法一起使用。

### 详细解释

基于需求测试法（Requirements-Based Testing）是最基础、最重要的测试方法。它直接以需求规格说明书（PRD/SRS/用户故事）为依据，逐条分析需求中的功能点和非功能要求，提炼测试条件并设计测试用例。基于需求测试法是所有其他测试方法的基础——每个测试用例都要追溯到某个需求。

**基于需求测试的核心原则：**

| 原则 | 说明 |
|------|------|
| **需求可追溯性** | 每个测试用例必须对应一条或几条需求，建立需求→用例的追溯矩阵 |
| **需求完整性** | 确保每条需求至少有一个测试用例覆盖 |
| **需求正确性** | 验证系统行为与需求描述完全一致 |
| **需求明确性** | 需求本身不清晰、有歧义时，需在测试前和产品/需求方澄清 |

**需求测试的覆盖维度：**

| 维度 | 覆盖内容 |
|------|----------|
| **功能需求** | 业务流程、数据输入/输出、用户界面、接口交互 |
| **性能需求** | 响应时间、吞吐量、并发用户数、资源利用率 |
| **安全需求** | 认证、授权、数据加密、审计日志、隐私保护 |
| **可用性需求** | 易用性、可访问性、多语言、错误提示友好性 |
| **兼容性需求** | 浏览器、操作系统、设备、第三方系统版本 |
| **法规需求** | 数据保护法(GDPR/PIPL)、行业标准、审计合规 |
| **隐含需求** | 需求文档未明确但行业默认、用户合理预期的行为 |

**设计步骤：**
1. 获取最新版本的需求文档（PRD/用户故事/原型图/接口文档）
2. 需求评审：识别模糊、矛盾、遗漏的需求，与需求方确认
3. 需求分解：将每个需求拆解为原子级的可测试需求点
4. 测试点提炼：为每个可测试需求点提炼测试条件
5. 用例设计：基于测试条件，结合其他黑盒方法（等价类、边界值、场景法等）设计具体测试用例
6. 建立需求-用例追溯矩阵（RTM, Requirements Traceability Matrix）
7. 需求变更时同步更新测试用例和追溯矩阵
8. 测试执行后统计需求覆盖率（已测需求数/总需求数）

### 实际使用场景（5个）

**场景1：用户注册功能 - 逐条需求验证**

- **需求文档描述**（示例）：
  1. 用户可以使用手机号或邮箱注册
  2. 手机号需为11位中国大陆手机号，需短信验证
  3. 邮箱需为有效格式，需邮箱验证码验证
  4. 密码长度6-20位，必须包含字母和数字
  5. 需勾选"同意用户协议和隐私政策"方可注册
  6. 同一手机号/邮箱24小时内最多获取5次验证码
  7. 同一IP地址1小时内最多注册3个账号
  8. 注册成功后自动跳转到首页并处于登录状态
  9. 注册成功后发送欢迎短信/邮件

- **测试点提炼与用例设计**：

| 需求编号 | 需求点 | 测试用例 |
|----------|--------|----------|
| R1 | 手机号或邮箱注册 | TC1: 使用有效手机号注册 → 成功；TC2: 使用有效邮箱注册 → 成功 |
| R2 | 11位手机号+短信验证 | TC3: 输入10位手机号 → 提示格式错误；TC4: 输入正确手机号 → 收到验证码 |
| R3 | 邮箱格式+邮箱验证 | TC5: 输入无@邮箱 → 提示格式错误；TC6: 输入正确邮箱 → 收到验证码 |
| R4 | 密码6-20位含字母数字 | TC7: 123456(纯数字) → 提示需含字母；TC8: abcdef(纯字母) → 提示需含数字；TC9: abc123(6位混合) → 通过；TC10: a1(2位) → 提示过短 |
| R5 | 勾选协议 | TC11: 不勾选协议点击注册 → 提示需同意协议 |
| R6 | 24h内最多5次验证码 | TC12: 连续获取6次验证码 → 第6次提示"今日获取次数已达上限" |
| R7 | 同IP 1h内最多3个账号 | TC13: 同IP注册第4个账号 → 提示"操作频繁，请稍后再试" |
| R8 | 注册成功跳转+自动登录 | TC14: 注册成功 → 跳转首页 → 顶部显示用户昵称(已登录状态) |
| R9 | 欢迎短信/邮件 | TC15: 手机号注册成功 → 收到欢迎短信；TC16: 邮箱注册成功 → 收到欢迎邮件 |

- **衍生用例（结合其他方法）**：
  - 等价类+边界值：手机号无效等价类（12位/10位/含字母/空）、密码边界（5位/6位/20位/21位）
  - 场景法：注册全流程（选择注册方式→填写信息→获取验证码→输入验证码→设置密码→勾选协议→提交→验证跳转和登录态）

**场景2：报表导出功能 - 多维度需求验证**

- **需求描述**：
  1. 支持将当前查询结果导出为Excel(.xlsx)和PDF格式
  2. 导出的数据必须与页面显示数据完全一致（包括排序、筛选）
  3. Excel支持最多导出100万行数据，PDF最多导出100页
  4. 导出文件名格式：报表名称_导出时间_操作人.yyyyMMddHHmmss
  5. 超过1万行数据需异步导出，通过站内信/邮件通知用户下载
  6. 导出链接有效期24小时
  7. 需要有导出操作审计日志
  8. 单用户每小时最多导出10次

- **测试点与用例设计**：

| 需求编号 | 需求点 | 测试用例 | 所用方法 |
|----------|--------|----------|----------|
| R1 | xlsx/PDF两种格式 | TC1: 导出xlsx → 文件可正常打开，数据完整；TC2: 导出PDF → 文件可正常打开，排版正确 | 等价类(有效格式) |
| R2 | 数据一致性 | TC3: 页面按金额降序筛选 → 导出 → 验证导出文件的排序和筛选与页面一致 | 场景法 |
| R3 | 数据量上限 | TC4: 导出100万行xlsx → 成功；TC5: 导出1000001行 → 提示超限；TC6: 导出101页PDF → 截断或提示 | 边界值法 |
| R4 | 文件名格式 | TC7: 导出后检查文件名是否包含报表名称+时间戳+操作人 | 基于需求 |
| R5 | 异步导出+通知 | TC8: 导出15000行 → 提示"数据量较大，请稍后在通知中心下载"→ 导出完成后收到站内信 | 场景法+状态迁移 |
| R6 | 链接有效期24h | TC9: 导出完成后23h59min → 链接可下载；TC10: 导出完成后24h01min → 链接过期 | 边界值法 |
| R7 | 审计日志 | TC11: 检查日志表是否记录了：操作人、操作时间、导出参数、文件名、数据量、IP地址 | 基于需求 |
| R8 | 频次限制 | TC12: 1h内导出第10次 → 成功；TC13: 1h内导出第11次 → 提示"导出次数超限，请稍后再试" | 边界值法 |
| *补充* | 空数据导出 | TC14: 查询结果为空时点击导出 → 导出空文件还是提示"无数据可导出"？ | 错误推测法 |
| *补充* | 并发导出 | TC15: 同一账号同时触发2个导出任务 → 两个任务并行执行且互不影响 | 错误推测法 |

**场景3：消息推送功能 - 功能性+非功能性需求**

- **需求描述**：
  1. 支持按用户ID、用户标签、用户分群三种方式选择推送目标
  2. 推送内容支持文本+图片+链接
  3. 支持立即推送和定时推送
  4. 支持Android/iOS/Web三端同步推送

## 1.5 随机测试法

### 通俗释义

随机胡乱输入各类数据，多用于自动化跑稳定性，项目收尾查漏。

### 详细解释

随机测试法（Random Testing / Stochastic Testing）是指不按照预设的测试脚本或测试用例，通过随机生成输入数据、随机选择操作路径来测试系统的方法。虽然看似"胡乱测试"，但随机测试在特定场景下具有独特的价值——它能发现系统化测试方法覆盖不到的边界情况。

**随机测试的主要形式：**

| 形式 | 说明 | 典型应用 |
|------|------|----------|
| **Monkey Testing** | 模拟随机点击、滑动、输入等用户操作，长时间运行 | 移动APP稳定性测试 |
| **Fuzz Testing（模糊测试）** | 生成随机/半随机的畸形数据作为输入 | 安全测试、协议测试、API测试 |
| **随机数据驱动** | 在自动化框架中随机生成参数组合 | 回归测试、兼容性测试 |
| **探索性随机测试** | 测试人员凭直觉随机探索系统 | 项目收尾查漏补缺 |
| **混沌工程** | 在生产/类生产环境中随机注入故障 | 系统韧性/弹性验证 |

**随机测试的价值与限制：**

| 价值 | 限制 |
|------|------|
| 能发现意料之外的缺陷（unknowable unknowns） | 无法保证覆盖率，不能替代系统化测试 |
| 适合长时间跑稳定性、内存泄漏检测 | 用例不可复现（需要记录随机种子） |
| 低成本发现崩溃、挂起类严重缺陷 | 需要配套监控和日志记录机制 |
| 可自动化、可持续集成 | 随机数据可能触发不到深层业务逻辑bug |

**设计步骤：**
1. 明确随机测试的目标（稳定性/安全性/查漏补缺）
2. 确定随机范围（哪些模块、哪些输入域、哪些操作类型）
3. 设定随机参数：操作频率、测试时长、随机种子（用于复现）
4. 配置监控：崩溃日志、异常监控、资源监控（CPU/内存/磁盘）
5. 执行随机测试并记录所有异常
6. 对每个异常进行复现尝试（利用随机种子）→ 记录/提单
7. 将随机测试发现的高频缺陷模式反馈到其他系统化测试方法中

### 实际使用场景（5个）

**场景1：移动APP Monkey稳定性测试**

- **测试目标**：检测APP在随机操作下是否出现崩溃（Crash）、无响应（ANR）、内存泄漏
- **测试配置**：
  - 工具：Android Monkey / iOS UI AutoMonkey
  - 操作数：10000次随机事件
  - 事件比例：点击70% / 滑动15% / 按键10% / 系统事件5%
  - 限制包名：仅操作被测APP，不跳出
  - 运行时长：约2小时
- **重点关注**：
  - Crash日志捕获（native crash / java crash）
  - ANR日志（主线程阻塞超过5秒）
  - 内存使用趋势（是否存在持续增长不释放的内存泄漏）
  - 线程数变化（是否存在线程泄漏）
- **测试覆盖范围**：
  - 各页面随机跳转：首页→搜索→详情→购物车→结算→支付→我的→设置→...
  - 随机表单输入：搜索框输入随机字符串、评论框输入emoji、数量输入负值
  - 随机系统中断：来电、通知栏下拉、横竖屏切换、多任务切换、低电量弹窗
  - 随机网络状态：WiFi/4G/飞行模式/弱网切换
- **缺陷观察**：连续跑24小时后，内存从200MB涨到800MB → 内存泄漏；随机点击100次后闪退 → NullPointer异常

**场景2：API接口Fuzz测试**

- **测试目标**：检测API在畸形输入下是否出现500错误、信息泄露、拒绝服务
- **测试配置**：
  - 工具：自定义脚本 + REST Fuzzer
  - 被测接口：所有对外暴露的REST API接口
  - Fuzz策略：
    - 参数类型变异：期望int传入string、bool、null、空数组、超长string
    - 参数缺失：逐个删除必填参数
    - 参数重复：同一个参数传多次（如 `?id=1&id=2`）
    - 参数注入：传入SQL注入/命令注入/XSS payload
    - 边界Fuzz：传入 Integer.MAX_VALUE (2147483647)、Long.MIN_VALUE、NaN、Infinity
    - JSON结构变异：嵌套过深(>100层)、Key重复、不合法JSON
- **重点关注**：
  - 响应码：不应出现500 Internal Server Error（表示服务端未处理异常）
  - 错误信息：不应暴露堆栈跟踪、数据库结构、内部IP等敏感信息
  - 响应时间：输入不应导致服务端死循环或长时间阻塞
  - 数据一致性：畸形参数不应导致数据写入错误（如截断写入、乱码写入）
- **缺陷示例**：
  - `GET /api/user?id=-1` → 返回500 + 堆栈信息（暴露代码路径）
  - `POST /api/order` 传入超长商品名(100KB) → 数据库报错字段过长 → 返回500
  - `GET /api/search?q=%00` → 服务端C代码处理NULL字节崩溃

**场景3：搜索功能随机输入测试**

- **测试目标**：验证搜索功能对各类随机输入的容错性和稳定性
- **随机输入类型**：
  - 纯随机字符串：`aSDf23$@#!`、`ಠ_ಠ`（Unicode表情符）
  - 超长输入：10000个随机字符
  - 空字符：零宽空格(U+200B)、NULL字符(U+0000)、换行符\n、回车\r
  - SQL关键字随机组合：`SELECT FROM WHERE UNION DROP`
  - 编程语言关键字：`null undefined NaN Infinity true false`
  - URL编码字符：`%20%00%FF%FE`
  - 多种编码混合：中文+日文+韩文+阿拉伯文+emoji
  - 特殊正则符号：`.*+?^${}()|[]\`
- **验证点**：
  - 搜索不崩溃、不超时
  - 返回结果不包含代码注入的脚本执行
  - 高亮显示不因特殊字符而布局错乱
  - 搜索建议/自动补全不因随机输入而异常
  - 无结果时展示友好提示而非报错
- **缺陷示例**：
  - 搜索 `[]` → 后端正则编译异常，返回500
  - 搜索 emoji → 数据库字符集不支持，存储/搜索乱码
  - 搜索 100KB 超长字符串 → 搜索引擎OOM

**场景4：表单多字段随机数据填充**

- **测试目标**：验证复杂表单对各类随机输入的校验能力和容错性
- **随机数据生成规则**（基于字段类型）：
  - 文本字段：随机长度(0-10000)、随机字符集（ASCII/中文/emoji/特殊符号/全角）
  - 数字字段：0、负数、小数、极大值、科学计数法、非数字
  - 日期字段：0000-00-00、9999-12-31、2025-02-29、空字符串
  - 邮箱字段：无@符号、双@@、超长域名、中文域名、无.com
  - 手机号字段：随机长度数字、含字母、特殊分隔符、国际区号
  - 下拉框字段：不在选项中的值、空值、多个值、数字型值
  - 文件上传字段：随机大小(0-1GB)、随机格式、伪造MIME类型
  - 富文本字段：超长HTML、无限嵌套标签、自闭合标签、iframe嵌入
- **验证点**：
  - 前端校验：非法输入是否在前端有合理提示
  - 后端校验：绕过前端提交的数据是否在后端二次校验
  - 数据存储：合法数据是否正确写入数据库，非法数据是否被拒绝
  - 错误恢复：提交失败后表单数据是否保留（避免用户重新填写）
- **缺陷示例**：
  - 姓名字段填写1000个中文字符 → 前端未校验 → 后端VARCHAR(50)静默截断 → 存了被截断的姓名
  - 金额字段填写 `1e10` → Java BigDecimal解析异常 → 订单创建失败无提示

**场景5：多用户并发随机操作（混沌工程）**

- **测试目标**：模拟生产环境中的随机并发操作，检测系统在不可预测的多用户行为下的稳定性和数据一致性
- **测试配置**：
  - 虚拟用户数：100-500个并发用户
  - 操作类型（电商场景为例）：
    - 浏览商品（随机类目、随机翻页）
    - 加入购物车（随机商品、随机数量）
    - 下单（随机支付方式、随机地址）
    - 取消订单（随机选择待支付订单）
    - 退款申请（随机选择已完成订单）
  - 混沌注入：
    - 随机Kill一个服务实例
    - 随机网络延迟注入（增加200ms-5s延迟）
    - 随机数据库主从切换
    - 随机消息队列消息积压
- **验证点**：
  - 有无超卖（库存负数）
  - 有无订单丢失（下单成功但数据库无记录）
  - 有无重复扣款
  - 服务恢复后系统自愈能力
  - 所有异常是否正确记录日志
- **缺陷示例**：
  - 用户A和B同时购买最后1件商品 → A支付成功，B也支付成功 → 库存变为-1（超卖）
  - 下单服务实例被Kill的瞬间 → 已扣款但订单未创建 → 资金不一致
  - Redis主从切换瞬间 → 库存缓存脏数据 → 库存显示不准确

## 2.1 穷举组合法

> **缩写**：AC（All Combinations，全组合）

### 通俗释义

把所有参数的所有取值全部排列组合测一遍，保证100%覆盖所有组合，但用例数量随参数增长呈指数级爆炸，实际项目中仅适用于参数极少（2-3个）的简单场景。

### 详细解释

穷举组合法（AC, All Combinations / Exhaustive Combination Testing）是最原始的多参数组合测试方法，即将所有输入参数的所有可能取值进行完全组合（笛卡尔积），对每一种组合都设计一个测试用例。假设有n个参数，每个参数有m_i个取值，则全组合用例数 = m_1 × m_2 × ... × m_n。例如5个参数各3个取值 = 3^5 = 243个用例，10个参数各2个取值 = 2^10 = 1024个用例。

穷举组合法的优点在于覆盖率达到100%，不会遗漏任何参数组合。然而其致命缺点是"组合爆炸"——当参数数量增多或各参数取值增多时，用例数量呈指数级增长，在实际项目中往往不可行。因此穷举组合法通常只适用于参数极少（如2-3个参数）、每个参数取值也很少（2-3个值）的简单场景，或作为其他组合缩减方法（如正交试验法、Pairwise）的参照基准。

在工程实践中，穷举组合法更多是作为一个理论参照——用正交试验法或Pairwise生成的用例集，可以与穷举组合的全集做对比，计算覆盖率和缩减比例，帮助评估用例集的有效性。

| 参数数量 | 每参数取值数 | 全组合用例数 | 可行性 |
|----------|-------------|-------------|--------|
| 2 | 2 | 4 | 可行 |
| 3 | 3 | 27 | 可行 |
| 4 | 3 | 81 | 勉强可行 |
| 5 | 3 | 243 | 困难 |
| 6 | 3 | 729 | 不可行 |
| 10 | 2 | 1024 | 不可行 |

### 实际使用场景（2个）

**场景1：[登录模块] 登录方式三参数全组合**

- **需求**：登录页面支持用户选择登录方式（密码登录/短信验证码登录/扫码登录），设备类型（PC/移动端），以及是否勾选"记住我"（是/否）。
- **参数与取值**：A: 登录方式 → 3水平（密码/短信/扫码）；B: 设备类型 → 2水平（PC/移动端）；C: 记住我 → 2水平（是/否）
- **全组合用例数**：3 × 2 × 2 = 12个用例
- **用例设计**：密码+PC+记住、密码+PC+不记住、密码+移动端+记住、密码+移动端+不记住、短信+PC+记住、短信+PC+不记住、短信+移动端+记住、短信+移动端+不记住、扫码+PC+记住、扫码+PC+不记住、扫码+移动端+记住、扫码+移动端+不记住
- **验证点**：每种组合下的登录流程正确性、记住我功能在关闭浏览器后的持久性、移动端与PC端的UI适配

**场景2：[支付模块] 支付方式与优惠券小范围全组合**

- **需求**：订单支付时支持3种支付方式（微信支付/支付宝/银行卡），可使用2类优惠券（满减券/折扣券），优惠券来源分为2种（系统发放/活动领取）。支付方式和优惠券类型之间可能有互斥规则（如某些银行卡不支持折扣券）。
- **参数与取值**：A: 支付方式 → 3水平（微信/支付宝/银行卡）；B: 优惠券类型 → 2水平（满减券/折扣券）；C: 优惠券来源 → 2水平（系统发放/活动领取）
- **全组合用例数**：3 × 2 × 2 = 12个用例
- **用例设计**：覆盖A×B×C的所有12种组合，验证支付金额计算、优惠券核销、退款时优惠券退回逻辑

## 2.2 基准选择法

> **缩写**：BC（Basic Choice，基本选择/基准选择）

### 通俗释义

先选定一组"基准值"作为参照，每次只变化一个参数的值（其他参数保持基准值不变），观察该参数变化对结果的影响。用极少用例快速覆盖单参数变化的影响。

### 详细解释

基准选择法（BC, Basic Choice），也称基本选择法，是一种最简单的组合缩减策略。核心思想是：先确定每个参数的一个"基准值"（Base Value）——通常是该参数的默认值、最常用值或最安全值。然后生成：(1) **基准用例**（1个）——所有参数都取基准值的组合；(2) **变化用例**——依次只改变一个参数为非基准值，其余参数保持基准值不变。

**用例数公式**：`1 + Σ(每个参数的非基准值个数)` = 线性增长 O(k×m)，k=参数数，m=每参数取值数。

**BC 与组合覆盖体系内其他方法的关系（AC / BC / EC / PC / OC）：**

| 简称 | 全称 | 含义 | 用例数规模 | 本文档对应章节 |
|------|------|------|-----------|---------------|
| **AC** | All Combinations | 全组合穷举，覆盖所有笛卡尔积 | 指数级 m₁×m₂×...×mₙ | [§2.1 穷举组合法](#21-穷举组合法) |
| **BC** | Basic Choice | 基准+单参数替换，每次只变1个参数 | 线性 1+Σ(mᵢ-1) | 本节 |
| **EC** | Extreme Combination | 全覆盖取值最多的参数，其余参数固定 | 等于 max(mᵢ) | [§2.3 极值组合法](#23-极值组合法) |
| **PC** | Pairwise Combination | 覆盖所有两两参数取值组合 | 通常 10~20 个 | [§2.5 Pairwise/All-Pairs](#25-pairwiseall-pairs-配对测试法) |
| **OC** | Orthogonal Combination | 正交表均匀采样，"整齐可比" | L_n(m^k) 由正交表决定 | [§2.4 正交试验法](#24-正交试验法) |

> **术语纠正**：部分国内资料将本节描述的"基准+单参数替换"策略称为"AC（All-Committed）"，这是**错误用法**。在国际标准术语体系中，**AC = All Combinations（全组合穷举）**，BC = Basic Choice（基准选择）。本文档严格遵循国际标准缩写，请读者注意区分。

基准选择法的优势在于用例数量可控（线性增长而非指数增长），适合在测试资源有限时快速识别参数变化的影响。但它的局限在于无法发现两个或多个参数同时偏离基准时的交互缺陷，因此不应作为唯一的组合测试策略。

> **与单一因素组合法的关系**：BC 与 [§2.9 单一因素组合法](#29-单一因素组合法)（OFAT/控制变量法）本质上是同一方法论——两者都遵循"固定其他参数、仅变化一个参数"的模式，用例数公式相同。区别在于 BC 强调"基准值"的系统选取策略，而 OFAT 强调"隔离变量"的因果推断视角。实际使用中二选一即可。

### 实际使用场景（2个）

**场景1：[配置测试] 数据库连接池参数调优**

- **需求**：系统连接池有4个关键参数——最小连接数（默认10）、最大连接数（默认50）、连接超时（默认30s）、空闲超时（默认15min）。需要在各参数变化时验证系统行为。
- **基准值设定**：最小连接=10, 最大连接=50, 连接超时=30s, 空闲超时=15min
- **BC策略用例设计**（1个基准 + 各参数非基准值）：
  - 基准用例：全部取默认值 → 验证连接池正常运行
  - 变化最小连接：5/20/30（其余三个参数保持基准值）→ 验证不同最小连接下的初始连接数
  - 变化最大连接：30/100/200（其余保持基准值）→ 验证不同最大连接上限
  - 变化连接超时：10s/60s/120s（其余保持基准值）→ 验证不同超时配置

- **总用例数**：1 + 3 + 3 + 3 + 3 = 13个（全组合：3^4 = 81个，缩减84%）

  - 基准用例：浅色+中+中文 → 基准截图保存

- **总用例数**：1 + 1 + 2 + 1 = 5个（全组合：2×3×2 = 12个，缩减58%）
- **验证点**：各配置下页面布局不错位、文字不截断、颜色对比度达标

## 2.3 极值组合法

> **缩写**：EC（Extreme Combination，极值组合）

### 通俗释义

找出取值最多的那个参数，把它的所有取值全部覆盖一遍，其他参数随便固定一个值。用最少的用例先把"最复杂、取值最多"的参数全覆盖。

### 详细解释

极值组合法（EC, Extreme Combination）是组合覆盖体系中最简单的一种缩减策略。核心思想是：在所有参数中，找出取值个数最多的那个参数 Pmax，为 Pmax 的每一个取值设计一个用例——在此过程中，其他所有参数各固定取一个值（通常选默认值或最常用值）即可。

**EC 的核心公式：**

```
用例数 = max(m₁, m₂, ..., mₙ)  // 即取值最多的那个参数的取值个数
```

**EC 与 AC / BC 的对比示例（3参数）：**

设参数1（系统）：A、B、C → 3个值；参数2（摄像头）：A、B、C、D → 4个值（最多）；参数3（环境）：A、B → 2个值

| 策略 | 简称 | 做法 | 用例 | 用例数 |
|------|------|------|------|--------|
| **全组合** | AC | 所有排列（笛卡尔积） | 3×4×2=24个 | 24 |
| **基准选择** | BC | 基准+单参数替换 | AA+A, BA+A, AB+A, AC+A（以AA+A为基准） | 1+(3-1)+(4-1)+(2-1)=6 |
| **极值组合** | EC | 全覆盖取值最多的参数（摄像头4个值），其余固定 | AA+A, AB+A, AC+A, AD+A（系统固定=A, 环境固定=A） | 4（=max(3,4,2)） |

> **一句话区分**：EC 照顾取值最多的参数、其他固定 → 最简、优先覆盖复杂项；BC 基准+逐个替换 → 经典最简覆盖；AC 全排列 → 最完整。

**EC 的适用场景与局限：**

| 优势 | 局限 |
|------|------|
| 用例数极少（等于最大取值数） | 除 Pmax 外的参数几乎无组合覆盖 |
| 优先保证最复杂参数的完整性 | 完全无法检测参数间的交互效应 |
| 适合快速冒烟验证、初筛 | 遗漏风险高，不可作为唯一策略 |

**设计步骤：**
1. 列出所有参数及其取值，统计每个参数的取值个数
2. 找出取值个数最多的参数 Pmax
3. 为其余每个参数选定一个固定值（默认值/最常用值）
4. 为 Pmax 的每一个取值生成一个用例（其余参数取固定值）

### 实际使用场景（2个）

**场景1：[移动端] 拍照功能——摄像头型号极值覆盖**

- **需求**：APP拍照功能需兼容4种摄像头配置——单摄/双摄/三摄/四摄（4水平，取值最多）。其他参数：操作系统 iOS/Android（2水平）、屏幕分辨率 HD/FHD/QHD（3水平）、是否开启HDR 是/否（2水平）。
- **EC策略**：摄像头=4个值全覆盖，其余固定（OS=Android, 分辨率=FHD, HDR=是）
- **用例（4个）**：单摄+Android+FHD+HDR；双摄+Android+FHD+HDR；三摄+Android+FHD+HDR；四摄+Android+FHD+HDR
- **全组合**：4×2×3×2 = 48个，EC仅需4个，缩减92%
- **验证点**：每种摄像头配置下拍照、录像、变焦功能正常；确认4种摄像头硬件兼容性无遗漏
- **补充说明**：EC保证最复杂的摄像头参数全覆盖，但未覆盖OS×分辨率×HDR的组合交互。后续需用PC（Pairwise）补充，覆盖约8-10个用例

**场景2：[支付] 支付渠道——渠道类型极值覆盖**

- **需求**：支付系统对接5种支付渠道——微信/支付宝/银行卡/花呗/白条（5水平，取值最多）。其他参数：订单金额区间 小额/中额/大额（3水平）、用户类型 新用户/老用户（2水平）、是否使用优惠券 是/否（2水平）。
- **EC策略**：支付渠道=5个值全覆盖，其余固定（金额=中额, 用户=老用户, 优惠券=否）
- **用例（5个）**：微信+中额+老用户+无券；支付宝+中额+老用户+无券；银行卡+中额+老用户+无券；花呗+中额+老用户+无券；白条+中额+老用户+无券
- **全组合**：5×3×2×2 = 60个，EC仅需5个，缩减92%
- **验证点**：每种支付渠道的基本支付流程正确（创建订单→支付→回调→状态更新）；渠道特有的限额、费率规则正确（如花呗分期、白条免息）
- **补充说明**：EC未覆盖"新用户+银行卡+大额"等组合，高风险组合需用BC或PC额外补充

## 2.4 正交试验法

> **缩写**：OC（Orthogonal Combination，正交组合）

### 通俗释义

参数很多、全组合测不完时，借助正交表挑选少量代表性用例，用最少用例覆盖绝大多数搭配，极端数据再单独补测。

### 详细解释

正交试验法（OC, Orthogonal Combination / Orthogonal Array Testing）利用数学中的正交表来高效设计多因素、多水平的测试用例。当被测系统有多个参数且每个参数有多个取值时，全组合（笛卡尔积）的用例数会爆炸性增长（如5个参数各3个值 = 3^5 = 243个用例）。正交试验法通过"均匀分散、整齐可比"的原则，从全组合中挑选少量具有代表性的用例，在保持充分覆盖的同时大幅减少用例数量。

**核心概念：**

| 概念 | 说明 |
|------|------|
| **因素（Factor）** | 测试中需要变化的参数/变量 |
| **水平（Level）** | 每个因素可能的取值个数 |
| **正交表** | 标准化的因素-水平组合表，记作 L_n(m^k)，n=试验次数，m=水平数，k=因素数 |
| **均匀分散** | 各因素的各水平在试验中出现的次数相同 |
| **整齐可比** | 任意两个因素的所有水平组合在试验中出现的次数相同 |

**常用正交表举例：**

```
L_4(2^3)：3个2水平因素，只需4次试验（全组合需8次）
L_9(3^4)：4个3水平因素，只需9次试验（全组合需81次）
L_16(4^5)：5个4水平因素，只需16次试验（全组合需1024次）
L_18(3^7)：7个3水平因素，只需18次试验（全组合需2187次）
```

**设计步骤：**
1. 确定测试中的因素（参数）和各因素的水平（取值）
2. 选择合适的正交表（因素数≥实际因素数，水平数≥最大水平数）
3. 将各因素的各水平映射到正交表中
4. 对于水平数不一致的情况，用"拟水平法"补足（将重要水平复用）
5. 将正交表每一行转换为一个测试用例
6. 对正交表中未覆盖但风险较高的极端组合，额外补充用例

- **注意事项**：收入范围×学历的某些组合在业务逻辑上可能有特殊校验（如"高中及以下+收入>30000"可能标记为异常），需额外验证

**场景5：电商促销规则组合测试**

- **因素与水平**：
  - A: 优惠券类型 → 4水平（满减券 / 折扣券 / 免邮券 / 现金券）
  - B: 会员等级 → 3水平（普通 / 银卡 / 金卡）
  - C: 支付方式 → 3水平（微信 / 支付宝 / 银行卡/信用卡）
  - D: 商品类型 → 3水平（普通商品 / 预售商品 / 虚拟商品）
  - E: 是否参与满减活动 → 2水平（是 / 否）
  - F: 是否叠加积分抵扣 → 2水平（是 / 否）

- **全组合**：4×3×3×3×2×2 = 432个用例
- **选用正交表**：约需24-36个用例
- **关键验证点**：
  - 优惠券叠加规则（哪些券可叠加，哪些互斥）
  - 会员折扣与优惠券的叠加顺序（先会员折扣再优惠券 vs 先优惠券再会员折扣，金额可能不同）
  - 虚拟商品是否可用免邮券（不应可用）

## 2.5 Pairwise/All-Pairs 配对测试法

> **缩写**：PC（Pairwise Combination，两两组合/全配对）

### 通俗释义

全排列太多、正交表不会选时，用Pairwise工具自动生成最少用例集——保证任意两个参数的取值组合都被覆盖到。核心逻辑是：大多数缺陷由单个参数或两个参数的组合触发，覆盖所有两两组合即可发现大部分Bug。

### 详细解释

Pairwise测试（PC, Pairwise Combination / All-Pairs Testing，全配对测试）是一种组合测试技术。其核心假设是：软件中绝大多数缺陷是单个参数或两个参数交互导致的（而非三个及以上参数的复杂交互）。因此，无需覆盖所有N个参数的全部组合（笛卡尔积），只需保证每两个参数的所有取值组合在用例集中至少出现一次。

**Pairwise与正交表的关系与区别：**

| 对比维度 | 正交试验法 | Pairwise/All-Pairs |
|----------|-----------|-------------------|
| **理论基础** | 正交拉丁方 / 数学正交表 | 贪心算法 / 图论覆盖 |
| **选表方式** | 查标准正交表（L_4(2^3), L_9(3^4)等） | 自动生成（工具自动计算） |
| **参数水平要求** | 所有参数水平数最好一致（不一致需拟水平法） | 参数水平数无需一致 |
| **覆盖程度** | 任意两因素的所有水平组合出现次数相同（整齐可比） | 保证每个两两组合至少出现一次 |
| **用例数** | 固定（由正交表决定） | 自动优化生成，通常等于或略多于正交表 |
| **灵活性** | 受限于可用正交表的规格 | 极高——任意数量参数、任意水平数都可 |

> **关键区别**：正交表要求"整齐可比"（任意两个因素间的所有水平组合出现次数相同），而Pairwise只要求"覆盖"（每个两两组合至少出现一次）。Pairwise在实践中更灵活，且现代工具有力地支持了其使用。

**设计步骤：**
1. 列出所有参数及每个参数的可能取值
2. 确定必含的约束条件（如某些参数组合不可能出现，如"支付方式=货到付款"与"地区=境外"不可共存）
3. 使用Pairwise工具（如PICT、AllPairs、Jenny等）自动生成测试用例集
4. 验证生成的用例集是否满足两两覆盖
5. 对工具无法处理的约束，手动补充或调整用例
6. 基于风险补充高维组合（如已知某3个参数交互会导致Bug，额外补充该3参数的全组合）

### 实际使用场景（3个）

**场景1：浏览器配置兼容性（8参数，229种两两组合→仅需11个用例）**

- **参数列表**：
  - 浏览器: Chrome / Firefox / Edge / Safari（4水平）
  - 操作系统: Windows / macOS / Linux（3水平）
  - 屏幕分辨率: 1920×1080 / 1366×768 / 2560×1440 / 3840×2160（4水平）
  - JavaScript版本: ES2019 / ES2020 / ES2021（3水平）
  - 语言: 中文 / English（2水平）
  - Cookie策略: 全部接受 / 仅会话（2水平）
  - 时区: CST / EST / UTC（3水平）
  - 网络: 在线 / 离线刚恢复（2水平）

- **全组合**：4×3×4×3×2×2×3×2 = 3456个用例（完全不可行）
- **Pairwise后**：约11-15个用例即可覆盖所有两两组合
- **工具示例（PICT）**：

```text

OS: Windows, macOS, Linux
Resolution: 1920x1080, 1366x768, 2560x1440, 3840x2160
JS: ES2019, ES2020, ES2021
Language: Chinese, English

Network: online, reconnect

```

**场景2：注册表单多字段组合（6参数，约12个用例）**

- **参数与取值**：
  - 账户类型：个人 / 企业（2水平）
  - 证件类型：身份证 / 护照 / 营业执照（3水平）
  - 手机号归属：国内 / 海外（2水平）
  - 密码强度：弱 / 中 / 强（3水平）
  - 验证方式：短信 / 语音 / 邮件（3水平）
  - 协议勾选：已勾选 / 未勾选（2水平）

- **全组合**：2×3×2×3×3×2 = 216个用例
- **Pairwise后**：约11-13个用例即可覆盖所有两两组合
- **补充用例**：企业账户 + 身份证（非法组合，但应测系统是否拦截）

**场景3：API请求参数组合（7参数，约15个用例）**

- **参数与取值**：
  - HTTP方法: GET / POST / PUT（3水平）
  - 认证Token: 有效 / 过期 / 缺失 / 错误（4水平）
  - Content-Type: JSON / XML / Form（3水平）
  - 分页: offset / cursor / 无分页（3水平）
  - 排序: asc / desc / 无（3水平）
  - 压缩: gzip / none（2水平）
  - 字段筛选: 无 / 部分字段 / 全部字段（3水平）

- **全组合**：3×4×3×3×3×2×3 = 1944个用例
- **Pairwise后**：约14-18个用例
- **关键约束**：GET方法不允许有Body（传了Content-Type=JSON也应被忽略）

> **工程建议**：Pairwise/All-Pairs不是全组合的完全替代品。对于已知的三参数及以上交互风险（如"支付金额+会员等级+优惠券类型"），应在Pairwise基础上额外补充高维组合用例。实际项目中，Pairwise通常与等价类划分法组合使用：先用等价类确定每个参数的典型取值，再用Pairwise生成组合用例。
>
> **与t-way的关系**：Pairwise/All-Pairs本质上就是t=2的t-way组合覆盖。如需更强的组合覆盖强度（如金融核心系统使用t=3），参见 [§2.6 t-way组合覆盖](#26-t-way组合覆盖2-way3-wayn-way)。

## 2.6 t-way组合覆盖（2-way/3-way/N-way）

### 通俗释义

Pairwise（2-way）的泛化版本——不仅覆盖任意2个参数的所有组合，还可以覆盖任意3个、4个甚至N个参数的组合。t值越大覆盖越全、用例越多，t=N时退化为穷举组合。

### 详细解释

t-way组合覆盖（t-way Combinatorial Testing）是Pairwise/All-Pairs测试的推广形式。其核心思想是：保证任意t个参数的所有取值组合在用例集中至少出现一次。t=1时即"每个参数的每个取值至少出现一次"（基础覆盖）；t=2时即Pairwise/All-Pairs（任意两参数组合覆盖）；t=3时即"任意三个参数的所有取值组合至少出现一次"，以此类推。当t等于参数总数N时，t-way退化为穷举全组合。

研究表明，约70%的软件缺陷由单个参数触发，约90%的缺陷由单参数或双参数组合触发，约97%的缺陷由不超过3个参数交互触发。因此在实际工程中，t=2（Pairwise）已能覆盖绝大多数缺陷，t=3可进一步提升覆盖率但用例数显著增加。对于安全关键系统（如航空、医疗、金融核心系统），建议使用t=3甚至t=4的组合覆盖强度。

| t值 | 覆盖要求 | 用例数规模 | 适用场景 |
|-----|---------|-----------|---------|
| t=1 | 每个参数的每个取值至少出现一次 | 极少 | 基础冒烟测试 |
| t=2（Pairwise） | 任意两参数的所有组合 | 少 | 常规业务系统 |
| t=3 | 任意三参数的所有组合 | 中等 | 金融/医疗核心系统 |
| t=4 | 任意四参数的所有组合 | 较大 | 航空/国防安全苛求系统 |
| t=N | 所有参数全部组合（穷举） | 极大 | 仅作理论参照 |

t-way测试用例生成通常借助工具完成，常用工具有ACTS（NIST开发）、PICT、CASA等。

### 实际使用场景（2个）

**场景1：[安全关键系统] 飞机仪表盘告警逻辑（t=3）**

- **需求**：飞机告警系统有5个关键参数，每个参数有多个离散取值。需验证任意3个参数组合下的告警行为正确性（安全关键系统要求更高的组合覆盖强度）。
- **参数与取值**：A: 高度传感器 → 3水平（正常/偏高/偏低）；B: 速度传感器 → 3水平（正常/超速/失速）；C: 姿态传感器 → 3水平（水平/倾斜/倒飞）；D: 引擎状态 → 2水平（正常/异常）；E: 燃油状态 → 2水平（充足/不足）
- **全组合**：3×3×3×2×2 = 108个；2-way约需12-15个用例；3-way约需27-33个用例
- **t=2 vs t=3对比验证**：
  - t=2用例集（约14个）→ 覆盖所有两参数组合 → 可能遗漏三个传感器同时异常时的告警叠加逻辑
  - t=3用例集（约30个）→ 额外覆盖"高度偏低+速度失速+姿态倒飞"等三参数危险组合
- **验证点**：任意3个异常参数的组合能否触发正确的多级告警（警告/注意/提示），告警优先级排序是否正确（如引擎异常+燃油不足应优先于高度偏高）

**场景2：[金融核心系统] 交易风控规则（t=3）**

- **需求**：交易风控引擎有6个参数，需使用t=3组合覆盖验证任意3个风控因子的组合行为。
- **参数与取值**：A: 交易金额区间 → 3水平（小额<1000/中额1000-50000/大额>50000）；B: 交易时间 → 3水平（工作日白天/工作日夜间/节假日）；C: 交易地点 → 3水平（常用地/非常用地/境外）；D: 交易频率 → 2水平（正常/高频）；E: 账户年龄 → 2水平（新账户<30天/老账户）；F: 设备 → 2水平（常用设备/新设备）
- **全组合**：3×3×3×2×2×2 = 216个；2-way约需12-15个；3-way约需30-40个
- **关键验证**：大额+境外+新设备（t=3）→ 是否触发最高级风控拦截；大额+高频+新账户（t=3）→ 是否触发洗钱嫌疑标记
- **验证点**：t=3覆盖下，风控规则的组合逻辑无遗漏；t=2覆盖下可能遗漏的3因子组合缺陷

## 2.7 部分组合法

### 通俗释义

不是全覆盖也不是随机选，而是按照某种规则（如"每个值至少出现一次"、"重点值多出现几次"）有策略地选取部分组合来测。是穷举组合和覆盖阵列之间的折中方案，适合测试时间紧张的场景。

### 详细解释

部分组合法（Partial Combination Testing）是介于穷举组合与系统化组合缩减方法之间的一类策略的统称。它不追求完全覆盖（如穷举法），也不依赖数学工具保证两两覆盖（如Pairwise），而是基于测试人员的判断和策略，有选择地选取一部分参数组合进行测试。

部分组合法的常见选取策略包括：
- **每个值至少出现一次策略**：保证每个参数的每个取值至少在用例集中出现一次，但不保证组合覆盖
- **重点值复现策略**：对业务上重要或风险较高的取值，在多个用例中反复出现
- **随机抽样策略**：从全组合中随机抽取一定比例的用例
- **分层抽样策略**：按参数的取值重要性分层后抽样

| 策略 | 覆盖保证 | 用例数 | 适用场景 |
|------|---------|--------|---------|
| 每个值至少一次 | 每个取值出现≥1次 | 极少 | 冒烟测试 |
| 重点值复现 | 高优先级值多次出现 | 少 | 核心业务验证 |
| 随机抽样 | 统计意义上的覆盖 | 灵活 | 回归测试补充 |
| 分层抽样 | 各层均有代表 | 中等 | 多维度配置测试 |

部分组合法适用于参数数量适中（4-8个）、测试时间紧张、且没有工具支持正交/Pairwise生成时的场景。其局限性在于覆盖率不可量化，可能遗漏重要的参数交互缺陷。

### 实际使用场景（2个）

**场景1：[快速迭代] 营销活动页面配置**

- **需求**：运营活动落地页有5个可配置项——Banner样式（3种）、优惠券展示位置（顶部/底部/弹窗）、商品列表样式（大图/列表/网格）、按钮颜色（红/蓝/橙/绿）、背景主题（节日/简约/品牌）。产品要求2小时内完成测试。
- **策略选择**：每个值至少出现一次 + 重点值复现
- **用例设计步骤**：列出所有取值：Banner=3种，位置=3种，列表样式=3种，按钮=4种，背景=3种（全组合3×3×3×4×3=324个——不可能2小时完成）。保证每个值出现一次的最小用例数 = max(各参数取值数) = 4个（取按钮颜色的4个取值为主轴）
- **实际设计6个用例**：按钮4种颜色各1个用例 + 重要组合额外2个（节日背景+弹窗+红色按钮；品牌背景+顶部+蓝色按钮）
- **验证点**：每个可配置项的各种取值都能正确渲染、配置组合之间无样式冲突

**场景2：[回归测试] 订单查询筛选条件**

- **需求**：订单查询页面有7个筛选条件——订单状态（6种）、支付方式（4种）、时间范围（7天/30天/自定义）、金额范围（单选/区间）、商品类目（3个一级类目）、会员等级（4级）、是否开发票（是/否）。全组合极大，回归测试只能跑部分组合。
- **策略选择**：分层抽样——核心条件全遍历，辅助条件随机匹配
- **用例设计**：核心层：订单状态（6种）+时间范围（2种固定）= 12个用例（每个状态在7天和30天各测一次）。辅助层：支付方式+会员等级+发票=每个核心用例随机匹配一种辅助条件组合
- **总用例数**：12个（覆盖所有状态×时间范围的核心组合，其余条件随机搭配）
- **验证点**：不同订单状态下的筛选结果正确性、核心条件组合的查询性能、筛选结果与列表展示的一致性

## 2.8 配对组合法

### 通俗释义

把互相之间影响比较大的参数两两配对锁定，"捆绑"在一起重点测试，然后再和别的参数组合。不等于Pairwise工具方法——本方法强调基于业务知识的人工配对，而非数学工具的自动覆盖。

### 详细解释

配对组合法（Paired Combination Testing）是一种基于业务语义的组合策略。与Pairwise/All-Pairs（数学工具保证任意两参数组合的覆盖）不同，配对组合法强调的是：根据业务知识，识别出哪些参数之间存在强关联/强交互关系，将这些参数"配对"在一起作为一个整体来设计组合用例。

配对组合法的工作步骤：
1. 识别强关联参数对：通过业务分析，找出哪些参数组合在一起会触发特殊的业务逻辑。例如"会员等级"和"折扣系数"强关联，"支付方式"和"手续费率"强关联
2. 将强关联对"绑定"：将两个或多个强关联参数视为一个组合单元，先确保该单元内部的取值组合被充分测试
3. 跨对组合：将不同的参数对与其他参数进行组合

| 对比维度 | Pairwise（工具法） | 配对组合法（业务法） |
|----------|-------------------|---------------------|
| 驱动方式 | 数学工具自动生成 | 测试人员业务判断 |
| 覆盖目标 | 任意两参数的数学组合 | 有业务语义的参数对 |
| 约束处理 | 通过工具语法声明 | 通过业务分析识别 |
| 适用场景 | 参数多、取值多、业务关系简单 | 业务规则复杂、参数间强关联 |

### 实际使用场景（2个）

**场景1：[保险业务] 保费计算参数配对**

- **需求**：保费计算涉及6个参数——险种（车险/健康险/寿险）、被保人年龄（<18/18-30/31-50/51-65/>65）、保额（5档）、缴费期限（趸交/5年/10年/20年/30年）、是否有既往症（是/否）、是否吸烟（是/否）。各参数之间存在不同的强关联关系。
- **强关联对识别**：
  - 配对1：险种 × 年龄 → 不同险种的年龄限制不同（如寿险最高投保年龄65岁，健康险最高55岁）
  - 配对2：年龄 × 缴费期限 → 年龄+缴费期限之和不能超过一定上限（如65岁不能选30年缴）
  - 配对3：是否有既往症 × 险种 → 健康险对有既往症的可能拒保或加费
- **用例设计**：先设计每个配对内部的合法/非法组合 → 再将3个配对交叉组合。配对1典型用例：健康险+55岁（边界）、车险+<18（非法）、寿险+66（非法）；配对2典型用例：51-65岁+30年缴（可能非法）；配对3典型用例：有既往症+健康险（加费或拒保）
- **验证点**：每个配对内部的业务规则正确、跨配对组合时无规则冲突、保费计算结果正确

**场景2：[电商促销] 优惠叠加规则配对**

- **需求**：一个订单可以同时使用多种优惠——平台满减（2档）、店铺优惠券（3种）、会员折扣（4级）、积分抵扣、支付立减（2种支付方式）。存在复杂的叠加/互斥规则。
- **强关联对识别**：
  - 配对1：平台满减 × 店铺优惠券 → 是否可叠加（业务规则：部分活动可叠加，部分不可）
  - 配对2：会员折扣 × 店铺优惠券 → 叠加时的计算顺序（先会员折扣再优惠券 vs 先优惠券再会员折扣，结果不同）
  - 配对3：积分抵扣 × 支付立减 → 支付立减是在积分抵扣后还是抵扣前
- **用例设计**：每个配对设计3-5个典型组合 → 再选取代表性的跨配对组合做端到端验证
- **配对1用例**：满200减30 + 店铺满100减10（可叠加）→ 验证折扣计算；满200减30 + 店铺满100减10（不可叠加规则）→ 验证系统自动取最优
- **配对2用例**：金卡8折 + 满200减30 → 验证是(原价×0.8-30)还是(原价-30)×0.8
- **验证点**：叠加计算顺序的执行正确性、互斥规则的拦截提示、边界情况（刚好达到满减门槛）

## 2.9 单一因素组合法

### 通俗释义

一次测试只变化一个参数看效果，其他参数全部固定不变。和科学实验里的"控制变量法"一样，好处是能清晰定位是哪个参数导致的问题，坏处是完全无法检测参数之间的交互效应。

### 详细解释

单一因素组合法（Single-Factor Combination Testing），也称为"一次一个因素法"（One-Factor-at-a-Time, OFAT），是最简单直观的组合测试策略。其做法是：选定一个基准配置（所有参数取默认值或常用值），然后每次只改变一个参数的值（从一个水平变到另一个水平），观察该变化对输出的影响。

这种方法的最大优点是因果关系清晰——当测试结果发生变化时，可以直接归因于那个被改变的参数，便于缺陷定位和责任划分。同时用例数量极少：如果有k个参数，每个参数m个取值，则仅需 1 + k×(m-1) 个用例（基准 + 每个参数的非基准取值），呈线性增长。

然而单一因素组合法有一个致命缺点：它完全无法检测参数之间的交互效应（Interaction Effect）。如果缺陷只有在两个或多个参数同时取特定值时才触发，单一因素法几乎一定会漏掉。因此在实践中，单一因素组合法不应作为唯一的组合测试策略，而应与Pairwise、正交试验等多因素组合方法配合使用。

> **与基准选择法的关系**：单一因素组合法与 [§2.2 基准选择法（BC）](#22-基准选择法) 本质上是同一方法论——两者用例数公式相同（1 + Σ非基准值个数），都遵循"每次只变一个参数"的模式。区别在于表述视角不同：单一因素组合法强调科学实验的"控制变量"逻辑，基准选择法强调"基准值"的系统选取。实际使用中二选一即可。

| 优点 | 缺点 |
|------|------|
| 因果关系清晰，缺陷易于定位 | 完全无法检测参数交互效应 |
| 用例数极少，线性增长 | 可能遗漏多参数组合缺陷 |
| 无需工具支持，简单易行 | 不适合参数间有复杂依赖的场景 |
| 适合性能调优、单变量影响分析 | 覆盖率无保证 |

### 实际使用场景（2个）

**场景1：[性能调优] JVM参数对GC行为的影响**

- **需求**：需要确定4个JVM参数对垃圾回收停顿时间的影响——堆大小（256MB/512MB/1GB/2GB）、GC算法（G1/Parallel/Serial）、年轻代比例（1/4, 1/3, 1/2）、是否开启字符串去重（是/否）。
- **基准配置**：堆=512MB, GC=G1, 年轻代=1/4, 去重=否
- **单一因素变化用例（10个）**：
  - 基准：512MB+G1+1/4+否 → 记录GC停顿时间作为基线
  - 变化堆大小：256MB+其他同基准 / 1GB+其他同基准 / 2GB+其他同基准 → 记录停顿时间变化
  - 变化GC算法：Parallel+其他同基准 / Serial+其他同基准 → 对比不同算法停顿时间
  - 变化年轻代：1/3+其他同基准 / 1/2+其他同基准 → 对比不同比例
  - 变化去重：是+其他同基准 → 对比去重开销
- **验证点**：每次变化后GC停顿时间的定量对比、单参数变化对吞吐量的影响
- **注意事项**：单一因素法能清晰定位每个参数的影响，但无法发现"堆=1GB且GC=Parallel且年轻代=1/2时出现异常Full GC"这种多参数交互缺陷

**场景2：[功能验证] 邮件通知模板配置**

- **需求**：系统邮件通知有3个可配置参数——模板语言（中文/英文）、发送方式（SMTP/API/队列）、是否包含附件（是/否）。需要在配置变更后快速验证通知功能。
- **基准配置**：语言=中文, 发送方式=SMTP, 附件=否
- **单一因素变化用例（4个）**：
  - 基准：中文+SMTP+不含附件 → 验证邮件正常发送和内容正确
  - 变化语言：英文+SMTP+不含附件 → 验证英文模板格式、变量替换正确
  - 变化发送方式：中文+API+不含附件 / 中文+队列+不含附件 → 验证不同通道送达
  - 变化附件：中文+SMTP+含附件 → 验证附件上传和接收
- **全组合**：仅2×3×2=12个，单一因素法4个用例缩减67%
- **验证点**：每种单一变化下邮件内容、格式、附件均正确
- **注意事项**：如果中文+队列+附件组合有特殊限制（如队列不支持附件），单一因素法可能遗漏

## 2.10 混合组合法

### 通俗释义

面对不同参数间关系不同，混合使用多种组合策略——重要的参数组合全测，中等重要的用正交或Pairwise，不重要的随意搭配。是多种组合策略的组合，也是实际项目中最务实的做法。

### 详细解释

混合组合法（Mixed Combination Testing / Hybrid Combination Strategy）是一种务实的测试策略。它的核心理念是：不同参数之间的重要程度和交互风险是不同的，没有一种单一的"最佳"组合策略能适应所有情况。因此，将多种组合策略混合使用：

> **方法定位说明**：严格来说，混合组合法是一种**元策略（Meta-Strategy）**而非独立的测试设计方法——它本身不定义新的组合规则，而是指导如何在不同风险等级的参数上选用前述各种组合方法。纳入本文档是为了提供完整的工程实践视角。

- 对高风险/核心参数组合 → 使用穷举组合法，确保100%覆盖
- 对中等风险参数组合 → 使用正交试验法或Pairwise/All-Pairs，保证系统化覆盖
- 对低风险参数组合 → 使用单一因素法或随机组合，减少用例数
- 对有明确业务关联的参数 → 使用配对组合法，按业务规则设计

混合组合法的设计步骤：
1. 列出所有参数及各参数取值
2. 对参数进行风险分级（高/中/低），风险判断依据：该参数涉及资金安全？核心主流程？历史缺陷频率高？
3. 将高风险参数组合划为"全组合区"，中风险划为"系统化覆盖区"，低风险划为"精简区"
4. 分别应用不同的组合策略
5. 合并去重，形成最终用例集

混合组合法的优势在于平衡了覆盖率、风险控制和用例数量三者的关系，是实际项目中最常用的做法。

| 风险等级 | 参数特征 | 推荐策略 | 覆盖目标 |
|----------|---------|---------|---------|
| **高风险** | 涉及资金、安全、核心主流程 | 穷举组合法 | 100%覆盖 |
| **中风险** | 影响用户体验、业务流程 | 正交/Pairwise | 两两覆盖 |
| **低风险** | 辅助功能、展示类 | 单一因素/随机 | 基础覆盖 |
| **业务关联** | 有明显业务配对关系 | 配对组合法 | 业务规则覆盖 |

### 实际使用场景（2个）

**场景1：[电商下单] 下单参数混合策略**

- **需求**：下单接口涉及8个参数。需合理分配测试资源。
- **参数风险分级**：
  - 高风险参数（涉及金额和库存）：支付方式（3种）、优惠券类型（3种）、是否使用积分（是/否）→ 穷举组合：3×3×2 = 18个用例
  - 中风险参数（影响用户体验）：配送方式（3种）、收货地址类型（2种）→ Pairwise与高风险参数合并生成约12个用例
  - 低风险参数（对核心流程影响小）：发票类型（2种）、订单备注（2种）、是否匿名购买（2种）→ 随机搭配
- **最终用例集**：高风险全组合18个 + 中风险Pairwise补充8个（扣除与高风险重复的）= 约26个用例
- **全组合**：3×3×2×3×2×2×2×2 = 864个，缩减97%
- **验证点**：高风险参数的所有组合均被完整测试、中低风险参数的代表性组合被覆盖

**场景2：[配置管理] 系统参数配置变更混合验证**

- **需求**：系统有10个运行时配置参数，需要验证配置变更后的系统行为。不同配置项的影响范围和风险不同。
- **参数风险分级**：
  - 高风险（变更后可能导致服务不可用）：数据库连接池大小（3档）、缓存过期时间（3档）、线程池大小（3档）→ Pairwise覆盖（约9-12个用例）
  - 中风险（影响性能但不影响可用性）：日志级别（4档）、限流阈值（3档）、超时时间（3档）→ 单一因素法（1基准+各参数变化=约8个）
  - 低风险（仅影响展示）：页面分页条数（3档）、日期格式（2档）、货币符号（3档）、语言（2档）→ 随机组合（约5个）
- **最终用例集**：约22-25个用例（全组合极大）
- **验证点**：高风险配置组合的系统稳定性、中风险配置变更的性能影响、配置热更新是否生效

## 2.11 等价组合法

### 通俗释义

先把每个参数的取值划成几个"等价类组"（一组里的值效果差不多），然后对等价类组之间做组合。是等价类划分法和组合测试的结合，特别适合每个参数取值空间很大的场景。

### 详细解释

等价组合法（Equivalence Combination Testing）是等价类划分思想在组合测试中的延伸应用。当每个参数的可能取值非常多（如连续数值、大量枚举值）时，可以先对每个参数的取值进行等价类划分，然后对划分后的等价类进行组合测试，从而大幅减少组合空间。

等价组合法的核心步骤：
1. 对每个参数进行等价类划分：将参数的取值空间划分为若干等价类（如"低/中/高"三个等价类），每个等价类内取1-2个代表值
2. 等价类映射：将原始的大量取值映射到少数几个等价类标签
3. 以等价类为单位进行组合测试：使用Pairwise、正交表或其他策略对等价类标签进行组合
4. 为每种等价类组合选取具体代表值：从对应等价类中选取具体测试数据

等价组合法特别适用于参数取值空间很大的场景（如金额、年龄、坐标等连续值参数），通过等价类的抽象，将无限或大量取值缩减为有限的几个类别，然后在此基础上进行高效的组合测试。

### 实际使用场景（2个）

**场景1：[金融系统] 转账金额多参数组合**

- **需求**：转账接口有4个参数——转账金额（0.01-500000元）、收款银行（20家银行）、转账方式（实时/普通/次日）、是否跨境（境内/境外）。金额参数取值无限，需要等价类划分后才能做组合测试。
- **等价类划分**：
  - 转账金额 → 5个等价类：小额(0.01-100)、中额(100-10000)、大额(10000-50000)、超大额(50000-500000)、无效(0/负数/超大)
  - 收款银行 → 4个等价类：国有大行、股份制银行、城商行/农商行、外资银行
  - 转账方式 → 3个等价类：实时、普通、次日
  - 是否跨境 → 2个等价类：境内、境外
- **等价类组合空间**：5×4×3×2 = 120种等价类组合，使用Pairwise约需15-18个用例
- **验证点**：不同金额等价类在各银行/跨境组合下的手续费计算、到账时间、限额策略

**场景2：[配置测试] 系统超时参数组合**

- **需求**：系统有3个超时参数——连接超时（1ms-60000ms）、读取超时（1ms-120000ms）、空闲超时（1s-3600s）。取值空间极大，需等价类划分后组合。
- **等价类划分**：
  - 连接超时 → 4个等价类：极短(1-100ms)、正常(100ms-5s)、较长(5s-30s)、超长(30s-60s)
  - 读取超时 → 4个等价类：极短(1-100ms)、正常(100ms-10s)、较长(10s-60s)、超长(60s-120s)
  - 空闲超时 → 4个等价类：短(1-60s)、正常(60s-600s)、长(600s-1800s)、超长(1800s-3600s)
- **等价类组合空间**：4×4×4 = 64种，使用正交表L_16约需16个用例
- **验证点**：不同超时等价类组合下的连接行为、超时优先级（哪个超时先生效）、资源释放正确性

## 2.12 边界组合法

### 通俗释义

对多个参数，每个参数都取边界值（最小值、最小值+1、最大值-1、最大值等），然后把各参数的边界值组合起来测。是边界值分析法和组合测试的结合，"边界值 × 边界值"的组合往往触发系统最深层的逻辑错误。

### 详细解释

边界组合法（Boundary Combination Testing）是边界值分析思想在组合测试中的延伸应用。当多个参数都有明确的边界范围时，将对每个参数的边界值关注与参数间的组合覆盖结合起来。研究表明，边界值和参数组合的交汇处是最容易发现缺陷的区域——"边界值 × 边界值"的组合往往触发系统最深层的逻辑错误。

边界组合法的核心步骤：
1. 对每个参数进行边界值分析：确定每个参数的关键边界值（min-1, min, min+1, normal, max-1, max, max+1）
2. 边界值标记：将每个参数的取值分为"边界值"和"非边界值（正常值）"两类
3. 组合策略：重点覆盖"多个参数同时取边界值"的组合，辅以"一边界一正常"的组合
4. 可配合Pairwise或正交表对边界值集合进行组合

边界组合法特别适用于有明确数值范围的多参数系统（如金融计算、物理仿真、工程计算），这些系统中多个参数同时处于边界时最容易出现精度丢失、溢出、逻辑判断错误等问题。

### 实际使用场景（2个）

**场景1：[金融计算] 贷款计算器多参数边界**

- **需求**：贷款计算器有4个参数——贷款金额（1万-500万）、年利率（2.00%-24.00%）、贷款期限（1个月-360个月）、还款方式（等额本息/等额本金）。需验证多个参数同时取边界值时的计算结果。
- **各参数边界值**：
  - 贷款金额边界：1万(min)、1.0001万(min+0.0001)、499.9999万(max-0.0001)、500万(max)、0.9999万(min-1非法)、500.0001万(max+1非法)
  - 年利率边界：2.00%(min)、2.01%(min+0.01)、23.99%(max-0.01)、24.00%(max)、1.99%(min-1非法)、24.01%(max+1非法)
  - 贷款期限边界：1个月(min)、2个月(min+1)、359个月(max-1)、360个月(max)、0个月(min-1非法)、361个月(max+1非法)
  - 还款方式：等额本息、等额本金（2种合法值）
- **边界组合策略**：选取"双边界组合"（贷款金额边界+年利率边界）→ 将两个参数的边界值做Pairwise组合
- **典型边界组合用例**：500万(max) + 24.00%(max) + 360个月(max) + 等额本息 → 验证大数计算精度；1万(min) + 2.00%(min) + 1个月(min) + 等额本金 → 验证小数计算精度；500万(max) + 2.00%(min) + 360个月(max) → 验证边界交叉
- **验证点**：各边界组合下月供金额精确到分、总利息计算正确、提前还款的剩余本金计算

**场景2：[图像处理] 图片压缩参数边界组合**

- **需求**：图片压缩工具提供3个参数——输出宽度（50px-4096px）、输出高度（50px-4096px）、压缩质量（1%-100%）。多个参数同时取边界值可能导致OOM或压缩失败。
- **各参数边界值**：
  - 宽度边界：50px(min)、51px(min+1)、4095px(max-1)、4096px(max)、49px(min-1非法)、4097px(max+1非法)
  - 高度边界：50px(min)、51px(min+1)、4095px(max-1)、4096px(max)、49px(min-1非法)、4097px(max+1非法)
  - 质量边界：1%(min)、2%(min+1)、99%(max-1)、100%(max)、0%(min-1非法)、101%(max+1非法)
- **边界组合重点用例**：4096px×4096px×100% → 最大尺寸+最高质量，验证是否OOM、处理时间是否可接受；4096px×4096px×1% → 最大尺寸+最低质量，验证极端压缩的视觉效果；50px×50px×1% → 最小尺寸+最低质量，验证是否过度压缩导致完全无法辨认；4096px×50px×100% → 宽高比极端不对称，验证是否变形或溢出
- **验证点**：边界组合下无内存溢出、无处理超时、输出尺寸和质量符合预期

## 2.13 判定表法

### 通俗释义

多个条件互相影响结果时，罗列所有条件组合，合并重复规则生成用例；分只有是/否、多取值两种表格格式。

### 详细解释

判定表法（Decision Table Testing）是一种系统化处理多条件组合的测试设计技术。当输入条件之间存在逻辑关系、不同条件组合产生不同结果时，判定表可以清晰完整地列出所有条件组合及其对应动作，避免遗漏。

**判定表的两种形式：**

| 形式 | 说明 | 适用场景 |
|------|------|----------|
| **有限条目判定表** | 所有条件取值只有"是/否"（True/False）两种状态 | 条件为布尔类型或可二值化 |
| **扩展条目判定表** | 条件可以有多个取值（如高/中/低、大/中/小） | 条件非二值，有多种离散值 |

**判定表的组成结构：**

```
+-------------------------------------+
|        条件桩（Condition Stub）       |  — 列出所有条件
+-------------------------------------+
|        条件项（Condition Entry）      |  — 每个条件的取值组合
+-------------------------------------+
|        动作桩（Action Stub）          |  — 列出所有可能的动作
+-------------------------------------+
|        动作项（Action Entry）         |  — 对应每种组合的动作结果
+-------------------------------------+
```

**设计步骤：**
1. 分析需求，识别所有条件（输入条件）和动作（输出结果）
2. 列出条件桩和动作桩
3. 计算规则数 = 条件1取值个数 × 条件2取值个数 × …（如有4个二值条件，规则数=2^4=16）
4. 填写条件项（所有组合）
5. 填写动作项（每种组合对应的结果）
6. 合并简化：如果两条规则的条件取值只有一项不同且产生的动作完全相同，则该项可以合并（标记为"—"无关）

## 2.14 因果图法

### 通俗释义

画图梳理输入条件和输出结果的逻辑关系，标注逻辑运算和约束规则，画完图的数据可以整理成判定表。

### 详细解释

因果图法（Cause-Effect Graphing）通过图形化的方式直观表达输入条件（因）和输出结果（果）之间的逻辑关系。它特别适用于输入条件之间存在复杂逻辑关系（与、或、非及各类约束）的场景。因果图是判定表的前置分析工具——画完因果图后，将其转换为判定表来设计测试用例。

**因果图的核心元素：**

| 元素 | 符号 | 说明 |
|------|------|------|
| **恒等** | `=` | 因出现，果必出现；因不出现，果必不出现 |
| **非（NOT）** | `~` | 因出现，果不出现；因不出现，果才出现 |
| **或（OR）** | `V` | 多个因中至少一个出现，果出现 |
| **与（AND）** | `^` | 多个因全部出现，果才出现 |

**五类约束关系（因与因之间）：**

| 约束 | 符号 | 说明 | 示例 |
|------|------|------|------|
| **E（互斥/异）** | Exclude | 多个因中最多只有一个为真 | 支付方式：不能同时选"微信支付"和"支付宝" |
| **I（包含）** | Include | 多个因中至少有一个为真 | 联系方式：至少填手机号或邮箱之一 |
| **O（唯一）** | Only One | 多个因中有且仅有一个为真 | 性别：男或女，必选其一 |
| **R（要求）** | Require | 因A为真时，因B必须也为真 | 使用优惠券 → 必须登录 |
| **M（屏蔽/强制）** | Mask | 因A为真时，因B必须为假 | 选择"不开发票"→ 发票类型选项必须不可用 |

**设计步骤：**
1. 分析需求规格，识别所有的"因"（输入条件）和"果"（输出结果/动作）
2. 分析因和果之间的逻辑关系（与/或/非）
3. 分析因与因之间的约束关系（E/I/O/R/M）
4. 画出因果图：左边画因、右边画果、中间用逻辑符号连接、因之间标注约束
5. 将因果图转换为判定表（每个因作为判定表的条件，每个果作为判定表的动作）
6. 根据判定表设计测试用例

### 实际使用场景（5个）

**场景1：ATM自动取款机**

- **因（输入条件）**：
  1. 插入银行卡
  2. 密码正确
  3. 选择取款
  4. 余额充足
  5. ATM内现金充足
- **果（输出结果）**：
  1. 吐钞
  2. 打印凭条
  3. 退卡
  4. 显示"密码错误"
  5. 显示"余额不足"
  6. 显示"ATM现金不足"
  7. 吞卡（密码连续3次错误）

- **逻辑关系**：
  - (密码正确 AND 选择取款 AND 余额充足 AND ATM现金充足) → (吐钞 AND 打印凭条 AND 退卡)
  - (密码正确 AND 选择取款 AND 余额不足) → 显示"余额不足" AND 退卡
  - (密码正确 AND 选择取款 AND 余额充足 AND ATM现金不足) → 显示"ATM现金不足" AND 退卡

**场景2：自动售货机**

- **因（输入条件）**：
  1. 投币金额 ≥ 商品价格
  2. 投币金额 < 商品价格
  3. 按下了商品选择按钮
  4. 商品库存 > 0
  5. 商品库存 = 0
- **果（输出结果）**：
  1. 推出商品
  2. 找零
  3. 退币
  4. 显示"金额不足"
  5. 显示"已售罄"

- **约束**：
  - 因1和因2互斥（E约束）—金额不能既≥价格又<价格
  - 因4和因5互斥（E约束）—库存不能既>0又=0

- **逻辑关系**：
  - (投币≥价格 AND 按下按钮 AND 库存>0) → (推出商品 AND 找零)
  - (投币<价格 AND 按下按钮) → 显示"金额不足"

  - 因1和因2互斥（E约束）
  - 因3为真时 → 因1被屏蔽（M约束，超重时忽略楼层按钮）
  - 因5为真时 → 所有其他因被屏蔽（M约束，紧急停止最高优先级）

- **因（输入条件）**：

  5. 人体感应检测到无人

  9. 模式选择：制热
  10. 模式选择：送风

  1. 启动压缩机（制冷）
  2. 启动加热（制热）
  3. 仅送风
  4. 停止运行
  5. 进入节能模式（温度容差调大到±3°C）
  6. 自动关机

- **约束**：
  - 因8/9/10互斥（O约束—必须且只能选一个模式）

  - 因4/5互斥（E约束—有人/无人二选一）
  - 因6/7互斥（E约束—峰电/谷电二选一）

**场景5：在线考试系统自动判分**

- **因（输入条件）**：
  1. 客观题总分 ≥ 60
  2. 客观题总分 < 60
  3. 主观题总分 ≥ 60（百分制换算后）
  4. 主观题总分 < 60
  5. 缺考标记 = True
  6. 作弊标记 = True
  7. 考试时长是否用完（是/否）
  8. 是否提前交卷
- **果（输出结果）**：
  1. 考试通过（总分≥60）
  2. 考试不通过
  3. 成绩无效（缺考）
  4. 成绩无效 + 标记作弊
  5. 需重考/补考
  6. 可查看成绩单
  7. 不可查看成绩单

- **约束**：
  - 因5（缺考）为真 → 屏蔽因1/2/3/4（M约束，缺考时客观主观分无意义）
  - 因6（作弊）为真 → 屏蔽所有正常评分逻辑（M约束）
  - 因5和因6互斥（E约束—不可能同时缺考和作弊）

- **逻辑关系**：
  - (缺考=True) → 成绩无效(缺考)
  - (作弊=True) → 成绩无效+标记作弊
  - (客观≥60 AND 主观≥60 AND NOT缺考 AND NOT作弊) → 通过
  - (客观<60 OR 主观<60) AND NOT缺考 AND NOT作弊 → 不通过/补考

## 3.1 场景分析法

### 通俗释义

顺着用户实际操作走完整业务，测正常走完流程、可选分支、出错中断、中途退出、做完再撤销五类操作。

### 详细解释

场景分析法（Scenario Analysis）从最终用户的真实使用视角出发，模拟业务操作的完整流程。它不是孤立的测试单个功能点，而是将多个功能串在一起，形成端到端（End-to-End）的业务场景。场景分析法特别适用于业务流程复杂、操作路径多样的系统。

**场景的五大类型：**

| 类型 | 说明 | 测试重点 |
|------|------|----------|
| **基本流** | 用户最常用的正常路径，从头到尾顺利完成目标 | 核心业务闭环、数据一致性 |
| **备选流** | 在主流程中用户可以选择的不同分支 | 分支逻辑正确性、选择后流程衔接 |
| **异常流** | 流程中遇到系统异常/业务规则限制而中断 | 错误提示、数据回滚、容错处理 |
| **中途终止** | 用户主动在流程中间取消/退出 | 已操作步骤的状态处理、未完成数据的清除 |
| **逆向回滚** | 已完成操作后进行撤销、退款、退货等逆操作 | 逆向流程完整性、相关数据联动回滚 |

**设计步骤：**
1. 梳理业务流程图，标注所有路径
2. 识别基本流（一条完整的成功路径）
3. 识别各节点的备选流（如不同支付方式、不同配送方式）
4. 识别异常流（余额不足、网络超时、库存不足等）
5. 识别中途终止场景（取消订单、退出流程）
6. 识别逆向回滚场景（退款、退换货、撤回审批）
7. 为每种场景设计测试用例，覆盖"前置条件 → 操作步骤 → 预期结果 → 后置验证"

### 实际使用场景（5个）

- **逆向回滚**：请假已生效 → 员工申请销假 → 恢复年假余额 → 考勤记录更新

**场景3：银行跨行转账**

- **基本流**：登录网银 → 选择"跨行转账"→ 输入收款方姓名 → 输入收款账号 → 选择收款银行 → 输入转账金额（10000元）→ 确认信息 → 输入交易密码 → 输入短信验证码 → 转账成功 → 余额扣减 → 生成电子回单
- **备选流1**：选择"实时到账"vs"普通到账（2小时内）"vs"次日到账"
- **备选流2**：从"常用收款人"中选择，无需重复录入
- **异常流1**：转账金额超过单日限额 → 提示"超出当日累计限额"→ 转账失败
- **异常流2**：收款账号与姓名不匹配 → 退回转账
- **异常流3**：收款银行系统维护中 → 提示"对方银行暂不可用，请稍后再试"
- **中途终止**：输入密码前点击取消 → 退出转账流程，无金额变动
- **逆向回滚**：转账已完成 → 24小时内申请撤回（需银行客服介入，非标准退款流程）

**场景4：机票预订系统**

- **基本流**：打开订票系统 → 搜索（出发:北京, 到达:上海, 日期:2025-06-20）→ 展示航班列表 → 选择航班 → 选择舱位（经济舱）→ 填写乘客信息（姓名、身份证号、手机号）→ 选择增值服务（行李额、选座）→ 确认订单 → 支付 → 出票成功 → 发送行程单到邮箱
- **备选流1**：往返程预订（非单程）
- **备选流2**：多人同时预订（添加多个乘客）
- **备选流3**：使用里程积分抵扣部分票价
- **异常流1**：支付时该舱位只剩1张票，2人预订 → 出票失败，自动退款
- **异常流2**：乘客姓名与身份证号不匹配 → 提示修正
- **异常流3**：所选航班因天气原因取消 → 免费改签或全额退款
- **中途终止**：选择航班后退出 → 订单不保留（机票无"待支付"缓存机制）
- **逆向回滚**：已出票 → 申请退票（收取手续费）→ 退款 → 退票凭证 → 座位释放；或申请改签 → 费用重新计算

**场景5：社交平台内容发布**

- **基本流**：登录APP → 点击"发布"→ 选择发布类型（图文）→ 编辑文字内容 → 添加图片（从相册选择3张）→ 添加话题标签（#旅行）→ 添加位置信息 → 设置可见范围（公开/好友可见/仅自己）→ 点击"发布"→ 发布成功 → 出现在个人主页和关注者信息流
- **备选流1**：发布纯文字无图片 → 无需选图步骤
- **备选流2**：发布视频内容 → 需等待视频上传和转码
- **备选流3**：定时发布 → 设置未来某个时间点自动发布
- **异常流1**：内容包含敏感词 → 发布时提示"内容不符合社区规范"→ 需修改后重试
- **异常流2**：视频上传过程中网络中断 → 自动保存草稿 → 网络恢复后可继续
- **异常流3**：图片超9张限制 → 提示"最多选择9张"
- **中途终止**：编辑内容后点击返回 → 弹出"是否保存草稿"→ 保存草稿/放弃
- **逆向回滚**：已发布内容 → 删除 → 确认删除 → 从信息流和主页移除 → 缓存清理

## 3.2 状态迁移法

### 通俗释义

梳理单据、订单所有状态，测试正常切换状态、非法强制跳转能否被系统拦截。

### 详细解释

状态迁移法（State Transition Testing）用于测试系统在不同状态之间的转换行为。许多业务对象（如订单、工单、审批单、账户等）都有明确的生命周期，在生命周期中状态按照特定规则进行转换。状态迁移法通过状态图来可视化所有可能的状态转换路径，并针对合法和非法转换设计测试用例。

**状态迁移的核心概念：**

| 概念 | 说明 |
|------|------|
| **状态（State）** | 系统/对象在某一时刻所处的情况（如：待支付、已支付、已发货） |
| **迁移（Transition）** | 从一个状态变为另一个状态的动作/事件 |
| **事件/触发（Event/Trigger）** | 引发状态迁移的行为（如：用户支付、商家发货、超时） |
| **守卫条件（Guard）** | 迁移发生前必须满足的条件（如：余额充足、库存足够） |
| **动作（Action）** | 迁移发生时的附带操作（如：发送通知、扣减库存） |

**状态迁移测试的两个维度：**

| 维度 | 测试内容 |
|------|----------|
| **合法正向流转** | 验证所有符合业务规则的状态转换能否正确执行，包括触发条件、守卫条件、附带动作 |
| **非法跳转拦截** | 验证系统能否正确拦截不允许的状态转换（如待支付→已完成，跳过已支付和已发货），包括前端绕过、接口直调、数据库直接修改等攻击方式 |

**设计步骤：**
1. 识别被测对象的所有可能状态
2. 确定初始状态和终态（可能多个）
3. 绘制状态图（UML状态图或状态迁移矩阵）
4. 列出所有合法的状态迁移路径（包括迁移的触发事件和守卫条件）
5. 列出所有非法的状态迁移路径
6. 为每条合法路径设计正向测试用例
7. 为每条非法路径设计负向测试用例（验证拦截）
8. 额外验证：并发迁移、超时自动迁移、数据持久化后状态一致性

> **扩展阅读**：上述设计步骤覆盖的是基本状态迁移（0-Switch，单步直接迁移）。如需更强的覆盖强度——如验证连续两步迁移序列（1-Switch）或三步迁移序列（2-Switch）——参见 [3.6 N-Switch测试法](#36-n-switch测试法0-switch1-switchn-switch)，它是状态迁移法的精细化扩展。

### 实际使用场景（5个）

**场景1：电商订单状态全生命周期**

- **状态列表**：待支付 → 已支付 → (已取消) → (已退款) → 待发货 → 已发货 → 运输中 → 已签收 → 已完成 / (退货中) → (已退款) / 交易关闭

- **合法正向流转测试**：
  1. 待支付 → 已支付：用户成功付款，订单状态变为"已支付"，库存扣减
  2. 待支付 → 已取消：用户主动取消，或30分钟超时未支付自动取消，库存回滚
  3. 已支付 → 待发货：支付回调成功，进入发货准备状态
  4. 待发货 → 已发货：商家填写物流单号，状态更新，触发发货通知
  5. 已发货 → 运输中：物流信息同步，各物流节点更新
  6. 已发货 → 已签收：物流显示签收，或用户手动确认收货
  7. 已签收 → 已完成：签收后7天无售后自动完成，或用户手动确认

- **非法跳转拦截测试**：
  1. 待支付 → 已发货：绕过支付环节直接发货（应被拦截）
  2. 待支付 → 已完成：跳过所有中间状态（应被拦截）
  3. 已完成 → 待支付：已完成订单不应回到待支付（应被拦截）
  4. 已退款 → 已发货：退款后不应再发货（应被拦截）
  5. 通过API直接修改状态：构造非法请求修改订单状态字段（应有权限+状态机校验）

- **状态迁移矩阵**（合法/非法）：

| 从\到 | 待支付 | 已支付 | 待发货 | 已发货 | 已签收 | 已完成 | 已取消 | 已退款 |
|-------|--------|--------|--------|--------|--------|--------|--------|--------|
| **待支付** | — | 合法 | 非法 | 非法 | 非法 | 非法 | 合法 | 非法 |
| **已支付** | 非法 | — | 合法 | 非法 | 非法 | 非法 | 合法 | 合法 |
| **待发货** | 非法 | 非法 | — | 合法 | 非法 | 非法 | 非法 | 非法 |
| **已发货** | 非法 | 非法 | 非法 | — | 合法 | 非法 | 非法 | 非法 |
| **已签收** | 非法 | 非法 | 非法 | 非法 | — | 合法 | 合法 | 非法 |
| **已完成** | 非法 | 非法 | 非法 | 非法 | 非法 | — | 非法 | 非法 |

**场景2：请假审批流程状态**

- **状态列表**：草稿 → 已提交 → 部门审批中 → (已驳回) → HR审批中 → 已通过 → (已撤回) → 已生效 → 已归档 / 已销假

- **合法正向流转测试**：
  1. 草稿 → 已提交：填写完请假信息后提交
  2. 已提交 → 部门审批中：自动分配给直属领导
  3. 部门审批中 → HR审批中：领导审批通过后流转至HR
  4. HR审批中 → 已通过：HR审批通过
  5. 已通过 → 已生效：请假开始时间到达，自动生效
  6. 已生效 → 已归档：请假结束后自动归档

- **分支与回退流转测试**：
  1. 部门审批中 → 已驳回：领导驳回，需填写驳回原因
  2. 已驳回 → 草稿（可修改后重新提交）：员工修改后重新提交
  3. 已提交/部门审批中/HR审批中 → 已撤回：员工主动撤回申请

- **非法跳转拦截测试**：
  1. 草稿 → 已生效：未提交审批直接生效（应拦截）
  2. 已驳回 → 已通过：被驳回后跳过重新审批直接通过（应拦截）
  3. 已归档 → 草稿：已归档的申请不应复活（应拦截）
  4. HR审批中 → 已撤回但跳过部门审批（应拦截—HR审批中撤回应先撤回HR审批再撤回部门审批）

**场景3：会员卡账户状态**

- **状态列表**：未激活 → 正常 → (冻结) → 已注销 / (挂失) → 已补卡

- **合法流转测试**：
  1. 未激活 → 正常：首次消费/充值后激活
  2. 正常 → 冻结：账户异常（如多次密码错误、可疑交易）或主动冻结
  3. 冻结 → 正常：客服解冻、身份验证通过后解冻
  4. 正常 → 挂失：用户主动挂失
  5. 挂失 → 已补卡：补办新卡后旧卡注销
  6. 正常 → 已注销：用户主动注销账户
  7. 冻结 → 已注销：冻结状态下注销（余额需处理）

- **非法跳转拦截测试**：
  1. 未激活 → 已注销：绕过正常使用直接注销（视业务规则可能允许）
  2. 已注销 → 正常：已注销账户复活（应拦截，除非有恢复机制）
  3. 挂失 → 正常：跳过补卡直接恢复（应拦截—需补办新卡）
  4. 已注销状态的卡被POS机刷卡消费（应返回"卡片已注销"）

- **自动化状态处理验证**：
  - 会员卡有效期到期 → 自动从未激活/正常 → 已过期（自动迁移）
  - 长期不活跃（如3年无消费）→ 自动从正常 → 冻结（静默冻结，保护性处理）

**场景4：运维工单/事件单状态**

- **状态列表**：新建 → 已分派 → 处理中 → (待补充信息) → (待审批变更) → 已解决 → (已驳回验收) → 已关闭 / (重新打开)

- **合法流转测试**：
  1. 新建 → 已分派：客服分派给对应工程师
  2. 已分派 → 处理中：工程师接受工单，开始处理
  3. 处理中 → 待补充信息：工程师需要客户提供更多信息
  4. 待补充信息 → 处理中：客户补充信息后继续处理
  5. 处理中 → 待审批变更：需要审批的变更操作
  6. 待审批变更 → 处理中：审批通过，继续执行
  7. 处理中 → 已解决：问题解决，等待确认
  8. 已解决 → 已关闭：客户确认/超时自动关闭
  9. 已解决 → 重新打开：客户不满意，重新打开工单
  10. 已关闭 → 重新打开：已关闭工单因相关原因重新打开

- **非法跳转拦截测试**：
  1. 新建 → 已解决：跳过处理直接解决（应拦截）
  2. 新建 → 已关闭：跳过所有环节直接关闭（应拦截）
  3. 已关闭 → 新建：重置工单状态（应拦截，用"重新打开"替代）
  4. 处理中 → 新建：回退到初始状态（应拦截）

**场景5：Git代码合并请求（Merge Request / Pull Request）状态**

- **状态列表**：Draft（草稿）→ Open（待评审）→ In Review（评审中）→ Changes Requested（需修改）→ Approved（已批准）→ Merged（已合并）/ Closed（已关闭）→ Reverted（已回滚）

- **合法流转测试**：
  1. Draft → Open：开发者完成代码，点击"Ready for Review"
  2. Open → In Review：评审者开始评审代码
  3. In Review → Changes Requested：评审者发现问题，要求修改
  4. Changes Requested → In Review：开发者修改后重新请求评审
  5. In Review → Approved：所有评审者批准
  6. Approved → Merged：代码合并到目标分支
  7. Open/In Review/Approved → Closed：主动关闭PR（不合并）
  8. Merged → Reverted：发现严重问题，回滚合并

- **非法跳转拦截测试**：
  1. Draft → Merged：跳过评审直接合并（应有分支保护规则拦截）
  2. Changes Requested → Merged：评审未通过但强制合并（需管理员权限）
  3. Merged → Open：已合并PR不应重新打开（应拦截）
  4. Closed → Merged：已关闭PR重新合并（应重新打开后再合并）
  5. 绕过UI直接调用API修改MR状态（后端应做状态机校验）

## 3.3 功能图法

### 通俗释义

结合场景流程 + 状态变化两种思路，既测业务步骤又测状态变更，多用于复杂复合型业务。

### 详细解释

功能图法（Function Map / Function Diagram）是一种综合性的测试设计方法，它融合了场景分析法（关注业务流程）和状态迁移法（关注状态转换）的特点。功能图法通过绘制功能流程图，同时描述系统功能的执行逻辑和数据/对象的状态变化，适用于业务逻辑复杂、功能和状态交织的系统。

> **方法来源说明**：功能图法并非ISTQB/ISO 29119标准中定义的标准黑盒测试设计方法。在ISTQB分类中，标准的黑盒/基于规格的测试设计技术为：等价类划分、边界值分析、判定表测试、状态转换测试、用例测试。功能图法主要见于国内部分测试教材和实践中，其实质是"场景分析法+状态迁移法"的复合运用。如果要在标准框架下归类，建议将对应场景拆分为场景分析法（基于业务流程）和状态迁移法（基于状态转换）分别应用。

**功能图的核心组成：**

| 组成部分 | 说明 | 来源 |
|----------|------|------|
| **功能节点** | 表示一个功能操作或处理步骤 | 场景分析法 |
| **状态节点** | 表示系统/对象在某个时点的状态 | 状态迁移法 |
| **流转路径** | 连接功能节点和状态节点的有向边 | 两者的结合 |
| **条件分支** | 根据不同条件走向不同路径 | 场景分析法 |
| **数据节点** | 标注关键数据的输入/输出/存储 | 扩展元素 |

**功能图的类型：**
1. **功能流程图（本地）**：描述单个模块内部的功能流转和状态变化
2. **功能层次图（全局）**：描述跨模块/跨系统的业务编排和数据流转
3. **功能-状态混合图**：同时标注功能步骤和对应的状态变化

**设计步骤：**
1. 分析需求，识别被测业务的所有功能节点
2. 识别各功能节点的输入数据、输出数据、处理逻辑
3. 识别业务涉及的对象及其状态变化点
4. 绘制功能-状态混合图：纵向为功能步骤，横向为状态变化
5. 识别图中的所有路径（从起始到各终点）
6. 对每条路径设计测试用例（包含操作步骤、数据、预期状态）
7. 重点验证功能节点之间的数据传递准确性和状态一致性

### 实际使用场景（5个）

**场景1：在线教育平台 - 课程学习全流程**

- **功能节点**：
  - F1: 浏览课程目录 → F2: 课程详情 → F3: 购买/报名 → F4: 开始学习 → F5: 观看视频 → F6: 完成课后作业 → F7: 参加阶段测验 → F8: 期末考试 → F9: 获得证书

- **状态节点**（学员课程状态）：
  - S1: 未报名 → S2: 已报名 → S3: 学习中 → S4: 已完成课时 → S5: 已通过测验 → S6: 已通过考试 → S7: 已获得证书

- **功能-状态混合路径测试**：
  1. **主路径**：F1→F2→F3→F4→F5→F6→F7→F8→F9，状态 S1→S2→S3→S4→S5→S6→S7
     - 验证点：每步完成后学习进度%更新，状态正确切换
  2. **跳课路径**：F4→直接播放第5课视频（跳过前4课），验证是否允许（视课程设置："学完解锁"还是"自由选择"）
  3. **未完成作业路径**：F5完成后跳过F6直接进入F7，验证是否拦截
  4. **重复考试路径**：F8不通过 → 重新学习 → 再次F8，验证重考次数限制和状态回退S6→S5
  5. **数据结构验证**：
     - F5观看视频50%退出 → 下次进入从50%继续（断点续播）
     - F6提交作业后，教师评分，成绩反写回学员记录
     - F9生成证书 → 证书编号、姓名、课程名、日期落库并防篡改

**场景2：仓库管理系统 - 出入库全流程**

- **功能节点**：

## 3.4 NS图测试法

### 通俗释义

把程序的处理逻辑画成NS图（Nassi-Shneiderman结构化流程图），然后根据图中的每个分支设计测试用例。是一种用图形化方式表达程序逻辑、强制使用结构化控制流的黑盒测试方法。

### 详细解释

NS图（Nassi-Shneiderman Diagram，也称盒图/结构化流程图）是由Ike Nassi和Ben Shneiderman于1972年提出的一种程序逻辑表达工具。与传统流程图不同，NS图完全禁止使用GOTO/跳转线，强制使用三种基本控制结构（顺序、选择、循环）的嵌套组合来表达程序逻辑，从而天然符合结构化程序设计原则。

NS图的核心元素：
- 顺序结构：上下堆叠的矩形框，按序执行
- 选择结构（if-then-else）：一个大的矩形被对角线分为左右两部分（或上下两部分），左边/上边是then分支，右边/下边是else分支
- 循环结构（while/do-while）：矩形内部包含一个循环体小矩形
- 多分支选择（switch/case）：矩形内部水平分割为多个分支

NS图测试法的设计思路：
1. 将需求规格或程序逻辑转化为NS图
2. 遍历NS图中的每一条可能路径（从顶部入口到底部出口）
3. 确保每条分支路径至少被一个测试用例覆盖
4. 尤其关注嵌套分支中深层路径的覆盖

NS图测试法与场景分析法的区别：场景分析法关注的是用户在系统层面的操作流程（端到端），而NS图测试法关注的是程序内部处理逻辑的结构化表达。

> **分类说明**：NS图（Nassi-Shneiderman Diagram）传统上属于**结构化程序设计工具**，在ISTQB/ISO 29119标准分类中更接近基于结构的白盒测试技术。本文档将其纳入黑盒测试方法，是基于以下实践考量：当NS图从**需求规格**（而非源代码）绘制时，其本质是对业务规则的结构化建模，属于黑盒范畴；但当NS图从代码逻辑反推绘制时，则属于白盒测试。读者应根据NS图的**绘制来源**（需求 vs 代码）判断其实际分类归属。

### 实际使用场景（2个）

**场景1：[业务规则] 优惠券使用资格判断逻辑**

- **需求**：判断用户是否可以使用某张优惠券，规则如下——(1)判断优惠券是否在有效期内，不在有效期则提示"已过期"；(2)在有效期内则判断订单金额是否满足最低消费门槛，不满足则提示"未达使用门槛"；(3)满足门槛则判断用户是否在黑名单中，在则提示"无使用资格"；(4)不在黑名单则判断优惠券是否已被使用，已使用则提示"已使用"；(5)未使用则允许使用，计算优惠后金额。
- **NS图结构**：这是一个顺序嵌套选择结构——从上到下5个判断节点，每个节点有"是→继续"/"否→终止并提示"两个分支。
- **测试路径设计（从NS图中提取）**：
  - 路径1（全部通过）：有效期内→满足门槛→不在黑名单→未使用→优惠生效
  - 路径2：不在有效期内→提示"已过期"→终止
  - 路径3：有效期内→不满足门槛→提示"未达使用门槛"→终止
  - 路径4：有效期内→满足门槛→在黑名单中→提示"无使用资格"→终止
  - 路径5：有效期内→满足门槛→不在黑名单→已使用→提示"已使用"→终止
- **共5条路径 → 5个用例**
- **验证点**：每条路径的预期提示文案正确、优惠计算金额正确、各路径之间互不干扰

**场景2：[算法验证] 电商价格计算逻辑**

- **需求**：商品最终价格的计算逻辑——(1)先判断是否参与满减活动，参与则先减满减金额；(2)再判断是否有品类折扣（如服装类8折），有则对满减后价格打折；(3)再判断是否可用优惠券，可用则对折后价格减优惠券金额；(4)最后判断是否包邮，不包邮则加运费。
- **NS图结构**：顺序嵌套选择——4个判断节点逐层嵌套，每个节点的计算影响下一节点的输入。
- **测试路径设计**（代表性子集，全组合 2^4=16 条路径，以下选取4条典型路径覆盖核心计算逻辑）：
  - 路径1：参与满减+有品类折扣+有优惠券+包邮 → 最终价格 = ((原价-满减)×折扣)-优惠券
  - 路径2：参与满减+无品类折扣+有优惠券+不包邮 → 最终价格 = (原价-满减)-优惠券+运费
  - 路径3：不参与满减+有品类折扣+无优惠券+包邮 → 最终价格 = 原价×折扣
  - 路径4：不参与满减+无品类折扣+无优惠券+不包邮 → 最终价格 = 原价+运费
- **验证点**：每种路径下计算顺序严格执行（顺序错误可能导致金额差异）、金额精度到分、各步骤计算结果正确

## 3.5 NS图转换法

### 通俗释义

把业务需求文档里的文字描述或流程图，按照固定规则转换成NS图格式，然后基于NS图生成测试用例。本质是先把需求结构化，再从结构化模型中提取用例——"转换即评审"，转换过程本身就能暴露需求中的逻辑漏洞。

### 详细解释

NS图转换法（NS Diagram Conversion Method）的核心是将非结构化的需求描述（自然语言、原始流程图、业务规则文档）通过一定的转换规则，转化为标准化的NS图，然后基于NS图设计测试用例。这个过程本质上是在做"需求结构化"——将模糊的文字描述转化为精确的逻辑表达。

NS图转换的常见规则：
- 将"如果...则...否则..."句式映射为选择结构
- 将"重复...直到..."、"当...时循环..."映射为循环结构
- 将"第一步...第二步..."依序执行的描述映射为顺序结构
- 将嵌套的条件判断映射为嵌套选择结构

NS图转换法的核心价值在于：转换过程本身就会暴露需求中的逻辑漏洞（如缺少else分支、循环终止条件不明确、条件之间矛盾等）。这种"转换即评审"的效果使得NS图转换法不仅是测试设计工具，也是需求验证工具。

NS图转换法与NS图测试法的关系：NS图测试法是直接基于已有的NS图设计用例；NS图转换法关注的是如何从原始需求生成NS图。两者通常配合使用——先用NS图转换法将需求转为NS图，再用NS图测试法从NS图中提取测试路径。**本质上，两者是同一方法论的两个阶段（建模阶段 vs 用例生成阶段），而非两种独立的方法。**

### 实际使用场景（2个）

**场景1：[需求评审] 订单退款规则文本转换**

- **需求原文**："用户申请退款后，系统判断订单状态。如果订单未发货，直接退款；如果订单已发货，需判断是否已签收。已签收的需用户先退货再退款，未签收的需拦截物流后退款。退款金额需扣除已使用的优惠券和积分。如果优惠券已过期，按优惠券面额扣除。"
- **转换过程**：
  - 步骤1：识别判断节点 → 订单状态判断（未发货/已发货）、签收判断（已签收/未签收）、优惠券过期判断（已过期/未过期）
  - 步骤2：绘制NS图结构 → 顶层选择（订单状态），分支1（未发货→直接退款），分支2（已发货→嵌套选择（签收状态）→ 子分支1（已签收→退货后退款），子分支2（未签收→拦截物流后退款）），退款金额计算模块（嵌套选择：优惠券是否过期）
  - 步骤3：转换中发现的逻辑漏洞：需求未说明"订单已完成"状态如何处理（如已过售后期）；"拦截物流"失败怎么办？没有异常处理分支；"退款金额扣除优惠券"和之前的"拦截物流后退款"是什么关系？顺序不明确
- **测试路径提取**：共识别5条有效路径 + 2个需求模糊点需澄清
- **验证点**：转换后的NS图逻辑完整、无遗漏分支、异常场景有对应处理

**场景2：[业务流程] 审批流条件转换**

- **需求原文**："员工提交请假申请后，由直属领导审批。如果请假天数≤3天，领导审批通过后直接生效；如果请假天数>3天，领导审批通过后还需HR审批。如果领导审批驳回，退回员工修改。如果HR审批驳回，也退回员工修改。如果员工在审批过程中撤回，流程结束。"
- **转换过程**：
  - 步骤1：识别判断节点 → 天数判断（≤3天/>3天）、领导审批结果（通过/驳回）、HR审批结果（通过/驳回）、是否撤回
  - 步骤2：绘制NS图 → 顶层顺序结构（提交→领导审批判断→嵌套），领导审批判断为选择结构（通过/驳回），通过分支嵌套天数判断（≤3→生效，>3→HR审批判断），HR审批判断也为选择结构（通过→生效，驳回→退回修改），驳回分支和撤回分支均为终止路径
  - 步骤3：转换中发现的逻辑漏洞：>3天时领导通过但HR审批超时的处理未定义；员工修改后重新提交，天数重新判断（修改可能导致天数变化）；撤回的时间和状态约束（如在HR审批中能否撤回？）
- **测试路径提取**：共识别6条路径（含各分支组合）
- **验证点**：NS图完整覆盖所有条件分支、异常路径（超时、并发审批等）在图中是否遗漏

## 3.6 N-Switch测试法（0-Switch/1-Switch/N-Switch）

### 通俗释义

状态迁移法的精细化版本——不仅测直接的状态切换（0-Switch/单步迁移），还要测隔一个状态的间接切换（1-Switch）、甚至隔N个状态的切换（N-Switch），以确保状态机在各种跳转路径下都正确。

### 详细解释

N-Switch测试法（N-Switch Testing / N-Switch Coverage）是状态迁移测试的扩展覆盖准则，来源于状态机测试理论。其核心概念是按"转换步数"定义不同的覆盖强度：

- **0-Switch覆盖**：覆盖状态机中所有的单步（直接）状态迁移。即从状态A经过1次迁移到达状态B的所有可能路径。这是基本的状态迁移覆盖，等同于传统的状态迁移测试。
- **1-Switch覆盖**：覆盖状态机中所有的两步迁移序列。即从状态A经过状态B再到状态C（A→B→C）的所有可能路径。这能检测状态迁移的"传递效应"——连续两次迁移后的状态是否正确。
- **N-Switch覆盖**：覆盖状态机中所有N+1步的迁移序列。N越大，覆盖越完整，但路径数呈指数增长（状态迁移路径的指数爆炸）。

| 覆盖级别 | 覆盖内容 | 路径数规模 | 适用场景 |
|----------|---------|-----------|---------|
| 0-Switch | 所有单步直接迁移 | 少（约等于迁移边数） | 基本功能验证 |
| 1-Switch | 所有两步迁移序列 | 中等 | 常规业务系统 |
| 2-Switch | 所有三步迁移序列 | 较大 | 金融/通信协议 |
| N-Switch | 所有N+1步迁移序列 | 指数增长 | 安全苛求系统 |

N-Switch测试法在通信协议测试、嵌入式系统测试中应用广泛。研究表明，大多数状态相关的缺陷可以通过0-Switch和1-Switch覆盖发现，N>=2的覆盖主要用于安全苛求系统。

### 实际使用场景（2个）

**场景1：[通信协议] TCP连接状态机的N-Switch覆盖**

- **需求**：验证TCP协议栈的状态机实现。TCP连接状态包括：CLOSED、LISTEN、SYN_SENT、SYN_RCVD、ESTABLISHED、FIN_WAIT_1、FIN_WAIT_2、CLOSE_WAIT、LAST_ACK、TIME_WAIT、CLOSING。
- **0-Switch覆盖用例**：验证每个合法的单步状态迁移
  - CLOSED → LISTEN（被动打开）
  - LISTEN → SYN_RCVD（收到SYN）
  - CLOSED → SYN_SENT（主动打开）
  - SYN_SENT → ESTABLISHED（收到SYN+ACK）
  - ESTABLISHED → FIN_WAIT_1（主动关闭）
  - ... 共约15个单步迁移
- **1-Switch覆盖用例**：验证所有两步迁移序列
  - CLOSED → LISTEN → SYN_RCVD（被动打开后收到SYN）
  - CLOSED → SYN_SENT → ESTABLISHED（主动打开并建立连接）
  - ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2（主动关闭的两步）
  - ... 共约30-40个两步序列
- **验证点**：0-Switch覆盖所有直接状态迁移、1-Switch覆盖连续两个迁移的中间状态是否正确处理、TIME_WAIT状态的2MSL等待计时

**场景2：[业务系统] 订单状态机的N-Switch覆盖**

- **需求**：订单状态包括：待支付(PENDING)、已支付(PAID)、处理中(PROCESSING)、已发货(SHIPPED)、已签收(DELIVERED)、已完成(COMPLETED)、已取消(CANCELLED)、已退款(REFUNDED)、退货中(RETURNING)。
- **0-Switch覆盖**（已验证的传统状态迁移测试）→ 约12个直接迁移
- **1-Switch覆盖重点用例**（两步连续迁移）：
  - PENDING → PAID → PROCESSING：支付后自动进入处理中，验证处理中状态的初始化（如库存锁定、物流单预生成）
  - PAID → CANCELLED → REFUNDED：支付后取消并自动退款，验证退款金额、优惠券退回
  - PROCESSING → SHIPPED → DELIVERED：发货后自动/手动签收，验证中间状态（运输中）的物流信息同步
  - SHIPPED → RETURNING → REFUNDED：已发货商品退货退款，验证退货物流和退款联动
  - DELIVERED → COMPLETED → （7天后自动状态封闭）：验证完成状态不可再变更
- **2-Switch覆盖补充用例**（三步迁移）：
  - PENDING → PAID → CANCELLED → REFUNDED：支付后用户主动取消→退款，验证退款流水
  - PROCESSING → SHIPPED → DELIVERED → COMPLETED：完整的正向三步流程，验证状态历史记录完整性
- **验证点**：1-Switch覆盖可发现"连续两次快速迁移导致中间状态时间戳错误"类缺陷、2-Switch补充长链路状态一致性

## 附录：黑盒测试设计方法对比总结

### 一、单参数基础分析类（5种）

| 序号 | 方法名称 | 核心思想 | 适用场景 | 与其他方法的关系 |
|------|----------|----------|----------|------------------|
| 1.1 | **等价类划分法** | 分组选代表，减少用例数；分输入域/输出域/存储域三个维度 | 有明确输入域的功能点 | 常与边界值法搭配使用，是最基础的用例设计方法 |
| 1.2 | **边界值分析法** | BUG多出在边界，分常规边界/健壮边界/输出边界/隐式边界四类 | 有数值范围/长度限制/时间范围的场景 | 等价类法的补充，两者互补、互不归属 |
| 1.3 | **错误推测法** | 经验驱动，查漏补缺（ISTQB归类为基于经验的技术，非基于规格） | 高风险模块、项目收尾阶段 | 作为其他系统化方法的补充，不可单独依赖 |
| 1.4 | **基于需求测试法** | 逐条对照需求文档，保证需求全覆盖 | 所有测试的基础和起点 | 所有其他方法的基础，配合使用确保需求追溯 |
| 1.5 | **随机测试法** | 随机输入/操作，发现意外缺陷 | 稳定性测试、安全测试、查漏补缺 | 不可替代系统化方法，作为补充和最后一关 |

### 二、多条件参数组合类（14种）

| 序号 | 方法名称 | 核心思想 | 适用场景 | 与其他方法的关系 |
|------|----------|----------|----------|------------------|
| 2.1 | **穷举组合法（AC）** | 所有参数取值全排列，100%覆盖（All Combinations） | 参数极少（2-3个）的简单场景 | 其他组合方法的理论参照基准 |
| 2.2 | **基准选择法（BC）** | 选基准值+每次只变一个参数，线性增长（Basic Choice） | 参数多但需快速验证单参数影响 | 无法检测参数交互，需配合其他组合法；与OFAT等价 |
| 2.3 | **极值组合法（EC）** | 全覆盖取值最多的参数，其余固定（Extreme Combination） | 快速冒烟验证、初筛最复杂参数 | 用例数=最大参数取值数；不覆盖交互效应 |
| 2.4 | **正交试验法（OC）** | 正交表均匀采样，"整齐可比"（Orthogonal Combination） | 多参数多取值的配置/兼容性测试 | 极端组合需额外用边界值法补测 |
| 2.5 | **Pairwise/All-Pairs（PC）** | 覆盖所有两两参数组合，工具自动生成（Pairwise Combination） | 参数水平数不一致的场景、任意数量参数的组合测试 | PC即t=2；不保证三参数及以上交互组合的覆盖 |
| 2.6 | **t-way组合覆盖** | Pairwise泛化，覆盖任意t个参数的所有组合 | t=3用于金融/医疗核心系统，t>=4用于安全苛求系统 | t=2即PC(Pairwise)，t=N退化为AC(穷举) |
| 2.7 | **部分组合法** | 按策略（每值至少一次/重点复现/随机抽样）选取部分组合 | 测试时间紧张、无工具支持时的折中方案 | 覆盖率不可量化，可能遗漏交互缺陷 |
| 2.8 | **配对组合法** | 基于业务知识识别强关联参数对，"捆绑"测试 | 参数间有明确业务关联（如会员等级×折扣系数） | 不同于PC（Pairwise工具法），强调业务语义 |
| 2.9 | **单一因素组合法** | 一次只变一个参数（控制变量法），因果清晰 | 性能调优、单变量影响分析 | 与BC等价，完全无法检测参数交互效应 |
| 2.10 | **混合组合法** | 按参数风险分级，混合使用多种组合策略（元策略） | 实际项目中最务实的做法 | 高风险AC+中风险PC+低风险随机 |
| 2.11 | **等价组合法** | 先对参数取值做等价类划分，再对等价类做组合 | 参数取值空间很大（连续值、大量枚举）的场景 | 等价类划分法+组合测试的结合 |
| 2.12 | **边界组合法** | 对多参数的边界值做组合，"边界值×边界值" | 有明确数值范围的多参数系统 | 边界值分析法+组合测试的结合 |
| 2.13 | **判定表法** | 穷举条件组合，合并化简 | 多个条件互相影响结果的逻辑判断 | 因果关系复杂时可先用因果图法分析，再转换为判定表 |
| 2.14 | **因果图法** | 图形化分析条件和结果的逻辑关系 | 条件间有复杂逻辑约束的场景 | 是判定表的前置分析工具，画完后转换为判定表 |

### 三、流程/状态迁移组合类（6种）

| 序号 | 方法名称 | 核心思想 | 适用场景 | 与其他方法的关系 |
|------|----------|----------|----------|------------------|
| 3.1 | **场景分析法** | 模拟用户真实操作流程，分基本流/备选流/异常流/终止/回滚 | 业务流程长的端到端测试 | 可结合状态迁移法用于复杂场景 |
| 3.2 | **状态迁移法** | 验证状态转换的正向和非法拦截 | 有明确状态生命周期的业务对象 | 可与场景分析法结合用于复合业务 |
| 3.3 | **功能图法** | 同时覆盖功能步骤和状态变化 | 复杂复合型业务系统（国内教材方法，非ISTQB标准方法） | 是场景分析法+状态迁移法的综合运用 |
| 3.4 | **NS图测试法** | 将处理逻辑画成NS结构化流程图，遍历所有分支路径 | 程序内部处理逻辑复杂的业务规则验证 | 与场景分析法互补：场景法关注端到端流程，NS图法关注模块内部逻辑 |
| 3.5 | **NS图转换法** | 将非结构化需求文本转换为标准NS图，"转换即评审" | 需求文档中逻辑描述不清晰、存在歧义时 | 与NS图测试法配合：先转换后测试，转换过程暴露需求漏洞 |
| 3.6 | **N-Switch测试法** | 按转换步数（0-Switch/1-Switch/N-Switch）逐级提高状态机覆盖强度 | 通信协议测试、安全苛求系统、多状态连续迁移场景 | 状态迁移法的精细化扩展，0-Switch即传统状态迁移测试 |

> **分类备注**：ISTQB将测试技术分为三大类——(1)基于规格的黑盒技术（等价类/边界值/判定表/状态迁移/用例测试）；(2)基于结构的白盒技术（语句/判定/条件/MC/DC等）；(3)基于经验的技术（错误推测/探索性测试）。本文档中的"错误推测法"严格属于第(3)类，"功能图法"为国内实践产物而非ISTQB标准方法，纳入本文档旨在提供全面的实践参考。

---

> **使用建议**：实际项目中，通常以基于需求测试法为基础框架，根据被测功能的特点灵活组合其他方法。例如：一个用户注册需求，可以先用基于需求法梳理测试点，再用等价类+边界值设计输入校验用例，用场景法设计全流程用例，最后用错误推测法补充安全性用例。
