## 1. 等价类划分法

### 通俗释义

把数据分成好几组，同一组内数据程序处理逻辑一样，每组随便选 1 个数据做用例；分合法数据、非法数据，同时核对入参、页面返回、数据库存储三处数据。

### 详细解释

等价类划分法（Equivalence Partitioning）是一种系统化的测试用例设计技术，其核心思想是：将程序的输入域（也包括输出域和存储域）划分为若干个子集（等价类），每个子集中的任意数据对程序的处理逻辑是等效的。测试时，从每个等价类中选取一个代表性数据进行测试，就能代表该等价类中所有数据的测试效果。

**等价类的分类维度：**

| 维度 | 说明 | 示例 |
|------|------|------|
| **有效等价类** | 符合程序要求的、有意义的输入数据集合 | 年龄输入框接受1-120，则1-120范围内任意整数为有效等价类 |
| **无效等价类** | 不符合程序要求的、无意义的输入数据集合 | 年龄输入框的范围外数据：0、-1、121、字母、特殊符号、小数、空值 |
| **输入域等价** | 针对输入参数的等价类划分 | 用户名输入框的各类合法/非法字符组合 |
| **输出域等价** | 针对输出结果的等价类划分 | 根据输入值系统返回的不同结果类型分组 |
| **存储等价** | 针对数据存储（数据库字段）的等价类划分 | 数据库字段长度限制、类型约束、约束条件等 |

**设计步骤：**
1. 分析需求规格，确定输入条件
2. 为每个输入条件划分有效等价类和无效等价类（至少1个有效、2个以上无效）
3. 为每个等价类编号
4. 设计测试用例：覆盖所有有效等价类（尽可能合并到一个用例中）；每个无效等价类单独设计一个用例（避免无效数据之间互相屏蔽）
5. 同步检查：入参校验 → 页面/接口返回校验 → 数据库落库校验

### 实际使用场景（5个）

**场景1：用户注册 - 手机号输入框**

- **需求**：手机号为11位中国大陆手机号，以1开头，第二位为3/4/5/6/7/8/9
- **有效等价类**：合法的11位手机号（如13800138000）
- **无效等价类**：
  - 少于11位数字（如1380013800，10位）
  - 多于11位数字（如138001380001，12位）
  - 非1开头（如23800138000）
  - 第二位非3-9（如12000138000）
  - 包含非数字字符（如13800a38000）
  - 包含特殊符号（如138-0013-8000）
  - 为空（不输入任何内容）
  - 全角数字（如１３８００１３８０００）
- **用例设计**：有效等价类合并为1个用例；8个无效等价类各1个用例，共9个用例

**场景2：商品价格输入**

- **需求**：商品价格范围为 0.01 - 999999.99 元，保留两位小数
- **有效等价类**：
  - 最小金额 0.01
  - 中间金额（如 99.99、1234.56）
  - 最大金额 999999.99
- **无效等价类**：
  - 0（免费商品需特殊标记）
  - 负数（如 -1.00）
  - 超过最大值（如 1000000.00）
  - 超出小数位限制（如 99.999，三位小数）
  - 非数字（字母、特殊符号、中文数字）
  - 空值
  - 科学计数法（如 1e5）
- **用例设计**：3个有效等价类各1个用例（也可合并），7个无效等价类各1个用例

**场景3：学历下拉框选择**

- **需求**：学历可选择 博士/硕士/本科/大专/高中及以下
- **有效等价类**：5个合法选项各为一个等价类
- **无效等价类**：
  - 不选择（空值/null）
  - 通过前端绕过传非法值（如"博士后"、"其他"）
  - 传数字型值（如 1、2、3）
- **输出域等价**：选择不同学历时，后端应正确存储对应的枚举值（如博士=5, 硕士=4, 本科=3, 大专=2, 高中及以下=1）
- **用例设计**：5个有效等价类各自至少1个用例 + 3个无效等价类各1个用例

**场景4：文件上传类型校验**

- **需求**：仅支持上传 JPG/PNG/PDF 格式文件，单文件不超过10MB
- **有效等价类**：
  - JPG 格式文件（如 photo.jpg）
  - PNG 格式文件（如 screenshot.png）
  - PDF 格式文件（如 document.pdf）
- **无效等价类**：
  - 不允许的类型（如 .exe、.zip、.docx、.xlsx）
  - 无后缀名文件
  - 伪造后缀名（将 .exe 改为 .jpg 上传）
  - 空文件（0KB）
  - 文件名含特殊字符（如 ../path/traversal.jpg）
- **用例设计**：3个有效等价类 + 5个无效等价类，共8个用例

**场景5：身份证号校验**

- **需求**：18位公民身份证号（最后一位可为X/x）
- **有效等价类**：
  - 末位为数字的合法身份证号（如110101199001010012）
  - 末位为X的合法身份证号（如11010119900101001X）
  - 末位为x的合法身份证号（小写x）
- **无效等价类**：
  - 17位号码（老版身份证）
  - 19位号码
  - 包含字母（除末位X外，如A10101199001010012）
  - 末位为Y等非X字母
  - 包含特殊符号
  - 空值
  - 全角字符
  - 前17位含非数字（如11010119900101001a）
- **用例设计**：3个有效等价类（可合并为1-2个）+ 8个无效等价类

## 2. 边界值分析法

### 通俗释义

BUG 大多出在数值临界点，除正常临界值外，还要测远超范围的极端值，同时检查返回结果临界、前后端校验规则不一样带来的隐藏边界。

### 详细解释

边界值分析法（Boundary Value Analysis）是对等价类划分法的强力补充。实践证明，大多数软件缺陷都集中在输入/输出域的边界值附近。边界值分析的核心是"边界值、边界值±1"——程序员在写判断条件时最容易在 `<`、`<=`、`>`、`>=`、`==` 这些比较符上出错。

**边界值的四种类型：**

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **常规7点边界** | min-1, min, min+1, normal(中间值), max-1, max, max+1 | 数值范围校验 |
| **健壮边界** | 远超范围的极端值：极大值、极负值、0、空值、特殊字符 | 系统抗极端输入能力 |
| **输出边界** | 系统输出/返回结果的临界值 | 分页、报表、列表数量临界 |
| **隐式前后端边界** | 前端校验和后端校验规则不一致导致的隐藏边界 | js校验通过但后端校验不通过、数据库字段长度>接口限制等 |

**设计步骤：**
1. 确定输入/输出的边界值范围
2. 对每个边界取：边界值、边界值-1、边界值+1
3. 额外补充：空值、极端值（如 Integer.MAX_VALUE、Long.MIN_VALUE）、特殊值（如 0、-1、null）
4. 输出边界同样方法取值
5. 对比前后端校验规则，找出不一致的隐藏边界

### 实际使用场景（5个）

**场景1：分页列表 - 每页显示条数边界**

- **需求**：列表每页默认显示20条，用户可选择10/20/50/100条，数据总量不固定
- **边界测试点**：
  - 数据总量为0条时，列表展示空状态（输出边界：0条）
  - 数据总量为1条时，不应出现分页（输出边界：1条）
  - 数据总量恰好为20条时，应只有1页（输出边界：=pageSize）
  - 数据总量为21条时，应出现第2页（输出边界：pageSize+1）
  - 每页选择10条，数据总量为100条，验证第10页为最后页
  - 每页选择50条，数据总量为100条，验证共2页
  - 每页选择100条，大数据量场景（如10000条），验证分页性能
  - 手动输入每页0条、-1条、非数字、超大值（前端校验但后端默认值）

**场景2：优惠券满减规则边界**

- **需求**：全场满200减30，满500减100，满1000减250（每档优惠不叠加，自动取最高优惠）
- **边界测试点**：
  - 订单金额 199元 → 不享受优惠（min-1）
  - 订单金额 200元 → 满200减30，实付170（=min）
  - 订单金额 200.01元 → 满200减30，实付170.01（min+0.01）
  - 订单金额 499元 → 满200减30，应只减30而非减100（跨档边界验证 min-1）
  - 订单金额 500元 → 满500减100，实付400（=min）
  - 订单金额 500.01元 → 满500减100（min+0.01 档位切换）
  - 订单金额 999元 → 满500减100，验证是否误用1000档（跨档）
  - 订单金额 1000元 → 满1000减250，实付750
  - 订单金额 0元 → 异常场景（健壮边界）
  - 订单金额为负数 → 系统应拦截（健壮边界）

**场景3：文件上传大小限制**

- **需求**：支持上传图片，单文件 ≤ 5MB，一次最多上传9张
- **边界测试点**：
  - 文件大小恰好 4.99MB → 上传成功（max-0.01）
  - 文件大小恰好 5.00MB → 上传成功（=max）
  - 文件大小恰好 5.01MB → 上传失败，提示超出限制（max+0.01）
  - 文件大小恰好 0KB → 空文件，能否上传？（min边界）
  - 文件大小 1字节 → 极小文件，能否上传？（min+1）
  - 文件大小 100MB → 远超限制（健壮边界，后端拦截）
  - 文件大小 1GB → 极端大文件（健壮边界，可能引发OOM）
  - 上传数量：9张 → 正好满（=max）
  - 上传数量：10张 → 超出限制（max+1）
  - **隐式前后端边界**：前端限制5MB，但后端Nginx限制1MB → 前端可选中5MB文件但上传到1MB时报413错误

**场景4：密码长度校验**

- **需求**：用户密码长度 6-20 位，必须包含字母和数字
- **边界测试点**：
  - 5位合法密码（如 a1b2c）→ 提示太短（min-1）
  - 6位合法密码（如 a1b2c3）→ 通过（=min）
  - 7位合法密码（如 a1b2c3d）→ 通过（min+1）
  - 19位合法密码 → 通过（max-1）
  - 20位合法密码 → 通过（=max）
  - 21位合法密码 → 提示太长（max+1）
  - 空密码（0位）→ 提示不能为空（健壮边界）
  - 1位密码 → 提示太短（健壮边界）
  - 100位密码 → 远超限制，后端可能截断导致绕过（健壮边界 + 隐式边界）
  - **隐式边界**：数据库字段设为 VARCHAR(20)，如果后端漏校验，写入时报错或静默截断

**场景5：活动时间范围**

- **需求**：限时抢购活动，2025-06-01 10:00:00 至 2025-06-03 23:59:59
- **边界测试点**：
  - 2025-06-01 09:59:59 → 活动未开始，商品显示原价（min-1秒）
  - 2025-06-01 10:00:00 → 活动开始，显示活动价（=min）
  - 2025-06-01 10:00:01 → 活动进行中（min+1秒）
  - 2025-06-03 23:59:58 → 活动进行中（max-1秒）
  - 2025-06-03 23:59:59 → 活动最后一秒，活动价（=max）
  - 2025-06-04 00:00:00 → 活动已结束，恢复原价（max+1秒）
  - **跨时区边界**：用户在不同时区（如UTC+0），系统时间边界是否正确
  - **隐式边界**：前端倒计时和后端时间可能不一致（服务器时间 vs 用户本地时间），导致前端显示活动进行中但后端拒绝下单

## 3. 场景分析法

### 通俗释义

顺着用户实际操作走完整业务，测正常走完流程、可选分支、出错中断、中途退出、做完再撤销五类操作。

### 详细解释

场景分析法（Scenario Analysis）从最终用户的真实使用视角出发，模拟业务操作的完整流程。它不是孤立的测试单个功能点，而是将多个功能串在一起，形成端到端（End-to-End）的业务场景。场景分析法特别适用于业务流程复杂、操作路径多样的系统。

**场景的五大类型：**

| 类型 | 说明 | 测试重点 |
|------|------|----------|
| **基本流** | 用户最常用的正常路径，从头到尾顺利完成目标 | 核心业务闭环、数据一致性 |
| **备选流** | 在主流程中用户可以选择的不同分支 | 分支逻辑正确性、选择后流程衔接 |
| **异常流** | 流程中遇到系统异常/业务规则限制而中断 | 错误提示、数据回滚、容错处理 |
| **中途终止** | 用户主动在流程中间取消/退出 | 已操作步骤的状态处理、未完成数据的清除 |
| **逆向回滚** | 已完成操作后进行撤销、退款、退货等逆操作 | 逆向流程完整性、相关数据联动回滚 |

**设计步骤：**
1. 梳理业务流程图，标注所有路径
2. 识别基本流（一条完整的成功路径）
3. 识别各节点的备选流（如不同支付方式、不同配送方式）
4. 识别异常流（余额不足、网络超时、库存不足等）
5. 识别中途终止场景（取消订单、退出流程）
6. 识别逆向回滚场景（退款、退换货、撤回审批）
7. 为每种场景设计测试用例，覆盖"前置条件 → 操作步骤 → 预期结果 → 后置验证"

### 实际使用场景（5个）

**场景1：电商下单全流程**

- **基本流**：打开APP → 搜索"蓝牙耳机" → 点击商品进入详情 → 选择规格（颜色:黑色, 数量:1）→ 加入购物车 → 进入购物车 → 勾选商品 → 点击结算 → 选择收货地址 → 选择支付方式（微信支付）→ 确认订单 → 支付 → 支付成功 → 查看订单状态为"待发货"→ 商家发货 → 确认收货 → 发表评价
- **备选流1**：不通过购物车，直接"立即购买"
- **备选流2**：选择货到付款而非微信支付
- **备选流3**：使用优惠券/积分抵扣后支付
- **异常流1**：支付时余额不足 → 提示"余额不足"→ 订单变为"待支付"→ 可在30分钟内继续支付
- **异常流2**：支付过程中订单被其他人抢购导致库存不足 → 支付失败 → 自动退款 → 订单关闭
- **异常流3**：支付时网络超时 → 订单状态变为"待支付"→ 需防重复支付（幂等性）
- **中途终止**：下单后在支付页面点击返回 → 订单保留在"待支付"状态 → 超时自动取消
- **逆向回滚**：已收货商品申请退货 → 填写退货原因 → 商家审核 → 退货地址下发 → 寄回 → 商家收货确认 → 退款到账

**场景2：OA请假审批流程**

- **基本流**：员工登录OA → 进入请假模块 → 点击"新建请假申请"→ 选择请假类型（年假）→ 选择起止时间（2025-06-15至2025-06-17共3天）→ 填写请假原因 → 选择工作交接人 → 提交 → 直属领导收到审批通知 → 领导审批通过 → HR审批通过 → 请假生效 → 系统自动扣减年假余额
- **备选流1**：请假类型选择"事假"（不扣年假余额但影响薪资）
- **备选流2**：请假半天而非全天
- **备选流3**：部门领导不在，自动转交上级领导审批
- **异常流1**：请假天数超过剩余年假天数 → 提示"年假余额不足"→ 无法提交
- **异常流2**：领导审批驳回 → 员工收到驳回通知（含驳回原因）→ 可修改后重新提交
- **异常流3**：HR审批超时 → 自动通过/提醒（视规则配置）
- **中途终止**：员工提交后、领导审批前 → 员工主动撤回 → 流程结束
- **逆向回滚**：请假已生效 → 员工申请销假 → 恢复年假余额 → 考勤记录更新

**场景3：银行跨行转账**

- **基本流**：登录网银 → 选择"跨行转账"→ 输入收款方姓名 → 输入收款账号 → 选择收款银行 → 输入转账金额（10000元）→ 确认信息 → 输入交易密码 → 输入短信验证码 → 转账成功 → 余额扣减 → 生成电子回单
- **备选流1**：选择"实时到账"vs"普通到账（2小时内）"vs"次日到账"
- **备选流2**：从"常用收款人"中选择，无需重复录入
- **异常流1**：转账金额超过单日限额 → 提示"超出当日累计限额"→ 转账失败
- **异常流2**：收款账号与姓名不匹配 → 退回转账
- **异常流3**：收款银行系统维护中 → 提示"对方银行暂不可用，请稍后再试"
- **中途终止**：输入密码前点击取消 → 退出转账流程，无金额变动
- **逆向回滚**：转账已完成 → 24小时内申请撤回（需银行客服介入，非标准退款流程）

**场景4：机票预订系统**

- **基本流**：打开订票系统 → 搜索（出发:北京, 到达:上海, 日期:2025-06-20）→ 展示航班列表 → 选择航班 → 选择舱位（经济舱）→ 填写乘客信息（姓名、身份证号、手机号）→ 选择增值服务（行李额、选座）→ 确认订单 → 支付 → 出票成功 → 发送行程单到邮箱
- **备选流1**：往返程预订（非单程）
- **备选流2**：多人同时预订（添加多个乘客）
- **备选流3**：使用里程积分抵扣部分票价
- **异常流1**：支付时该舱位只剩1张票，2人预订 → 出票失败，自动退款
- **异常流2**：乘客姓名与身份证号不匹配 → 提示修正
- **异常流3**：所选航班因天气原因取消 → 免费改签或全额退款
- **中途终止**：选择航班后退出 → 订单不保留（机票无"待支付"缓存机制）
- **逆向回滚**：已出票 → 申请退票（收取手续费）→ 退款 → 退票凭证 → 座位释放；或申请改签 → 费用重新计算

**场景5：社交平台内容发布**

- **基本流**：登录APP → 点击"发布"→ 选择发布类型（图文）→ 编辑文字内容 → 添加图片（从相册选择3张）→ 添加话题标签（#旅行）→ 添加位置信息 → 设置可见范围（公开/好友可见/仅自己）→ 点击"发布"→ 发布成功 → 出现在个人主页和关注者信息流
- **备选流1**：发布纯文字无图片 → 无需选图步骤
- **备选流2**：发布视频内容 → 需等待视频上传和转码
- **备选流3**：定时发布 → 设置未来某个时间点自动发布
- **异常流1**：内容包含敏感词 → 发布时提示"内容不符合社区规范"→ 需修改后重试
- **异常流2**：视频上传过程中网络中断 → 自动保存草稿 → 网络恢复后可继续
- **异常流3**：图片超9张限制 → 提示"最多选择9张"
- **中途终止**：编辑内容后点击返回 → 弹出"是否保存草稿"→ 保存草稿/放弃
- **逆向回滚**：已发布内容 → 删除 → 确认删除 → 从信息流和主页移除 → 缓存清理

## 4. 错误推测法

### 通俗释义

依靠过往项目踩坑经验、同类产品常见 bug，预判容易出错位置，额外补充测试用例，用来填补别的方法漏测的点。

### 详细解释

错误推测法（Error Guessing）是一种基于经验和直觉的测试技术。它不依赖系统化的规则或模型，而是依赖测试人员对系统的深入理解、对过往项目中积累的缺陷模式的认知，以及对行业中常见错误类型的了解。错误推测法的效果高度依赖于测试人员的经验水平。

> **ISTQB分类说明**：在ISTQB标准测试分类中，错误推测法（Error Guessing）被归类为**基于经验的测试技术（Experience-Based Testing）**，而非**基于规格的测试技术（Specification-Based Testing / 黑盒测试）**。基于经验的测试技术还包括探索性测试（Exploratory Testing）等。本文档将其纳入黑盒测试方法汇总，是因为国内测试实践中习惯将其作为黑盒测试的补充方法使用，但读者应了解在标准分类体系中的正确定位。

**错误推测的知识来源：**

| 来源 | 说明 |
|------|------|
| **历史缺陷数据库** | 本团队过往项目中记录的高频缺陷类型和位置 |
| **同类产品已知问题** | 竞品或同类型系统的已知漏洞和缺陷 |
| **技术栈常见陷阱** | 特定语言、框架、中间件的已知坑点 |
| **通用错误模式** | 如空指针、数组越界、并发问题、精度丢失等 |
| **边界条件遗漏** | 等价类和边界值分析可能覆盖不全的特殊情况 |
| **业务规则冲突** | 看似不相关但实际联动时产生冲突的规则 |

**常见的高风险测试靶点：**

- 输入框：SQL注入、XSS脚本注入、超长字符串、特殊Unicode字符、emoji
- 时间处理：闰年/非闰年、跨时区、夏令时、2038年问题、2月29日
- 金额计算：浮点数精度、负数金额、科学计数法、大额溢出
- 并发操作：重复提交、并发修改同一数据、库存超卖
- 网络异常：超时、断网、弱网、重连后的状态恢复
- 编码问题：不同字符编码乱码、特殊字符处理

**设计步骤：**
1. 回顾历史项目缺陷库和同类产品问题清单
2. 分析被测系统的技术栈和架构，列出常见陷阱
3. 识别业务规则中容易产生冲突的地方
4. 推测每个高风险位置可能的错误类型
5. 设计针对性测试用例，明确输入、操作路径、预期行为
6. 将错误推测用例作为其他系统化方法的补充，而非替代

### 实际使用场景（5个）

**场景1：搜索功能 - 注入与特殊字符攻击**

- **推测错误点**：搜索框是最常见的攻击入口，程序员可能未做参数化查询，或未对特殊字符做转义
- **测试用例**：
  1. SQL注入：输入 `' OR '1'='1` 或 `'; DROP TABLE products; --`，验证是否被防注入过滤
  2. XSS注入：输入 `<script>alert(1)</script>` 或 `<img src=x onerror=alert(1)>`，验证搜索结果页是否执行了脚本（应做HTML转义）
  3. 正则注入（ReDoS）：输入 `((((a)*)*)*)*b` 超长嵌套正则，验证是否导致服务CPU跑满
  4. 空字符注入：输入 `%00`（NULL字符），验证后端对不同语言（C/C++/Java/Python）的空字符处理
  5. LDAP注入：输入 `*)(uid=*))(|(uid=*`，如果搜索底层用到LDAP需验证
  6. 超长输入：输入10000个字符的搜索词，验证是否有长度限制和性能问题
  7. Unicode混淆：输入看起来像正常字符的Unicode特殊字符（如全角字母、零宽空格）

**场景2：支付系统 - 金额精度与并发**

- **推测错误点**：金额计算使用浮点数导致精度丢失，或并发处理不当导致资金问题
- **测试用例**：
  1. 浮点数精度：支付金额 0.1+0.2，验证是否等于0.3而非0.30000000000000004（应用BigDecimal/整数分）
  2. 并发重复支付：模拟用户在网络延迟下连点两次"支付"按钮，验证是否只扣款一次（幂等性验证）
  3. 支付回调与查询并发：模拟第三方支付回调到达的同时用户查询订单，验证订单状态最终一致性
  4. 负金额支付：传入金额=-100，验证系统是否拦截
  5. 科学计数法金额：传入金额=1e10，验证系统是否正确解析或拒绝
  6. 超时后成功：模拟支付请求在服务端已成功但客户端超时，用户再次支付，验证不会重复扣款
  7. 退款金额大于支付金额：验证退款校验逻辑
  8. 零金额支付：金额=0，验证是否允许（如0元购漏洞）

**场景3：登录认证 - 安全绕过**

- **推测错误点**：认证逻辑是安全的核心，程序员可能在Session管理、Token校验、密码策略上出现疏忽
- **测试用例**：
  1. 暴力破解：连续输入错误密码100次，验证是否有锁定机制和验证码
  2. Session未清除：登录后退出，使用之前的Session ID访问接口，验证是否被拒绝
  3. Token篡改：修改JWT Token中的用户ID字段，验证后端是否校验签名而非仅解析Payload
  4. 验证码绕过：抓包后删除验证码参数或不传验证码字段，验证后端是否严格校验
  5. 密码明文传输：抓包检查登录请求中密码是否明文传输（应使用加密或HTTPS）
  6. 记住我功能：勾选"记住我"登录后，删除持久化Cookie，验证是否仍能自动登录
  7. 多端登录：同一账号在手机和电脑同时登录，互踢逻辑是否正确
  8. 弱密码策略：使用123456、password等弱密码注册，验证密码强度校验

**场景4：数据导入导出 - 编码与格式**

- **推测错误点**：导入导出涉及文件格式、编码、大小等多种边界，容易在特殊数据上出问题
- **测试用例**：
  1. 编码冲突：导入UTF-8编码的文件但含GBK特有字符（如中文繁体字），验证是否乱码
  2. BOM头处理：导入带BOM头的UTF-8文件，验证首行是否出现乱码字符
  3. 空文件导入：导入0KB的空文件，验证提示信息是否友好
  4. 超大数据量导入：导入100万行Excel，验证是否超时、OOM、或部分成功部分失败
  5. 循环引用：导入含公式的Excel，单元格A1引用B1，B1引用A1，验证处理逻辑
  6. 特殊字符：单元格中包含换行符、逗号、引号（CSV格式），验证导入后字段是否错位
  7. 合并单元格：导入含合并单元格的Excel，验证数据映射是否正确
  8. 超大字段：某单元格内容超过数据库字段长度（如VARCHAR(255)但导入了500字符），验证是截断还是报错

**场景5：时间日期处理**

- **推测错误点**：日期时间处理BUG无处不在——时区、闰年、跨月、跨年、夏令时等
- **测试用例**：
  1. 闰年2月29日：选择2024-02-29（闰年），验证日期合法性；选择2025-02-29（非闰年），验证是否拒绝
  2. 跨时区：用户在UTC+8创建的数据，在UTC+0时区查看，创建时间显示是否正确（相差8小时）
  3. 夏令时切换：在美国时区选择夏令时切换日（2025-03-09 02:30:00——这个时刻不存在！），验证系统行为
  4. 2038年问题：在32位系统中设置日期为2038-01-20（超过int32范围2038-01-19），验证时间戳溢出
  5. 12月31日→1月1日跨年：在12月31日23:59:59秒下单，验证年份切换后数据统计、订单号生成是否正确
  6. 月末特殊日期：设置日期为每月31日，但在2月/4月/6月/9月/11月，验证自动调整为当月最后一天还是报错
  7. 服务器时间和数据库时间不一致：系统时间不同步导致的时间处理异常

## 5. 判定表法

### 通俗释义

多个条件互相影响结果时，罗列所有条件组合，合并重复规则生成用例；分只有是/否、多取值两种表格格式。

### 详细解释

判定表法（Decision Table Testing）是一种系统化处理多条件组合的测试设计技术。当输入条件之间存在逻辑关系、不同条件组合产生不同结果时，判定表可以清晰完整地列出所有条件组合及其对应动作，避免遗漏。

**判定表的两种形式：**

| 形式 | 说明 | 适用场景 |
|------|------|----------|
| **有限条目判定表** | 所有条件取值只有"是/否"（True/False）两种状态 | 条件为布尔类型或可二值化 |
| **扩展条目判定表** | 条件可以有多个取值（如高/中/低、大/中/小） | 条件非二值，有多种离散值 |

**判定表的组成结构：**

```
┌─────────────────────────────────────┐
│        条件桩（Condition Stub）       │  ← 列出所有条件
├─────────────────────────────────────┤
│        条件项（Condition Entry）      │  ← 每个条件的取值组合
├─────────────────────────────────────┤
│        动作桩（Action Stub）          │  ← 列出所有可能的动作
├─────────────────────────────────────┤
│        动作项（Action Entry）         │  ← 对应每种组合的动作结果
└─────────────────────────────────────┘
```

**设计步骤：**
1. 分析需求，识别所有条件（输入条件）和动作（输出结果）
2. 列出条件桩和动作桩
3. 计算规则数 = 条件1取值个数 × 条件2取值个数 × …（如有4个二值条件，规则数=2⁴=16）
4. 填写条件项（所有组合）
5. 填写动作项（每种组合对应的结果）
6. 合并简化：如果两条规则的条件取值只有一项不同且产生的动作完全相同，则该项可以合并（标记为"—"无关）

### 实际使用场景（5个）

**场景1：用户登录验证 - 有限条目判定表**

- **条件**（3个，二值）：
  - C1: 用户名是否正确？（是/否）
  - C2: 密码是否正确？（是/否）
  - C3: 验证码是否正确？（是/否）
- **动作**：
  - A1: 登录成功，跳转首页
  - A2: 提示"用户名不存在"
  - A3: 提示"密码错误"
  - A4: 提示"验证码错误"
  - A5: 账号锁定（密码错误超次数）

| 规则编号 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|----------|---|---|---|---|---|---|---|---|
| **C1:用户名正确** | 是 | 是 | 是 | 是 | 否 | 否 | 否 | 否 |
| **C2:密码正确** | 是 | 是 | 否 | 否 | — | — | — | — |
| **C3:验证码正确** | 是 | 否 | 是 | 否 | — | — | — | — |
| **A1:登录成功** | ✓ |   |   |   |   |   |   |   |
| **A2:用户名不存在** |   |   |   |   | ✓ | ✓ | ✓ | ✓ |
| **A3:密码错误** |   |   | ✓ | ✓ |   |   |   |   |
| **A4:验证码错误** |   | ✓ |   | ✓ |   |   |   |   |

> 合并简化：规则5-8中，用户名不正确时，密码和验证码结果无关紧要（标记为"—"），可合并为一条规则

**场景2：车险保费计算 - 扩展条目判定表**

- **条件**（3个，多值）：
  - C1: 出险记录（0次/1-2次/3次及以上）
  - C2: 车龄（1年以内/1-3年/3年以上）
  - C3: 驾龄（1年以内/1-5年/5年以上）
- **动作**：

  - C1: 会员等级（普通/银卡/金卡/钻石）

## 6. 因果图法

### 通俗释义

画图梳理输入条件和输出结果的逻辑关系，标注逻辑运算和约束规则，画完图的数据可以整理成判定表。

### 详细解释

因果图法（Cause-Effect Graphing）通过图形化的方式直观表达输入条件（因）和输出结果（果）之间的逻辑关系。它特别适用于输入条件之间存在复杂逻辑关系（与、或、非及各类约束）的场景。因果图是判定表的前置分析工具——画完因果图后，将其转换为判定表来设计测试用例。

**因果图的核心元素：**

| 元素 | 符号 | 说明 |
|------|------|------|
| **恒等** | `=` | 因出现，果必出现；因不出现，果必不出现 |
| **非（NOT）** | `~` | 因出现，果不出现；因不出现，果才出现 |
| **或（OR）** | `∨` | 多个因中至少一个出现，果出现 |
| **与（AND）** | `∧` | 多个因全部出现，果才出现 |

**五类约束关系（因与因之间）：**

| 约束 | 符号 | 说明 | 示例 |
|------|------|------|------|
| **E（互斥/异）** | Exclude | 多个因中最多只有一个为真 | 支付方式：不能同时选"微信支付"和"支付宝" |
| **I（包含）** | Include | 多个因中至少有一个为真 | 联系方式：至少填手机号或邮箱之一 |
| **O（唯一）** | Only One | 多个因中有且仅有一个为真 | 性别：男或女，必选其一 |
| **R（要求）** | Require | 因A为真时，因B必须也为真 | 使用优惠券 → 必须登录 |
| **M（屏蔽/强制）** | Mask | 因A为真时，因B必须为假 | 选择"不开发票"→ 发票类型选项必须不可用 |

**设计步骤：**
1. 分析需求规格，识别所有的"因"（输入条件）和"果"（输出结果/动作）
2. 分析因和果之间的逻辑关系（与/或/非）
3. 分析因与因之间的约束关系（E/I/O/R/M）
4. 画出因果图：左边画因、右边画果、中间用逻辑符号连接、因之间标注约束
5. 将因果图转换为判定表（每个因作为判定表的条件，每个果作为判定表的动作）
6. 根据判定表设计测试用例

## 7. 正交试验法

### 通俗释义

参数很多、全组合测不完时，借助正交表挑选少量代表性用例，用最少用例覆盖绝大多数搭配，极端数据再单独补测。

### 详细解释

正交试验法（Orthogonal Array Testing）利用数学中的正交表来高效设计多因素、多水平的测试用例。当被测系统有多个参数且每个参数有多个取值时，全组合（笛卡尔积）的用例数会爆炸性增长（如5个参数各3个值 = 3⁵ = 243个用例）。正交试验法通过"均匀分散、整齐可比"的原则，从全组合中挑选少量具有代表性的用例，在保持充分覆盖的同时大幅减少用例数量。

**核心概念：**

| 概念 | 说明 |
|------|------|
| **因素（Factor）** | 测试中需要变化的参数/变量 |
| **水平（Level）** | 每个因素可能的取值个数 |
| **正交表** | 标准化的因素-水平组合表，记作 Lₙ(m^k)，n=试验次数，m=水平数，k=因素数 |
| **均匀分散** | 各因素的各水平在试验中出现的次数相同 |
| **整齐可比** | 任意两个因素的所有水平组合在试验中出现的次数相同 |

**常用正交表举例：**

```
L₄(2³)：3个2水平因素，只需4次试验（全组合需8次）
L₉(3⁴)：4个3水平因素，只需9次试验（全组合需81次）
L₁₆(4⁵)：5个4水平因素，只需16次试验（全组合需1024次）
L₁₈(3⁷)：7个3水平因素，只需18次试验（全组合需2187次）
```

**设计步骤：**
1. 确定测试中的因素（参数）和各因素的水平（取值）
2. 选择合适的正交表（因素数≥实际因素数，水平数≥最大水平数）
3. 将各因素的各水平映射到正交表中
4. 对于水平数不一致的情况，用"拟水平法"补足（将重要水平复用）
5. 将正交表每一行转换为一个测试用例
6. 对正交表中未覆盖但风险较高的极端组合，额外补充用例

### 实际使用场景（5个）

## 8. Pairwise/All-Pairs 配对测试法

### 通俗释义

全排列太多、正交表不会选时，用Pairwise工具自动生成最少用例集——保证任意两个参数的取值组合都被覆盖到。核心逻辑是：大多数缺陷由单个参数或两个参数的组合触发，覆盖所有两两组合即可发现大部分Bug。

### 详细解释

Pairwise测试（也称All-Pairs Testing，全配对测试）是一种组合测试技术。其核心假设是：软件中绝大多数缺陷是单个参数或两个参数交互导致的（而非三个及以上参数的复杂交互）。因此，无需覆盖所有N个参数的全部组合（笛卡尔积），只需保证每两个参数的所有取值组合在用例集中至少出现一次。

**Pairwise与正交表的关系与区别：**

| 对比维度 | 正交试验法 | Pairwise/All-Pairs |
|----------|-----------|-------------------|
| **理论基础** | 正交拉丁方 / 数学正交表 | 贪心算法 / 图论覆盖 |
| **选表方式** | 查标准正交表（L₄(2³), L₉(3⁴)等） | 自动生成（工具自动计算） |
| **参数水平要求** | 所有参数水平数最好一致（不一致需拟水平法） | 参数水平数无需一致 |
| **覆盖程度** | 任意两因素的所有水平组合出现次数相同（整齐可比） | 保证每个两两组合至少出现一次 |
| **用例数** | 固定（由正交表决定） | 自动优化生成，通常等于或略多于正交表 |
| **灵活性** | 受限于可用正交表的规格 | 极高——任意数量参数、任意水平数都可 |

> **关键区别**：正交表要求"整齐可比"（任意两个因素间的所有水平组合出现次数相同），而Pairwise只要求"覆盖"（每个两两组合至少出现一次）。Pairwise在实践中更灵活，且现代工具有力地支持了其使用。

**设计步骤：**
1. 列出所有参数及每个参数的可能取值
2. 确定必含的约束条件（如某些参数组合不可能出现，如"支付方式=货到付款"与"地区=境外"不可共存）
3. 使用Pairwise工具（如PICT、AllPairs、Jenny等）自动生成测试用例集
4. 验证生成的用例集是否满足两两覆盖
5. 对工具无法处理的约束，手动补充或调整用例
6. 基于风险补充高维组合（如已知某3个参数交互会导致Bug，额外补充该3参数的全组合）

### 实际使用场景（3个）

**场景1：浏览器配置兼容性（8参数，344种两两组合→仅需11个用例）**

- **参数列表**：
  - 浏览器: Chrome / Firefox / Edge / Safari（4水平）
  - 操作系统: Windows / macOS / Linux（3水平）
  - 屏幕分辨率: 1920×1080 / 1366×768 / 2560×1440 / 3840×2160（4水平）
  - JavaScript版本: ES2019 / ES2020 / ES2021（3水平）
  - 语言: 中文 / English（2水平）

- **Pairwise后**：约14-18个用例
- **关键约束**：GET方法不允许有Body（传了Content-Type=JSON也应被忽略）

> **工程建议**：Pairwise/All-Pairs不是全组合的完全替代品。对于已知的三参数及以上交互风险（如"支付金额+会员等级+优惠券类型"），应在Pairwise基础上额外补充高维组合用例。实际项目中，Pairwise通常与等价类划分法组合使用：先用等价类确定每个参数的典型取值，再用Pairwise生成组合用例。

## 9. 状态迁移法

### 通俗释义

梳理单据、订单所有状态，测试正常切换状态、非法强制跳转能否被系统拦截。

### 详细解释

状态迁移法（State Transition Testing）用于测试系统在不同状态之间的转换行为。许多业务对象（如订单、工单、审批单、账户等）都有明确的生命周期，在生命周期中状态按照特定规则进行转换。状态迁移法通过状态图来可视化所有可能的状态转换路径，并针对合法和非法转换设计测试用例。

**状态迁移的核心概念：**

| 概念 | 说明 |
|------|------|
| **状态（State）** | 系统/对象在某一时刻所处的情况（如：待支付、已支付、已发货） |
| **迁移（Transition）** | 从一个状态变为另一个状态的动作/事件 |
| **事件/触发（Event/Trigger）** | 引发状态迁移的行为（如：用户支付、商家发货、超时） |
| **守卫条件（Guard）** | 迁移发生前必须满足的条件（如：余额充足、库存足够） |
| **动作（Action）** | 迁移发生时的附带操作（如：发送通知、扣减库存） |

**状态迁移测试的两个维度：**

| 维度 | 测试内容 |
|------|----------|
| **合法正向流转** | 验证所有符合业务规则的状态转换能否正确执行，包括触发条件、守卫条件、附带动作 |
| **非法跳转拦截** | 验证系统能否正确拦截不允许的状态转换（如待支付→已完成，跳过已支付和已发货），包括前端绕过、接口直调、数据库直接修改等攻击方式 |

**设计步骤：**
1. 识别被测对象的所有可能状态
2. 确定初始状态和终态（可能多个）
3. 绘制状态图（UML状态图或状态迁移矩阵）
4. 列出所有合法的状态迁移路径（包括迁移的触发事件和守卫条件）
5. 列出所有非法的状态迁移路径
6. 为每条合法路径设计正向测试用例
7. 为每条非法路径设计负向测试用例（验证拦截）
8. 额外验证：并发迁移、超时自动迁移、数据持久化后状态一致性

### 实际使用场景（5个）

**场景1：电商订单状态全生命周期**

- **状态列表**：待支付 → 已支付 → (已取消) → (已退款) → 待发货 → 已发货 → 运输中 → 已签收 → 已完成 / (退货中) → (已退款) / 交易关闭

- **合法正向流转测试**：
  1. 待支付 → 已支付：用户成功付款，订单状态变为"已支付"，库存扣减
  2. 待支付 → 已取消：用户主动取消，或30分钟超时未支付自动取消，库存回滚
  3. 已支付 → 待发货：支付回调成功，进入发货准备状态
  4. 待发货 → 已发货：商家填写物流单号，状态更新，触发发货通知
  5. 已发货 → 运输中：物流信息同步，各物流节点更新
  6. 已发货 → 已签收：物流显示签收，或用户手动确认收货
  7. 已签收 → 已完成：签收后7天无售后自动完成，或用户手动确认

- **非法跳转拦截测试**：
  1. 待支付 → 已发货：绕过支付环节直接发货（应被拦截）
  2. 待支付 → 已完成：跳过所有中间状态（应被拦截）
  3. 已完成 → 待支付：已完成订单不应回到待支付（应被拦截）
  4. 已退款 → 已发货：退款后不应再发货（应被拦截）
  5. 通过API直接修改状态：构造非法请求修改订单状态字段（应有权限+状态机校验）

- **状态迁移矩阵**（合法√ 非法×）：

| 从\到 | 待支付 | 已支付 | 待发货 | 已发货 | 已签收 | 已完成 | 已取消 | 已退款 |
|-------|--------|--------|--------|--------|--------|--------|--------|--------|
| **待支付** | — | √ | × | × | × | × | √ | × |
| **已支付** | × | — | √ | × | × | × | √ | √ |
| **待发货** | × | × | — | √ | × | × | × | × |
| **已发货** | × | × | × | — | √ | × | × | × |
| **已签收** | × | × | × | × | — | √ | √ | × |
| **已完成** | × | × | × | × | × | — | × | × |

**场景2：请假审批流程状态**

- **状态列表**：草稿 → 已提交 → 部门审批中 → (已驳回) → HR审批中 → 已通过 → (已撤回) → 已生效 → 已归档 / 已销假

- **合法正向流转测试**：
  1. 草稿 → 已提交：填写完请假信息后提交
  2. 已提交 → 部门审批中：自动分配给直属领导
  3. 部门审批中 → HR审批中：领导审批通过后流转至HR
  4. HR审批中 → 已通过：HR审批通过
  5. 已通过 → 已生效：请假开始时间到达，自动生效
  6. 已生效 → 已归档：请假结束后自动归档

- **分支与回退流转测试**：
  1. 部门审批中 → 已驳回：领导驳回，需填写驳回原因
  2. 已驳回 → 草稿（可修改后重新提交）：员工修改后重新提交
  3. 已提交/部门审批中/HR审批中 → 已撤回：员工主动撤回申请

- **非法跳转拦截测试**：
  1. 草稿 → 已生效：未提交审批直接生效（应拦截）
  2. 已驳回 → 已通过：被驳回后跳过重新审批直接通过（应拦截）
  3. 已归档 → 草稿：已归档的申请不应复活（应拦截）
  4. HR审批中 → 已撤回但跳过部门审批（应拦截—HR审批中撤回应先撤回HR审批再撤回部门审批）

**场景3：会员卡账户状态**

- **状态列表**：未激活 → 正常 → (冻结) → 已注销 / (挂失) → 已补卡

- **合法流转测试**：
  1. 未激活 → 正常：首次消费/充值后激活
  2. 正常 → 冻结：账户异常（如多次密码错误、可疑交易）或主动冻结
  3. 冻结 → 正常：客服解冻、身份验证通过后解冻
  4. 正常 → 挂失：用户主动挂失
  5. 挂失 → 已补卡：补办新卡后旧卡注销
  6. 正常 → 已注销：用户主动注销账户
  7. 冻结 → 已注销：冻结状态下注销（余额需处理）

- **非法跳转拦截测试**：
  1. 未激活 → 已注销：绕过正常使用直接注销（视业务规则可能允许）
  2. 已注销 → 正常：已注销账户复活（应拦截，除非有恢复机制）
  3. 挂失 → 正常：跳过补卡直接恢复（应拦截—需补办新卡）
  4. 已注销状态的卡被POS机刷卡消费（应返回"卡片已注销"）

- **自动化状态处理验证**：
  - 会员卡有效期到期 → 自动从未激活/正常 → 已过期（自动迁移）
  - 长期不活跃（如3年无消费）→ 自动从正常 → 冻结（静默冻结，保护性处理）

**场景4：运维工单/事件单状态**

- **状态列表**：新建 → 已分派 → 处理中 → (待补充信息) → (待审批变更) → 已解决 → (已驳回验收) → 已关闭 / (重新打开)

- **合法流转测试**：
  1. 新建 → 已分派：客服分派给对应工程师
  2. 已分派 → 处理中：工程师接受工单，开始处理
  3. 处理中 → 待补充信息：工程师需要客户提供更多信息
  4. 待补充信息 → 处理中：客户补充信息后继续处理
  5. 处理中 → 待审批变更：需要审批的变更操作
  6. 待审批变更 → 处理中：审批通过，继续执行
  7. 处理中 → 已解决：问题解决，等待确认
  8. 已解决 → 已关闭：客户确认/超时自动关闭
  9. 已解决 → 重新打开：客户不满意，重新打开工单
  10. 已关闭 → 重新打开：已关闭工单因相关原因重新打开

- **非法跳转拦截测试**：
  1. 新建 → 已解决：跳过处理直接解决（应拦截）
  2. 新建 → 已关闭：跳过所有环节直接关闭（应拦截）
  3. 已关闭 → 新建：重置工单状态（应拦截，用"重新打开"替代）
  4. 处理中 → 新建：回退到初始状态（应拦截）

**场景5：Git代码合并请求（Merge Request / Pull Request）状态**

- **状态列表**：Draft（草稿）→ Open（待评审）→ In Review（评审中）→ Changes Requested（需修改）→ Approved（已批准）→ Merged（已合并）/ Closed（已关闭）→ Reverted（已回滚）

- **合法流转测试**：
  1. Draft → Open：开发者完成代码，点击"Ready for Review"
  2. Open → In Review：评审者开始评审代码
  3. In Review → Changes Requested：评审者发现问题，要求修改
  4. Changes Requested → In Review：开发者修改后重新请求评审
  5. In Review → Approved：所有评审者批准
  6. Approved → Merged：代码合并到目标分支
  7. Open/In Review/Approved → Closed：主动关闭PR（不合并）
  8. Merged → Reverted：发现严重问题，回滚合并

- **非法跳转拦截测试**：
  1. Draft → Merged：跳过评审直接合并（应有分支保护规则拦截）
  2. Changes Requested → Merged：评审未通过但强制合并（需管理员权限）
  3. Merged → Open：已合并PR不应重新打开（应拦截）
  4. Closed → Merged：已关闭PR重新合并（应重新打开后再合并）
  5. 绕过UI直接调用API修改MR状态（后端应做状态机校验）

## 10. 功能图法

### 通俗释义

结合场景流程 + 状态变化两种思路，既测业务步骤又测状态变更，多用于复杂复合型业务。

### 详细解释

功能图法（Function Map / Function Diagram）是一种综合性的测试设计方法，它融合了场景分析法（关注业务流程）和状态迁移法（关注状态转换）的特点。功能图法通过绘制功能流程图，同时描述系统功能的执行逻辑和数据/对象的状态变化，适用于业务逻辑复杂、功能和状态交织的系统。

> **方法来源说明**：功能图法并非ISTQB/ISO 29119标准中定义的标准黑盒测试设计方法。在ISTQB分类中，标准的黑盒/基于规格的测试设计技术为：等价类划分、边界值分析、判定表测试、状态转换测试、用例测试。功能图法主要见于国内部分测试教材和实践中，其实质是"场景分析法+状态迁移法"的复合运用。如果要在标准框架下归类，建议将对应场景拆分为场景分析法（基于业务流程）和状态迁移法（基于状态转换）分别应用。

**功能图的核心组成：**

| 组成部分 | 说明 | 来源 |
|----------|------|------|
| **功能节点** | 表示一个功能操作或处理步骤 | 场景分析法 |
| **状态节点** | 表示系统/对象在某个时点的状态 | 状态迁移法 |
| **流转路径** | 连接功能节点和状态节点的有向边 | 两者的结合 |
| **条件分支** | 根据不同条件走向不同路径 | 场景分析法 |
| **数据节点** | 标注关键数据的输入/输出/存储 | 扩展元素 |

**功能图的类型：**
1. **功能流程图（本地）**：描述单个模块内部的功能流转和状态变化
2. **功能层次图（全局）**：描述跨模块/跨系统的业务编排和数据流转
3. **功能-状态混合图**：同时标注功能步骤和对应的状态变化

**设计步骤：**
1. 分析需求，识别被测业务的所有功能节点
2. 识别各功能节点的输入数据、输出数据、处理逻辑
3. 识别业务涉及的对象及其状态变化点
4. 绘制功能-状态混合图：纵向为功能步骤，横向为状态变化
5. 识别图中的所有路径（从起始到各终点）
6. 对每条路径设计测试用例（包含操作步骤、数据、预期状态）
7. 重点验证功能节点之间的数据传递准确性和状态一致性

### 实际使用场景（5个）

**场景1：在线教育平台 - 课程学习全流程**

- **功能节点**：
  - F1: 浏览课程目录 → F2: 课程详情 → F3: 购买/报名 → F4: 开始学习 → F5: 观看视频 → F6: 完成课后作业 → F7: 参加阶段测验 → F8: 期末考试 → F9: 获得证书

- **状态节点**（学员课程状态）：
  - S1: 未报名 → S2: 已报名 → S3: 学习中 → S4: 已完成课时 → S5: 已通过测验 → S6: 已通过考试 → S7: 已获得证书

- **功能-状态混合路径测试**：
  1. **主路径**：F1→F2→F3→F4→F5→F6→F7→F8→F9，状态 S1→S2→S3→S4→S5→S6→S7
     - 验证点：每步完成后学习进度%更新，状态正确切换
  2. **跳课路径**：F4→直接播放第5课视频（跳过前4课），验证是否允许（视课程设置："学完解锁"还是"自由选择"）
  3. **未完成作业路径**：F5完成后跳过F6直接进入F7，验证是否拦截
  4. **重复考试路径**：F8不通过 → 重新学习 → 再次F8，验证重考次数限制和状态回退S6→S5
  5. **数据结构验证**：
     - F5观看视频50%退出 → 下次进入从50%继续（断点续播）
     - F6提交作业后，教师评分，成绩反写回学员记录
     - F9生成证书 → 证书编号、姓名、课程名、日期落库并防篡改

**场景2：仓库管理系统 - 出入库全流程**

- **功能节点**：
  - F1: 创建采购单 → F2: 供应商发货 → F3: 到货登记 → F4: 质检 → F5: 上架 → F6: 分配储位 → F7: 订单波次 → F8: 拣货 → F9: 复核 → F10: 打包 → F11: 出库 → F12: 交接物流

- **状态节点**（商品库存状态）：
  - S1: 在途 → S2: 待质检 → S3: 合格品（可售）→ S4: 不合格品（待退）→ S5: 已锁定（订单占用）→ S6: 已出库 → S7: 退货回库

- **复合路径测试**：
  1. **正向入出库**：F1→F2→F3→F4→F5→F6→F7→F8→F9→F10→F11→F12；状态 S1→S2→S3→S5→S6
  2. **质检不合格路径**：F3→F4(不合格)→退货处理，状态 S2→S4→（退货出库）
  3. **取消订单释放库存**：F7锁定库存S5→用户取消订单→库存释放回S3，验证库存数量正确回加
  4. **退货回库**：已出库S6→客户退货→S7退货回库→质检→S3或S4
  5. **并发验证**：
     - F7波次时100件商品同时被两个订单锁定，验证库存锁定不超卖
     - F8拣货和F9复核之间，另一订单取消释放库存，验证状态一致性

**场景3：招聘管理系统 - 候选人全流程**

- **功能节点**：
  - F1: 发布职位 → F2: 接收简历 → F3: HR初筛 → F4: 电话沟通 → F5: 笔试 → F6: 技术面试 → F7: HR面试 → F8: 背景调查 → F9: 发放Offer → F10: Offer确认 → F11: 入职办理

- **状态节点**（候选人状态）：
  - S1: 简历投递 → S2: 初筛通过 → S3: 面试中 → S4: 面试通过 → S5: Offer待确认 → S6: 已入职 → S7: 淘汰 → S8: 候选人放弃 → S9: 进入人才库

- **关键路径测试**：
  1. **完整录用路径**：F1→F2→F3→F4→F5→F6→F7→F8→F9→F10→F11；S1→S2→S3→S4→S5→S6
  2. **中途淘汰路径**：F6技术面试不通过 → S7淘汰 → 发送拒信 → 3个月冷却期 → 可重新投递
  3. **候选人放弃路径**：F9发放Offer → F10候选人拒接 → S8放弃 → 记录放弃原因

  5. **复合验证**：
     - 每个功能节点的"操作人"权限：F3只能HR操作，F6只能面试官操作
     - Offer金额变更 → 审批流触发 → 审批通过/驳回 → Offer状态更新

**场景4：保险理赔系统**

- **功能节点**：
  - F1: 报案登记 → F2: 资料上传 → F3: 初审 → F4: 现场勘察 → F5: 定损 → F6: 核赔 → F7: 理算 → F8: 审批 → F9: 支付赔款 → F10: 结案

- **状态节点**（理赔案件状态）：
  - S1: 已报案 → S2: 资料待补充 → S3: 审核中 → S4: 勘察中 → S5: 定损完成 → S6: 核赔通过 → S7: 待支付 → S8: 已支付 → S9: 已结案 → S10: 拒赔 → S11: 撤销

- **数据+状态复合路径测试**：
  1. **正常理赔路径**：F1→F2→F3→F4→F5→F6→F7→F8→F9→F10；S1→S3→S4→S5→S6→S7→S8→S9
  2. **资料补充路径**：F3初审发现资料不全 → S2资料待补充 → 客户补传 → F3重新初审 → 进入正常流程
     - 验证点：补传后案件不应重新编号，补传次数限制，超时未补传自动撤销
  3. **拒赔路径**：F6核赔不通过 → S10拒赔 → 发送拒赔通知书 → 客户可申诉 → 重新进入F6核赔
  4. **撤回路径**：F3初审中客户来电撤销 → S11撤销 → 案件归档但保留记录
  5. **数据一致性验证**：
     - F5定损金额 → F7理算金额（扣除免赔额后）→ F9实际支付金额，三者关系正确
     - 赔款支付后，保单的"已赔付金额"字段更新
     - F4勘察上传的现场照片绑定到案件编号，结案后照片存储不丢失

**场景5：内容管理系统 - 文章发布全流程**

- **功能节点**：
  - F1: 创建文章 → F2: 编辑内容 → F3: 保存草稿 → F4: 提交审核 → F5: 编辑审核 → F6: 主编终审 → F7: 定时发布 → F8: 即时发布 → F9: 发布后修改 → F10: 下架/删除

- **状态节点**（文章状态）：
  - S1: 草稿 → S2: 待编辑审核 → S3: 待主编终审 → S4: 已驳回 → S5: 审核通过待发布 → S6: 已定时发布 → S7: 已发布 → S8: 已下架 → S9: 已删除

- **复合路径测试**：
  1. **完整发布路径**：F1→F2→F3→F4→F5→F6→F8；S1→S2→S3→S5→S7
     - 验证点：每一步操作人、操作时间戳记录；发布后URL可访问；搜索引擎索引更新
  2. **驳回修改路径**：F4→F5(拒绝)→S4驳回(附修改意见)→F2修改→F3→F4→F5(通过)→F6→F8；S4→S1→S2→S3→S5→S7
     - 验证点：修改版本记录（不是覆盖，是新增版本），驳回原因完整保留
  3. **定时发布路径**：F6通过→F7设置定时发布时间（如明天8:00）；S5→S6定时等待→S7已发布
     - 验证点：定时任务准时触发、发布前支持取消定时、定时发布失败时告警
  4. **发布后修改路径**：S7已发布→F9发布后修改→需重新走审核流F4→F5→F6(视配置)；修改期间S7状态保持不变（线上还是旧版本），审核通过后新版本覆盖
  5. **多环境一致性验证**：
     - 预发布环境：审核通过后先发布到预发布环境验证，确认无误后发布到生产环境
     - CDN缓存刷新：发布新内容/修改后，CDN缓存是否在合理时间内刷新

## 11. 随机测试法

### 通俗释义

随机胡乱输入各类数据，多用于自动化跑稳定性，项目收尾查漏。

### 详细解释

随机测试法（Random Testing / Stochastic Testing）是指不按照预设的测试脚本或测试用例，通过随机生成输入数据、随机选择操作路径来测试系统的方法。虽然看似"胡乱测试"，但随机测试在特定场景下具有独特的价值——它能发现系统化测试方法覆盖不到的边界情况。

**随机测试的主要形式：**

| 形式 | 说明 | 典型应用 |
|------|------|----------|
| **Monkey Testing** | 模拟随机点击、滑动、输入等用户操作，长时间运行 | 移动APP稳定性测试 |
| **Fuzz Testing（模糊测试）** | 生成随机/半随机的畸形数据作为输入 | 安全测试、协议测试、API测试 |
| **随机数据驱动** | 在自动化框架中随机生成参数组合 | 回归测试、兼容性测试 |
| **探索性随机测试** | 测试人员凭直觉随机探索系统 | 项目收尾查漏补缺 |
| **混沌工程** | 在生产/类生产环境中随机注入故障 | 系统韧性/弹性验证 |

**随机测试的价值与限制：**

| 价值 | 限制 |
|------|------|
| 能发现意料之外的缺陷（unknowable unknowns） | 无法保证覆盖率，不能替代系统化测试 |
| 适合长时间跑稳定性、内存泄漏检测 | 用例不可复现（需要记录随机种子） |
| 低成本发现崩溃、挂起类严重缺陷 | 需要配套监控和日志记录机制 |
| 可自动化、可持续集成 | 随机数据可能触发不到深层业务逻辑bug |

**设计步骤：**
1. 明确随机测试的目标（稳定性/安全性/查漏补缺）
2. 确定随机范围（哪些模块、哪些输入域、哪些操作类型）
3. 设定随机参数：操作频率、测试时长、随机种子（用于复现）
4. 配置监控：崩溃日志、异常监控、资源监控（CPU/内存/磁盘）
5. 执行随机测试并记录所有异常
6. 对每个异常进行复现尝试（利用随机种子）→ 记录/提单
7. 将随机测试发现的高频缺陷模式反馈到其他系统化测试方法中

### 实际使用场景（5个）

**场景1：移动APP Monkey稳定性测试**

- **测试目标**：检测APP在随机操作下是否出现崩溃（Crash）、无响应（ANR）、内存泄漏
- **测试配置**：
  - 工具：Android Monkey / iOS UI AutoMonkey
  - 操作数：10000次随机事件
  - 事件比例：点击70% / 滑动15% / 按键10% / 系统事件5%
  - 限制包名：仅操作被测APP，不跳出
  - 运行时长：约2小时
- **重点关注**：
  - Crash日志捕获（native crash / java crash）
  - ANR日志（主线程阻塞超过5秒）
  - 内存使用趋势（是否存在持续增长不释放的内存泄漏）
  - 线程数变化（是否存在线程泄漏）
- **测试覆盖范围**：
  - 各页面随机跳转：首页→搜索→详情→购物车→结算→支付→我的→设置→...
  - 随机表单输入：搜索框输入随机字符串、评论框输入emoji、数量输入负值
  - 随机系统中断：来电、通知栏下拉、横竖屏切换、多任务切换、低电量弹窗
  - 随机网络状态：WiFi/4G/飞行模式/弱网切换
- **缺陷观察**：连续跑24小时后，内存从200MB涨到800MB → 内存泄漏；随机点击100次后闪退 → NullPointer异常

**场景2：API接口Fuzz测试**

- **测试目标**：检测API在畸形输入下是否出现500错误、信息泄露、拒绝服务
- **测试配置**：
  - 工具：自定义脚本 + REST Fuzzer
  - 被测接口：所有对外暴露的REST API接口
  - Fuzz策略：
    - 参数类型变异：期望int传入string、bool、null、空数组、超长string
    - 参数缺失：逐个删除必填参数
    - 参数重复：同一个参数传多次（如 `?id=1&id=2`）
    - 参数注入：传入SQL注入/命令注入/XSS payload
    - 边界Fuzz：传入 Integer.MAX_VALUE (2147483647)、Long.MIN_VALUE、NaN、Infinity
    - JSON结构变异：嵌套过深(>100层)、Key重复、不合法JSON
- **重点关注**：
  - 响应码：不应出现500 Internal Server Error（表示服务端未处理异常）
  - 错误信息：不应暴露堆栈跟踪、数据库结构、内部IP等敏感信息
  - 响应时间：输入不应导致服务端死循环或长时间阻塞
  - 数据一致性：畸形参数不应导致数据写入错误（如截断写入、乱码写入）
- **缺陷示例**：
  - `GET /api/user?id=-1` → 返回500 + 堆栈信息（暴露代码路径）
  - `POST /api/order` 传入超长商品名(100KB) → 数据库报错字段过长 → 返回500
  - `GET /api/search?q=%00` → 服务端C代码处理NULL字节崩溃

**场景3：搜索功能随机输入测试**

- **测试目标**：验证搜索功能对各类随机输入的容错性和稳定性
- **随机输入类型**：
  - 纯随机字符串：`aSDf23$@#!`、`ಠ_ಠ`（Unicode表情符）
  - 超长输入：10000个随机字符
  - 空字符：零宽空格(U+200B)、NULL字符(U+0000)、换行符\n、回车\r
  - SQL关键字随机组合：`SELECT FROM WHERE UNION DROP`
  - 编程语言关键字：`null undefined NaN Infinity true false`
  - URL编码字符：`%20%00%FF%FE`
  - 多种编码混合：中文+日文+韩文+阿拉伯文+emoji
  - 特殊正则符号：`.*+?^${}()|[]\`
- **验证点**：
  - 搜索不崩溃、不超时
  - 返回结果不包含代码注入的脚本执行
  - 高亮显示不因特殊字符而布局错乱
  - 搜索建议/自动补全不因随机输入而异常
  - 无结果时展示友好提示而非报错
- **缺陷示例**：
  - 搜索 `[]` → 后端正则编译异常，返回500
  - 搜索 emoji `🤣🤣🤣🤣` → 数据库字符集不支持，存储/搜索乱码
  - 搜索 100KB 超长字符串 → 搜索引擎OOM

**场景4：表单多字段随机数据填充**

- **测试目标**：验证复杂表单对各类随机输入的校验能力和容错性
- **随机数据生成规则**（基于字段类型）：
  - 文本字段：随机长度(0-10000)、随机字符集（ASCII/中文/emoji/特殊符号/全角）
  - 数字字段：0、负数、小数、极大值、科学计数法、非数字
  - 日期字段：0000-00-00、9999-12-31、2025-02-29、空字符串
  - 邮箱字段：无@符号、双@@、超长域名、中文域名、无.com
  - 手机号字段：随机长度数字、含字母、特殊分隔符、国际区号
  - 下拉框字段：不在选项中的值、空值、多个值、数字型值
  - 文件上传字段：随机大小(0-1GB)、随机格式、伪造MIME类型
  - 富文本字段：超长HTML、无限嵌套标签、自闭合标签、iframe嵌入
- **验证点**：
  - 前端校验：非法输入是否在前端有合理提示
  - 后端校验：绕过前端提交的数据是否在后端二次校验
  - 数据存储：合法数据是否正确写入数据库，非法数据是否被拒绝
  - 错误恢复：提交失败后表单数据是否保留（避免用户重新填写）
- **缺陷示例**：
  - 姓名字段填写1000个中文字符 → 前端未校验 → 后端VARCHAR(50)静默截断 → 存了被截断的姓名
  - 金额字段填写 `1e10` → Java BigDecimal解析异常 → 订单创建失败无提示

**场景5：多用户并发随机操作（混沌工程）**

- **测试目标**：模拟生产环境中的随机并发操作，检测系统在不可预测的多用户行为下的稳定性和数据一致性
- **测试配置**：
  - 虚拟用户数：100-500个并发用户
  - 操作类型（电商场景为例）：
    - 浏览商品（随机类目、随机翻页）
    - 加入购物车（随机商品、随机数量）
    - 下单（随机支付方式、随机地址）
    - 取消订单（随机选择待支付订单）
    - 退款申请（随机选择已完成订单）
  - 混沌注入：
    - 随机Kill一个服务实例
    - 随机网络延迟注入（增加200ms-5s延迟）
    - 随机数据库主从切换
    - 随机消息队列消息积压
- **验证点**：
  - 有无超卖（库存负数）
  - 有无订单丢失（下单成功但数据库无记录）
  - 有无重复扣款
  - 服务恢复后系统自愈能力
  - 所有异常是否正确记录日志
- **缺陷示例**：
  - 用户A和B同时购买最后1件商品 → A支付成功，B也支付成功 → 库存变为-1（超卖）
  - 下单服务实例被Kill的瞬间 → 已扣款但订单未创建 → 资金不一致
  - Redis主从切换瞬间 → 库存缓存脏数据 → 库存显示不准确

## 12. 基于需求测试法

### 通俗释义

逐条对照需求文档提炼测试点，是所有黑盒方法的基础，一般搭配其余方法一起使用。

### 详细解释

基于需求测试法（Requirements-Based Testing）是最基础、最重要的测试方法。它直接以需求规格说明书（PRD/SRS/用户故事）为依据，逐条分析需求中的功能点和非功能要求，提炼测试条件并设计测试用例。基于需求测试法是所有其他测试方法的基础——每个测试用例都要追溯到某个需求。

**基于需求测试的核心原则：**

| 原则 | 说明 |
|------|------|
| **需求可追溯性** | 每个测试用例必须对应一条或几条需求，建立需求→用例的追溯矩阵 |
| **需求完整性** | 确保每条需求至少有一个测试用例覆盖 |
| **需求正确性** | 验证系统行为与需求描述完全一致 |
| **需求明确性** | 需求本身不清晰、有歧义时，需在测试前和产品/需求方澄清 |

**需求测试的覆盖维度：**

| 维度 | 覆盖内容 |
|------|----------|
| **功能需求** | 业务流程、数据输入/输出、用户界面、接口交互 |
| **性能需求** | 响应时间、吞吐量、并发用户数、资源利用率 |
| **安全需求** | 认证、授权、数据加密、审计日志、隐私保护 |
| **可用性需求** | 易用性、可访问性、多语言、错误提示友好性 |
| **兼容性需求** | 浏览器、操作系统、设备、第三方系统版本 |
| **法规需求** | 数据保护法(GDPR/PIPL)、行业标准、审计合规 |
| **隐含需求** | 需求文档未明确但行业默认、用户合理预期的行为 |

**设计步骤：**
1. 获取最新版本的需求文档（PRD/用户故事/原型图/接口文档）
2. 需求评审：识别模糊、矛盾、遗漏的需求，与需求方确认
3. 需求分解：将每个需求拆解为原子级的可测试需求点
4. 测试点提炼：为每个可测试需求点提炼测试条件
5. 用例设计：基于测试条件，结合其他黑盒方法（等价类、边界值、场景法等）设计具体测试用例
6. 建立需求-用例追溯矩阵（RTM, Requirements Traceability Matrix）
7. 需求变更时同步更新测试用例和追溯矩阵
8. 测试执行后统计需求覆盖率（已测需求数/总需求数）

### 实际使用场景（5个）

**场景1：用户注册功能 - 逐条需求验证**

- **需求文档描述**（示例）：
  1. 用户可以使用手机号或邮箱注册
  2. 手机号需为11位中国大陆手机号，需短信验证
  3. 邮箱需为有效格式，需邮箱验证码验证
  4. 密码长度6-20位，必须包含字母和数字
  5. 需勾选"同意用户协议和隐私政策"方可注册
  6. 同一手机号/邮箱24小时内最多获取5次验证码
  7. 同一IP地址1小时内最多注册3个账号
  8. 注册成功后自动跳转到首页并处于登录状态
  9. 注册成功后发送欢迎短信/邮件

- **测试点提炼与用例设计**：

| 需求编号 | 需求点 | 测试用例 |
|----------|--------|----------|
| R1 | 手机号或邮箱注册 | TC1: 使用有效手机号注册 → 成功；TC2: 使用有效邮箱注册 → 成功 |
| R2 | 11位手机号+短信验证 | TC3: 输入10位手机号 → 提示格式错误；TC4: 输入正确手机号 → 收到验证码 |
| R3 | 邮箱格式+邮箱验证 | TC5: 输入无@邮箱 → 提示格式错误；TC6: 输入正确邮箱 → 收到验证码 |
| R4 | 密码6-20位含字母数字 | TC7: 123456(纯数字) → 提示需含字母；TC8: abcdef(纯字母) → 提示需含数字；TC9: abc123(6位混合) → 通过；TC10: a1(2位) → 提示过短 |
| R5 | 勾选协议 | TC11: 不勾选协议点击注册 → 提示需同意协议 |
| R6 | 24h内最多5次验证码 | TC12: 连续获取6次验证码 → 第6次提示"今日获取次数已达上限" |
| R7 | 同IP 1h内最多3个账号 | TC13: 同IP注册第4个账号 → 提示"操作频繁，请稍后再试" |
| R8 | 注册成功跳转+自动登录 | TC14: 注册成功 → 跳转首页 → 顶部显示用户昵称(已登录状态) |
| R9 | 欢迎短信/邮件 | TC15: 手机号注册成功 → 收到欢迎短信；TC16: 邮箱注册成功 → 收到欢迎邮件 |

- **衍生用例（结合其他方法）**：
  - 等价类+边界值：手机号无效等价类（12位/10位/含字母/空）、密码边界（5位/6位/20位/21位）
  - 场景法：注册全流程（选择注册方式→填写信息→获取验证码→输入验证码→设置密码→勾选协议→提交→验证跳转和登录态）

**场景2：报表导出功能 - 多维度需求验证**

- **需求描述**：
  1. 支持将当前查询结果导出为Excel(.xlsx)和PDF格式
  2. 导出的数据必须与页面显示数据完全一致（包括排序、筛选）
  3. Excel支持最多导出100万行数据，PDF最多导出100页
  4. 导出文件名格式：报表名称_导出时间_操作人.yyyyMMddHHmmss
  5. 超过1万行数据需异步导出，通过站内信/邮件通知用户下载
  6. 导出链接有效期24小时
  7. 需要有导出操作审计日志
  8. 单用户每小时最多导出10次

- **测试点与用例设计**：

| 需求编号 | 需求点 | 测试用例 | 所用方法 |
|----------|--------|----------|----------|
| R1 | xlsx/PDF两种格式 | TC1: 导出xlsx → 文件可正常打开，数据完整；TC2: 导出PDF → 文件可正常打开，排版正确 | 等价类(有效格式) |
| R2 | 数据一致性 | TC3: 页面按金额降序筛选 → 导出 → 验证导出文件的排序和筛选与页面一致 | 场景法 |
| R3 | 数据量上限 | TC4: 导出100万行xlsx → 成功；TC5: 导出1000001行 → 提示超限；TC6: 导出101页PDF → 截断或提示 | 边界值法 |
| R4 | 文件名格式 | TC7: 导出后检查文件名是否包含报表名称+时间戳+操作人 | 基于需求 |
| R5 | 异步导出+通知 | TC8: 导出15000行 → 提示"数据量较大，请稍后在通知中心下载"→ 导出完成后收到站内信 | 场景法+状态迁移 |
| R6 | 链接有效期24h | TC9: 导出完成后23h59min → 链接可下载；TC10: 导出完成后24h01min → 链接过期 | 边界值法 |
| R7 | 审计日志 | TC11: 检查日志表是否记录了：操作人、操作时间、导出参数、文件名、数据量、IP地址 | 基于需求 |
| R8 | 频次限制 | TC12: 1h内导出第10次 → 成功；TC13: 1h内导出第11次 → 提示"导出次数超限，请稍后再试" | 边界值法 |
| *补充* | 空数据导出 | TC14: 查询结果为空时点击导出 → 导出空文件还是提示"无数据可导出"？ | 错误推测法 |
| *补充* | 并发导出 | TC15: 同一账号同时触发2个导出任务 → 两个任务并行执行且互不影响 | 错误推测法 |

**场景3：消息推送功能 - 功能性+非功能性需求**

- **需求描述**：
  1. 支持按用户ID、用户标签、用户分群三种方式选择推送目标
  2. 推送内容支持文本+图片+链接
  3. 支持立即推送和定时推送
  4. 支持Android/iOS/Web三端同步推送
  5. 用户免打扰时段（22:00-8:00）不推送营销类消息
  6. 推送送达率≥95%（1小时内）
  7. 推送失败自动重试3次，间隔5分钟
  8. 需要推送数据统计面板（发送量/送达量/点击量/点击率）

- **测试点与用例设计**：

| 需求编号 | 需求点 | 测试用例 | 验证方法 |
|----------|--------|----------|----------|
| R1 | 三种推送目标选择 | TC1: 选择指定用户ID(100个) → 仅这100人收到；TC2: 选择"VIP用户"标签 → 所有VIP收到；TC3: 选择"近30天未活跃"分群 → 仅为该分群用户收到 | 等价类+数据库对比 |
| R2 | 文本+图片+链接 | TC4: 仅文本推送 → 正确展示；TC5: 文本+图片 → 图片正常加载；TC6: 含链接 → 点击链接正确跳转 | 场景法 |
| R3 | 立即+定时推送 | TC7: 立即推送 → 30秒内送达；TC8: 设置明天9:00推送 → 准时推送 | 场景法+时间边界 |
| R4 | 三端同步 | TC9: 同一用户登录Android+iOS+Web → 三端均收到推送且可打开（已读状态同步） | 多端验证 |
| R5 | 免打扰时段 | TC10: 营销推送定时22:01 → 应延迟到次日8:00推送或提示不可定时；TC11: 非营销类(交易通知)22:30 → 应正常推送 | 边界值+业务规则 |
| R6 | 送达率≥95% | TC12: 推送10000人 → 1h内送达≥9500人（统计面板+数据库验证） | 性能/数据统计 |
| R7 | 失败重试3次 | TC13: 模拟推送失败 → 5min后重试 → 再失败→ 5min后再重试 → 再失败→ 5min后第3次 → 仍失败则标记"推送失败" | 状态迁移 |
| R8 | 数据统计面板 | TC14: 推送后查看统计：发送量=推送目标数、送达量=实际推到的设备数、点击量=点击通知的人数、计算点击率=点击量/送达量 | 数据一致性 |

**场景4：权限管理系统 - 功能+安全需求**

- **需求描述**：
  1. 系统支持角色管理（创建/编辑/删除角色）
  2. 每个角色可配置菜单权限（可见哪些菜单项）
  3. 每个角色可配置按钮权限（增/删/改/查/导出/审批等操作按钮）
  4. 每个角色可配置数据权限（本人/本部门/本部门及下级/全部）
  5. 用户可拥有多个角色，权限取并集
  6. 角色之间有互斥规则（如：申请人和审批人不能是同一角色）
  7. 权限变更后，已登录用户下次请求时生效（或15分钟内缓存过期）
  8. 所有权限变更操作记录审计日志
  9. 超级管理员角色不可删除、不可编辑

- **测试点与用例设计**：

| 需求编号 | 需求点 | 测试用例 |
|----------|--------|----------|
| R1 | 角色CRUD | TC1: 创建角色"部门经理"→ 成功；TC2: 编辑角色名称 → 成功；TC3: 删除角色（无用户关联）→ 成功；TC4: 删除角色（已有用户关联）→ 提示"角色使用中，无法删除" |
| R2 | 菜单权限 | TC5: 角色A仅配置"用户管理"菜单 → 用户登录后仅能看到"用户管理"菜单，其他菜单不可见 |
| R3 | 按钮权限 | TC6: 角色B配置"用户列表"可查看但无"新增"按钮权限 → 界面上无"新增用户"按钮；TC7: 直接调用新增用户API → 后端拒绝(403) |
| R4 | 数据权限 | TC8: 角色C配置"本部门"数据权限 → 仅能看到本部门的数据；TC9: 角色D配置"本部门及下级" → 能看到本部门和所有下级部门数据 |
| R5 | 多角色并集 | TC10: 用户同时有角色E(菜单A)和角色F(菜单B) → 可看到菜单A+B |
| R6 | 角色互斥 | TC11: 给已有"审批人"角色的用户再分配"申请人"角色 → 提示"角色互斥，无法分配" |
| R7 | 权限缓存过期 | TC12: 管理员修改用户权限 → 用户下次API请求时权限更新；TC13: 等15分钟后缓存过期 → 用户权限自动更新 |
| R8 | 审计日志 | TC14: 每次权限变更后检查审计日志：操作人、操作时间、操作内容(变更前/变更后)、IP地址 |
| R9 | 超级管理员保护 | TC15: 尝试删除/编辑超级管理员角色 → 按钮置灰或操作被拒绝 |

**场景5：订单超时自动取消 - 需求+隐含需求验证**

- **需求描述**：
  1. 待支付订单超过30分钟未支付，系统自动取消
  2. 取消时恢复商品库存
  3. 取消时退回已使用的优惠券
  4. 取消时已使用的积分退回
  5. 自动取消后发送APP推送和短信通知用户
  6. 自动取消的订单在订单列表显示为"已取消（超时）"

- **测试点与用例设计**：

| 需求编号 | 需求点 | 测试用例 | 所用方法 |
|----------|--------|----------|----------|
| R1 | 30分钟超时 | TC1: 下单后29分59秒 → 订单仍为"待支付"；TC2: 下单后30分00秒 → 订单变为"已取消" | 边界值 |
| R2 | 库存恢复 | TC3: 商品原库存10件，下单1件(库存9件)→超时取消→库存恢复为10件 | 数据验证 |
| R3 | 优惠券退回 | TC4: 使用满200减30优惠券→超时取消→优惠券退回用户账户，状态恢复为"未使用" | 状态迁移 |
| R4 | 积分退回 | TC5: 使用1000积分抵扣10元→超时取消→1000积分退回账户 | 数据验证 |
| R5 | 双通道通知 | TC6: 超时取消后30秒内→收到APP推送(标题+内容正确)；TC7: 收到短信(内容正确) | 端到端验证 |
| R6 | 状态展示 | TC8: 订单列表→该订单状态为"已取消"，取消原因"超时未支付" | UI验证 |

- **隐含需求补充测试**：

| 隐含需求 | 测试用例 |
|----------|----------|
| 定时任务稳定性 | TC9: 同时有10000个待支付订单到期 → 定时任务能否在合理时间内(如5分钟)全部处理完成 |
| 定时任务幂等性 | TC10: 定时任务执行一半时重启 → 不会重复取消已取消的订单 |
| 与人工取消并行 | TC11: 在自动取消的同一秒，用户手动点击支付 → 支付应成功还是被拒绝？（需和产品确认策略：建议"支付成功"优先，因为用户意愿明确） |
| 与自动取消并行 | TC12: 用户在29分50秒发起支付，支付处理了20秒(30分10秒完成)→支付成功→不应被自动取消（以支付回调时间为准判断超时） |
| 分布式时钟一致性 | TC13: 订单服务时间(服务器A)和定时任务时间(服务器B)不一致 → 超时判断以哪个为准？（应使用同一时钟源或数据库时间） |
| 库存恢复的并发安全 | TC14: 自动取消恢复库存的同时，另一个用户正在下单购买最后一件 → 库存数量正确，不超卖 |

## 附录：黑盒测试设计方法对比总结

| 序号 | 方法名称 | 核心思想 | 适用场景 | 与其他方法的关系 |
|------|----------|----------|----------|------------------|
| 1 | **等价类划分法** | 分组选代表，减少用例数 | 有明确输入域的功能点 | 常与边界值法搭配使用，是最基础的用例设计方法 |
| 2 | **边界值分析法** | BUG多出在边界 | 有数值范围/长度限制/时间范围的场景 | 等价类法的补充，两者互补、互不归属 |
| 3 | **场景分析法** | 模拟用户真实操作流程 | 业务流程长的端到端测试 | 可结合状态迁移法用于复杂场景 |
| 4 | **错误推测法** | 经验驱动，查漏补缺（ISTQB归类为基于经验的技术，非基于规格） | 高风险模块、项目收尾阶段 | 作为其他系统化方法的补充，不可单独依赖 |
| 5 | **判定表法** | 穷举条件组合，合并化简 | 多个条件互相影响结果的逻辑判断 | 因果关系复杂时可先用因果图法分析，再转换为判定表 |
| 6 | **因果图法** | 图形化分析条件和结果的逻辑关系 | 条件间有复杂逻辑约束的场景 | 是判定表的前置分析工具，画完后转换为判定表 |
| 7 | **正交试验法** | 用正交表均匀采样，大幅减少组合数 | 多参数多取值的配置/兼容性测试（参数水平数一致） | 极端组合需额外用边界值法补测 |
| 8 | **Pairwise/All-Pairs** | 覆盖所有两两参数组合，工具自动生成 | 参数水平数不一致的场景、任意数量参数的组合测试 | 不保证三参数及以上交互组合的覆盖 |
| 9 | **状态迁移法** | 验证状态转换的正向和非法拦截 | 有明确状态生命周期的业务对象 | 可与场景分析法结合用于复合业务 |
| 10 | **功能图法** | 同时覆盖功能步骤和状态变化 | 复杂复合型业务系统（国内教材方法，非ISTQB标准方法） | 是场景分析法+状态迁移法的综合运用 |
| 11 | **随机测试法** | 随机输入/操作，发现意外缺陷 | 稳定性测试、安全测试、查漏补缺 | 不可替代系统化方法，作为补充和最后一关 |
| 12 | **基于需求测试法** | 逐条对照需求文档，保证需求全覆盖 | 所有测试的基础和起点 | 所有其他方法的基础，配合使用确保需求追溯 |

> **分类备注**：ISTQB将测试技术分为三大类——(1)基于规格的黑盒技术（等价类/边界值/判定表/状态迁移/用例测试）；(2)基于结构的白盒技术（语句/判定/条件/MC/DC等）；(3)基于经验的技术（错误推测/探索性测试）。本文档中的"错误推测法"严格属于第(3)类,"功能图法"为国内实践产物而非ISTQB标准方法，纳入本文档旨在提供全面的实践参考。

---

> **使用建议**：实际项目中，通常以**基于需求测试法**为基础框架，根据被测功能的特点灵活组合其他方法。例如：一个用户注册需求，可以先用基于需求法梳理测试点，再用等价类+边界值设计输入校验用例，用场景法设计全流程用例，最后用错误推测法补充安全性用例。
