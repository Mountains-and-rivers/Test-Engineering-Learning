## 1. 语句覆盖法

### 通俗释义

语句覆盖就是让程序里的每一行代码至少被执行一次。可以把它理解为"代码行走到没"——如果某行代码从未被运行过，就等于没有测试到它。

### 详细解释

语句覆盖法（Statement Coverage）是最基础、最底线的白盒测试方法。它的核心思想是：设计测试用例，使程序中的每一条可执行语句（不包括注释、空行、声明）至少被执行一次。语句覆盖率是衡量测试充分性最基础的指标。

**覆盖率的计算公式：**

```
语句覆盖率 = 已执行的可执行语句数 / 所有可执行语句总数 × 100%
```

**语句覆盖的局限性：**

语句覆盖虽然是入门标准，但它是白盒测试中最弱的一种覆盖。仅仅满足100%语句覆盖，往往会漏掉大量逻辑错误。最经典的例子是：一个if条件判断，即使只执行True分支，False分支里的bug也不会被发现。

**设计步骤：**
1. 分析被测代码，识别所有可执行语句（排除注释、声明语句、空行等）
2. 梳理代码的控制流图（Control Flow Graph），标记每个基本块
3. 设计初始测试用例，从入口走到出口
4. 记录已覆盖的语句行号
5. 对未覆盖的语句，分析需要什么样的输入才能走到
6. 补充测试用例，直到所有语句都被执行过
7. 输出语句覆盖率报告

### 实际使用场景（5个）

**场景1：计算器程序的基础验证**

```java
public int calculate(int a, int b, String op) {
    int result = 0;                          // 语句1
    if (op.equals("add")) {                  // 语句2
        result = a + b;                      // 语句3
    } else if (op.equals("sub")) {           // 语句4
        result = a - b;                      // 语句5
    } else if (op.equals("mul")) {           // 语句6
        result = a * b;                      // 语句7
    } else {                                 // 语句8
        result = a / b;                      // 语句9
    }
    return result;                           // 语句10
}
```

- **语句覆盖目标**：设计用例走到语句3/5/7/9中的每一个
- **用例1**：`calculate(3, 2, "add")` → 覆盖语句1, 2, 3, 10（覆盖add分支）
- **用例2**：`calculate(3, 2, "sub")` → 覆盖语句1, 2, 4, 5, 10（覆盖sub分支）
- **用例3**：`calculate(3, 2, "mul")` → 覆盖语句1, 2, 4, 6, 7, 10（覆盖mul分支）
- **用例4**：`calculate(3, 2, "div")` → 覆盖语句1, 2, 4, 6, 8, 9, 10（覆盖div分支）
- **4个用例实现100%语句覆盖**。但注意：没有任何用例覆盖`op`为null时语句2调用`op.equals()`抛出NullPointerException的情况

**场景2：用户登录验证代码**

```java
public LoginResult login(String username, String password) {
    if (username == null || username.isEmpty()) {      // 语句1
        return LoginResult.USERNAME_EMPTY;              // 语句2
    }
    if (password == null || password.isEmpty()) {      // 语句3
        return LoginResult.PASSWORD_EMPTY;              // 语句4
    }
    User user = userDao.findByUsername(username);      // 语句5
    if (user == null) {                                 // 语句6
        return LoginResult.USER_NOT_FOUND;              // 语句7
    }
    if (!passwordEncoder.matches(password, user.getPassword())) { // 语句8
        return LoginResult.PASSWORD_ERROR;              // 语句9
    }
    return LoginResult.SUCCESS;                        // 语句10
}
```

- **语句覆盖设计**：
  - TC1: `login(null, "123456")` → 覆盖语句1, 2（用户名为空）
  - TC2: `login("test", null)` → 覆盖语句1, 3, 4（密码为空）
  - TC3: `login("nouser", "123456")` → 覆盖语句1, 3, 5, 6, 7（用户不存在）
  - TC4: `login("user1", "wrong")` → 覆盖语句1, 3, 5, 6, 8, 9（密码错误）
  - TC5: `login("user1", "correct")` → 覆盖语句1, 3, 5, 6, 8, 10（登录成功）
- **5个用例实现100%语句覆盖**。但注意：语句8中`passwordEncoder.matches()`本身的内部代码是否被覆盖？这涉及函数嵌套调用的覆盖粒度问题

**场景3：数组查找功能**

```java
public int findMax(int[] arr) {
    if (arr == null || arr.length == 0) {      // 语句1
        throw new IllegalArgumentException("数组为空"); // 语句2
    }
    int max = arr[0];                           // 语句3
    for (int i = 1; i < arr.length; i++) {     // 语句4
        if (arr[i] > max) {                     // 语句5
            max = arr[i];                       // 语句6
        }
    }
    return max;                                 // 语句7
}
```

- **语句覆盖设计**：
  - TC1: `findMax(null)` → 覆盖语句1, 2（空数组异常）
  - TC2: `findMax(new int[]{5, 3, 8, 1, 6})` → 覆盖语句1, 3, 4(多次), 5(多次), 6(当8>5时), 7（正常查找最大值）
  - 注意：TC2中语句6在元素等于/小于当前最大值时不会执行（如3<5时不进入if），但语句6本身已经被覆盖过一次（8>5时），所以达到100%语句覆盖
- **2个用例实现100%语句覆盖**。但仅2个用例，判断语句5的两种结果（true/false）并未都被覆盖

**场景4：订单价格计算（含折扣逻辑）**

```java
public double calcOrderPrice(double amount, boolean isVip, boolean hasCoupon) {
    double discount = 1.0;                            // 语句1
    if (isVip) {                                       // 语句2
        discount = 0.85;                               // 语句3
    }
    if (hasCoupon) {                                   // 语句4
        discount = discount * 0.9;                     // 语句5
    }
    if (amount > 1000) {                               // 语句6
        discount = discount * 0.8;                     // 语句7
    }
    double finalPrice = amount * discount;              // 语句8
    return finalPrice;                                  // 语句9
}
```

- **语句覆盖设计**：只需1个用例即可覆盖全部语句：
  - TC1: `calcOrderPrice(1500, true, true)` → VIP且有优惠券且金额超1000，覆盖所有语句1-9
  - 但仅此1个用例，虽然100%语句覆盖，却完全没有验证"非VIP"、"无优惠券"、"金额<=1000"这些分支的行为！这在测试中是非常危险的"伪充分"
- **教训**：语句覆盖100%不代表测试充分

**场景5：简单的数值范围分类器**

```java
public String classifyNumber(int num) {
    String result;                                      // 声明不可执行
    if (num < 0) {                                      // 语句2
        result = "负数";                                 // 语句3
    } else if (num == 0) {                              // 语句4
        result = "零";                                   // 语句5
    } else if (num > 0 && num <= 100) {                 // 语句6
        result = "正小值";                               // 语句7
    } else {                                            // 语句8
        result = "正大值";                               // 语句9
    }
    System.out.println("分类结果: " + result);           // 语句10
    return result;                                      // 语句11
}
```

- **语句覆盖设计**：
  - TC1: `classifyNumber(-5)` → 覆盖语句2, 3, 10, 11
  - TC2: `classifyNumber(0)` → 覆盖语句2, 4, 5, 10, 11
  - TC3: `classifyNumber(50)` → 覆盖语句2, 4, 6, 7, 10, 11
  - TC4: `classifyNumber(200)` → 覆盖语句2, 4, 6, 8, 9, 10, 11
- **4个用例实现100%语句覆盖**。但边界值如-1、1、99、100、101没有覆盖；多条件语句6的短路逻辑也没有单独测试

## 2. 判定覆盖法

### 通俗释义

判定覆盖就是让程序里每个if/else的分支都走过一遍——条件为真走过，条件为假也要走过。可以理解为"每个路口向左转和向右转都要走至少一次"。

### 详细解释

判定覆盖法（Decision Coverage / Branch Coverage）要求每个判断的取真分支和取假分支都至少执行一次。判定可以是if条件、while条件、for循环条件、switch分支等。判定覆盖比语句覆盖更强——因为要覆盖每个分支，自然会顺带覆盖大部分语句。

**覆盖率的计算公式：**

```
判定覆盖率 = 已执行的判定结果分支数 / 所有判定结果分支总数 × 100%
```

其中，每个判定有2个结果分支（True/False），switch判定有N个结果分支。判定覆盖率100%也被称为"分支覆盖100%"。

**判定覆盖与语句覆盖的关系：**

- 100%判定覆盖 必然包含 100%语句覆盖（每个分支内的语句必然被执行）
- 100%语句覆盖 不一定包含 100%判定覆盖（可能只走了if的True分支）

**设计步骤：**
1. 绘制程序的控制流图，标注所有判定节点
2. 统计所有判定节点数N和各自的分支数
3. 计算总分支数 = sum(每个判定的分支数)
4. 设计测试用例，覆盖每个判定的每个分支
5. 记录已覆盖的分支
6. 补充用例直到所有分支都被覆盖

### 实际使用场景（5个）

**场景1：电梯楼层调度逻辑**

```java
public String elevatorAction(int currentFloor, int targetFloor, boolean isOverloaded) {
    if (isOverloaded) {                                    // 判定1 (T/F)
        return "超重报警,请减少人数";
    }
    if (currentFloor == targetFloor) {                     // 判定2 (T/F)
        return "已在目标楼层";
    }
    String direction;
    if (targetFloor > currentFloor) {                      // 判定3 (T/F)
        direction = "上行";
    } else {
        direction = "下行";
    }
    return "电梯" + direction + "中, 从" + currentFloor + "楼到" + targetFloor + "楼";
}
```

- **判定覆盖分析**：3个判定，6个分支（每个判定T/F各一）
- **用例设计**：
  - TC1: `elevatorAction(1, 5, true)` → 判定1=T，覆盖"超重报警"分支
  - TC2: `elevatorAction(3, 3, false)` → 判定1=F, 判定2=T，覆盖"已在目标楼层"分支
  - TC3: `elevatorAction(1, 5, false)` → 判定1=F, 判定2=F, 判定3=T，覆盖"上行"
  - TC4: `elevatorAction(5, 1, false)` → 判定1=F, 判定2=F, 判定3=F，覆盖"下行"
- **4个用例实现100%判定覆盖**（6/6分支全部覆盖）。判定覆盖率 = 100%

**场景2：贷款资格审查**

```java
public LoanResult evaluateLoan(int creditScore, double income, boolean hasGuarantor) {
    if (creditScore < 600) {                                       // 判定1
        return LoanResult.REJECTED_LOW_CREDIT;
    }
    if (income < 50000) {                                          // 判定2
        if (hasGuarantor) {                                        // 判定3
            return LoanResult.APPROVED_WITH_GUARANTOR;
        } else {
            return LoanResult.REJECTED_LOW_INCOME;
        }
    }
    if (creditScore >= 750 && income >= 100000) {                  // 判定4
        return LoanResult.APPROVED_PREMIUM;
    }
    return LoanResult.APPROVED_STANDARD;
}
```

- **判定覆盖分析**：4个判定，8个分支
- **用例设计**：
  - TC1: `evaluateLoan(500, 30000, false)` → 判定1=T → REJECTED_LOW_CREDIT
  - TC2: `evaluateLoan(650, 30000, true)` → 判定1=F, 判定2=T, 判定3=T → APPROVED_WITH_GUARANTOR
  - TC3: `evaluateLoan(650, 30000, false)` → 判定1=F, 判定2=T, 判定3=F → REJECTED_LOW_INCOME
  - TC4: `evaluateLoan(800, 120000, false)` → 判定1=F, 判定2=F, 判定4=T → APPROVED_PREMIUM
  - TC5: `evaluateLoan(700, 60000, false)` → 判定1=F, 判定2=F, 判定4=F → APPROVED_STANDARD
- **5个用例实现100%判定覆盖**。但注意判定4是复合条件（有AND），判定覆盖只看整体true/false结果，不管`creditScore>=750`和`income>=100000`各自的true/false组合——这是条件覆盖要解决的问题

**场景5：在线支付回调处理**

```java
public PaymentResult handlePaymentCallback(String orderId, String status, double amount) {
    Order order = orderDao.findById(orderId);                    // 语句1
    if (order == null) {                                         // 判定1
        return PaymentResult.ORDER_NOT_FOUND;
    }
    if (!order.getStatus().equals("PENDING")) {                  // 判定2
        return PaymentResult.ORDER_STATUS_INVALID;
    }
    if (!status.equals("SUCCESS") && !status.equals("FAIL")) {  // 判定3
        return PaymentResult.UNKNOWN_STATUS;
    }
    if (status.equals("SUCCESS")) {                              // 判定4
        if (Math.abs(amount - order.getAmount()) > 0.01) {      // 判定5
            return PaymentResult.AMOUNT_MISMATCH;
        }
        order.setStatus("PAID");
    } else {
        order.setStatus("PAY_FAILED");
    }
    orderDao.save(order);
    return PaymentResult.SUCCESS;
}
```

- **判定覆盖分析**：5个判定，10个分支
- **用例设计**：
  - TC1: 传入不存在订单号 → 判定1=T → ORDER_NOT_FOUND
  - TC2: 传入已支付订单 → 判定1=F, 判定2=T → ORDER_STATUS_INVALID
  - TC3: 传入非法status值 → 判定1=F, 判定2=F, 判定3=T → UNKNOWN_STATUS
  - TC4: 支付成功但金额不匹配 → 判定1=F, 判定2=F, 判定3=F, 判定4=T, 判定5=T → AMOUNT_MISMATCH
  - TC5: 支付成功金额匹配 → 判定1=F, 判定2=F, 判定3=F, 判定4=T, 判定5=F → SUCCESS (PAID)
  - TC6: 支付失败 → 判定1=F, 判定2=F, 判定3=F, 判定4=F → SUCCESS (PAY_FAILED)
- **6个用例实现100%判定覆盖**（10/10分支全部覆盖）

## 3. 条件覆盖法

### 通俗释义

条件覆盖是深入到判定里面的每一个条件——判定是由多个条件组成的（比如 `a && b` 里有两个条件），每个条件都要分别取真和取假各一次。可以理解为"不仅路口左右转弯走了，还要验证路口每个信号灯的红绿灯都亮过"。

### 详细解释

条件覆盖法（Condition Coverage）要求每个判定中的每个简单布尔条件都至少取一次真值和一次假值。这里"简单条件"指不能再分解的原子条件（如 `a > 5`、`b == true`、`c != null` 等），不含逻辑运算符&&、||、!。

**覆盖率的计算公式：**

```
条件覆盖率 = 已覆盖的条件取值数 / 所有条件的总可能取值数 × 100%
```

例如判定 `if (a > 0 && b < 10)`，有2个条件：
- 条件1: `a > 0` 需覆盖 T 和 F
- 条件2: `b < 10` 需覆盖 T 和 F
- 总共需要覆盖 4 个条件取值

**条件覆盖 vs 判定覆盖：**

- 100%条件覆盖 不一定 100%判定覆盖（经典例子：两个条件都为T和都为F时整体判定结果可能相同）
- 100%判定覆盖 不一定 100%条件覆盖（判定覆盖只看整体true/false，不关心内部条件各自的取值）

**设计步骤：**
1. 分析代码中的所有判定
2. 将每个判定分解为多个简单条件
3. 列出所有简单条件及其 T/F 取值要求
4. 设计测试用例，使每个条件都至少取一次T和一次F
5. 记录已覆盖的条件取值
6. 补充用例直到所有条件取值都被覆盖

### 实际使用场景（5个）

**场景1：用户注册参数校验**

```java
public boolean validateRegistration(String username, String password, int age) {
    if (username != null && username.length() >= 4 && password != null 
        && password.length() >= 6 && age >= 18) {       // 判定包含5个简单条件
        return true;
    }
    return false;
}
```

- **条件分解**：
  - C1: `username != null` (T/F)
  - C2: `username.length() >= 4` (T/F)
  - C3: `password != null` (T/F)

## 4. 判定-条件覆盖法

### 通俗释义

判定-条件覆盖是判定覆盖和条件覆盖的结合——既要每个判定整体取真取假都走过，又要每个判定内部的每个条件取真取假都走过。可以理解为"既保证每个路口的左右转都走了，又保证每个信号灯的红绿灯都亮过"。

### 详细解释

判定-条件覆盖法（Decision/Condition Coverage）同时满足判定覆盖和条件覆盖的要求。它是两者的交集：既要求100%判定覆盖，又要求100%条件覆盖。这是比前两者都更强的覆盖标准。

**覆盖要求：**

1. 每个判定取真一次、取假一次（判定覆盖要求）
2. 每个简单条件取真一次、取假一次（条件覆盖要求）

**判定-条件覆盖的局限性：**

虽然比单独的判定覆盖或条件覆盖更强，但判定-条件覆盖仍然有缺陷：它不要求每个判定的内部条件之间的组合都被测试到。例如 `if (a && b)`，判定-条件覆盖只要满足：
- 判定整体结果为T和F各一次
- a取T和F各一次
- b取T和F各一次
但不要求验证 a=T+b=F 或 a=F+b=T 等具体组合。

**设计步骤：**
1. 列出所有判定及其内部条件
2. 列出判定覆盖的要求（每个判定T/F各一次）
3. 列出条件覆盖的要求（每个条件T/F各一次）
4. 合并两个要求，找出能同时满足的最小用例集
5. 设计测试用例
6. 验证每个用例是否同时满足判定和条件覆盖
7. 补充未能同时满足的用例

## 5. 条件组合覆盖法

### 通俗释义

条件组合覆盖是最严格的基于判定条件的覆盖——把每个判定里所有条件的T/F排列组合全部测一遍。比如判定有3个条件，就有8种组合（2^3），8种全都要测到。可以理解为"把每个路口信号灯的所有可能颜色组合都试一遍"。

### 详细解释

条件组合覆盖法（Multiple Condition Coverage / Branch Condition Combination Coverage）要求每个判定中的所有简单条件的T/F取值组合全部被覆盖。对于一个有N个简单条件的判定，需要覆盖2^N种条件组合。

**覆盖率的计算公式：**

```
条件组合覆盖率 = 已覆盖的条件组合数 / 所有可能的条件组合总数 x 100%
```

**条件组合覆盖的优势与代价：**

| 方面 | 说明 |
|------|------|
| **优势** | 能发现判定内部条件之间的微妙交互问题；覆盖最详尽 |
| **代价** | 组合爆炸——N个条件需要2^N个组合；对于短路语言（Java/C/C++），某些组合不可达 |
| **工程实践** | 通常只对关键业务逻辑使用100%条件组合覆盖；非关键代码使用判定-条件覆盖即可 |

**设计步骤：**
1. 分析代码中的每个判定，提取所有简单条件
2. 对每个判定，计算条件组合总数 = 2^n（n = 条件数）
3. 列出每个判定的所有条件组合（真值表）
4. 检查每种组合是否可达（不可达的组合标记为N/A）
5. 为每种可达组合设计测试用例
6. 验证覆盖率并输出报告

## 6. MC/DC 修正条件判定覆盖法

### 通俗释义

MC/DC要求每个条件都能"独立影响"判定结果——即使其他条件保持不变，单个条件的翻转就能改变整个判定的结果。可以理解为"每个条件都有话语权，单独一个能改变最终决定"。

### 详细解释

修正条件判定覆盖（MC/DC, Modified Condition/Decision Coverage）是比条件组合覆盖更精简但同样严格的覆盖标准。它在航空软件 DO-178C 标准中被强制要求（Level A 级别软件），也在汽车功能安全标准 ISO 26262、轨道交通 EN 50128 等安全关键领域中被广泛采用。

**MC/DC的三大核心要求：**

| 要求 | 说明 |
|------|------|
| **判定覆盖** | 每个判定至少有一次True和一次False的结果 |
| **条件覆盖** | 判定中的每个条件至少有一次True和一次False的取值 |
| **独立影响** | 每个条件必须能独立地影响判定结果（在其他条件固定不变的前提下，该条件的翻转导致整体判定结果的改变） |

**MC/DC与条件组合覆盖的关系：**

- 条件组合覆盖要求 2^N 个用例（N为条件数），MC/DC仅要求 N+1 个用例（最少情况）
- MC/DC基于"每个条件独立影响判定"这一原则来选取最小用例集
- MC/DC在安全关键领域的覆盖充分性与条件组合覆盖相当，但用例数量大幅减少

**设计步骤：**
1. 识别判定中的所有简单条件（原子条件）
2. 列出能同时满足判定覆盖和条件覆盖的候选用例
3. 对每个条件，找到一对用例：该条件取值不同而其他条件取值相同，且判定结果不同（证明该条件的独立影响力）
4. 汇总所有N+1个用例（基于最小化原则）
5. 对于不可达的条件组合，标记为N/A并记录原因
6. 验证MC/DC覆盖率 = 已证独立影响的条件数 / 总条件数 x 100%

### 实际使用场景（5个）

**场景1：安全关键系统——飞控软件（2个条件，3个用例）**

## 7. 路径覆盖法

### 通俗释义

路径覆盖就是把程序从头到尾所有可能的执行路径全部走一遍——不是只看每个if/else分叉，而是把所有分叉串起来的"全程路线"都跑一遍。可以理解为"不仅要走每个路口，还要把每条从起点到终点的完整路线都走一遍"。

### 详细解释

路径覆盖法（Path Coverage）要求测试用例覆盖程序控制流图中从入口到出口的所有可能路径。一条路径是从程序入口到出口的一条完整的执行轨迹。

**路径数量的计算：**

程序中的循环会导致路径数爆炸（每次循环迭代都是一条不同的路径），所以实际工作中通常区分：

| 路径覆盖级别 | 说明 |
|-------------|------|
| **全路径覆盖** | 覆盖所有可能路径（包含循环的每次迭代组合）— 通常不可达 |
| **基本路径覆盖** | 覆盖所有基本路径（线性无关路径）— McCabe方法 |
| **有限循环路径** | 将循环展开有限次（如0次、1次、多次）|

**设计步骤：**
1. 绘制程序的控制流图（CFG）
2. 标识入口节点和出口节点
3. 枚举所有从入口到出口的路径
4. 对包含循环的路径，设定展开上限（如循环0次、1次、2次）
5. 为每条路径设计测试用例
6. 记录每条路径的覆盖情况

### 实际使用场景（5个）

**场景1：简单的if-else多分支路径**


  - 路径2: 80<=score<90 → "B"
  - 路径3: 70<=score<80 → "C"
  - 路径4: 60<=score<70 → "D"

## 8. 基本路径测试法

### 通俗释义

基本路径测试是路径覆盖的简化版——用数学方法（圈复杂度）算出最少需要测几条"独立路径"，确保每条代码之间的连线都被走过至少一次。可以理解为"保证每个路口都被经过，但不用把所有路线组合都走完"。

### 详细解释

基本路径测试法（Basis Path Testing）由Thomas McCabe在1976年提出，是解决路径组合爆炸问题的工程化方法。其核心思想是：从程序的圈复杂度（Cyclomatic Complexity）计算出线性无关路径的最小集合，覆盖这个最小集合就能保证所有语句和所有分支都被覆盖。

**圈复杂度的三种计算方法：**

| 方法 | 公式 | 说明 |
|------|------|------|
| **边-节点法** | V(G) = E - N + 2 | E = 控制流图边数, N = 节点数 |
| **判定节点法** | V(G) = P + 1 | P = 判定节点数（每个if/while/for/case算1个判断） |
| **区域法** | V(G) = R | R = 控制流图将平面划分成的区域数（含外部区域） |

**圈复杂度的工程意义：**

| V(G) | 含义 | 测试建议 |
|------|------|----------|
| 1-10 | 简单程序，低风险 | 基本路径即可 |
| 11-20 | 中等复杂度，中等风险 | 基本路径 + 关键路径组合 |
| 21-50 | 复杂程序，高风险 | 建议重构；如无法重构则需大量测试 |
| >50 | 极高复杂度，不可测试 | 必须重构 |

**设计步骤：**
1. 将代码转换为控制流图（CFG）
2. 计算圈复杂度 V(G)
3. 确定基本路径集合（数量 = V(G)）
4. 为每条基本路径设计测试用例（包括输入数据和预期输出）
5. 执行测试并验证每条基本路径都被覆盖
6. 确保V(G)条基本路径覆盖了所有节点和所有边

### 实际使用场景（5个）

**场景1：简单条件判断程序（V(G)=4）**


## 9. 循环测试法

### 通俗释义

循环测试就是专门针对程序里的各种循环（for/while/do-while）设计测试——循环执行0次、1次、2次、最大次、超出最大次各测一遍。可以理解为"不仅要看循环转不转，还要看转0圈、转1圈、转到极限、转超了会怎样"。

### 详细解释

循环测试法（Loop Testing）是专门针对循环结构的测试方法。程序中大部分的复杂度和潜在缺陷都集中在循环中（如数组越界、死循环、Off-by-One错误等）。循环测试法将循环分为四类，每类有不同的测试策略。

**循环的四种类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **简单循环** | 单一循环，内无嵌套 | `for(int i=0; i<n; i++)` |
| **嵌套循环** | 循环内部包含另一个循环 | 二维数组遍历 |
| **串联循环** | 多个循环顺序排列，无嵌套 | 先遍历再筛选 |
| **非结构化循环** | 使用goto/break等非标准跳转的循环 | 跨循环跳转 |

**简单循环测试策略（7个测试点）：**

| 测试点 | 循环次数 | 测试目的 |
|--------|---------|----------|
| 跳过循环 | 0次 | 循环体不执行（如空数组） |
| 一次通过 | 1次 | 循环体执行一次 |
| 两次通过 | 2次 | 循环体执行两次（验证两次之间状态传递） |
| 典型次数 | m次（m<n, m<max） | 正常循环场景 |
| 边界-1 | n-1次 | 边界前一格 |
| 边界 | n次 | 正好上限 |
| 边界+1 | n+1次 | 超出上限 |

**嵌套循环的测试策略：**
1. 固定所有外层循环为最小值，对内层做简单循环测试
2. 固定所有内层循环为"典型值"，对外层做简单循环测试
3. 所有循环都取最小值（全为0）
4. 所有循环都取最大值
5. 交叉边界：内层最大+外层最小，内层最小+外层最大

**设计步骤：**
1. 识别代码中的所有循环，分类为简单/嵌套/串联/非结构化
2. 对简单循环：设计0, 1, 2, m, n-1, n, n+1次（7个用例）
3. 对嵌套循环：由内向外或由外向内逐步测试
4. 对串联循环：验证前一个循环的输出是否正确传递给后一个循环
5. 对非结构化循环：优先建议重构，无法重构则做全路径分析

```java
public double[] calculateClassAverages(int[][] scores) {

    for (int c = 0; c < numClasses; c++) {        // 外层循环
```

```java
public List<String> processItems(List<Item> items) {

    List<Item> filtered = new ArrayList<>();
    for (Item item : items) {                              // 循环A
        if (item.getPrice() > 0 && item.getStock() > 0) {
            filtered.add(item);
        }
    }
    // 循环2: 排序
    for (int i = 0; i < filtered.size() - 1; i++) {        // 循环B(外层)

                filtered.set(j + 1, temp);

        }

    // 循环3: 格式化
```

```java
public int findInMatrix(int[][] matrix, int target) {
    int row = -1, col = -1;
    outerLoop:
    for (int i = 0; i < matrix.length; i++) {
        for (int j = 0; j < matrix[i].length; j++) {
            if (matrix[i][j] == target) {

                col = j;
```

## 10. 数据流测试法

### 通俗释义

数据流测试关注的是代码中数据的"来龙去脉"——一个变量在哪里被定义了（赋值），在哪里被使用了（读取），中间有没有被错误修改。可以理解为"跟踪每一笔数据的生命轨迹，确保从出生到使用没有意外"。

### 详细解释

数据流测试法（Data Flow Testing）基于程序的数据流分析，通过追踪变量从定义（Definition）到使用（Use）的路径来设计测试用例。它关注的是变量状态的变化，而非控制流的走向。

**核心概念 - 定义-使用对（DU-Pair）：**

| 概念 | 符号 | 说明 | 示例 |
|------|------|------|------|
| **定义（Definition）** | def(x) | 变量x被赋值的语句 | `x = 5;` |
| **使用（Use）** | c-use(x) / p-use(x) | 变量x被读取的语句 | 计算使用: `y = x + 1`; 判定使用: `if (x > 0)` |
| **定义-使用链（DU-Chain）** | def -> use | 从一个定义点到它的使用点的路径 | `x=5; ... y=x+1;` |
| **定义清除（Kill）** | kill(x) | 变量x被重新定义（旧值失效）| 再次赋值`x=10;` 会清除x=5的定义 |

**数据流覆盖的三种级别：**

| 级别 | 覆盖要求 |
|------|----------|
| **全定义覆盖（All-Defs）** | 每个定义至少有一条路径到达某个使用点 |
| **全使用覆盖（All-Uses）** | 每个定义到每个使用点之间的路径都要覆盖 |
| **全定义-使用路径覆盖（All-DU-Paths）** | 所有定义到使用的每条可能路径都覆盖（含循环去环） |

**常见的数据流异常：**

| 异常类型 | 说明 | 示例 |
|----------|------|------|
| **定义后未使用** | 变量被赋值但从未被读取 | `int x = 5; // 后续从未使用` |
| **使用前未定义** | 变量在赋值前被读取 | `int y = x + 1; // x未初始化` |
| **定义后重复定义** | 第一次赋值后尚未使用又被赋值 | `x=5; x=10; // x=5从未被使用` |

**设计步骤：**
1. 构建程序的控制流图（CFG）
2. 识别所有变量的定义点和计算使用点(c-use)、判定使用点(p-use)
3. 构建定义-使用对（DU-Pairs）列表
4. 分析每个DU-Pair之间的路径是否可达
5. 选择覆盖级别（All-Defs / All-Uses / All-DU-Paths）
6. 为每个DU-Pair设计测试用例
7. 执行并验证数据流覆盖

### 实际使用场景（5个）

**场景1：变量定义后被中间赋值覆盖的Bug检测**

```java
public double calculateFinalPrice(double basePrice, boolean isVip) {
    double discount = 0.1;                          // def(discount) @语句1
    // ... 省略若干行代码 ...
    discount = 0.05;                                // def(discount) @语句2 - 覆盖了语句1的定义
    if (isVip) {
        discount = 0.2;                             // def(discount) @语句3
    }
    return basePrice * (1 - discount);              // use(discount) @语句4
}
```

- **数据流分析**：
  - def(discount)@语句1的值(0.1)从未被使用——被语句2的定义杀死
  - use(discount)@语句4实际使用的是语句2或语句3的定义
  - 暴露了"死代码"或逻辑缺陷：语句1的`discount=0.1`可能是意图使用的初始值，但被语句2误覆盖
- **数据流测试用例**：
  - TC1 (VIP): basePrice=100, isVip=true → 期望discount=0.2 → 最终价=80
  - TC2 (非VIP): basePrice=100, isVip=false → 期望???
    - 期望可能是0.1折扣(90元)但实际执行的是0.05折扣(95元)
    - 数据流测试暴露了这个问题：语句1的定义从未到达使用点

**场景2：使用前未赋值的危险路径检测**

- **测试用例设计**：

**场景3：订单处理中的数据流完整验证**

**场景4：全局变量的数据流污染检测**

**场景5：数据库连接池的生命周期数据流**

```java
public class ConnectionPool {
    private Connection conn = null;                     // def(conn)=null
    private boolean closed = false;
    
    public Connection getConnection() {
        if (closed) {
            throw new IllegalStateException("连接池已关闭");
        }
        if (conn == null) {                             // p-use(conn)
            conn = createNewConnection();               // def(conn) @语句1
        } else if (!conn.isValid(5)) {                  // p-use(conn)
            conn.close();
            conn = createNewConnection();               // def(conn) @语句2 - 重建
        }
        return conn;                                    // use(conn)
    }
    
    public void closePool() {
        if (conn != null && !conn.isClosed()) {         // p-use(conn)
            conn.close();
        }
        conn = null;                                    // kill+def(conn)=null @语句3
        closed = true;
    }
}
```

- **关键DU-Pair追踪**：
  - DU1: def(conn)@初始null -> p-use@getConnection中的conn==null检查
  - DU2: def(conn)@语句1 -> p-use@getConnection中的isValid检查 -> p-use@return
  - DU3: def(conn)@语句1 -> p-use@closePool -> kill@语句3
  - DU4: def(conn)@语句2（重连）-> use@return -> 后续被其他方法使用

- **数据流测试用例**：
  - TC1: 首次getConnection -> def@1 -> 返回新连接
  - TC2: 连接失效后getConnection -> 检测失效 -> def@2(重建) -> 返回新连接
  - TC3: closePool -> use(close) -> def@3(null)
  - TC4: closePool后再调用getConnection -> 应抛出异常
  - TC5: 获取连接 -> 外部关闭连接 -> getConnection再调用 -> 重建

## 11. 变异测试法

### 通俗释义

变异测试就是故意往代码里"种虫子"（做一点小改动），然后跑一遍测试用例，看测试用例能不能发现这个改动导致的错误。如果测试用例没发现，说明测试用例不够好——有漏网之鱼。可以理解为"故意埋雷看排雷兵能不能找到"。

### 详细解释

变异测试法（Mutation Testing）是一种评估测试用例质量的白盒方法。它不是直接测试程序，而是通过测试"测试用例本身"的有效性：对被测程序做微小改动（称为"变异"），生成一系列变异体（Mutant），然后运行现有测试用例集，看能"杀死"多少变异体。

**核心概念：**

| 概念 | 说明 |
|------|------|
| **变异体（Mutant）** | 对原程序做单一小改动后的版本 |
| **变异算子（Mutation Operator）** | 进行改动的规则（如把`>`改成`>=`、把`+`改成`-`） |
| **杀死变异体（Kill）** | 至少有一个测试用例在变异体上运行失败 |
| **存活变异体（Survive）** | 所有测试用例在变异体上都通过——测试用例存在盲区 |
| **等效变异体（Equivalent Mutant）** | 语法上不同但语义上与程序等价（如`a=a+1`改为`a=1+a`） |

**变异分数：**

```
变异分数 = 被杀死的变异体数 / (总变异体数 - 等效变异体数) x 100%
```

变异分数越接近100%，说明测试用例集的质量越高。

**常见变异算子：**

| 类别 | 原操作 | 变异替换 | 示例 |
|------|--------|----------|------|
| **算术运算符** | `+` | `-`, `*`, `/` | `a+b` -> `a-b` |
| **关系运算符** | `>` | `>=`, `<`, `==`, `!=` | `a>b` -> `a>=b` |
| **逻辑运算符** | `&&` | `\|\|` | `a&&b` -> `a\|\|b` |
| **常量修改** | `0` | `1`, `-1` | `x=0` -> `x=1` |
| **语句删除** | 删除一条语句 | — | 删除验证码校验步骤 |
| **返回值修改** | `return x;` | `return null;` | — |

**设计步骤：**
1. 准备高质量的被测代码和已有测试用例集
2. 选择变异测试工具（如PITest for Java, Stryker for JS, MutPy for Python）
3. 选择适用的变异算子集合
4. 运行变异测试：工具自动生成变异体 + 对每个变异体运行全部测试用例
5. 分析报告：查看存活变异体，分析是测试用例缺失还是等效变异体
6. 针对存活变异体补充新的测试用例
7. 重复直到变异分数达到目标（通常>85%为良好，>95%为优秀）

### 实际使用场景（5个）

**场景1：关系运算符变异 - 边界条件测试不足**

```java
// 原代码
public String getAgeGroup(int age) {
    if (age < 18) {
        return "未成年";
    } else if (age >= 18 && age < 60) {
        return "成年";
    } else {
        return "老年";
    }
}
```

- **现有测试用例**：
  - TC1: age=10 -> "未成年"
  - TC2: age=30 -> "成年"
  - TC3: age=70 -> "老年"
- **变异测试结果**：
  - 变异体M1: `age < 18` -> `age <= 18` → TC1通过(10<=18), TC2通过(30<=18? false), TC3通过 → **存活！**
  - 变异体M2: `age < 60` -> `age <= 60` → TC2通过(30<=60), TC3通过(70<=60? false) → **存活！**
- **分析**：测试用例没有覆盖边界值18和60，导致变异体存活
- **改进**：补充边界值测试
  - TC4: age=18 -> 期望"成年"
  - TC5: age=17 -> 期望"未成年"
  - TC6: age=60 -> 期望"老年"
  - TC7: age=59 -> 期望"成年"
- 补充后M1和M2都被杀死，变异分数提升

**场景2：算术运算符变异 - 金额计算精度**

```java
public double calculateDiscount(double price, int discountPercent) {
    double discount = price * (discountPercent / 100.0);
    return price - discount;
}
```

- **现有测试用例**：
  - TC1: price=100, discountPercent=20 -> 期望80
  - TC2: price=200, discountPercent=50 -> 期望100
- **变异测试的变异体**：
  - M1: `price * (...)` -> `price / (...)` → TC1得500 -> 被杀死
  - M2: `price - discount` -> `price + discount` → TC1得120 -> 被杀死
  - M3: `discountPercent / 100.0` -> `discountPercent / 100`（整数除法）→ 100的20/100=0 → TC1仍得100而非80 -> 被杀死
- 现有测试用例杀死了这些变异体，说明算术运算符级别的变异能被检测到

**场景3：逻辑运算符变异 - 权限校验漏洞**

```java
public boolean canAccess(User user, Resource resource) {
    if (user.isAdmin() || (user.isOwnerOf(resource) && user.hasPermission("READ"))) {
        return true;
    }
    return false;
}
```

- **现有测试用例**：
  - TC1: 管理员 -> 期望true
  - TC2: 资源所有者且有READ权限 -> 期望true
  - TC3: 非管理员非所有者 -> 期望false
- **变异测试 - 逻辑运算符变异**：
  - M1: `||` -> `&&` → TC1中isAdmin=true, 后面条件未知 -> 取决于后面条件 => 如果TC1的user不是owner → M1中返回false → TC1失败 → M1被杀死
  - M2: `&&` -> `||` → TC2中isOwner=true, hasPermission=true -> `true || true` -> true -> TC2通过
    - TC3中isAdmin=false, isOwner=false, hasPermission=false -> `false || false || false` = false -> TC3通过 -> **M2存活！**
- **分析**：没有覆盖"非管理员+是所有者但没有READ权限"和"非管理员+非所有者+有READ权限"的情况
- **改进**：
  - TC4: 非管理员 + 是所有者 + 无READ权限 -> 原代码期望false; M2中true -> 失败 -> M2被杀死
  - TC5: 非管理员 + 非所有者 + 有READ权限 -> 原代码期望false; M2中true -> 失败

**场景4：语句删除变异 - 关键验证被跳过**

```java
public TransferResult transfer(Account from, Account to, double amount) {
    if (from == null || to == null) { return TransferResult.NULL_ACCOUNT; }
    if (amount <= 0) { return TransferResult.INVALID_AMOUNT; }
    if (from.getBalance() < amount) { return TransferResult.INSUFFICIENT_BALANCE; }
    from.debit(amount);                                   // 语句4
    to.credit(amount);                                    // 语句5
    auditLog.record(from, to, amount);                    // 语句6
    return TransferResult.SUCCESS;
}
```

- **现有测试用例**：
  - TC1: 正常转账100元 -> 期望SUCCESS, 余额正确
  - TC2: 余额不足 -> 期望INSUFFICIENT_BALANCE
  - TC3: amount=0 -> 期望INVALID_AMOUNT
- **变异测试 - 语句删除**：
  - M1: 删除语句4 `from.debit(amount)` -> TC1: from余额未减, to余额加100 -> 失败 -> M1被杀死
  - M2: 删除语句5 `to.credit(amount)` -> TC1: from余额减100, to余额未加 -> 失败 -> M2被杀死
  - M3: 删除语句6 `auditLog.record(...)` -> TC1: 转账成功, 余额正确 -> **TC1仍通过！** -> M3存活！
- **分析**：审计日志的缺失不影响业务功能，但现有测试用例没有验证审计日志
- **改进**：TC4: 正常转账后 -> 验证auditLog中是否有该转账记录（Mock审计组件检查调用次数和参数）-> M3被杀死

**场景5：常量变异 - 默认值/魔法数字的正确性**

```java
public class RateLimiter {
    private static final int MAX_REQUESTS_PER_MINUTE = 100;
    private static final int BLOCK_DURATION_SECONDS = 60;
    private static final double PREMIUM_MULTIPLIER = 2.0;
    
    public boolean isAllowed(User user) {
        int recentRequests = getRecentRequestCount(user);
        int limit = MAX_REQUESTS_PER_MINUTE;
        if (user.isPremium()) {
            limit = (int)(MAX_REQUESTS_PER_MINUTE * PREMIUM_MULTIPLIER);
        }
        if (recentRequests >= limit) {
            blockUser(user, BLOCK_DURATION_SECONDS);
            return false;
        }
        return true;
    }
}
```

- **常量变异**：
  - M1: `MAX_REQUESTS_PER_MINUTE = 100` -> `99` -> TC: 普通用户请求99次, 原代码99>=100=false -> 允许; M1中99>=99=true -> 阻止 -> 如果TC覆盖了99次则M1被杀死
  - M2: `PREMIUM_MULTIPLIER = 2.0` -> `1.9` -> TC: VIP用户199次, 原代码limit=200, 199<200 -> 允许; M1中limit=190, 199>=190 -> 阻止 -> 被杀死
  - M3: `BLOCK_DURATION_SECONDS = 60` -> `59` -> 需要精确验证封锁时长的测试用例

- **变异测试改进价值**：
  - 为每个常量创建边界测试：MAX_REQUESTS-1, MAX_REQUESTS, MAX_REQUESTS+1
  - 验证PREMIUM用户的精确限额（100*2.0=200的边界）
  - 验证封锁时长的精确值

## 附录：白盒测试设计方法对比总结

### 方法覆盖能力递进关系

| 覆盖强度 | 方法名称 | 覆盖维度 | 力度排序 |
|----------|----------|----------|----------|
| 最弱 | 语句覆盖 | 每条可执行语句至少执行一次 | 1 |
| 弱 | 判定覆盖 | 每个判定的T/F分支至少各一次 | 2 |
| 中 | 条件覆盖 | 每个简单条件的T/F至少各一次 | 3 |
| 中强 | 判定-条件覆盖 | 判定覆盖 + 条件覆盖 | 4 |
| 很强 | 条件组合覆盖 | 每个判定的所有条件T/F排列组合 | 5 |
| 精强 | MC/DC | 每个条件独立影响判定结果（安全关键领域标准） | 5.5 |
| 强 | 路径覆盖 | 所有从入口到出口的完整路径 | 6 |
| 实用强 | 基本路径测试 | 线性无关路径全覆盖（圈复杂度驱动） | 7 |
| 偏结构 | 循环测试 | 循环0/1/2/m/n-1/n/n+1次 | 补充 |
| 偏数据 | 数据流测试 | 定义-使用对的全部路径 | 补充 |
| 质量评估 | 变异测试 | 现有测试用例的质量评估与改进 | 质量度量 |

### 各方法详细对比

| 序号 | 方法名称 | 核心思想 | 适用场景 | 局限性 |
|------|----------|----------|----------|--------|
| 1 | **语句覆盖法** | 每行代码至少执行一次 | 最基础的代码覆盖率检查 | 最弱的覆盖标准，100%语句覆盖仍可能漏掉大量逻辑错误 |
| 2 | **判定覆盖法** | 每个if/else分支都走一遍 | 分支逻辑较多的代码 | 不检查复合条件内部细节 |
| 3 | **条件覆盖法** | 每个条件取T/F至少一次 | 复合条件较多的判定 | 100%条件覆盖不一定100%判定覆盖 |
| 4 | **判定-条件覆盖法** | 同时满足判定覆盖和条件覆盖 | 中等复杂度的判定逻辑 | 不检查条件之间的组合 |
| 5 | **条件组合覆盖法** | 所有条件T/F排列组合全覆盖 | 关键安全/金融业务逻辑 | 组合爆炸（N个条件=2^N个组合） |
| 6 | **MC/DC修正条件判定覆盖** | 每个条件独立影响判定结果 | 航空DO-178C/汽车ISO 26262/医疗IEC 62304等安全关键软件 | 需要严格识别独立影响对，不可达组合需标记 |
| 7 | **路径覆盖法** | 所有完整执行路径全覆盖 | 路径数可控的小型函数 | 循环导致路径爆炸，全路径覆盖常不可达 |
| 8 | **基本路径测试法** | 圈复杂度驱动的独立路径最小集 | 工程中最实用的白盒方法 | 不保证条件组合和循环充分覆盖 |
| 9 | **循环测试法** | 循环0/1/2/m/n-1/n/n+1次专项测试 | 包含循环结构的代码 | 仅覆盖循环，需配合其他方法 |
| 10 | **数据流测试法** | 追踪变量定义-使用路径 | 数据依赖复杂的逻辑 | 分析耗时，需借助工具 |
| 11 | **变异测试法** | 注入缺陷验证测试用例发现能力 | 测试用例质量评估与改进 | 运行耗时长，需排除等效变异体 |

### 覆盖标准之间的包含关系

```
条件组合覆盖 => MC/DC => 判定-条件覆盖 => 判定覆盖 => 语句覆盖
                                          => 条件覆盖 => 语句覆盖
                    路径覆盖 => 判定覆盖
```

> 说明：A => B 表示"100% A覆盖必然包含100% B覆盖"。例如：100%判定覆盖必然100%语句覆盖，但100%条件覆盖不一定100%判定覆盖。MC/DC要求每个条件独立影响判定结果，因此严格强于判定-条件覆盖，但在用例数量上远少于条件组合覆盖。

### 工程实践中的白盒测试策略

| 代码类型 | 推荐覆盖级别 | 方法组合 |
|----------|-------------|----------|
| **普通业务代码** | 判定覆盖 + 基本路径 | 先用基本路径测试法设计核心用例，再补充判定覆盖 |
| **核心计算逻辑** | 条件组合覆盖 + 循环测试 | 对涉及金额、安全的判定使用条件组合覆盖，循环使用专项测试 |
| **安全敏感代码(Web/App)** | 条件组合覆盖 + 数据流测试 | 认证授权等代码需要最严格的条件组合+数据流双重验证 |
| **安全关键系统(航空/汽车/医疗)** | MC/DC + 条件组合覆盖 | DO-178C/ISO 26262/IEC 62304等标准强制要求MC/DC |
| **数据密集型代码** | 数据流测试 + 语句覆盖 | 以数据流覆盖为主，语句覆盖作为兜底 |
| **测试用例验收** | 变异测试 | 用变异测试评估已有测试用例的质量，变异分数<85%则需补充 |

### 通用设计步骤（融合各方法）

1. **分析代码结构**：绘制控制流图（CFG），计算圈复杂度 V(G)
2. **选择覆盖目标**：根据代码重要性和复杂度选择覆盖级别
3. **设计基本路径**：使用基本路径测试法设计 V(G) 条核心用例（保证判定和语句覆盖）
4. **补充条件测试**：对于复合条件，补充条件组合覆盖用例
5. **补充循环测试**：对循环结构补充 0/1/2/n-1/n/n+1 次用例
6. **补充数据流测试**：对关键变量验证定义-使用路径
7. **执行与覆盖率分析**：运行测试，使用工具分析覆盖率缺口
8. **变异测试验证**：对关键模块运行变异测试，确保变异分数 >= 85%
9. **持续改进**：根据发现的问题和覆盖率缺口不断补充测试用例

---

> **使用建议**：白盒测试不是孤立使用的——在实际项目中，白盒测试通常与黑盒测试互补。黑盒测试确保"做对了事"（功能正确），白盒测试确保"做事的方式对了"（代码逻辑正确）。两者结合才能构建完整的测试体系。对于关键系统，建议语句覆盖率>=90%、判定覆盖率>=85%、基本路径100%、关键模块变异分数>=90%。
