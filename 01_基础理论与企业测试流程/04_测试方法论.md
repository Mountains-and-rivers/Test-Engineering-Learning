## 1. 基于风险的测试

### 通俗释义

时间永远不够，全量测试不可能，那就把时间和精力花在"最容易出问题、出了问题影响最大"的地方。基于风险的测试就是识别风险、评估风险、按风险高低分配测试资源。

### 详细解释

#### 风险的定义

在测试语境中，风险 = **发生的概率** x **造成的影响**

| 风险要素 | 说明 | 评价标准 |
|----------|------|----------|
| **概率（Probability）** | 该模块/功能出现缺陷的可能性 | 复杂度、变更频率、历史Bug数、开发者经验 |
| **影响（Impact）** | 缺陷如果发生，对业务/用户的影响程度 | 业务重要性、用户影响面、法律/合规风险 |

#### 风险评估矩阵

```
影响
  │
高 │  中风险        高风险        极高风险
  │  (P低 x I高)    (P高 x I高)    (必须测)
  │
中 │  低风险        中风险        高风险
  │  (可少测)                     (重点测)
  │
低 │  极低风险      低风险        中风险
  │  (可不测)       (可抽测)
  │
  └─────────────────────────────────→ 概率
      低            中            高
```

#### 基于风险的测试流程

```
步骤1：风险识别
  - 与产品、开发、运维一起识别风险
  - 参考历史缺陷数据、线上故障数据
  - 输出：风险清单

步骤2：风险评估
  - 对每个风险评估概率和影响
  - 计算风险等级 = 概率 × 影响
  - 输出：风险等级列表

步骤3：风险排序
  - 按风险等级从高到低排序
  - P0（极高风险）、P1（高风险）、P2（中风险）、P3（低风险）
  - 输出：优先级排序的风险清单

步骤4：测试资源分配
  - 按风险等级分配测试时间和设计深度
  - 高风险：深度测试（全覆盖、多组合）
  - 中风险：常规测试（主要场景+边界）
  - 低风险：轻量测试（烟雾测试+基本功能）

步骤5：持续更新
  - 随着项目推进和新信息出现，更新风险评估
  - 测试中发现的Bug会更新对某些模块的风险认知
```

#### 风险识别Checklist

```
技术风险：
□ 新引入的技术/框架/组件
□ 复杂算法/业务逻辑
□ 历史Bug高发模块
□ 多人频繁修改的模块
□ 集成点（外部系统接口）
□ 性能瓶颈点

业务风险：
□ 核心业务流程（支付、下单）
□ 涉及资金的任何计算
□ 用户数据隐私相关
□ 合规/法律强相关功能
□ 品牌敏感的页面（首页、登录页）

项目风险：
□ 时间紧张的模块
□ 新人开发的模块
□ 需求频繁变更的模块
□ 不同团队协作的边界模块
```

## 2. 基于模型的测试

### 通俗释义

把系统行为用"模型"描述出来（状态图、流程图、决策表等），然后让工具根据模型自动生成测试用例。模型好不好，直接决定测试质量。

### 详细解释

#### 基于模型测试的核心概念

```
系统行为
    │
    ▼
┌──────────┐    自动生成    ┌──────────┐
│  模型    │ ──────────→  │  测试用例  │
│ (Model)  │              │ (Test Case)│
└──────────┘              └──────────┘
    │                          │
    │ 人工/半自动              │ 执行
    ▼                          ▼
┌──────────┐              ┌──────────┐
│ 需求/设计  │              │ 测试结果   │
└──────────┘              └──────────┘
```

#### 常用模型类型

| 模型类型 | 描述 | 适用场景 | 工具 |
|----------|------|----------|------|
| **有限状态机** | 状态+迁移+条件+动作 | 订单状态流转、登录状态管理、审批流程 | GraphWalker, Spec Explorer |
| **流程图/活动图** | 业务流程的步骤和分支 | 购物流程、注册流程、理赔流程 | BPMN工具 |
| **决策表** | 条件组合与对应动作 | 费率计算、优惠规则、权限判断 | Excel, DMN |
| **序列图** | 对象间的交互顺序 | API调用链、消息传递 | UML工具 |
| **Petri网** | 并发和同步行为 | 并发系统、资源竞争场景 | CPN Tools |

#### 状态机模型示例（订单系统）

```
状态机模型：电商订单状态流转

                  ┌─────────┐
                  │  待支付  │
                  └────┬────┘
                       │ 支付成功
                       ▼
                  ┌─────────┐
          ┌───────│  已支付  │───────┐
          │       └────┬────┘       │
          │ 申请退款    │ 商家发货   │ 申请退款
          ▼            ▼            ▼
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │ 退款中   │ │ 已发货   │ │ 退款中   │
     └────┬────┘ └────┬────┘ └────┬────┘
          │            │ 确认收货  │
          ▼            ▼            ▼
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │ 已退款   │ │  完成   │ │ 已退款   │
     └─────────┘ └─────────┘ └─────────┘
```

**从状态机自动生成的测试用例：**
1. 待支付 → 支付成功 → 已支付（正向路径）
2. 已支付 → 商家发货 → 已发货 → 确认收货 → 完成（正向路径）
3. 已支付 → 申请退款 → 退款中 → 已退款（退款路径）
4. 待支付 → 超时取消（取消路径）
5. 已发货 → 申请退款 → 退款中（发货后退款）
6. 待支付 → 支付失败 → 待支付（支付失败恢复）
7. 已支付 → 支付重复通知 → 已支付（幂等性验证）

#### 基于模型测试的优缺点

| 优点 | 缺点 |
|------|------|
| 自动生成海量用例，覆盖度高 | 建模本身工作量大，对技能要求高 |
| 模型即文档，易于理解和维护 | 模型错误导致生成的全部用例都错 |
| 变更影响分析（改模型就知道哪些路径变了） | 对复杂系统的完整建模非常困难 |
| 可重复生成，适应需求变更 | 生成的用例可能过多，需要筛选 |

## 3. 探索式测试

### 通俗释义

不按预设的测试用例来，边测边想、边想边学、边学边测。就像玩一个游戏——你在探索这个系统，看看有哪些"有趣的"行为。探索式测试是"即兴创作"，但不是"随便点点"。

### 详细解释

#### 探索式测试 vs 脚本化测试

| 对比维度 | 脚本化测试 | 探索式测试 |
|----------|-----------|-----------|
| 测试执行 | 按预先设计的步骤 | 边测试边设计下一步 |
| 测试设计 | 在执行之前完成 | 与执行同时进行 |
| 灵活性 | 低（固定脚本） | 高（随时调整方向） |
| 可重复性 | 高 | 低（需记录路径） |
| 文档产出 | 测试用例、执行记录 | 测试笔记、录屏 |
| 适合场景 | 回归测试、验收测试 | 新功能探索、复杂场景挖掘 |
| 发现Bug类型 | 预期的Bug | 非预期的、复杂的Bug |

#### 基于Session的探索式测试（SBTM）

SBTM是探索式测试的结构化管理方法，核心单元是"Session（会话）"：

**Session的五要素：**

| 要素 | 说明 | 示例 |
|------|------|------|
| **Charter（宪章）** | 本次探索的目标和范围 | "探索购物车在添加/删除/修改商品时的数量和金额变化" |
| **Time Box（时间盒）** | 固定的探索时间 | 90分钟（不可中断） |
| **Reviewable Result（可评审的结果）** | Session结束后交付的记录 | 测试笔记、发现的问题、覆盖的路径 |
| **Debriefing（汇报）** | 与团队分享发现 | Session结束后5-10分钟口头汇报 |

**SBTM Session报告模板：**

```
Session #: 2024-001
Charter: 探索登录功能的密码重置流程
时间: 90分钟
日期: 2024-06-15

# 测试覆盖
- 正常重置密码流程（邮箱验证码方式）
- 验证码过期后重新获取
- 验证码输入错误3次后的锁定
- 重置密码后新旧密码均无法登录（预期行为）
- 网络中断时重置密码

# 发现的问题
Bug-1: 验证码输入错误5次后账号未锁定（需求说3次）
Bug-2: 重置密码成功后收到2封通知邮件
Bug-3: 在弱网下点击"获取验证码"按钮无响应，也无超时提示

# 测试数据
- 使用的账号: test_001@example.com
- 使用的验证码: 模拟过期、模拟错误

# 时间分配
- Charter设计: 5分钟
- 测试执行: 70分钟
- Bug记录: 10分钟
- 总结: 5分钟

# 后续建议
- 建议对密码重置进行专项安全测试
- 验证码相关的并发场景需要补充测试
```

#### 探索式测试的常用方法（测试漫游）

| 漫游方法 | 策略描述 | 适用场景 |
|----------|----------|----------|
| **地标漫游** | 先识别系统的关键功能点，再逐一探索 | 熟悉新系统时 |
| **极限漫游** | 挑战系统的极限值 | 测试边界和约束 |
| **强迫症漫游** | 重复执行同一个操作多次 | 测试资源泄漏、计数错误 |
| **出租车漫游** | 不按最佳路径，尝试"绕路"操作 | 测试非典型使用路径 |
| **深夜漫游** | 在系统维护/后台任务运行时测试 | 测试定时任务、数据清理 |
| **破坏者漫游** | 故意执行非法操作 | 测试异常处理和容错能力 |
| **博物馆漫游** | 测试旧版本/遗留功能 | 回归遗留功能 |
| **组合漫游** | 将两个看似无关的功能组合使用 | 测试功能间相互干扰 |

## 4. 上下文驱动测试

### 通俗释义

没有"最佳实践"，只有"在当前的上下文中最合适的做法"。上下文驱动测试（Context-Driven Testing, CDT）强调测试策略必须根据项目的具体情况来制定。

### 详细解释

#### CDT的7大核心原则

| 原则 | 解释 |
|------|------|
| **1. 任何实践的价值取决于其上下文** | 同样的方法在项目A可行，项目B可能灾难 |
| **2. 在特定上下文中有好的实践，但没有最佳实践** | "最佳实践"是伪概念，只有"适合这个场景的好实践" |
| **3. 人，在一起工作，是项目最重要的部分** | 工具和流程服务于人，而非相反 |
| **4. 项目会随时间产生不可预测的结果** | 计划无法覆盖所有意外，测试要随时适应变化 |
| **5. 产品是一个解决方案，如果问题没解决，产品就不行** | 测试关注的是"产品是否解决了用户的问题" |
| **6. 好的软件测试是一个具有挑战性的智力过程** | 测试不是机械操作，是智力密集型活动 |
| **7. 只有通过整个项目过程中收集的判断和技能，我们才能在正确的时间做正确的事** | 依赖专业的判断力，而非机械执行流程 |

#### 上下文驱动 vs 传统过程驱动

| 维度 | 过程驱动（传统） | 上下文驱动（CDT） |
|------|-----------------|-------------------|
| 对标准的态度 | 遵循行业标准/最佳实践 | 根据上下文裁剪标准 |
| 测试方法选择 | 按流程规定的方法 | 根据项目需要选择方法 |
| 文档 | 完整详细的文档 | 够用即可，不过度文档化 |
| 测试人员角色 | 按规定的步骤执行测试 | 运用专业判断设计测试策略 |
| 对变化的态度 | 变化是计划的敌人 | 变化是常态，拥抱变化 |
| 成功定义 | 按计划完成测试 | 发现了关键信息并帮助决策 |

#### 上下文的构成要素

```
上下文的维度：
┌─────────────────────────────────────────────┐
│ 项目上下文                                    │
│ ├─ 项目类型（Web/移动/嵌入式/IoT/AI）         │
│ ├─ 开发生命周期（传统/敏捷/DevOps）           │
│ ├─ 团队规模与结构                             │
│ ├─ 时间与预算约束                             │
│ ├─ 技术和工具栈                               │
│ └─ 组织文化与流程成熟度                       │
│                                              │
│ 产品上下文                                    │
│ ├─ 产品类型（ToC/ToB/内部工具）               │
│ ├─ 风险容忍度                                 │
│ ├─ 用户画像与使用场景                         │
│ ├─ 质量要求（安全关键/商业/娱乐）              │
│ └─ 合规性要求                                 │
│                                              │
│ 利益相关者上下文                              │
│ ├─ 管理层对质量的态度                         │
│ ├─ 客户/用户对Bug的容忍度                     │
│ ├─ 竞争态势（市场对质量的压力）               │
│ └─ 团队士气与工作状态                         │
└─────────────────────────────────────────────┘
```

## 5. 测试驱动开发与行为驱动开发

### 5.1 测试驱动开发（TDD）

#### 通俗释义

先写测试，再写代码。这个"反直觉"的做法能让你先想清楚"代码应该怎么用"，再动手实现。TDD的核心循环：Red（写一个失败的测试）→ Green（写最少的代码使其通过）→ Refactor（重构优化）。

#### TDD核心循环

```
     ┌─────────────────┐
     │  Red（红色）      │
     │  写一个失败的测试  │
     └────────┬────────┘
              │
              ▼
     ┌─────────────────┐
     │  Green（绿色）    │
     │  写最少代码让测试通过│
     └────────┬────────┘
              │
              ▼
     ┌─────────────────┐
     │  Refactor（重构） │
     │  优化代码结构      │
     └────────┬────────┘
              │
              ▼
         (循环继续)
```

#### TDD的三大法则

| 法则 | 说明 |
|------|------|
| **法则1** | 除非是为了让失败的测试通过，否则不能写任何产品代码 |
| **法则2** | 只写刚好导致失败的测试（编译失败也算失败） |
| **法则3** | 只写刚好让失败的测试通过的产品代码 |

#### TDD的价值与误区

| 价值 | 误区 |
|------|------|
| 代码天然可测试（因为测试先于代码） | "TDD可以替代所有测试"——不行，还要集成测试等 |
| 减少过度设计（只写刚好让测试通过的代码） | "TDD 100%覆盖就没有Bug"——逻辑错误仍可能存在 |
| 快速反馈（几秒内知道代码是否正确） | "TDD会拖慢开发速度"——短期可能慢，但Bug修复时间的节省更大 |
| 重构有安全感（测试兜底） | "每个方法都要写测试"——只测公共行为，不测实现细节 |

---

### 5.2 行为驱动开发（BDD）

#### 通俗释义

BDD是TDD的"升级版"：用所有人都能看懂的自然语言（Given/When/Then）描述系统行为，让产品、开发、测试在同一套"活文档"上协作。

#### BDD的核心语法：Gherkin

```gherkin
功能: 用户登录

  场景: 使用正确的账号密码登录
    假设 用户在登录页面
    当 用户输入正确的邮箱 "test@example.com"
    并且 用户输入正确的密码 "Password@123"
    并且 用户点击"登录"按钮
    那么 页面跳转到首页
    并且 页面右上角显示用户名 "Test User"

  场景: 使用错误的密码登录
    假设 用户在登录页面
    当 用户输入正确的邮箱 "test@example.com"
    并且 用户输入错误的密码 "WrongPass"
    并且 用户点击"登录"按钮
    那么 页面显示错误提示 "邮箱或密码错误"
    并且 用户仍然停留在登录页面

  场景大纲: 密码格式验证
    假设 用户在登录页面
    当 用户输入邮箱 "test@example.com"
    并且 用户输入密码 "<密码>"
    并且 用户点击"登录"按钮
    那么 页面显示错误提示 "<错误信息>"
    
    例子:
      | 密码       | 错误信息           |
      | ""         | 密码不能为空        |
      | "123"      | 密码长度不能少于8位  |
      | "password" | 密码必须包含大写字母  |
```

#### TDD vs BDD vs ATDD

| 维度 | TDD | BDD | ATDD |
|------|-----|-----|------|
| 关注层级 | 代码层（单元） | 行为层（功能） | 业务层（验收） |
| 编写者 | 开发者 | 产品+开发+测试 | 产品+测试+用户代表 |
| 语言 | 编程语言 | 自然语言(Gherkin) | 业务语言 |
| 粒度 | 细（函数/方法） | 中（功能/场景） | 粗（业务流程） |
| 受众 | 开发者 | 全团队 | 业务方+全团队 |

---

## 6. 关键字驱动测试

### 通俗释义

把测试操作包装成"关键字"（如"打开浏览器"、"输入文本"、"点击按钮"），测试用例就是关键字的组合。不懂编程的人也能看懂和编写测试用例。

### 详细解释

#### 关键字驱动的层次结构

```
第3层：业务关键字
  ┌─────────────────────────┐
  │ "用户登录成功"            │
  │ "添加商品到购物车"        │
  │ "完成支付流程"            │
  └───────────┬─────────────┘
              │ 由多个第2层关键字组合
第2层：功能关键字
  ┌─────────────────────────┐
  │ "输入账号" "输入密码"     │
  │ "点击登录按钮"            │
  │ "验证页面包含xxx文本"     │
  └───────────┬─────────────┘
              │ 调用底层操作
第1层：操作关键字
  ┌─────────────────────────┐
  │ "Click" "Type" "Wait"   │
  │ "Verify" "Select"       │
  └─────────────────────────┘
```

#### 关键字驱动测试用例示例（表格形式）

| 序号 | 关键字 | 参数1 | 参数2 | 参数3 |
|------|--------|-------|-------|-------|
| 1 | OpenBrowser | Chrome | https://test.example.com | — |
| 2 | Click | 登录按钮 | — | — |
| 3 | InputText | 邮箱输入框 | test@example.com | — |
| 4 | InputText | 密码输入框 | Password@123 | — |
| 5 | Click | 登录提交按钮 | — | — |
| 6 | VerifyText | 页面标题 | 我的首页 | — |
| 7 | VerifyElement | 用户名显示 | TestUser | — |
| 8 | CloseBrowser | — | — | — |

#### 关键字驱动 vs 数据驱动

| 维度 | 关键字驱动 | 数据驱动 |
|------|-----------|----------|
| 核心变量 | 操作（做什么） | 数据（输入什么） |
| 用例结构 | 关键字序列 | 同一流程+不同数据 |
| 复用粒度 | 操作级别 | 脚本级别 |
| 可读性 | 高（接近自然语言） | 中（需要看数据表） |
| 维护性 | 关键字维护成本较高 | 数据维护成本低 |
| 适合人员 | 业务人员和测试人员 | 测试人员和自动化工程师 |

## 7. 数据驱动测试

### 通俗释义

把测试数据和测试逻辑分离：写一套测试脚本，用不同的数据去跑。Excel表/CSV/JSON里的一行数据 = 一个测试用例。

### 详细解释

#### 数据驱动测试的架构

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  数据源       │    │  测试脚本     │    │  测试结果     │
│  Excel/CSV   │───→│  (一套逻辑)   │───→│  逐条记录     │
│  JSON/XML    │    │               │    │  通过/失败    │
│  数据库      │    │ 读取数据      │    │  详细日志     │
│              │    │ 执行操作      │    │              │
│              │    │ 比对结果      │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
```

#### 数据驱动测试用例设计（登录功能）

**数据文件（CSV）：**

```csv
用例ID, 邮箱, 密码, 期望结果, 期望消息
TC-LOGIN-001, test@example.com, Password@123, 成功, 登录成功
TC-LOGIN-002, test@example.com, WrongPass, 失败, 邮箱或密码错误
TC-LOGIN-003, "", Password@123, 失败, 邮箱不能为空
TC-LOGIN-004, test@example.com, "", 失败, 密码不能为空
TC-LOGIN-005, notexist@example.com, Password@123, 失败, 邮箱未注册
TC-LOGIN-006, test@example.com, 123, 失败, 密码长度不能少于8位
TC-LOGIN-007, test@, Password@123, 失败, 邮箱格式不正确
TC-LOGIN-008, test@example.com, Password@12345678901234567890, 失败, 密码长度不能超过20位
```

**一条脚本，8个用例自动执行：**
1. 读取第1行数据 → 执行登录 → 比对结果 → 记录通过/失败
2. 读取第2行数据 → 执行登录 → 比对结果 → 记录通过/失败
3. ...

#### 数据驱动测试的优缺点

| 优点 | 缺点 |
|------|------|
| 新增用例只需添加数据，无需改脚本 | 测试脚本需要足够的灵活性处理各种数据 |
| 数据与逻辑分离，降低维护成本 | 复杂场景的数据准备可能很繁琐 |
| 非技术人员可以补充测试数据 | 只能覆盖流程固定的场景 |
| 大幅减少重复代码 | 数据文件本身需要维护 |
| 方便参数化和批量测试 | 一条数据出错可能导致后续数据未执行 |

## 8. 5种方法论的实际应用案例

---

### 案例1：基于风险的测试——银行核心系统升级（方法论：基于风险的测试）

**背景：** 某银行核心系统升级，涉及存款、贷款、支付、风控4个核心模块。测试周期仅45天，无法对所有功能进行深度测试。

**风险评估过程：**

| 模块 | 复杂度 | 变更量 | 历史Bug | 业务影响 | 风险等级 |
|------|:------:|:------:|:-------:|:-------:|:--------:|
| 存款计算引擎 | 极高 | 大 | 多 | 极高 | **极高** |
| 贷款审批流程 | 高 | 中 | 中 | 高 | **高** |
| 支付网关 | 高 | 大 | 多 | 极高 | **极高** |
| 风控规则引擎 | 极高 | 大 | 中 | 高 | **高** |
| 账户管理 | 中 | 小 | 少 | 中 | **中** |
| 报表统计 | 低 | 小 | 少 | 低 | **低** |

**测试资源分配：**

| 风险等级 | 测试深度 | 用例比例 | 自动化要求 |
|----------|----------|----------|-----------|
| 极高（存款+支付） | 全覆盖测试（3轮+） | 60%的用例 | 100%自动化回归 |
| 高（贷款+风控） | 深度测试（2轮+） | 25%的用例 | 核心场景自动化 |
| 中（账户） | 常规测试（1轮+） | 10%的用例 | BVT自动化 |
| 低（报表） | 轻量测试（BVT） | 5%的用例 | 基本不自动化 |

**结果：**
- 在45天内完成了基于风险优先级的测试
- 发现P0级Bug 12个，全部在极高/高风险模块
- 中低风险模块0个P0/P1 Bug → 验证了风险判断的准确性
- 上线后6个月零生产事故

### 案例2：基于模型的测试——航空订票系统（方法论：基于模型的测试）

**背景：** 某航空订票系统，机票状态和流程极其复杂（预订→支付→出票→改签→退票→取消等）。

**建模过程：**

```
状态机模型（简化版）：

机票生命周期：
[搜索] → [预订(锁定)] → [支付中] → [已出票] → [飞行中] → [已使用]
                ↓           ↓          ↓
            [已取消]    [支付失败]   [改签中] → [已改签]
                            ↓          ↓
                        [已取消]   [退票中] → [已退票]
```

**模型效果：**

| 指标 | 传统设计 | 基于模型 |
|------|----------|----------|
| 测试用例数量 | ~300（手工设计） | ~1200（自动生成） |
| 用例设计时间 | 2周 | 3天（建模+审核） |
| 状态组合覆盖率 | ~60%（人工难穷举） | ~95%（模型保证） |
| 发现问题 | 23个Bug | 41个Bug（多发现18个） |

**关键发现：**
- 模型自动生成了"改签后退票"（人工容易遗漏的边缘路径）的测试用例
- 发现了改签后退票时，燃油费计算错误的Bug
- 该Bug在传统手工测试中未被覆盖

### 案例3：探索式测试——电商APP新功能验证（方法论：探索式测试SBTM）

**背景：** 某电商APP上线"直播带货"功能，团队需要在3天内完成验证。没有时间编写完整的测试用例。

**SBTM方案：**

| Session | Charter | 时间 | 测试人员 |
|---------|---------|------|----------|
| S1 | 探索主播开播→讲解→上架商品→下播全流程 | 90min | 测试A |
| S2 | 用户端进入直播间→互动→加购→下单 | 90min | 测试B |
| S3 | 极端场景：弱网/断网/来电中断/切换App | 60min | 测试A |
| S4 | 并发场景：多人同时加购、千人观看 | 90min | 测试B |
| S5 | 支付链路：直播间下单→支付→订单状态同步 | 60min | 测试A |
| S6 | 兼容性：不同手机型号+系统版本 | 90min | 测试B |
| S7 | 自由探索（基于前6个Session的发现） | 60min | 两人 |
| S8 | Bug验证+回归测试 | 60min | 两人 |

**结果（3天共10小时探索时间）：**
- 发现Bug 23个（P0:3个, P1:8个, P2:12个）
- 关键Bug：弱网下直播间画面卡顿但音画不同步（P0）
- 关键Bug：用户加购商品后在直播间外删除，再次进入直播间购物车状态不一致（P0）
- 如果使用传统脚本化测试，预计只能发现大约10个Bug

### 案例4：BDD实践——保险理赔系统（方法论：BDD + TDD）

**背景：** 某保险公司开发新理赔系统，业务规则极其复杂（不同险种、不同理赔类型、不同金额阶梯）。

**BDD实践：**

**Step 1：业务方+开发+测试一起写Feature文件**

```gherkin
功能: 车险理赔审批

  场景大纲: 根据理赔金额自动审批
    假设 用户提交了车险理赔申请
    并且 理赔类型为 "<理赔类型>"
    并且 理赔金额为 <金额> 元
    当 系统执行自动审批
    那么 审批结果为 "<审批结果>"
    
    例子:
      | 理赔类型 | 金额   | 审批结果 |
      | 刮擦维修 | 500    | 自动通过 |
      | 刮擦维修 | 3000   | 人工审核 |
      | 重大事故 | 2000   | 人工审核 |
      | 玻璃单独 | 800    | 自动通过 |
      | 全损     | 100000 | 专家审核 |
```

**Step 2：开发基于这些Scenario写自动化测试并开发功能（TDD方式）**

**Step 3：相同Scenario用于手工测试和自动化回归**

**效果：**
- 大大减少了需求理解偏差（同一个Scenario被产品、开发、测试共享）
- Scenario数量：237个场景覆盖6种险种
- 自动化率：95%（237个场景中225个自动化）
- 沟通成本降低：需求澄清时间从3周降到1周

### 案例5：数据驱动+关键字驱动混合——ERP系统测试（方法论：混合驱动）

**背景：** 某ERP系统有500+个功能页面，涉及大量CRUD操作。测试脚本的维护成本成为瓶颈。

**混合驱动方案：**

**第1层（数据层）：** Excel管理测试数据

| 用例ID | 模块 | 功能 | 输入数据 | 期望结果 |
|--------|------|------|----------|----------|
| ERP-001 | 供应商 | 新增 | 供应商名:ABC公司, 联系人:张三 | 新增成功, 列表中可查 |
| ERP-002 | 供应商 | 新增 | 供应商名:"" | 提示"供应商名不能为空" |
| ERP-003 | 供应商 | 修改 | 原名称→新名称 | 修改成功 |
| ... | ... | ... | ... | ... |

**第2层（关键字层）：** 封装通用操作

```
关键字：新增记录
  1. 点击"新增"按钮
  2. 遍历填入表单数据
  3. 点击"保存"
  4. 验证结果

关键字：查询记录
  1. 输入查询条件
  2. 点击"查询"
  3. 验证查询结果数量和内容

关键字：修改记录
  1. 查询到目标记录
  2. 点击"编辑"
  3. 修改字段
  4. 保存并验证
```

**效果：**
- 新增测试场景只需在Excel中添加一行（无需编写代码）
- 关键字库从0到50个，支撑了500+功能页面
- 测试数据维护交给业务分析师，测试人员专注于关键字和脚本优化
- 自动化用例从200增长到1800+（仅用了3个月）

## 总结

1. **基于风险的测试**是资源受限时的最优策略——把有限的测试资源用在"刀刃"上
2. **基于模型的测试**通过建模自动生成用例——适合复杂的状态流转和规则组合场景
3. **探索式测试**是脚本化测试的重要补充——适合新功能验证和复杂Bug的挖掘
4. **上下文驱动测试**提醒我们——没有银弹，根据项目具体情况选择合适的方法
5. **TDD和BDD**将测试从"后端"移到"前端"——提升代码质量和团队沟通效率
6. **关键字驱动和数据驱动**是自动化测试的两种经典模式——降低门槛、提升复用
7. **方法论的选择取决于上下文**——可以混合使用多种方法论来应对不同的测试挑战
8. **结构化探索最有效**——SBTM让探索式测试既不失控又有产出
