# 单元测试 (Unit Testing)

> **单元测试**（Unit Testing）：对软件中最小的可测试单元（通常是函数、方法或类）进行正确性验证的测试方法。它是测试金字塔的基座，是保证代码质量的第一道防线。

## 1. 单元测试的定义与价值

### 通俗释义

单元测试就是开发写完一个函数、一个方法或一个类之后，写一段测试代码来验证这段业务逻辑在各种输入下是否输出预期结果。它不依赖数据库、不依赖网络、不依赖其他服务，只测"纯代码逻辑"。

### 详细解释

单元测试（Unit Testing）是软件测试中最基础的层级，它将软件分解为最小的可测试单元（Unit），并对每个单元进行独立验证。在面向对象编程中，一个"单元"通常是一个类的方法；在函数式编程中，一个"单元"通常是一个函数；在过程式编程中，一个"单元"通常是一个过程/子程序。

**单元测试的核心特征：**

| 特征 | 说明 |
|------|------|
| **隔离性** | 每个单元测试独立运行，不依赖外部资源（数据库、网络、文件系统等） |
| **快速性** | 单个单元测试应在毫秒级完成，整个单元测试套件应在几分钟内完成 |
| **自动化** | 单元测试应能在CI/CD流水线中自动执行，无需人工干预 |
| **可重复** | 同一测试在任何时间、任何环境、任何执行顺序下结果一致 |
| **白盒性** | 测试人员（通常是开发者）了解被测试单元的内部实现 |

**单元测试的价值矩阵：**

| 价值维度 | 具体收益 |
|----------|----------|
| **缺陷预防** | 在编码阶段即发现逻辑错误，修复成本是集成测试阶段的1/10，是生产环境的1/100 |
| **回归保护** | 修改代码后快速验证是否破坏了已有功能，给重构以信心 |
| **设计反馈** | 难以编写单元测试的代码往往存在设计问题（耦合过紧、职责不清） |
| **活文档** | 单元测试本身就是代码行为的最佳文档，描述函数的输入输出契约 |
| **开发效率** | 减少调试时间——测试失败时可以直接定位到具体函数，而非在全链路中排查 |
| **团队协作** | 新成员可以通过阅读和运行单元测试快速理解模块行为 |

**单元测试 vs 其他测试层级的定位：**

```
          ┌─────────────┐
          │  E2E / UAT  │  ← 少量：验证核心业务流程
          ├─────────────┤
          │  集成测试    │  ← 适量：验证模块间交互
          ├─────────────┤
          │  单元测试    │  ← 大量：验证每个函数/方法的正确性
          └─────────────┘
```

## 2. 单元测试框架

### JUnit (Java)

JUnit 是 Java 生态中最广泛使用的单元测试框架，当前主流版本为 JUnit 5 (Jupiter)。

**核心注解：**

| 注解 | 说明 | JUnit版本 |
|------|------|-----------|
| `@Test` | 标记测试方法 | JUnit 4/5 |
| `@BeforeEach` / `@Before` | 每个测试方法执行前运行 | JUnit5 / JUnit4 |
| `@AfterEach` / `@After` | 每个测试方法执行后运行 | JUnit5 / JUnit4 |
| `@BeforeAll` / `@BeforeClass` | 所有测试方法执行前运行一次（需为static） | JUnit5 / JUnit4 |
| `@AfterAll` / `@AfterClass` | 所有测试方法执行后运行一次（需为static） | JUnit5 / JUnit4 |
| `@DisplayName` | 为测试方法指定可读性更好的名称 | JUnit5 |
| `@ParameterizedTest` | 参数化测试，用不同参数多次运行同一测试 | JUnit5 |
| `@ValueSource` / `@CsvSource` / `@MethodSource` | 为参数化测试提供数据源 | JUnit5 |
| `@Disabled` / `@Ignore` | 临时禁用测试方法 | JUnit5 / JUnit4 |
| `@Nested` | 创建嵌套测试类，表达测试层级关系 | JUnit5 |
| `@Tag` | 为测试打标签，便于分类执行 | JUnit5 |

**JUnit 5 断言示例：**

```java
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
    
    @BeforeEach
    void setUp() {
        // 每个测试前的初始化
    }
    
    @Test
    @DisplayName("两数相加应返回正确和")
    void add_TwoPositiveNumbers_ReturnsSum() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));
        assertNotEquals(6, calc.add(2, 3));
    }
    
    @Test
    @DisplayName("除数为零时应抛出异常")
    void divide_ByZero_ThrowsException() {
        Calculator calc = new Calculator();
        assertThrows(ArithmeticException.class, 
            () -> calc.divide(10, 0));
    }
    
    @ParameterizedTest
    @CsvSource({"2,3,5", "0,0,0", "-1,1,0", "100,200,300"})
    @DisplayName("参数化加法测试")
    void add_Parameterized(int a, int b, int expected) {
        Calculator calc = new Calculator();
        assertEquals(expected, calc.add(a, b));
    }
}
```

### TestNG (Java)

TestNG 是受 JUnit 和 NUnit 启发的测试框架，在配置灵活性和高级特性方面提供了更多选择。

**TestNG vs JUnit 对比：**

| 特性 | TestNG | JUnit 5 |
|------|--------|---------|
| 测试依赖 (`dependsOnMethods`) | 内置支持 | 需第三方实现 |
| 测试分组 (`groups`) | 内置支持 | `@Tag` 替代 |
| 并行执行 | XML配置即可 | 需编程式配置 |
| 参数化 | `@DataProvider` / XML | `@ParameterizedTest` + Source |
| 测试优先级 (`priority`) | 内置支持 | `@Order` 替代 |
| 监听器机制 | 丰富的内置监听器 | 扩展模型 (Extensions) |

### pytest (Python)

pytest 是 Python 生态中最流行的测试框架，以简洁著称。测试函数名以 `test_` 开头，文件名以 `test_` 开头或 `_test` 结尾。

**核心特性：**

| 特性 | 说明 |
|------|------|
| **fixture** | 比 setUp/tearDown 更灵活的依赖注入机制，支持 scope(function/class/module/session) |
| **conftest.py** | 共享 fixture 的配置文件，按目录层级自动继承 |
| **mark** | 标记测试（如 `@pytest.mark.slow`, `@pytest.mark.skip`），用于分类执行 |
| **parametrize** | 参数化测试，语法简洁：`@pytest.mark.parametrize("a,b,expected", [(2,3,5), (0,0,0)])` |
| **插件体系** | 丰富的第三方插件：pytest-cov（覆盖率）、pytest-xdist（并行）、pytest-mock、pytest-django |
| **assert 重写** | 直接用 Python `assert`，失败时 pytest 会显示详细的差异对比 |

**pytest 示例：**

```python
import pytest

# conftest.py 中的 fixture
@pytest.fixture
def sample_calculator():
    """创建一个计算器实例供测试使用"""
    return Calculator()

# 测试文件 test_calculator.py
class TestCalculator:
    
    def test_add_positive_numbers(self, sample_calculator):
        result = sample_calculator.add(2, 3)
        assert result == 5
        assert isinstance(result, int)
    
    def test_divide_by_zero(self, sample_calculator):
        with pytest.raises(ZeroDivisionError, match="division by zero"):
            sample_calculator.divide(10, 0)
    
    @pytest.mark.parametrize("a,b,expected", [
        (2, 3, 5),
        (0, 0, 0),
        (-1, 1, 0),
        (100, 200, 300),
    ])
    def test_add_parametrized(self, sample_calculator, a, b, expected):
        assert sample_calculator.add(a, b) == expected
    
    @pytest.mark.slow
    def test_large_computation(self, sample_calculator):
        # 标记为慢速测试，日常开发中可跳过
        result = sample_calculator.fibonacci(30)
        assert result == 832040
```

### Jest (JavaScript/TypeScript)

Jest 是 Facebook 开源的 JavaScript 测试框架，内置断言、Mock、覆盖率等功能，零配置即可使用。

**核心特性：**

| 特性 | 说明 |
|------|------|
| **零配置** | 开箱即用，默认支持 Babel/TypeScript |
| **快照测试** | `toMatchSnapshot()` 对比UI组件的渲染输出 |
| **内置Mock** | `jest.fn()` / `jest.mock()` 强大的模块/函数Mock |
| **并行执行** | 默认多进程并行运行测试文件（--maxWorkers 控制） |
| **覆盖率** | 内置 `--coverage` 生成 Istanbul 覆盖率报告 |
| **Watch模式** | `--watch` 仅重新运行变更相关的测试 |

**Jest 示例：**

```javascript
// calculator.test.js
const Calculator = require('./calculator');

describe('Calculator', () => {
    let calc;
    
    beforeEach(() => {
        calc = new Calculator();
    });
    
    describe('add', () => {
        test('两正数相加应返回正确和', () => {
            expect(calc.add(2, 3)).toBe(5);
        });
        
        test('正负数混合相加', () => {
            expect(calc.add(-1, 5)).toBe(4);
        });
        
        test.each([
            [2, 3, 5],
            [0, 0, 0],
            [-1, 1, 0],
            [100, 200, 300],
        ])('add(%i, %i) 应等于 %i', (a, b, expected) => {
            expect(calc.add(a, b)).toBe(expected);
        });
    });
    
    describe('divide', () => {
        test('除数为零时应抛出错误', () => {
            expect(() => calc.divide(10, 0)).toThrow('Division by zero');
        });
    });
});
```

### Go Testing (Go)

Go 语言内置 `testing` 包，测试文件以 `_test.go` 结尾，函数签名固定为 `func TestXxx(t *testing.T)`。

**核心特性：**

| 特性 | 说明 |
|------|------|
| **表驱动测试** | Go社区推崇的测试风格：将测试用例组织为结构体切片，循环执行 |
| **子测试 (t.Run)** | 支持嵌套测试，每个子测试可独立运行 |
| **TestMain** | 全局测试入口，用于测试前的全局设置和测试后的清理 |
| **覆盖率** | `go test -cover` 原生支持覆盖率统计 |
| **基准测试** | `func BenchmarkXxx(b *testing.B)` 原生性能基准测试 |
| **模糊测试** | Go 1.18+ 原生支持 Fuzz Testing (`func FuzzXxx(f *testing.F)`) |

**Go 表驱动测试示例：**

```go
func TestAdd(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"两正数相加", 2, 3, 5},
        {"零值相加", 0, 0, 0},
        {"正负数相加", -1, 1, 0},
        {"大数相加", 100, 200, 300},
    }
    
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result := Add(tt.a, tt.b)
            if result != tt.expected {
                t.Errorf("Add(%d, %d) = %d; want %d", 
                    tt.a, tt.b, result, tt.expected)
            }
        })
    }
}
```

## 3. 测试替身

测试替身（Test Double）是在单元测试中用于替代真实依赖对象的技术统称。当被测单元依赖外部组件（数据库、网络服务、文件系统、其他类等）时，使用测试替身可以保持测试的隔离性和快速性。

**五种测试替身对比：**

| 类型 | 英文 | 是否记录调用 | 是否有返回值 | 核心作用 |
|------|------|-------------|-------------|----------|
| **Dummy** | 哑对象 | 否 | 无意义返回值 | 仅用于填充参数列表，被测代码不调用它 |
| **Stub** | 桩 | 否 | 返回预设值 | 提供固定的间接输入 |
| **Spy** | 间谍 | 是 | 可调用真实方法或返回预设值 | 记录调用信息，用于事后验证 |
| **Mock** | 模拟 | 是（自动验证） | 返回预设值 | 预设调用期望，自动验证调用是否正确 |
| **Fake** | 伪造 | 否 | 有真实逻辑 | 用轻量级实现替代重依赖（如内存数据库替代MySQL） |

### Dummy（哑对象）

最简单的替身，仅用于满足参数传递，实际在测试中不会被使用。

```java
// Dummy 示例：测试 UserService 时需要传入一个不会被调用的 Logger
@Test
void updateUserName_ValidName_UpdatesSuccessfully() {
    Logger dummyLogger = new Logger() {
        @Override public void log(String msg) { /* 不会被调用 */ }
    };
    UserService service = new UserService(dummyLogger, userRepo);
    // ... 测试逻辑中不会用到 dummyLogger
}
```

### Stub（桩）

为被测代码提供预设的间接输入值，不记录调用过程。

```java
// Stub 示例：固定返回一个用户列表
@Test
void getActiveUsers_ReturnsOnlyActiveUsers() {
    UserRepository stubRepo = new UserRepository() {
        @Override
        public List<User> findAll() {
            return Arrays.asList(
                new User("1", "Alice", true),
                new User("2", "Bob", false),
                new User("3", "Charlie", true)
            );
        }
    };
    UserService service = new UserService(stubRepo);
    List<User> result = service.getActiveUsers();
    assertEquals(2, result.size());
}
```

### Spy（间谍）

记录被调用的情况（调用次数、传入参数），测试后验证调用行为是否正确。

```java
// Spy 示例：验证邮件发送服务被正确调用
@Test
void registerUser_SendsWelcomeEmail() {
    EmailServiceSpy spyEmail = new EmailServiceSpy();
    UserService service = new UserService(userRepo, spyEmail);
    
    service.register(new User("alice@example.com", "Alice"));
    
    assertEquals(1, spyEmail.getSendCount());
    assertEquals("alice@example.com", spyEmail.getLastRecipient());
    assertTrue(spyEmail.getLastSubject().contains("欢迎"));
}
```

### Mock（模拟）

预设期望值并在测试结束时自动验证。如果调用不匹配预期（次数不对、参数不对、方法未调用等），测试自动失败。

```java
// Mockito Mock 示例
@Test
void placeOrder_WhenPaymentSucceeds_OrderStatusConfirmed() {
    // 创建 Mock
    PaymentGateway mockPayment = mock(PaymentGateway.class);
    InventoryService mockInventory = mock(InventoryService.class);
    
    // 预设行为
    when(mockPayment.charge(anyString(), eq(new BigDecimal("99.99"))))
        .thenReturn(new PaymentResult(true, "txn_123"));
    when(mockInventory.isInStock("SKU_001")).thenReturn(true);
    
    OrderService service = new OrderService(mockPayment, mockInventory);
    Order order = service.placeOrder("SKU_001", 1, new BigDecimal("99.99"));
    
    // 验证结果
    assertEquals(OrderStatus.CONFIRMED, order.getStatus());
    
    // 自动验证行为
    verify(mockPayment, times(1)).charge(anyString(), any(BigDecimal.class));
    verify(mockInventory, times(1)).deduct("SKU_001", 1);
    verify(mockInventory, never()).refund(anyString()); // 不应调用退款
}
```

### Fake（伪造）

用轻量级替代实现来替换重量级依赖，拥有真实的工作逻辑。典型例子：用内存数据库（如H2）替代MySQL进行单元测试。

```java
// Fake 示例：用内存Map替代真实数据库
class FakeUserRepository implements UserRepository {
    private Map<String, User> storage = new HashMap<>();
    
    @Override
    public User save(User user) {
        storage.put(user.getId(), user);
        return user;
    }
    
    @Override
    public Optional<User> findById(String id) {
        return Optional.ofNullable(storage.get(id));
    }
    
    // 完整实现接口的所有方法...
}

@Test
void updateProfile_SavesToRepository() {
    FakeUserRepository fakeRepo = new FakeUserRepository();
    fakeRepo.save(new User("1", "Alice", "alice@example.com")); // 预置数据
    
    UserService service = new UserService(fakeRepo);
    User updated = service.updateProfile("1", "Alice Updated");
    
    assertEquals("Alice Updated", fakeRepo.findById("1").get().getName());
}
```

### 测试替身选择指南

| 场景 | 推荐替身 | 原因 |
|------|----------|------|
| 被测代码完全不调用该依赖 | Dummy | 最轻量，仅满足参数要求 |
| 需要固定返回某些数据 | Stub | 简单直接，无需验证交互 |
| 需要验证"是否被调用"、"调用了几次"、"参数是什么" | Spy / Mock | 关注行为验证 |
| 依赖服务很重（数据库、消息队列） | Fake | 提供真实但轻量的替代实现 |
| 需要精确控制返回序列（第一次返回A，第二次返回B） | Mock | 编程式控制更方便 |

## 4. 单元测试覆盖率目标

### 覆盖率类型

| 覆盖率类型 | 计算方式 | 说明 |
|-----------|----------|------|
| **行覆盖率** (Line Coverage) | 已执行行数 / 总代码行数 | 最基础的度量，但一行可能含多个分支 |
| **分支覆盖率** (Branch Coverage) | 已覆盖分支数 / 总分支数 | 每个 if/else、switch case 是否都覆盖到 |
| **函数覆盖率** (Function Coverage) | 已调用函数数 / 总函数数 | 是否每个函数都有测试 |
| **条件覆盖率** (Condition Coverage) | 已覆盖条件组合数 / 总条件组合数 | 更严格：`if (a && b)` 需要测 (T,T)/(T,F)/(F,T)/(F,F) |
| **路径覆盖率** (Path Coverage) | 已覆盖执行路径数 / 总路径数 | 最严格但路径数可能爆炸性增长 |
| **变异覆盖率** (Mutation Coverage) | 杀死变异体数 / 总变异体数 | 测试质量度量：修改代码后测试能否发现 |

### 覆盖率目标建议

| 项目类型 | 建议行覆盖率 | 建议分支覆盖率 | 说明 |
|----------|-------------|---------------|------|
| **核心业务库/框架** | 90%+ | 85%+ | 被广泛依赖，缺陷影响面大 |
| **金融/支付系统** | 90%+ | 85%+ | 涉及资金，零容忍逻辑错误 |
| **普通业务服务** | 80%+ | 70%+ | 平衡投入产出比 |
| **前端UI组件** | 70%+ | — | UI测试成本高，关注交互逻辑 |
| **配置/常量类** | 30%+ | — | 纯数据定义，低风险 |
| **遗留系统** | 渐进式提升 | — | 不改不动，改动时补充测试 |

**覆盖率不是银弹：**

```
高覆盖率 ≠ 高质量测试

以下代码覆盖率100%但毫无意义：
@Test
void test() {
    service.doWork();
    // 没有进行任何断言！测试永远通过
}

更有价值的衡量标准：
- 断言的有效性：是否真正验证了业务逻辑？
- 边界条件覆盖：等价类 + 边界值是否充分？
- 错误路径覆盖：异常分支是否都被测试？
- 变异测试得分：修改代码后测试能否检测到？
```

## 5. TDD 测试驱动开发

### TDD 核心循环：红 - 绿 - 重构

TDD（Test-Driven Development）不是测试技术，而是**设计方法**。它要求开发者在写实现代码之前先写测试代码，通过测试来驱动设计。

```
  ┌──────────────────────────────────┐
  │                                  │
  ▼                                  │
┌────┐    ┌────┐    ┌────────┐     │
│ RED │───▶│GREEN│───▶│REFACTOR│─────┘
│ 红  │    │ 绿  │    │  重构   │
└────┘    └────┘    └────────┘
 写一个     写最少      优化代码
 失败的     代码让      而不改变
 测试      测试通过    外部行为
```

### 三个阶段详解

**第一阶段：RED（红）—— 写一个失败的测试**

- 基于需求/用户故事，写出一个最小、最具体的测试用例
- 测试代码本身也是"代码"，应该清晰表达意图
- 此阶段只关注"What"（做什么），不关注"How"（怎么做）
- 运行测试，确认它失败（红色）—— 如果不是红色，说明这个测试没有在测新功能

**第二阶段：GREEN（绿）—— 写最少的代码让测试通过**

- 只写刚好能让测试通过的代码，不过度设计
- 哪怕用硬编码、复制粘贴 —— 第一步是让测试绿
- 此阶段证明"需求已实现，无非更多"

**第三阶段：REFACTOR（重构）—— 优化代码而不改变行为**

- 消除重复代码、改善命名、提取公共逻辑
- 应用设计模式、优化算法
- 测试始终保持绿色 —— 重构的安全性由测试保障
- 此阶段将"能工作的代码"升级为"干净的代码"

### TDD 三定律（Uncle Bob）

| 定律 | 内容 |
|------|------|
| **第一定律** | 在编写不能通过的单元测试前，不可编写任何生产代码 |
| **第二定律** | 只编写恰好无法通过的单元测试（不能编译也是一种失败） |
| **第三定律** | 只编写恰好能使当前失败测试通过的生产代码 |

### TDD 的实际价值

| 价值 | 说明 |
|------|------|
| **驱动设计** | 写测试的过程强迫你思考接口设计——如何调用这个函数？参数和返回值是什么？ |
| **可测试性** | TDD天然产生高可测试性的代码（低耦合、依赖注入） |
| **减少调试** | 每次只改少量代码，出问题立刻知道哪个改动导致的 |
| **回归安全网** | 每次重构都有全部测试的保护 |
| **开发节奏** | 红-绿-重构的短循环（通常2-5分钟）保持高度专注 |

### TDD 常见误区

| 误区 | 正确理解 |
|------|----------|
| "TDD浪费时间" | 短期看多写了测试代码，但大幅减少了调试和后期bug修复时间 |
| "先写测试限制创造力" | 测试只定义行为契约，不限制内部实现方式 |
| "100% TDD" | 并非所有场景都适合（如UI布局、探索性编码），能做70%就很好 |
| "不需要设计就直接TDD" | TDD不能替代架构设计，宏观设计仍需提前规划 |

## 6. 优秀单元测试的特征 (FIRST原则)

### F - Fast（快速）

单元测试必须在**毫秒级**完成。如果单元测试需要访问数据库、网络、文件系统，它就不是真正的"单元"测试。整个单元测试套件应在**几分钟内**全部执行完毕。

```
反例：一个"单元测试"需要5秒（访问了数据库）
好例：通过Mock替代数据库，测试在10ms内完成
```

**实践建议：**
- 避免在单元测试中使用真实的数据库连接、HTTP调用、文件I/O
- 如果有少量不可避免的慢测试，将它们标记为集成测试并分离执行
- 在CI中设置单元测试超时（如每个测试方法500ms）

### I - Independent（独立）

每个测试必须**完全独立**于其他测试。测试A的结果不应影响测试B的执行。测试的执行顺序不应影响结果。

```java
// 反例：测试之间有状态共享
static int counter = 0; // 危险！测试B依赖测试A对counter的修改

// 好例：每个测试自给自足
@BeforeEach
void setUp() {
    counter = 0; // 每个测试前重置状态
}
```

**实践建议：**
- 每个测试自己创建所需的对象和数据
- 不依赖测试执行顺序（JUnit不保证执行顺序）
- 不在测试之间共享可变状态
- 使用 `@BeforeEach` 确保每个测试的初始状态一致

### R - Repeatable（可重复）

同一测试在任何时间、任何环境、任何执行次数下都必须产生**相同的结果**。不依赖外部环境（系统时间、时区、网络状态、随机数等）。

```java
// 反例：依赖系统时间
Order order = new Order(LocalDateTime.now()); // 每次运行时间不同

// 好例：注入可控的时间源
@Test
void orderHasCorrectCreationTime() {
    Clock fixedClock = Clock.fixed(Instant.parse("2025-06-15T10:00:00Z"), ZoneId.of("UTC"));
    Order order = new Order(fixedClock);
    assertEquals("2025-06-15T10:00:00Z", order.getCreatedAt().toString());
}
```

**实践建议：**
- 使用依赖注入传递时间源（`java.time.Clock`）
- 随机数的测试使用固定的种子
- 避免依赖环境变量、系统属性（如需要，在测试中显式设置）

### S - Self-Validating（自我验证）

测试必须自己判断通过还是失败，**不需要人工判断输出**。测试的结果必须是二元的：通过（绿）或失败（红）。

```java
// 反例：需要人工看日志输出
@Test
void processOrder() {
    orderService.process(order);
    System.out.println(order.getStatus()); // 需要人眼看输出对不对！
}

// 好例：通过断言自动验证
@Test
void processOrder_SufficientStock_StatusConfirmed() {
    orderService.process(order);
    assertEquals(OrderStatus.CONFIRMED, order.getStatus());
    assertEquals(9, inventoryService.getStock("SKU001")); // 库存从10减为9
}
```

### T - Timely（及时）

单元测试应该在**生产代码之前或同时**编写（TDD的理想），而不是在生产代码完成很久之后再补。及时编写的测试能驱动设计，事后补的测试往往是"为覆盖率而写"。

**实践建议：**
- 将写测试作为"完成"定义的一部分（DoD, Definition of Done）
- 在Code Review中同时审查测试代码
- CI流水线在代码合并时强制运行所有单元测试

## 7. 常见反模式

### 反模式1：测试没有断言（Invisible Assertion）

```java
// 反模式：测试永远通过，因为没有任何验证
@Test
void testCalculateDiscount() {
    discountService.calculate(new Order(100));
    // 没有assert —— 这个测试毫无意义
}

// 好例
@Test
void calculateDiscount_OrderOver200_Applies20Percent() {
    BigDecimal discount = discountService.calculate(new Order(300));
    assertEquals(new BigDecimal("60.00"), discount);
}
```

### 反模式2：一个测试验证多个行为（Multiple Assertions / Eager Test）

```java
// 反模式：一个测试方法测了N个独立行为，失败时难以定位问题
@Test
void testUserService() {
    User user = userService.create("Alice", "alice@example.com");
    assertNotNull(user.getId());          // 测创建
    assertEquals("Alice", user.getName()); // 测命名
    userService.updateEmail(user.getId(), "new@example.com");
    assertEquals("new@example.com", userService.findById(user.getId()).getEmail()); // 测更新
    userService.delete(user.getId());
    assertTrue(userService.findById(user.getId()).isEmpty()); // 测删除
}

// 好例：拆分为独立的测试方法
@Test void createUser_ValidInput_ReturnsUserWithId() { /* ... */ }
@Test void updateEmail_ValidEmail_UpdatesSuccessfully() { /* ... */ }
@Test void deleteUser_ExistingUser_RemovesFromRepository() { /* ... */ }
```

### 反模式3：测试依赖执行顺序（Test Order Dependency）

**现象：** 单独运行测试A失败，但按某个顺序运行就通过。  
**原因：** 测试之间通过共享状态互相依赖。  
**解决：** 每个测试独立创建所需状态，使用 `@BeforeEach` 清理。

### 反模式4：过度Mock（Mockery / Over-Mocking）

```java
// 反模式：Mock了所有依赖，但实际没测到任何真实逻辑
@Test
void testOrderService() {
    when(mockUserRepo.findById(any())).thenReturn(Optional.of(mockUser));
    when(mockInventory.check(any())).thenReturn(true);
    when(mockPayment.charge(any())).thenReturn(mockPaymentResult);
    when(mockNotifier.send(any())).thenReturn(true);
    // ... 这个测试变成Mock配置的测试，而非业务逻辑的测试
}

// 好例：使用Fake替代Mock去测真正逻辑
// 或：只Mock外部边界（支付、通知），不Mock内部领域对象
```

**Mock使用原则：**
- Mock你不拥有的接口（外部服务、第三方库的边界）
- 不要Mock你拥有的值对象和实体
- 如果发现需要Mock太多东西，这是设计有问题——被测类承担了太多职责

### 反模式5：测试依赖时间（Time Dependency）

```java
// 反模式：依赖系统时间导致测试不稳定
@Test
void isPromotionActive_CurrentTimeWithinRange_ReturnsTrue() {
    // 测试是否通过取决于"现在"是不是活动时间！
    assertTrue(promotionService.isActive("PROMO_001"));
}
// 解决：注入时间源，让时间可控
```

### 反模式6：在测试中加入条件逻辑（Conditional Test Logic）

```java
// 反模式：测试中有if/for/while —— 测试逻辑本身可能有bug
@Test
void processOrder(Order order) {
    if (order.isVIP()) {
        assertEquals(0.8, order.getDiscountRate());
    } else {
        assertEquals(1.0, order.getDiscountRate());
    }
}
// 好例：拆成两个独立的测试方法
@Test void processOrder_VIP_AppliesDiscount() { /* ... */ }
@Test void processOrder_Regular_NoDiscount() { /* ... */ }
```

### 反模式7：测试的实现细节而非行为（Testing Implementation Details）

```java
// 反模式：测试绑定到具体实现（调用了哪些私有方法）
@Test
void calculatePrice_CallsPrivateCalculateTax() {
    // 不应该验证私有方法是否被调用！
    verify(priceCalculator, times(1)).calculateTax(any()); 
}

// 好例：测试可观察的行为/返回值
@Test
void calculatePrice_ProductWithTax_ReturnsPricePlusTax() {
    Price price = priceCalculator.calculate(product);
    assertEquals(new BigDecimal("118.00"), price.getTotal()); // 验证结果而非实现
}
```

### 反模式8：测试命名不清晰

```java
// 反模式
@Test void test1() { /* ... */ }
@Test void testCalculate() { /* ... */ }
@Test void checkOrder() { /* ... */ }

// 好例：命名规范 - MethodName_Scenario_ExpectedBehavior
@Test void calculateDiscount_VIPCustomerDuringPromotion_AppliesExtra20Percent() { }
@Test void placeOrder_InsufficientStock_ThrowsOutOfStockException() { }
@Test void validateEmail_EmptyString_ReturnsFalse() { }
```

### 案例1：Java + JUnit 5 + Mockito —— 订单服务单元测试

**被测代码：**

```java
public class OrderService {
    private final ProductRepository productRepo;
    private final OrderRepository orderRepo;
    private final PaymentGateway paymentGateway;
    private final NotificationService notifier;
    
    public OrderResult placeOrder(OrderRequest request) {
        // 1. 校验商品存在且有库存
        Product product = productRepo.findById(request.getProductId())
            .orElseThrow(() -> new ProductNotFoundException(request.getProductId()));
        if (product.getStock() < request.getQuantity()) {
            throw new InsufficientStockException(product.getId(), request.getQuantity());
        }
        // 2. 计算订单金额
        BigDecimal totalAmount = product.getPrice().multiply(BigDecimal.valueOf(request.getQuantity()));
        // 3. 调用支付网关扣款
        PaymentResult payment = paymentGateway.charge(request.getUserId(), totalAmount);
        if (!payment.isSuccess()) {
            throw new PaymentFailedException(payment.getError());
        }
        // 4. 创建订单
        Order order = new Order(UUID.randomUUID().toString(), request.getUserId(),
            product.getId(), request.getQuantity(), totalAmount, OrderStatus.CONFIRMED);
        orderRepo.save(order);
        // 5. 扣减库存
        productRepo.decrementStock(product.getId(), request.getQuantity());
        // 6. 发送通知
        notifier.sendOrderConfirmation(order);
        return new OrderResult(order.getId(), OrderStatus.CONFIRMED, totalAmount);
    }
}
```

### 案例2：Python + pytest —— 用户认证服务

**被测代码：**

```python
# auth_service.py
import hashlib
import re
from datetime import datetime, timedelta
import jwt

class AuthService:
    def __init__(self, user_repo, email_sender):
        self.user_repo = user_repo
        self.email_sender = email_sender
        self.secret = "my-secret-key"
    
    def register(self, email, password):
        self._validate_email(email)
        self._validate_password(password)
        if self.user_repo.find_by_email(email):
            raise ValueError("邮箱已被注册")
        hashed = hashlib.sha256(password.encode()).hexdigest()
        user = self.user_repo.create(email, hashed)
        self.email_sender.send_welcome(email)
        return user
    
    def login(self, email, password):
        user = self.user_repo.find_by_email(email)
        if not user:
            raise ValueError("用户不存在")
        hashed = hashlib.sha256(password.encode()).hexdigest()
        if user.password_hash != hashed:
            raise ValueError("密码错误")
        return self._generate_token(user)
    
    def _validate_email(self, email):
        if not email or "@" not in email:
            raise ValueError("邮箱格式无效")
    
    def _validate_password(self, password):
        if len(password) < 8:
            raise ValueError("密码至少8位")
        if not re.search(r'[A-Za-z]', password):
            raise ValueError("密码需包含字母")
        if not re.search(r'[0-9]', password):
            raise ValueError("密码需包含数字")
    
    def _generate_token(self, user):
        payload = {
            "user_id": user.id,
            "email": user.email,
            "exp": datetime.utcnow() + timedelta(hours=24)
        }
        return jwt.encode(payload, self.secret, algorithm="HS256")
```

// cart.js
class ShoppingCart {
    constructor() {
        this.items = [];
        this.discountCodes = new Map();
        this.discountCodes.set('SAVE10', { type: 'percentage', value: 10, minAmount: 50 });
        this.discountCodes.set('FLAT20', { type: 'fixed', value: 20, minAmount: 100 });
    }
    
    addItem(product, quantity = 1) {
        if (!product || !product.id || !product.price) {
            throw new Error('Invalid product');
        }
        if (quantity <= 0 || !Number.isInteger(quantity)) {
            throw new Error('Quantity must be a positive integer');
        }
        if (product.price < 0) {
            throw new Error('Price cannot be negative');
        }
        
        const existing = this.items.find(item => item.product.id === product.id);
        if (existing) {
            existing.quantity += quantity;
        } else {
            this.items.push({ product, quantity });
        }
    }
    
    removeItem(productId) {
        this.items = this.items.filter(item => item.product.id !== productId);
    }
    
    updateQuantity(productId, quantity) {
        if (quantity <= 0) {
            this.removeItem(productId);
            return;
        }
        const item = this.items.find(item => item.product.id === productId);
        if (item) {
            item.quantity = quantity;
        }
    }
    
    getSubtotal() {
        return this.items.reduce((sum, item) => 
            sum + item.product.price * item.quantity, 0
        );
    }
    
    applyDiscountCode(code) {
        const discount = this.discountCodes.get(code);
        if (!discount) {
            throw new Error('Invalid discount code');
        }
        const subtotal = this.getSubtotal();
        if (subtotal < discount.minAmount) {
            throw new Error(`Minimum order amount ${discount.minAmount} not met`);
        }
        if (discount.type === 'percentage') {
            return (subtotal * discount.value) / 100;
        } else if (discount.type === 'fixed') {
            return discount.value;
        }
        return 0;
    }
    
    getTotal(discountCode = null) {
        const subtotal = this.getSubtotal();
        if (!discountCode) return subtotal;
        const discount = this.applyDiscountCode(discountCode);
        return Math.max(0, subtotal - discount);
    }
    
    getItemCount() {
        return this.items.reduce((sum, item) => sum + item.quantity, 0);
    }
    
    clear() {
        this.items = [];
    }
}

package points

import "errors"

type MembershipLevel int

const (
    Regular   MembershipLevel = 0
    Silver    MembershipLevel = 1
    Gold      MembershipLevel = 2
    Platinum  MembershipLevel = 3
)

type PointsCalculator struct{}

// CalculateOrderPoints 根据订单金额和会员等级计算应得积分
// 规则:
//   - 每消费1元得1基础积分
//   - Silver 会员额外 +10%
//   - Gold 会员额外 +30%
//   - Platinum 会员额外 +50%
//   - 订单金额必须 > 0
func (pc *PointsCalculator) CalculateOrderPoints(orderAmount float64, level MembershipLevel) (int, error) {
    if orderAmount <= 0 {
        return 0, errors.New("订单金额必须大于0")
    }
    
    basePoints := int(orderAmount)
    var bonusRate float64
    
    switch level {
    case Regular:
        bonusRate = 0
    case Silver:
        bonusRate = 0.1
    case Gold:
        bonusRate = 0.3
    case Platinum:
        bonusRate = 0.5
    default:
        return 0, errors.New("无效的会员等级")
    }
    
    totalPoints := float64(basePoints) * (1 + bonusRate)
    return int(totalPoints), nil
}

// validator.ts
export interface ValidationRule {
    field: string;
    type: 'string' | 'number' | 'email' | 'phone' | 'array';
    required?: boolean;
    min?: number;
    max?: number;
    pattern?: RegExp;
    enum?: any[];
    custom?: (value: any) => string | null; // 返回null表示通过，返回string为错误信息
}

export class RequestValidator {
    private rules: ValidationRule[];

    constructor(rules: ValidationRule[]) {
        this.rules = rules;
    }

    validate(data: Record<string, any>): { valid: boolean; errors: Record<string, string> } {
        const errors: Record<string, string> = {};

        for (const rule of this.rules) {
            const value = data[rule.field];

            // 必填校验
            if (rule.required && (value === undefined || value === null || value === '')) {
                errors[rule.field] = `${rule.field} 是必填字段`;
                continue; // 必填不满足时跳过后续校验
            }

            // 如果非必填且值为空，跳过后续校验
            if (!rule.required && (value === undefined || value === null || value === '')) {
                continue;
            }

            // 类型校验
            const typeError = this.validateType(rule.field, value, rule);
            if (typeError) {
                errors[rule.field] = typeError;
                continue;
            }

            // min / max 校验（字符串长度或数字大小）
            if (rule.min !== undefined || rule.max !== undefined) {
                const rangeError = this.validateRange(rule.field, value, rule);
                if (rangeError) {
                    errors[rule.field] = rangeError;
                    continue;
                }
            }

            // 正则校验
            if (rule.pattern && !rule.pattern.test(String(value))) {
                errors[rule.field] = `${rule.field} 格式不正确`;
                continue;
            }

            // 枚举值校验
            if (rule.enum && !rule.enum.includes(value)) {
                errors[rule.field] = `${rule.field} 值不在允许范围内`;
                continue;
            }

            // 自定义校验
            if (rule.custom) {
                const customError = rule.custom(value);
                if (customError) {
                    errors[rule.field] = customError;
                }
            }
        }

        return {
            valid: Object.keys(errors).length === 0,
            errors,
        };
    }

    private validateType(field: string, value: any, rule: ValidationRule): string | null {
        switch (rule.type) {
            case 'string':
                if (typeof value !== 'string') return `${field} 必须是字符串类型`;
                break;
            case 'number':
                if (typeof value !== 'number' || isNaN(value)) return `${field} 必须是数字类型`;
                break;
            case 'email':
                if (typeof value !== 'string' || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value))
                    return `${field} 必须是有效的邮箱格式`;
                break;
            case 'phone':
                if (typeof value !== 'string' || !/^1[3-9]\d{9}$/.test(value))
                    return `${field} 必须是有效的手机号`;
                break;
            case 'array':
                if (!Array.isArray(value)) return `${field} 必须是数组类型`;
                break;
        }
        return null;
    }

    private validateRange(field: string, value: any, rule: ValidationRule): string | null {
        if (rule.type === 'number') {
            if (rule.min !== undefined && value < rule.min)
                return `${field} 不能小于 ${rule.min}`;
            if (rule.max !== undefined && value > rule.max)
                return `${field} 不能大于 ${rule.max}`;
        } else {
            const len = String(value).length;
            if (rule.min !== undefined && len < rule.min)
                return `${field} 长度不能小于 ${rule.min}`;
            if (rule.max !== undefined && len > rule.max)
                return `${field} 长度不能大于 ${rule.max}`;
        }
        return null;
    }
}

| **依赖管理** | 需要完整环境 | 隔离外部依赖 |
| **谁编写** | 测试工程师 | 开发工程师 |
| **可重复性** | 中（环境可能变化） | 高（隔离外部变化） |

**各框架核心对比：**

| 框架 | 语言 | 核心优势 | 典型场景 |
|------|------|----------|----------|
| JUnit 5 | Java | 生态最成熟，Spring Boot默认集成 | 企业级后端服务 |
| TestNG | Java | 并行/分组/依赖管理更强大 | 复杂测试编排 |
| pytest | Python | 语法极简，插件丰富 | 数据科学/后端/工具 |
| Jest | JS/TS | 零配置，快照测试，内置Mock | 前端/Node.js全栈 |
| Go Testing | Go | 语言内置，表驱动测试惯用法 | 云原生/微服务 |

---

> **核心要点回顾**：单元测试是软件质量的基石。它追求的是"快"和"隔离"——每个测试在毫秒级完成，不依赖任何外部资源。FIRST原则（Fast, Independent, Repeatable, Self-Validating, Timely）是评判单元测试质量的标准。TDD不是测试技术而是设计方法。测试替身（Mock/Stub/Fake/Spy/Dummy）是实现隔离的核心工具。覆盖率是必要但不充分的度量，真正重要的是断言的有效性和边界条件的覆盖。
