# 集成测试 (Integration Testing)

> **集成测试**（Integration Testing）：在单元测试之后，将多个已通过单元测试的模块组合在一起，验证它们之间的接口、交互、数据传递是否正确的测试方法。它位于测试金字塔的中间层，是连接单元测试和系统测试的桥梁。

## 1. 集成测试的定义与目的

### 通俗释义

集成测试就是把已经通过单元测试的各个模块按照设计要求"拼装"起来，测试模块之间的接口是否正常、数据传递是否正确、协同工作是否符合预期。如果说单元测试验证的是"每个零件是否合格"，集成测试验证的就是"零件组装后能否协同运转"。

### 详细解释

集成测试（Integration Testing）是软件测试层级中的关键一环。在单元测试确认每个独立模块内部逻辑正确后，集成测试开始验证模块之间的交互。模块组合后最常见的问题包括：接口不匹配、数据传递丢失/错乱、时序问题、资源竞争、全局数据结构的破坏等。

**集成测试的核心关注点：**

| 关注维度 | 说明 | 典型缺陷 |
|----------|------|----------|
| **接口匹配** | 调用方和被调用方的参数类型、数量、顺序、返回值是否一致 | 参数类型不匹配、缺少必填参数 |
| **数据传递** | 数据在模块间传递时是否完整、正确（包括边界值、特殊字符、编码） | 数据截断、编码转换乱码、精度丢失 |
| **调用时序** | 模块间的调用顺序是否符合业务逻辑 | 先支付后回调但支付状态还未更新 |
| **错误处理** | 下游模块异常时，上游模块是否正确处理 | 异常被吞掉、超时未处理、重试逻辑不生效 |
| **性能阈值** | 集成后是否出现性能下降 | N+1查询、连接池耗尽、死锁 |
| **事务一致性** | 跨模块操作的事务边界是否正确 | 部分成功部分失败、数据不一致 |

**集成测试 vs 单元测试 vs 系统测试：**

```
单元测试：测单个函数/方法  ← 开发最熟悉，速度最快
    ↓
集成测试：测模块间交互    ← 关注接口、数据流、异常传播
    ↓
系统测试：测端到端业务流程 ← 关注用户体验、全链路功能
```

## 2. 集成测试策略

### 2.1 大爆炸集成 (Big Bang)

**策略描述：** 将所有模块一次性全部组装在一起进行测试。

```
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│ A │ │ B │ │ C │ │ D │ │ E │  所有模块同时集成
└─┬─┘ └─┬─┘ └─┬─┘ └─┬─┘ └─┬─┘
  └──────┴──────┴──────┴──────┘
               │
          ┌────▼────┐
          │ 整体测试  │ ← 一次性测试整个系统
          └─────────┘
```

| 优势 | 劣势 |
|------|------|
| 方法简单，无需额外编写桩/驱动 | 缺陷定位极其困难——失败时无法判断是哪个模块的哪个接口出问题 |
| 适用于小型系统（模块<5个） | 接口缺陷容易互相掩盖（A模块的bug导致B模块走不到异常分支） |
| 无需规划集成顺序 | 测试不充分——部分接口路径因前置条件不满足而未被触发 |

**建议：** 仅适用于小而简单的系统，或作为快速冒烟测试的补充。对中大型系统坚决不推荐。

### 2.2 自顶向下集成 (Top-Down)

**策略描述：** 从顶层控制模块开始，逐步向下集成。下一层未就绪的模块用"桩"（Stub）替代。每集成一个下层模块，替换掉对应的桩。

```
       ┌──────┐
       │ Main │  ← 第1步：先测顶层
       └──┬───┘
    ┌─────┼─────┐
    ▼     ▼     ▼
 ┌────┐┌────┐┌────┐
 │Stub││Stub││Stub│  ← 下层用桩替代
 └────┘└────┘└────┘

       ┌──────┐
       │ Main │
       └──┬───┘
    ┌─────┼─────┐
    ▼     ▼     ▼
 ┌────┐┌────┐┌────┐
 │ A  ││Stub││Stub│  ← 第2步：替换Stub为真实模块A
 └──┬─┘└────┘└────┘
    ▼
 ┌────┐
 │Stub│
 └────┘
```

| 优势 | 劣势 |
|------|------|
| 早期即可看到系统整体框架和控制流程 | 桩（Stub）的编写工作量较大 |
| 高层逻辑和关键接口优先被验证 | 底层模块的高风险逻辑验证较晚 |
| 适合GUI驱动、控制流清晰的应用 | 底层异常场景难以通过桩模拟 |

### 2.3 自底向上集成 (Bottom-Up)

**策略描述：** 从最底层的模块开始集成测试，逐步向上层推进。上层未就绪的模块用"驱动"（Driver）替代。

```
                                    ┌──────┐
                                    │ Main │  ← 最后集成
                                    └──┬───┘
    ┌─────────────┬─────────────┬──────┼──────┐
    ▼             ▼             ▼      ▼      ▼
 ┌────┐       ┌────┐       ┌────┐  ┌────┐
 │ A  │       │ B  │       │ C  │  │ D  │  ← 第2步：中间层
 └──┬─┘       └──┬─┘       └──┬─┘  └──┬─┘
    ▼             ▼             ▼      ▼
 ┌────┐       ┌────┐       ┌────┐  ┌────┐
 │ E  │       │ F  │       │ G  │  │ H  │  ← 第1步：底层先集成
 └────┘       └────┘       └────┘  └────┘
```

| 优势 | 劣势 |
|------|------|
| 底层模块的缺陷容易被精确定位 | 系统的整体框架和控制流程最后才能看到 |
| 无需编写桩（Stub），驱动（Driver）编写相对简单 | 顶层设计缺陷发现较晚 |
| 适合底层模块承担核心计算或数据处理的系统 | 修改成本高——顶层缺陷可能导致底层需重新设计 |

### 2.4 三明治集成 (Sandwich)

**策略描述：** 结合自顶向下和自底向上两种策略的混合方法。将系统分为三层：顶层用自顶向下，底层用自底向上，中间层最后集成。

```
┌─────────────────────────────────────┐
│         自顶向下 (Top-Down)          │
│    ┌──────┐                         │
│    │ Main │ ──────────────── 顶层    │
│    └──┬───┘                         │
├───────┼─────────────────────────────┤
│       │         三明治层 (Target)     │
│    ┌──▼──┐  ┌──────┐               │
│    │  A  │  │  B   │  ←── 中间层    │
│    └──┬──┘  └──┬───┘               │
├───────┼────────┼────────────────────┤
│       │   自底向上 (Bottom-Up)       │
│    ┌──▼──┐ ┌──▼───┐                │
│    │  C  │ │  D   │  ←── 底层       │
│    └─────┘ └──────┘                │
└─────────────────────────────────────┘
```

| 优势 | 劣势 |
|------|------|
| 兼具两种策略的优点 | 集成前的规划和分层需要较多设计 |
| 关键路径的接口优先得到验证 | 中间层（Target Layer）测试覆盖可能不足 |
| 适合分层架构（如MVC、三层架构） | 桩和驱动都需要编写，工作量较大 |

### 2.5 持续集成 (Continuous Integration / CI)

在现代敏捷开发中，集成测试不再是阶段性的活动，而是通过CI/CD流水线**持续进行**。每次代码提交都自动触发集成测试。

```
代码提交 → 构建 → 单元测试 → 集成测试 → 部署到测试环境 → 冒烟测试
   │                                              │
   └──────────────── 反馈 < 10分钟 ────────────────┘
```

**持续集成中的集成测试实践：**

| 实践 | 说明 |
|------|------|
| **每提交即集成** | 每个PR/MR触发集成测试，避免"集成地狱" |
| **集成测试环境容器化** | 使用Docker Compose / Testcontainers 快速搭建测试依赖 |
| **数据库版本管理** | 使用Flyway/Liquibase管理数据库schema，集成测试自动迁移 |
| **测试数据管理** | 每个测试套件使用独立的数据集，避免互相干扰 |
| **快速失败** | 单元测试失败则跳过集成测试，节省CI资源 |

### 集成策略选择指南

| 系统特征 | 推荐策略 |
|----------|----------|
| 小团队、简单系统（<5个模块） | 大爆炸 + 冒烟测试 |
| 分层架构清晰（如Spring Boot三层架构） | 三明治集成 |
| 控制流驱动的GUI应用 | 自顶向下 |
| 计算密集型、数据驱动应用 | 自底向上 |
| 微服务架构 | 持续集成 + 契约测试 |
| 遗留系统改造 | 自底向上（先稳定底层） |

## 3. 接口集成测试

### 3.1 RESTful API 集成测试

**测试目标：** 验证不同服务/模块之间通过HTTP接口交互时的正确性。

**核心测试维度：**

| 维度 | 测试内容 |
|------|----------|
| **请求构造** | HTTP方法(GET/POST/PUT/DELETE/PATCH)、URL路径参数、Query参数、请求体(JSON/XML/Form) |
| **请求头** | Content-Type、Authorization、自定义Header、Accept-Language |
| **响应验证** | 状态码(2xx/3xx/4xx/5xx)、响应体结构/字段/类型、响应头 |
| **错误处理** | 参数缺失、参数非法、业务异常、服务不可用时的HTTP状态码和错误响应体 |
| **鉴权校验** | 无Token/过期Token/错误Token的401/403响应 |
| **超时处理** | 设置合理超时，验证超时后的行为 |
| **幂等性** | 相同请求发送多次，结果一致（特别关注POST的幂等性处理） |

**API集成测试示例（Spring Boot Test + MockMvc）：**

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureMockMvc
class UserControllerIntegrationTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private UserRepository userRepository;

    @BeforeEach
    void setUp() {
        userRepository.deleteAll();
    }

    @Test
    @DisplayName("创建用户 → POST /api/users → 返回201 + 用户JSON")
    void createUser_ValidRequest_Returns201() throws Exception {
        String requestBody = """
            {
                "username": "alice",
                "email": "alice@example.com",
                "age": 25
            }
            """;

        mockMvc.perform(post("/api/users")
                .contentType(MediaType.APPLICATION_JSON)
                .content(requestBody))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.id").exists())
            .andExpect(jsonPath("$.username").value("alice"))
            .andExpect(jsonPath("$.email").value("alice@example.com"))
            .andExpect(jsonPath("$.age").value(25));

        // 验证数据库落库
        assertEquals(1, userRepository.count());
    }

    @Test
    @DisplayName("查询用户 → GET /api/users/{id} → 返回200 + 用户JSON")
    void getUserById_ExistingUser_Returns200() throws Exception {
        // 预置数据到数据库
        User saved = userRepository.save(new User(null, "bob", "bob@example.com", 30));

        mockMvc.perform(get("/api/users/{id}", saved.getId())
                .accept(MediaType.APPLICATION_JSON))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.username").value("bob"))
            .andExpect(jsonPath("$.email").value("bob@example.com"));
    }

    @Test
    @DisplayName("查询不存在的用户 → GET /api/users/{id} → 返回404")
    void getUserById_NonExisting_Returns404() throws Exception {
        mockMvc.perform(get("/api/users/{id}", 99999))
            .andExpect(status().isNotFound())
            .andExpect(jsonPath("$.message").value("用户不存在"));
    }

    @Test
    @DisplayName("参数校验 → 缺少必填字段 → 返回400")
    void createUser_MissingRequiredField_Returns400() throws Exception {
        String requestBody = """
            {
                "username": "",
                "email": "invalid-email"
            }
            """;

        mockMvc.perform(post("/api/users")
                .contentType(MediaType.APPLICATION_JSON)
                .content(requestBody))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.errors").isArray());
    }
}
```

### 3.2 契约测试 (Contract Testing)

**什么是契约测试：** 在微服务架构中，每个服务独立演进。契约测试是消费者（Consumer）和提供者（Provider）之间的一种集成测试方式——消费者定义自己期望的接口格式（契约），提供者验证自己能满足该契约。

```
┌──────────────┐      契约       ┌──────────────┐
│  Consumer    │ ◄──────────── ► │  Provider     │
│  (调用方)    │   期望/满足      │  (提供方)     │
│              │                │              │
│ 生成契约文件  │ ──────────────▶ │ 验证契约      │
│ (期望的行为)  │                │ (满足期望)    │
└──────────────┘                └──────────────┘
```

**Pact 框架的契约测试流程：**

| 步骤 | 消费者端 | 提供者端 |
|------|----------|----------|
| 1 | 编写单元测试，定义对提供者接口的期望（请求是什么、响应应是什么） | — |
| 2 | 运行测试，Pact自动生成契约文件（JSON格式） | — |
| 3 | 将契约文件发布到Pact Broker | — |
| 4 | — | 从Pact Broker拉取契约文件 |
| 5 | — | 运行契约验证：重放消费者定义的请求，验证提供者实际响应是否匹配 |

**消费者端契约定义（JavaScript + Jest）：**

```javascript
const { Pact } = require('@pact-foundation/pact');
const path = require('path');

describe('User API Consumer Contract', () => {
    const provider = new Pact({
        consumer: 'OrderService',
        provider: 'UserService',
        port: 1234,
        log: path.resolve(process.cwd(), 'logs', 'pact.log'),
        dir: path.resolve(process.cwd(), 'pacts'),
    });

    beforeAll(() => provider.setup());
    afterAll(() => provider.finalize());

    describe('get user by ID', () => {
        beforeAll(() => {
            return provider.addInteraction({
                state: 'user with ID 1 exists',
                uponReceiving: 'a request for user ID 1',
                withRequest: {
                    method: 'GET',
                    path: '/api/users/1',
                    headers: { Accept: 'application/json' },
                },
                willRespondWith: {
                    status: 200,
                    headers: { 'Content-Type': 'application/json' },
                    body: {
                        id: 1,
                        username: 'Alice',
                        email: 'alice@example.com',
                    },
                },
            });
        });

        test('应返回用户信息', async () => {
            const userClient = new UserClient('http://localhost:1234');
            const user = await userClient.getUserById(1);
            expect(user.username).toBe('Alice');
            expect(user.email).toBe('alice@example.com');
        });
    });
});
```

**提供者端契约验证（Java + JUnit）：**

```java
@Provider("UserService")
@PactBroker(url = "http://pact-broker:9292")
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class UserServicePactVerificationTest {

    @LocalServerPort

    @BeforeEach

        // 确保提供者状态"user with ID 1 exists"存在
        // 即：准备一个ID为1的用户
    }

    @TestTemplate
    @ExtendWith(PactVerificationInvocationContextProvider.class)
    void pactVerificationTestTemplate(PactVerificationContext context) {
        context.verifyInteraction();
    }

    @State("user with ID 1 exists")

        // 在数据库中创建对应的用户数据
        userRepository.save(new User(1L, "Alice", "alice@example.com"));
    }
}
```

**契约测试的价值：**

| 价值 | 说明 |
|------|------|
| **独立演进** | 消费者和提供者可以独立开发、独立测试、独立部署 |
| **变更可见** | 提供者修改接口时，契约测试会立即发现哪些消费者受影响 |
| **减少E2E** | 替代部分昂贵的端到端测试，在集成层面提供信心 |
| **文档活化石** | 契约本身就是接口的"活文档" |

## 4. 数据库集成测试

### 4.1 核心测试目标

数据库集成测试验证应用程序与数据库之间的交互是否正确——SQL语句的正确性、ORM映射的准确性、事务的完整性、数据的一致性。

**数据库集成测试的核心维度：**

| 测试维度 | 验证内容 | 常见问题 |
|----------|----------|----------|
| **CRUD操作** | 增/删/改/查的SQL是否正确执行 | SQL语法错误、字段映射错误 |
| **ORM映射** | 实体类与数据库表的映射是否正确 | 字段类型不匹配、关联关系配置错误 |
| **事务管理** | 事务提交、回滚、传播行为是否正确 | 事务不回滚、脏数据、死锁 |
| **约束校验** | 唯一键、外键、非空约束是否正确触发 | 违反约束时返回不友好的错误 |
| **查询性能** | 复杂查询是否生成N+1问题 | 关联查询性能差 |
| **数据迁移** | Schema变更脚本是否正确执行 | 迁移脚本在特定数据下失败 |
| **并发控制** | 乐观锁/悲观锁是否生效 | 并发更新丢失 |

### 4.2 数据库集成测试策略

**方案一：使用真实数据库 + Testcontainers**

```java
@SpringBootTest
@Testcontainers
class UserRepositoryIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15")
            .withDatabaseName("testdb")
            .withUsername("test")
            .withPassword("test");

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    private UserRepository userRepository;

    @Test
    @DisplayName("保存用户 → 查询可获取 → 与保存数据一致")
    void saveAndFindById_ReturnsSavedUser() {
        User user = new User(null, "Alice", "alice@example.com", 25);
        User saved = userRepository.save(user);

        Optional<User> found = userRepository.findById(saved.getId());

        assertTrue(found.isPresent());
        assertEquals("Alice", found.get().getUsername());
        assertEquals("alice@example.com", found.get().getEmail());
        assertEquals(25, found.get().getAge());
        assertNotNull(found.get().getCreatedAt()); // 自动填充的时间戳
    }

    @Test
    @DisplayName("批量插入 → 查询数量匹配")
    void saveAll_ReturnsCorrectCount() {
        List<User> users = List.of(
            new User(null, "User1", "u1@example.com", 20),
            new User(null, "User2", "u2@example.com", 21),
            new User(null, "User3", "u3@example.com", 22)
        );
```

## 5. 第三方服务集成测试

### 5.1 测试策略总览

第三方服务（支付网关、短信服务、云存储、邮件服务等）不在我们的控制范围内，集成测试需要在"真实验证"和"测试效率"之间取得平衡。

**四种策略层次：**

```
          真实调用真环境
              ▲
              │  高可信度 + 高成本 + 慢 + 不隔离
          契约测试
              │
          沙箱/测试环境
              │
          WireMock 模拟
              │  低可信度 + 低成本 + 快 + 完全隔离
              │
```

| 策略 | 可信度 | 成本 | 速度 | 适用场景 |
|------|--------|------|------|----------|
| **WireMock/模拟** | 低 | 低 | 快 | 日常开发、CI流水线中的大多数测试 |
| **第三方沙箱环境** | 中 | 低 | 中 | 关键业务流程的集成验证 |
| **契约测试** | 中高 | 中 | 中 | 第三方提供OpenAPI规范时 |
| **真实环境（预发布）** | 高 | 高 | 慢 | 上线前最终验证、周期性巡检 |

### 5.3 第三方服务的容错测试

**核心容错测试场景：**

| 异常类型 | 模拟方式 | 期望行为 |
|----------|----------|----------|
| **连接超时** | WireMock延迟 > 客户端超时 | 抛出超时异常，触发降级逻辑 |
| **读取超时** | WireMock分块响应，中间停顿 | 抛出超时异常，不阻塞调用方 |
| **服务不可用 (503)** | WireMock返回503 | 触发重试，重试耗尽后触发降级 |
| **返回畸形数据** | WireMock返回非JSON/空响应 | 客户端解析失败但不崩溃，记录日志 |
| **慢响应** | WireMock添加2s-5s固定延迟 | 触发断路器（Circuit Breaker），快速失败 |
| **并发限制** | WireMock限制并发连接数 | 调用方排队/拒绝，不丢失请求 |

## 6. 消息队列集成测试

### 6.1 消息队列测试的核心关注点

消息队列（Kafka / RabbitMQ / RocketMQ / Pulsar）在分布式系统中扮演着解耦、削峰、异步处理的关键角色。消息队列的集成测试需要关注以下维度：

| 测试维度 | 验证内容 | 常见缺陷 |
|----------|----------|----------|
| **消息生产** | 消息是否能正确发送到Topic/Queue | 发送失败无补偿、序列化错误 |
| **消息消费** | 消费者是否能正确接收并处理消息 | 消费失败无重试、死信队列未配置 |
| **消息顺序** | 需要顺序处理的场景是否保证了顺序 | 分区策略导致顺序错乱 |
| **消息幂等** | 重复消费是否导致数据异常 | 重复处理重复扣款 |
| **消息可靠性** | Broker宕机后消息是否丢失 | 未配置持久化、ACK策略设置错误 |
| **消息积压** | 消费者落后时系统的表现 | 积压导致内存溢出、临时队列满 |
| **死信处理** | 消息多次消费失败后的处理 | 死信被丢弃无人关注 |

```java
@SpringBootTest
@Testcontainers
class OrderEventIntegrationTest {

    @Container
    static KafkaContainer kafka = new KafkaContainer(
        DockerImageName.parse("confluentinc/cp-kafka:7.5.0")
    );

    @DynamicPropertySource
    static void configureKafka(DynamicPropertyRegistry registry) {
        registry.add("spring.kafka.bootstrap-servers", kafka::getBootstrapServers);
    }

    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;

    @Autowired
    private OrderEventConsumer orderEventConsumer;

    @Test
    @DisplayName("发送订单事件 → 消费者收到消息并正确处理")
    void sendOrderEvent_ConsumerReceivesAndProcesses() throws Exception {
        // Given: 准备事件
        OrderEvent event = new OrderEvent("ORDER_001", "user_1", 
            OrderStatus.CREATED, new BigDecimal("199.99"));

        // When: 发送消息
        // Spring Kafka 3.x / Spring Boot 3.x returns CompletableFuture, not ListenableFuture
        CompletableFuture<SendResult<String, OrderEvent>> future = 
            kafkaTemplate.send("order-events", event.getOrderId(), event);
        future.get(10, TimeUnit.SECONDS); // 确保发送成功

        // Then: 等待消费者处理（最多等30秒）
        boolean processed = orderEventConsumer.awaitProcessed(event.getOrderId(), 30);
        assertTrue(processed, "消息应在30秒内被消费并处理");
        
        // 验证处理结果
        OrderEvent processedEvent = orderEventConsumer.getProcessedEvent(event.getOrderId());
        assertEquals(OrderStatus.CREATED, processedEvent.getStatus());
        assertEquals(new BigDecimal("199.99"), processedEvent.getTotalAmount());
    }

    @Test
    @DisplayName("消息发送失败重试 → 至少一次成功投递")
    void sendMessage_WithRetry_AtLeastOnceDelivery() throws Exception {
        // 模拟broker短暂不可用（需要额外工具如Toxiproxy）
        // 此处验证消息发送的重试机制

        OrderEvent event = new OrderEvent("ORDER_002", "user_2", 
            OrderStatus.CREATED, new BigDecimal("50.00"));

        // 即使有瞬时故障，重试后成功投递
        SendResult<String, OrderEvent> result = 
            kafkaTemplate.send("order-events", event.getOrderId(), event)
                .get(15, TimeUnit.SECONDS);
        
        assertNotNull(result.getRecordMetadata());
        assertTrue(result.getRecordMetadata().offset() >= 0);
    }

    @Test
    @DisplayName("重复消费 → 幂等处理不应影响数据一致性")
    void duplicateConsumption_IdempotentProcessing_DataConsistent() throws Exception {
        // Given: 同一消息发送两次
        OrderEvent event = new OrderEvent("ORDER_003", "user_3", 
            OrderStatus.CREATED, new BigDecimal("100.00"));

        kafkaTemplate.send("order-events", event.getOrderId(), event).get(10, TimeUnit.SECONDS);
        // 稍后再发送一次同样的消息（模拟重试导致的重复）
        Thread.sleep(1000);
        kafkaTemplate.send("order-events", event.getOrderId(), event).get(10, TimeUnit.SECONDS);

        // When: 等待处理
        Thread.sleep(5000);

        // Then: 幂等机制应确保只处理一次（如通过数据库唯一约束消息ID）
        // 这里验证订单只被创建了一次
        int count = orderEventConsumer.getProcessedCount(event.getOrderId());
        assertEquals(1, count, "重复消息应被幂等机制过滤，只处理一次");
    }
}
```

```go
    defer cancel()

    t.Run("完整下单流程：库存检查 → 下单 → 库存扣减", func(t *testing.T) {
        // 1. 先确认库存
        stockResp, err := inventoryClient.CheckStock(ctx, &pb_inventory.CheckStockRequest{
            ProductId: "PROD_001",
        })
        require.NoError(t, err)
        initialStock := stockResp.GetAvailableQuantity()
        require.Greater(t, initialStock, int32(0), "初始库存应大于0")

        // 2. 下单
        orderResp, err := orderClient.PlaceOrder(ctx, &pb_order.PlaceOrderRequest{
            UserId:    "USER_001",
            ProductId: "PROD_001",
            Quantity:  1,
        })
        require.NoError(t, err)
        assert.Equal(t, pb_order.OrderStatus_CONFIRMED, orderResp.GetStatus())
        assert.NotEmpty(t, orderResp.GetOrderId())

        // 3. 验证库存减少
        stockResp2, err := inventoryClient.CheckStock(ctx, &pb_inventory.CheckStockRequest{
            ProductId: "PROD_001",
        })
        require.NoError(t, err)
        assert.Equal(t, initialStock-1, stockResp2.GetAvailableQuantity(), 
            "下单后库存应减少1")
    })

    t.Run("库存不足时下单应失败", func(t *testing.T) {
        _, err := orderClient.PlaceOrder(ctx, &pb_order.PlaceOrderRequest{
            UserId:    "USER_002",
            ProductId: "PROD_001",
            Quantity:  99999, // 远超实际库存
        })
        require.Error(t, err)
        // gRPC错误码应为资源耗尽
        assert.Contains(t, err.Error(), "stock")
    })
}
```

```java
@SpringBootTest
@Testcontainers
class OrderEventDrivenIntegrationTest {

    @Container
    static KafkaContainer kafka = new KafkaContainer(
        DockerImageName.parse("confluentinc/cp-kafka:7.5.0"));

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

    @DynamicPropertySource
    static void configure(DynamicPropertyRegistry registry) {
        registry.add("spring.kafka.bootstrap-servers", kafka::getBootstrapServers);
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    private OrderCommandHandler orderHandler;
    @Autowired
    private InventoryEventHandler inventoryHandler;
    @Autowired
    private NotificationEventHandler notificationHandler;
    @Autowired
    private OrderRepository orderRepo;
    @Autowired
    private ProductRepository productRepo;

    @Test
    @DisplayName("端到端事件流：订单创建 → 库存扣减事件 → 通知发送事件")
    void orderCreated_TriggersInventoryAndNotificationEvents() throws Exception {
        // Given: 预置商品库存
        Product product = productRepo.save(
            new Product(null, "SKU_100", "Test Product", new BigDecimal("99.00"), 10));

        // When: 创建订单命令
        CreateOrderCommand command = new CreateOrderCommand(
            "user_1", product.getSku(), 2, new BigDecimal("198.00"));
        String orderId = orderHandler.handle(command);

        // Then: 订单已创建并持久化
        assertNotNull(orderId);
        Order order = orderRepo.findById(orderId).orElseThrow();
        assertEquals(OrderStatus.CREATED, order.getStatus());

        // Then: 等待库存扣减事件被处理（轮询等待，最多20秒）
        await().atMost(20, TimeUnit.SECONDS).until(() -> {
            Product updated = productRepo.findById(product.getId()).orElseThrow();
            return updated.getStock() == 8; // 10 - 2
        });

        // Then: 订单状态已更新为 CONFIRMED（库存扣减成功后更新）
        order = orderRepo.findById(orderId).orElseThrow();
        assertEquals(OrderStatus.CONFIRMED, order.getStatus());

        // Then: 通知事件已发送（验证通知处理器收到了事件）
        assertTrue(notificationHandler.wasNotificationSent(orderId),
            "确认订单后应发送通知事件");
    }

    @Test
    @DisplayName("库存不足 → 订单失败 → 发送失败通知")
    void insufficientStock_OrderFails_NotificationSent() throws Exception {
        // Given: 库存仅1件
        Product product = productRepo.save(
            new Product(null, "SKU_200", "Rare", new BigDecimal("999.00"), 1));

        // When: 尝试下单3件
        CreateOrderCommand command = new CreateOrderCommand(
            "user_2", product.getSku(), 3, new BigDecimal("2997.00"));
        String orderId = orderHandler.handle(command);

        // Then: 订单进入失败状态
        await().atMost(15, TimeUnit.SECONDS).until(() -> {
            Order order = orderRepo.findById(orderId).orElseThrow();
            return order.getStatus() == OrderStatus.FAILED;
        });

        // Then: 库存未变
        Product unchanged = productRepo.findById(product.getId()).orElseThrow();
        assertEquals(1, unchanged.getStock());

        // Then: 失败通知已发送
        assertTrue(notificationHandler.wasFailureNotificationSent(orderId),
            "订单失败应发送失败通知");
    }
}
```

## 8. 附录：集成测试策略对比总结

| 策略 | 适用场景 | 缺陷定位 | 桩/驱动工作量 | 早期发现架构问题 |
|------|----------|----------|-------------|-----------------|
| **大爆炸** | 小型系统(<5个模块) | 困难 | 无 | 否 |
| **自顶向下** | 控制流清晰的应用 | 中等 | 桩编写工作量大 | 是 |
| **自底向上** | 计算/数据驱动应用 | 较容易 | 驱动编写相对简单 | 否 |
| **三明治** | 分层架构系统 | 中等 | 两者都要 | 是（上层） |
| **持续集成** | 所有现代项目 | 容易（增量变更） | 自动化 | 是 |

**集成测试的测试替身 vs 单元测试的测试替身：**

| 替身类型 | 单元测试中使用 | 集成测试中使用 |
|----------|---------------|---------------|
| **Mock** | 大量使用（Mockito/EasyMock） | 仅Mock外部边界（第三方API） |
| **Stub** | 偶尔使用 | 在自顶向下策略中常用 |
| **Fake** | 不常用 | 大量使用（H2、Embedded Kafka等） |
| **真实组件** | 不使用 | 被测模块间使用真实实例 |

## 本章小结

1. **集成测试连接单元测试与系统测试**：它验证模块之间的"接口匹配、数据传递、调用时序、错误传播"，关注的是模块组装后的协同工作能力，而非单个模块的内部逻辑。
2. **集成策略需匹配系统架构**：大爆炸仅适合小型系统；自顶向下优先验证高层逻辑；自底向上便于定位底层缺陷；三明治结合两者优点；持续集成是现代敏捷项目的标配。
3. **契约测试是微服务时代的核心手段**：通过消费者定义期望、提供者验证满足，契约测试实现服务间独立演进，降低端到端测试的依赖和成本。
4. **数据库集成测试应使用真实数据库**：Testcontainers 提供轻量级的真实数据库实例，避免 H2 等内存数据库与生产数据库方言不一致导致的"假通过"风险。
5. **第三方服务测试需平衡可信度与效率**：日常开发使用 WireMock 模拟获得快速反馈，关键业务流使用沙箱环境验证，上线前在预发布环境做最终确认。

> **核心要点回顾**：集成测试验证的是模块之间的"连接处"——接口匹配、数据传递、调用时序、错误传播。选择合适的集成策略（大爆炸/自顶向下/自底向上/三明治）取决于系统架构特征。在现代微服务架构中，契约测试（Contract Testing）是一种轻量高效的替代方案。数据库集成测试应使用真实数据库（Testcontainers）而非内存数据库以保证可靠性。第三方服务集成测试要在可信度和效率之间取得平衡 —— WireMock用于日常开发，真实沙箱环境用于上线前验证。消息队列的集成测试必须关注消息的可靠性、幂等性和死信处理。
