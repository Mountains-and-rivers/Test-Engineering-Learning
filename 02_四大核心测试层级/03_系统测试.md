# 系统测试 (System Testing)

> **系统测试**（System Testing）：在集成测试完成后，将整个软件系统作为一个完整的整体，在模拟或真实的生产环境中验证其是否满足需求规格说明书中定义的功能和非功能需求。它是测试金字塔中覆盖面最广、最接近用户视角的测试层级。

## 1. 系统测试的定义与范围

### 通俗释义

系统测试就是把所有功能都部署到跟(生成)环境一样，模拟真实用户从头到尾用一遍，看整个系统是否满足需求，不仅是功能是否正确，还有快不快、稳不稳、安不安全、好不好用。

### 详细解释

系统测试（System Testing）是软件开发生命周期中第一个将软件作为**完整系统**进行验证的测试阶段。与单元测试关注"零件质量"、集成测试关注"零件装配"不同，系统测试关注"整机性能"——从最终用户的视角验证软件的端到端行为。

**系统测试在测试金字塔中的位置：**

```
         ┌──────────────────────┐
         │     验收测试 (UAT)    │ ← 用户验收，确认是否满足业务需求
         ├──────────────────────┤
         │     系统测试          │ ← 【本阶段】验证完整系统是否满足规格说明
         ├──────────────────────┤
         │     集成测试          │ ← 验证模块间交互
         ├──────────────────────┤
         │     单元测试          │ ← 验证单个函数/方法
         └──────────────────────┘
```

**系统测试的六大范围：**

| 测试类型 | 关注点 | 核心问题 |
|----------|--------|----------|
| **功能测试** | 业务功能是否按需求实现 | "系统做了该做的事吗？" |
| **性能测试** | 系统在不同负载下的响应能力 | "系统够快吗？能扛住多少人？" |
| **安全测试** | 系统的安全防护能力 | "系统能防住攻击吗？" |
| **兼容性测试** | 系统在不同环境下的表现 | "换个浏览器/设备还能用吗？" |
| **可用性测试** | 用户使用体验 | "系统好用吗？用户能找到功能吗？" |
| **可靠性测试** | 系统长时间运行的稳定性 | "系统能连续运行7天不崩溃吗？" |

**系统测试与其他测试层级的对比：**

| 维度 | 单元测试 | 集成测试 | 系统测试 |
|------|----------|----------|----------|
| **测试对象** | 函数/方法 | 模块间接口 | 整个系统 |
| **测试视角** | 开发者（白盒） | 架构师（灰盒） | 用户（黑盒） |
| **环境要求** | 无外部依赖 | 部分依赖（DB/MQ等） | 类生产完整环境 |
| **执行耗时** | 毫秒级/个 | 秒级/个 | 分钟/小时级/个 |
| **维护成本** | 低 | 中 | 高 |
| **缺陷修复成本** | 最低 | 中等 | 较高 |

## 2. 功能测试 (Functional Testing)

### 2.1 定义与范围

功能测试是系统测试中最核心的部分，验证系统是否按照需求规格实现了所有功能——使用有效输入得到预期的输出，使用无效输入得到合理的错误提示。

**功能测试的核心验证矩阵：**

| 验证维度 | 测试重点 | 方法支撑 |
|----------|----------|----------|
| **输入校验** | 所有输入点的有效/无效数据处理 | 等价类划分、边界值分析 |
| **业务流程** | 多条端到端业务路径的完整执行 | 场景分析法 |
| **数据处理** | 数据计算、转换、存储的正确性 | 基于需求测试 |
| **状态管理** | 业务对象的状态流转 | 状态迁移法 |
| **权限控制** | 不同角色的功能可见性和可操作性 | 基于需求测试 |
| **异常处理** | 系统在异常输入下的行为 | 错误推测法 |


### 2.3 功能测试的自动化分层策略

| 层级 | 自动化率目标 | 说明 |
|------|-------------|------|
| **核心业务流程** (P0) | 95%+ | 登录、下单、支付等核心路径必须自动化 |
| **重要功能** (P1) | 70-80% | 次要功能、高频功能的自动化 |
| **一般功能** (P2) | 30-50% | 低频功能可依赖手工测试 |
| **边缘功能** (P3) | 0-10% | 极少触达的功能手工做一轮即可 |

## 3. 非功能测试

### 3.1 性能测试

性能测试验证系统在不同负载条件下的响应速度、吞吐量、资源利用率和稳定性。

**性能测试的四种类型：**

| 类型 | 目的 | 典型场景 |
|------|------|----------|
| **负载测试** (Load Testing) | 验证系统在预期负载下的性能 | 模拟正常峰值并发用户数，验证响应时间是否达标 |
| **压力测试** (Stress Testing) | 找到系统的性能拐点和崩溃点 | 逐步增加负载，观察何时响应时间急剧恶化或开始报错 |
| **容量测试** (Capacity Testing) | 验证系统能处理的最大数据量/用户量 | 大数据量场景下的查询/导出/计算性能 |
| **稳定性测试** (Soak/Endurance Testing) | 验证长时间运行的稳定性 | 以预期负载连续运行24-72小时，观察内存泄漏/连接池耗尽等 |
| **峰值测试** (Spike Testing) | 验证突增负载后的恢复能力 | 秒杀场景：瞬时从100并发飙升至10000并发 |

**性能测试关键指标：**

| 指标 | 说明 | 典型目标 |
|------|------|----------|
| **响应时间 (RT)** | 请求从发出到收到响应的时间 | P95 < 500ms (Web), P99 < 200ms (API) |
| **吞吐量 (TPS/QPS)** | 系统每秒处理的请求数 | 视业务而定，关注能否达到预期峰值的1.5倍 |
| **并发用户数** | 系统同时服务的用户数 | 视业务而定 |
| **错误率** | 失败请求占总请求的比例 | < 0.1% (目标), < 1% (预警), >5% (严重故障) |
| **CPU使用率** | 服务器的CPU占用 | < 70% (稳态), < 90% (峰值) |
| **内存使用率** | 服务器的内存占用 | < 80%，无持续增长（内存泄漏检测） |
| **GC频率/耗时** | JVM等语言的垃圾回收 | Full GC频率 < 1次/小时，单次STW < 200ms |

**性能测试工具选型：**

| 工具 | 语言/技术栈 | 优势 | 适用场景 |
|------|-----------|------|----------|
| JMeter | Java (UI/CLI) | 功能最全面，插件丰富，支持多种协议 | HTTP/数据库/JMS/MQ 全场景 |
| Gatling | Scala (DSL) | 代码化场景，报告精美，高性能 | HTTP API 性能测试 |
| k6 | Go + JavaScript | 开发者友好，云原生，CI/CD集成好 | 微服务、API、Kubernetes |
| Locust | Python | Python生态，分布式简单 | Web应用、API |
| wrk/wrk2 | C | 极高性能，极简使用 | 快速基准测试、单接口压测 |

### 3.2 安全测试

安全测试验证系统在面对恶意攻击、数据泄露、未授权访问时的防护能力。

**安全测试十大检查维度：**

| 维度 | 检查内容 | 常见漏洞 |
|------|----------|----------|
| **认证 (Authentication)** | 身份验证机制是否正确 | 弱密码策略、Session固定、Token泄露 |
| **授权 (Authorization)** | 权限控制是否完备 | 越权访问、IDOR(不安全的直接对象引用) |
| **输入校验** | 所有输入点是否防注入 | SQL注入、XSS、命令注入、XXE |
| **数据传输** | 敏感数据是否加密传输 | 明文密码、HTTPS未强制、证书配置错误 |
| **会话管理** | Session/Token管理是否安全 | 登出未失效、超时未过期、未绑定IP/设备 |
| **敏感数据** | 敏感信息是否妥善保护 | 日志打印密码、响应返回多余字段、前端暴露密钥 |
| **文件安全** | 文件上传下载是否安全 | 任意文件上传(WebShell)、路径遍历、文件类型伪造 |
| **加密算法** | 加密是否使用安全算法 | MD5/SHA1用于密码、硬编码密钥、弱随机数 |
| **日志审计** | 关键操作是否有审计 | 缺少审计日志、日志可被篡改、日志含敏感信息 |
| **依赖安全** | 第三方库是否存在已知漏洞 | Log4Shell、Fastjson反序列化、过时组件 |

**安全测试工具链：**

| 工具 | 类型 | 说明 |
|------|------|------|
| OWASP ZAP | DAST (动态扫描) | 开源Web漏洞扫描器，适合CI/CD集成 |
| Burp Suite | DAST + 手工渗透 | 业界标准的Web渗透测试工具 |
| SonarQube | SAST (静态分析) | 代码级安全漏洞检测 |
| sqlmap | SQL注入检测 | 自动化SQL注入检测与利用 |
| Nmap | 端口/服务扫描 | 网络层面发现开放端口和服务 |
| Metasploit | 渗透测试框架 | 漏洞利用验证 |

**OWASP Top 10 (2021) 与对应测试策略：**

| 排名 | 风险 | 系统测试验证方法 |
|------|------|-----------------|
| A01 | 访问控制失效 | 用低权限用户访问高权限接口，验证403返回 |
| A02 | 加密机制失效 | 检查是否使用HTTPS，检查Cookie的Secure/HttpOnly标记 |
| A03 | 注入攻击 | 在所有输入框输入SQL注入/XSS payload，验证过滤 |
| A04 | 不安全的设计 | 审查API的限额、频率限制是否生效 |
| A05 | 安全配置错误 | 检查错误页面是否暴露堆栈信息、中间件版本 |
| A06 | 脆弱的组件 | SCA扫描第三方依赖的CVE漏洞 |
| A07 | 认证失效 | 测试弱密码注册、暴力破解锁定 |
| A08 | 软件和数据完整性 | 验证文件下载的哈希校验 |
| A09 | 日志和监控失效 | 检查关键操作是否有审计日志 |
| A10 | SSRF | 测试URL参数是否可访问内网地址 |

### 3.3 兼容性测试

兼容性测试验证系统在不同硬件、操作系统、浏览器、网络环境下的表现。

**兼容性测试矩阵：**

| 兼容维度 | 测试内容 | 策略 |
|----------|----------|------|
| **浏览器兼容** | Chrome/Firefox/Safari/Edge 近3个主版本 | 使用Selenium Grid / BrowserStack 并行执行 |
| **操作系统兼容** | Windows 10/11, macOS, Linux (Ubuntu), iOS, Android | 关键业务覆盖主OS，次要功能覆盖Top2 |
| **设备兼容** | 不同品牌/型号手机，不同分辨率/屏幕尺寸 | 使用正交法减少组合数，重点覆盖Top设备 |
| **分辨率兼容** | 1920×1080, 1366×768, 2560×1440, 移动端常见分辨率 | 响应式设计测试 |
| **网络兼容** | WiFi, 4G, 5G, 弱网(2G/3G), 断网恢复 | Charles/Fiddler限速模拟 |
| **数据库兼容** | MySQL/PostgreSQL的不同版本 | 用Testcontainers切换数据库版本测试 |
| **JDK/运行时兼容** | JDK 8/11/17/21, Node.js LTS版本 | CI矩阵构建 |

**浏览器兼容性测试优先级策略：**

| 优先级 | 浏览器 | 覆盖率 | 原因 |
|--------|--------|--------|------|
| P0 (全覆盖) | Chrome 最新版 | 100%用例 | 中国市场份额 > 60% |
| P1 (重点覆盖) | Safari 最新版 (iOS) | 80%用例 | 移动端用户主要浏览器 |
| P2 (核心覆盖) | Edge 最新版 | 核心业务流程 | Windows默认浏览器 |
| P3 (基本覆盖) | Firefox 最新版 | 冒烟级别 | 开发者用户群 |

### 3.4 可用性测试

可用性测试关注系统是否易学、易用、高效、容错。

**可用性测试指标体系：**

| 指标 | 衡量方式 | 目标值 |
|------|----------|--------|
| **任务完成率** | 用户完成指定任务的比例 | > 90% |
| **任务完成时间** | 用户完成指定任务的平均时间 | 与竞品/预期对比 |
| **错误率** | 用户操作出错的频率 | < 5% |
| **可学习性** | 新用户首次使用后独立完成操作的比率 | > 80% |
| **满意度** | SUS(系统可用性量表)评分 | > 68 (行业平均) |

**可用性测试的尼尔森十大原则检查清单：**

| 原则 | 系统测试中的验证方法 |
|------|---------------------|
| **1. 系统状态可见** | 加载中有进度条/spinner，操作结果有明确反馈 |
| **2. 系统与真实世界匹配** | 语言符合用户习惯，图标直观，错误提示说人话 |
| **3. 用户控制与自由** | 支持撤销操作，关键操作有二次确认 |
| **4. 一致性与标准** | 同类功能操作方式一致，术语统一，遵循平台规范 |
| **5. 错误预防** | 输入框有格式提示，危险操作按钮颜色区分，必填标记清晰 |
| **6. 识别而非回忆** | 导航可见，搜索支持建议，历史记录可查 |
| **7. 灵活高效** | 支持快捷键，支持批量操作，常用功能入口浅 |
| **8. 简洁设计** | 信息层级清晰，不相关/低频信息折叠 |
| **9. 帮助识别错误** | 错误提示指出问题+给出解决方案，表单校验实时显示 |
| **10. 帮助文档** | 帮助入口易找，搜索功能可用 |

## 4. 端到端测试 (E2E Testing)

### 4.1 定义与定位

端到端测试从最终用户视角出发，验证完整的业务流程在真实环境中是否能正确执行。它横跨前端、后端、数据库、第三方服务——整个技术栈。

```
用户操作APP → 前端 → API网关 → 后端服务 → 数据库
                                    │
                              ┌─────┴─────┐
                              │  消息队列   │
                              │  缓存       │
                              │  第三方API  │
                              └───────────┘
         └──────────── E2E测试覆盖 ────────────┘
```

### 4.2 E2E测试框架选型

| 框架 | 语言 | 优势 | 劣势 | 适用 |
|------|------|------|------|------|
| **Playwright** | JS/TS/Python/Java/.NET | 速度最快，自动等待，多浏览器，移动端模拟 | 相对年轻 | Web端E2E首选 |
| **Cypress** | JavaScript | 开发者体验优秀，调试方便，实时重载 | 仅支持Chromium系，多Tab限制 | 前端密集型应用 |
| **Selenium** | 多语言 | 浏览器支持最全，社区最成熟 | 速度慢，API较底层，需手动等待 | 多浏览器兼容测试 |
| **Appium** | 多语言 | 跨平台移动测试 | 配置复杂，执行慢 | 移动端自动化 |
| **Puppeteer** | JavaScript | 轻量，Headless Chrome控制 | 仅Chrome | 页面截图、PDF生成 |

### 4.3 E2E测试的"测试金字塔"反模式

```
【反模式】冰淇淋甜筒 —— 过度依赖E2E测试

     ┌──────────┐
     │ E2E 测试  │  ← 大量！慢、不稳定、维护成本高
     ├──────────┤
     │ 集成测试  │  ← 少量
     ├──────────┤
     │ 单元测试  │  ← 极少
     └──────────┘

【推荐】测试金字塔 —— E2E测试只在关键路径

     ┌──────────┐
     │ E2E 测试  │  ← 少量：5-10个核心业务流程
     ├──────────┤
     │ 集成测试  │  ← 适量：模块间交互
     ├──────────┤
     │ 单元测试  │  ← 大量：业务逻辑覆盖
     └──────────┘
```

**E2E 测试的严格控制：**

| 控制策略 | 说明 |
|----------|------|
| **数量限制** | 全系统的E2E测试不应超过20-30个场景 |
| **只测核心** | 仅覆盖P0级的端到端业务流程（如：注册→登录→下单→支付→收货） |
| **环境稳定** | E2E测试环境必须稳定，避免因环境问题导致的"假阳性"失败 |
| **失败必查** | E2E测试失败必须第一时间排查原因，不能"重跑看看" |
| **定期而非每提交** | E2E测试不需要在每个PR触发（太长），每日/每版本执行即可 |

## 5. 回归测试策略

### 5.1 定义与挑战

回归测试（Regression Testing）是在软件变更（新功能、Bug修复、配置变更）后，重新执行已有测试用例，以确认变更未引入新的缺陷、未破坏已有功能。

**回归测试的核心挑战：**
- 随着版本迭代，回归用例数量不断膨胀（从100个增长到1000+）
- 全部回归耗时越来越长（从30分钟增长到8小时+）
- "全回归"不可持续，必须有选择地执行

### 5.2 回归测试的六大策略

| 策略 | 说明 | 适用时机 | 覆盖率 |
|------|------|----------|--------|
| **全量回归** | 执行全部已有的测试用例 | 大版本发布前的最终验证 | 100% |
| **选择性回归** | 仅执行与变更模块相关的用例 | 小版本迭代、日常开发 | 40-70% |
| **基于风险的回归** | 优先执行高风险模块的用例 | 时间紧迫或变更影响面不确定 | 30-50% |
| **自动化回归** | 自动化脚本替代手工回归 | 每次提交触发 | 60-80% |
| **冒烟回归** | 仅执行最核心的P0用例 | 每次部署后的快速验证 | 10-20% |
| **混合策略** | 日常自动化冒烟回归 + 每周选择性回归 + 发布前全量回归 | 推荐！| 平衡 |

### 5.3 回归测试的选择策略矩阵

```
变更影响面分析：

                    │ 变化范围广 (多个模块)  │ 变化范围窄 (单个模块)
────────────────────┼────────────────────────┼────────────────────────
变更频率高 (频繁迭代) │ 自动化冒烟 + 全模块回归  │ 影响模块深度回归 + 冒烟
────────────────────┼────────────────────────┼────────────────────────
变更频率低 (偶尔修订) │ 关联模块回归 + 冒烟      │ 仅变更模块回归
```

### 5.4 回归测试自动化的ROI分析

| 用例类型 | 手工执行频率 | 自动化成本 | 建议 |
|----------|-------------|-----------|------|
| 每版本必测的P0用例 | 每次迭代（2周） | 中 | **必须自动化** |
| 仅大版本全回归的P0用例 | 每季度 | 高 | 选择性自动化核心部分 |
| 每次提交都触发的冒烟用例 | 每天 | 低 | **必须自动化** |
| 一次性验证用例 | 1次 | — | 不需要自动化 |

## 6. 冒烟测试与健全测试

### 6.1 冒烟测试 (Smoke Testing)

**定义：** 在软件构建/部署完成后，执行一组最少但最关键的核心测试用例，以快速确认系统的"骨架"是否立得住——如果冒烟测试失败，系统进入"拒绝测试"状态，不再进行深入测试。

**冒烟测试的"最少集"标准：**

```
一个合格的冒烟测试套件应该：
1. 用例数：10-30个（绝不超过50个）
2. 执行时间：< 30分钟
3. 覆盖范围：每个核心模块至少1条用例
4. 通过标准：100%通过，任何一个失败都应阻断后续测试
```

**冒烟测试用例选择方法：**

| 来源 | 选取策略 |
|------|----------|
| **功能测试用例** | 从每个模块选取1-2条最核心的正向流程用例 |
| **API用例** | 主要接口的 "happy path" 请求 |
| **环境检查用例** | 数据库连接、缓存可用、关键配置加载 |
| **第三方依赖检查** | 依赖的关键外部服务是否能连通 |

**冒烟测试在日常流程中的位置：**

```
开发提交代码 → 构建 → 部署测试环境 → 冒烟测试
                                         │
                           ┌─────────────┼──────────────┐
                           ▼              ▼              ▼
                       全部通过        部分失败         全部失败
                           │              │              │
                        继续完整     定位失败原因      版本拒绝
                        功能测试     修复后重新部署    回退
```

### 6.2 健全测试 (Sanity Testing)

**定义：** 在冒烟测试通过后，对变更的具体功能点进行窄而深的验证，确认变更本身是否正确、基本的业务逻辑是否可行。

**冒烟测试 vs 健全测试对比：**

| 维度 | 冒烟测试 (Smoke) | 健全测试 (Sanity) |
|------|-----------------|-------------------|
| **范围** | 宽而浅 (Wide but Shallow) | 窄而深 (Narrow but Deep) |
| **目的** | 确认系统能跑通 | 确认变更功能正确 |
| **触发时机** | 每次部署/构建后 | 特定变更后 |
| **执行顺序** | 先于健全测试 | 冒烟通过后 |
| **用例量** | 10-30个 | 视变更而定 |
| **谁执行** | CI自动触发 | 测试人员 |

**示例说明：**

```
场景：开发修复了"订单详情页金额显示错误"的Bug

冒烟测试 (部署后)：
  ✓ 系统能否正常启动
  ✓ 登录功能是否正常
  ✓ 商品列表能否打开
  ✓ 购物车能否添加商品
  ✓ 订单列表能否正常展示

健全测试 (冒烟通过后)：
  ✓ 进入订单详情页，验证普通金额显示
  ✓ 进入订单详情页，验证大额订单金额显示（千分位）
  ✓ 进入订单详情页，验证优惠后金额显示
  ✓ 进入订单详情页，验证退款金额显示
  ✓ 检查订单列表页的金额显示是否也受影响（关联影响）
```

## 7. 用户验收测试前的前置条件

在将系统交付给用户/业务方进行验收测试(UAT)之前，系统测试阶段必须完成以下所有检查项，确保UAT的质量和效率。

### 7.1 UAT准入检查清单

| 序号 | 检查项 | 通过标准 | 检查方式 |
|------|--------|----------|----------|
| 1 | **功能测试完成** | 所有P0/P1用例100%通过，P2用例通过率>95% | 测试报告审查 |
| 2 | **严重缺陷清零** | P0（崩溃/阻塞）缺陷为0，P1（严重功能缺失）缺陷为0 | 缺陷管理系统 |
| 3 | **已知缺陷已记录** | 所有遗留缺陷已评级、有责任人、有修复计划、UAT环境可复现 | 缺陷清单 |
| 4 | **冒烟测试通过** | UAT环境冒烟测试100%通过 | 冒烟测试报告 |
| 5 | **核心流程E2E通过** | 至少3个核心端到端业务流程全部通过 | E2E测试报告 |
| 6 | **基础性能达标** | 关键接口在预期并发下P95响应时间达标 | 性能测试报告 |
| 7 | **安全扫描通过** | 高危漏洞为0，中危漏洞有修复计划 | 安全扫描报告 |
| 8 | **数据准备就绪** | UAT环境的数据（用户、商品、配置等）已初始化，满足验收场景 | 数据准备清单 |
| 9 | **环境配置同步** | UAT环境配置与生产环境一致（或差异已知且已记录） | 配置对比表 |
| 10 | **测试文档交付** | 测试用例、测试数据、操作手册已交付给UAT团队 | 文档清单 |
| 11 | **用户培训完成** | UAT参与者已接受系统使用培训，了解基本操作和验收流程 | 培训签到 |
| 12 | **回滚方案就绪** | 如UAT发现阻塞性缺陷，有快速修复和重新部署的方案 | 应急预案 |

### 7.2 UAT环境准备标准

| 维度 | 具体要求 |
|------|----------|
| **环境隔离** | UAT环境与开发/测试环境物理隔离，不受开发活动影响 |
| **数据接近生产** | 数据量、数据分布、数据多样性接近生产（脱敏后） |
| **版本锁定** | UAT环境的代码版本冻结，非修复阻塞性缺陷不更新 |
| **日志监控** | 全链路日志可查，关键指标有监控 |
| **用户开通** | UAT参与者的账号、权限已配置完毕 |
| **通知机制** | 环境宕机或异常时能及时通知相关人员 |

**性能测试方案：**

```
测试目标：
1. 验证系统在5000 QPS下的性能表现
```

**性能测试场景设计：**

|----------|----------|----------|---------|----------|------|

| S3 | 压力测试 | 阶梯增长 (1000→10000) | — | 每阶梯3分钟 | 找性能拐点 |
| S4 | 峰值测试 | 500→5000(5秒内) | — | 10分钟 | 验证弹性 |
| S5 | 稳定性测试 | 稳定3000 QPS | 3000 | 8小时 | 检测内存泄漏 |
| S6 | 限流验证 | 稳定6000 QPS | 6000 | 10分钟 | 验证限流生效 |

**场景S1结果分析示例：**

```
| QPS | P50 响应时间 | P95 响应时间 | P99 响应时间 | 错误率 | CPU |
|-----|-------------|-------------|-------------|--------|-----|
| 500 | 15ms | 30ms | 50ms | 0% | 15% |
| 1000| 18ms | 35ms | 60ms | 0% | 25% |
| 2000| 25ms | 50ms | 90ms | 0% | 40% |
| 3000| 35ms | 75ms | 130ms | 0.01% | 55% |
| 4000| 50ms | 110ms| 200ms | 0.05% | 70% |
| 5000| 80ms | 180ms| 350ms | 0.1% | 82% |
| 6000| 150ms| 500ms| 1200ms| 2.5% | 95% ← 拐点
| 7000| 超时大量 | — | — | 15% | 99% ← 崩溃
```

**测试结论：**
- 系统在5000 QPS下表现稳定，P95响应时间 < 200ms，满足目标
- 性能拐点在6000 QPS附近，CPU接近100%，开始出现大量超时
- 限流机制在6000 QPS时生效，正确返回429状态码
- 8小时稳定性测试未发现内存泄漏，GC表现正常

### 案例5：系统可靠性测试 —— 故障注入与恢复

**场景：** 验证微服务系统在基础设施故障下的自动恢复能力。

**故障注入计划（混沌工程）：**

```yaml
测试配置:
  环境: 预发布环境 (Staging)
  工具: Chaos Mesh
  监控: Prometheus + Grafana + 告警通知
  回滚: 立即终止故障注入的脚本

故障场景:
  - name: "数据库主库宕机"
    target: mysql-primary
    action: kill-pod
    duration: 60s
    expected_behavior:
      - 应用应自动切换到从库
```

## 9. 附录：系统测试方法对比总结

| 测试类型 | 核心关注点 | 执行时机 | 关键工具 | 人员 |
|----------|-----------|----------|----------|------|
| **功能测试** | 功能是否按需求实现 | 每个Sprint/迭代 | Selenium, Playwright, Postman | 测试工程师 |
| **性能测试** | 系统速度和稳定性 | 大版本前、架构变更后 | JMeter, Gatling, k6 | 性能测试工程师 |
| **安全测试** | 系统防护能力 | 每个版本、重大变更后 | OWASP ZAP, Burp Suite | 安全测试工程师 |
| **兼容性测试** | 不同环境下的表现 | 大版本前、新平台支持时 | BrowserStack, Selenium Grid | 测试工程师 |
| **可用性测试** | 用户使用体验 | 设计阶段、上线前 | 用户调研, 可用性量表 | UX + 测试工程师 |
| **E2E测试** | 端到端业务完整性 | 每日/Nightly | Playwright, Cypress | 自动化测试工程师 |
| **回归测试** | 变更未破坏已有功能 | 每次变更后 | 自动化测试框架 | 测试工程师 |
| **冒烟测试** | 系统基本可用性 | 每次部署后 | CI/CD Pipeline | CI自动执行 |
| **可靠性测试** | 故障恢复能力 | 上线前、架构变更后 | Chaos Mesh, Gremlin | SRE + 测试工程师 |

---

> **核心要点回顾**：系统测试是检验整个软件系统是否满足需求的综合验证活动。它以功能测试为核心基座，扩展到性能、安全、兼容性、可用性和可靠性多个非功能维度。E2E测试提供了从用户视角的端到端验证，但应严格控制数量（5-10个核心场景），避免出现"冰淇淋甜筒"反模式。回归测试需要采用混合策略：日常自动化冒烟、每周选择性回归、发布前全量回归。冒烟测试（宽而浅）和健全测试（窄而深）是质量控制流水线中的两个关键质量门。系统测试完成后，必须满足12项UAT准入检查，确保软件质量和文档准备就绪，才能进入用户验收测试阶段。
