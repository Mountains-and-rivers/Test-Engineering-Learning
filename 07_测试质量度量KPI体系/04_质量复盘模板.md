# 质量复盘模板 (Quality Review / Retrospective Templates)

> **质量复盘**不是"批判大会"——它是团队从过去的经验（尤其是失败）中学习，系统性地提升质量能力的过程。好的复盘让团队"不重复犯错"，坏的复盘让团队"不敢犯错"。

#### 1.2 迭代复盘敏捷版（站会式复盘）

当迭代较短或变化较少时，可使用精简版：

```
迭代复盘敏捷版 (15分钟)

Round 1: 数据快照 (3分钟)
  - 本迭代发现了多少缺陷？
  - 带到了线上几个缺陷？
  - 用例执行率和通过率？

Round 2: 三问 (每人1分钟，最多10分钟)
  1. 这个迭代的质量上，最满意的一件事？
  2. 质量上，最不满意的一件事？
  3. 如果只改一件事来提升质量，会改什么？

Round 3: 行动 (2分钟)
  - 确定1-2个下迭代的改进项
  - 指定负责人
```

## 4. 复盘方法

### 4.1 Start / Stop / Continue

这是最简单也最常用的复盘方法，适合迭代级别的快速复盘。

**方法框架：**

```
Start (开始做什么 - 我们还没做但应该开始做的)
  ├── 开始做XXX，因为...
  ├── 开始做YYY，因为...
  └── ...

Stop (停止做什么 - 我们在做但不该继续做的)
  ├── 停止做XXX，因为...
  ├── 停止做YYY，因为...
  └── ...

Continue (继续做什么 - 我们已经在做且应该坚持的)
  ├── 继续保持XXX，因为...
  ├── 继续保持YYY，因为...
  └── ...
```

**质量复盘中的Start/Stop/Continue示例：**

```
Start（开始做）：
  - 开始在每个迭代初进行测试策略评审 → 提前对齐测试重点
  - 开始在PR中自动运行覆盖率检查 → 防止覆盖率退化
  - 开始每周统计线上监控数据作为测试输入 → 补充测试场景

Stop（停止做）：
  - 停止在用例评审中只关注数量 → 改为关注覆盖率和有效性
  - 停止跳过非功能需求（性能/安全）的测试 → 改为必须纳入计划
  - 停止用手工方式统计KPI数据 → 改为自动化采集

Continue（继续做）：
  - 继续保持每日缺陷Triage的习惯 → 保证缺陷及时处理
  - 继续保持每迭代的回归自动化维护 → 避免自动化腐烂
  - 继续保持测试左移（需求阶段参与评审）→ 已见成效
```

### 4.2 4L法 (Liked / Learned / Lacked / Longed For)

4L法比Start/Stop/Continue更全面，增加了"学到了什么"和"渴望什么"的维度。

```
Liked（喜欢/满意的地方）:
  - 哪些事情让我们感到满意？
  - 哪些成果让我们引以为豪？

Learned（学到的东西）:
  - 我们从这次经历中学到了什么？
  - 有哪些技术/流程上的新认知？

Lacked（缺乏/不足的地方）:
  - 我们在哪些方面感到力不从心？
  - 缺少了什么资源或支持？

Longed For（渴望/期待的东西）:
  - 我们渴望在下一个阶段拥有/实现什么？
  - 如果有魔法棒，你最想改变什么？
```

**4L法实战示例（版本复盘）：**

```
Liked:
  1. 支付模块重构的回归测试自动化套件充分发挥了作用
  2. 本版本的缺陷响应速度明显提升（修复周期从3.2天降到1.8天）
  3. 团队在性能测试方面的能力有明显进步

Learned:
  1. 重构前编写"特征测试"非常有效——先记录现有行为，再重构
  2. 契约测试对微服务间的接口变更风险控制非常有效
  3. 灰度发布 + 监控告警的组合对线上问题早发现非常关键

Lacked:
  1. 测试环境搭建仍然太慢（平均等半天）
  2. 积分模块的需求文档质量不高，导致测试设计反复修改
  3. 缺少专项的安全测试能力，安全测试依赖外部团队

Longed For:
  1. 一套测试数据自助平台——输入参数就能生成测试数据
  2. 更稳定的测试环境——希望环境问题不再是测试的阻碍
  3. 有更多时间做探索性测试——而不是被机械的回归占据
```

### 4.3 5-Why 根因分析法

5-Why是分析缺陷逃逸和质量问题的核心方法。关键不是恰好问5次，而是追问到根本原因。

**5-Why分析模板：**

```
问题描述：[清晰描述问题——什么、何时、影响范围]

Why 1: [为什么这个问题会发生？]
  → [回答]

Why 2: [为什么上面的回答会发生？]
  → [回答]

Why 3: [为什么上面的回答会发生？]
  → [回答]

Why 4: [为什么上面的回答会发生？]
  → [回答]

Why 5: [为什么上面的回答会发生？]
  → [回答 - 这应该是根本原因]

根本原因：[总结]
改进措施：[基于根本原因制定，非表面修复]
```

**5-Why实战案例：**

```
问题描述：
2026-06-08 订单状态显示错误，影响约2000用户，持续3小时。

Why 1: 为什么订单状态显示错误？
→ 因为前端显示的订单状态和后端数据库中的不一致

Why 2: 为什么前后端状态不一致？
→ 因为后端更新了订单状态码的定义，但前端没有同步更新

Why 3: 为什么后端修改了状态码定义但前端不知道？
→ 因为后端开发者认为这只是一个内部优化，没想到会影响前端

Why 4: 为什么后端开发者不知道会影响前端？
→ 因为没有接口字段的变更影响分析流程，也没有契约文档记录这个字段的使用方

Why 5: 为什么没有接口变更影响分析流程？
→ 因为团队没有建立接口契约管理和变更通知机制

根本原因：
团队缺少接口契约管理流程，接口字段的变更缺乏影响分析和通知机制。

改进措施（基于根因）：
1. 建立接口契约文档（使用OpenAPI/Swagger规范）
2. 接口字段变更必须触发影响分析
3. 引入Pact契约测试，前后端契约变更自动发现
4. 将契约测试纳入CI流水线

（而非表面措施：修复这个具体Bug → 修好了，但下次换个字段还会出现）
```

### 5.1 改进措施的SMART原则

每个改进措施都应该是SMART的：

```
SMART改进措施模板：

S (Specific - 具体的):
  ❌ "提高测试覆盖率" → 太模糊
  ✅ "将支付模块的单元测试行覆盖率从45%提升到75%"

M (Measurable - 可衡量的):
  ❌ "加强代码审查" → 无法衡量
  ✅ "所有PR的代码审查覆盖率检查通过率达到95%"

A (Achievable - 可达成的):
  ❌ "下个迭代实现100%自动化覆盖" → 不现实
  ✅ "下个迭代自动化回归用例数量从120条增加到150条"

R (Relevant - 相关的):
  ❌ "学习Kubernetes部署" → 与当前质量问题无关
  ✅ "学习Playwright以提升Web自动化稳定性"

T (Time-bound - 有时限的):
  ❌ "尽快完成" → 没有截止时间
  ✅ "在Sprint-09结束前(2026-06-28)完成"

## 6. 5个质量复盘实战案例

### 案例1：线上支付故障复盘

**背景：**
某电商平台在促销期间支付服务出现30分钟故障，导致约5000笔订单支付失败。事后进行深度复盘。

**复盘过程摘要：**

```
时间线回顾：
14:00 - 大促开始，流量进入峰值
14:05 - 支付服务响应时间开始升高
14:10 - 支付渠道A开始返回超时
14:12 - 自动切换到支付渠道B（备用渠道）
14:15 - 支付渠道B因为未预热也出现超时
14:20 - 支付服务整体不可用
14:25 - 手动介入，重启支付服务
14:30 - 支付服务恢复
14:35 - 确认所有渠道恢复正常

5-Why分析：
  Why1: 为什么支付服务不可用？→ 主备渠道都超时
  Why2: 为什么备用渠道也超时？→ 备用渠道切换后无预热，直接承受全量流量
  Why3: 为什么没有预热？→ 切换逻辑中没有预热步骤
  Why4: 为什么切换逻辑缺少预热？→ 降级方案设计时未考虑冷启动场景
  Why5: 为什么设计评审没发现？→ 评审清单缺少"降级方案冷启动"检查项

改进措施：
  1. 降级切换增加预热逻辑（技术改进）
  2. 降级方案评审清单增加"冷启动"检查项（流程改进）
  3. 每季度进行一次全链路故障演练（组织改进）
```

### 案例2：测试遗漏的深度复盘

**背景：**
一个关于订单金额计算的Bug逃逸到了线上，影响到了部分用户的退款金额。测试阶段为什么没有发现？

**复盘分析：**

```
缺陷详情：
  订单金额 = 商品金额 - 优惠券 - 积分抵扣 + 运费
  当订单中同时使用了"满减券"和"积分抵扣"时，
  积分抵扣的计算使用了"优惠后的金额"而非"商品原价"，
  导致积分多扣。

测试阶段为什么没发现？

测试用例覆盖分析：
  商品金额 - 优惠券:        ✅ 有覆盖
  商品金额 - 积分抵扣:      ✅ 有覆盖
  商品金额 - 优惠券 - 积分: ✗ 无覆盖 ← 组合场景遗漏
  运费计算:                ✅ 有覆盖
  全部组合:                ✗ 无覆盖 ← 完全组合爆炸，不可能全覆盖

根因分析：
  Why1: 为什么组合场景遗漏？→ 测试设计时未识别到这个组合
  Why2: 为什么未识别到？→ 等价类划分时，把"优惠券"和"积分抵扣"分到了
                          不同的等价类，认为它们互不影响
  Why3: 为什么认为它们互不影响？→ 对业务规则理解不够深入
                          优惠券影响商品金额，积分抵扣基于商品金额→互相影响

改进措施：
  1. 对涉及金额/计算的多个因素，使用Pairwise（成对组合）方法设计测试
  2. 建立"金额计算类"需求的专项评审检查清单
  3. 与产品确认所有涉及金额计算的业务规则并文档化
  4. 在单元测试层面补充组合场景（成本更低）
```

### 案例3：版本质量复盘驱动流程优化

**背景：**
某团队连续3个版本DRE低于95%，线上缺陷频发。通过版本复盘驱动了系统性的流程优化。

**3个版本的数据对比：**

```
版本     DRE    线上S1+S2   发布后Hotfix次数
V2.1    94.5%      3            2
V2.2    93.8%      4            3
V2.3    94.0%      2            2

触发复盘的根本问题：
  为什么连续3个版本DRE不达标？

复盘发现的问题链：
  1. 版本周期被压缩 (从4周到3周)
  2. 但测试策略没有相应调整 (还是按4周设计)
  3. 导致测试时间不够 → 牺牲了兼容性测试和异常场景测试
  4. 而这些正是线上缺陷的主要来源

改进措施：
  1. 测试策略评审前置：版本计划期就评审测试策略
  2. 基于风险的测试范围调整：时间不足时，主动评估哪些测试可以裁剪
  3. 测试自动化加速：投入资源将最耗时的回归测试自动化
  4. 版本周期与测试策略强制关联：版本周期压缩超过20%时必须重新评审测试策略

效果：
  V2.4版本DRE 96.5%，线上S1+S2 = 0
```

### 案例4：跨团队质量复盘

**背景：**
某项目涉及3个团队（前端、后端、数据平台），线上故障频繁但复盘发现各自团队都没有"明显错误"。问题是接口之间的交互造成的。

**复盘方法：跨团队联合复盘**

```
参与方：
  - 前端团队 (Web + 移动端)
  - 后端团队 (订单 + 支付 + 用户)
  - 数据平台团队
  - 测试团队
  - 运维/SRE团队

复盘工具：系统故障树分析 (Fault Tree Analysis)

顶层事件：订单数据不一致
    ├── 分支1: 前端展示与后端数据不一致
    │     ├── 原因1.1: 缓存未刷新 (后端更新了缓存策略)
    │     └── 原因1.2: API版本不匹配 (前端用旧版API)
    │
    ├── 分支2: 数据库与缓存不一致
    │     ├── 原因2.1: 缓存更新失败未重试
    │     └── 原因2.2: 主从同步延迟 + 读写分离未正确处理
    │
    └── 分支3: 数据平台延迟
          ├── 原因3.1: 数据管道ETL延迟
          └── 原因3.2: 实时与离线数据口径不一致

跨团队改进措施：
  1. 建立"接口契约"三方签字机制（前端、后端、测试）
  2. 合约测试覆盖所有跨团队接口
  3. 数据一致性监控告警（而非各团队独立监控）
  4. 每月一次跨团队质量同步会
```

### 案例5：从复盘到知识库沉淀

**背景：**
团队的复盘发现总是类似的问题重复出现。"每次复盘都得出同样的结论"说明改进没有被固化为知识。

**解决方案：建立"质量知识库"**

```
复盘 → 知识库转化流程：

  复盘报告
    │
    ├→ 提取: 根本原因 (Root Cause)
    │
    ├→ 分类: 放入知识库对应分类
    │     ├── 需求缺陷类
    │     ├── 设计缺陷类
    │     ├── 编码缺陷类
    │     ├── 测试遗漏类
    │     ├── 环境/配置类
    │     └── 第三方依赖类
    │
    ├→ 连接: 关联到对应的流程检查清单
    │     例如: "金额计算组合场景遗漏" → 关联到
    │          "测试用例设计检查清单-金额类需求"
    │
    ├→ 发布: 团队内分享 (技术周报/知识库)
    │
    └→ 验证: 下个类似需求时检查清单是否生效

知识库条目示例：

## 总结

质量复盘的核心原则：

| 原则 | 说明 |
|------|------|
| **对事不对人** | 复盘的目的是找到系统性问题，而非追究个人责任 |
| **数据说话** | 基于客观数据，而非主观感受 |
| **追根究底** | 不只停留在表面原因，使用5-Why找到根因 |
| **可行动** | 每个改进措施都是SMART的，明确负责人和截止日期 |
| **闭环验证** | 改进措施完成后验证效果，形成知识沉淀 |
| **节奏合理** | 迭代级(2周)、版本级(1-3月)、年级(年度)层层递进 |
